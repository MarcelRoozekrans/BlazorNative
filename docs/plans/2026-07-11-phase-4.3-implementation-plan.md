> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** one command from edit to rendered tree (M4 DoD #5): `make devloop` watches the
.NET source, republishes win-x64, and re-runs the new Kotlin `PreviewHost` (console tree
dump); `make devloop-android` does publish → install → launch → logcat on a device — all
round-trip times measured and recorded. Fast-restart, not hot-reload, documented.

## Gate 1 — PreviewHost (JVM, TDD)

### Task 1: TreeSnapshot + PreviewHost + gradle task

1. TDD `TreeSnapshotTest.kt` (JVM): synthetic patch sequences → expected indented text —
   create (parent/child order), text set/replace, props, styles, InsertIndex mid-list
   (bucket-local semantics like the WidgetMapper pins), remove (subtree gone), multi-frame
   accumulation. Then implement `TreeSnapshot.kt` (main/kotlin, pure JVM).
2. `PreviewHost.kt` — thin `main(args)`: component from args (default BnDemo), boot via
   `BlazorNativeRuntime`, drain mount frames into TreeSnapshot, print tree + stage
   timings (load/init/mount/total ms), exit 0 / non-zero + stderr on failure. Placement:
   main/kotlin acceptable (document the APK dead-code note) or a JVM-only arrangement —
   implementer's call, documented.
3. Gradle `runPreviewHost` JavaExec: same `jna.library.path` as the test JVM,
   `-Pcomponent=` pass-through, depends on nothing .NET (the script owns publishing).
4. Manual smoke: `gradlew runPreviewHost` against the current publish → BnDemo tree
   visible. Record the output in the report.
5. Full JVM suite `--rerun` → record count (34 + new); update ci.yml JVM assertion in
   the SAME commit. `dotnet test` must stay 203/0 (no .NET change — verify).
6. Commit: `feat(phase-4.3): PreviewHost — console tree dump over the native dll (TDD)`.
