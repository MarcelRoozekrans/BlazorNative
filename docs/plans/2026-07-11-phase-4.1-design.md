## 1. Survivors — rescoped

Category strings replace `BlazorNative.WasiCompatibility`:
**`BlazorNative.MobilePolicy`** (advisory architecture policy) and **`BlazorNative.Interop`**
(boundary correctness). Help links move off the dead `github.com/your-org/...` to
`https://github.com/MarcelRoozekrans/BlazorNative/blob/main/docs/analyzers.md#bn00xx`
(one new doc page, one anchor per rule — created this phase).

| ID | New severity | Rescoped meaning (all wording rewritten off the WASI premise) |
|---|---|---|
| **BN0004** ThreadSleep | Warning (was Error) | MobilePolicy: `Thread.Sleep` blocks a runtime thread — on the single Kotlin dispatch lane this stalls every queued frame/event; prefer `await Task.Delay`. |
| **BN0010** Sockets | Warning (was Error) | MobilePolicy: raw `System.Net.Sockets.*` bypasses the host bridge — network I/O should ride `IMobileBridge.FetchAsync`/`AddBlazorNativeHttp` so the host owns permissions, proxies, and lifecycle. |
| **BN0011** HttpClient | Warning | MobilePolicy: **narrowed to the parameterless ctor only** — `new HttpClient()` uses the socket handler; `new HttpClient(handler)` stays legal (BlazorNative.Http itself constructs over `BridgeHttpHandler`). Message: prefer the injected client from `AddBlazorNativeHttp()`. |
| **BN0013** Process | Error | MobilePolicy→correctness: `System.Diagnostics.Process` is unsupported on Android app sandboxes; still flagged as error. |
| **BN0014** AsyncBridgeHandler | Error | Interop: wording drops "Mono-WASI single-threaded scheduler"; the surviving truth: `NativeEvents` handlers run synchronously inside a native callback window (DevHost multicast is sync; the production lane is the single-threaded dispatch lane) — an `async` handler becomes fire-and-forget whose continuation escapes the window; exceptions vanish. Same detection logic; note in the rule doc that `NativeEvents`' own redesign is a ledgered open item (NativeShellBridge stubs it no-op). |

## 2. New rules — the real NativeAOT constraint surface

Both **Error**, category `BlazorNative.Interop`, syntactic + cheap:

- **BN0020 — exceptions must not escape `[UnmanagedCallersOnly]`.** A method carrying the
  attribute whose body can complete exceptionally without a catch-all (`catch` with no filter
  or `catch (Exception)`) is flagged. An exception crossing the C-ABI boundary is UB under
  NativeAOT (process abort on Android). This pins the pattern `Exports.cs` already follows
  everywhere (rc-code + capture-slot contract) so the 9th export can't forget it.
  Heuristic scope (documented): flags only method bodies whose top-level statements are not
  fully wrapped; expression-bodied UCO methods that call a single `SafeX(...)` wrapper are
  fine if the wrapper call itself is inside a try — to keep the rule honest and simple, the
  rule requires the top-level try/catch shape; `Exports.cs` conforms today (verify at Gate 1;
  if it doesn't everywhere, fixing Exports.cs to conform is in-scope and behavior-neutral).
- **BN0021 — explicit cdecl on exports.** A `[UnmanagedCallersOnly]` method missing
  `CallConvs = new[] { typeof(CallConvCdecl) }` or missing `EntryPoint =` is flagged. The
  JNA side binds by name to cdecl; an implicit-callconv or unnamed export is a silent ABI
  drift vector. (The C# compiler already enforces static + blittability — CS8894/CS8932
  family — so those need no analyzer.)

## 3. Attachment + zero-warning bar

- All five src/ projects get the analyzer ProjectReference. The solution build must end
  **0 BN warnings** — any legitimate in-library use gets a scoped
  `#pragma warning disable BNxxxx` with a justification comment (expected: none; BN0011's
  narrowing and BN0012's deletion remove the known would-be hits; `DevHostBridge`/
  `BlazorNative.Blazor` subscribe synchronously today).
- `BlazorNative.Blazor` (the browser DevHost app) is a src/ project too — include it; its
  `Home.razor` sync subscription must stay green under BN0014 (it's the rule's original
  Phase 2.0 target).

## 4. Tests + release tracking

- `Microsoft.CodeAnalysis.Testing` (harness exists; patterns per `BridgeAsyncHandlerAnalyzerTests.cs`:
  Net90 reference assemblies, inline `BridgeStub` for Core types, `{|BNxxxx:...|}` markup).
- Per surviving/new rule (7 rules): fires-on-bad, silent-on-good, plus rule-specific edges
  (BN0011: handler-ctor negative; BN0014: the 3 existing tests stay; BN0020: wrapped body
  negative + rethrow-inside-catch positive-or-documented; BN0021: full attribute negative).
  Existing 3 BN0014 tests keep passing (wording change must not break the markup tests —
  they assert IDs, not messages; verify).
- **Release tracking:** create `AnalyzerReleases.Shipped.md` (a `1.0.0-phase-4.1` release
  block: 6 removed rules recorded as removed, survivors recorded with changed
  category/severity, 2 new rules) + `AnalyzerReleases.Unshipped.md`; **drop the RS2008
  `<NoWarn>`**. Fix the stale desynced header comment in `WasiBclGapsAnalyzer.cs` (renamed —
  see below).
- File/class renames to match the new world: `WasiThreadingAnalyzer` shrinks to the BN0004
  survivor → merge survivors into a renamed `MobilePolicyAnalyzer.cs`; `WasiBclGapsAnalyzer`
  merges likewise; `BridgeAsyncHandlerAnalyzer` stays; new `InteropBoundaryAnalyzer.cs`
  (BN0020/BN0021). One-type-per-file where practical (M6 ledger nods).

## 5. CI + baselines

- Attaching analyzers changes nothing in CI *mechanics* (they ride `dotnet build`), but the
  **.NET test-count assertion in `ci.yml` must be updated** in the same PR (the analyzer
  test project grows from 3 tests to ~20+; exact number recorded at the gate).
- Version string stays `1.0.0-phase-3.5` — it tracks the runtime binary, which this phase
  does not change. (JVM/instrumented baselines untouched: 32 / 32.)
