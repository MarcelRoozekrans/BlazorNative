## Gate A — #120: the reported runtime version is governed, not stale *(DoD #3)*

`Exports.VersionNumber` was `"1.4.0-phase-5.4"` — an ungoverned 8th literal, consumer-visible
via the version export string, the `PlatformInfo` JSON, and `GetPlatformInfoAsync`'s
`AppVersion` — so a consumer read a version four milestones stale.

- **Red-first:** `PackageVersionPinTests.TheRuntimeVersionExport_AgreesWithTheProps` asserts
  `Exports.VersionNumber == <props Version>`. It **FAILED** on `1.4.0-phase-5.4 ≠ 0.1.0` and
  passes after.
- **Fix:** value → `0.1.0` with the `// x-release-please-version` annotation (mirroring the
  props convention); `src/BlazorNative.Runtime/Exports.cs` added to
  `release-please-config.json` `extra-files`. It is now a **governed mirror** — every release
  bumps it in lockstep with the package (the 0.1.1 release PR will move it to 0.1.1), and the
  drift pin reds in `build-test` if it ever falls behind. The reported version string changes
  from `BlazorNative.Runtime 1.4.0-phase-5.4` to `BlazorNative.Runtime 0.1.0`.

## Gate B — #122: the load-bearing runtime-pack version is drift-guarded *(DoD #4)*

`RuntimeFrameworkVersion 10.0.9` — the pin that makes bionic/iOS NativeAOT compile at all —
lived in 3 csproj spots (sample ×2, template ×1) with no test linking them, while every
cosmetic literal was pinned.

- **Fix:** `PackageVersionPinTests.EveryRuntimeFrameworkVersion_AgreesAcrossSampleAndTemplate`
  parses every occurrence, derives the canonical value from occurrence #1 (no hardcoded
  `10.0.9` second copy), and asserts the rest agree (non-vacuity floor ≥ 3).
- **Mutation-proven it bites:** setting one occurrence to `10.0.99-MUTANT` failed the pin
  (offender named); reverted clean.
