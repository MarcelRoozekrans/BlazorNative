# Phase 2.4 Streaming-Path Investigation

**Date:** 2026-05-27
**Status:** complete (rung-1 validated)
**Scope:** identify a streaming transport mechanism for Phase 2.5+ render-frame flow.

The Phase 2.4 design ships one-shot snapshot transport (host parses captured
stdout after `wasi:cli/run.run` returns). This investigation finds the
cheapest viable mechanism for Phase 2.5+ to add streaming — host sees
`[FRAME]` lines as they happen during the run, not after `func_call` returns.

## Investigation ladder

Work down until one rung passes on JVM. Each rung gets a recorded result
with concrete evidence (test code, observed timestamps, error messages).

### Rung 1 — Tee'd stdout (cheapest) — **PASS**

**Hypothesis:** `wasi_config_set_stdout_file` (and equivalently
`wasmtime --stdout=<file>` redirection on the CLI) flushes line-by-line; a
background poller can read the file while the run is in progress.

**Test:** `src/BlazorNative.Jni/src/test/kotlin/io/blazornative/jni/StreamingSpike_Rung1Test.kt`

The test launches `wasmtime.exe` as a subprocess with stdout redirected to
a temp file via `ProcessBuilder.redirectOutput(stdoutFile)`. A background
Kotlin thread polls the file every 10ms, recording `System.nanoTime()` the
first time it sees one and then two `[FRAME]` substrings. The main thread
records `System.nanoTime()` when `process.waitFor()` returns. PASS = the
first-frame timestamp precedes the run-completion timestamp.

**Setup:** the sentinel emits in milliseconds — too fast for tee'd observation
without prolonging the run. Solution: gated a second `renderer.Mount<_BridgeFrameSelfTest>()`
+ `Thread.Sleep(200)` in `WasiEntryPoint.Main` behind env var `BLAZOR_STREAMING_SPIKE=1`
(the test sets it via `--env BLAZOR_STREAMING_SPIKE=1`). Two `[FRAME]` lines
emit ~200ms apart; the second is the load-bearing observation because it
proves the host saw a `[FRAME]` that the wasm hadn't yet committed when the
first one appeared.

**Result:** **PASS** — across three back-to-back runs the poller consistently
observed both `[FRAME]` lines before wasmtime exited.

| Run | First frame before exit | Second frame before exit |
|-----|------------------------:|-------------------------:|
| 1   | 221.7 ms                | 7.7 ms                   |
| 2   | 221.9 ms                | 2.4 ms                   |
| 3   | 214.7 ms                | 14.2 ms                  |

Reading: the first `[FRAME]` line was already visible on the host ~215-222 ms
before the wasmtime process exited (matches the 200 ms spike sleep + ~15-22 ms
mount/render overhead). The second `[FRAME]` line was observable 2-14 ms
before exit — the poller catches it within one or two 10 ms tick cycles after
the wasm writes it, well before `wasi:cli/run.run` unwinds and exits the process.

**Latency observed:** ≤ ~10-15 ms from wasm `Console.WriteLine` to host visibility
(bounded by the poller tick interval of 10 ms). This is the natural ceiling
on Rung 1 streaming latency; the poll interval is configurable and could be
tightened to 1-5 ms if frame budget demands it.

**Sample captured stdout** (run 1, full 808 bytes):
```

[BOOT] di-ok bridge=WasiBridge renderer=NativeRenderer
[BOOT] event-ok fired=True name=self-test payload=phase-2.0
```

**Notes on the mechanism:**

- The Kotlin test uses the wasmtime **CLI subprocess** for simplicity (no JNI
  ceremony). But the in-process JNA path in `WasiHost.kt` uses
  `wasi_config_set_stdout_file` — the same underlying file-backed stdout
  primitive that the wasmtime CLI's `>file` redirection turns into. The
  flush semantics observed here therefore transfer directly to the in-process
  flow: any consumer that opens a separate `FileInputStream`/`RandomAccessFile`
  pointed at the same file can stream lines line-by-line.
- `Console.WriteLine` in .NET on Mono-WASI ends each line with a flush of
  the underlying `wasi_fd_write` — we added explicit `Console.Out.Flush()`
  calls in the spike to make this deterministic, but no flush was needed
  to make Rung 1 pass (the default newline-triggered flush is enough).
- The test launches wasmtime as a subprocess; both stdout and stderr are
  merged via `redirectErrorStream(true)` so a Mono crash would still be
  captured.

## Recommendation for Phase 2.5+

**Rung 1 is viable.** Phase 2.5+ should build streaming on top of tee'd
file-backed stdout (`wasi_config_set_stdout_file`) with a background polling
thread reading the same file.

### Concrete next steps for Phase 2.5+

1. **WasiHost host-side change:** instead of (or in addition to) reading the
   stdout file after `wasmtime_component_func_call` returns, kick off a
   background polling task at the start of `runWasm` that tails the file
   and dispatches `[FRAME]` lines to `handlers.onFrame` as they arrive.
   Join the polling thread after `func_call` returns and parse any tail
   bytes the poller hadn't yet seen.
2. **Latency budget:** 10 ms poll interval keeps end-to-end frame visibility
   under ~15 ms. For a 60 Hz target (16.6 ms/frame) this is workable; for
   higher refresh, drop the interval to 1-5 ms or switch to a pipe-based
   transport (Rung 2 — not needed for current targets).
3. **Backpressure / partial-line handling:** the poller may read a half-written
   `[FRAME]` line. Use `FrameStreamParser` (which already exists for the
   one-shot path) on accumulated bytes, advance the file offset only past
   complete newline-terminated frames, and re-attempt on the next tick.
4. **Android consideration:** Android's `wasi_config_set_stdout_file` uses
   the same libwasmtime path. The poller on JVM uses the standard
   `java.nio.file.Files` API — translates 1:1 to Android. Recommend
   replicating `StreamingSpike_Rung1Test` as an Android instrumented test
   in Phase 2.5 (using `context.cacheDir` for the stdout file, same as the
   one-shot tests already do).

### What this does NOT validate

- We did not confirm pipe-FD streaming (Rung 2) is faster or lower-overhead
  than file polling. The file polling overhead is one syscall per 10 ms, which
  is negligible — but if Phase 2.5+ measurements show poll overhead under high
  frame rate, Rung 2 remains an unexplored optimization.
- We did not test on Android. The host primitives are the same (`wasi_config_set_stdout_file`
  → file FD inside the WASI sandbox), but per-platform validation should be
  part of Phase 2.5 acceptance.
