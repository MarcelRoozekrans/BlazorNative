## What shipped

**The permission pattern, on a bridge that grew for it.** A permission-gated call now flows
request → OS prompt → grant/deny → completion callback, and **denial reaches .NET as data —
a status integer, never an exception, never a hang**. The machinery is generic (an op-enum +
flat-JSON args on one struct slot; a capability-agnostic completion export), proven **headless
first** on `DevHostBridge`, then on both device shells. `docs/bridge-extension.md` grew
**section (f)** — the reusable pattern 9.1/9.2/9.3 copy, with geolocation as its worked example.

**Geolocation, in a new 7th package.** `BlazorNative.Device` houses the `IGeolocation` facade,
a thin delegate over `IMobileBridge.GetCurrentPositionAsync` — which stays in **Core** so
`DevHostBridge` can mock the tri-state result headless. Both shells implement the real flow:
**Android** `LocationManager` + `requestPermissions` (the no-Play-Services path `adb emu geo fix`
feeds directly); **iOS** `CLLocationManager` when-in-use, `NSLocationWhenInUseUsageDescription`
in `Info.plist`. The sample app grows a **`/geolocation` demo page** (`BnGeolocationDemo`,
the 10th routed page).

## THE HEADLINE — the ABI grew for the first time since Phase 3.1, deliberately and argued

**The bridge has been frozen since Phase 3.1.** Clipboard/share (5.4) proved a host capability
could live behind a struct slot without touching the export set. Every phase since has extended
the wire *vocabulary* (node types, event names, flat-JSON fields) on the existing exports and
left the ABI's shape untouched. **Phase 9.0 is the first phase to add an export since 3.1 —
because an honest async-permission completion cannot ride any existing one, and we could not
find a reading of the existing surface that made it fit.**

The three candidates and why each fails:

- **`fetch_complete`** is **fetch-typed** — its payload is an HTTP response (status code, headers,
  body). A permission result is a different shape, and overloading it would make "the user said no"
  masquerade as an HTTP outcome. Rejected.
- **`host_event`** is **contractually synchronous** — it drives `StateHasChanged` inline on the
  dispatch lane. A permission completion arrives on a **background thread after an arbitrary
  suspension** (the OS held the app to show its dialog); routing it through `host_event` would
  violate that export's threading contract. Rejected.
- **Polling** re-invents a push the bridge already knows how to do (`fetch_complete` is exactly a
  deferred push keyed by requestId). Rejected as a step backward.

**So the ABI grows — and grows GENERICALLY, paid once.** The rule the design set and the review
verified: **9.0 pays the export-grow event once, and 9.1/9.2/9.3 reuse it with ZERO further
struct/export/gate/drift change.**

- **Bridge struct 72 → 80 bytes** — one new slot, `HostCallBegin` at **offset 72** (the
  `FetchBegin` twin: an op-enum selects the capability, flat-JSON carries the args). Verified:
  `BridgeProtocolNativeTests` asserts `Marshal.SizeOf<BlazorNativeBridgeCallbacks>() == 80` and
  `OffsetOf(HostCallBegin) == 72`, the ten prior offsets unchanged.
- **Exports 9 → 10** — `blazornative_host_call_complete` (the `fetch_complete` twin: `requestId`,
  `int status`, optional flat-JSON payload). Contract, from its own doc comment: **0 = delivered ·
  1 = unknown/already-completed id (benign cancellation race, ignore) · 2 = internal bridge
  failure (log loudly, detail on stderr)**. **Never throws across the ABI.**
- **No geolocation specifics anywhere in the struct or the export.** The op-enum and the flat-JSON
  are the whole capability surface; even request-vs-check `mode` rides the JSON. This is the
  **"pay once, reuse thrice" thesis**, and the Gate 1 review confirmed it holds — 9.1 adds an op
  constant + a host handler and touches no struct, no export, no gate, no drift pin.

**Every hard-coded-9 site moved in lockstep** — the grow is only honest if every gate that pinned
the old count moved with it: `ci.yml` ×4 (win-x64, bionic-x64, bionic-arm64, the iOS archive nm
gate RID arrays), `ios.yml`, `android-instrumented.yml`, the template-smoke, `BootSmokeNativeTest`
(9 → 10), and the three-mirror drift pin (72 → 80 on .NET / Kotlin / Swift).

**A stale shell fails LOUDLY, for the new capability only.** A 72-byte shell run against the
80-byte runtime does not silently misread the new slot — the `RequireSlot` guard throws
`NotSupportedException` **for geolocation and nothing else**; every prior capability keeps working.
Tested (the struct-72 mutations below).

## The permission model, as proven (DoD #1)

**Denial is DATA — the milestone's law.** The completion status is a **wire-mirrored 6-value
enum** — `Granted` / `Denied` / `DeniedPermanently` / `Restricted` / `LocationUnavailable` /
`Error` (0..5), mirrored byte-for-byte across .NET / Kotlin / Swift. **Every terminal outcome
resolves the awaiting `ValueTask` with a status** — never an exception thrown across the C
boundary, never a dropped completion, never a hang. A test drives all six through `DevHostBridge`
within a **bounded await**; the same is proven on both device shells (below). A non-`Granted`
status carries a **null payload and never parses** — `ParseGeolocationResult` returns the status
alone before it ever touches the JSON.

**The pending-call registry** (`s_pendingHostCalls`) is the `s_pendingFetches` twin:
process-scoped, an `Interlocked` id generator, `RunContinuationsAsynchronously`, the
`CancellationToken` threaded through (kill-during-prompt → `TrySetCanceled`), and an unknown id is
**benign** (return 1, never a throw).

**The named risk — surviving the OS suspending the app — is PROVEN:**

- **Android** recreates the Activity mid-prompt. The .NET registry is process-scoped and never
  notices; the only host-side state that must survive is the `requestCode → requestId` map, so it
  is a **static** field on `AndroidShellBridge` (a per-instance map would die with the Activity that
  opened the dialog). A recreated Activity builds a fresh bridge that still routes the OS result to
  the same in-flight `requestId`. Proven on the AVD: the pending request survives
  `scenario.recreate()` and the recreated Activity completes the **same** .NET continuation
  (`host_call_complete` rc 0).
- **iOS** routes async `CLLocationManager` authorization by `requestId` the same way; the shell
  holds the handler for the app lifetime (the delegate is a **weak** reference — the 5.3/7.3
  retention lesson restated for CoreLocation), so nothing can be deallocated while the prompt +
  fix are in flight.

`docs/bridge-extension.md` **section (f)** records this as the reusable pattern.

## Geolocation (DoD #2)

`IGeolocation` (in `BlazorNative.Device`) is a thin facade over
`IMobileBridge.GetCurrentPositionAsync` — the typed surface stays in **Core** on purpose, so
`DevHostBridge` mocks the tri-state headless (`GeolocationStatus` + `GeolocationPosition`
settable). The fix payload key set is pinned **exactly** — `lat / lng / accuracy / altitude /
timestamp` (`lng`, not `lon`) — the same set `NativeShellBridge.ParseGeolocationResult` reads and
both shells write; a unit test reddens if a shell drifts a name. Android obtains the fix from the
platform `LocationManager`; iOS via a one-shot `requestLocation`. When-in-use only;
background/always is out of scope for M9.

## A milestone-first worth recording — the first off-lane TCS completion the iOS shell has done

Until now the iOS shell had **never** completed an async TCS off the dispatch lane. `fetch`'s iOS
path fails **synchronously** (the stub), and `host_event` is **undeclared** on iOS. The grown ABI
exercised a path the shell **did not previously have**: `blazornative_host_call_complete` arrives
on a ThreadPool continuation, renders, and the frame hops to main via the **existing** off-lane
`CommitFrame`. It works — proven by the two BOOT tests that drive `/geolocation` through the real
`host_call_complete` into a live .NET continuation (a grant echoes `fix:lat,lng`, a denial echoes a
status, both within a bounded await, rc 0 proving the routing, no hang).

## Verification honesty — the PROVEN / UNPROVEN split

**PROVEN on CI:** the emulator's mocked-`LocationManager` Granted fix + the mocked denial +
Activity-recreation survival (Android AVD); the simulator's mocked-`CLLocationManager` fix + the
mocked denial (iOS). The 80-byte/10-export ABI, its offset pins, and the loud-fail of a stale shell.
The full permission-completion path — request → prompt → grant/deny → completion → await resolved —
on both shells within a bounded await.

**UNPROVEN until the owner's physical Android phone** — named, not smuggled:

- **The real GPS fix** — CI feeds `adb emu geo fix`, a set coordinate, not a satellite fix.
- **The real system permission-dialog UX** — both shells **bypass the real OS dialog gesture in
  CI**: it is not drivable in Android instrumentation or hosted XCTest, so the shells drive the
  delegate/result callbacks directly behind an authorization-status override + a suppress switch
  (production CoreLocation / `requestPermissions` calls are unsuppressed and untouched). **The
  design's split named exactly this.**
- **`DeniedPermanently` across real restarts** — the "don't ask again" / Settings round-trip is a
  real-OS behaviour no mock reproduces.

**iOS real-device stays DEFERRED** — the Apple Developer account is the trigger (a TestFlight phase
opens when the owner acquires it), exactly as M5–M8 shipped.

## Ledger out

- **M4 — the malformed-Granted parse fault.** A malformed *Granted* payload surfaces as a
  `FormatException` on the awaiting side: `ParseGeolocationResult` calls `ParseFlatJsonObject`, which
  throws `Malformed` on bad JSON. This is **NOT a denial-as-data violation** — non-`Granted` carries a
  null payload and never parses, and an empty/partial Granted degrades to zeros (`double.TryParse`).
  It is a **host-contract violation on the success path** (a host that claims Granted but sends
  garbage), and it throws to the caller instead of being contained the way `CompleteFetch` contains
  its own parse faults. A future hardening pass moves the parse inside a fault boundary. One line, one
  item.
- **A two-concurrent-in-flight-host-calls test** (Gate 1 review nice-to-have) — cheap insurance for
  9.1+, which adds ops that could be in flight at the same time as geolocation. Not needed for the
  single op 9.0 wires; worth landing before the second op does.
