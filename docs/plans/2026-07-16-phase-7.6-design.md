## The route surfaces as they actually are (verified, not recalled)

Four surfaces carry page knowledge today:

1. **`HostSession.s_components`** (`src/BlazorNative.Runtime/HostSession.cs:44`) — the
   mount registry, name → `Func<NativeRenderer,int>` (a statically-rooted `Mount<T>`
   lambda per row, the trim-law shape). **14 entries**: the nine routed pages + five
   test probes (`HelloComponent`, `CompositionProbe`, `FocusProbe`, `HostEventProbe`,
   `ClipboardProbe`). Its own header calls it "the single component-name authority."
2. **`NativeNavigationManager.s_routes`** (`NativeNavigationManager.cs:37`) — route →
   name, **9 entries** (`"/"` → `BnDemo` through `"/imagepolish"` →
   `BnImagePolishDemo`). Every VALUE is a hand-repeated mount-registry key; every phase
   since 6.1 has appended a row here AND a KDoc paragraph promising the Kotlin mirror
   moves in the same gate.
3. **`MainActivity.DEEP_LINK_COMPONENTS`** (`MainActivity.kt:116`) — the Android shell's
   deep-link route → name map, **8 entries** (everything but `"/"`; the default rides
   the separate `?: "BnDemo"` fallback literal at `MainActivity.kt:217`). It exists for
   a structural reason recorded in 5.1: the shell resolves the deep-link component at
   **Intent-parse time, before the runtime boots**, and .NET's startup-route honor only
   fires on a session's FIRST mount — which never holds under Activity recreation or the
   shared instrumented session.
4. **iOS has NO route table at all** — verified: `BnRuntime.start(component:)` mounts by
   NAME, there is no deep-link surface, no route map, nothing to drift (the 7.5 Gate 3
   report recorded it; the grep confirms it). The day iOS gains a deep-link story, it
   gains a mirror AND joins the pin below — stated here so that day is a decision, not
   an accident.

What already guards this: `NavigationTests.EveryRoute_ResolvesToAComponentTheMountRegistryKnows`
(route values ⊆ registry keys — the 6.3 typo-catcher) and the nine-page ordered baseline
literal (`NavigationTests.cs:294`); each demo page's Android test mounts its route once
on-device ("so the row itself is asserted"). What guards **nothing**: Kotlin's map
against .NET's table. A pair that drifts (`"/form"` → the wrong page) fails no compile,
no test, no lane — a deep link quietly opens the wrong screen on one platform. That is
the exact silent-cross-shell-drift class the style tables closed in 6.1, and the fix is
the same fix.

## Decisions (recorded at phase-open; #1 flagged for veto)

### 1. One .NET manifest; Kotlin's map becomes a PINNED MIRROR, never a derivation ⚠

*The call to veto: manifest-and-drift-test (a) vs. generate/transmit the table (b).*

- **(b) generate or transmit** — rejected twice over. *Transmit* is structurally
  impossible for the one consumer that matters: `DEEP_LINK_COMPONENTS` is consulted at
  Intent-parse time, **before the `.so` is even loaded** — there is no runtime to ask,
  and giving the frozen 72-byte bridge a table channel is a wire change this phase
  forbids. *Generate* (a Gradle task emitting Kotlin from a checked-in JSON, or parsing
  the `.cs`) buys same-commit lockstep at the price of build machinery in the one file
  every instrumented test touches, a generated-sources dependency in `MainActivity`, and
  a THIRD serialization of an 8-row map — overkill for a POC, said honestly. The drift
  test buys the identical guarantee (drift fails the required lane in the commit that
  causes it) with zero machinery, and it is the established house pattern: the style
  tables, the frame tables, the fixture constants, the Yoga version pin are ALL held
  this way.
- **(a) one .NET-owned manifest + a required-lane drift pin** — the pick.

**The normative rule, stated once:**

> **A page is declared ONCE — one row in the .NET page manifest.** Every other surface
> is either a **derived view** (same assembly, same object graph — cannot drift) or a
> **pinned mirror** (a hand-written copy a required-lane drift test compares
> pair-for-pair). Today that means: `NativeNavigationManager`'s route table and
> `HostSession`'s mount registry are DERIVED; Android's `DEEP_LINK_COMPONENTS` is the
> one surviving PINNED MIRROR (it must be a copy — the shell needs it before the
> runtime exists); iOS has no route surface, and the day it grows one it joins the pin.

**The shape.** A new `PageManifest.cs` (`BlazorNative.Runtime`, internal static — its
own class, deliberately, so neither `HostSession` nor `NativeNavigationManager` feeds
the other's static initializer):

```
internal readonly record struct Page(string? Route, string Name, Func<NativeRenderer,int> Mount);
internal static readonly Page[] Pages = [ /* 9 routed rows + 5 probe rows (Route: null) */ ];
```

- Each row keeps the `r => r.Mount<T>()` lambda **exactly as written today** — the
  statically-rooted generic instantiation is the trim law's shape; nothing goes
  reflective, the 4 accepted IL2072s must not grow (Gate 1's publish asserts it as
  always).
- `HostSession.s_components` is built from ALL rows; `NativeNavigationManager.s_routes`
  from the routed rows. `DefaultRoute`/`DefaultComponent`/`ResolveComponent`/
  `TryGetRouteForComponent`/`NotifyMounted` keep their signatures — consumers see no
  change; the per-phase provenance comments consolidate onto the manifest rows.
- **The .NET count reaches Kotlin the way it reaches every other mirror: it doesn't.**
  No checked-in constant both sides pin (a third copy), no JVM test reading `.cs` source
  (the JVM lane is not where drift tests live) — the pin sits in **build-test**, the one
  required lane where every file is checkout-visible, quoting
  `ShellStyleTableDriftTests`' own header for why.

**The drift test** — `RouteTableDriftTests.cs`, `Runtime.Tests` (the manifest is
Runtime's; the style precedent lives in Renderer.Tests because `YogaStyleAttributes` is
the renderer's — same rule, applied). It regex-parses `MainActivity.kt` out of the
checkout as text, declaration anchored at line start (the style-table parser's
anti-comment-match rule), and pins **pairs, not name sets** — for routes, the drift that
matters is a route mapped to the WRONG page, which set-equality cannot see. Facts:

1. `DEEP_LINK_COMPONENTS` == the manifest's routed rows **minus the default route**
   (`"/"` is deliberately absent from the Kotlin map), pair-for-pair, failure message
   naming the offending pairs in both directions.
2. The Kotlin default-fallback literal (`?: "BnDemo"`, `MainActivity.kt:217`) ==
   `PageManifest`'s default row's name — the one row the map doesn't carry, pinned where
   it actually lives.
3. The nine-page ordered baseline (today `NavigationTests.cs:294`) retargets to the
   manifest's routed rows — the content pin that catches a page silently vanishing from
   the source itself (a drift test comparing two surfaces is blind to a row deleted from
   both).

`EveryRoute_ResolvesToAComponentTheMountRegistryKnows` becomes true **by construction**
(both dictionaries are views of one array) and is **retired with a comment**, not kept
as a green tautology — the house rule against assertions that cannot fail. The on-device
route-row assertions (each `Bn*DemoAndroidTest` mounting its route once) stay: the pin
proves the tables agree, they prove the row actually resolves on a device.

**Touch points, counted:** `PageManifest.cs` (new), `HostSession.cs`,
`NativeNavigationManager.cs`, `NavigationTests.cs` (retarget + retire),
`RouteTableDriftTests.cs` (new), `MainActivity.kt` (KDoc only — the map's content is
already correct; its comment stops promising unification and starts naming its pin).
Six files, two of them new tests. iOS: **zero files** — recorded, expected.

## Risks

| Risk | Mitigation |
|---|---|
| Regex-parsing a Kotlin `mapOf` literal is brittle | The style-table parser's rules, reused: declaration anchored at line start (comments that MENTION the map cannot match), pairs parsed inside the balanced parens, and a parse that finds zero pairs FAILS the test (never vacuously green) |
| The manifest move breaks trim rooting (the `Mount<T>` lambdas are the roots) | Lambdas move verbatim, one per row, still statically instantiated; Gate 1's bar includes the publish with IL2072 == 4 and 9 exports on win-x64 — the same assertion every phase carries |
| A static-init cycle between `HostSession` and `NativeNavigationManager` | The manifest is its OWN class with no dependencies; both consumers read one static array — the dependency is a fan-out, not a cycle |
| The drift pin goes red at Gate 1 against the existing Kotlin map | Then the tables already disagree and the phase just found DoD #8's bug — that is the pin working; fix the pair, record it in the conclusion. (Verified ahead: today's 8 pairs + fallback DO agree — the pin should land green.) |
| H3's retry masks a real regression | Bounded (3), attempt count in the failure text, and the content assertion is unchanged — a deterministic break fails all three attempts identically |
| The audit finds a DoD row it cannot verify | The M6 precedent: record PARTIAL honestly, ledger the gap, decide the tag on capability truth — never rewrite the finding |
