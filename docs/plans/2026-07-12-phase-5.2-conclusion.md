**Executive summary (2026-07-12).** The **first non-Kotlin shell** exists.
BnDemo's widget tree renders on the CI iOS simulator through a **Swift/UIKit**
shell over the NativeAOT runtime, linked as a **static `.a`** into an Xcode app.
The shell is the imperative twin of the Android `WidgetMapper` — `UIStackView`↔
`LinearLayout`, `UILabel`/`UIButton`/`UITextField`↔`TextView`/`Button`/`EditText`
— reading read-only rt→shell frames over the exact 48/24-byte wire the JVM and
.NET shells decode. A hosted **XCTest** boots the runtime, mounts BnDemo, and
asserts the real `UIView` tree; a **new `.github/workflows/ios.yml`** lane
publishes the static archive, verifies the nine `blazornative_*` exports, and
runs that XCTest on the simulator — **informational-first**, mirroring the
Android emulator lane.

**The load-bearing discovery** (the reason this phase existed): the recipe for
linking a NativeAOT **static** archive into an Xcode app/XCTest bundle —
`bootstrapperdll.o` direct-link + `-force_load` the app `.a` + a merged
`libBnRuntimeSupport.a` (the runtime pack's `native/*.a`) + the 5.0 spike's
framework set. This is the foundation 5.3 and eventual device builds inherit.

No Runtime **source** changed → version stays **`1.3.0-phase-5.1`** (documented
below). .NET 220 / JVM 78 / Android 38 untouched; the iOS **XCTest count = 2**
is the new asserted surface.

## 1. The static-`.a`-into-Xcode link recipe (the centerpiece)

The 5.0 spike proved a NativeAOT **`.dylib`** links + boots on the simulator. 5.2
needed the **app-embed** shape — a static `.a` linked into the app binary — and
that is a materially different problem, because a `NativeLib=Static` archive
carries **only the compiled managed→native app code**. NativeAOT bundles the
runtime + platform native libraries *itself* only when it emits a `.dylib`; for a
static embed the app must supply them. The working link (BnHost target only, so
the hosted test bundle — which links against the app via `BUNDLE_LOADER` — never
re-links the runtime):

```
vendor/bootstrapperdll.o                       # DIRECT object — its constructor inits the runtime
-force_load vendor/libBlazorNative.Runtime.a   # the app .a: exports + module cctors
vendor/libBnRuntimeSupport.a                   # merged support archive, ON-DEMAND
-lc++ -lz -licucore -lobjc
-framework Foundation -framework Security -framework CoreFoundation
```

Three facts were established the hard way, each in its own diagnosis commit:

1. **`bootstrapperdll.o` must be a DIRECT object.** The crash (`BnHost-*.ips`,
   SIGSEGV) was `blazornative_init` → `Thread::ReversePInvokeAttachOrTrapThread`
   → `ThreadStore::AttachCurrentThread` → `RuntimeInstance::GetThreadStore()` on a
   **NULL** `RuntimeInstance`: the runtime was never initialised. NativeAOT
   bootstraps a library via a `__attribute__((constructor))` in
   `libbootstrapperdll.o`. Merged into an **archive**, that object is on-demand and
   a *constructor is not a referenced symbol*, so it was never pulled. Link it as a
   **direct object** → its constructor lands in the app's `init_array` and runs at
   launch, before `blazornative_init`. **This is the load-bearing fix.**
   (`libbootstrapper.o` — the EXE variant, needs `__managed__Main` — is dropped.)
2. **The support libs must be present but ON-DEMAND.** The runtime bootstrapper/GC
   (`_Rh*`) and the platform native libs (`_AppleCryptoNative_*`, `System.Native`,
   …) live in the runtime pack's
   `runtimes/iossimulator-arm64/native/*.a`. `ios.yml` merges them into one
   `libBnRuntimeSupport.a` via `xcrun libtool -static`, **excluding** the
   non-default variants (`ServerGC`, `eventpipe-enabled`, `standalonegc-enabled`)
   — the app is ILC-compiled for WorkstationGC + `*-disabled` (established from the
   win-x64 `link.rsp`). **Force-loading this archive breaks the link** (it drags in
   the exe bootstrapper's `___managed__Main` + the `Network`-framework transport
   `_nw_*`); on-demand pulls only the library-appropriate objects via `_Rh*`.
3. **`PublishAot` must stay in the csproj, not on the CLI.** A CLI
   `-p:PublishAot=true` becomes a **global** MSBuild property inherited by the P2P
   refs (Core/Renderer/Http/Components), which lack the runtime-pack bypass →
   `NETSDK1203`. Publishing `-r iossimulator-arm64` alone uses the csproj's local
   `PublishAot` and the ios-gated bypass — the same posture as the win-x64/bionic
   ci.yml legs.

**Deviations from the 5.0 spike's bare-clang flags:** the spike link-probed a
self-contained `.dylib` **positionally**; the static embed additionally needs the
direct-linked `bootstrapperdll.o` + the merged on-demand support archive
(items 1–2). The framework/lib set carried over **unchanged** (Foundation /
Security / CoreFoundation, `c++`/`z`/`icucore`/`objc`). `GSS`/`Network` are **not**
needed because the support archive is on-demand (networking/Kerberos code is not
pulled by boot+render). The only build-surface tool the lane adds is **XcodeGen**
(`brew install xcodegen`), which materialises `BnHost.xcodeproj` from a readable
`project.yml` instead of a hand-rolled, merge-hostile `.pbxproj`.

## 2. The Swift shell as-built (`src/BlazorNative.Apple/`, design §1 a–e)

- **(a) C-ABI bridging** — `BnHost/BlazorNativeRuntimeC.h` (the Swift
  `SWIFT_OBJC_BRIDGING_HEADER`): the 4 boot+render exports + `shutdown`, the
  by-value `bn_init_options`/`bn_init_result` structs (offset-matched to
  `Exports.cs`), the `bn_frame_callback` typedef. Swift's native C interop
  **replaces the JNA layer entirely** — no reflection, no Structure classes. The
  other 4 exports (dispatch/bridge/fetch/host_event) are present in the archive
  (asserted by `nm`) but undeclared — boot+render only (5.3 wires them).
- **(b) `BnFrameAdapter`** — `UnsafeRawPointer.loadUnaligned` reads at the exact
  offsets (Kind@0 NodeId@4 ParentNodeId@8 NodeType@12 AuxInt@16 Reserved0@20
  Text@24 PropName@32 PropValue@40; frame Patches@0 PatchCount@8 FrameId@12
  TimestampMs@16); strings copied in-callback (`String(cString:)` — arena-safe);
  throwing decode; `0..65536` count guard; unknown-kind (incl. dormant 2) log+skip.
  `loadUnaligned` (not `load`) so a non-8-aligned arena can never trap.
- **(c) `BnWidgetMapper`** (UIKit) — `view→UIStackView`, `text→UILabel`,
  `button→UIButton`, `input→UITextField` (image/scroll/picker → placeholder
  `UIView` + log, so container indices stay consistent); the text-collapse
  (a `text` child of a `UILabel`/`UIButton`/`UITextField` aliases onto the parent,
  `ReplaceText`→`setTitle:`/`.text`); mid-list `insertArrangedSubview(_:at:)`;
  `backgroundColor` (#RRGGBB/#AARRGGBB→`UIColor`), `fontSize`→`.systemFont`,
  `padding`→`layoutMargins`, `value`/`placeholder`; `RemoveNode` by identity
  (alias-safe); Attach/Detach log+skip; `[Int32: UIView]` registry.
- **(d) `BnRuntime`** — `init(os:"ios")` → status 0 → `register_frame_callback`
  → `mount("BnDemo")`. The `@convention(c)` callback can't capture → routes through
  a `BnRuntime.shared` singleton (the Swift form of JNA's strong-ref hazard); the
  runtime + the global trampoline outlive registration. Decode wrapped in
  `do/catch` → error sink (drop the frame loudly, never crash in the C callback).
- **(e) Xcode project + host** — `AppDelegate` + a minimal `HostViewController`
  (boots the shell into its root view for a real launch; **inert under XCTest** via
  `NSClassFromString("XCTestCase")` so the test owns the single native session) +
  the XcodeGen `project.yml`.

## 3. Threading (the render path) — the Kotlin twin

The frame callback fires on the **mount thread** (sync-mount: BnDemo's full frame
arrives before `mount` returns). `BnFrameAdapter` decodes + **copies strings on
that thread** (arena lifetime). `BnWidgetMapper` **buffers** patches until the
`CommitFrame` patch, then `DispatchQueue.main.async` to build/mutate the UIKit
tree — the exact twin of the Kotlin `mainHandler.post(applyBatch)` batch. The
hosted test pumps `RunLoop.current` (draining the main queue) until the form
appears — the async-hop analog of the Android `pollForForm`.

## 4. The tests — DoD #2 proof + the drift guard (count = 2)

- **`BnRenderTests.testBnDemoRendersCanonicalTree`** (hosted; mirrors
  `BnDemoAndroidTest`'s pins): root has exactly the form stack; the form has **6
  arranged subviews in order** — `[0]` `UILabel` "BnDemo" fontSize 24, `[1]`
  `UITextField` placeholder "Type here...", `[2]` echo `UIStackView` with one empty
  `UILabel` (the **mid-list insertIndex 2** — the shape only lands right if the
  mapper honors it), `[3][4][5]` `UIButton` titles "Clear"/"Theme"/"Settings →";
  `backgroundColor == #FFEEAA` on the form + echo panel. The simulator log shows
  the boot: `native init ok — BlazorNative.Runtime 1.3.0-phase-5.1` → `mounted
  BnDemo` → test passed.
- **`BnDriftTests.testWireLayoutMatchesContract`** — the **third** offset/size
  guard, alongside `NativeFrameAdapterTest.kt` (JVM) and `PatchProtocolNativeTests.cs`
  (.NET, `Marshal.OffsetOf`). Pins `BnFrameAdapter`'s offset constants to the
  48/24-byte layout + the nodeType table + the 65536 ceiling, so a future protocol
  change breaks **all three shells** loudly.

`ios.yml` asserts the baseline **2 passed / 0 failed** (asserted, not observed —
the house style; count drift fails loudly).

## 5. The iOS CI lane (DoD #4)

`.github/workflows/ios.yml` — `macos-latest`, **single job** (iOS publishes AND
tests on macOS, unlike Android's split): publish `-r iossimulator-arm64` (csproj
bypass + `NativeLib=Static`; assert 4 IL2072) → `nm -gU` asserts the **9**
`blazornative_*` exports → assemble the static-embed link inputs (§1) → XcodeGen +
`xcodebuild test` on a **runner-selected** simulator (`simctl list … | jq` picks
an available iPhone — robust to Xcode's device-name drift) → assert the XCTest
baseline. **INFORMATIONAL** (not a required check), mirroring
`android-instrumented.yml`; **promotion criteria** documented (≈10 consecutive
green runs on main → promotable). Triggers: `workflow_dispatch` + `pull_request`
(iOS paths); public-repo macOS minutes are free. Uploads everything each run
(publish log, nm dump, xcodebuild log, crash reports, `.xcresult`).

Wall times (green run [29196852504](https://github.com/MarcelRoozekrans/BlazorNative/actions/runs/29196852504)):
total **3m28s** — publish ≈30s, xcodebuild+test ≈2min, the render test itself
≈3s. Xcode 26.5 / iPhoneSimulator 26.5 / SDK 10.0.301 / ILC+packs 10.0.9.

## 6. Baselines + version

The only .NET change is **`<NativeLib>Static</NativeLib>`** added to the **iOS-only**
RID `PropertyGroup` (RID-gated — the condition never matches win-x64/bionic, which

IL2072, 9 exports). **No Runtime source changed**, so the version stays

documented choice (design §5). No-regression (local): **.NET 220/0/0**,
**JVM 78/0/0** (`--rerun-tasks`); Android **38** untouched (existing CI green is the
backstop). The new asserted surface is the **iOS XCTest count = 2**.

## Carryovers / ledger

- **Interactivity / bridge / navigation = Phase 5.3.** The mount frame's
  `AttachEvent` patches are decoded then **skipped** (logged) — 5.3 wires
  `dispatch_event` + the shell bridge + nav parity.
- **Device / signing / App Store = later** (needs an Apple Developer account). The
  lane is simulator-only; `ios-arm64` (device) publishes (5.0 audit) but is not a
  lane surface yet.
- **The support-lib denylist → allowlist + RID generalization for device = 5.3.**
  The merge currently *excludes* the non-default variants; a device build (and a
  cleaner posture) should *include* an explicit allowlist and parameterize the RID.
- **The mapper's stubbed node types** (image/scroll/picker → placeholder `UIView`)
  — BnDemo uses none; wired as real `UIImageView`/`UIScrollView`/`UIPickerView`
  when a component needs them.
- **`UIScene` lifecycle warning** — the classic `UIApplicationDelegate` app logs a
  future-deprecation note; harmless for the test host, revisit if the 5.3 demo app
  adopts scenes.

## Decisions captured

- **UIKit + static `.a`.** UIKit is the imperative twin of the Kotlin View mapper;
  the static archive is the app-embed shape an iOS app wants (the 5.0 spike
  deferred this packaging decision).
- **The static-embed link recipe** (§1): `bootstrapperdll.o` direct-link +
  `-force_load` app `.a` + on-demand merged support archive + the spike frameworks.
  The load-bearing discovery for 5.3 and device.
- **`PublishAot` stays in the csproj** (never on the CLI — global-property
  propagation → NETSDK1203 on the P2P refs).
- **Version unchanged (`1.3.0-phase-5.1`)** — no Runtime source changed; only the
  csproj (RID-gated) + the new `src/BlazorNative.Apple/` dir.
- **Lane informational-first**, XcodeGen for the project, runner-selected
  simulator, upload-everything — the Android-emulator-lane posture, ported.
