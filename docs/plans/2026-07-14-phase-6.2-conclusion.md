## What shipped

The `scroll` NodeType, stubbed since Phase 2.5, is real. `<BnScroll Height="200">` is a
**viewport**; its content is laid out by Yoga, may exceed it, and **actually scrolls** — on
Android and iOS, from one runtime, with identical computed frames.

**No ABI change**: still 9 exports and the 72-byte bridge. `scroll` was already NodeType 6 on
the wire; `overflow` is set shell-side from the NodeType and is not a routed style name.

**Counts (all CI-asserted):** .NET **304/0** · JVM **79/0** · Android instrumented **96/96** ·
iOS XCTest **50/50**.

## The model: a synthetic content node

```
wire tree:   scroll → row0..row9
view/Yoga:   scroll → CONTENT → row0..row9      ← the content node is SYNTHETIC (shell-created)

scroll node  = the VIEWPORT (needs a DEFINITE height)
content node = height:auto, width:100%, flexDirection:column, flexShrink NEVER SET
               → its computed height IS the content size, read straight out of Yoga
```

Wire child *i* is the content node's child *i* — **in the Yoga tree and the view tree**. This is
the second index-mapping rule after 6.1's text-collapse invariant, and it fails the same way:
silently, as a skew. It matters more on iOS than Android, because Android *throws* on an
out-of-range index while iOS *clamps* — a skew that is loud there is silent here. All four cases
(front, mid-list, append, child-removal) plus the text-collapse × scroll interaction are pinned
**by frame** on both platforms.

`contentSize` comes from Yoga, never from a shell-side union of child frames. Two shells
deriving it independently is exactly where they drift apart.

## The proof: identical numbers on both devices

`BnScrollDemo` (`/scroll`) — a 300×200 viewport over ten 80-tall rows, with a flex row *inside*
a scrolled row and the scroll *inside* a flex column:

| node | x | y | w | h |
|---|---|---|---|---|
| viewport | 0 | 0 | 300 | 200 |
| **content (synthetic)** | 0 | 0 | 300 | **800** |
| row *i* | 0 | 80·*i* | 300 | 80 |
| box A · **box B (`Grow=1`)** · box C | 0 · **50** · 250 | 0 | 50 · **200** · 50 | 80 |
| back row (outside the viewport) | 0 | 200 | 300 | *measured (oracle)* |

`contentSize` 300×800 · viewport 300×200 · **scrollable range 600**. At offset 600 the visible
window is content y 600..800: row 7 is *partially* visible (its bottom 40dp), rows 8 and 9 fully.
**Zero numeric divergence between the platforms** — asserted number-for-number in
`BnScrollDemoAndroidTest` and `BnScrollDemoTests`.

And it *scrolls*: both suites drive the scroll position and assert the rows move relative to the
viewport. Android gets a second, independent witness for free — `ScrollView` clamps a 10 000dp
scroll request to exactly **600**, derived from the content view's *laid-out* height, so the
framework and Yoga agree about the 800 by two different paths.

## The four false mechanisms

This phase's real lesson. **Four plausible causal stories were written into this design — by us —
and every one was wrong.** Each was caught only by writing the test that checked it, and each had
already been repeated in two or three files by the time it was caught.

1. **"`overflow: scroll` is what stops Yoga clamping the content to the viewport."**
   No. **Yoga's `flexShrink` default is 0** (CSS's is 1). The free space is `200 − 800 = −600`;
   negative free space is distributed by the *shrink* pass; `0` refuses it. The content node keeps
   its 800 — and computes to 800 with `overflow: visible` too, to the dp. We still *set*
   `overflow: scroll` (it is what a scrolling viewport means), but it is not what produces the
   number. **A `flexShrink: 1` copied out of CSS or an RN stylesheet collapses 800 → 200** — and
   the rows keep their 80 and *every frame stays correct*, so a screenshot looks perfect while
   600dp of content is permanently unreachable. The only corrupted number is `contentSize`.
2. **"`BnYogaFrameLayout.onMeasure`'s UNSPECIFIED fallback is what makes the page scroll."**
   No — the demo still scrolls without it. What it actually protects: `ScrollView.onLayout` ends
   with a `scrollTo` that re-clamps the offset against the content child's just-laid-out height,
   **on every commit**. With a broken fallback the content is 0-tall at that moment, so **any
   re-render while the user is scrolled snaps the page back to the top.** Real, user-visible,
   silent — and it was untested.
3. **"`Grow=\"1\"` in a bounded parent bounds a scroll viewport."** No. `Grow` leaves
   `flexBasis: auto`, so the basis is *the content's* height, so the free space is **negative** —
   and `flexGrow` distributes only *positive* free space. The negative goes to the shrink pass,
   where `flexShrink: 0` refuses it (mechanism 1, one level up). The viewport keeps its content's
   height and never scrolls. The correct shapes are `Height`, or **`Grow="1" Basis="0"`** — which
   is precisely why CSS's `flex: 1` shorthand sets basis to `0`. This one had been recommended by
   the design, by `BnScroll`'s docs, by `BnScrollDemo`'s header, **and by the diagnostic's own
   warning message.**
4. **"The iOS `contentOffset` clamp fires when the content shrinks."** No — it fired on *every*
   commit. `UIScrollView.bounces` is on whenever content is scrollable, so a mid-drag rubber-band
   leaves `contentOffset.y` legitimately negative; any commit landing in that window clamped it to
   0 and **killed the bounce under the user's finger**. Invisible in this phase (nothing in the
   demo re-renders while you drag) and constant in the next: **M7's virtualized list commits
   continuously while scrolling**, and is the named customer of this exact mechanism.

The pattern is worth carrying into M7: *a mechanism nobody tested is a mechanism nobody knows.*
Every one of these was stated confidently, propagated by copy, and survived review-by-reading.
None survived a test.

## The guards, proven by mutation

Every guard in this phase was verified by breaking the thing it guards and confirming the *right*
test goes red — and several of the results are the point:

| Mutation | What happened |
|---|---|
| `flexShrink: 1` on the content node | Android 6 red · iOS 8 red — and **the row frames are not among them.** Every frame stays correct; only `contentSize` breaks. |
| Delete the content-node purge | Android: exactly 1 red (the lifecycle test) — **no frame assertion can see it.** iOS: 2 red, and **it does not crash** — the stale entry *is* a freed `YGNodeRef`, and only a `guard let scroll = views[scrollId]` that reads like bookkeeping stands between it and a dereference. It now `assertionFailure`s. |
| Revert `onMeasure`'s UNSPECIFIED fallback | The demo **still scrolls** (disproving the claim it was written for); only the new scroll-preservation test reddens — expected 600, got 0. |
| The definite-height warning's `>=` → `==` | Only the new "viewport taller than its content" test reddens, printing the exact falsehood the shipped code emitted: *"it computed to 200.0dp, which is exactly its content's height"* — when the content was 100. |
| Drop the `max(0, …)` floor from iOS's clamp | The simulator reads `contentOffset.y` back as **−120.0 at rest** — `UIScrollView` really does keep a page scrolled above its own content. |
| Delete `contentInsetAdjustmentBehavior = .never` | **49 of 50 tests stay green** — every frame, every `contentSize`, the whole demo table — because the safe-area inset is *zero in a detached test host*. Only a direct property assertion catches it. The knob whose justification is "the tests cannot see this" needed an assertion *on the property*, not on a number. |

## Contracts pinned across the mirrors

The six-name container-style **ignore list** (`flexDirection`, `justifyContent`, `alignItems`,
`flexWrap`, `gap`, `padding` — ignored and logged on a `scroll` node, because the scroll node's
only child is the synthetic content node) is hand-written in Kotlin **and** Objective-C++. It is
pinned by `ShellStyleTableDriftTests` in the required lane: Kotlin ≡ `.mm` set-equality, plus
`ignoreList ⊆ YogaStyleAttributes` — not ceremony, because the ignore rule lives behind
`owns(property)`, so a name *off* the routing table would never reach it and would be silently
dropped instead.

`BnScroll` is a flex **item**, not a flex **container**: the container-layout family
(`Direction`, `Justify`, `Align`, `Wrap`, `Gap`, `Padding`) is absent by construction. Every one
of them would style the *scroll* node, whose only Yoga child is the content node — and
`Justify="Center"` over 800 of content in a 200 viewport offsets it to **y = −300**, making the
top of the page permanently unreachable. An author who wants to shape the content composes a
`BnColumn Gap="8"` **inside** the scroll: React Native's `contentContainerStyle` escape hatch,
without a second style surface. The shells enforce the same rule at the wire (the ignore list),
so the raw-element hatch is closed by the same sentence.

## Honest boundaries (ledgered)

- **Horizontal scroll.** Android's `ScrollView` is vertical-only — horizontal is a *different
  widget class*, which would have to be chosen at `CreateNode` from a `flexDirection` that arrives
  in a *later* `SetStyle` patch. An ordering problem for an axis nothing needs yet.
- **`onScroll` / `scrollTo` / offset-restore across navigation.** `onScroll` fires at 60Hz and
  would be the first high-frequency producer on a wire designed for taps; it needs its own design
  (coalescing, throttling). Its real customer is **M7's virtualized list**, which should own all
  three together.
- **`ContentPadding` / `ContentGap`** — an RN-style content-container style surface, if composing
  a `BnColumn` inside proves insufficient in practice.
- **The definite-height diagnostic has a known false negative:** `Height="100%"` against an
  *indefinite* parent resolves to auto, hugs its content, and dies silently — but `PERCENT` exits
  at the first condition. Yoga does not expose whether a percent actually resolved.
- **`picker` still does not flex its children** (the last framework `ViewGroup` that runs its own
  layout). `image` has no source path — **6.3 owns it**.
- **iOS-only mechanics, deliberately not mirrored to Android:** the `contentOffset` clamp on
  shrink (Android gets it free from `ScrollView.onLayout`'s re-clamp);
  `contentInsetAdjustmentBehavior = .never` (a parity knob — `.automatic` would shift iOS's scroll
  range by a device-specific safe-area constant Android has no notion of); `BnScrollContentView`
  as a marker type (`UIScrollView` keeps scroll-indicator subviews, so `subviews[0]` is a lie).
  **Android-only, deliberately not mirrored to iOS:** the `onMeasure` UNSPECIFIED fallback,
  `isFillViewport`, and `ScrollView`'s per-layout offset re-clamp.

## What 6.3 inherits

A working scroll engine and a demo that scrolls. `image` is the last stubbed leaf: it creates a
widget on both platforms and measures to zero, because neither shell has a source-loading path.
That is the whole of Phase 6.3.
