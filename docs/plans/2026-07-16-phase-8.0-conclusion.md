## The inversion as proven

**The API** (`028d47d`, Gate 1 — the .NET side lands whole because it cannot land
half-done without breaking build-test):

- `BlazorNativeApp.RegisterPages(params BlazorNativePage[])` +
  `BlazorNativePage.Routed<T>(route, name)` / `Named<T>(name)` — the two `DAM(All)`
  factories are the **only** constructors of the mount thunk, the lambda verbatim the
  7.6 row shape (`static r => r.Mount<T>()`). Concrete `T` at every app call site
  satisfies the annotation statically; nothing goes reflective.
- **The app registers via `[ModuleInitializer]`** — `SampleAppPages.All` (the 14 rows,
  provenance comments moved verbatim) registered eagerly inside `blazornative_init`
  under NativeAOT, with an idempotent `EnsureRegistered()` for lazy-initializing
  CoreCLR test hosts (the `StrictModeTestDefaults` pattern).
- **`PageManifest` = the internal STORE**, rows gone: loud validation
  (`ArgumentException` naming the row — non-empty, factory-built, unique names, unique
  routes, `"/"` required iff any routed row), register-once, never-after-freeze. The
  derived views (`HostSession` mount registry, `NativeNavigationManager` route table)
  went **lazy-after-freeze**: ILC may pre-initialize static ctors, and a pre-baked
  EMPTY dictionary is exactly the silent failure this phase cannot afford. An empty
  registry rides the existing rc 1 with a distinguished stderr diagnostic naming the
  fix — no ABI change.
- **Runtime's Components `ProjectReference` is DELETED.** With the manifest rows and
  the five probes gone, no Runtime source names a Components type; the library
  dependency graph no longer knows the app exists. SampleApp owns the
  Runtime+Components join the shells embed.
- **The publish head moved** — `PublishAot`, the RIDs, the 3.0c/5.0 runtime-pack
  bypasses, `NativeLib=Static`, the `BionicNativeAot.targets` import — verbatim into
  `samples/BlazorNative.SampleApp.csproj`. The 9 exports stay in Runtime's
  `Exports.cs`, emitted via `<UnmanagedEntryPointsAssembly Include="BlazorNative.Runtime" />`;
  an `AfterTargets="Publish"` target (`CanonicalizeNativeArtifactName`, covering both
  `publish/` and the `native/` intermediate) freezes the artifact names at the shell
  contract (`BlazorNative.Runtime.dll` / `libBlazorNative.Runtime.so` /
  `libBlazorNative.Runtime.a`), so every consumer re-pointed a **directory** and
  nothing else.
- **Zero shell code, asserted:** the filtered diff over Kotlin/Swift/ObjC++/
  `project.yml` on this branch is EMPTY; `build.gradle.kts` moved 4 path strings only
  (4 insertions, 4 deletions — `runtimePubRoot`, the win-x64 `jna.library.path`, the
  two publish hint strings). The re-point checklist was counted, not hoped
  (`062a1a0` = the required lanes, `755791c` = the device lanes + devloop/Makefile).

## THE HEADLINE — the trim finding (the named risk, materialized)

The design's premise — *"module initializers are unconditional ILC roots"* — was
**FALSE in `--nativelib` mode.** The first clean win-x64 publish of the re-pointed head
showed **IL2072 == 0, not 4**, a 400KB-smaller binary, and the page names ABSENT from
the native image: read off the `ilc.rsp` pair, nativelib ILC roots only the **input
module's** `UnmanagedCallersOnly` exports, and the exports now ride
`--generateunmanagedentrypoints:BlazorNative.Runtime` — so nothing in the image
referenced SampleApp code and ILC trimmed the **entire app**, module initializer
included. The failure is *silent until rc-1 at first mount on a device*: the shipped
binary would have carried an empty registry, on every shell.

It was caught by the standing **stop-and-analyze rule** (IL2072 drift is never
re-baselined silently — a rule written for drift UP that paid off on drift DOWN), and
fixed minimally in `de16b28`:

```xml
<TrimmerRootAssembly Include="BlazorNative.SampleApp" />
```

**The app roots ITSELF** — it is the composition root, and its manifest already names
every page statically, so the root only restores the startup edge ILC cannot see. The
shipped libraries stay fully trimmable — the post-fix publish shows the SAME four
framework IL2072s as main and zero others, which is the proof the trim law survived.
This is NOT the design's rejected sledgehammer (that rejection was about rooting the
app to make an attribute *scan* find unreferenced types; no scan exists here).

**Recorded as a design-doc correction (Gate 1 review I-1):** the falsified premise
stands corrected as *"module initializers are unconditional roots only for modules that
survive into the image; in nativelib mode, an app whose exports live in a referenced
assembly must root itself."*

**Two NAMED 8.3-template inputs ride out of this finding** (tracked in MILESTONE.md
under DoD #4):

1. **The template's csproj MUST carry the `TrimmerRootAssembly` line** — a fresh
   `dotnet new blazornative` app has exactly this shape (exports in Runtime, pages in
   the app) and would be trimmed to an empty registry without it.
2. **Review M-1 rides the same list:** `EnsureRegistered` sets its once-guard *before*
   calling `RegisterPages` — if registration throws, the guard is already set and a
   retry silently no-ops. Harmless in the sample (a throwing manifest is a dead app
   either way); flip the order or note it when 8.3 copies the pattern.

## The tripwire posture

The silent-trim failure mode is now pinned **three ways**, so it cannot recur quietly:

1. **The IL2072 `-ne 4` gate** — a whole-module trim drives the count to 0, and the
   publish gate reds on any value other than exactly 4 (all three ci.yml RIDs + the
   ios.yml simulator publish).
2. **The JVM lane's real mount** — `BootSmoke`/`BnDemo` mount through the C ABI
   against the published win-x64 binary; a trimmed app is dead on arrival there, in
   the required lane.
3. **The page-name presence probe** (review I-2, landed `b96a927`) — three page
   name-strings asserted present in the published native image at the export-check
   step, with the failure message naming `TrimmerRootAssembly` as the likely cause.
