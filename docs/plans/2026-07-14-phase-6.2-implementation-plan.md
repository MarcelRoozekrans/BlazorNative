# Phase 6.2 Implementation Plan — Real Scrolling

> **For Claude:** REQUIRED SUB-SKILL: `superpowers:subagent-driven-development` — fresh implementer
> subagent per gate, spec review then code-quality review after each.

**Goal:** the `scroll` NodeType stops being a stub. A `BnScroll` is a viewport; its content is laid
out by Yoga and may exceed it; and it actually scrolls on both platforms.

**Design:** [2026-07-14-phase-6.2-design.md](2026-07-14-phase-6.2-design.md) — the four locked
decisions and the synthetic-content-node model. Read it; it is not up for re-litigation.

**Architecture:** the scroll node gets a **synthetic content node** (`height: auto`) whose computed
height *is* the content size. Wire children parent into the content node in **both** trees. **No ABI
change** (9 exports + the 72-byte bridge, untouched; `scroll` is NodeType 6, already on the wire
since Phase 2.5).

**Branch:** `phase-6.2-scroll`.

---

## Non-negotiables (violating any fails spec review)

1. **No C-ABI / wire change.** No new export, no new NodeType, no new patch kind. `BnScroll` emits
   the existing `scroll` element (`MapElementToNodeType` already maps it to NodeType 6).
2. **The synthetic content node is in BOTH trees or NEITHER.** A scroll node's wire child at index
   *i* is the content node's child at index *i* — in the Yoga tree **and** the view tree. This is
   the second index-mapping rule after the text-collapse invariant, and it fails the same way:
   silently, as a skew. Pin it with an insert-at-index test on both platforms. (A `CreateNodePatch`
   with `InsertIndex == -1` means **append to the CONTENT node's children** — never to the scroll
   node's, whose only child *is* the content node.)
3. **`contentSize` comes from YOGA**, never from a shell-side union of child frames. The content
   node's computed height *is* the content size. Two shells deriving it independently is precisely
   what the drift tests exist to prevent.
4. **The 6.1 parity table must not move.** `BnDemo`, `BnSettingsPage` and `BnLayoutDemo` keep their
   goldens on .NET/JVM and their frames on both shells. Scroll must not perturb an un-scrolled tree.
5. **Everything Phase 6.1 pinned still holds** — read its [conclusion](2026-07-13-phase-6.1-conclusion.md).
   In particular: the measure func attaches **by NodeType**; Yoga's own rounding is **off**
   (`pointScaleFactor = 0`); one `RemoveNodePatch` means a whole **subtree** (and on iOS a missed
   descendant is a **dangling `YGNodeRef`**, not a leak — the synthetic content node is now part of
   that subtree); the style value grammar is strict and normative; `alignContent` is never set.
6. **NEVER set `flexShrink` on the synthetic content node.** *(Added at the Gate 1 review — the
   design's "Why the content node computes to 800" section is the argument.)* Yoga's `flexShrink`
   **default is 0** where CSS's is 1, and that default is the ENTIRE mechanism by which a content
   node keeps its 800 against a 200-high viewport (free space `200 − 800 = −600` is distributed by
   the shrink pass, and `0` means "shrink me by none of it"). It is **not** `overflow: scroll` that
   does this. A `flexShrink: 1` copied out of CSS or an RN style sheet collapses 800 → 200 and the
   page stops scrolling with no error anywhere.
7. **A `BnScroll` is a flex ITEM, not a flex CONTAINER** *(Gate 1 review, design decision 5)* — and
   the shells enforce the same rule at the wire: **on a `scroll` node, the style names
   `flexDirection`, `justifyContent`, `alignItems`, `flexWrap`, `gap` and `padding` are IGNORED and
   LOGGED** (design decision 6, §"Container styles on a scroll node"). Item styles and
   `backgroundColor` apply normally. This is normative for BOTH shells; it joins the definite-height
   warning as a scroll-node diagnostic.

## Gate 1 — .NET: `BnScroll`, the demo page, the golden ✅ (shipped; amended by its review)

**Files:** create `src/BlazorNative.Components/BnScroll.cs`, `BnScrollDemo.cs`; register the
`/scroll` route (`HostSession` mount registry + `NativeNavigationManager` — **all mirrors move
together**); test `tests/BlazorNative.Runtime.Tests/BnScrollDemoTests.cs`.

- `BnScroll` emits the `scroll` element and forwards **`BnView`'s ITEM surface — and none of its
  CONTAINER surface** (non-negotiable #7 / design decision 5): no `Direction`, `Justify`, `Align`,
  `Wrap`, `Gap` or `Padding`. It *is* a flex item that scrolls its content; the content's layout
  belongs to the synthetic content node, and an author who wants to shape it puts a `BnColumn`
  inside. Tests: it emits NodeType `scroll`; its item params ride `SetStyle`; the container family
  is absent in **both** directions (reflectively, and behaviourally as an exact wire table).
- **`BnScrollDemo`** — every number predictable, because Gates 2/3 assert real frames against it:
  a viewport `Width="300" Height="200"`, containing **ten rows** each `Height="80"` (so the content
  node computes to **800**; scrollable range **600**). The four inputs are `internal const`
  (`RowCount`, `RowHeightDp`, `ViewportWidthDp`, `ViewportHeightDp`) and the golden asserts the
  **products** — the content size is computed by the contract, not restated in prose. Put a flex row
  *inside* one of the scrolled rows and the scroll *inside* a flex column, so the demo proves both
  nestings (design §Verification #4). Add a "← Back" button outside the scroll.
- The patch golden is the contract Gates 2/3 derive their frame expectations from: it pins the
  creates **and their count** (19 — a twentieth means the content node leaked onto the wire) **and
  the style count** (33 — the mirror pin: the per-node tables cover 18 of the 19 nodes, so a stray
  `SetStyle` on the nineteenth would otherwise pass green). Document the expected frames in the
  component's header, as `BnLayoutDemo` does — **you are writing the spec the shells will be held
  to**, so every number in that header must be *checked*, not narrated (the review found "at offset
  600 rows 7, 8 and 9 fill the viewport", which is false: row 7 spans content y 560..640, so at
  offset 600 only its bottom 40dp is visible).
- Gate 1 close: `dotnet test` green — **302** (asserted in `ci.yml`, with provenance), JVM
  unchanged at 79.

## Gate 2 — Android: `ScrollView` + the synthetic content node

**Files:** `YogaLayout.kt`, `WidgetMapper.kt`; instrumented tests.

1. **A Yoga-only unit test FIRST — and it OWNS THE MECHANISM QUESTION** (design §"Gate 2's
   Yoga-only unit test"). Build scroll(200 tall) → content(`auto`) → ten 80-tall children,
   calculate, and assert **all four** of:
   - the **content node computes to 800** — before any widget is involved;
   - it **still computes to 800 with `overflow: visible`** — recording that the mechanism is Yoga's
     **`flexShrink` default of 0**, *not* the overflow flag (the design's original claim, corrected
     at the Gate 1 review);
   - **`flexShrink: 1` on the content node collapses it to 200** — the CSS-instinct mutation, made
     visible in a test instead of on a device (non-negotiable #6);
   - what an **auto-height scroll node** computes to — the definite-height warning is written
     against that answer.

   Keep `overflow: scroll` if it is semantically right (it is what RN does) — but the contract does
   not claim it is what produces the 800, and this test is what settles it.
2. `scroll` → `ScrollView` (vertical) + a `BnYogaFrameLayout` content child. Wire children parent
   into the content view (non-negotiable #2; `InsertIndex == -1` = append **to the content view**).
   The synthetic content node gets `height: auto`, `width: 100%`, `flexDirection: column`, and
   **never `flexShrink`**.
   - **`ScrollView` measures its single child with an `UNSPECIFIED` height spec.**
     `BnYogaFrameLayout.onMeasure`'s Yoga-size fallback (added by the 6.1 review) is what answers it
     correctly — do not regress it; pin it here with a test that says so.
3. **The two scroll-node diagnostics**, both warn-once, both pinned by a test:
   - **definite height** — a scroll node with `auto` height takes its height *from* its content, so
     nothing scrolls. (The symptom — a page that just doesn't scroll — is otherwise baffling.)
   - **container styles IGNORED-AND-LOGGED** (non-negotiable #7 / design decision 6): on a `scroll`
     node, `flexDirection`, `justifyContent`, `alignItems`, `flexWrap`, `gap` and `padding` are
     dropped with one warning naming the node and the style. Item styles and `backgroundColor` apply
     normally. `BnScroll` cannot produce them, but the raw-element hatch can (`OpenElement("scroll")
     + AddAttribute("gap", …)` — a .NET test pins that it reaches the wire), and a `flexDirection`
     on a scroll node kills scrolling.
4. `BnScrollDemo` on the AVD: assert the content size (800), the row frames, that rows below the
   viewport are outside it, and — **the load-bearing bit** — drive the scroll position and assert
   the rows **actually move** relative to the viewport. `contentSize > viewport`, from Yoga.
   At offset 600 the visible window is content y **600..800**: row 7 is **partially** visible (its
   bottom 40dp), rows 8 and 9 fully — assert *those* numbers, not "rows 7-9 fill the viewport".
5. Extend the **node-lifecycle test**: removing a scroll node must free the synthetic content node
   too (the 6.1 purge walks the Yoga tree, and the synthetic node is in it — prove it).
6. Bump + assert the instrumented count in `android-instrumented.yml` (house style: provenance).

## Gate 3 — iOS: `UIScrollView`, the same numbers

**Files:** `BnYogaLayout.{h,mm}` (only if the C surface needs an `overflow` setter — it will),
`BnWidgetMapper.swift`; XCTests.

- **Yoga's headers must never be visible to Swift** (the 6.0 rule; six red CI runs bought it). Any
  new Yoga call goes in the `.mm` behind the plain-C surface.
- `scroll` → `UIScrollView` + a plain `UIView` content child; wire children parent into the content
  view; `contentSize` = the content node's computed size (**from Yoga**, non-negotiable #3). The
  content node gets `height: auto`, `width: 100%`, `flexDirection: column`, and **never
  `flexShrink`** (non-negotiable #6 — Yoga's default 0 is what produces the 800).
- **Both scroll-node diagnostics, same rules as Gate 2**: the definite-height warning AND the
  container-style **ignore-and-log** rule (non-negotiable #7) — `flexDirection`, `justifyContent`,
  `alignItems`, `flexWrap`, `gap`, `padding` dropped-and-logged on a `scroll` node; item styles and
  `backgroundColor` applied normally. One rule, two shells, or it is not a rule.
- **The same assertions as Gate 2, with the same numbers** — content size 800, the same row frames
  (at offset 600: row 7 partially visible — its bottom 40dp — rows 8 and 9 fully), and a real scroll
  (`setContentOffset`) that moves the rows. That pairing is what DoD #4 means by "on both
  platforms".
- **Extend `ShellStyleTableDriftTests`** if `overflow` becomes a routed style name — it must land in
  all three mirrors (.NET, Kotlin, `.mm`) or be in none.
- Bump + assert the XCTest count in `ios.yml`.
