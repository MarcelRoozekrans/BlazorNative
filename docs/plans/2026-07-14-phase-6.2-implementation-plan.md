# Phase 6.2 Implementation Plan — Real Scrolling

> **For Claude:** REQUIRED SUB-SKILL: `superpowers:subagent-driven-development` — fresh implementer
> subagent per gate, spec review then code-quality review after each.

**Goal:** the `scroll` NodeType stops being a stub. A `BnScroll` is a viewport; its content is laid
out by Yoga and may exceed it; and it actually scrolls on both platforms.

**Design:** [2026-07-14-phase-6.2-design.md](2026-07-14-phase-6.2-design.md) — the four locked
decisions and the synthetic-content-node model. Read it; it is not up for re-litigation.

**Architecture:** the scroll node gets Yoga's `overflow: scroll` and a **synthetic content node**
(`height: auto`) whose computed height *is* the content size. Wire children parent into the content
node in **both** trees. **No ABI change** (9 exports + the 72-byte bridge, untouched; `scroll` is
NodeType 6, already on the wire since Phase 2.5).

**Branch:** `phase-6.2-scroll`.

---

## Non-negotiables (violating any fails spec review)

1. **No C-ABI / wire change.** No new export, no new NodeType, no new patch kind. `BnScroll` emits
   the existing `scroll` element (`MapElementToNodeType` already maps it to NodeType 6).
2. **The synthetic content node is in BOTH trees or NEITHER.** A scroll node's wire child at index
   *i* is the content node's child at index *i* — in the Yoga tree **and** the view tree. This is
   the second index-mapping rule after the text-collapse invariant, and it fails the same way:
   silently, as a skew. Pin it with an insert-at-index test on both platforms.
3. **`contentSize` comes from YOGA**, never from a shell-side union of child frames. The content
   node's computed height *is* the content size. Two shells deriving it independently is precisely
   what the drift tests exist to prevent.
4. **The 6.1 parity table must not move.** `BnDemo`, `BnSettingsPage` and `BnLayoutDemo` keep their
   goldens on .NET/JVM and their frames on both shells. Scroll must not perturb an un-scrolled tree.
5. **Everything Phase 6.1 pinned still holds** — read its [conclusion](2026-07-13-phase-6.1-conclusion.md).
   In particular: the measure func attaches **by NodeType**; Yoga's own rounding is **off**
   (`pointScaleFactor = 0`); one `RemoveNodePatch` means a whole **subtree** (and on iOS a missed
   descendant is a **dangling `YGNodeRef`**, not a leak — the synthetic content node is now part of
   that subtree); the style value grammar is strict and normative; `alignContent` is never set.

## Gate 1 — .NET: `BnScroll`, the demo page, the golden

**Files:** create `src/BlazorNative.Components/BnScroll.cs`, `BnScrollDemo.cs`; register the
`/scroll` route (`HostSession` mount registry + `NativeNavigationManager` — **all mirrors move
together**); test `tests/BlazorNative.Runtime.Tests/BnScrollDemoTests.cs`.

- `BnScroll` emits the `scroll` element and forwards the same flex surface as `BnView` (it *is* a
  flex item that happens to scroll its content). Test: it emits NodeType `scroll`, and its flex
  params ride `SetStyle` exactly as `BnView`'s do.
- **`BnScrollDemo`** — every number predictable, because Gates 2/3 assert real frames against it:
  a viewport `Width="300" Height="200"`, containing **ten rows** each `Height="80"` (so the content
  node computes to **800**; scrollable range **600**). Put a flex row *inside* one of the scrolled
  rows and the scroll *inside* a flex column, so the demo proves both nestings (design §Verification
  #4). Add a "← Back" button outside the scroll.
- The patch golden is the contract Gates 2/3 derive their frame expectations from. Document the
  expected frames in the component's header, as `BnLayoutDemo` does — **you are writing the spec the
  shells will be held to**.
- Gate 1 close: `dotnet test` green (record the count; it is asserted in `ci.yml`), JVM unchanged.

## Gate 2 — Android: `ScrollView` + the synthetic content node

**Files:** `YogaLayout.kt`, `WidgetMapper.kt`; instrumented tests.

1. **A Yoga-only unit test FIRST** (design §Risks): build scroll(200 tall, `overflow: scroll`) →
   content(`auto`) → ten 80-tall children, calculate, and assert the **content node computes to 800**
   — before any widget is involved. The whole phase rests on Yoga's `overflow: scroll` semantics
   being what we think they are; prove it in isolation, and the widget bugs that follow can't hide
   behind it.
2. `scroll` → `ScrollView` (vertical) + a `BnYogaFrameLayout` content child. Wire children parent
   into the content view (non-negotiable #2). `overflow: scroll` on the scroll node; the synthetic
   content node gets `height: auto`, `width: 100%`, `flexDirection: column`.
   - **`ScrollView` measures its single child with an `UNSPECIFIED` height spec.**
     `BnYogaFrameLayout.onMeasure`'s Yoga-size fallback (added by the 6.1 review) is what answers it
     correctly — do not regress it; pin it here with a test that says so.
3. **The definite-height rule**: a scroll node with `auto` height sizes to its content and never
   scrolls. Warn once at runtime; pin the warning with a test. (The symptom — a page that just
   doesn't scroll — is otherwise baffling.)
4. `BnScrollDemo` on the AVD: assert the content size (800), the row frames, that rows below the
   viewport are outside it, and — **the load-bearing bit** — drive the scroll position and assert
   the rows **actually move** relative to the viewport. `contentSize > viewport`, from Yoga.
5. Extend the **node-lifecycle test**: removing a scroll node must free the synthetic content node
   too (the 6.1 purge walks the Yoga tree, and the synthetic node is in it — prove it).
6. Bump + assert the instrumented count in `android-instrumented.yml` (house style: provenance).

## Gate 3 — iOS: `UIScrollView`, the same numbers

**Files:** `BnYogaLayout.{h,mm}` (only if the C surface needs an `overflow` setter — it will),
`BnWidgetMapper.swift`; XCTests.

- **Yoga's headers must never be visible to Swift** (the 6.0 rule; six red CI runs bought it). Any
  new Yoga call goes in the `.mm` behind the plain-C surface.
- `scroll` → `UIScrollView` + a plain `UIView` content child; wire children parent into the content
  view; `contentSize` = the content node's computed size (**from Yoga**, non-negotiable #3).
- **The same assertions as Gate 2, with the same numbers** — content size 800, the same row frames,
  and a real scroll (`setContentOffset`) that moves the rows. That pairing is what DoD #4 means by
  "on both platforms".
- **Extend `ShellStyleTableDriftTests`** if `overflow` becomes a routed style name — it must land in
  all three mirrors (.NET, Kotlin, `.mm`) or be in none.
- Bump + assert the XCTest count in `ios.yml`.
