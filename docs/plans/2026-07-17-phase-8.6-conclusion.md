## What release-please actually buys here, honestly — **not the version**

**Traced through `versioning-strategies/prerelease.ts` AND executed.** From `1.0.0-preview.1`:

```
fix:     ->  1.0.0-preview.2
feat:    ->  1.0.0-preview.2
feat!:   ->  1.0.0-preview.2
```

**All three. The commit type cannot move the version at all.** `isPreMajor = major < 1`, so at
major=1 **both** of the reference's `*-pre-major` flags are **dead no-ops** — and dead **at the
reference's own 1.3.3 too**, so mirroring AdoNet.Async's config faithfully would have imported **two
no-ops**. They are omitted: *config that cannot fire is a comment pretending to be a decision.*

> ### **THE VERSION IS A COUNTER. THE COMMITS COMPUTE THE CHANGELOG.**

**So the honest sales pitch is the changelog and the ritual, not the arithmetic.** 8.2's reason 2 —
*"we'd want to walk patches 1.0.1, 1.0.2"* — is **measurably false**; it walks the prerelease
counter. Corrected in the design rather than quietly dropped.

**The graduation trap, and it is the sharp end.** Reaching `1.0.0` needs `Release-As: 1.0.0` **AND a
config change**. Stopping at the commit **silently re-enters preview** — and the design named only
`fix:`; **Gate 1 corrected it by executing**:

```
fix:     ->  1.0.1-preview
feat:    ->  1.1.0-preview
feat!:   ->  2.0.0-preview
```

**All three commit types, not one.** **Nothing in the tooling warns**: the tag and the props would
agree, so **every pin stays green while the version goes backwards.** It is documented **loud, at
`GITHUB-SETUP.md`'s graduation section — where the person typing `Release-As` is standing**, not in
a design doc nobody opens.

## The dry-run caught a crash Gate 1 would have shipped

**Gate 2's bar was "run the dry-run, then diff the 7 literals". The dry-run never reached the diff.
It threw:**

```
SyntaxError: Expected double-quoted property name in JSON at position 1699 (line 48 column 42)
    at JSON.parse (<anonymous>)
    at GenericJson.updateContent (release-please/build/src/updaters/generic-json.js:32:27)
```

**Line 48 column 42 of `template.json` is exactly where Gate 1's `// x-release-please-version`
annotation comment starts.**

**The cause is a dispatch nobody can see at the config.** A **bare path string** in `extra-files`
does not *request* an updater — it asks release-please to **guess one from the file extension**:

```
.json          -> CompositeUpdater(GenericJson("$.version"), Generic)   <- strict JSON.parse
.yaml / .yml   -> CompositeUpdater(GenericYaml, Generic)
.toml / .xml   -> ...
else           -> Generic          <- the ONLY arm decision 3 wanted
```

**So the four bare strings Gate 1 shipped resolved to TWO different updaters.** `.props` and the two
`.csproj` files matched **no** arm and fell to `else` — **they got the right updater BY ACCIDENT OF
THEIR EXTENSIONS.** `template.json` got a **strict JSON parser pointed at a JSONC file**.

> **release-please would have crashed on EVERY run. No release PR, ever.**

**Decision 3's normative sentence — *"one updater kind everywhere"* — was true for three files by
luck and false for the fourth**, and no amount of *reading* the config could tell you which. The
object form makes it true **by construction**:

```json
{ "type": "generic", "path": "src/Directory.Build.props" }
```

**Converted at all four sites, not just the broken one** — three would work bare, but only because
`.props`/`.csproj` are **unknown to a tool that may learn them.** *Depending on a third-party tool's
ignorance is not a design.*

**The lesson arrived twice in one phase and was not carried the first time.** `release-please.yml:74`
**already said**, in Gate 1's own words: *"release-please reads it with `JSON.parse` … which is
STRICT — a `//` comment throws on load. Measured, not assumed."* That was measured for the **config**
file. **The identical fact was true of a SECOND json file release-please must WRITE**, and the design
checked that `dotnet new` and `TemplateDriftTests` tolerate JSONC — **without ever asking whether
release-please does.** It flagged the right line for the wrong reason.

**The pin carries NO copy of release-please's dispatch ladder** — deliberately: *that table is a
third party's and it drifts.* It asserts **one** invariant with **zero** extension knowledge:
**never use the form whose updater the extension decides.**

```
MUTATION (template.json reverted to Gate 1's bare string):
  BARE PATH STRINGS (1) — the updater is chosen by FILE EXTENSION:
    templates/.../.template.config/template.json
```

> ***A configuration that is right by accident of a file extension is not right; it is
> unfalsified.***

## THE LINE WRAP DECLARED A BREAKING CHANGE

**The dry-run was supposed to measure a formatting risk. It found the changelog announcing something
that never happened:**

```
npx release-please release-pr --dry-run
  title: chore(...): release 1.0.0-preview.2
  ### ⚠ BREAKING CHANGES
  * **release:** footers.
```

**There is no breaking change on this branch.** The text came from **`004179f`'s own WHY body — the
commit that INSTALLED the commit contract** — whose prose explains which footers release-please
reads. **The body is hard-wrapped at 80 columns, the wrap pushed the token to column 0, and the next
word was "footers."** ***So the line wrap declared a breaking change.*** One word earlier in the
paragraph and nothing would have happened.

> ### **DECISION 5'S PREMISE IS HALF TRUE, AND THAT IS THE FINDING.**
>
> Both `.commitlintrc.yml` and `commitlint.yml` said the branch's commits are *"DISCARDED — they
> never reach main and release-please never sees them"*. **TRUE OF SUBJECTS. FALSE OF BODIES:**
> `squash_merge_commit_message` is **`COMMIT_MESSAGES`**, so every branch body **concatenates into
> main's body** and release-please parses **that**. **The one surface that carries footers is the
> one surface nothing read before it landed.**

**The two hazards are not the same size, and that is measured:**

| Token | Where it fires | Blast radius |
|---|---|---|
| a **breaking** token at column 0 | **anywhere** in the body | **a changelog section** |
| a **release-as** token at column 0 | the message's **LAST paragraph** → `Version.parse` | ⚠ **THE VERSION ITSELF** |

**Mid-body with prose after it extracts nothing — which is the only reason `004179f`, which spells
that token too, did not also hand-pick a version.** The strategy scans for that note **FIRST**,
before any breaking or feature counting. **A WHY body documenting rung 3 by spelling it, and ending
near it, releases that version. Nothing warns.**

**So "a mis-typed subject can never cause a wrong version" — measured, true, shipped in both files —
is exact about the SUBJECT and must not be stretched over the BODY.**

## The footer check — and "column 0" is measurably the wrong boundary

**The rule, and it is machine-decidable in a way "did you mean it?" is not:**

> **A PR's commit BODIES may declare a release footer only if the PR's TITLE declares the same
> thing.**
> · a breaking token → the title must carry the conventional `!` marker
> · a release-as token → the title must contain the version it names

**What it asserts, stated rather than overclaimed: NOT that the author meant the footer.** It cannot
read minds and does not try. **It asserts that the SUBJECT and the BODY AGREE about what this PR
does** — two independent statements of intent, both by the same author. On disagreement it reds and
makes a human reconcile the two texts **they wrote**. *Exactly one is wrong; the check does not need
to know which.*

**No label, and the reason is the failure itself:** ***a label is a human attesting to their own
accident, and `004179f`'s author would have applied one without hesitating*** — they did not know it
had happened. **The title is evidence the author produced for another reason**, which is what makes
it worth comparing against.

> ### ⚠ **"COLUMN 0" WOULD HAVE SHIPPED FOUR HOLES.**
>
> **The controller's brief said column 0. The design said column 0. `61cfd7a`'s own body said column
> 0. Gate 3 measured against the real parser — `conventional-commits-parser` 6.4.0 with
> release-please's own options — and the parser disagrees:**
>
> ```
> <1-8 spaces or tab> then the token  -> NOTE   indentation saves NOTHING
> `* ` bulleted token                 -> NOTE   a list bullet saves nothing (and `* ` is what a squash writes)
> lowercase `breaking change:`        -> NOTE   IT IS CASE-INSENSITIVE
> token with NO colon                 -> NOTE   the colon is not required
> ```
>
> **The naive check — column-0, case-sensitive, colon-requiring — passes all four real hazards.**
> The case-insensitivity is the sharpest: **a body starting a line with the words "breaking change:"
> in ordinary lowercase prose declares one.**
>
> **The escape hatches are measured too**, because they are what the RED tells you to reach for:
> backticks, `> `, `# `, this repo's own `· ` bullet, or **any word before the token** — all produce
> no note. That is *"name the thing, do not spell it"* **with receipts instead of advice.**

**Run against the real commit — the whole branch, as a PR would see it:**

```
footer-check: 9 commit(s)
✗ commit #7, body line 28: 'BREAKING CHANGE: footers.' is a BREAKING-CHANGE footer,
  but the PR title does not carry the '!' breaking marker.                    exit=1
```

**Commit #7 IS `004179f`.** **One red, on the one poisoned commit, naming the line.** The other
eight pass — **including `61cfd7a`, which discusses these tokens at length and does not trip**,
because it named them without spelling them at a line start. ***The rule validates empirically on
the very commit that wrote it down.***

### Then it caught its own author

**Gate 3's final commit tripped it — in the commit whose body claimed its bodies were clean.**

```
commit #1, body line 24: '<a release-as token> ...' — IT PICKS THE RELEASED VERSION —
but the PR title does not name that version.                                  exit=1
```

**That is `004179f`'s failure, committed by the author of the check for it, within the hour.**
Measured before being believed — the parser really does emit the note from that line, **with no
colon** — so **the check was right and the body was wrong.** **Fixed the way the RED prescribes:
re-wrap; name the thing, do not spell it. Never by touching the title.**

**The RED leads with the right fix, and that is deliberate.** The rider below caught a pin whose
message told the reader **the wrong fix in the house's most persuasive voice**. So this one says
*re-wrap or quote the line*, and says **explicitly DO NOT add `!` to silence an accident** — that
makes the changelog announce what your wrap said by mistake.

**Self-test: 14/14** (7 pass / 7 RED), **row count asserted so a deleted row reds**. Row 1 is
`004179f`'s real body; the four measured holes are their own rows. **The self-test runs in the lane
BEFORE the check does** — *a pin that cannot see its subject must never pass vacuously, and that
applies to the pin's own logic.*

**`004179f` is NOT rewritten.** A pushed-history rewrite is the owner's call; **the controller
neutralises this instance at merge by setting the squash body explicitly.** This check stops the
**next** one — and it would have stopped this one.
