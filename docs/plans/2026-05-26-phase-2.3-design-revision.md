# Phase 2.3 Design Revision ŌĆö Env-Var Bridge (C-1)

**Date:** 2026-05-26 (same day as original design)
**Supersedes:** `docs/plans/2026-05-26-phase-2.3-design.md` Q1 (WIT-typed import) + Q2 implementation depth
**Status:** Approved 2026-05-26 after Task 2 (`3aa83c9`) surfaced three SDK gaps that block WIT-import revival

---

## Why the redesign

Task 2's deep dive (committed `3aa83c9`; full notes in `BlazorNative.WasiHost.csproj` comment block) found that the wasi-experimental SDK 10.0.8 has **three independent gaps** that prevent the original Phase 2.3 design from working end-to-end:

1. Mono-AOT's `_WasmPInvokeModules` auto-population covers only statically-linked native libs ŌĆö user modules need explicit MSBuild opt-in.
2. `$(WasmAllowUndefinedSymbols)` property is documented but the SDK never consumes it ŌĆö wasm-ld fails with "undefined symbol".
3. `wasm-component-ld --component-type` accepts only binary blobs produced by `wit-component`, not raw `.wit` text ŌĆö and no SDK helper exists to assemble that blob.

Closing all three is **possible** but requires 1-3 hours of MSBuild + Rust toolchain spike against undocumented surfaces, with real risk of surfacing a 4th gap mid-spike.

The original design's bet was that WIT-typed imports would be the "JSI-equivalent future direction." The bet stands long-term, but **wasi-experimental 10.0.8 isn't ready to host it today**. Phase 2.3 needs to ship M2's bridge-call DoD with what the SDK ACTUALLY supports.

## The revised approach: env-var bridge

Use the standard `wasi:cli/environment` interface (which wasi-experimental's component-adapter satisfies natively, the same way Phase 2.2 satisfied `wasi:cli/run`) to pass platform-info as an environment variable at instantiation. The .wasm reads via `Environment.GetEnvironmentVariable(...)`.

```
[Host (Kotlin/JVM or Android)]
  wasi_config_set_env(wasiConfig, 1, ["BLAZOR_PLATFORM_INFO"], ["""{"os":"Android",...}"""])
  Ōåō wasmtime_context_set_wasi
  Ōåō wasmtime_component_linker_add_wasip2 (no custom registration needed)
  Ōåō instantiate + invoke wasi:cli/run.run
[.NET-in-WASM Main]
  var info = Environment.GetEnvironmentVariable("BLAZOR_PLATFORM_INFO") ?? "{}";
  Console.WriteLine($"[BOOT] bridge-ok platform-info={info}");
```

**Why this works:**

- `wasi:cli/environment` is a STANDARD wasi import (not custom) ŌĆö wasi-experimental's component-adapter pre-includes it. Zero SDK gap.
- `wasmtime_component_linker_add_wasip2` (Phase 2.1) already satisfies all standard wasi:* imports including environment. Zero JNA work beyond one new binding.
- `Environment.GetEnvironmentVariable` is a standard .NET API that Mono-WASI implements via `wasi:cli/environment.get-environment`. Standard library; not a custom extern.
- The marker payload still varies per host (proves real round-trip, not a constant baked into the .wasm) ŌĆö the original GREEN CHECKPOINT criterion is preserved.

### What stays the same

- The `[BOOT] bridge-ok platform-info=<json>` marker shape in Main
- All 5 test assertions across JVM + Android (BridgePlatformInfoTest's sentinel approach works through env-var bridge the same way it would through WIT-import bridge)
- Three-way cross-validation: JVM stub + Android real + .NET CLI stub all assert the marker; payloads differ per host

### Trade-offs accepted

- **One-way only (host ŌåÆ .NET).** Env vars are initialization-time, immutable. For platform-info this is correct semantics (the OS doesn't change during runtime). For future dynamic bridges (user-event callbacks, async navigation), we'll need a different mechanism ŌĆö most likely the export-based "request/response" pattern (Option C-2 from the brainstorm), but that's Phase 2.5+ work when there's an actual UI event to bridge.
- **String-only payload.** Env vars are pairs of strings. Structured data crosses the boundary as JSON (or other text format). For Phase 2.3 this is fine ŌĆö JSON is the design's Q5=C hybrid contract anyway.
- **No symmetric dynamic API.** wit-bindgen + WIT imports remain the long-term direction (per Phase 2.1 design Q1 reasoning). When wasi-experimental ships first-class WIT integration (or when we close the 3 SDK gaps in a future spike), Phase 2.3+ can migrate. The current revision is YAGNI-correct for M2 scope.

## Carryover into BACKLOG

The original WIT-import design isn't wrong ŌĆö just blocked by SDK gaps. Three items for future work:

- **Phase 2.3b ŌĆö wasi-experimental WIT-import unblocker.** Investigate `_WasmPInvokeModules` + `LinkerArg --allow-undefined` + custom `wit-component` MSBuild target. ~1-3 hours, requires Rust toolchain + understanding the wit-component binary blob format. Reward: revive the wit-bindgen path for dynamic bridges.
- **Phase 2.5+ ŌĆö export-based dynamic bridge.** When user-event callbacks (button tap ŌåÆ .NET handler) need a runtime-resident bridge mechanism that env vars can't provide, design + ship the `[UnmanagedCallersOnly]` export pattern (Option C-2 from the brainstorm).
- **General ŌĆö track upstream wasi-experimental.** When the .NET runtime ships first-class WIT integration (likely .NET 11/12+), reconsider Phase 2.3's revised approach.
