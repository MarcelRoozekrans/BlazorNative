## 1. Navigation (DoD #7)

- **Core**: `INavigationManager` — `ValueTask NavigateToAsync(string route)`,
  `string CurrentRoute { get; }`, `event Action<string> RouteChanged`.
- **Runtime**: `NativeNavigationManager` —
  - Route table `"/"` → `"BnDemo"`, `"/settings"` → `"BnSettingsPage"` (values are mount-registry
    keys; the registry stays the single component-name authority).
  - `NavigateToAsync(route)`: resolve (unknown route → `ArgumentException`, surfaced per the
    strict/error conventions) → notify the host via the 3.1 `Navigate` bridge callback (host
    updates its `@Volatile` route + logs) → `RemoveRootComponent(currentRootId)` (Blazor's
    protected API on our `Renderer` subclass — **the 3.3 disposal machinery emits the
    RemoveNode patches that clear the screen**) → `TryMount` the new component → raise
    `RouteChanged`.
  - `CurrentRoute`: tracked .NET-side; initialized at session start by querying the host's
    `CurrentRoute` buffer callback — an existing route maps to its component at first mount
    (host-restored route), else `/`.
  - Host-INITIATED navigation (back button, deep links) is explicitly **M5**.
- **DI**: `HostSession` registers `NativeNavigationManager` as `INavigationManager`;
  `NativeRenderer.Mount` already returns the root componentId — the session tracks the current
  root for the swap.
- **Risk (verify first at Gate 1):** `RemoveRootComponent`'s exact shape on Blazor 10 from a
  custom Renderer subclass. If absent/unusable → fallback = dispose-and-recreate the renderer
  (heavier, still correct — the host clears on the RemoveNode sweep). RED-on-ambiguity.
