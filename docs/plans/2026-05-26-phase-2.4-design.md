# Phase 2.4 Design — Render-Frame Consumer (One-Shot)

**Date:** 2026-05-26
**Milestone:** M2 — P1 First End-to-End Demo on Android
**Status:** Approved 2026-05-26
**Predecessor:** [Phase 2.3 design revision](2026-05-26-phase-2.3-design-revision.md) (env-var bridge GREEN across 3 runtimes)
**Successor scope (named here, not built here):** Phase 2.5 (widget mapper), Phase 2.7 (real Hello + screenshot)

---

## Goal

Close the `.NET → host` runtime loop for a single render snapshot. Mount a sentinel component in `Main`, produce one `RenderFrame`, emit it via tagged stdout, parse it back into a Kotlin `RenderFrame` data class on the host side, and hand the parsed frame to a `MobileBridgeHandlers.onFrame` callback the way Phase 2.3 handed `platformInfo` in the opposite direction.

After this phase, Phase 2.5 can plug widget mapping into the `onFrame` slot without inventing transport, and Phase 2.7 can swap the sentinel for the real Hello component without restructuring the pipeline.

## Brainstorm decisions

| # | Question | Decision | Rationale |
|---|---|---|---|
| Q1 | Runtime shape — one-shot vs. streaming? | **C: one-shot snapshot for M2 demo, plus a solution-bound streaming spike at the end of the phase** | The M2 demo (DoD #7) doesn't need streaming. Doing the spike inside 2.4 (rather than as a follow-up phase) means Phase 2.5+ starts on a validated streaming path with no design churn. |
| Q2 | Transport channel? | **A: tagged `[FRAME] {json}` lines on stdout** | The stdout pipeline is the single most-validated piece of M2 — Phase 2.1/2.2/2.3 all proved it round-trips across wasmtime CLI, JVM JNA, and Android JNA. The line-prefix tagging convention is already established by `[BOOT]` markers. Zero new infrastructure. |
| Q3 | Host-side boundary? | **B: `MobileBridgeHandlers.onFrame: (RenderFrame) -> Unit` callback** | Matches the Phase 2.3 shape (handlers data class, defaults stub, host wires real impl). Cleanly separates transport (this phase) from widget mapping (Phase 2.5). |
| Q4 | What drives the frame? | **A: sentinel `_BridgeFrameSelfTest` component in `BlazorNative.WasiHost`** | Transport is the phase's job, not "what does Hello look like." Sentinel emits deterministic patches the tests can assert against, and stays as a regression fixture after Phase 2.7 lands the real Hello. |
| Q5 | Streaming spike sizing? | **B (revised): solution-bound investigation ladder, not time-bound** | The spike must produce a *validated streaming path* for Phase 2.5+, not just "we tried the cheapest one." Stops when one rung has a green test on JVM + Android; documents every failed rung explicitly so the doc is the artifact regardless of which rung succeeds. |

## Architecture & Data Flow

```
[.NET in WASM]
  Main → renderer.Mount<_BridgeFrameSelfTest>()
       → NativeRenderer.UpdateDisplayAsync(batch)
       → DispatchFrame(frame)
       → Console.WriteLine("[FRAME] " + json)    // sync; was async/broken
            ↓
        wasi:cli/stdout
            ↓
        wasi_config_set_stdout_file (existing host capture pipeline)
            ↓
[Host (JVM/Android/CLI)]
  WasiHost.loadAndRun reads captured stdout after run() returns
       → FrameStreamParser.parse(stdout) → List<RenderFrame>
       → handlers.onFrame(frame) for each
```

**Why this works on all three runtimes:**
- wasmtime CLI: `--stdout=...` redirects, our `WasmtimeRunner` already captures stdout to a string.
- JVM JNA: `wasi_config_set_stdout_file` writes to a temp file; existing `WasiHost.loadAndRun` reads it back.
- Android JNA: same `WasiHost.loadAndRun` (Phase 2.2 unified the JVM + Android code path); the captured stdout flows into `MainActivity` already.

## .NET-Side Changes

### New: `src/BlazorNative.WasiHost/_BridgeFrameSelfTest.cs`

Internal sentinel component. Hand-written `BuildRenderTree` (no `.razor` file so WasiHost stays free of Razor build infra):

```csharp
internal sealed class _BridgeFrameSelfTest : ComponentBase
{
    protected override void BuildRenderTree(RenderTreeBuilder b)
    {
        b.OpenElement(0, "div");
        b.AddContent(1, "frame-self-test");
        b.CloseElement();
    }
}
```

Deterministic emitted shape: one `CreateNodePatch("view")` + one `CreateNodePatch("text")` + one `ReplaceTextPatch("frame-self-test")` + one `CommitFramePatch`. Tests assert on these exact patch types.

### Modified: `src/BlazorNative.Renderer/NativeRenderer.cs`

**Add sync mount API** (replaces the GetResult / static-flag mitigation paths originally considered — user feedback during brainstorm: no `.GetAwaiter().GetResult()` calls anywhere):

```csharp
public int Mount<TComponent>(ParameterView parameters = default)
    where TComponent : IComponent
{
    var task = MountAsync<TComponent>(parameters);
    if (!task.IsCompletedSuccessfully)
    {
        throw new InvalidOperationException(
            "Mount<T> requires the first render to complete synchronously. " +
            "Component has async lifecycle work that needs a multi-threaded " +
            "scheduler — not supported on Mono-WASI (Phase 1.2 finding).");
    }
    return task.Result;
}
```

Why this is safe: `task.Result` on an *already-completed* task returns immediately without entering any wait path, so no `Task.InternalWaitCore` PNSE trap is possible. The assertion encodes the contract — components mounted via `Mount<T>` must have sync first-render. Async-lifecycle components throw with a clear diagnostic instead of trapping silently.

`MountAsync<T>` is retained for future async-capable callers (browser hosts, tests with multi-threaded schedulers).

**Replace `DispatchFrameAsync`** (currently calls `await _bridge.WriteStorageAsync` + `await _bridge.FetchAsync`, both of which throw `NotImplementedException` — dead code) with a sync emitter:

```csharp
private void DispatchFrame(RenderFrame frame)
{
    var json = JsonSerializer.Serialize(frame, RendererJsonContext.Default.RenderFrame);
    Console.WriteLine($"[FRAME] {json}");
}
```

`UpdateDisplayAsync` swaps `_ = DispatchFrameAsync(frame)` for `DispatchFrame(frame)`. The in-process `_frames` event still fires for renderer-tests; the `_bridge.WriteStorage` / `FetchAsync` calls go away entirely.

### Modified: `src/BlazorNative.WasiHost/WasiEntryPoint.cs`

After the existing `[BOOT] bridge-ok` marker:

```csharp
Console.WriteLine("[BOOT] mounting sentinel");
renderer.Mount<_BridgeFrameSelfTest>();
Console.WriteLine("[BOOT] frame-emitted");
```

## Host-Side Changes

### New: `src/BlazorNative.Jni/src/main/kotlin/io/blazornative/jni/RenderFrame.kt`

Kotlin mirror of `.NET` `RenderFrame` + `RenderPatch` using `kotlinx.serialization` with sealed-class polymorphism keyed on `"op"`:

```kotlin
@Serializable
data class RenderFrame(val frameId: Int, val timestampMs: Long, val patches: List<RenderPatch>)

@Serializable
@JsonClassDiscriminator("op")
sealed class RenderPatch {
    @Serializable @SerialName("create")  data class CreateNode(val nodeId: Int, val nodeType: String, val parentId: Int? = null) : RenderPatch()
    @Serializable @SerialName("prop")    data class UpdateProp(val nodeId: Int, val name: String, val value: String?) : RenderPatch()
    @Serializable @SerialName("append")  data class AppendChild(val parentId: Int, val childId: Int, val atIndex: Int = -1) : RenderPatch()
    @Serializable @SerialName("remove")  data class RemoveNode(val nodeId: Int) : RenderPatch()
    @Serializable @SerialName("text")    data class ReplaceText(val nodeId: Int, val text: String) : RenderPatch()
    @Serializable @SerialName("style")   data class SetStyle(val nodeId: Int, val property: String, val value: String?) : RenderPatch()
    @Serializable @SerialName("event")   data class AttachEvent(val nodeId: Int, val eventName: String, val handlerId: Int) : RenderPatch()
    @Serializable @SerialName("detach")  data class DetachEvent(val nodeId: Int, val handlerId: Int) : RenderPatch()
    @Serializable @SerialName("commit")  data class CommitFrame(val frameId: Int, val timestampMs: Long) : RenderPatch()
}
```

Adds `kotlinx-serialization-json` 1.7.x dependency to `build.gradle.kts` and the `kotlinx.serialization` plugin.

### New: `src/BlazorNative.Jni/src/main/kotlin/io/blazornative/jni/FrameStreamParser.kt`

```kotlin
object FrameStreamParser {
    private const val PREFIX = "[FRAME] "
    private val json = Json { ignoreUnknownKeys = true; classDiscriminator = "op" }

    fun parse(stdout: String): List<RenderFrame> =
        stdout.lineSequence()
            .filter { it.startsWith(PREFIX) }
            .map { json.decodeFromString<RenderFrame>(it.removePrefix(PREFIX)) }
            .toList()
}
```

### Modified: `MobileBridgeHost.kt`

```kotlin
data class MobileBridgeHandlers(
    val platformInfo: () -> String = { "{}" },
    val onFrame: (RenderFrame) -> Unit = {}
)
```

`Defaults.handlers` unchanged shape — both fields take their defaults.

### Modified: `WasiHost.kt`

After `loadAndRun` reads `stdoutFile` (post-`func_call`), parse and dispatch frames:

```kotlin
val frames = FrameStreamParser.parse(stdout)
frames.forEach { handlers.onFrame(it) }
```

Placed in `loadAndRun` (not `runWasm`) since `runWasm` shouldn't grow another concern.

### Modified: `AndroidPlatformInfo.kt`

Extend `handlers` with a logging `onFrame`:

```kotlin
val handlers = MobileBridgeHandlers(
    platformInfo = { /* existing Build.* JSON */ },
    onFrame = { frame -> Log.i("BlazorNative", "Frame ${frame.frameId}: ${frame.patches.size} patches") }
)
```

`MainActivity` automatically picks this up via `AndroidPlatformInfo.handlers` (no change there beyond the frame-log line appearing in the displayed TextView, which already echoes stdout — the `Log.i` adds a parallel structured line).

## Tests

### `.NET` — extend `tests/BlazorNative.Wasi.Tests/BootSmoke.cs`

After the existing `[BOOT] bridge-ok` assertion:

- `Assert.Contains("[BOOT] frame-emitted", stdout)` — proves `Mount` completed sync.
- `Assert.Matches(@"\[FRAME\] \{.*""op"":""commit""", stdout)` — proves at least one frame reached stdout with a commit patch.

### `.NET` — new `tests/BlazorNative.Wasi.Tests/FrameSelfTestParsesAsRenderFrame.cs`

Extracts the first `[FRAME] ` line from captured stdout, deserializes via `RendererJsonContext.Default.RenderFrame`, asserts:

- `Patches.Length >= 3` (view + text + replace-text minimum)
- `Patches.OfType<CommitFramePatch>().Count() == 1`
- One `CreateNodePatch` with `NodeType == "view"`

### JVM — new `tests/.../FrameDispatchTest.kt`

```kotlin
@Test fun dispatches_sentinel_frame_via_onFrame() {
    val captured = mutableListOf<RenderFrame>()
    val handlers = MobileBridgeHandlers(onFrame = { captured.add(it) })
    WasiHost.loadAndRun(wasmPath, handlers)
    assertTrue(captured.isNotEmpty(), "no frames captured")
    val frame = captured.first()
    assertTrue(frame.patches.any { it is RenderPatch.CommitFrame })
    assertTrue(frame.patches.any { it is RenderPatch.CreateNode && it.nodeType == "view" })
}
```

### JVM — new `tests/.../FrameStreamParserTest.kt`

Standalone parser test against synthetic stdout — guards the line-prefix contract without needing a wasmtime run.

### Android — extend `BootSmokeAndroidTest.kt`

Same callback assertion as JVM (`captured.isNotEmpty()` + one commit + one create), wired via `MobileBridgeHandlers(onFrame = ...)`. Confirms the parse path works on Android's Dalvik JSON pipeline.

## Streaming Spike (solution-bound)

Stops when one investigation rung has a green test on JVM + Android proving the host received a `[FRAME]` line *before* run-completion. Every failed rung is recorded with concrete evidence in `docs/plans/2026-05-26-phase-2.4-streaming-paths.md`.

### Investigation ladder

1. **Tee'd stdout.** Background poller reads `stdoutFile` during the run; check whether `[FRAME] ` lines appear before `wasmtime_component_func_call` returns. If `wasi_config_set_stdout_file` flushes line-by-line on the libwasmtime/Mono-WASI side, this is the answer.
2. **Stdout `pipe()` instead of file.** If file-backed capture buffers, swap to a JNA-bound `wasi_config_set_stdout` against a pipe FD (POSIX `pipe()` on JVM; `ParcelFileDescriptor.createPipe()` on Android). Background thread reads the pipe.
3. **Custom export pump (`pump_frames() → byte[]`).** `[UnmanagedCallersOnly]` export returns the next queued frame's bytes (or empty). Host calls in a loop on a background thread. Requires Phase 1.3 trim-root attention (`[DynamicDependency]`) and a multi-export call sequence we haven't exercised yet for non-stdlib exports.
4. **wasi:io stream binding.** Use the standard `wasi:io/streams` interface — proper component-model way. Most complex, but principled fallback.

### Driving longer-running observations

The sentinel as designed completes in milliseconds. To force a longer observation window, gate a second mount + `Thread.Sleep(200)` in `Main` behind an env var the host sets only for the streaming-spike test (`BLAZOR_STREAMING_SPIKE=1`). Avoids a second AOT publish.

### Done when

One rung has a green test on JVM + Android proving the host received a `[FRAME]` line *before* `func_call` returns, with measured latency. That rung is Phase 2.5+'s starting point. The doc names it.

### If all four rungs fail

The doc says so explicitly with the evidence for each, proposes a Phase 2.4b investigation phase (upstream wasmtime issue, .NET runtime change, wasi-experimental SDK bump). M2's one-shot path still ships — streaming becomes its own milestone-level concern.

## Out of Scope

- **Reviving `_bridge.WriteStorageAsync` / `_bridge.FetchAsync`.** Both still throw `NotImplementedException` after this phase. Phase 2.4 deletes the dead calls from `DispatchFrameAsync`; it does not implement the stubs. The shell-* deferred imports remain BACKLOG-tracked.
- **Real Hello component / screenshot.** Phase 2.7's job. The sentinel stays as a regression fixture.
- **Widget mapping (`NodeType` → Android widget).** Phase 2.5's job. Android's `onFrame` callback just logs.
- **Event flow (host → .NET button taps).** Phase 2.5+/M3. The event-ingress path from Phase 2.0 (`DispatchEventCore`) exists, but no host wiring for native gestures.
- **Genuinely-async component support.** `Mount<T>` throws if first render doesn't complete sync. Async-lifecycle support deferred to whenever a real component needs it (probably M3).
- **Streaming production wiring.** The spike validates a path; it does *not* ship streaming. Phase 2.5+ does the production wiring on the validated path.

## Risks

| # | Risk | Mitigation |
|---|---|---|
| 1 | The sentinel's mount-and-render chain returns a non-completed Task on Mono-WASI despite the sync-only renderer code path | `Mount<T>` throws a clear diagnostic naming the exact reason. Surfaces immediately during BootSmoke. Fallback: investigate which framework `await` is genuinely non-sync on Mono-WASI; potentially work around by driving via `Dispatcher.InvokeAsync` directly with a TaskCompletionSource. |
| 2 | `kotlinx.serialization` adds APK size or proguard friction on Android | Library is ~300 KB, well-understood, no reflection at runtime (compile-time codegen). Already the de-facto choice for KMP-style cross-module serialization. |
| 3 | All four streaming-spike rungs fail | Doc records the evidence; Phase 2.4b investigation phase becomes its own scope. M2 one-shot path still ships — DoD #7 doesn't depend on streaming. |
| 4 | `[FRAME] ` line prefix collides with future Console output that happens to start with the same string | The probability is essentially zero (we control all stdout writes inside the .wasm boundary) but the parser ignores malformed JSON gracefully and the contract is documented in `PatchProtocol.cs`. |
