## As-builts

- **`BlazorNativeTestRunner`** (Gate 0, the PRE-3.5 MUST): custom `AndroidJUnitRunner` sets
  `BLAZORNATIVE_STRICT=1` in `onCreate` **before** `super.onCreate` — before the instrumented
  Application exists and before any test class loads. Both per-class `@BeforeClass` `Os.setenv`
  copies and all alphabetical-ordering KDoc contracts deleted; every instrumented process is
  strict, filtered runs included (the 3.4-flagged gap on `EventRoundTripAndroidTest` /
  `WidgetMapperTest` is closed by construction, not by convention).
- **`INavigationManager` (Core) + `NativeNavigationManager` (Runtime)**: route table
  `"/"` → `BnDemo`, `"/settings"` → `BnSettingsPage` (values are mount-registry keys — the
  registry stays the single component-name authority). Unknown route → `ArgumentException`
  per the strict conventions. The package lift into `BlazorNative.Navigation` is M6 (ledgered
  below); the contract deliberately lives beside `IMobileBridge` in Core for now.
- **The swap sequence**: host-notify (3.1 `Navigate` callback) → `Unmount`
  (`RemoveRootComponent` — verified FIRST per the design's named risk: `protected internal
  void RemoveRootComponent(Int32)` on Blazor 10.0.9's `Renderer`, callable from
  `NativeRenderer`; the disposal batch synchronously emits RemoveNodePatches covering every
  host-root view) → fresh `TryMount` → **afterSwap** (`CurrentRoute` update + `RouteChanged`
  raise). Current-root tracking clears **before** Unmount (unmount-as-best-effort: a disposal
  fault can't wedge every subsequent swap on a dead root id). Route state and `RouteChanged`
  ride the swap unit — they track **the screen, not the intent** (`04ebc79`): a failed
  deferred swap leaves `CurrentRoute` on the page that's actually mounted and never fires the
  event.
- **`RunAfterDispatch` deferral** (the phase's mid-dispatch finding): Blazor 10 holds
  `_isBatchInProgress` open across event handlers, so `RemoveRootComponent` inside a click
  dispatch throws "Cannot start a batch when one is already in progress". Mid-dispatch swaps
  therefore queue via `NativeRenderer.RunAfterDispatch` and drain at the **outermost** dispatch
  unwind in a `finally` — still before `blazornative_dispatch_event` returns, so the
  dispatch-window pin holds at all three layers. Drain faults map to the rc-2 capture slot,
  and **every** drain fault is stderr-logged (a second fault is no longer silently discarded
  when the slot is taken).
- **Startup-route honor is narrowed** (documented, ABI unchanged): only a session's **first
  mount of the routed default component** (`"BnDemo"`) resolves through the host-restored
  `CurrentRoute` buffer; unknown/empty → `/`. The mount ABI stays name-based.
- **`BnSettingsPage` + the two-page demo**: themed `BnView` + title `BnText` "Settings" +
  "← Back" `BnButton`; `BnDemo` gains `[Inject] INavigationManager` (the first real DI consumer
  in Components) + the "Settings →" `BnButton`. Canonical header + all four pinned shape
  surfaces updated once per the documented procedure (now 4 attaches).
- **Probe exports retired** (`a23114a`): `blazornative_run_trim_probes` (3.0c Gate 4) +
  `blazornative_run_bridge_probes` (3.1) deleted with their runners (`TrimProbes.cs` /
  `BridgeProbes.cs`), Kotlin bindings, and every test that called them — with **per-test
  judgment**: `ShellBridgeAndroidTest.storage_persists_via_sharedpreferences` survives (it
  drives the handlers directly, never the export). `TrimValidationProbes` (test project) +
  `CompositionProbe` (registry) STAY — scaffolding, ledgered in the audit. Verified: exactly
  the 8 `blazornative_*` symbols on all three RIDs, exactly the 4 accepted IL2072s each.
- **Guarded-catch coverage re-pinned** (`1b9f44a`): the probe-test deletion had removed the
  only pin on the ABI's exception posture — review caught the false "covered elsewhere" claim.
  Re-pinned as two focused wire-leg tests: .NET (`FetchBegin` host return code −1 →
  `HostError` `InvalidOperationException`) and Kotlin (`guarded()` maps a handler throw to −1 +
  onError, never a raw throw into JNA).

## Honest boundaries (recorded, not fixed)

- **Non-strict drain faults are log-only.** With `StrictErrors` off (the production default),
  a deferred swap action's renderer fault at depth 0 hits the log-only path: the swap
  "succeeds" from the export's perspective (returns 0) and the fault is stderr detail only.
  Same POC posture as 3.3's strict-mode default; the diagnostics surface is M4+.
- **A failed swap leaves a blank screen while `CurrentRoute` reads the old route.** Unmount
  is best-effort and tracking clears first, so a target-mount failure after a successful
  unmount leaves no root mounted; `CurrentRoute` still reports the (now unmounted) old route
  and `RouteChanged` never fired. Pinned by
  `FailedSwap_InsideClickHandler_FaultsDispatch_RouteStateUntouched` (`a1d57f6`).
- **The host's route buffer updates BEFORE the swap.** Host-notify is deliberately first in
  the sequence, so a failed deferred swap leaves the host's route (`/settings`) ≠ .NET
  `CurrentRoute` (`/`) — the documented host-notify-first divergence, asserted by the
  failed-swap pin.
- **A throwing `RouteChanged` subscriber faults the swap unit** — after the screen already
  swapped (afterSwap runs post-mount). Subscriber isolation is a carryover (below).

## Carryovers

### M5 — host-initiated navigation

`INavigationManager` covers .NET-initiated navigation only. **Back button, deep links, and
any host-initiated route change are explicitly M5** (Android shell completeness: lifecycle,
predictive back, deep links). The 3.1 `CurrentRoute` buffer + `Navigate` callback are the
plumbing it will ride.

### M6 ledger (ships with the packaged component library) — intact, +1 new entry

- **Stringly-typed `FontSize`/`Padding`** — conscious 3.4 decision; switching to numeric
  parameters later is source-breaking for consumers — decide at M6 packaging.
- **Stale-echo/IME-composition artifact under fast typing** — distinguishing a stale echo
  from a genuine programmatic overwrite is sequence-stamping territory (frame/sequence ids on
  the wire).
- **Theme colors literal-pinned in 4 files / 6 sites** — `#FFEEAA`/`#334455` across
  `BnDemo.cs` (canonical), `BnDemoTests.cs`, `BnDemoTest.kt`, `BnDemoAndroidTest.kt`; a
  palette change is a multi-site edit. Shared fixture source when the components package ships.
- **`FrameTestPins`/`FramePins` paired harness extraction** — the structural pin helpers are
  duplicated between the .NET and JVM twins (plus partial copies in the probe/navigation
  twins). Extract the paired harness when the next phase would add another copy.
- **One-type-per-file for the packaged Components project** — `BnDemo.cs` still holds
  `BnThemedPanel` + `BnDemo` (M6 packaging sweep).
- **NEW: navigation lift into `BlazorNative.Navigation`** — the contract lives in Core, the
  impl in Runtime (this phase's locked decision); M6 lifts it into the dedicated package per
  the BACKLOG P5 theme.

### Still open from 3.2/3.3 (ledger continuity)

- **Async-handler dispatch window** — capture-window guarantees assume synchronous handlers;
  `await` in consumer handlers gets more likely every phase.
- **Dispatch-lane starvation watch** — one slow handler head-of-line-blocks the single Kotlin
  dispatch lane.
- **Focus/blur events** — mapped .NET-side since 3.2, never wired to Android listeners.
- **Attach-over-attach stale-watcher re-attach** — a change re-attach leaves the previous
  TextWatcher registered (benign waste under the at-most-once contract); key watchers on
  (node, event) when the path is next open.
- **`RemoveComponent` bucket scan** — O(total live buckets) per disposal; index by
  componentId if keyed lists dispose many children per frame.
- **`TranslateToViewIndex` memoization** — O(subtree slots) per mid-list insert; memoize
  per-diff if list sizes make it measurable.

### New this phase

- **`RouteChanged` subscriber isolation (M4+)** — a throwing subscriber currently faults the
  swap unit after the screen swapped (see honest boundaries); per-subscriber isolation (the
  `DispatchEventCore` multicast pattern) when the diagnostics surface lands.
- **`firstMatch` duplicated across 4 instrumented classes** — the tree-walk helper now has a
  fourth copy in `NavigationAndroidTest`; joins the harness-extraction ledger above.
- **Kotlin `value`-prop posture unchanged** — the write-back arm still does
  `setSelection(length)` (cursor-to-end) and carries the 3.4 IME caveats; navigation didn't
  touch it.
