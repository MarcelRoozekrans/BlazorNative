## 3. Gate 2 — PR CI (the safety net)

`.github/workflows/ci.yml`, trigger: push to `main` + all PRs. Single
`windows-latest` job (everything we verify locally today, verbatim):

1. Pin toolchain: .NET SDK per `global.json` (10.0.3xx band); JDK 21 (Temurin);
   NDK `26.3.11579264` via `sdkmanager` (runner images carry recent NDKs but the
   pinned revision is installed explicitly — setup.ps1 §5 is the reference).
2. `dotnet build` (analyzers ride the build) + full `dotnet test` (expect 177/2).
3. Three RID publishes (win-x64, linux-bionic-x64, linux-bionic-arm64) — expect
   exactly the 4 accepted IL2072s each; **eight-export verification** (dumpbin via
   vswhere on win-x64; `llvm-readelf` from the pinned NDK on both `.so`) — the
   local Gate-4 procedure from 3.5, now on every PR.
4. Gradle `testDebugUnitTest` against the freshly published win-x64 dll (expect 32).
5. Upload artifacts: all three published binaries + the jniLibs staging layout the
   emulator job consumes.
6. NuGet + Gradle caches keyed on lockfiles/csproj hashes; target < ~20 min wall.

Known runner risks (RED-on-ambiguity applies): NativeAOT link on `windows-latest`
needs the VS C++ toolset (present on the image, but first run proves it);
`sdkmanager` NDK install ~2 GB (cacheable).

## 4. Gate 3 — Instrumented job (nightly + manual)

Second workflow (`android-instrumented.yml`): `workflow_dispatch` + nightly cron.
Job A (`windows-latest`): bionic-x64 publish → artifact. Job B (`ubuntu-latest`,
KVM): download artifact into the jniLibs path, `reactivecircus/android-emulator-runner`
(API 34, x86_64, same profile as the local AVD), `connectedAndroidTest` (expect
32/32; the suite is strict-by-runner since 3.5). Gradle's `copyNativeHostSo` task
must accept a pre-staged `.so` (it copies from the publish dir today — a small
`ciSoDir` property override keeps local behavior unchanged).

Honest boundary: emulator-on-CI has known flake modes (boot timeouts); the job is
**informational at first** (not a required check) until a stability baseline exists.

## 5. Gate 4 — Issue structure + branch protection

Run the refreshed script against the repo (labels, milestones, issues per Gate 0).
Branch protection on `main`: require PR, require the CI check, require conversation
resolution, no direct pushes. **Workflow consequence, adopted deliberately: from
Phase 4.1 on, phases merge via PR (`gh pr create` → CI green → merge) instead of
local `merge --no-ff`** — same review rhythm, the merge just happens on GitHub.
Project board: manual per the guide (Backlog/Up Next/In Progress/Done), user-owned.

## 6. Gate 5 — Public flip + close-out

Preconditions: CI green on `main` (the badge is real), instrumented job has ≥1 successful
run, README/issue structure reviewed. Flip `--visibility public` (**controller
confirms with the user immediately before** — publishing is irreversible in
spirit). Conclusion doc per house style.

## Risks

| Risk | Mitigation |
|---|---|
| `gh` auth / org missing | Gate 1 STOPs and reports; user creates org / runs `gh auth login` |
| NativeAOT link fails on the runner image | RED-on-ambiguity: one diagnostic iteration (vswhere paths differ), then stop; fallback = self-hosted runner decision escalated |
| NDK 26.3 unavailable via sdkmanager on the image | It's a pinned archived revision — sdkmanager serves it; cache post-install |
| Emulator CI flake | Informational-first; retry policy documented; never a required check this phase |
| Branch protection breaks the established merge flow | Deliberate switch to PR merges from 4.1; documented in the conclusion + memory |
| Private-repo Actions minutes (Windows = 2× multiplier) | Short-lived: the repo flips public at Gate 5 (public repos = free minutes) |
