# Phase 2.8 Runtime Architecture Evaluation

**Date:** 2026-05-28
**Research method:** WebSearch + WebFetch (not training-data recall)
**Purpose:** Inform M3 starting-point runtime decision

## Verdict per option

- **Option A — wasi-experimental:** 🟡 still officially experimental; .NET 10 GA shipped it but the same architectural pain points (single-thread Mono-WASI, SDK gaps) are still open. Stable enough for M2-style work, not improving fast enough to remove the M2 pain by M3.
- **Option B — componentize-dotnet:** ⚠ uncertain — latest release `v0.7.0-preview00010` is dated **2025-03-20** (over a year stale as of 2026-05-28). Active repo metrics could not be verified from public README/releases. Carries explicit "heavy development, missing features" warning. Targets WASI 0.2 / Wasmtime ~v23 — older than our wasmtime v45.
- **Option C — NativeAOT-per-ABI:** 🟡 preview / "nearly ready" for `linux-bionic-arm64` / `linux-bionic-x64`. Microsoft docs still classify Android NativeAOT as a "limited experience". **Cross-OS compile from Windows is not supported by NativeAOT** — would require WSL or a Linux/macOS host.

## Evidence per question

### Option A: wasi-experimental

**Q: Workload changes / improvements since .NET 10.0.8?**
- The `wasi-experimental` workload tracking issue `dotnet/runtime#65895` (open since 2022-02-25) is still the authoritative tracker; no items I could verify mark "Support .NET threads on threading-enabled wasm runtimes" as complete. Source: <https://github.com/dotnet/runtime/issues/65895>
- A blog post titled ".NET 10: No WASI" frames the .NET 10 cycle as one where WASI work stalled relative to expectations. Source: <https://henrikrxn.github.io/blog/Wasi-dotnet-10/>
- Microsoft's official position: the workload "will remain an experiment at least until WASI itself has a stable 1.0 release." Confirmed via the discussion threads cited from search results.

**Q: Progress on the three Phase 2.3 SDK gaps?**
- (a) `_WasmPInvokeModules` auto-population: no specific issue closing this gap surfaced in search. **No evidence found in current sources.**
- (b) `$(WasmAllowUndefinedSymbols)` actually being consumed: **No evidence found** of a fix in recent .NET 10.0.x patches.
- (c) `wasm-component-ld --component-type` accepting raw `.wit` text: the `wasm-component-ld` tooling design embeds component-type metadata via the `component-type` custom section produced by `wit-bindgen` — i.e. the tool expects the binary blob path; no plans surfaced to accept raw WIT text. Source: <https://github.com/bytecodealliance/wasm-component-ld>, <https://github.com/bytecodealliance/wasmtime/issues/13190>

**Q: Active issues / recent PRs on Mono-WASI threading or async?**
- `dotnet/runtime#102894` "[wasi] async/await on WASIp2" — marked Closed, milestone 10.0.0. Resolution comments were not retrievable from the visible page summary; the close may be re-scoping rather than implementation. Source: <https://github.com/dotnet/runtime/issues/102894>
- `dotnet/runtime#117848` ".NET 10 Preview 6 wasiconsole cannot run out-of-the-box" (opened 2025-07-19, assigned to Pavel Savara). Indicates the wasi-experimental template was broken at the .NET 10 preview boundary and the fix path was still being worked. Source: <https://github.com/dotnet/runtime/issues/117848>
- `dotnet/runtime#112926` "Building with WasmBuildNative and threading enabled fails with assertion" — confirms WASI + threading is still not a working combination. Source: <https://github.com/dotnet/runtime/issues/112926>
- `dotnet/runtime#121849` ".NET 10 working WASM project migrated to .NET 10 — MONO interpreter NIY / out of sync" — recent migration breakage report on Mono-WASI. Source: <https://github.com/dotnet/runtime/issues/121849>

**Bottom line for A:** the workload still works (M2 proved it) but **none of the three Phase 2.3 SDK gaps have visible, datable fixes**, and the boot-thread async limitation is still architectural — the WASIp3 async story (component-model async imports/exports) is the long-term answer, not a 10.0.x patch.

### Option B: componentize-dotnet

**Q: Active development state? Last commit / release date?**
- Latest release: `v0.7.0-preview00010`, **dated 2025-03-20**. Source: <https://github.com/bytecodealliance/componentize-dotnet/releases>
- Latest NuGet `BytecodeAlliance.Componentize.DotNet.Wasm.SDK` version aligns with the same `0.7.0-preview00010` / `0.6.0-preview00009` line. Source: <https://www.nuget.org/packages/BytecodeAlliance.Componentize.DotNet.Wasm.SDK>
- README shows "181 Commits" on main; exact last-commit timestamp was not retrievable from the WebFetch summary. **Cannot confirm whether the repo is dormant or quietly active.**

**Q: Works with .NET 10? Supported SDK range?**
- README states ".NET 10+ preview SDK" is required for getting started. The 2025-03-20 release notes specifically called out "Upgraded to .NET 10 preview SDK". Source: <https://github.com/bytecodealliance/componentize-dotnet/blob/main/README.md>, <https://github.com/bytecodealliance/componentize-dotnet/releases>

**Q: Production-ready or experimental?**
- README carries explicit warning: "all the underlying technologies are under heavy development and are missing features." Experimental. Source: README.

**Q: Documentation + sample completeness?**
- Templates package (`BytecodeAlliance.Componentize.DotNet.Templates`) exists; the 2025-03-20 release "updated documentation to use templates for getting started". Sample-level coverage was not deeply audited in this research pass.

**Q: Known limitations vs `wasi-experimental`?**
- Targets Wasmtime v23.0.2 (the M2 baseline is wasmtime v45 — would require either a wasmtime downgrade on the Android shell or hoping v45 stays compatible with WASI 0.2 components produced against the v23 baseline).
- Dynamic memory allocation with strings requires "additional configuration".
- Some WASI imports must be manually implemented by the host.
- Uses **NativeAOT-LLVM** under the hood (not Mono interpreter), so the Phase 2.3 `[UnsafeAccessor]` + `Renderer.DispatchEventAsync` interop pattern would need re-validation under a NativeAOT-LLVM trimmer.

**Q: Does it close the 3 SDK gaps?**
- Custom WIT-typed imports/exports: **Yes** — "any .wit files in your project will automatically generate corresponding C# sources, allowing you to import or export functionality". This is the headline feature and directly addresses the gap that forced us into the env-var bridge in Phase 2.3. Source: README via WebFetch.
- The `_WasmPInvokeModules` and `WasmAllowUndefinedSymbols` issues are wasi-experimental-specific MSBuild concerns; componentize-dotnet sidesteps them by using a different build pipeline entirely.

**Bottom line for B:** the **headline value (real WIT-typed bridge)** is exactly what Phase 2.3 needed. The **price** is: (1) a 14-month-stale latest release, (2) NativeAOT-LLVM compilation (means re-validating `[UnsafeAccessor]` Blazor interop), (3) Wasmtime v23 baseline mismatch with our v45 host, (4) the same "experimental" warning the wasi-experimental workload carries.

### Option C: NativeAOT-per-ABI for Android

**Q: Does .NET 10.0.x ship NativeAOT for `linux-bionic-arm64` / `linux-bionic-x64`?**
- Yes, those are the two supported Bionic RIDs. Standard `android-arm64` / `android-x64` RIDs are **not** supported. Source: <https://github.com/dotnet/runtime/blob/main/src/coreclr/nativeaot/docs/android-bionic.md>
- Status is "limited experience" per the official doc — not full Android support.
- Performance evidence: Galaxy Note 10+ cold start 271–331 ms (NativeAOT) vs 1.27–1.41 s (MonoAOT) on the same project. Source: <https://code.soundaranbu.com/state-of-nativeaot-net10>
- Android NativeAOT for MAUI itself is **post-.NET 10** (slated for servicing, with the path widening when MAUI moves to CoreCLR in .NET 11). Source: <https://devblogs.microsoft.com/dotnet/dotnet-maui-moves-to-coreclr-in-dotnet-11/>

**Q: Cross-compilation from Windows — possible directly?**
- **No.** Official docs: "Native AOT does not support cross-OS compilation. Cross-OS compilation with Native AOT requires some form of emulation, like a virtual machine or Windows WSL." Source: <https://learn.microsoft.com/en-us/dotnet/core/deploying/native-aot/cross-compile> (page updated 2025-08-22)
- Android targets Bionic (Linux-based), so building from a Windows host means **WSL or Docker Linux is mandatory**.
- Counter-data-point: the `NativeAOT-AndroidHelloJniLib` sample claims "This tool works on both Windows and Linux, so WSL environment is not needed" — but it relies on **BFlat** (a third-party AOT toolchain), not the Microsoft NativeAOT pipeline. So the "Windows-direct" path exists only outside the supported MS toolchain. Source: <https://github.com/josephmoresena/NativeAOT-AndroidHelloJniLib>

**Q: `[UnsafeAccessor]` + NativeAOT?**
- NativeAOT has **the best behavior** of the three toolchains (NativeAOT, ILC, Trimmer) for `[UnsafeAccessor]` — it produces warnings only when the accessor target itself would warn under direct access. Properly annotated accessors compile clean. Source: <https://github.com/dotnet/runtime/issues/101195>
- Caveat: .NET 10's new `MobileAggressiveAttributeTrimming` strips `[DynamicallyAccessedMembers]` before ILC sees them, producing spurious NativeAOT trim warnings on mobile builds. This is documented and tracked (`dotnet/macios#22065`).

**Q: `[DynamicDependency]` + NativeAOT?**
- Generally supported (it's the canonical way to preserve members across the trimmer). Specific Android-bionic-NativeAOT issues not found in this search pass. **Treat as low-risk pending validation.**

**Q: Microsoft-published "NativeAOT for Android" template / sample?**
- The closest official artifact is the `android-bionic.md` doc in `dotnet/runtime`, which references the community sample `josephmoresena/NativeAOT-AndroidHelloJniLib`. **No first-party Microsoft template was found.**

**Q: State of `NativeAOT-LLVM` (runtimelab feature branch)?**
- Active experimental branch at `dotnet/runtimelab` `feature/NativeAOT-LLVM`. Targets browser-wasm and wasi-wasm. Known rough edges: manual Emscripten install, intermittent build failures, `STATUS_ACCESS_VIOLATION` reports (issues `#2522`, `#2378`, `#2384`). Source: <https://github.com/dotnet/runtimelab/tree/feature/NativeAOT-LLVM>
- This is the toolchain componentize-dotnet builds on top of. So Options B and C-via-LLVM share a stability story.

**Bottom line for C:** NativeAOT-per-ABI sidesteps every wasi-experimental pain point (real threading, real async, ~5× startup speed, mature `[UnsafeAccessor]` story) **but** costs us (1) the "single-binary-runs-anywhere" property — back to per-ABI native libs alongside libwasmtime, (2) a mandatory WSL/Linux build host, (3) "limited experience" official status until at least .NET 11 servicing for MAUI-adjacent scenarios.

## Recommendation for M3 kickoff

**Start M3 on Option A (wasi-experimental) and budget Phase 3.0 as a 1-week timeboxed Option B spike.**

Rationale:
- M2 already proved Option A boots end-to-end on Android. Switching toolchains *between* milestones (rather than within) is more disciplined.
- The Phase 2.3 env-var bridge is functional. The 6 deferred `mobile_bridge` exports can be re-implemented over the env-var transport without blocking on a real WIT-typed bridge, which buys us time to validate Option B properly rather than rushing the switch.
- Option B (`componentize-dotnet`) is the only candidate that **directly closes the Phase 2.3 SDK-gap-(c) pain** (custom WIT-typed imports). It deserves a real evaluation but **not before we confirm its repo is still alive** — the 2025-03-20 latest release is a red flag.
- Option C (NativeAOT-per-ABI) is a **strategic re-platform**, not a tactical improvement. Worth keeping on the roadmap for an eventual M4/M5 but premature for M3 — would require WSL build infrastructure we don't have, and Android NativeAOT is still "limited experience" per Microsoft's own doc.

**Gating questions for the Phase 3.0 Option-B spike:**
1. Is the `componentize-dotnet` repo actually still maintained? (Check commit graph directly via `gh api`.)
2. Does a `BytecodeAlliance.Componentize.DotNet.Wasm.SDK` build produce a `.wasm` (or `.wasm` component) that wasmtime v45 can load on Android, given the v23 baseline mismatch?
3. Does the existing `[UnsafeAccessor]`-based `BlazorInterop.DispatchEventAsync` survive NativeAOT-LLVM compilation?
4. What is the resulting `.wasm` size vs the current 14 MB Mono-AOT artifact? (If similar or smaller — strong "switch" signal. If much larger — re-think.)

**Spike exit criteria:** GREEN BootSmoke on Android emulator with a single WIT-typed `host.log` import wired through the componentize-dotnet pipeline. If GREEN in ≤ 1 week → switch M3's remaining phases to Option B. If RED or inconclusive → continue M3 on Option A and revisit at M4.

## Notes on uncertainty

**Well-evidenced:**
- NativeAOT cross-OS compilation is not supported (Microsoft docs, updated 2025-08-22).
- `linux-bionic-arm64`/`x64` are the only Android NativeAOT RIDs.
- componentize-dotnet's latest release is dated 2025-03-20.
- `wasi-experimental` is still classified as experimental and the tracking issue `dotnet/runtime#65895` has open threading items.

**Uncertain / not retrievable from this research pass:**
- Whether `dotnet/runtime#102894` (async/await on WASIp2) actually shipped working code or was closed as re-scoped. The visible page summary did not include resolution comments.
- Exact last-commit date and contributor activity on `bytecodealliance/componentize-dotnet` main branch — the WebFetch summary showed "181 Commits" but no timestamp. A direct `gh api repos/bytecodealliance/componentize-dotnet/commits` call would resolve this.
- Concrete `.NET 10.0.x` patch-by-patch changelog entries for the three Phase 2.3 SDK gaps. **No evidence of fixes found**, but absence of evidence is not evidence of absence — a deeper read of `dotnet/sdk` and `dotnet/runtime` release notes per 10.0.1 through 10.0.N could surface fixes this search missed.
- `[DynamicDependency]` + NativeAOT-bionic specific gotchas. No specific issues surfaced; assume low-risk pending Phase 3.0 spike validation.

**Sources flagged as weak:**
- The "State of NativeAOT in .NET 10" article (<https://code.soundaranbu.com/state-of-nativeaot-net10>) is a single developer's blog — good corroboration but not authoritative on its own.
- `henrikrxn.github.io/blog/Wasi-dotnet-10` ("No WASI") is similarly a personal blog; the *framing* of WASI stalling in .NET 10 matches the public issue tracker evidence but is one person's view.
- WebFetch did not retrieve comment threads on GitHub issues, only the OP body — so issue-resolution status is consistently weaker evidence than issue-existence.
