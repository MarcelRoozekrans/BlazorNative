# Phase 10.1 — Version governance (M10 DoD #3, #4)

**Milestone:** 10. **Phase:** 10.1. **Closes:** DoD #3 (#120 stale `Exports.VersionNumber`)
and DoD #4 (#122 unguarded `RuntimeFrameworkVersion`). `.NET`/config only — no shells, no ABI.

## Gate A — #120: the consumer-visible runtime version is stale and ungoverned

### The defect
`src/BlazorNative.Runtime/Exports.cs:98` — `internal const string VersionNumber =
"1.4.0-phase-5.4";` — an **8th, ungoverned** version literal, frozen ~4 milestones back. It
is consumer-visible in THREE places:
- `Exports.cs:102` — the version string `"BlazorNative.Runtime {VersionNumber}"` returned via
  the version/init path;
- `NativeShellBridge.cs:909` — the `PlatformInfo` JSON version field;
- `NativeShellBridge.cs:924` — `GetPlatformInfoAsync`'s `AppVersion`.
The package is **0.1.0** (0.1.1 pending), so a consumer reading the runtime version gets
`1.4.0-phase-5.4`. It is in **no test** and **not** in release-please's `extra-files`.

### The fix — bring it INTO the apparatus as a governed mirror
Make it a release-please mirror like `<Version>` in the props:
- Change to the CURRENT package version with the annotation:
  `internal const string VersionNumber = "0.1.0"; // x-release-please-version`
  (release-please's generic updater rewrites the first semver on an annotated line — mirror
  the props' `<!-- x-release-please-version -->` convention).
- Add `{ "type": "generic", "path": "src/BlazorNative.Runtime/Exports.cs" }` to
  `release-please-config.json` `extra-files`. Now every release bumps it in lockstep with the
  props (the open 0.1.1 release PR #142 will re-generate to bump it too).
- **Drift pin:** extend `PackageVersionPinTests` — assert `Exports.VersionNumber` equals the
  props `<Version>` (the same manifest↔props equality the props already gets). Update any
  literal/file-count invariant in that test suite to include the new governed file.

### Red-first
A test asserting `Exports.VersionNumber == <props Version>` **FAILS** on the current
`1.4.0-phase-5.4` (≠ `0.1.0`) and passes after the value + governance change. That is the
natural red-first: the stale value is the bug.

### Watch
- Any test asserting the OLD version string format (`1.4.0-phase-5.4` / the `BlazorNative.
  Runtime …` string) — search and update to the governed value. The version *string* the
  export returns changes from `BlazorNative.Runtime 1.4.0-phase-5.4` to `… 0.1.0`.
- The value must equal the props on `main` (0.1.0) so the pin is green at merge; release-please
  moves both to 0.1.1 together in #142.

## Gate B — #122: the one load-bearing version that isn't guarded

### The defect
`RuntimeFrameworkVersion 10.0.9` is hardcoded in **3** csproj spots with **no** test linking
them:
- `samples/BlazorNative.SampleApp/BlazorNative.SampleApp.csproj:66` and `:86`
- `templates/BlazorNative.Templates/content/BlazorNative.App/MyBlazorNativeApp.csproj:57`
Its own comment says it exists *"so a servicing release can't silently change the toolchain
under us"* — it is the version that makes bionic/iOS NativeAOT compile at all — yet it is the
one version the repo does not drift-guard, while every cosmetic literal is pinned.

### The fix — a drift pin linking the occurrences
Add a drift test (extend `PackageVersionPinTests` or a focused new test) that parses the
`RuntimeFrameworkVersion` value out of all occurrences across the sample + template csproj and
asserts they are **identical** (and, ideally, names the canonical value so a deliberate bump
is a one-line, test-visible change). Pattern-derive the value (do not hardcode `10.0.9` as a
second copy the way the finding warns against — read it from the files and assert agreement).

### Red-first (mutation)
The occurrences currently AGREE, so the pin is a guard, not a bug-fix. Prove it BITES: a
throwaway mutation setting one occurrence to a different version must make the new test FAIL
(demonstrated + reverted), so the guard is not vacuously green.
