# Design — Mono-WASI Async-Trap Remediation (Phase 2.0)

*Brainstormed: 2026-05-25 — Milestone 2, Phase 2.0*

## Problem

.NET 10 Mono-WASI throws `PlatformNotSupportedException` from `Task.InternalWaitCore` whenever code synchronously waits on an incomplete `Task` — there is no thread to block on in single-threaded WASI. Phase 1.2 confirmed this by removing `async Main`. M1's final audit (risk #1) flagged that the same trap would fire on the first real bridge event in M2 because two production paths still use `.AsTask().GetAwaiter().GetResult()` and `await` chains that originate inside an `[UnmanagedCallersOnly]` callback.

Detailed analysis during the Phase 2.0 brainstorm narrowed the actual trap surface:

- **All `WasiBridge` methods (`NavigateAsync`, `FetchAsync`, `WriteStorageAsync`, etc.)** return `ValueTask` shapes but complete **synchronously** (the underlying `Native.shell_*` extern is sync). Awaiting them in `NativeRenderer.DispatchFrameAsync` is safe today — completed `ValueTask`s don't suspend.
- **The only trap site is `WasiBridge.DispatchEvent`'s `.AsTask().GetAwaiter().GetResult()`** — and it only fires when a subscriber's handler returns an *incomplete* `ValueTask`. Current sole subscriber (`Home.razor`) returns `ValueTask.CompletedTask` synchronously, so the trap is dormant today. **First real failure point:** the first time any developer writes `Bridge.NativeEvents += async (e, ct) => { await SomethingReal(); ... };`.

The trap is therefore conditional on subscriber behaviour, not guaranteed. But the conditional path is the *natural* path for a Blazor developer (handlers that do real work tend to await). Phase 2.0 closes the trap structurally so it cannot fire regardless of subscriber code shape.

## Decisions

| # | Decision | Chosen | Rejected alternatives |
|---|---|---|---|
| 1 | Should bridge-event handlers be allowed to genuinely `await`? | **No** — `IMobileBridge.NativeEvents` shape changes from `event AsyncEvent<NativeEvent>` to `event Action<NativeEvent>`. Handlers complete synchronously; async follow-up uses `Dispatcher.InvokeAsync(...)` fire-and-forget. | A: keep `AsyncEvent<NativeEvent>` + queue-and-drain with a pump export (more plumbing, relies on uncertain Mono-WASI scheduler-tick semantics); C: renderer-in-shell architectural pivot (out of M2 scope; doesn't actually solve the underlying trap). |
| 2 | Compile-time enforcement of "no async handlers" | **Yes** — new `BN0014` analyzer rule in `BlazorNative.Analyzers` flags async lambdas (and async-void/Task-returning methods) registered against `event Action<NativeEvent>`. Severity: error. | B: API shape only, trust convention (insufficient — `Action<T>` accepts `async void` lambdas which silently reintroduce the trap); C: warning instead of error (warnings get ignored). |
| 3 | Test strategy | **Three layers:** (a) pure managed unit tests via the new `DispatchEventCore` entry point — millisecond feedback; (b) WASI end-to-end via a new `[BOOT] event-ok` marker emitted by `Main`'s self-test, asserted by extended `BootSmoke`; (c) `Microsoft.CodeAnalysis.Testing`-based smoke tests for BN0014 in a new `BlazorNative.Analyzers.Tests` project. | Just (b) (no fast feedback loop for bridge logic); just (a) (doesn't exercise actual Mono-WASI runtime); just (c) (doesn't prove the trap is closed). |

## Architecture

```
                            BEFORE (M1):
                            ─────────────

   Android shell ──── blazornative_dispatch_event(name, payload) ────┐
                                                                      │
                                                                      ▼
   [UnmanagedCallersOnly] public static void DispatchEvent(...)
                          {
                              ...
                              _events.InvokeAsync(...)             ← AsyncEventHandler
                                     .AsTask()
                                     .GetAwaiter()
                                     .GetResult();                 ← TRAP if any
                          }                                            subscriber handler
                                                                       is async


                            AFTER (Phase 2.0):
                            ──────────────────

   Android shell ──── blazornative_dispatch_event(name, payload) ────┐
                                                                      │
                                                                      ▼
   [UnmanagedCallersOnly] public static void DispatchEventNative(...)
                          {
                              ...
                              DispatchEventCore(name, payload);    ← refactored,
                          }                                            managed-callable

                          internal static void DispatchEventCore(name, payload)
                          {
                              if (Current is not { } c || c._events is null) return;
                              foreach (var h in c._events.GetInvocationList())
                                  try { ((Action<NativeEvent>)h)(evt); }
                                  catch (Exception ex) { Console.Error.WriteLine(...); }
                          }                                          ← sync multicast,
                                                                       no Task, no await,
                                                                       no trap. Per-handler
                                                                       try/catch so one
                                                                       subscriber's bug
                                                                       doesn't strand
                                                                       siblings.


                            IMobileBridge contract change:
                            ──────────────────────────────

                            BEFORE:  event AsyncEvent<NativeEvent> NativeEvents;
                            AFTER:   event Action<NativeEvent>     NativeEvents;

                            Subscribers in Home.razor:
                              (e, ct) => { ...; return ValueTask.CompletedTask; }
                                → e => { ...; }

                            Async follow-up (the existing fire-and-forget pattern):
                              e => { _ = InvokeAsync(StateHasChanged); }


                            New analyzer rule:
                            ──────────────────

   BlazorNative.Analyzers/BridgeAsyncHandlerAnalyzer.cs
   Rule BN0014:
       Fires on:   Bridge.NativeEvents += async e => { ... };
                   Bridge.NativeEvents += async (e) => { ... };
                   Bridge.NativeEvents += SomeAsyncMethod;   // method symbol is async
       Silent on:  Bridge.NativeEvents += e => { ... };       // sync lambda
                   Bridge.NativeEvents += e => { _ = SomeAsync(); }
       Severity:   Error (consistent with BN0001-BN0004, BN0010, BN0013)
       Message:    "Bridge event handlers must complete synchronously on
                    Mono-WASI. Use a synchronous lambda and fire-and-forget
                    async work via Dispatcher.InvokeAsync."
```

## Components — file-by-file

### `src/BlazorNative.Core/IMobileBridge.cs` — contract change

```csharp
// before
event AsyncEvent<NativeEvent> NativeEvents;

// after
event Action<NativeEvent> NativeEvents;
```

Drop `using ZeroAlloc.AsyncEvents;`. The `ZeroAlloc.AsyncEvents` package itself stays — `NativeRenderer` still uses `AsyncEventHandler<RenderFrame>` for its `Frames` event (that's a WASM-internal event, never invoked from an unmanaged callback).

### `src/BlazorNative.Core/WasiBridge.cs` — refactor DispatchEvent + sync event accessor

```csharp
private Action<NativeEvent>? _events;

public event Action<NativeEvent>? NativeEvents
{
    add    => _events += value;
    remove => _events -= value;
}

[UnmanagedCallersOnly(EntryPoint = "blazornative_dispatch_event")]
public static unsafe void DispatchEventNative(byte* namePtr, int nameLen, byte* payloadPtr, int payloadLen)
{
    var name = Encoding.UTF8.GetString(namePtr, nameLen);
    var payload = payloadLen > 0 ? Encoding.UTF8.GetString(payloadPtr, payloadLen) : null;
    DispatchEventCore(name, payload);
}

/// <summary>Managed-callable entry into the bridge event multicast. Invoked by
/// DispatchEventNative from the native shell and by WasiEntryPoint.cs's
/// self-test during boot. Subscribers MUST complete synchronously per the
/// Phase 2.0 sync-contract decision — see this design doc.</summary>
internal static void DispatchEventCore(string name, string? payload)
{
    if (Current is not { } c || c._events is null) return;

    var evt = new NativeEvent(name, payload);
    foreach (var handler in c._events.GetInvocationList())
    {
        try { ((Action<NativeEvent>)handler)(evt); }
        catch (Exception ex)
        {
            Console.Error.WriteLine($"[NativeEvents] subscriber threw: {ex.GetType().Name}: {ex.Message}");
        }
    }
}
```

The Phase 1.3 `[DynamicDependency(All, typeof(WasiBridge))]` on `Program.Main` keeps both methods rooted through Mono-AOT.

### `src/BlazorNative.Core/DevHostBridge.cs` — parallel migration

Same field/accessor/invoke changes as WasiBridge. Two `_events.OnNext`-style call sites (line ~36 in NavigateAsync and line ~106 in InjectEvent) become `_events?.Invoke(...)`. Catch-and-continue multicast happens via a shared internal helper or duplicated foreach — both bridges share the same shape.

### `src/BlazorNative.Blazor/Pages/Home.razor` — subscriber update

```csharp
// before
private AsyncEvent<NativeEvent>? _eventHandler;
...
_eventHandler = (e, ct) =>
{
    Events.Insert(0, e);
    if (Events.Count > 20) Events.RemoveAt(Events.Count - 1);

// after
```

Drop `@using ZeroAlloc.AsyncEvents;`. The add/remove pattern via `Bridge.NativeEvents += _eventHandler;` / `-= _eventHandler;` is unchanged.

### `src/BlazorNative.WasiHost/WasiEntryPoint.cs` — bridge self-test in Main

Insert between the `[BOOT] di-ok` and `[BOOT] done` markers (~10 lines):

```csharp
// Phase 2.0 self-test: prove bridge round-trip works without trapping.
// Registers a sync subscriber, invokes DispatchEventCore from managed code
// (the same path the unmanaged blazornative_dispatch_event export calls
// into), confirms the subscriber fired. This is the Mono-WASI end-to-end
// check that the [Task.InternalWaitCore PNSE] trap from M1 is closed.
var selfTestFired = false;
NativeEvent receivedEvent = default;
Action<NativeEvent> probe = e => { selfTestFired = true; receivedEvent = e; };
bridge.NativeEvents += probe;
WasiBridge.DispatchEventCore("self-test", "phase-2.0");
bridge.NativeEvents -= probe;
Console.WriteLine($"[BOOT] event-ok fired={selfTestFired} name={receivedEvent.Name}");
```

BootSmoke gains one new assertion line; the sequence becomes 4 markers.

### `src/BlazorNative.Analyzers/BridgeAsyncHandlerAnalyzer.cs` — new file

Implements rule BN0014. Hooks `SyntaxKind.AddAssignmentExpression` (and reasonably also `RemoveAssignmentExpression` for symmetry — though we don't need to flag removes). Checks:

1. Left operand is a member access targeting `IMobileBridge.NativeEvents` (or any `event Action<NativeEvent>` on a type implementing `IMobileBridge`). Symbol-based check via `IPropertySymbol.ContainingType` and event-symbol comparison.
2. Right operand is either:
   - A `SimpleLambdaExpressionSyntax` or `ParenthesizedLambdaExpressionSyntax` with `AsyncKeyword` set
   - An `IdentifierNameSyntax` or `MemberAccessExpressionSyntax` whose symbol is a method with `IsAsync == true` (covers `Bridge.NativeEvents += SomeAsyncMethod;`)

If both conditions hold, report BN0014 on the right-operand location with the message above. ~80 lines + the existing `WasiThreadingAnalyzer.cs` provides a template for the boilerplate.

### `tests/BlazorNative.Renderer.Tests/BridgeEventTests.cs` — Layer 1 tests

Four `[Fact]`s (single sync subscriber fires; multi-subscriber multicast; throwing-subscriber catch-and-continue; unsubscribe semantics). Lives in the renderer-tests project — no WASI fixture needed. Imports `BlazorNative.Core` and uses `WasiBridge.DispatchEventCore` directly.

### `tests/BlazorNative.Wasi.Tests/BootSmoke.cs` — Layer 2 assertion

Single line added to `WasmModule_BootsCleanly_UnderWasmtime`:

```csharp
Assert.Contains("[BOOT] event-ok fired=True name=self-test", stdout);
```

### `tests/BlazorNative.Analyzers.Tests/` — new project + Layer 3 tests

- `BlazorNative.Analyzers.Tests.csproj` (xUnit + `Microsoft.CodeAnalysis.Testing.MSTest` + a project reference to the analyzers project for `Microsoft.CodeAnalysis.Testing`'s analyzer-discovery)
- `BridgeAsyncHandlerAnalyzerTests.cs` — at minimum two `[Fact]`s: `BN0014_FiresOnAsyncLambdaSubscription`, `BN0014_SilentOnSyncLambda`. Uses the standard `{|BN0014:...|}` markup syntax to assert diagnostic location.
- Added to `BlazorNative.sln`.

This is the **infrastructure-establishing** part of the test work — once it's in place, BACKLOG P3's "Analyzer unit tests" task can drop in tests for BN0001-BN0013 incrementally without re-scaffolding.

## Data flow

### Flow A — Boot self-test (Phase 2.0, runs in WASI under wasmtime)

```
wasmtime BlazorNative.WasiHost.wasm
      │
      ▼
Program.Main()
      │
      │ [BOOT] runtime-start
      ▼
Build DI graph, resolve IMobileBridge (→ WasiBridge instance)
      │
      │ [BOOT] di-ok bridge=WasiBridge renderer=NativeRenderer
      ▼
bridge.NativeEvents += probe;                  // Action<NativeEvent>
      ▼
WasiBridge.DispatchEventCore("self-test", "phase-2.0")
      │  (managed call, same code path as unmanaged callback)
      ▼
WasiBridge.Current._events.GetInvocationList() → [probe]
      ▼
probe(evt)                                     // runs to completion synchronously
      │                                            (no await, no scheduler interaction)
      ▼
bridge.NativeEvents -= probe;
      ▼
[BOOT] event-ok fired=True name=self-test       ← new marker
      ▼
[BOOT] done
      ▼
return 0
```

### Flow B — Real native event (M2 Phase 2.3+, runs in the Android shell)

```
Android UI thread (Looper)
      │
      │ user taps Button → Kotlin onClick → onClickFromBlazor(routePtr...)
      ▼
JNI / wasmtime-java
      ▼
wasmtime invokes export "blazornative_dispatch_event"
      ▼
[UnmanagedCallersOnly] DispatchEventNative(byte* namePtr, ...)
      ▼
string allocate, UTF8 decode
      ▼
DispatchEventCore(name, payload)               ← same managed entry point as
      │                                          the self-test in Flow A
      ▼
foreach (handler in _events.GetInvocationList())
      ▼
Each subscriber runs to completion synchronously.
      │
      │ Home.razor's subscriber:
      │   1. mutates Events list  (sync)
      │   2. _ = InvokeAsync(StateHasChanged)  (fire-and-forget — schedules
      │                                         a Blazor re-render on the
      │                                         Renderer.Dispatcher, returns
      │                                         immediately without awaiting)
      ▼
DispatchEventNative returns control to wasmtime
      ▼
wasmtime returns control to the Android UI thread

(The actual Blazor re-render happens later, when the cooperative scheduler
 ticks during the next host re-entry — e.g. when Android calls another
 export, or via the renderer's own write path.)
```

### Key invariants

1. **`DispatchEventCore` returns synchronously.** Multicast finishes before control returns to caller.
2. **`InvokeAsync(StateHasChanged)` is fire-and-forget on `Renderer.Dispatcher`.** Enqueues a re-render onto the dispatcher's queue and returns an incomplete Task; the handler discards via `_ = ...`.
3. **The trap mechanism is structurally absent.** No `Task` / `ValueTask` / `Wait` anywhere from `DispatchEventNative` → `DispatchEventCore` → `Invoke` → subscriber. `Task.InternalWaitCore` cannot be reached.
4. **BN0014 closes the async-lambda hole.** Compile-time enforcement that subscribers cannot reintroduce a Task-returning shape.

## Error handling

| # | Failure mode | Mitigation in this design |
|---|---|---|
| 1 | Subscriber throws an exception | `DispatchEventCore` wraps each subscriber invocation in `try/catch`, logs to `Console.Error` with subscriber's exception type+message, continues to the next subscriber. .NET's default multicast-delegate semantics (first exception terminates the chain) are explicitly overridden. |
| 2 | Subscriber writes an `async` lambda | **BN0014 fires at compile time** — build fails before the binary ships. Belt-and-braces: `Action<NativeEvent>` shape compiles async lambdas as `async void`, which doesn't suspend in the chain but does swallow exceptions; BN0014 is the real defense. |
| 3 | `WasiBridge.Current` is null when `DispatchEventCore` is called | `if (Current is not { } c || c._events is null) return;` — early return, no observable effect. |
| 4 | Self-test subscriber never fires (`[BOOT] event-ok fired=False`) | BootSmoke asserts `fired=True` literally — test fails with a clear message; CI/dev catches before ship. |

### Deliberately not handled (Phase 2.0 scope boundary)

- **Re-entrant DispatchEventCore.** If a subscriber's body synchronously triggers another bridge event, the multicast iterates a stable `GetInvocationList()` snapshot — re-entry doesn't iterate-while-modifying. Future events queue at the dispatcher level. Not a concern.
- **Backpressure.** No queueing means no overflow. If the native shell pushes events faster than subscribers consume, subscribers run to completion before the next event lands — natural backpressure via the call stack. Phase 2.0 is fine; M7 hardening may add a debug-mode burst counter.
- **Cross-thread safety.** WASI is single-threaded.

## Out of scope (deferred)

- **Queue-and-drain pump infrastructure** — option (a) from the brainstorm. If async handlers ever become a real need post-1.0, promote to (a). Tracked: not a backlog item; this design doc itself is the record that it was considered and rejected for YAGNI reasons.
- **BN0001-BN0013 unit tests** — BACKLOG P3 "Analyzer unit tests". Phase 2.0 sets up the analyzer-test infrastructure; BN0014 has tests; the rest stays a P3 task.
- **`BlazorNative.Analyzers` NuGet packaging** — BACKLOG P3 / P5 work. Phase 2.0 ships the analyzer as a ProjectReference (per Phase 1.5's wiring decision).
- **Renderer-in-shell architectural pivot** — option (c) from the brainstorm. Explicitly out of M2 scope.

## Risks

1. **`internal` visibility of `DispatchEventCore` from a different assembly.** `BridgeEventTests.cs` lives in `BlazorNative.Renderer.Tests`, which doesn't have access to `BlazorNative.Core`'s `internal` members by default. Mitigation: add `[assembly: InternalsVisibleTo("BlazorNative.Renderer.Tests")]` to `BlazorNative.Core` (or use `AssemblyInfo.cs`). Alternative: make `DispatchEventCore` `public` (smaller blast radius than expected — it's already on a `public sealed class WasiBridge`).
2. **`Microsoft.CodeAnalysis.Testing` package selection.** Multiple flavors exist (`Microsoft.CodeAnalysis.CSharp.Analyzer.Testing.{XUnit,MSTest,NUnit}`). We'll use the xUnit variant to match the existing test projects.
3. **BN0014 false-positive on intentional `async` handler.** If a developer genuinely wants to register an async handler against a non-bridge event with the same `Action<NativeEvent>` shape, BN0014 might fire. Mitigation: BN0014's predicate is strict — only fires when the LHS event symbol is `IMobileBridge.NativeEvents` (matched by full type+name), not any `Action<NativeEvent>`.
