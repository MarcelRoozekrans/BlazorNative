# Phase 11.0 — Deep-link route codegen (M11 DoD #1)

**Milestone:** 11 — Production Readiness. **Phase:** 11.0 (first). **Closes:** DoD #1 — the
hand-written `DEEP_LINK_COMPONENTS` mirror is replaced by a build-time-generated map, so an app
author adding a routed page never hand-edits it. Plus a consumer-footgun audit.

## The problem (the seed finding)
`MainActivity.kt`'s `DEEP_LINK_COMPONENTS` is a hand-written `mapOf(route → componentName)`
read at Intent-parse time (a cold deep-link launch resolves route→component *before* the .NET
runtime loads, to seed the mount). Everything else derives from the app's page registration
(`SampleAppPages.All` → `PageManifest` → the mount registry + route table); this map does not.
`RouteTableDriftTests` (required lane) catches a mismatch *in the repo* — but that guard is
**ours, not the consumer's**: a `dotnet new` app author who adds a routed page and forgets to
hand-edit their `DEEP_LINK_COMPONENTS` gets no drift test, and their deep link silently opens
the wrong screen. Production-grade = no footguns; this map must be **generated**, not written.

## The constraint that shapes the fix
The map must be readable by **Kotlin, before .NET loads** (Intent-parse). So the generated
artifact is an **Android resource** (or generated Kotlin), produced at **build time** (where
.NET *is* available). Only the *source* of the map changes — hand-written → generated. The
Intent-parse read stays.

## The mechanism (decided: build-time codegen over the app's own registry)
A build step that reads the app's **actual registered routed pages** (general — works for any
consumer, not a named `SampleAppPages`) and emits the map:

1. **A codegen tool** (`BlazorNative.RouteGen`, a small `dotnet` tool shipped with the
   framework) invoked by an MSBuild `<Target>` in the framework's build targets
   (`build/BionicNativeAot.targets`, which already imports into every consumer app):
   - loads the built **app assembly**, triggers its `[ModuleInitializer]` (which calls
     `BlazorNativeApp.RegisterPages`), then reads the framework's own `PageManifest` routed rows
     — `(route, componentName)` for every routed page. This is why it's general: it consumes the
     framework registry, not a convention-named array.
   - writes `src/androidMain/res/raw/blazornative_routes.json` (a flat `{ "/route": "Component" }`
     + the default row) into the Android shell's resources, before Android packaging (hook:
     `AfterTargets="Build"`, before the AGP resource-merge / `IlcCompile`).
2. **`MainActivity.kt`** reads `R.raw.blazornative_routes` at Intent-parse (via the runtime
   `getIdentifier` pattern Phase-parity already established, so it stays package-agnostic and the
   template mirror stays byte-identical), parses the JSON into the route→component map + the
   default fallback — replacing the hand-written `mapOf` **and** the template's empty-map
   divergence. No .NET needed to read it.
3. **The drift test** (`RouteTableDriftTests`): flip from "the hand-written Kotlin map mirrors
   `SampleAppPages`" to "the **generated resource** matches `SampleAppPages`'s routed rows" —
   still the required-lane guard, now guarding the generator's output. (If generation makes the
   pair-pin vacuous, retire it with a written rationale; keep the default-fallback pin.)

### Considered + rejected
- **Roslyn source generator** parsing `Routed<T>(route, name)` literals → generators emit C#,
  not Android resources; routing through a C# const + an MSBuild file-write is more moving parts
  and breaks if a route/name is ever non-literal. The registry-read is truer to "the real pages."
- **Keep the map, ship the drift test to consumers** → doesn't remove the hand-edit; still a
  footgun, just a louder one.

## Gates
- **Gate A** — the codegen tool + the MSBuild target + resource emission, wired in the **repo**
  app; `MainActivity` reads the generated resource; `RouteTableDriftTests` flipped to guard the
  generated output. Prove: the generated `blazornative_routes.json` equals today's hand-written
  map (byte-for-byte semantic parity — a red-first that the generator reproduces the exact 12
  pairs + default). Repo build + JVM/instrumented lanes green (the deep-link tests still pass).
- **Gate B** — the **template**: the generated app gets the codegen target + reads the resource;
  a `dotnet new` app with an added routed page gets its deep-link mapping **for free** (proven in
  the template-smoke lane — a generated app compiles and its routes.json contains the added
  page). Template mirror discipline: sync + `TemplateDriftTests`.
- **Gate C** — **the consumer-footgun audit** (DoD #1's second half): enumerate every place a
  consumer must hand-edit a shell file when adding a page/capability (deep-link map — now fixed;
  the AndroidManifest permissions/queries for a capability; the Info.plist keys; any `when`/map
  in the shells). For each: derive it, or document it as a known required step with a rationale.
- **Gate D** — conclusion + planning + PR.

## Risks / honest unknowns
- **Loading the app assembly at build time** to run its registration: the app is a managed
  assembly pre-AOT, so a normal .NET host can load it — but it pulls Blazor + the framework as
  build-time deps. If that proves fragile (e.g. trimming/AOT interactions), the fallback is the
  source-generator route (parse the literal `Routed<T>` calls). Gate A proves the load works
  before we commit.
- **Ordering:** the resource must be generated BEFORE AGP merges Android resources and before the
  gradle build reads them — the target's `Before/AfterTargets` must be pinned and proven in CI.
- **The generated resource is a build artifact** — gitignore it (like other generated output);
  the drift test regenerates + compares, so there's no checked-in copy to drift.

## Out of scope for 11.0
Dogfooding (11.1), real-device (11.2), API stability (11.3). This phase is the routing
single-source-of-truth fix + the footgun audit only.
