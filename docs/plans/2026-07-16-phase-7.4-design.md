# Phase 7.4 Design — `BnModal` + the React Native Parity Survey

**Date:** 2026-07-16
**Milestone:** M7 — Components + Razor (closes DoD #5 and DoD #7)
**Predecessor:** Phase 7.3 — form controls + a real `picker` (PR #97)
**Owns:** `BnModal` — the first overlay surface (show/hide + `ChildContent`) — and the RN
core-component parity survey with its cheap wins. The milestone's out-of-scope list binds:
no animation system, no gestures beyond tap/change/focus/scroll.

---

## Decisions (recorded at phase-open; #1 and #3 flagged for veto)

1. **`BnModal` is an overlay in the EXISTING root — NodeType `modal = 11` — not a native
   dialog window.** ⚠ *The call to veto.* The two options, weighed against the actual
   codebase:

   - **(a) Native dialog primitives** (Android `Dialog`/`DialogFragment`, iOS presented
     view controller) is **rejected on feasibility**, concretely: both shells apply frames
     in ONE walk from the synthetic `hostRoot` against `root.bounds`
     (`YogaLayout.calculateAndApply` / `BnWidgetMapper.calculateAndApply`) — and a
     `Dialog`'s content view lives in **its own Window**, a presented view controller's
     view **outside `HostViewController.view`**. Blazor-rendered, Yoga-laid-out content
     inside either means a **second Yoga root, a second calculate pass, a second resize
     hook, a second frame-application surface** — RN's own `Modal` pays exactly this (a
     `ReactRootView` inside the dialog). Worse than the cost is the **state-owner
     violation**: a native dialog OWNS its dismissal (back button, outside-touch,
     accessibility escape all call `dismiss()` on the WINDOW) — the shell would self-close
     while .NET still holds `Visible = true`, the precise desync the 7.3 picker precedent
     exists to forbid. And the thesis check: "renders natively" is about **widgets**; a
     modal is a **z-order and placement concept**, and placement is Yoga's — on both
     shells, since 6.1, without exception. The button INSIDE the modal is native; the
     window chrome a `Dialog` adds is not something any assertion reads.
   - **(b) The overlay**: the modal's content is ordinary nodes in the existing tree,
     hosted by a full-root overlay stacked last. One tree, one layout pass, one
     frame-application path; every event inside the modal rides the existing wire because
     its views are ordinary entries in the shells' `nodes` maps. This is the pick.

   **The model — the 6.2 synthetic-node machinery, pointed at the root.** A `modal` node
   materializes as TWO shell-side pieces, in both trees, in the same breath:

   - **The anchor** — a 0-sized view at the modal's WIRE position (its parent's child list,
     at the patch's insertIndex), with shell-fixed Yoga styles `position: absolute`,
     `width: 0`, `height: 0` — out of the flex flow entirely, contributing nothing to any
     sibling's frame. It exists for exactly one reason, and the rule is **normative — the
     THIRD index-mapping rule**, after the text collapse and the scroll content node:
     *the anchor occupies the modal node's slot in its wire parent, in the view tree AND
     the Yoga tree, so sibling insert indices never skew; the modal's wire child at index
     i is the OVERLAY's child at index i.*
   - **The overlay** — a container attached as the **LAST child of the host root** (view
     tree: `widget_root` / `HostViewController.view`; Yoga tree: `hostRoot`), with
     shell-fixed styles `position: absolute; top: 0; left: 0; width: 100%; height: 100%`.
     The modal's wire children parent into it (the `containerFor`/`scrollContents`
     redirection pattern, keyed by the modal's id in a `modalOverlays` map). Stacking
     order is creation order — **the shell attaches the overlay last at creation and
     never re-orders it**; a re-shown modal is a re-created one and lands on top.

   **One purge difference from `scroll`, stated so it cannot be an assumption:** the
   scroll content node is a Yoga CHILD of its scroll node, so the subtree purge finds it
   for free. The overlay is NOT a descendant of the anchor — `RemoveNode(modalId)` must
   purge **both** subtrees, and the `modalOverlays` entry is what names the second one
   (evicted in the same breath, the `contentNodes` discipline). The lifecycle test
   (overlay count back to 0 after mount → remove — the `contentNodeCount` pattern) is
   what keeps that true; on iOS a missed purge is a dangling `YGNodeRef`, not a leak.

   **What the three mirrors get:** `modal = 11` and (decision 5) `activityindicator = 12`
   on the existing int32 field of the 48-byte patch — a **wire-vocabulary extension, not
   an ABI change**: exports stay 9, the bridge stays 72 bytes. `FrameEncoder.MapNodeType`
   throws on unknown; both shells' `nodeTypes` arrays gain the entries; all three pins
   extend (including the Kotlin content-and-length pin 7.3 added). **The measure story:**
   `modal` is NOT in `MEASURED_NODE_TYPES` — anchor and overlay are containers with
   shell-fixed styles, and a measure func on a container is the 6.1 law's named violation.
   **`SetStyle` on a `modal` node is diagnosed-and-ignored on both shells** (the scroll
   container-style rule's shape): every style would land on the anchor or the overlay,
   neither of which the author owns. `BnModal`'s surface cannot emit one, but the
   hand-rolled-wire hatch is open, so a test pins the rule live (the 6.2 method).

2. **Hide unmounts. .NET owns visibility; dismissal is a REQUEST on the wire, never a
   shell self-close.** `BnModal.razor` is internally `@if (Visible) { <modal …> … }` —
   `Visible = false` means **zero wire presence**: the whole overlay subtree leaves as
   one `RemoveNodePatch`, which is precisely the disposal shape 7.2's removes-first fix
   hardened. The alternative (keep-mounted, hidden) needs a `display`/`hidden` style
   concept that does not exist and would keep native state alive that the author cannot
   see. Consequence, stated once: **state inside `ChildContent` does not survive hide
   unless bound** — the 7.2 eviction rule verbatim (an app that wants it binds it,
   exactly as on the web). Surface: `Visible`/`VisibleChanged` (`bool`, the `@bind-`
   pair), `ChildContent`, `ScrimColor` (default `#80000000`), and the content box's
   declared-size parameters (`ContentWidth`/`ContentHeight`/`Padding`/`BackgroundColor` —
   the `BnScroll` forwarding precedent). Dismissal (scrim tap, Android back, the app's
   own button) flows to .NET as an event; `BnModal` invokes `VisibleChanged(false)`; a
   parent that ignores it keeps the modal open — correct, no self-closing state anywhere
   in the shell (the 7.3 state-owner lesson, applied to the component that made it
   urgent).

3. **Android's back button dismiss-requests the topmost modal — the shell consults the
   modal stack BEFORE navigation-back.** ⚠ *The call to veto: this changes shell back
   semantics.* The rule, stated once: *on back-invoked, if the mapper has ≥ 1 live
   overlay, the shell dispatches on the topmost modal's `click` wire (the modal element's
   own attach — `BnModal` always subscribes it, so it is always live) and consumes the
   back event; navigation-back runs only when no overlay is live.* The dispatch is a
   request — .NET flips `Visible`, the re-render removes the overlay. A back that races
   an in-flight dismissal is absorbed by construction: `Visible` is already false and the
   second `VisibleChanged(false)` moves nothing. iOS ships nothing here — no hardware
   back, and sheet/swipe gestures are out of scope. Recorded as the second
   "same wire, platform-appropriate trigger" divergence (the checkbox precedent's shape).

4. **`click` attaches to plain `view` nodes on both shells — and the scrim-tap rule is
   normative.** Android's `setOnClickListener` is already generic; iOS's `click` arm is
   **UIButton-only today** (`BnWidgetMapper` guards `control is UIButton`) — Gate 3 grows
   a `UITapGestureRecognizer` arm for non-control views, with the delegate filter
   `touch.view === the attached view` so a recognizer never fires for a descendant's
   touch. The rule, stated once: **a tap dismiss-requests ONLY when the touch lands on
   the scrim view itself, never on a descendant.** iOS enforces it with the touch-view
   filter; Android — where a non-clickable child's tap falls THROUGH to the scrim — by
   the content box swallowing its own taps (`BnModal` attaches a no-op `click` to it, the
   RN inner-Touchable pattern; that dispatch is real and the device counters account for
   it). Same observable wire behavior, different platform mechanism — verified per shell,
   never assumed (the 7.3 guard lesson). This extension is also the survey's down payment
   on `Pressable` (decision 5's table).

5. **The survey's cheap wins: `BnActivityIndicator` ships; `SafeAreaView` is ledgered
   with its problem named.** `ActivityIndicator` is genuinely cheap: one NodeType
   (`activityindicator = 12`), a measured leaf (`ProgressBar` indeterminate /
   `UIActivityIndicatorView.medium`, animating while mounted — presence is `@if`, the
   decision-2 posture; no props, no new wire surface), intrinsic size platform-owned and
   asserted by oracle (the 6.3 method). `SafeAreaView` is **not** cheap, and the honest
   assessment is in the shells: insets are HOST data that arrive asynchronously and
   change at runtime — a container NodeType whose Yoga padding the SHELL owns
   (WindowInsets / `safeAreaInsets`), re-solved on every inset change. On iOS that is
   real work against real numbers; on Android **the current shell is not edge-to-edge, so
   the framework consumes the insets and the component provably pads 0** — shipping it
   means either a component that does nothing on one platform or an edge-to-edge shell
   posture change that re-litigates every device frame assertion (`BnScrollDemo`'s iOS
   twin already sets `contentInsetAdjustmentBehavior = .never` — "the safe area is the
   app's business" — for exactly this class of reason). **Ledgered with its trigger:**
   ships with the edge-to-edge/`StatusBar` work (M9-adjacent), as a container whose
   padding the shell owns. Nothing else on the table is one-NodeType cheap — the phase
   stays lean; `BnModal` is the main course.

## The React Native parity survey (DoD #7)

RN's core set mapped to `src/BlazorNative.Components/`. `*` = moved out of RN core to a
community package (`@react-native-community/slider`, `@react-native-picker/picker`).

|---|---|---|

| ScrollView | **have it** | `BnScroll` (vertical; horizontal ledgered since 6.2) |

| Switch | **have it** | `BnSwitch` (7.3) |

| ActivityIndicator | **ships in 7.4** | `BnActivityIndicator` (decision 5) |

| StatusBar | **ledgered** | host chrome control; the same edge-to-edge posture question; milestone names it explicitly |
| RefreshControl | **ledgered** | pull gesture + a `refreshing` round-trip on the scroll wire; gestures out of scope — milestone names it |
| KeyboardAvoidingView | **ledgered** | keyboard insets are the safe-area problem's hard case, plus animation; milestone names it |
| Alert | **ledgered** | an imperative host API, not a render-tree node — the shell-bridge (clipboard precedent) is its natural home; deferred for a customer |

## The proof surface: `/modal` (`BnModalDemo.razor` — the EIGHTH root page)

Both route mirrors extend (`NativeNavigationManager` + `MainActivity.DEEP_LINK_COMPONENTS`
— unification stays 7.6 work). The page: a trigger `BnButton` (`@bind-Visible` flips
true), a `BnModal` whose `ChildContent` is a `BnText`, a **`BnSwitch`** (the form control
that proves interactivity INSIDE the overlay rides the existing wire), and a dismiss
`BnButton`; plus a `BnActivityIndicator` on the page AND one inside the modal (the cheap
win, proven in both hosting contexts). The modal sits BETWEEN two declared-size siblings
on the page, so the anchor's zero-footprint rule is a frame assertion, not a comment.

**Frame assertions (both shells):** the content box's DECLARED 280 × 180 exactly; the
scrim = the ROOT's own bounds, read at assert time (`widget_root` / host view — full-root
numbers are a DEVICE assertion; detached-root synthetic tests pin structure and creation
shape, not percent-of-auto); centered content at x = (W − 280)/2, y = (H − 180)/2 against
those bounds — the 6.3 oracle discipline: declared sizes cross-platform, host-derived
placement per-platform against the shell's own root. The overlay is the root's LAST
subview; the anchor's siblings hold the exact frames they hold with `Visible = false`.
The activity indicator's intrinsic size by oracle (a fresh `ProgressBar` /
`UIActivityIndicatorView` measured the same way).

**On-device round trips:** show → toggle the switch inside the modal (the change
round-trips, the echo re-renders) → dismiss-button hide → overlay count 0, node purged;
show → scrim-tap → the dismissal event on the wire → hide; a content-box tap does NOT
dismiss (decision 4's rule, asserted on the dispatch counter and on `Visible`); Android
only: back-invoked with the modal open dismiss-requests and does NOT navigate; back with
it closed navigates as before. Every dismissal is a counted wire dispatch — the shell
never closes anything itself.

## Gates

| Gate | GREEN bar |

| **2 — Android** | Anchor + overlay wired (fixed styles, overlay LAST at root, `containerFor` redirection, the named two-subtree purge + overlay-count lifecycle pin); the Kotlin nodeTypes entries + content/length pin (**+2 on JVM 90**); back-before-navigation routing; the style-ignore rule live; `/modal` on the AVD: the frame table, all round trips, the counter assertions. **≥ +10 on 147.** |
| **3 — iOS** | The mirror (overlay under `HostViewController.view`, same fixed styles, same purge shape); Swift nodeTypes + pin; **the `click`-on-view tap-recognizer arm with the touch-view filter, verified not assumed** (a control inside the modal must still receive its own touches); same declared-size frames, same round trips minus back. **≥ +10 on 111.** |
| **4 — Close** | Conclusion (the overlay-vs-dialog record, the survey verdict, the ledger); DoD #5 + #7 checked in MILESTONE.md; counts; PR. |

Mutation expectations (run, not claimed — the 7.2/7.3 discipline, iOS mutations ON CI):
drop the two Kotlin nodeTypes entries → the JVM content pin reds before any emulator
boots; drop the anchor's absolute/zero styles → the sibling frame pins red; drop the
overlay purge → the lifecycle count red; drop iOS's touch-view filter → the
content-tap-does-not-dismiss test red; drop the back routing → the back-dismiss test red
while navigation-back stays green.

## Risks

| Risk | Mitigation |
|---|---|
| The anchor rule broken on one shell → silent sibling index skew (iOS clamps, remember) | The rule is normative and the modal sits between declared-size siblings in the demo — the skew is a frame-table red, mirrored on both shells |
| The overlay purge misses one of the two subtrees | The purge is named in the design (NOT the scroll shape); the overlay-count lifecycle test is the pin — on iOS the failure is a dangling `YGNodeRef`, which is why the test exists on both |
| The iOS tap recognizer eats touches meant for controls inside the modal | The `shouldReceive` filter scopes it to the attached view's OWN touches; the in-modal switch round-trip is the device proof |
| Scrim-tap semantics drift between shells (fall-through vs filter) | One normative rule, two named mechanisms, the same three device assertions on both (scrim-tap dismisses, content-tap does not, the counter agrees) |
| Back routing regresses navigation-back | The consult-then-fall-through rule is one branch; both behaviors asserted in the same device test file |
| A second modal, or show-during-dismiss, scrambles stacking | Creation-order stacking is the stated rule; re-show = re-create = on top; a race resolves through .NET state (`Visible`), which is the only writer |
| Host resize/rotation while the modal is open | Free by construction — the overlay is `100%/100%` in the ONE tree the existing resize hooks already re-solve; no new hook, no second surface |
