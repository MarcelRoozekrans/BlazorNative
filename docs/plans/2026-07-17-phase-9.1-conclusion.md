## THE HEADLINE — the 9.0 ABI bet paid off on its first draw: local notifications cost the ABI NOTHING

**Phase 9.0 grew the bridge to 80 bytes / 10 exports ONCE and argued it would be paid once** —
9.1/9.2/9.3 add an op constant and touch the ABI at nothing else. Local notifications are the
first reuse, and **they hold the bet exactly**:

- **Bridge stayed 80 bytes.** No new slot. Notifications ride the existing `HostCallBegin` at
  offset 72 (the last slot; nothing appended).
- **Exports stayed 10.** No new symbol. The calls reuse `host_call_complete`; warm tap-through
  reuses `host_event`. The win-x64/bionic `dumpbin`/`readelf` gates and the iOS archive `nm` gate
  all show the SAME ten `blazornative_*` symbols.
- **No drift-pin moved, no gate arithmetic.** The three-mirror struct pin (80 on .NET / Kotlin /
  Swift), the export gate (10), and the offset pins are byte-for-byte what 9.0 left. Notifications
  added only **wire vocabulary**: `HostCallOp.Notifications = 1`, the `NotificationStatus` enum,
  and the `"navigate"` host-event name.

**The reuse proof is made falsifiable, not asserted.** `NotificationsAbiUnchangedTests` pins
`Marshal.SizeOf<BlazorNativeBridgeCallbacks>() == 80`, `OffsetOf(HostCallBegin) == 72`, and
`Notifications == 1`, *inside the notifications suite* so the claim lives next to the code it
guards. The named mutations prove it bites: assert-81-bytes reds the .NET pin (`Actual: 80`), and
the iOS **struct-grow mutant failed to COMPILE** (`missing argument 'mutSlot11'`, run
**29612929731**) — the Swift type system itself forbids a silent grow. **"Pay once, reuse thrice"
is now demonstrated by construction, not promised.**

## Tap-through — an inbound event, solved without a new mechanism

Tap-through was the design's hardest question. A tapped notification is an **unsolicited INBOUND
event**: the OS wakes the app on a tap with **no in-flight .NET call to complete**, so it
correctly does NOT use `host_call_complete` (there is nothing to complete). It does not force an
ABI change either — the owner chose **both cases**, each on machinery that already existed:

- **Cold (app killed):** reuses the **Phase 5.1 launch deep-link**. The notification carries a
  `blazornative://<route>` Intent (Android) / a launch route in `userInfo` (iOS); on a cold tap
  the shell resolves it through the `DEEP_LINK_COMPONENTS` mirror **before the .so loads** and
  seeds the initial mount by name, verbatim.
- **Warm (app alive):** wires `onNewIntent` (Android, `singleTop`) / `didReceive` (iOS) — both
  **unhandled since 5.1** — to CALL the pre-existing `blazornative_host_event` export with the
  reserved name `"navigate"` and the route payload → `DispatchHostEventCore` maps it to
  `NavigateToAsync`, re-routing the LIVE session (rc 0). This is the exact shape `"back"` took in
  5.1, extended to a route. **The iOS shell CALLS `host_event` for the FIRST TIME here** — its
  precedent is 9.0's `host_call_complete`, the other export the iOS shell learned to call this
  milestone. No ABI, no new export.

## The three ops + the permission model

**Ops.** `schedule` (inexact `AlarmManager` firing a manifest-declared receiver on Android — no
`SCHEDULE_EXACT_ALARM`; a one-shot `UNTimeIntervalNotificationTrigger` on iOS) / `show`
(`NotificationManager.notify` on the once-created `blazornative_default` channel, Android 8+;
`UNMutableNotificationContent` with a nil trigger on iOS) / `cancel` (drops the posted
notification and any pending alarm/request by the app-chosen int id).

**Permission.** Android points the 9.0 machinery verbatim at `POST_NOTIFICATIONS`, with the
**implicit-grant fast path below API 33** (the permission did not exist pre-Tiramisu). iOS calls
`requestAuthorization`, folding `.provisional` → `Granted`. **Denial is DATA on both shells** —
every non-`Granted` outcome returns a `NotificationStatus` integer via `host_call_complete`, never
a thrown error across the C boundary, never a dropped completion, never a hang, bounded (the 9.0
law, mutation-proven here on the AVD and the simulator).

## Two doc-vs-reality findings from Gate 3, recorded as honesty rows

1. **The unauthorized-simulator `getPendingNotificationRequests` seam.** An unauthorized simulator
   is all CI can offer (the notification-permission alert is the un-grantable owner-device gesture,
   exactly as 9.0's location dialog was). On an unauthorized center, `UNUserNotificationCenter.add`
   does **not** register a pending request, so `getPendingNotificationRequests` is empty — the
   design's "assert schedule via getPendingNotificationRequests" cannot hold on CI. The production
   `add`/`remove` still run (the real code path); the assertion moved to a **construction seam**
   (`lastAddedRequestForTest` / `lastCancelledIdForTest`), which is authorization-independent and
   deterministic. **The posted-in-the-shade proof is physical-device UNPROVEN**, named in the split
   below.
2. **`UNAuthorizationStatus` has NO `.restricted` case.** `restricted = 3` is a **wire-parity
   constant only** — iOS's reachable statuses are `0/1/2/4` (`authorized`/`provisional` → Granted,
   `notDetermined` → Denied, `denied` → DeniedPermanently, throw → Error). The design's status
   table copied the `restricted` row from `CLLocationManager`, which is the type that actually has
   it. The Swift enum defines `restricted` for byte-parity with .NET/Kotlin, but no iOS code path
   can reach it.

## Verification honesty — the PROVEN / UNPROVEN split

**PROVEN on CI:** the AVD's **real** notification post + cancel, both tap-through paths (cold mount
+ warm re-route), and denial-as-data-no-hang; the simulator's schedule/cancel/denial on the
construction seam + tap-through. The 80-byte / 10-export ABI unmoved on all three RIDs.

**UNPROVEN until the owner's physical Android phone** — named, not smuggled:

- **The real notification in the shade** — CI asserts a real `notify` on an emulator; a phone's
  actual pulled-down notification is device territory.
- **A real tap from a genuinely killed process** — instrumentation relaunches; a true cold tap
  after the OS reaped the process is not drivable in CI.
- **The real `POST_NOTIFICATIONS` dialog gesture** — the shells bypass the un-grantable OS dialog
  behind an authorization override + suppress switch (production calls unsuppressed and untouched),
  exactly as 9.0's location dialog.
- **`DeniedPermanently` across restarts** — the Settings round-trip no mock reproduces.

**iOS real-device stays DEFERRED** — Apple Developer account is the trigger (a TestFlight phase
opens when the owner acquires it), as M5–M9 have shipped. The iOS proofs here are
simulator-scoped and labeled.
