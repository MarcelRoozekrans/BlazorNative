# Design — WASI Entry Point + DI Bootstrap (Phase 1.2)

*Brainstormed: 2026-05-23 — Milestone 1, Phase 1.2*

## Problem

`src/BlazorNative.Core/WasiEntryPoint.cs` currently contains a no-op `Main` that prints a placeholder line and exits. Phase 1.2 makes it a real WASI entry point: build the DI graph, resolve the renderer + bridge, do a cooperative `await Task.Delay(1)` round-trip, exit 0. This satisfies `MILESTONE.md` DoD #3 (WASI entry point exists) + #5 (cooperative async round-trip works).

Phase 1.1 finding that simplifies this phase: .NET 10's Mono-WASI runtime handles scheduler bootstrap transparently. The `dotnet new wasiconsole` template is two lines (`Console.WriteLine` + `using System;`). The pre-net10 BACKLOG wording about `WasiEventLoop.Run()` is no longer applicable — `async Task<int> Main()` just works.

## Decisions

| # | Decision | Chosen | Rejected alternatives |
|---|---|---|---|
| 1 | Scope of Phase 1.2 | **DI registration + scheduler round-trip** — build the service graph, resolve `IMobileBridge` + `NativeRenderer`, do not invoke any bridge method (avoids the unresolved `mobile_bridge` extern trap) | A: minimalist `await Task.Delay(1)` only (too thin); C: also mount a component (trips bridge externs that don't exist until M2) |
| 2 | DI container | **ZeroAlloc.Inject** — source-generated, AOT-safe, zero reflection | A: Microsoft.Extensions.DI (works but doesn't carry the source-gen + zero-reflection wins the rest of the ZA adoption already commits us to) |
| 3 | ZA.Inject mode | **MS DI Extension** — generated `IServiceCollection.AddXxxServices()` methods that work in both DevHost (ASP.NET, mandatory MS DI) and WASI Core; one set of `[Singleton]`/`[Transient]` attributes drives both | Standalone (two DI shapes — code duplication for DevHost + WASI); Hybrid (more complex, no win without MS DI extras we don't use) |
| 4 | Verification strategy | **xUnit subprocess test in new `tests/BlazorNative.Wasi.Tests/`** — publishes Core for `wasi-wasm`, runs result via `wasmtime`, asserts stdout markers + exit code | Shell script (no `dotnet test` integration); both (over-engineering for one spike) |
| 5 | Success signature | **Structured 4-line trace** — `[BOOT] scheduler-start` / `[BOOT] di-ok bridge=... renderer=...` / `[BOOT] delay-ok Δ=...ms` / `[BOOT] done`; failure markers `[BOOT] FAIL stage=<...> msg=<...>` with distinct exit codes per stage | Minimal `BOOT OK` (no diagnostic signal on failure); JSON output (overkill, drags serializer into boot path) |

## Architecture

```
                wasmtime <BlazorNative.Core.wasm>
                                  │
                                  ▼
              ┌───────────────────────────────────────┐
              │  WasiEntryPoint.cs::Main (async)      │
              │                                       │
              │  1. emit  [BOOT] scheduler-start      │
              │  2. build ServiceCollection           │
              │     ├─ AddBlazorNativeCoreServices()  │  ← ZA.Inject-generated
              │     ├─ AddBlazorNativeRendererServices()  │
              │     └─ AddBlazorNativeHttpServices()   │
              │  3. provider = services.BuildSP()     │
              │  4. resolve IMobileBridge + NativeRenderer (don't INVOKE) │
              │  5. emit  [BOOT] di-ok bridge=... renderer=...            │
              │  6. await Task.Delay(1)               │
              │  7. emit  [BOOT] delay-ok Δ=...       │
              │  8. emit  [BOOT] done                 │
              │  9. return 0                          │
              └───────────────────────────────────────┘

    Generated method per assembly:
       BlazorNative.Core      → services.AddBlazorNativeCoreServices()
       BlazorNative.Renderer  → services.AddBlazorNativeRendererServices()
       BlazorNative.Http      → services.AddBlazorNativeHttpServices()

    DevHost composition (unchanged behaviour):
       same generated methods called from Program.cs;
       ASP.NET Core's WebApplicationBuilder uses MS DI under the hood.
```

Three things to call out:
- **No `[UnmanagedCallersOnly]` work.** Phase 1.2 doesn't invoke any bridge methods. Phase 1.3 separately validates `WasiBridge.DispatchEvent` is visible via `wasm-tools dump`.
- **No component mount.** `NativeRenderer.MountAsync<HelloComponent>` would trigger `_bridge.WriteStorageAsync` → unresolved `shell_storage_write` extern → trap. Component mounting waits until M2 when the Android shell provides those imports.
- **`Main` is `async Task<int>`.** .NET 10 supports async `Main`; we need it for `await Task.Delay`.

## Components

### New: `src/BlazorNative.Core/WasiEntryPoint.cs` (rewrites the existing stub)

```csharp
using Microsoft.Extensions.DependencyInjection;
using BlazorNative.Core;
using BlazorNative.Renderer;
using BlazorNative.Http;

namespace BlazorNative.Core;

public static class Program
{
    public static async Task<int> Main()
    {
        try
        {
            var sw = System.Diagnostics.Stopwatch.StartNew();
            Console.WriteLine("[BOOT] scheduler-start");

            IServiceProvider provider;
            IMobileBridge bridge;
            NativeRenderer renderer;
            try
            {
                var services = new ServiceCollection();
                services.AddBlazorNativeCoreServices();
                services.AddBlazorNativeRendererServices();
                services.AddBlazorNativeHttpServices();
                provider = services.BuildServiceProvider();
                bridge = provider.GetRequiredService<IMobileBridge>();
                renderer = provider.GetRequiredService<NativeRenderer>();
            }
            catch (BlazorVersionMismatchException ex)
            {
                Console.WriteLine($"[BOOT] FAIL stage=blazor-drift msg={ex.Message}");
                return 2;
            }
            catch (Exception ex)
            {
                Console.WriteLine($"[BOOT] FAIL stage=di msg={ex.GetType().Name}: {ex.Message}");
                return 1;
            }

            Console.WriteLine($"[BOOT] di-ok bridge={bridge.GetType().Name} renderer={renderer.GetType().Name}");

            var t0 = sw.ElapsedMilliseconds;
            try { await Task.Delay(1); }
            catch (Exception ex)
            {
                Console.WriteLine($"[BOOT] FAIL stage=scheduler msg={ex.GetType().Name}: {ex.Message}");
                return 3;
            }
            var dt = sw.ElapsedMilliseconds - t0;
            Console.WriteLine($"[BOOT] delay-ok Δ={dt}ms");

            Console.WriteLine("[BOOT] done");
            return 0;
        }
        catch (Exception ex)
        {
            Console.WriteLine($"[BOOT] FAIL stage=unknown msg={ex.GetType().Name}: {ex.Message}");
            return 99;
        }
    }
}
```

### Annotations on existing services

Three one-line additions:
- `src/BlazorNative.Core/WasiBridge.cs` — add `[Singleton(typeof(IMobileBridge))]` above the class declaration. (WASI is single-threaded; lifetime is process-scoped.)
- `src/BlazorNative.Renderer/NativeRenderer.cs` — add `[Singleton]` above the class.
- `src/BlazorNative.Http/BridgeHttpHandler.cs` — add `[Transient]` (HttpMessageHandler isn't safe to share across `HttpClient`s).

`DevHostBridge` does **not** get an attribute — it's only used in the DevHost (where it's manually constructed in `Program.cs`). Annotating both `WasiBridge` and `DevHostBridge` as `[Singleton(typeof(IMobileBridge))]` would cause ZAI001 duplicate-registration errors.

### Generated methods

ZA.Inject's source generator emits one extension method per assembly, named from the assembly: `AddBlazorNativeCoreServices()`, `AddBlazorNativeRendererServices()`, `AddBlazorNativeHttpServices()`. If the auto-derived names don't match, override via `[assembly: ZeroAllocInject("AddBlazorNativeCoreServices")]`.

### Removed / re-exported

`ServiceCollectionExtensions.AddBlazorNativeRenderer()` in `RendererServices.cs:18-25` becomes a thin re-export (`=> services.AddBlazorNativeRendererServices();`) for backwards compatibility with DevHost's `Program.cs`. Same for `BlazorNative.Http.AddBlazorNativeHttp()`.

### Package additions

`BlazorNative.Core.csproj`:

## Data flow

Under wasmtime, a single straight line — no callbacks, no native shell interaction, no Blazor render cycle:

```
wasmtime BlazorNative.Core.wasm
      │
      ▼
Mono runtime loads BlazorNative.Core.dll
      │
      ▼
dispatches to Program.Main()
      │
      ▼
stdout: "[BOOT] scheduler-start"
      │
      ▼
new ServiceCollection()
+ AddBlazorNativeCoreServices()       ─┐
+ AddBlazorNativeRendererServices()    │ ZA.Inject-generated calls
+ AddBlazorNativeHttpServices()       ─┘ (pure new()/typeof())
.BuildServiceProvider()
      │
      ▼
provider.GetRequiredService<IMobileBridge>()
   → WasiBridge..ctor()
      - allocates AsyncEventHandler<NativeEvent>()
      - sets WasiBridge.Current = this
      ✗ no `mobile_bridge` extern called yet
      │
provider.GetRequiredService<NativeRenderer>()
   → NativeRenderer..ctor(bridge, provider)
      - allocates _frames, _tree
      - base Renderer..ctor() runs (creates Dispatcher)
      - triggers BlazorInterop.cctor → version + drift probe
      ✗ no Mount, no UpdateDisplayAsync
      │
      ▼
stdout: "[BOOT] di-ok bridge=WasiBridge renderer=NativeRenderer"
      │
      ▼
await Task.Delay(1)
   → Mono WASI scheduler yields, sleeps ~1ms, resumes Main
      │
      ▼
stdout: "[BOOT] delay-ok Δ=<measured ms>"
      ▼
stdout: "[BOOT] done"
      ▼
return 0 → wasmtime exits with code 0
```

**Key invariants:**
- **No `[DllImport("mobile_bridge")]` extern is called.** Lazy lookup means unresolved imports don't trap until first call. None of the boot path calls bridge methods — only constructs `WasiBridge`.
- **`BlazorInterop` static-ctor fires** when `NativeRenderer..ctor` first references the type. The version + drift probe runs *during* the `[BOOT] di-ok` step. If Blazor drift is detected, `BlazorVersionMismatchException` bubbles up out of `BuildServiceProvider` — caught and reported as exit code 2.
- **`Stopwatch.StartNew()` is the time source** (monotonic clock, works under WASI's `clock_time_get`).

## Error handling

Three failure categories, each with a defined exit code and a stdout marker so the test can disambiguate:

| Category | Trigger | Exit code | Stdout signal |
|---|---|---|---|
| **Boot / DI failure** | `BuildServiceProvider()` throws, or `GetRequiredService<T>()` returns/throws | `1` | `[BOOT] FAIL stage=di msg=...` |
| **Blazor version drift** | `BlazorInterop` static-ctor throws `BlazorVersionMismatchException` | `2` | `[BOOT] FAIL stage=blazor-drift msg=...` |
| **Scheduler failure** | `await Task.Delay(1)` throws or never resumes | `3` (throw) or timeout-kill (deadlock) | `[BOOT] FAIL stage=scheduler msg=...` (throw) — or empty (deadlock) |
| **Unknown** | Last-resort `catch` covering anything thrown from `Console.WriteLine` etc. | `99` | `[BOOT] FAIL stage=unknown msg=...` |

Deliberately not handled in Phase 1.2:
- **WASI module load failures** (corrupt .wasm, unresolved required imports). No `Main` code runs; test sees wasmtime stderr + non-zero exit.
- **OOM / stack overflow inside the runtime.** Mono's responsibility. Test detects via timeout + non-zero exit.
- **`Console.WriteLine` failure.** Theoretically possible if WASI's `fd_write` is denied. Out of scope — wasmtime grants stdout by default.

## Testing

**New project: `tests/BlazorNative.Wasi.Tests/BlazorNative.Wasi.Tests.csproj`** (net10.0, xUnit). Added to `BlazorNative.sln`. No project reference to `BlazorNative.Core` — the test shells out to a published artifact.

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <IsPackable>false</IsPackable>
    <IsTestProject>true</IsTestProject>
    <Nullable>enable</Nullable>
  </PropertyGroup>
  <ItemGroup>
    <PackageReference Include="Microsoft.NET.Test.Sdk" Version="17.*" />
    <PackageReference Include="xunit" Version="2.*" />
    <PackageReference Include="xunit.runner.visualstudio" Version="2.*" />
  </ItemGroup>
</Project>
```

**One xUnit `IClassFixture` for the expensive publish step**, two tests:

- `WasiPublishFixture` — runs once per test class. Invokes `dotnet publish src/BlazorNative.Core ... -r wasi-wasm --output artifacts/wasi-publish-test/`. Throws if publish fails (with hint about installing `wasi-sdk`). Exposes `WasmPath`.
- `WasmModule_BootsCleanly_UnderWasmtime` — runs `wasmtime <WasmPath>`, asserts all four `[BOOT]` markers present in order, no `FAIL`, exit 0. 10-second subprocess timeout.
- `WasmModule_ExitsWithinTwoSeconds` — same wasmtime invocation, asserts wall-clock < 2s. Regression catch for startup ballooning or scheduler deadlock.

Both tests `[Trait("Category", "Integration")]` so fast inner-loop runs can skip via `dotnet test --filter "Category!=Integration"`.

**Coverage matrix:**

| Test | What it proves |
|---|---|
| `WasmModule_BootsCleanly_UnderWasmtime` | All 4 boot markers fire in order, no FAIL marker, exit 0. End-to-end: scheduler + DI + delay + clean exit. |
| `WasmModule_ExitsWithinTwoSeconds` | Mono-AOT startup + Main + exit completes in under 2s. Catches startup regression and scheduler deadlock. |

**Deferred to later phases:**
- Asserting specific `[UnmanagedCallersOnly]` exports exist — Phase 1.3 `wasm-tools dump` check.
- Triggering a render cycle and asserting patches reach `WriteStorageAsync` — needs the native shell or a stubbed wasmtime imports module, M2 work.
- Performance baselines for boot time — once we have a number worth defending, M7 telemetry.

## Prerequisites

1. **`wasi-experimental` workload** — already installed in Phase 1.1.
2. **`wasi-sdk`** — Clang/LLVM toolchain. Download from https://github.com/WebAssembly/wasi-sdk/releases, install, and set `WASI_SDK_PATH` env var. Required for `dotnet publish` to AOT-compile the IL to the app-specific `.wasm`. Without it, `WasiPublishFixture` throws with a clear error message.
3. **`wasmtime` CLI** — on PATH. Already required by Phase 1.1 / Makefile.
