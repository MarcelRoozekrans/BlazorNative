# Phase 11.0 — the consumer-footgun audit (M11 DoD #1, second half)

**Date:** 2026-07-20
**Phase:** 11.0 (Gate C). **Milestone:** 11 — Production Readiness, DoD #1.
**Predecessors:** Gate A (`0bcf8e8` — deep-link map generated from source), Gate B (`c4f6745`
— RouteGen packed in the Runtime package; a `dotnet new` app derives its own map).
**Design:** [2026-07-20-phase-11.0-design.md](2026-07-20-phase-11.0-design.md) (Gate C).

## Why this audit exists

The deep-link route map was **one** place a consumer had to hand-edit a shell file when they
added a page — and it was a silent footgun (add a routed page, forget the map, the deep link
opens the wrong screen with no compile error and no test). Gates A/B closed that one. DoD #1's
second half asks the honest follow-up: **is that the only such place?** This audit enumerates
every other surface a real app author touches when they add a page or use a capability, and for
each one either confirms it is **DERIVED** (no consumer action) or records it as a **DOCUMENTED**
required step with the rationale for why it cannot be derived.

## The packaging fact that governs the whole audit

**The Android shell is copied *source*, not a linked library.** `dotnet new blazornative` emits a
single-module `com.android.application` Gradle project whose `src/androidMain/` holds a
byte-identical copy of the reference shell (Kotlin, manifest, resources). There is **no `.aar`
today**, so there is **no library manifest-merge** — the consumer's app manifest *is* the
template's `AndroidManifest.xml`. That is the key to the Android rows below: every permission,
`<queries>` block, `<receiver>`, and the camera `<provider>` is **already in the manifest the
consumer receives**, so "use a capability" requires **no manifest hand-add** on Android. (The day
the shell ships as an `.aar`, these move from "template-supplied" to "merged from the library" —
the classification stays DERIVED; only the mechanism changes. `build.gradle.kts` names that future
explicitly.)

**iOS has no template at all** — the shell is copied manually from `src/BlazorNative.Apple/BnHost/`
per [Shells → iOS](../../website/docs/shells/ios.md). So iOS's per-app surfaces are inherently
DOCUMENTED, not derived: there is no generator and no template rewrite to lean on.

## The audit

| # | Surface | What the consumer must edit | Verdict | Fix / rationale |
|---|---|---|---|---|
| 1 | **Deep-link route→component map** (Android, `res/raw/blazornative_routes.json`) | *Was:* hand-add a `mapOf` pair per routed page in `MainActivity.kt`. *Now:* nothing. | **DERIVED** (fixed, Gates A/B) | `BlazorNative.RouteGen` parses the app's C# source for `Routed<T>(route, name)` and emits the resource at build time; `MainActivity` reads it at Intent-parse. Adding a routed page is one row in `AppPages.All`. Drift-guarded by `RouteTableDriftTests.GeneratedRoutesJson_ReproducesTheManifestsRoutedRows_PairForPair`. |
| 2 | **A per-page `when`/`switch`/map keyed on component name or route** anywhere else in the shells | none | **DERIVED** (none exists) | Grep of `src/BlazorNative.Jni/src/androidMain/kotlin/**` for `when(`/`mapOf`/component-name literals: every `when` is generic (node-type, patch-type, style property, bridge op) — **not one is keyed per page**. The generated map (#1) was the only page-keyed surface. Confirmed by inspection, not just claimed. |
| 3 | **Add a routed page — the .NET side** (`AppPages.All`) | one `Routed<T>(route, name)` row | **DERIVED by design** — this IS the single source of truth | The mount registry, the route table, and now the Android deep-link map are all views over this one array. This is the *intended* single edit; every other row here confirms nothing shadows it. |
| 4 | **Android permissions** — `INTERNET`, `POST_NOTIFICATIONS`, `USE_BIOMETRIC` | none to add | **DERIVED** (template-supplied) | All three are declared in the template `AndroidManifest.xml` with inline comments. `USE_BIOMETRIC` is a *normal* permission (install-time, no runtime prompt); `POST_NOTIFICATIONS` is the Android-13+ runtime permission the shell requests itself. The consumer adds nothing to *use* the capability. The inverse concern (shipping an unused permission) is DOCUMENTED in-manifest ("Remove this only if your app never…") — see #10. |
| 5 | **Camera package visibility** — the `<queries>` for `ACTION_IMAGE_CAPTURE` | none | **DERIVED** (template-supplied) | The `<queries><intent><action .../></intent></queries>` block is in the template manifest. On API 30+ the shell needs it to *resolve* the camera app for its availability `check`; it ships pre-declared. No runtime `CAMERA` permission is needed at all (the system camera app owns the sensor). |
| 6 | **Camera FileProvider** — the `<provider>` + `res/xml/file_paths.xml` | none | **DERIVED** (template-supplied) | Both the `<provider>` and `file_paths.xml` are in the template. The authority is `${applicationId}.fileprovider`: **AGP substitutes the consumer's own `applicationId` at build**, and `AndroidShellBridge.fileProviderAuthority` reads `packageName` at runtime — the two agree automatically. This is the "authority embeds the appId" case the design flagged, and it derives cleanly because the substitution is a build variable, not a hand-typed string. |
| 7 | **The scheduled-notification `<receiver>`** (`AndroidShellBridge$NotificationPublisher`) | none | **DERIVED** (template-supplied) | Manifest-declared (not runtime-registered) so a scheduled alarm fires after the process is killed. Fully qualified name in the template; no consumer action. |
| 8 | **iOS usage-description strings** — `NSCameraUsageDescription`, `NSLocationWhenInUseUsageDescription`, `NSFaceIDUsageDescription` | rewrite the string copy to your app | **DOCUMENTED** (per-app copy, un-derivable) | The reference `BnHost/Info.plist` carries real sentences, but they are **BlazorNative-branded demo copy** ("BlazorNative uses your camera to take a photo in the camera demo"). The *keys* are copied for free when the consumer copies the shell; the *text* is app-specific and the App Store rejects boilerplate. iOS `SIGABRT`s at the capability call if a key is absent, so the key's presence matters — it is present; only the sentence is the consumer's. **Fixed the doc:** `shells/ios.md` now calls this out explicitly (it previously documented the ATS exemption and the bundle id but not the usage strings). |
| 9 | **iOS keychain / entitlements** | none (for the shipped capabilities) | **DERIVED / N-A** | Secure storage uses the raw `AndroidKeyStore` (Android — no manifest permission) and the app's default keychain access group (iOS — no explicit entitlement needed for the shipped `getWithAuth` path). No `keychain-access-groups` entitlement is required for the reference capability set; a consumer sharing a keychain across apps would add one, which is genuinely their integration and out of the shipped surface. |
| 10 | **Over-permissioning** (shipping a permission for a capability the app never uses) | optionally *remove* an unused `<uses-permission>` | **DOCUMENTED** (in-manifest + shells/android.md) | Because #4–#7 pre-declare everything, an app that never authenticates still ships `USE_BIOMETRIC`, etc. This is the *inverse* footgun and it is deliberately chosen (a working-by-default template beats a per-capability opt-in dance). Each manifest entry carries a "Remove this only if your app never…" comment; `shells/android.md` now has a **Capabilities and permissions** section summarizing what is pre-declared and what to trim before a store submission. |
| 11 | **Deep-link URI scheme** — `blazornative://` | change it for a production app | **DOCUMENTED** (per-app choice) | The scheme is hard-coded in two paired places — the manifest `<data android:scheme="blazornative"/>` and `MainActivity.DEEP_LINK_SCHEME`. Two BlazorNative apps on one device both claiming `blazornative://` collide (Android disambiguates). This is a genuine per-app decision (a custom scheme, not a verified `https` App Link), documented in `shells/android.md`. Not derivable — there is no "the app's scheme" input to derive it from, and a custom scheme needs no domain verification by design. |
| 12 | **App identity** — `namespace` / `applicationId` (`com.example.starterapp`) | set to your own | **DOCUMENTED** (per-app identity) | Already documented in `shells/android.md` ("The app identity"). Cannot be derived — it *is* the input. Note it is deliberately separate from the shell's `io.blazornative.shell` source package. |
| 13 | **iOS root-component name** — `HostViewController.swift` + `BnRuntime.swift` | edit two source lines to your registered page name | **DOCUMENTED** (no iOS template/generator) | The Android analog (`MainActivity`'s `?: "BnStarterPage"` fallback) **is** templated — rewritten at `dotnet new` time and pinned by `RouteTableDriftTests.AndroidDefaultFallbackLiteral…`. iOS has neither a template nor a generator, so "on iOS you are the drift test." Already documented in `shells/ios.md §3`. |
| 14 | **iOS csproj / link recipe** — the iOS `PropertyGroup`, `TrimmerRootAssembly`, `bootstrapperdll.o` link order | copy the iOS `PropertyGroup`; keep `TrimmerRootAssembly` | **DOCUMENTED** (asymmetry with Android) | Android ships `build/BionicNativeAot.targets` (auto-imported); iOS has no equivalent shim. Already documented in `shells/ios.md §5/§7`. Moving the recipe into the `BlazorNative.Runtime` package is backlog (it would auto-import for both platforms) — noted, not done here (out of 11.0 scope). |
| 15 | **`TrimmerRootAssembly` (.NET side, both platforms)** | do not delete it | **DERIVED / DOCUMENTED** (template ships it) | The template csproj carries `<TrimmerRootAssembly Include="$(AssemblyName)"/>` with a comment. Without it ILC trims the whole app: green build, zero warnings, failed mount. Not a hand-*add* (the template supplies it); documented as a "do not delete" in `getting-started/quick-start.md`. |

## What was fixed vs documented

**Fixed (Gates A/B, before this audit):** row 1 — the one derivable, silent footgun.

**Fixed in this Gate (docs, consumer-facing):**
- `getting-started/quick-start.md` — the stale "The one place that does not derive" caution (it
  still told consumers to hand-edit `DEEP_LINK_COMPONENTS`) is **replaced** with the truth: the map
  is generated; adding a routed page is one row in `AppPages.All`.
- `shells/android.md` — the stale "Deep links → the one place that does not derive" caution is
  **replaced**; the "launcher page" framing is corrected (the launcher component comes from the
  generated `/` row, with the `MainActivity` literal as the resource-less fallback the drift test
  pins); a new **Capabilities and permissions** section documents rows 4–7, 10, 11.
- `shells/ios.md` — a new note documents row 8 (the usage-description strings are app-specific copy
  to rewrite, not just keys to keep).

**Documented, not fixed (genuinely un-derivable, by row):** 8, 10, 11, 12, 13, 14 — app-specific
copy, app identity, a per-app scheme choice, and the iOS no-template surfaces. Each is a *known,
intentional* required step with the rationale above, and each is now stated where a consumer will
look (the shell docs and the in-manifest / in-plist comments).

## The bottom line

After Gate C, **there is no Android shell file a consumer must hand-edit to add a page** (the map
derives; nothing else is page-keyed), and **no manifest entry they must hand-add to use a
capability** (the template pre-declares all of them). The surfaces that remain are (a) *identity and
copy* that are inputs, not derivable outputs, and (b) the *iOS manual path*, which has no template to
derive from and is documented step-by-step. The seed finding — the last hand-written
single-source-of-truth violation — is closed, and the audit confirms it was genuinely the last one
of its kind.
