# Phase 6.1 Design — Flexbox Layout Core

**Date:** 2026-07-13
**Milestone:** M6 — Real-UI Foundation (closes DoD #2, #3, #6)
**Predecessor:** Phase 6.0 — Yoga-integration spike, **GREEN** on both shells
([conclusion](2026-07-13-phase-6.0-spike-conclusion.md), PR #54)
**Owns:** the engine. Yoga stops being a probe and becomes the thing that places every view.

---

## What changes, in one sentence

Today a `view` node is a vertical `LinearLayout` / `UIStackView` and children are appended in
order; after this phase **every node lives in a Yoga tree, containers are plain
`FrameLayout` / `UIView`, and every child is placed at a Yoga-computed frame** — with text
and other leaves measured natively so their intrinsic size drives the layout.

## Decisions (locked at brainstorm, 2026-07-13)

1. **Yoga owns all placement** (not opt-in). One model everywhere — the React Native design.
   Yoga's default `flexDirection` is `column`, so an *un-styled* tree lays out exactly like
   today's vertical stack: the demo pages keep looking the same while the engine underneath
   is replaced. The cost is honest and is paid in this phase: the tests that pin
   `LinearLayout` / `UIStackView` *by type* get rewritten to pin **frames** instead — which is
   the stronger assertion anyway.
2. **The flex demo is a third page** (`/layout` → `BnLayoutDemo`). `BnDemo` and
   `BnSettingsPage` keep their goldens on all four surfaces, so layout-engine bugs never
   arrive mixed with golden-rewrite noise. It exercises the nav path for free.
3. **`BnView` gains the flex parameter surface; `BnRow` / `BnColumn` are thin presets.**
   **No `BnStack`** — it would be a synonym for `BnColumn`, and two names for one thing is a
   library smell on day one. (The milestone contract named it; this design consciously drops
   it. Recorded in the DoD #6 note.)
4. **Typed C# params, strings on the wire.** Public params become enums/numerics
   (`FlexDirection.Row`, `float? FlexGrow`) that stringify at `BuildRenderTree` into the
   **existing** `SetStyle` wire. Compile-time safety for the app author; **zero ABI change**
   (still 9 exports + the 72-byte bridge); the shells still parse strings.

## No ABI change — the wire is already there

`NativeRenderer.StyleAttributes` already routes `flexDirection`, `alignItems`, `justifyContent`,
`width`, `height`, `margin` to `SetStylePatch` — the shells simply ignore them today
(`else -> Log.w("not yet supported")`). This phase **extends that allow-list**, **partitions it**
(see §"The allow-list is a routing table") and **teaches the shells to honour it**. Patch kind
`SetStyle = 6`, the 48-byte patch struct, the 9 exports and the 72-byte bridge are all untouched.

### The null-reset bug (found by the 6.1 codebase survey — fix it here)

Removing a style attribute on re-render emits `UpdatePropPatch(name, null)`, **not**
`SetStylePatch(name, null)` ([NativeRenderer.cs:589](../../src/BlazorNative.Renderer/NativeRenderer.cs#L589)).
Harmless-ish today (nobody unsets `Padding`); **not** harmless once `FlexGrow` can go
conditionally null — the prop would silently keep its old value forever. Fix: branch on
`StyleAttributes.Contains(removedName)` so a removed style emits `SetStyle(name, null)`, which
already means "reset to default" on the wire. Both shells must reset the Yoga property to
`undefined`/default on a null value. **This gets its own regression test on .NET + both shells.**

## The layout model

### One Yoga node per view node

The shell keeps `nodeId -> (view, yogaNode)`. `CreateNode` creates both and inserts the Yoga
child at the same index the view goes to; `RemoveNode` removes both; `SetStyle` sets a Yoga
style property; `CommitFrame` triggers **one** layout pass.

**The text-collapse invariant is preserved.** A `text` node whose parent is a text-bearing
non-container (`TextView`/`UILabel`/`Button`) gets **no view** today
([WidgetMapper.kt:267](../../src/BlazorNative.Jni/src/androidMain/kotlin/io/blazornative/shell/WidgetMapper.kt#L267)) —
it must also get **no Yoga node**, or child indices skew between the two trees. The Yoga tree
mirrors the *view* tree, not the *patch* tree.

### The layout pass

```
CommitFrame → applyBatch (on the main thread, as today)
            → YGNodeCalculateLayout(root, hostWidth, hostHeight, LTR)
            → walk the tree, apply each node's computed frame to its view
```

Plus a **re-layout on host resize** (Android `onLayout` / iOS `viewDidLayoutSubviews`) — same
calculate + apply, no patch involved. Rotation therefore works for free.

**Units.** Yoga computes in **density-independent units** on both platforms: iOS points map
1:1; Android multiplies computed frames by `density` at apply time (and divides incoming px
never — style values arrive as dp already). Consequence: **the same tree yields the same
numbers on both platforms**, which is exactly what the 6.0 parity assertion depends on and
what DoD #2's "lays out identically" means.

### Native measurement (DoD #3)

Leaves (`text`, `button`, `input`, `image`) get a Yoga **measure function** — the mechanism the
6.0 spike proved end-to-end in both channels:

| | Android | iOS |
|---|---|---|
| measure | `view.measure(spec(w, wMode), spec(h, hMode))` → `measuredWidth/Height` (÷ density) | `view.sizeThatFits(CGSize(w, h))` |
| mode map | `Exactly→EXACTLY`, `AtMost→AT_MOST`, `Undefined→UNSPECIFIED` | clamp per mode; `Undefined → .greatestFiniteMagnitude` |

So a long label wraps and reports its wrapped height, and that height drives the row it sits
in.

**The measure func attaches BY NODETYPE — `text` / `button` / `input` / `image` — and never by
"this node happens to have no children."** They are not the same set: `BnLayoutDemo`'s row is
three *childless* `view`s, and attaching a measure func to them would let a `UIView`'s intrinsic
size (0×0, or worse) speak over Yoga — destroying the `Grow=1` box's computed width and every
number the parity assertion rests on. A childless container is a container.

### iOS: Objective-C++ behind a plain-C surface (from the 6.0 lesson)

Non-negotiable, and the reason 6.0 cost six red CI runs: **Yoga's headers must never be
visible to Swift** (Xcode's Swift explicit-module dependency scanner reads the bridging header
with a path-less search that honours neither `HEADER_SEARCH_PATHS` nor `-Xcc -I`). So the iOS
Yoga layer is a new `BnYogaLayout.mm` (Objective-C++) exposing a plain-C node API declared in
`BnYogaLayout.h`:

```c
typedef void* bn_yoga_node;                      // opaque YGNodeRef
bn_yoga_node bn_yoga_node_new(void);
void  bn_yoga_node_insert_child(bn_yoga_node parent, bn_yoga_node child, int32_t index);
void  bn_yoga_node_remove_child(bn_yoga_node parent, bn_yoga_node child);
void  bn_yoga_node_free(bn_yoga_node node);
int32_t bn_yoga_node_set_style(bn_yoga_node node, const char* name, const char* value); // NULL value = reset
void  bn_yoga_node_set_measure(bn_yoga_node node, bn_yoga_measure_fn fn, void* context);
void  bn_yoga_calculate(bn_yoga_node root, float availableWidth, float availableHeight);
bn_yoga_frame bn_yoga_node_get_frame(bn_yoga_node node);   // {x, y, w, h}
```

Swift holds the opaque pointers and the `void* context` is the node's identity for the measure
callback (a C function pointer cannot capture — the same constraint the runtime's frame
callback lives under). **Style-name parsing lives in the `.mm`, in C**, so the *string → Yoga
setter* mapping is written once per platform and is directly comparable to Kotlin's.

Android has no such constraint: `com.facebook.yoga:yoga`'s `YogaNode` Java API is called
straight from Kotlin, and its measure function is a capturing lambda (it captures the `View`).

## The style surface (what a developer can write)

`BnView` params (all nullable — null means "don't emit", i.e. Yoga's default):

```csharp
FlexDirection? Direction;  FlexJustify? Justify;  FlexAlign? Align;  FlexAlign? AlignSelf;
float? Grow;  float? Shrink;  string? Basis;   // basis: "auto" | "50%" | "120"
FlexWrap? Wrap;   string? Gap;
string? Width;  string? Height;  string? MinWidth;  string? MaxWidth; …
FlexPosition? Position;  string? Top/Right/Bottom/Left;
BackgroundColor;  Padding;  Margin;   // Padding/BackgroundColor exist today
```

The public *type* names are `Flex`-prefixed on purpose. These ship on nuget.org in M8, and a
bare `Align` / `Wrap` / `Justify` / `Position` in a library's root namespace collides with
app-side types. The *parameter* names stay short (`Align=`, `Wrap=`) — that is the ergonomic
surface.

`BnRow` = `BnView Direction=Row`; `BnColumn` = `BnView Direction=Column`. Nothing else — they
forward every other param, **nulls included** (a `ParameterView` only writes *supplied*
parameters, so a guarded forward would leave `BnView` holding a stale value).

### The allow-list is a routing table, and it is PARTITIONED

`NativeRenderer` splits the SetStyle allow-list in two, because after this phase a shell that
receives `SetStyle(name, value)` must send the name to **exactly one** of two destinations —
and "which one?" must not be a judgement call taken twice, in two hand-written parsers:

| Set | Destination in the shell | Names |
|---|---|---|
| `YogaStyleAttributes` | the node's **Yoga node** | `flexDirection`, `justifyContent`, `alignItems`, `flexWrap`, `gap`, `alignSelf`, `flexGrow`, `flexShrink`, `flexBasis`, `width`, `height`, `minWidth`, `maxWidth`, `minHeight`, `maxHeight`, **`padding`**, **`margin`**, `position`, `top`, `right`, `bottom`, `left` |
| `VisualStyleAttributes` | the **View / UIView** | `backgroundColor`, `color`, `fontSize`, `fontWeight`, `background`, `style` |

`StyleAttributes` is their union; a test pins union-equality *and* disjointness
(`StyleAttributePartitionTests`).

**`padding` is LAYOUT.** Yoga places a container's children inside its padding box, so padding
belongs to the *Yoga node* — a shell that *also* calls `view.setPadding(...)` (as Android does
today) **double-applies** it. Same for `width`/`height`/`margin`. Gates 2 and 3 must delete
those view-level calls; the implementation plan says so in as many words.

Matching is **ordinal (case-sensitive)**, because both shells match style names
case-sensitively. An ignore-case list on .NET would classify `FlexGrow` as a style that the
shells then silently drop — .NET promising routing it cannot deliver. Ordinal means a mis-cased
name falls onto the *prop* wire, where the shells already log "unknown prop": visible, not silent.

Deliberately **not** on the list (ledgered): `alignContent`, `rowGap`, `columnGap` — no typed
param, no producer, and each accepted name is a name two hand-written parsers must implement for
nothing. (The wrap demo *relies* on Yoga's `alignContent` default of `flex-start`; **not setting
it** is precisely how it gets that.) Likewise `display` and `flex`, dropped from the pre-6.1
list: nothing types them and no shell ever implemented them.

### Style value grammar (normative)

**This section is the single source of truth for the value half of the wire.** Both shell
parsers (Kotlin's `YogaLayout`, iOS's `BnYogaLayout.mm`) are written *from here*; `FlexStyles.cs`
and the implementation plan *reference* it rather than restating it, because three copies of a
grammar is how two parsers drift.

| Kind | Applies to | Accepted values |
|---|---|---|
| **Layout length** | `width`, `height`, `minWidth`, `maxWidth`, `minHeight`, `maxHeight`, `flexBasis`, `margin`, `padding`, `gap`, `top`, `right`, `bottom`, `left` | `<number>` — **density-independent units** (dp on Android, points on iOS) · `<number>%` · `auto` (where Yoga allows it: `width`/`height`/`flexBasis`/`margin`) |
| **Unitless number** | `flexGrow`, `flexShrink` | a plain float (`"1"`, `"0.5"`). **Not a length** — no `%`, no `auto`, no units |
| **Enum word** | `flexDirection`, `justifyContent`, `alignItems`, `alignSelf`, `flexWrap`, `position` | exactly the strings `FlexStyleValues.ToStyleValue()` produces: `row`, `column`, `row-reverse`, `column-reverse`, `flex-start`, `center`, `flex-end`, `space-between`, `space-around`, `space-evenly`, `auto`, `stretch`, `baseline`, `nowrap`, `wrap`, `wrap-reverse`, `relative`, `absolute` |
| **null** | any | **reset to Yoga's default** (`YGUndefined` / the property's default). The wire already carries null (`PatchProtocol`); the removal path emits it (the null-reset fix) |

**Numbers are invariant-cultured.** `1.5`, never `1,5` — the shells parse with a C/Java float
parser. `.NET` guarantees this at the stringify site (`ToStyleValue(this float?)`).

**A number is exactly this production, and THE WHOLE STRING MUST BE CONSUMED:**

```
number := [+-]? ( digit+ ( '.' digit* )? | '.' digit+ ) ( [eE] [+-]? digit+ )?
```

**Trailing garbage is a REJECTION, never a prefix parse.** This is normative for both shells
and it is the rule the two natural implementations get wrong in *opposite directions*:

- C's `strtof(value, nullptr)` returns `12.0` for `"12px"`, accepts `"12abc"`, skips leading
  whitespace, and accepts its own extensions (`0x10`, `inf`, `nan`). **A `.mm` parser must
  therefore check `endptr` reached the terminating NUL** — and screen the value against the
  production above, which excludes hex/inf/nan.
- Java's float grammar (what Kotlin's `toFloatOrNull` screens with) accepts a **trailing `f`/`d`**
  — `"12f"` parses as `12.0`. So the Kotlin parser screens against the production above *before*
  parsing, too.

Left to their natural implementations the two shells would HONOUR values the other IGNORES, on
the wire's *rejection* path, and the two frame tables would disagree for a reason that has
nothing to do with the engine. The two parsers can never be bit-identical by construction —
**the TEST is the contract, not the implementation**: `WidgetMapperSetStyleTest`'s
`unparseable_layout_value_is_logged_and_ignored` (`12px`) and
`a_trailing_float_suffix_is_rejected_like_any_other_garbage` (`12f`) are the pair every shell
must pass.

**Negative numbers.** Accepted **only** for `margin` and the position offsets `top`/`right`/
`bottom`/`left` — Yoga defines both (a negative margin pulls a node toward its sibling; a
negative offset shifts it against the inset direction). Everywhere else (`width`, `height`,
`min*`/`max*`, `flexBasis`, `padding`, `gap`, `flexGrow`, `flexShrink`) a negative value is
**rejected and logged**, never clamped: Yoga's own behaviour for a negative width is undefined
territory, and a silently-clamped 0 is a frame the two platforms may not agree on.

**NO UNIT SUFFIXES.** `12dp`, `12px`, `12sp` are **not** in this grammar, and **.NET never emits
one** — every string-valued param reaches the wire as a bare number, `"50%"`, or `"auto"`. The
reason is non-negotiable #5: Yoga computes in density-independent units on both platforms with
**one conversion site** (Android's frame-apply). `px` is *not* dp (Android must not multiply it
by density), `sp` is font-scaled, and neither exists on iOS — accepting them would put a
platform-specific unit into a cross-platform layout contract, and the failure mode is the DoD #2
parity assertion going red for a reason that has nothing to do with the engine.

> Shells **may** defensively strip a trailing `dp`/`px`/`sp` and **log a warning** — the existing
> `fontSize`/`padding` handlers do exactly that today and must keep working for the legacy
> *visual* props. That is a tolerance, not part of the grammar.

**Anything else** — an unparseable value, a suffix, an unknown enum word, a negative where
negatives are rejected — is **logged and ignored**. Never silently guessed.

## The proof surface: `BnLayoutDemo` at `/layout`

A component whose **computed frames are fully predictable**, so all four test surfaces assert
the *same numbers* — the discipline the 6.0 spike established (twelve numbers, both rungs):

- a **row** of three boxes: fixed 50 · `flexGrow:1` · fixed 50 → the grow box absorbs the rest
- a **column** with `justifyContent: space-between` and an `alignSelf: center` child
- a **wrap** row whose children overflow onto a second line — four **90dp** boxes in a 300dp row
- a **text leaf** whose *measured* size drives its frame (the DoD #3 proof in the demo itself)

The canonical frame table lives in `BnLayoutDemo.cs`'s file header. Two numbers in it are
load-bearing in a way that is easy to undo:

1. **The wrap boxes are 90dp, not 100.** Four 100s in a 300 row would put three on line 1 *only*
   because Yoga's break test is `consumed + item > available` and `300 > 300` is false — the demo
   would sit exactly *on* the wrap boundary, where half a dp of rounding drift on either shell
   flips box 3 onto line 2 and the two platforms "disagree" for a non-engine reason. 3 × 90 = 270
   leaves 30dp of slack; the break is a fact.
2. **Yoga's `alignContent` default is `flex-start`, which DEVIATES from CSS** (where it is
   `stretch`). The wrap row's second line sits at `y = 40` — the line-1 cross size — *because* of
   that default. A shell that "helpfully" applies the CSS default instead silently shifts the
   frame, and the demo does not set `alignContent` at all (it is not even on the allow-list), so
   there is nothing in the patch stream to catch it. It is asserted in the frame table and
   nowhere else.

Assertions live on: .NET (the emitted `SetStyle` patch golden), Android instrumented (real
`View` frames after layout), iOS XCTest (real `UIView` frames). **The Android and iOS frame
expectations are literally the same numbers** — that is DoD #2's "identically on both".

## Test churn (known, budgeted — the honest cost of decision #1)

These pin `LinearLayout`/`UIStackView` *by type* and get rewritten to pin frames:

| Surface | Files |
|---|---|
| Android instrumented | `WidgetMapperTest`, `BnDemoAndroidTest`, `CompositionAndroidTest`, `WidgetMapperSetStyleTest` (its "unknown property → still a LinearLayout" premise is exactly what this phase inverts) |
| iOS XCTest | `BnRenderTests`, `BnInteractionTests` (their *helpers* hard-cast to `UIStackView`) |
| JVM | `TreeSnapshotTest` — container semantics only if `CONTAINER_TYPES` moves |

The .NET/JVM **patch goldens** (`BnDemoTests.cs`, `BnDemoTest.kt`) only churn if `BnView` emits
new attributes for the *existing* demo — it won't (all new params default null → no patch).
**That is a design constraint, not an accident: adding flex must not perturb an un-styled tree.**

Yoga's JNI binding needs an Android runtime, so **Android Yoga tests are instrumented-only**
(no JVM-unit rung) — as in 6.0.

## Verification (the GREEN bar)

1. `BnLayoutDemo` renders with **identical computed frames** on the AVD and the iOS simulator,
   asserted number-for-number on both.
2. A text leaf's **measured** size drives its frame on both (a long label wraps; the row it
   sits in grows).
3. An un-styled tree (`BnDemo`, `BnSettingsPage`) still renders as a vertical stack —
   goldens unchanged on .NET/JVM, frames equivalent on the shells.
4. A style **reset** (flex prop → null on re-render) returns the node to its default, on both.
5. CI counts bumped and asserted on every surface; `build-test` green.

## Risks

| Risk | Mitigation |
|---|---|
| The container swap breaks interaction/nav tests via their `UIStackView`/`LinearLayout` helpers | Expected and budgeted (table above); rewrite helpers to frame-based accessors *first*, then assert |
| Measure-func re-entrancy / layout thrash (measuring a view that is mid-layout) | Measure on the main thread inside the layout pass only (both platforms already batch to main at `CommitFrame`); no view mutation inside the measure func |
| Android dp/px confusion (Yoga in dp, views in px) | One conversion site: frames are multiplied by `density` at apply time; style values are dp by contract. A unit test pins a known dp→px frame |
| `ScrollView`/`Spinner` accept one child / have their own layout | Out of scope here — 6.2 owns scroll; picker stays as-is; document the boundary |
| Percent / `auto` value parsing diverging between Kotlin and the `.mm` | §"Style value grammar (normative)" is the ONE table both parsers are written from — not restated in the plan or in `FlexStyles.cs`, which reference it. The parity test (same numbers, both platforms) is what catches divergence anyway |
| A shell attaches the measure func to childless `view`s ("no children = leaf") | It attaches **by nodetype** (`text`/`button`/`input`/`image`) — see §"Native measurement". The row demo's boxes are childless views; measuring them destroys the `Grow=1` frame |
