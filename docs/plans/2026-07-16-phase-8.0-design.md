# Phase 8.0 Design — The Samples/Library Separation + Registration Inversion (M8 DoD #1)

**Date:** 2026-07-16
**Milestone:** M8 — Developer Ecosystem (owns **DoD #1**: "the demo app is a consumer,
not a tenant" — the milestone's named risk, verified first)
**Predecessor:** Milestone 7 — complete, tagged `v7.0`
([final audit](2026-07-16-milestone-7-final-audit.md)); milestone opened at `f5acd43`
**Owns:** the registration inversion (`PageManifest` stops naming app types), the
`samples/` app project the shells embed, the test retargeting, and the package-purity
CI pin. **Design only** — implementation follows in this phase's gates.

---

## The one sentence the whole phase hangs on

**Phase 8.0 adds ZERO capability and changes the wire NOT AT ALL** — still 9 exports +
the 72-byte bridge, thirteen NodeTypes on three mirrors, every golden and frame table
passing **unchanged or retargeted-verbatim, never edited**. What it inverts is one
dependency arrow: today the library names the app (`PageManifest.cs` in
`BlazorNative.Runtime` hardcodes 14 demo/probe types and the Runtime csproj references
`BlazorNative.Components` to reach them); after 8.0 the app names itself through a
public registration API and the shells embed **the sample app's** publish. The
inversion, not the file move, is the phase (MILESTONE.md's own words) — it is the
consumer story 8.1 packages, 8.3 templates, and 8.4 documents, and if it fights the
trim laws, this phase finds out before anything is packaged.

Current bars (the standing law: counts with provenance): .NET **539/0** (ci.yml
build-test) · JVM **106/0** (ci.yml) · Android instrumented **184/0**
(android-instrumented.yml) · iOS XCTest **154/0** (ios.yml).

## The coupling as it actually is (verified end to end, not recalled)

1. **`PageManifest.cs`** (`src/BlazorNative.Runtime/PageManifest.cs`) — the 7.6
   unification: 14 rows (9 routed pages + 5 unrouted probes), each a
   `new(route, name, r => r.Mount<T>())` where `T` is a **demo type the library names**
   (`BlazorNative.Components.BnDemo` …). The lambdas are the trim-law shape: statically
   rooted `Mount<T>` instantiations, no reflection, the 4 accepted IL2072s. Plus
   `DefaultComponent` = the `"/"` row's name (`Pages.Single(...)`).
2. **Derived views** — `HostSession.s_components` (all rows → mount registry) and
   `NativeNavigationManager.s_routes` (routed rows → route table) are static-readonly
   projections of that array. The 7.6 normative rule ("a page is declared ONCE") is
   **kept intact by this phase — the ONE declaration just changes owners.**
3. **The pinned mirror** — `MainActivity.DEEP_LINK_COMPONENTS` + the `?: "BnDemo"`
   fallback, pinned pair-for-pair by `RouteTableDriftTests.cs` (Runtime.Tests, the
   required build-test lane) against `PageManifest.Pages` via internals.
4. **The probes live in Runtime itself** — `HelloComponent`, `CompositionProbe`,
   `FocusProbe`, `HostEventProbe`, `ClipboardProbe` are `.cs` files in
   `src/BlazorNative.Runtime/`, not in Components. "Demo pages + probes move out"
   therefore touches BOTH src projects.
5. **How the shells get the app:** `dotnet publish src/BlazorNative.Runtime -c Release
   -r <rid>` produces THE artifact — `BlazorNative.Runtime.dll` (win-x64, the JVM dev
   loop), `BlazorNative.Runtime.so` (linux-bionic ×2, copied to
   `jniLibs/**/libBlazorNative.Runtime.so` by gradle's `runtimePubRoot` machinery), and
   `BlazorNative.Runtime.a` (iossimulator-arm64, `NativeLib=Static`, staged to
   `src/BlazorNative.Apple/vendor/libBlazorNative.Runtime.a`). Because Runtime
   `ProjectReference`s Core + Renderer + Http + **Components**, the native binary IS
   Runtime+Components fused — the shells already embed the whole app; they just embed
   it under the library's name.
6. **The publish gates** — every publish must show exactly **4 IL2072** warnings and
   **9 `blazornative_*` exports** (dumpbin/llvm-readelf in ci.yml, `nm -gU` +
   visibility check in ios.yml). The 4 IL2072s originate in the Renderer's annotated
   internals (the 3.0a/3.0c inventory), **not** in the manifest's call sites — the
   basis of this phase's trim hypothesis (decision 1).
7. **`samples/ConsumerSmoke`** — the existing outside-the-solution precedent: consumes
   the five packages from a local feed, mounts a hand-written component through the
   renderer harness, deliberately WITHOUT HostSession ("the NativeAOT composition root
   is an app-shape concern, unpackaged by design" — the 4.5 record). 8.0 makes that
   4.5 sentence obsolete in the best way: the composition root becomes a *registered*
   concern, and the sample app is the app-shape reference.

## Decisions (recorded at phase-open; ⚠ marks the veto-flagged judgment calls)

### 1. The registration API — an explicit push, made by the app, at startup ⚠

*The call to veto: (a) explicit registration call vs (b) source-generator/attribute
scan vs (c) manifest-type-moves-and-Runtime-consumes-an-interface.*

- **(b) source generator or attribute scan — rejected.** An attribute scan is
  `Assembly.GetTypes()` at runtime: reflection over the whole app assembly, exactly
  what the trim law exists to forbid (every type not otherwise rooted is trimmed; the
  scan would find nothing, or force `TrimmerRootAssembly` on the entire app — the
  sledgehammer that hides real trim bugs). A source generator avoids the runtime
  reflection but buys a generator project, generator tests, and generator *debugging*
  to produce fourteen lines a human writes in two minutes — the 7.6 argument against
  generate-machinery for an 8-row map, applied again. Rejected twice over.
- **(c) the app exposes a manifest type Runtime discovers — rejected as stated,**
  because "Runtime discovers" has no trim-safe mechanism: Runtime cannot name the
  app's type (that is the coupling being deleted) and cannot reflect for it (b's
  problem). The salvageable half of (c) — *the manifest data moves to the app* — is
  absorbed into (a): the app owns the array, and **hands it over** instead of being
  asked for it.
- **(a) an explicit registration call the app makes at startup — the pick.** Push,
  not pull: no discovery problem exists when the app does the naming.

**The API surface, exactly** (new file `src/BlazorNative.Runtime/BlazorNativeApp.cs`;
public — this is the consumer API 8.3's template will call):

```csharp
namespace BlazorNative.Runtime;

/// One page declaration: route (null = mount-by-name only), mount-registry
/// name, and the statically-rooted mount thunk (created ONLY via the two
/// factories — the trim-law shape is not constructible any other way).
public readonly struct BlazorNativePage
{
    public string? Route { get; }
    public string  Name  { get; }
    internal Func<NativeRenderer, int> Mount { get; }

    public static BlazorNativePage Routed<[DynamicallyAccessedMembers(
        DynamicallyAccessedMemberTypes.All)] T>(string route, string name)
        where T : IComponent;   // => new(route, name, static r => r.Mount<T>())

    public static BlazorNativePage Named<[DynamicallyAccessedMembers(
        DynamicallyAccessedMemberTypes.All)] T>(string name)
        where T : IComponent;   // => new(null, name, static r => r.Mount<T>())
}

public static class BlazorNativeApp
{
    public const string DefaultRoute = "/";
    /// Called ONCE, at app startup, before the first mount. Validates loudly.
    public static void RegisterPages(params BlazorNativePage[] pages);
}
```

- **Where it lives: Runtime, not Core.** The mount thunk's type is
  `Func<NativeRenderer,int>` and its consumer is `HostSession` — putting the API in
  Core would force a Core→Renderer dependency that inverts the layering for nothing.
  Core stays pure contracts. (Stated once; every later phase inherits it.)
- **The trim story — the hypothesis Gate 1 verifies.** The lambdas keep their exact
  shape (`static r => r.Mount<T>()`), now written **in app code** through the
  factories. `NativeRenderer.Mount<T>` is already public with
  `[DynamicallyAccessedMembers(All)]` on `T`; the factories carry the same annotation
  on their own generic parameter, so the requirement flows statically — and every app
  call site instantiates with a **concrete** type (`Routed<BnDemo>(…)`), which
  satisfies DAM at compile time. Nothing goes reflective; every `Mount<T>`
  instantiation is rooted from the app's module initializer (decision 2), which ILC
  roots unconditionally. **Hypothesis: IL2072 stays exactly 4** (the accepted four
  live in Renderer code compiled once, same as today) **and no new IL2091/IL2026
  appears.** Gate 1's re-pointed publish gate asserts it on win-x64 +
  linux-bionic ×2; Gate 2's ios.yml run asserts it on iossimulator-arm64. Any drift
  stops the phase for analysis — IL2072 == 4 is re-baselined only with analysis,
  never silently (the standing law).
- **`PageManifest` survives as the internal STORE, its rows leave.** `RegisterPages`
  validates then freezes into `PageManifest.Pages`; `HostSession.s_components` and
  `NativeNavigationManager.s_routes` remain derived views of that one array — the 7.6
  normative rule holds verbatim, one owner swapped. **The views become lazy** (first
  materialization = the freeze point) instead of static-readonly: under NativeAOT, ILC
  may pre-initialize static ctors at compile time, and a pre-baked EMPTY dictionary is
  precisely the silent failure this phase cannot afford. Registration after the freeze
  → `InvalidOperationException`. A second `RegisterPages` → `InvalidOperationException`
  ("pages are registered once, at startup").
- **Validation, loud, at registration time** (`ArgumentException` naming the row):
  non-empty array; unique names; unique routes; and **if any routed row exists, a
  `"/"` row is required** — `DefaultComponent` (which survives, derived exactly as
  today: the `"/"` row's name) must exist for Android's no-extra default mount and
  the route-aware initial mount to mean anything.
- **If the app registers nothing:** `blazornative_mount` returns **rc 1** — no new
  return code, no ABI change — but the stderr diagnostic is distinguished and names
  the fix: *"no pages are registered — the app assembly must call
  BlazorNativeApp.RegisterPages at startup (a [ModuleInitializer]; see
  samples/BlazorNative.SampleApp)"*. `blazornative_init` semantics are deliberately
  unchanged ("the runtime loaded + accessors verify") — inventing an init failure
  class for the zero-pages case would be an observable ABI behavior change in a phase
  whose law is that the wire does not change at all. A .NET test pins the rc-1 +
  diagnostic path via a new internal `ResetRegistrationForTests` (serialized in the
  existing "host-session" collection, like every registry consumer).

### 2. Who calls it: the app's module initializer — because the app IS the binary

There is no managed `Main` in a `NativeLib` — the published binary's entry points are
the 9 exports, all of which live in Runtime, which cannot name the app. The one hook
where **app code runs without being named** is `[ModuleInitializer]`: NativeAOT
compiles module initializers of every assembly into the startup path and runs them
**eagerly at runtime initialization** — i.e. inside the first exported call
(`blazornative_init`), before it returns. So the sample app ships:

```csharp
// samples/BlazorNative.SampleApp/SampleAppPages.cs
public static class SampleAppPages
{
    public static readonly BlazorNativePage[] All = [ /* 14 rows, provenance
        comments moved verbatim from PageManifest.cs */ ];

    [ModuleInitializer]
    internal static void Init() => EnsureRegistered();

    /// Idempotent — tests on CoreCLR call this explicitly (see decision 4:
    /// CoreCLR runs module initializers lazily, on first touch of the
    /// assembly, which a mount-by-NAME test never performs).
    public static void EnsureRegistered() { /* once-guard → RegisterPages(All) */ }
}
```

`SampleAppPages.All` is deliberately **public**: it is the drift test's new target
(decision 4), the template's worked example (8.3), and sample apps are documentation.
The eager-at-startup claim is not taken on faith — it is **verified by the lanes
themselves**: the JVM suite loads the published win-x64 binary and mounts `BnDemo`
(dead on arrival if registration didn't run), and both device suites boot-mount on
hardware. If a future toolchain made initializers lazy, the failure is the loud rc-1
diagnostic above, in the first run that uses that toolchain.

### 3. The sample app project: `samples/BlazorNative.SampleApp`, IN the solution, and it owns the publish

  `BlazorNative.SampleApp`, SDK `Microsoft.NET.Sdk.Razor`
  (+ `StaticWebAssetsEnabled=false`, the 7.0 record). **IN the solution** — the rule,
  stated once: *a samples project is in the solution iff it is in the required
  compile/test path.* The sample app is the shells' publish source and the goldens'
  subject → in. `ConsumerSmoke` exists to prove package-only consumption → stays out.
  The sample app's pages remain subject to the 7.1 supported-`.razor`-subset laws
  (components referenced in markup must be `public` — the BnThemedPanel lesson — the
  pinned SDK pair, no web-only constructs).
- **The publish head moves.** The NativeAOT machinery — `PublishAot`,
  `RuntimeIdentifiers`, the 3.0c/5.0 runtime-pack-bypass PropertyGroups,

### 4. Test retargeting: the tests STAY; their references re-point ⚠

*The call to veto: (A) split a new samples test project vs (B) Runtime.Tests
references the sample app.*

- **(A) a `BlazorNative.SampleApp.Tests` project — rejected for churn without gain.**
  ~17 of Runtime.Tests' files name app types (the 8 demo-page golden suites,
  `HelloGoldenTests`, the 4 probe suites, `NavigationTests`, `RouteTableDriftTests`,
  the SpikeRazor pair, plus the session/dispatch/bridge machinery tests that mount
  probes). Moving them means a new project, new IVT grants from Runtime AND
  Components AND SampleApp, a split host-session serialization story, and a count
  ledger tracking tests across a project boundary — all to relocate tests that ship
  nowhere. Package purity is about **shipped assemblies**; test projects are not in
  the shipped set and never will be.
- **(B) Runtime.Tests adds a `ProjectReference` to SampleApp — the pick.** Zero test
  files move; every golden passes **byte-identical** because not even its file path
  changes. The retargets, exhaustively: (i) `using BlazorNative.Components` →
  `using BlazorNative.SampleApp` where a moved type is named (mechanical, not a
  golden edit); (ii) a one-line module initializer in Runtime.Tests (the
  `StrictModeTestDefaults` pattern, already in the house) calls
  `SampleAppPages.EnsureRegistered()` — CoreCLR runs module initializers lazily, and
  a mount-by-NAME test never touches a SampleApp type, so the explicit call is the
  deterministic hook; (iii) SampleApp grants `InternalsVisibleTo` to Runtime.Tests
  (the demo pages' `internal const` frame arithmetic — the 6.2 pattern, new grantor).
- **`RouteTableDriftTests` retargets one expression** — `PageManifest.Pages` →
  `SampleAppPages.All` — and otherwise survives verbatim: same file, same required
  lane (build-test, the one lane where every file is checkout-visible), same pair
  pin, same default-fallback pin, same nine-page content baseline. The drift-test
  discipline survives with the manifest's new owner, exactly as DoD #1 demands.
  `NavigationTests`' baseline: untouched (it drives routes through the same derived
  views).
- **Both device suites: unchanged, stated as a bar.** They mount by name/route
  against the published binary, which still CONTAINS the pages (same code, new
  assembly identity) — Android instrumented stays **184/0**, iOS stays **154/0**,
  with zero assertion edits; Gate 2's runs are the evidence.
- **The count story, with provenance:** .NET stays a single lane number — nothing
  moves between lanes. 539 existing (all pass retargeted) + new in Gate 1:
  registration validation ≥ 6 (empty array, dup name, dup route, missing `"/"`,
  double-register, register-after-freeze), empty-registry mount rc-1 + diagnostic
  ≥ 1, purity pins ≥ 3 (decision 5) → **.NET ≥ 549**. JVM 106, instrumented 184,
  iOS 154: all unchanged.

### 5. The package-purity pin: a two-sided roster + a pattern net, in the required lane

New `PackagePurityTests.cs` in Runtime.Tests (build-test — required, checkout-visible,
and it can load every shipped assembly by reference):

1. **The roster, both directions.** The 16 moved types (9 pages + `BnThemedPanel` +
   `SpikeRazor` + `HelloComponent` + 4 probes) as a literal list: each asserted
   **PRESENT** in `BlazorNative.SampleApp.dll` and **ABSENT** (by full enumeration)
   from every shipped assembly. One direction alone is gameable — deleting a type
   outright would green an absence-only pin; the presence side catches it.
2. **The pattern net** for future strays: zero types in any shipped assembly matching
   `.*Demo$ | .*Probe$ | ^SpikeRazor` — catches the NEXT demo page someone parks in
   the library, which the frozen roster cannot.
3. **The shipped set is itself a pinned literal** — `Core, Renderer, Http,
   Components, Runtime, Analyzers` — so a new shipped assembly must join the pin
   deliberately, not drift in unexamined.

**The mutation that proves it bites (run at Gate 1, recorded, reverted):** add
`internal sealed class BnScratchProbe {}` to Components → the pattern net red, naming
the type and assembly; re-declare `BnDemo` in Components → the roster's absence side
red. Extending the pin to interrogate the **packed nupkg** contents is 8.1's job by
construction (8.1 owns pack; there is no nupkg to interrogate in 8.0) — recorded as
8.1 input, and the assembly-level pin is the stronger daily guard anyway (it runs on
every PR in the required lane, not just when packing changes).

## The normative rules, stated once

> **A page is declared ONCE — one row in the app's manifest** (`SampleAppPages.All`),
> registered through `BlazorNativeApp.RegisterPages`. Runtime's store and its two
> derived views cannot drift from it by construction; Android's
> `DEEP_LINK_COMPONENTS` remains the one PINNED MIRROR, drift-tested pair-for-pair in
> the required lane. (The 7.6 rule, owner swapped.)

> **Shipped assemblies carry no app types.** The shipped set is a pinned literal; the
> roster + pattern net in `PackagePurityTests` enforce it on every PR.

> **The embedded artifact name is shell contract**: `libBlazorNative.Runtime.so` /
> `libBlazorNative.Runtime.a` / `BlazorNative.Runtime.dll`, frozen regardless of which
> project publishes it; the publish head canonicalizes.

## Risks

| Risk | Mitigation |
|---|---|
| **The trim risk (the named one, leads):** moving mount call sites to app code changes what ILC roots — new IL2072/IL2091, or trimmed pages | The lambdas keep their exact shape behind DAM(All)-annotated factories; concrete `T` at every call site satisfies DAM statically; module initializers are unconditional roots. Gate 1 verifies on three RIDs, Gate 2 on the fourth, before anything else builds on the API. Drift = stop and analyze — never re-baseline silently. |
| `UnmanagedEntryPointsAssembly` fails to emit — or Mach-O localizes — the 9 exports from a referenced assembly | The existing export-count checks (dumpbin/llvm-readelf/`nm -gU` + the ios.yml visibility check) fail same-commit; fallback is a thin re-export shim in the sample app, designed only if needed. |
| ILC pre-initializes the derived views to empty (the static-ctor preinit hazard) | The views go lazy-after-freeze by design (decision 1); the JVM + device boot mounts would fail loudly (rc 1 + diagnostic) if an empty registry ever shipped. |
| Module-initializer timing differs across runtimes | NativeAOT: eager at startup, verified by every lane that loads the binary. CoreCLR test host: lazy by spec — handled by the explicit `EnsureRegistered()` in the tests' module initializer, not by faith. |
| Razor SDK (`Microsoft.NET.Sdk.Razor`) + `NativeLib`/runtime-pack-bypass publish head friction (untested combination) | Components already builds AOT-compatible under the Razor SDK; if the publish head itself trips, the recorded fallback is a two-project split (SampleApp Razor lib + a plain-SDK publish head referencing it) — same API, same artifact, one extra csproj. |
| A missed re-point site ships a stale binary | Impossible silently: every consumer copies by explicit path; a stale/missing path is a loud file-not-found in the lane that owns it. The touch-point table (decision 6) is the checklist. |
| Golden churn sneaks in as "retargeting" | The law restated in the bar: goldens/frame tables pass unchanged or retargeted-verbatim (usings/paths only) — a golden VALUE edit fails review by definition; Gate 1 cites `git diff --stat` over the golden files. |
