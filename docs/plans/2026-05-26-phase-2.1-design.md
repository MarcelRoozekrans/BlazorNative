# Phase 2.1 Design — JVM Desktop Hosts BlazorNative.WasiHost via libwasmtime + JNA

**Date:** 2026-05-26
**Milestone:** M2 — First End-to-End Demo on Android
**Phase:** 2.1 (replaces previous "Android Kotlin shell scaffold + wasmtime-java Gradle integration" framing; see "Phase split" below)
**Status:** Design — approved 2026-05-26

---

## Goal

Prove `BlazorNative.WasiHost.wasm` can be loaded and booted in-process from a JVM host on Windows x86_64 via `libwasmtime` (cargo-built from source) bound through JNA. The runtime layer this phase produces ships unchanged to Android in Phase 2.2 — only the libwasmtime build target changes.

## Why this phase exists (and why it replaces the original Phase 2.1 framing)

The MILESTONE originally framed Phase 2.1 as "Android Kotlin shell scaffold + `wasmtime-java` Gradle integration." During Phase 2.1 brainstorming we discovered:

1. **`dev.wasmtime:wasmtime-java` does not exist** (404). The BACKLOG/MILESTONE bullet was aspirational.
2. **`kawamuray/wasmtime-java`** (the closest community effort) was last released October 2023, ships no Android ABI binaries.
3. **`chicory`** (pure-JVM WASM runtime) is WASIp1 only — explicitly no component model, no `wasi:http`. Our `.wasm` is currently a component (uses `wasi:http/types`).
4. **`wasmtime`'s own C API is mature** for core wasm + WASIp1, and basic component-model support has landed (issue #6987 closed); structured WIT-typed-import support is still being added (issues #11437, #11617 open as of mid-2025).
5. **`wasmtime` cross-compiles to Android via NDK** — proven by upstream issues and community examples.

### Chosen strategy: G (RN-Hermes pattern)

Cross-compile `libwasmtime` for each target ABI via NDK + cargo; bind from Kotlin via JNA. This mirrors how React Native ships Hermes (per-ABI `libhermes.so` in the APK, bound via JSI). RN's existence is the proof that this is a sane architecture.

The alternatives considered and rejected:

- **A (chicory):** Pure-JVM, no JNI. Cheapest engineering. Rejected because (i) interpreter-grade perf is the wrong default for "best future possibilities," (ii) WASIp1-only locks us out of typed-bridge upgrades (WIT-typed imports are the WASM-world's JSI), and (iii) requires the same format pivot anyway.
- **C (bundled wasmtime CLI + IPC):** Proven to work but architecturally wrong for an interactive UI bridge.
- **D (defer the runtime call):** Kicks the can without learning anything.

### Phase split (replaces original Phase 2.1 as a single phase)

Strategy G's surface area for "first end-to-end load" doesn't fit cleanly in one phase. Split:

| Phase | Old name | New scope |
|---|---|---|
| 2.1 | Android scaffold + module loads | **JVM desktop hosts `.wasm` via libwasmtime + JNA on Windows x86_64.** Format-pivot spike, cargo build of `wasmtime.dll`, JNA bindings, BootSmokeTest. **No Android.** |
| 2.2 | mobile_bridge symbol exports (Android) | **Android scaffold + cross-compile libwasmtime for android-arm64.** Reuses Phase 2.1's JNA layer unchanged. |
| 2.3 | render-frame consumer | mobile_bridge real symbol implementations (was Phase 2.2). |
| 2.4 | widget mapper | render-frame consumer (was Phase 2.3). |
| 2.5+ | … | … (renumber down by one) |

Rationale: desktop-first isolates runtime-layer risk (format pivot + JNA bindings + libwasmtime C-API quirks) from Android build pipeline risk (NDK + emulator + Gradle Android plugin). Each phase has a real green/red checkpoint.

## Locked decisions and rationale

| ID | Decision | Rationale |
|---|---|---|
| Q1 | Runtime strategy = **G (libwasmtime + JNA via NDK)** | RN-Hermes pattern; best perf + future possibilities; only path that keeps the component model alive without depending on a dead binding (kawamuray) or an unmaintained one (chicory's WASIp2 story). |
| Q2 | Sequence = **format-pivot spike first** (Phase 2.1.0 internal task) | Strategy G's whole premise rests on "wasi-experimental can emit a slim component with custom WIT imports." Unverified. Spike fails fast or unblocks rest of phase. |
| Q3 | Format pivot shape = **minimal strip (drop wasi:http, keep component, author mobile_bridge.wit)** | Preserves component model for future typed-bridge upgrades (the WIT→JSI-equivalent path). C-API exposes enough for our import shapes (after Q5). |
| Project layout | **pure-Gradle/Android Studio** (NOT dotnet-android) | NDK + cargo + JNA are all Gradle-native; dotnet-android would fight them. |
| Q4 | Scope = **split Phase 2.1 desktop / Phase 2.2 Android** | Isolates runtime-layer risk from Android build pipeline risk; each phase has a checkpoint. |
| Q5 | WIT contract = **hybrid** (typed primitives for simple imports, JSON-string payloads for structured ones) | Avoids betting Phase 2.1 on full WIT-record marshaling (C-API issues #11437 #11617 still open); still establishes typed-import muscle for the simple cases. Non-breaking upgrade path for structured imports later. |
| Q6 | mobile_bridge stubs = **pure no-ops returning sensible defaults** | Phase 2.1's goal is "module loads + boots"; nothing meaningful to assert about stub behavior yet. Real implementations land in Phase 2.3 (was 2.2). |
| Q7 | libwasmtime acquisition = **cargo build from source via setup.ps1** | Front-loads Rust toolchain so Phase 2.2 only adds NDK target on top of working cargo; avoids "two sources of libwasmtime" between desktop and Android. |
| Q8 | JVM structure = **Kotlin + Gradle from day one, `src/BlazorNative.Jni/`, JUnit 5 in `src/test/kotlin/`** | No rewrite tax in Phase 2.2 (same code targets Android); automated regression for the .wasm boot path; consistent with Q7 toolchain front-loading. |

### Limitations of Strategy G (documented per user request)

1. **C API component-model support is partially exposed.** Issue #6987 (basic) closed; #11437 (Resources) and #11617 (Byte arrays) open as of mid-2025. We mitigate by Q5 (hybrid WIT — only use C-API surfaces we know work).
2. **No prebuilt Android C-API artifact ships from wasmtime.** We cross-compile via NDK + cargo in Phase 2.2. Cost is real but well-trodden.
3. **JNA bindings are ours to write and maintain.** ~30 C-API calls. Not exotic but not free.
4. **WIT-typed-import marshaling for structured records is a research risk for future phases.** When we want to upgrade `shell-fetch` from JSON-string to typed records, we revisit C-API state then.
5. **Single-process, single-ABI desktop runtime only in Phase 2.1.** Linux/macOS support deferred to backlog; concurrency/render-loop deferred to Phase 2.4+.
6. **JNA crossing cost is non-zero.** Every C-API call pays a JNI boundary. Acceptable for boot path; revisit if Phase 2.4+ render loop shows it as a bottleneck.

## Architecture

```
[Phase 2.1.0 — format-pivot spike]
  src/BlazorNative.WasiHost/
    + wit/mobile_bridge.wit                  ← custom WIT (typed primitives + JSON-string structured)
    BlazorNative.WasiHost.csproj             ← drops wasi:http transitive, configures wasi-experimental
                                               to emit a slim component importing only mobile_bridge
    ↓ dotnet publish -r wasi-wasm -c Release
  bin/Release/net10.0/wasi-wasm/AppBundle/
    BlazorNative.WasiHost.wasm               ← slim component, 7 mobile_bridge imports

[Phase 2.1.1 — runtime layer]
  vendor/wasmtime/
    wasmtime.dll                             ← cargo-built by setup.ps1
  src/BlazorNative.Jni/                      ← Kotlin/Gradle module
    src/main/kotlin/.../WasmtimeBindings.kt  ← JNA bindings (~30 C-API calls)
    src/main/kotlin/.../MobileBridgeHost.kt  ← Kotlin-side no-op stubs for the 7 imports
    src/main/kotlin/.../WasiHost.kt          ← high-level load+boot façade
    build.gradle.kts

[Phase 2.1.2 — verification]
  src/BlazorNative.Jni/src/test/kotlin/.../BootSmokeTest.kt
    @Test fun boots_and_emits_markers() { ... 4 marker asserts ... }
```

**Key flow:** `dotnet publish` emits the slim component → `setup.ps1` (one-time) ensures `wasmtime.dll` exists in `vendor/` → `./gradlew test` runs `BootSmokeTest` → JNA loads `wasmtime.dll`, instantiates the component, links the 7 stub imports, calls `_start`, captures stdout, asserts the 4 `[BOOT]` markers from Phase 2.0.

## Components

### `src/BlazorNative.WasiHost/wit/mobile_bridge.wit` (new, ~40 lines)

```wit
package blazornative:mobile-bridge@0.1.0;

interface bridge {
  // Typed primitives (Phase 2.1)
  shell-navigate: func(route: string);
  shell-current-route: func() -> string;
  shell-platform-info: func() -> string;

  // JSON-string structured (deferred to typed records in a future phase)
  shell-fetch: func(request-json: string) -> string;
  shell-storage-read: func(key: string) -> string;
  shell-storage-write: func(key: string, value: string);
  shell-storage-delete: func(key: string);
}

world wasi-host {
  import bridge;
  export wasi:cli/run@0.2.0;  // for _start
}
```

- Kotlin JVM plugin, JUnit 5, JNA 5.x as Maven deps.
- `tasks.test { systemProperty("wasm.path", project.rootDir.resolve("src/BlazorNative.WasiHost/bin/Release/net10.0/wasi-wasm/AppBundle/BlazorNative.WasiHost.wasm").absolutePath) }`.
- `systemProperty("jna.library.path", project.rootDir.resolve("vendor/wasmtime").absolutePath)`.

### `src/BlazorNative.Jni/src/main/kotlin/io/blazornative/jni/WasmtimeBindings.kt` (new, ~300 lines)

JNA `Library` interface with the ~30 C-API functions we need:
- Engine/store: `wasm_engine_new`, `wasm_engine_delete`, `wasm_store_new`, `wasm_store_delete`, `wasmtime_config_new`, `wasmtime_config_wasm_component_model_set`.
- WASI config: `wasmtime_wasi_config_new`, `wasmtime_wasi_config_inherit_stdout` (or `_set_stdout_pipe`), `wasmtime_wasi_config_inherit_stderr`, `wasm_store_context_set_wasi`.
- Component: `wasmtime_component_new`, `wasmtime_component_delete`.
- Linker: `wasmtime_component_linker_new`, `wasmtime_component_linker_define_func`, `wasmtime_component_linker_delete`.
- Instance: `wasmtime_component_instantiate`, `wasmtime_component_instance_get_func`.
- Call: `wasmtime_component_func_call`.
- Error: `wasmtime_error_message`, `wasmtime_error_delete`, `wasmtime_trap_new`.

Struct wrappers (`Pointer`-typed): `Engine`, `Store`, `Component`, `Linker`, `Instance`, `Func`.

### `src/BlazorNative.Jni/src/main/kotlin/io/blazornative/jni/MobileBridgeHost.kt` (new, ~100 lines)

- `fun stubs(): MobileBridgeHandlers` — struct of 7 Kotlin lambdas implementing the `bridge` interface. Each returns a sensible default:
  - `shellCurrentRoute() → "/"`
  - `shellPlatformInfo() → """{"phase":"2.1","runtime":"jvm-desktop"}"""`
  - `shellNavigate(_) → Unit`
  - `shellStorageRead(_) → ""`
  - `shellStorageWrite(_, _) → Unit`
  - `shellStorageDelete(_) → Unit`
  - `shellFetch(_) → """{"status":501,"body":"stub"}"""`
- JNA callback wrappers marshal `(Pointer, Long)` UTF-8 buffer pairs ↔ Kotlin `String`.

### `src/BlazorNative.Jni/src/main/kotlin/io/blazornative/jni/WasiHost.kt` (new, ~80 lines)

- `fun loadAndRun(wasmPath: Path, handlers: MobileBridgeHandlers): String` — high-level façade.
- Sets up engine + store + component + linker, registers stub handlers on `blazornative:mobile-bridge/bridge`, captures stdout via wasi-preview2 stdio redirection (preferred) or `System.setOut` wrapper (fallback), calls `_start`, returns captured stdout.

### `src/BlazorNative.Jni/src/test/kotlin/io/blazornative/jni/BootSmokeTest.kt` (new, ~30 lines)

One JUnit 5 test:

```kotlin
@Test
fun boots_and_emits_markers() {

    val stdout = WasiHost.loadAndRun(wasmPath, MobileBridgeHost.stubs())
```

Direct parity with `tests/BlazorNative.Wasi.Tests/BootSmoke.cs` (same assertions, different host).

### `setup.ps1` (modify, ~40 lines added)

- Detect Rust → install via `winget install Rustlang.Rustup` if missing.
- `git clone https://github.com/bytecodealliance/wasmtime --depth 1 --branch v45.0.0 vendor/wasmtime-src` (if missing).
- `cargo build -p wasmtime-c-api --release --manifest-path vendor/wasmtime-src/Cargo.toml`.
- Copy `vendor/wasmtime-src/target/release/wasmtime.dll` → `vendor/wasmtime/wasmtime.dll`.
- Assert Java 17+ present (Marcel's Java 25 satisfies; no install).
- Gradle wrapper bootstraps itself on first `./gradlew` invocation; no script work needed.

## Data Flow

### Boot sequence (test execution)

```
1. ./gradlew test
     ↓ invokes
2. BootSmokeTest.boots_and_emits_markers
     ↓ calls
3. WasiHost.loadAndRun(wasmPath, MobileBridgeHost.stubs())
     ↓
4. JNA loads wasmtime.dll (lazy, once per JVM)
     ↓
5. wasm_engine_new(config with component-model=true, wasi=true)
6. wasmtime_store_new + wasi-preview2 stdio config (capture stdout to pipe)
7. wasmtime_component_new(engine, .wasm bytes)
8. wasmtime_component_linker_new(engine)
     for each of 7 mobile_bridge imports:
       wasmtime_component_linker_define_func(linker, "blazornative:mobile-bridge/bridge", "shell-...", &cb_...)
9. wasmtime_component_instantiate(store, component, linker) → instance
10. wasmtime_component_instance_get_func(instance, "_start") → func
11. wasmtime_component_func_call(func, [], [])
     ↓ runs Mono-AOT'd .NET inside wasm sandbox:
         Console.WriteLine("[BOOT] entry-point ...")
         (DI graph construction)
         Console.WriteLine("[BOOT] di-ok ...")
         WasiBridge.DispatchEventCore("self-test", "phase-2.0")
         Console.WriteLine("[BOOT] event-ok ...")
         Console.WriteLine("[BOOT] done ...")
     ↓ wasi-preview2 routes stdout writes to the captured pipe
12. WasiHost reads pipe → returns captured stdout
13. BootSmokeTest asserts 4 marker substrings present
```

### mobile_bridge call flow (if invoked)

```
.NET-side (inside .wasm):
   var info = WasiBridge.Current.PlatformInfo   // hits the import
     ↓ extern call via WIT import binding
     ↓ wasmtime resolves → invokes registered linker func
JNA-side:
   cb_platform_info callback invoked by wasmtime
     ↓ no args to marshal
     ↓ delegates to MobileBridgeHost.stubs().platformInfo()
     ↓ returns """{"phase":"2.1","runtime":"jvm-desktop"}"""
JNA marshaling out:
   Kotlin String → UTF-8 bytes → write into wasmtime-allocated buffer
     ↓ wasmtime copies into guest memory, returns to .NET caller
.NET-side:
   string value returned from extern call
```

### Stdout capture mechanism

wasi-preview2's `wasi:cli/stdout` is a `wasi:io/streams.output-stream` resource. Two strategies (spike picks the simpler one that works):

- **(a) wasmtime-side redirect:** `wasmtime_wasi_config_set_stdout_pipe` (if exposed by C-API). Clean.
- **(b) JVM-side capture:** wrap `System.setOut(PrintStream)` around a `ByteArrayOutputStream`. Works if wasmtime inherits the JVM's stdout. Fallback.

## Error Handling

### Boundary 1: Format-pivot spike (Phase 2.1.0)

Binary outcome. Spike's last step: `wasm-tools component wit BlazorNative.WasiHost.wasm`. Output must list `blazornative:mobile-bridge/bridge` and `wasi:cli`-shaped exports. Output must NOT list `wasi:http/types`. If the output is wrong, **the spike fails and we stop** — re-plan back to Q3 (fall back to full core-module strip, Q3=B).

### Boundary 2: JNA → wasmtime C-API errors

Every C-API call that can fail returns a `wasm_trap_t*` or `wasmtime_error_t*`. JNA wrapper:

```kotlin
class WasmtimeException(val source: String, val message: String) : RuntimeException("$source: $message")

inline fun <T> checkWasmtime(call: String, block: () -> Pointer?): T {
    val errPtr = block() ?: return Unit as T
    val msg = WasmtimeBindings.INSTANCE.wasmtime_error_message(errPtr)
    WasmtimeBindings.INSTANCE.wasmtime_error_delete(errPtr)
    throw WasmtimeException(call, msg)
}
```

No retries, no fallbacks. Wasmtime errors mean the .wasm is malformed, the binding is wrong, or the host is broken — all want immediate visibility.

### Boundary 3: Host-stub callback exceptions

JNA callbacks run on wasmtime-owned threads. Uncaught Kotlin exceptions can silently crash the JVM. We wrap each callback:

```kotlin
private fun safeCallback(name: String, body: (Args) -> Result): NativeCallback = NativeCallback { args ->
    try { body(args) }
    catch (t: Throwable) {
        System.err.println("[mobile_bridge stub '$name' threw] ${t.javaClass.simpleName}: ${t.message}")
        WasmtimeBindings.INSTANCE.wasmtime_trap_new("stub '$name' threw: ${t.message}", t.message?.length ?: 0)
    }
}
```

Mirrors Phase 2.0's `DispatchEventCore` catch-and-continue — one bad stub shouldn't kill the JVM.

### What we deliberately don't handle

- libwasmtime missing → JNA `UnsatisfiedLinkError`. Setup.ps1's responsibility.
- .wasm path missing → `gradle test` fails before any wasmtime call. Self-explanatory.
- Marker assertions fail → JUnit fails with captured stdout in the message. Stdout IS the diagnostic.
- Concurrency → Phase 2.1 is single-threaded. Bridge concurrency arrives with the render loop in 2.4+.

## Open risks (carried into implementation)

1. **wasi-experimental may not support custom WIT import authoring.** Mitigated by the Phase 2.1.0 spike as fail-fast.
2. **wasmtime C-API may not expose `wasmtime_component_linker_define_func` for arbitrary WIT signatures.** Discovered during JNA bindings work; fallback is to author the WIT with the smallest subset the C-API definitely supports.
3. **Stdout capture mechanism unverified.** Two strategies documented (wasmtime-side pipe redirect vs JVM-side `System.setOut`); spike picks the working one.
4. **JNA callback threading model on wasmtime-owned threads.** May require `Callback` attaching to JVM via `AttachCurrentThread` — JNA usually handles this transparently but worth verifying during first-run.

These are all "discovered during the work" risks, not "design choices we punted on." Each has a known fallback.

## Appendix: Performance expectations vs .NET MAUI

Captured during Phase 2.1 brainstorm (2026-05-26) so the project's perf positioning is unambiguous for future contributors.

**Short version: BlazorNative will not beat MAUI on conventional runtime perf. It is structural, not a tooling gap that closes over time.** The framework is positioned on a different axis — single-artifact portability + deploy velocity + sandboxing + Razor-to-native — not on raw CPU/memory perf.

### Where MAUI structurally wins

| Metric | MAUI | BlazorNative | Why |
|---|---|---|---|
| Cold start (fresh install) | 200–800ms | 500–1500ms | One-time Cranelift JIT compile of the .wasm at first load. |
| Cold start (cached `.cwasm`) | 200–800ms | 300–1000ms | Still pays "load WASM → instantiate → init Mono inside WASM" vs MAUI's "dlopen Mono → run". |
| Hot loop CPU (compute-bound) | 1.0× baseline | 1.05–1.30× slower | Cranelift-compiled WASM is well-optimized but pays for memory bounds checks, function-table indirection, linear-memory model. |
| Native call (bridge crossing) | ~50–200 ns (C# → JNI) | ~500–2000 ns (WASM → JNA → JVM, with marshaling) | We cross two boundaries; MAUI crosses one. Per-call overhead ~10× worse — death by a thousand cuts on chatty UIs. |
| Memory footprint | ~30–60 MB | ~50–90 MB | Two GCs (Mono-in-WASM + JVM), wasmtime engine state, JNA reflection overhead. ~20–30 MB structural overhead. |
| Binary size | 15–30 MB APK | 25–40 MB APK | .wasm (~13 MB Mono-AOT) + libwasmtime (~10 MB) + Kotlin shell. Trimming helps but doesn't close the gap. |
| GC pause predictability | Mono GC only | Mono-in-WASM GC has fewer optimizations than standalone Mono | Less predictable pause times under load. |

### Where BlazorNative wins (perf-adjacent, not runtime perf)

| Metric | MAUI | BlazorNative | Winner |
|---|---|---|---|
| Deploy a hotfix | Build APK + sign + upload + store review (hours–days) | Push new `.wasm` (~13 MB, delta-compressible to KB) (seconds, modulo store policy) | BlazorNative by orders of magnitude |
| A/B variant rollout | New APK per variant | Different `.wasm` per cohort, swapped at runtime | BlazorNative significantly |
| Dev iteration | Full rebuild + reinstall (~30–120s) | Rebuild `.wasm` + hot-swap (~5–10s, future hot-reload work) | BlazorNative if hot-reload ships |
| CI pipeline duration | N parallel platform builds | 1 `.wasm` build + N cached shell builds | BlazorNative significantly |
| Sandbox-isolated third-party code | Process-level only (heavy) | WASM-module-level (cheap) | BlazorNative structurally |

### Positioning consequence

The honest framing: **BlazorNative is the .NET answer to React Native's value proposition, not the .NET answer to Native development.** MAUI is the latter (Xamarin reborn — write .NET, get native). BlazorNative is the former (RN/Flutter shape, but Blazor — write Razor, get a portable artifact that renders native widgets via thin shells).

**When MAUI is the right call:**
- Performance-sensitive app (games, video editing, real-time DSP, large-data compute).
- Need deep platform APIs without bridge-marshaling cost.
- Shipping production within ~12 months.
- Team prefers Microsoft-supported, first-class IDE tooling.
- Per-platform builds are acceptable.

**When BlazorNative is the right call:**
- App is UI-heavy with conventional CRUD logic (~95% of mobile apps) where the 5–30% hot-loop tax + 500ms cold-start tax is imperceptible.
- OTA hotfix capability is valuable.
- Sandboxed/auditable app surface matters (financial, healthcare, government, multi-tenant).
- Want byte-identical artifact across mobile + desktop + browser.
- Razor → native widgets (not WebView, which is what MAUI Blazor Hybrid delivers).

**When neither is the right call:**
- Top-of-the-app-store game-grade perf required → go pure native (Swift / Kotlin).

### Why this matters for milestone planning

Future milestones (M5 platform coverage, M7 framework hardening, M8 enterprise readiness) should NOT have perf-parity-with-MAUI as a goal. They should have perf-parity-with-React-Native as the realistic benchmark, since that's the actual competitor in the "portable runtime + native widgets" category. M4 (production-shippable) DoD criteria should reflect this — "fast enough that users can't tell" is the target, not "as fast as MAUI."
