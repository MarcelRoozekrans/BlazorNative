### Diagnostic iterations (1 used, of 1 allowed)

**Iteration 1 — `android_clang.cmd` not found (x64 publish, first attempt).**

Verbatim error:

```
'"android_clang.cmd"' is not recognized as an internal or external command,
operable program or batch file.
C:\Users\MarcelRoozekrans\.nuget\packages\microsoft.dotnet.ilcompiler\10.0.9\build\Microsoft.NETCore.Native.Unix.targets(321,5): error MSB3073: The command ""android_clang.cmd" -fuse-ld=lld -Wl,--version" exited with code 9009.
```

Root cause: this machine sets the Windows hardening environment variable
`NoDefaultCurrentDirectoryInExePath=1`, which removes the current directory
from cmd.exe's executable search path. The vendored targets' Windows shim
relies on exactly that lookup: `PrepareNdkBatches` writes `android_clang.cmd`
into the project directory (CWD) and sets `CppCompilerAndLinker` to the bare
name `android_clang.cmd`. The `where /Q` linker probe in
`Microsoft.NETCore.Native.Unix.targets` still passed (where.exe always
searches CWD), so the failure surfaced only at the first real linker
invocation with exit code 9009. Confirmed by manual repro: with the variable
set, `cmd /c "cd /d <projdir> && android_clang.cmd --version"` fails
identically; with it removed, clang 17.0.2 runs.

Fix: remove `NoDefaultCurrentDirectoryInExePath` from the environment of the
`dotnet publish` invocation (`Remove-Item Env:\NoDefaultCurrentDirectoryInExePath`).
No file changes needed. Retry succeeded; the arm64 publish (run with the same
environment) succeeded on the first attempt.

**Consequence for later gates/CI:** any wrapper script that drives a bionic
publish on Windows must clear `NoDefaultCurrentDirectoryInExePath` (or the
targets must be changed to reference the shim by absolute path,
`$(MSBuildProjectDirectory)\android_clang.cmd`).

### Non-fatal noise

Each publish logs `'test' is not recognized as an internal or external
command` — a Unix-only probe inside `Microsoft.NETCore.Native.Unix.targets`
running under `IgnoreExitCode`; harmless on a Windows host.

### Repo housekeeping

The root `build/` directory was caught by the Gradle `build/` ignore rule in
`.gitignore`; added a `!/build/` negation (with comment) so the vendored
targets file is tracked.

### Step 0 deviation — BionicNativeAot.targets shim resolution (carry-forward from Gate 1 review)

The Gate 1 review prescribed absolute `$(MSBuildProjectDirectory)` shim paths
in `CppCompilerAndLinker`/`ObjCopyName`. That prescription is **not workable**:

1. The ILCompiler SDK probes the linker with `where /Q "$(CppLinker)"`
   (`SetupOSSpecificProps`), and where.exe rejects absolute paths:
   `EXEC : error : Invalid pattern is specified in "path:pattern"`.
2. Keeping the property bare for the probe and rewriting `CppLinker` to an
   absolute path in a follow-up target also fails: `SetupOSSpecificProps`
   itself invokes `"$(CppLinker)" -fuse-ld=lld -Wl,--version` (lld version
   detection, feeds the `sections.ld` `KEEP(__modules)` handling), so the bare
   name hits cmd.exe before any of our targets can run:
   `error MSB3073: The command ""android_clang.cmd" -fuse-ld=lld -Wl,--version" exited with code 9009`.

The only string that satisfies both where.exe (bare name) and cmd.exe under
`NoDefaultCurrentDirectoryInExePath=1` (path-qualified or PATH-resolvable) is
a **bare name resolvable via PATH**. Empirically verified: with the env var
set, `where.exe /Q android_clang.cmd` from the containing directory exits 0
while `cmd /c android_clang.cmd` exits 1 and `cmd /c <absolute path>` exits 0.

Final deviation (committed as `38d6171`): shim names stay bare; a
`RoslynCodeTaskFactory` inline task (`PrependShimDirToNodePath`, invoked from
`PrepareNdkBatches`) idempotently prepends `$(MSBuildProjectDirectory)` to the
MSBuild node process PATH, so every later SDK `Exec` (probe, lld version
check, link, objcopy) resolves the shims via PATH. Shim files are still
written/deleted with absolute paths, gitignored (`android_clang.cmd`,
`android_llvm-objcopy.cmd`) since `RemoveBatchFiles` only runs when linking
succeeds. Verified: the Gate 1 scaffold publishes `linux-bionic-x64` from the
**repo root** with `NoDefaultCurrentDirectoryInExePath=1` left set — the Gate 1
"clear the env var" workaround is no longer needed.

### Diagnostic iterations (1 used, of 1 allowed)

**Iteration 1 — `struct_sizes_match_c_abi` failed `expected:<24> but was:<8>`
on the options struct.** Not an ABI mismatch and not Android-specific: JNA's
