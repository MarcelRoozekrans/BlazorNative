## What this document is

The **reproducible zero-to-running walk** DoD #3 demands: the exact commands a stranger runs, in
order, from an empty directory to a dual-ABI Android APK — **touching no BlazorNative repo source
at any point.** Every package comes from nuget.org. There is no `git clone`, no `ProjectReference`,
no local pack step.

It is written from two walks actually performed: `bn-baseline` on published **0.3.0** (Gate A) and
`bn-zeroalloc-showcase` on published **0.4.0** (Gate D), both under `C:\Projects\Dogfood\`.

**Read "What this proves — and what it does not" at the bottom before quoting anything from here.**

## 0. Prerequisites

The SDK/JDK/NDK table lives in the published docs
([Installation → Prerequisites](https://marcelroozekrans.github.io/BlazorNative/docs/getting-started/installation)) and is
not duplicated here. What **is** duplicated here — because the dogfood hit all three, on two
separate machines — are the three **environment** settings whose absence produces a failure that
does not name its own cause:

| Set this | Or you get | Needed for |
|---|---|---|
| `vswhere` reachable — run from a **Developer Command Prompt for Visual Studio**, or put the VS Installer dir on `PATH` | **`MSB3073`** at the NativeAOT **link** step. The compile succeeds; the link cannot find the MSVC toolchain. | the `win-x64` publish |
| **`JAVA_HOME` → a JDK 21** (a JDK, not a JRE; not an older JDK) | Gradle fails the shell build | `gradlew` |
| **`ANDROID_NDK_ROOT`** → your installed NDK | the bionic cross-compile cannot run | the `linux-bionic-*` publishes |

> **Both Android RIDs are required, not just the one you plan to run on.** The shell's
> `verifyNativeAssets` gradle task demands **`linux-bionic-x64` *and* `linux-bionic-arm64`** before
> it will build an APK — deliberately, so you can never ship an APK with a missing or stale ABI. If
> you publish only `arm64` because "that's what real phones are", the build stops. It fails fast and
> prints the exact `dotnet publish` command for each missing `.so`, so the error is recoverable —
> but knowing it up front saves the round trip.
> (Task: `android/build.gradle.kts` → `verifyNativeAssets`.)

## 1. Install the template

```bash
dotnet new install BlazorNative.Templates
```

No version on that command — nuget.org is the only place that knows the current one. To pin:
`BlazorNative.Templates::<version>`.

**This step is new as of 0.4.0.** Before 0.4.0 the template pack had never been published, and this
command failed for everyone despite the docs promising it (friction **F0**). It was verified live
during this phase: a real install resolved `BlazorNative.Templates@0.4.0` from nuget.org.

## 2. Create the app

```bash
dotnet new blazornative -n MyApp
cd MyApp
```

You get the .NET app (pages, the `AppPages` manifest, the NativeAOT publish head) **and** a complete
Android/Gradle tree with the Kotlin shell already in it. The generated csproj references exactly
**three** packages directly — `BlazorNative.Runtime`, `.Components`, `.Analyzers` — at the template's
release-pinned version. `Renderer`, `Core`, `Http` and `Device` arrive **transitively** via `Runtime`;
you add nothing (friction **F1**). Confirm the closure yourself if you like:

```bash
dotnet restore
dotnet list package --include-transitive
```

## 3. Publish the native library — all three RIDs

The .NET app compiles to a **native library**; the Android shell loads it and renders its frames. The
APK is built from the publish output, so the publish always comes first.

```bash
dotnet publish . -c Release -r linux-bionic-x64      # emulator ABI      — REQUIRED by gradle
dotnet publish . -c Release -r linux-bionic-arm64    # real devices      — REQUIRED by gradle
dotnet publish . -c Release -r win-x64               # the loadable dll  — optional here
```

### What a clean publish looks like

**Exactly four `IL2072` warnings, and nothing else.** All four come from Blazor's own reflection
internals — `ComponentProperties` (×2), `CascadingParameterState`, `ComponentFactory` — never from
your pages. The template sets `TrimmerSingleWarn=false` on purpose so they appear individually
rather than collapsing into one aggregate; a **new** warning shape under NativeAOT is exactly the
thing you want to see.

This is the yardstick for the whole ecosystem story below: **4 IL2072, no more.** It held on all
three RIDs for the stock app, and it still held with **11 ZeroAlloc packages** added.

## 4. Build the APK

```bash
cd android
./gradlew assembleDebug      # → build/outputs/apk/debug/MyApp-debug.apk
# or
./gradlew installDebug       # build + install onto a running emulator/device
```

The APK carries **both** `.so`s (`arm64-v8a` + `x86_64`, renamed `libBlazorNative.Runtime.so` so
JNA's `Native.load` resolves) and the generated `res/raw/blazornative_routes.json`.

That routes resource is worth a look — `cat` it. It is the deep-link map, and it was **derived from
your pages at build time**, not written by anyone. On a stock app it reads:

```json
{"/":"BnStarterPage"}
```

## 5. Add a page — and watch the deep-link map follow

Add a `.razor` page, then **one row** in `AppPages.cs`:

```csharp
BlazorNativePage.Routed<BnAboutPage>("/about", "BnAboutPage"),   // route + deep link + mount name
BlazorNativePage.Named<SomeScreen>("SomeScreen"),                // mount name only, no route
```

Re-run the publish + gradle. `blazornative_routes.json` now contains the new row. **You edit no
shell file.** No Kotlin, no manifest, no map. This was verified on `bn-baseline` — a consumer app
built purely from published packages — which is the point: Phase 11.0's "the last hand-written
single-source-of-truth violation is gone" is confirmed *from outside the repo*, on the shipped
artifact, not just in our own sample.

## 6. Add your own services — the `ConfigureServices` seam (new in 0.4.0)

Until 0.4.0 the composition root was **sealed**: the framework built its `ServiceProvider` inside
`BlazorNative.Runtime` with no app hook, so an app author who wrote a service had nowhere to
register it and a page `[Inject]`ing it failed at mount. 0.4.0 opens the seam.

In `AppPages.cs`, beside the existing `RegisterPages` call:

```csharp
using Microsoft.Extensions.DependencyInjection;   // ← see the note below

public static void EnsureRegistered()
{
    if (s_registered)
        return;

    BlazorNativeApp.RegisterPages(All);

    BlazorNativeApp.ConfigureServices(services =>
    {
        services.AddSingleton<IGreetingService, GreetingService>();
        // ...or a ZeroAlloc.Inject-generated registration method:
        // services.AddMyAppServices();
    });

    s_registered = true;
}
```

Then inject it in a page exactly as you would a framework service:

```razor
@inject IGreetingService Greeting
<BnText Text="@Greeting.Greet("world")" />
```

**Three things that are true and worth knowing:**

- It is captured **once**, at the same `[ModuleInitializer]` that registers pages, and consumed
  **once** — after the framework's own registrations and immediately before the single
  `BuildServiceProvider()`. There is exactly **one** provider; this is not a second container.
- Because the app's delegate runs **last**, re-registering a framework contract is a deliberate
  last-write (MS.DI resolves the last registration), not an accident.
- Constructor dependencies work normally, including on framework services. In the showcase a
  4-deep chain resolved: a resilience proxy → a fetcher → a generated REST client → the
  framework's own bridge-backed `HttpClient`.

> **Footgun, fix in flight (F7):** the 0.4.0 template's `AppPages.cs` ships **without**
> `using Microsoft.Extensions.DependencyInjection;`, so your first attempt at the block above fails
> with **CS0246** on `IServiceCollection`. Add the `using`. **PR #165** adds it to the template
> along with a commented example, so this note expires.

## 7. Optional — layering the ZeroAlloc ecosystem

The showcase app added **11** ZeroAlloc packages on top and still published with exactly the 4
IL2072s. Two rules make that reproducible:

**(a) Three versions are mandatory pins, not suggestions.** BlazorNative itself already depends on
these; pulling a *different* version puts two `Microsoft.CodeAnalysis` versions into one compile and
the source generators diamond-conflict:

```bash
dotnet add package ZeroAlloc.Inject      --version 1.7.1
dotnet add package ZeroAlloc.Collections --version 1.1.2
dotnet add package ZeroAlloc.AsyncEvents --version 1.1.2
```

**(b) Reference the generator privately**, mirroring what the framework's own Renderer csproj does,
so it runs against *your* compile only and does not double-emit against the transitive copy:

```xml
<PackageReference Include="ZeroAlloc.Inject.Generator" Version="1.7.1"
                  PrivateAssets="all" IncludeAssets="analyzers;build" />
```

Both mitigations were verified: no `Microsoft.CodeAnalysis` diamond, and no duplicate-generator
emit.

**One package to skip: `ZeroAlloc.Cache` 1.1.15 does not compile** in this configuration (14 ×
CS1061 from its generated registration extension; its generator also emits proxies for interfaces
that never asked for caching). Upstream:
[ZeroAlloc.Cache#87](https://github.com/ZeroAlloc-Net/ZeroAlloc.Cache/issues/87).

**`ZeroAlloc.Serialisation` needs a `JsonSerializerContext`** in the same compilation or it stops
your build with **`ZASZ004`** — which is *good* (an AOT runtime failure turned into a compile error)
but undocumented in the package. The ~6 lines it wants:

```csharp
using System.Text.Json.Serialization;

[JsonSerializable(typeof(AppSettings))]
internal sealed partial class AppJsonContext : JsonSerializerContext;
```

**`ZeroAlloc.Validation` needs its generator package too** — `ZeroAlloc.Validation.Generator` — or
`[Validate]` silently generates nothing, with a green build. See the
[friction ledger](2026-07-21-phase-11.1-friction-ledger.md) Part 2 for the rest.

## 8. Keep two things

- **`<TrimmerRootAssembly Include="$(AssemblyName)" />`** in the csproj. Without it ILC trims your
  entire app: green build, zero trim warnings, failed mount at runtime.
- **`build/BionicNativeAot.targets`** and **`global.json`**. The first is the NDK cross-compile shim
  — there is no bionic publish without it. The second pins the SDK feature band the ILC host
  depends on.

## What this proves — and what it does not

### Proven

- A stranger with no access to this repository can install the published template, generate an app,
  publish NativeAOT for three RIDs, and build a dual-ABI Android APK, **entirely from nuget.org**.
- The publish is **trim-clean to the 4-IL2072 yardstick** — stock, and with 11 ZeroAlloc packages
  layered on.
- **Adding a routed page requires zero shell edits**; the Android deep-link map regenerates itself
  (verified on a consumer app, not just the in-repo sample).
- The **`ConfigureServices` seam works for a real consumer at runtime**, not merely at compile time:
  an ABI harness P/Invoked the *published* NativeAOT binary, replayed the shell's own call sequence
  (`blazornative_init` → `register_frame_callback` → `mount`), and the first frame's `ReplaceText`
  patches carried the **app service's own output** — including a count proving two services shared
  one singleton **instance**. All 6 pages mounted `rc = 0`.

### Not proven — read this before citing the above

- **UI events were never dispatched.** Pages **mount and render their initial frame**. Form submit,
  list add, camera capture, fetch button — all compile and are wired, none were fired at runtime.
  `OnInitialized` / `OnAfterRenderAsync` **did** run (that is how the app-service strings reached
  the frame), but nothing beyond first render was exercised.
- **No real device, and no capability actually fired.** Capability calls ran against the **DevHost
  bridge** (`PlatformKind.DevHost`). The APK was **built but not installed**. Real hardware and real
  camera/biometrics/geolocation/notifications are **Phase 11.2 / DoD #2** — a separate proof this
  document does not make and must not be read as making.
- **iOS was not walked.** Deferred by an explicit prior decision; iOS remains a manual,
  non-templated procedure with no template and no route generator.
- **`rc = 0` is not a health check.** A page with a bad parameter binding can throw, log to stderr,
  render partially, and still return `rc = 0` — and on Android that stderr is discarded. See
  friction **F8** / [#164](https://github.com/MarcelRoozekrans/BlazorNative/issues/164). Do not treat
  a zero return code as "the screen is correct".
