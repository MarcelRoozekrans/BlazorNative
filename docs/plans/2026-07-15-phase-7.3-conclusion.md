## What shipped

**The form is real on both shells, and the last stubbed widget is gone.** Four controls —
`BnCheckbox`, `BnSwitch`, `BnSlider`, and `BnPicker` (the NodeType-7 `Spinner` placeholder from
Phase 2.5, now a real `Spinner`/`UIPickerView`) — all authored in `.razor`, all two-way bound via
the `@bind-` pair convention, all on the **existing change wire**: no new event names, no new
export, no ABI change. The three new NodeTypes (`checkbox = 8`, `switch = 9`, `slider = 10`) are
a **wire-vocabulary extension** riding the existing int32 field of the 48-byte patch; the
vocabulary's three mirrors move in lockstep and are now pinned on all three surfaces — including
the Kotlin content-and-length pin Gate 1 found missing (Kotlin had pinned only byte offsets and
one synthetic decode; a missed vocabulary entry decoded to `"?"` and only a device golden could
see it, two layers from the bug).

The proof surface is **`/form`** (`BnFormDemo`, the seventh page): each control twice (bound +
disabled) in a `BnColumn`, an echo `BnText` re-rendered by every round-trip, the
`["Alpha","Bravo","Charlie"]` items literal through the strict parser into both shells' adapters.
Both device lanes drive it with real input: four round-trips = **exactly four** change dispatches
(the loop guards, end to end), and disabled controls dispatch **nothing** — asserted on the
dispatch counter under real touch (Android `dispatchTouchEvent`, never `performClick`, which
bypasses the enabled check by design; iOS on the property, because `sendActions` bypasses
`isEnabled` by design and UIKit's touch gate is the enforcement).

## The state-owner precedent (design decision 3, as proven)

The picker is the **first widget that owns state natively** — its items *and* its selection UI
live in the shell, unlike input, where the text lives in .NET. The precedent, as it now stands
proven:

- **`items` rides `UpdateProp` as strict flat-JSON** (the grammar below); `selectedIndex` rides
  `UpdateProp` as an invariant int string; selection comes **back on the existing change wire**
  with the new index as payload.
- **The normative clamp rule** (one prose home, `BnPicker.razor`'s header; mirrored verbatim by
  both shells): items empty → **-1**; non-empty → clamp into `[0, Count-1]`; shrink below the
  selection → **the last item**. Within a frame the shell applies `items` **before**
  `selectedIndex`, and **every items application re-clamps** the current selection against the
  new items — so a frame that shrinks the items and moves the selection lands identically on both
  shells regardless of which patch carried which prop (the clamp base falls back to the raw
  `requestedIndex`, so patch order is immaterial — pinned by test on iOS; Android's mechanism is
  code-identical, see the ledger).
- **The CLAMPED index is what's dispatched** — asserted on the dispatched payload, on both device
  lanes. This is the assertion that matters: .NET's benign inbound clamp makes a non-clamping
  shell *invisible* from the .NET side, so the wire is the only place the rule is provable. A
  pick during an item shrink never puts a stale index on the wire.

The scope boundary is recorded with it: the clamp-and-notify rule is **the picker's**, because a
picker's selection validity depends on data the picker owns (its items). A slider's range is the
author's declaration — an out-of-range `Value` is the author's bug, rides the wire raw, displays
natively clamped, and no notify or wire correction fires (recorded in `BnSlider.razor`).

## The checkbox divergence (design decision 2 — flagged for veto, not vetoed)

**`BnCheckbox` maps to `UISwitch` on iOS.** iOS has no native checkbox — UIKit's toggle is
`UISwitch` (React Native ships only `Switch` for the same reason), so `BnCheckbox` and `BnSwitch`
are semantically identical on iOS and visually distinct only on Android (`CheckBox` vs `Switch`).
This is a recorded **"renders natively, not identically"** divergence: same wire, same events,
same frames — different pixels. The alternative (a custom-drawn checkbox) buys pixel parity at
the cost of not being native — against the project's thesis.

## The per-control guard asymmetry — the phase's central empirical finding

The design said "verify the bind-loop guard per control, don't assume." Verification found the
platforms need **opposite guards** — and the controls differ *among themselves* on each platform:

| Control | Android | iOS |
|---|---|---|
| checkbox / switch | `applyingBatch` guard — `CompoundButton.setChecked` fires its listener **synchronously** (the 4.2 TextWatcher pattern) | **no guard** — `UISwitch.setOn` fires nothing (verified, not assumed) |
| slider | `applyingBatch` guard — `setProgress` fires synchronously; int progress dedups repeated samples structurally | **step-quantization dedup** at the dispatch site — float-native drags deliver a distinct `.valueChanged` per sample; without it one step dispatches many times |
| picker | **expected-selection compare** — `Spinner.setSelection` does NOT fire synchronously: AdapterView **posts** its notifier onto a later main-loop turn, when `applyingBatch` is long false; `appliedSelection` is recorded before every shell-side set and a callback reporting it is the echo of our own set | **same-row compare** — a wheel spun away and back is not a change (matching AdapterView); and the record-before-apply ordering **is structurally an expected-selection guard** (below) |

Two more Android findings the first device run forced into code:

- **`BnSpinner`, the self-layout fix** (the RN `ReactPicker` pattern, by name):
  `AbsSpinner.setSelection` applies a user pick's selection inside the **next framework layout
  pass** — which never comes in this shell (`BnYogaFrameLayout.onLayout` is deliberately empty;
  Yoga re-lays views only at patch commits). A stock `Spinner`'s pick sat pending forever — three
  controls echoed on the first run, the picker never did. `BnSpinner` answers its own
  `requestLayout` with a posted self measure+layout at the current Yoga frame. iOS must NOT copy
  this — `UIPickerView.selectRow` applies immediately.
- The picker guard tests run against an **attached** root: a detached view's posted notifier goes
  to a run-queue that only drains on window attach — i.e. never — so a detached-root picker test
  passes vacuously.

And the Gate 3 review's finding on the iOS side — an overclaim, corrected and made falsifiable:
the shell's comments claimed "no guard in this shell could swallow the evidence," but every
picker fires-nothing test moved the selection through the apply path, which records
`appliedSelection` **before** calling `selectRow`. If UIKit *did* fire `didSelectRow` on a
programmatic `selectRow`, the handler would see `row == appliedSelection` and swallow it — the
tests stayed green in that counterfactual, i.e. **record-before-apply IS, structurally, the
expected-selection guard Android's Spinner carries**. The fix is a discriminating test: raw
`selectRow(2)` on the view while `appliedSelection` is 0 — guard disarmed, a delegate fire would
dispatch `"2"` and redden. The fires-nothing claim is now evidence, not assumption.

## The two iOS 26 platform findings (found on CI, against the briefed diagnoses)

Both Gate 3 reds arrived with plausible diagnoses — a quantization/formatting bug, a
measure-mode mapping bug — and **both diagnoses were wrong**: the suspected code paths were
verified correct, and the platform's own widgets were doing something no earlier widget in either
shell had done. The recurring lesson, in its third telling this milestone-pair: *analysis says X,
construction disagrees* — run the construction.

1. **`UISlider` stores `value` as the normalized track fraction** `(value−min)/(max−min)` in
   Float32 and reconstructs `min + fraction·range` on read. `setValue(60)` on `0...100` stores
   `0.6f = 0.60000002…` and reads back `60.000004` — exactly one ulp up, the number in both
   failure messages. The dispatched payload was already the exact `"60.0"` (that assertion passed
   in the failing run — only the read-backs failed). Fix: **`BnSliderView` holds the last
   programmatic value exactly** and reports it verbatim; a real drag (`beginTracking`) clears the
   shim so live tracking stays the platform's own and the dispatch-site dedup keeps seeing fresh
   samples. Why no earlier control hit this: the slider is the shell's first float-carrying
   widget — `UITextField` stores `.text` verbatim, `UISwitch` stores a `Bool`.
2. **`UISwitch` refuses imposed frames** — it re-enforces its intrinsic size (63×39 on iOS 26)
   on every frame write, the **first widget in either shell to refuse the frame it is given**
   (`UILabel`/`UIButton`/`UITextField`/`UISlider`/`UIPickerView` all accept imposed frames, which
   is why 6.1's measured leaves never surfaced it). Yoga's arithmetic was right all along: Yoga
   3.2.1 imposes the available width on a StretchFit axis regardless of what a measure func
   returns (verified against the pinned tag's source), and Yoga computed 390 — the *view* read
   63. Per the layout-vs-intrinsic law, the fix asserts the stretch **where it lives**: the
   stretch LAW on the Yoga layout box (test-only `bnYogaFrame(of:)`), the view width by **oracle**
   (a fresh `UISwitch` handed the same stretched frame — the 6.3 method, platform-version-proof).
   The pin's job survives: a declared width smuggled onto the un-styled quartet still reddens the
   Yoga number, and Android's twin still asserts the view-level stretch its widgets honour.

## The strict items grammar — one acceptance set, three parsers

The Gate 1 review's mid-flight correction is why a dedicated parser exists: `FlatJson`'s reader
is object-only and **lenient** (whitespace, `\b`, `\f`, `\/`, uppercase hex all slide through) —
reusing it would have made the shells' acceptance sets drift on inputs the .NET writer never
emits, the exact failure one normative grammar exists to prevent. So the acceptance set is
**exactly `BnItemsJson.Write`'s image**, decided strict and made executable:

- **Three parsers, one set:** the reference parser `BnItemsStrictParser.cs` (the spec, ~100
  lines, deliberately NOT a JSON parser) → transcribed to `ItemsJson.kt` (shared source set, JVM
  lane) and `BnItemsJson.swift`, method for method.
- **15 normative malformed categories** (whitespace between tokens, trailing comma, prefix
  parses, `\b`-family escapes, `\u` for non-controls, the long `\u` spelling of a short-escape
  control, raw controls, non-array roots, …) plus — restored by the Gate 3 review — .NET's
  verbatim **backslash-u001F rejection vector on both shells** (uppercase hex of a *control*;
  built in code, never written literally — a decodable escape sitting raw in a source file is
  invisible to review): lowercase backslash-u001f is in the acceptance set, so **casing is the
  ONLY reason it rejects**. (The substituted backslash-u00AB vector had lost its discriminating
  power — it also trips the non-control rule, so a parser mutated to accept uppercase hex still
  passed.)
- **The boundary pins:** U+001F is the last char the writer escapes (lowercase-escape-only,
  one canonical spelling per input); **U+007F rides raw** — "control" provably means < U+0020 in
  all four writers and both shell parsers.
- Malformed input throws; the caller logs loudly and renders an **empty** picker rather than a
  wrong one.

## The Gate 1 red-first find: the un-guarded clamp notify hung Mount itself

The clamp's notify-on-move originally fired **every render** the raw value stayed out of range —
and the runaway needed **no cooperating parent at all**:
`EventCallback.Factory.Create(receiver, lambda)` infers the receiver from the delegate's target,
so any bound component is the implicit `IHandleEvent` receiver of its own lambda — every notify
auto-`StateHasChanged`s it, and **Mount itself spun forever** (observed: a 90s blame-hang inside
`Renderer.ProcessRenderQueue`; clamp → notify → re-render → same raw value → clamp → …). The fix
tracks the last-notified **(clamped, raw) pair**: a re-render carrying the same pair stays
silent, a new raw value or a new clamp result (items changed) notifies again, an in-range set
clears the tracker. Termination is now **by construction** — the header's "Terminates:" claim no
longer leans on the parent writing the clamp back. Mutation: the pre-fix guard shape reddens the
loop test `[2,2,2,2]` vs green `[2]`.

## The honest ledger (deferred with names, not dropped)

- **The Android reversed-order batch test** — iOS pins that `selectedIndex`-before-`items` in a
  single batch lands identically to the normative order (`requestedIndex` keeps the raw wire
  value; patch order is immaterial). Android's mechanism is **code-identical**; the instrumented
  twin was deliberately not re-run for it. Named so the asymmetry is a decision, not an oversight.
- **The step-quantization ceiling** — `n_max = round(range/step)` can emit a value above a
  declared max that is not a step multiple. The arithmetic is **identical on both shells** (one
  multiply, one add, integer n first), and Gate 1's recorded posture covers it: an out-of-range
  declaration is the author's bug. If a customer ever declares a non-multiple max and cares,
  the fix is one `min()` in two mirrored dispatch sites.
- **iOS `enabled` parse leniency** — `NSString.boolValue` accepts more spellings than Kotlin's
  `toBoolean()`. Pre-existing on `main` (not introduced this phase) and only reachable by
  hand-rolled wires — the .NET writer emits exactly `"true"`/`"false"`. Ledgered as a
  grammar-tightening candidate alongside any future prop-grammar pass.
