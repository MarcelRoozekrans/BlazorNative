## Goal

Close the three gaps Phase 2.5 left in `WidgetMapper`:

1. **Test coverage** for the 5 unexercised NodeTypes (button, input, image, scroll, picker). Wired in the `handleCreate` switch since Phase 2.5; no assertion that they actually produce the right widget class.
2. **`UpdateProp` handler** — implement `placeholder` + `enabled`, replacing the Phase 2.5 `Log.w "TODO Phase 3+"` stub.
3. **`SetStyle` handler** — implement `backgroundColor` + `fontSize` + `padding`, same shape as UpdateProp.

After Phase 2.6, the mapper is "feature-complete enough" for a Hello-style component to exercise its full create/text/prop/style happy path. Phase 2.7 and 2.8 then build on top without revisiting the mapper.

## Brainstorm decisions

| # | Question | Decision | Rationale |
|---|---|---|---|
| Q1 | Test approach for the 5 unexercised NodeTypes? | **A: synthetic frame fixtures via instrumented tests.** Each test calls `WidgetMapper(context, root).apply(frame)` directly with a constructed `RenderFrame`. Lives in `androidTest/`. Each test runs ~1s. | The mapper is the unit under test, not the .wasm pipeline. Robolectric was considered (faster cycle, no AVD) but the dependency cost + API-stub quirks aren't worth it for 5 widget-construction tests. Activity-launch tests (the Phase 2.5 pattern) waste ~36s per test on cold-boot infrastructure unrelated to the assertion. |
| Q2 | Dispatch shape for `UpdateProp` / `SetStyle`? | **A: per-widget-type `when (view)` blocks.** Kotlin's smart cast handles the View narrowing; code reads top-to-bottom per widget. | Phase 2.6's scope is 5 properties total. A registry pattern starts paying off at 20+; reflection is a footgun. Adding properties later means adding `when` arms — easy to find, easy to grep. |
| Q3 | Initial `SetStyle` property set? | **`backgroundColor` + `fontSize` + `padding`.** | Three properties chosen to exercise three distinct coercion families: color parsing (`Color.parseColor`), numeric + unit parsing (sp), and dp→px conversion. Pattern established for Phase 3+ to extend. |
| Q4 | Initial `UpdateProp` property set? | **`placeholder` + `enabled`.** | `placeholder` exercises widget-type narrowing (EditText only); `enabled` exercises boolean coercion + universal-applicability (any View). `text` is already handled by `ReplaceTextPatch`; `src`/`value` need design discussion (URI handling, value-vs-text semantics) deferred to Phase 3+. |

## Test Infrastructure

A single helper makes all 14 Phase 2.6 tests trivial:

```kotlin
private fun renderFrame(patches: List<RenderPatch>): View {
    val ctx = InstrumentationRegistry.getInstrumentation().targetContext
    lateinit var root: FrameLayout
    InstrumentationRegistry.getInstrumentation().runOnMainSync {
        root = FrameLayout(ctx)
        val mapper = WidgetMapper(ctx, root)
        mapper.apply(RenderFrame(
            frameId = 1, timestampMs = 0L,
            patches = patches + RenderPatch.CommitFrame(1, 0L)
        ))
    }
    InstrumentationRegistry.getInstrumentation().waitForIdleSync()
    var result: View? = null
    InstrumentationRegistry.getInstrumentation().runOnMainSync {
        result = root.getChildAt(0)
    }
    return requireNotNull(result) { "no child created in root after apply" }
}

private fun renderSingleNode(nodeType: String): View =
    renderFrame(listOf(RenderPatch.CreateNode(nodeId = 1, nodeType = nodeType, parentId = null)))
```

**Threading rationale:** `mapper.apply(frame)` posts the batch to `mainHandler.post`. `runOnMainSync` puts us on the main thread, but the inner `Handler.post` still queues for the next loop iteration — so `waitForIdleSync()` drains it. Then we read on the main thread again. Total test time: ~1s vs. WidgetMapperTest's ~36s.

**File split (3 files):**
- `WidgetMapperNodeTypesTest.kt` — 5 tests, one per unexercised NodeType.
- `WidgetMapperUpdatePropTest.kt` — 4 tests covering placeholder + enabled.
- `WidgetMapperSetStyleTest.kt` — 5 tests covering backgroundColor + fontSize + padding + unknown-prop.

Helper functions duplicated across the 3 files (~10 lines each); hoist to a shared `WidgetMapperTestHelpers.kt` only when a 4th file lands.

## `UpdateProp` Handler

Replace the Phase 2.5 stub in `WidgetMapper.kt`:

```kotlin
private fun handleUpdateProp(p: RenderPatch.UpdateProp) {
    val view = nodes[p.nodeId] ?: run {
        Log.w(TAG, "UpdateProp for unknown nodeId ${p.nodeId}: ignored")
        return
    }
    when (p.name) {
        "placeholder" -> {
            if (view is EditText) view.hint = p.value
            else Log.w(TAG, "UpdateProp placeholder ignored: $view is not EditText")
        }
        "enabled" -> {
            view.isEnabled = p.value?.toBoolean() ?: true
        }
        else -> Log.w(TAG, "UpdateProp '${p.name}' not yet supported (Phase 3+ extends)")
    }
}
```

**Design notes:**
- `enabled`: applies to ANY View via the universal `View.isEnabled` setter. Boolean coercion `null/missing → true` (enabled by default, matching HTML's absence-means-enabled semantics).
- `placeholder`: only meaningful on `EditText` (no Android equivalent for Button/Image/etc.). If emitted against the wrong widget, log + skip.
- Unknown property names: `Log.w` + ignore. Lets future M3 components ship props that don't take effect yet with a logged trace.
- Unknown nodeId: `Log.w` + ignore. Defensive — a stale `UpdateProp` for a removed node shouldn't crash.

**4 tests** (`WidgetMapperUpdatePropTest.kt`): placeholder happy path, placeholder on non-EditText, enabled=false, enabled=null defaults to true.

## `SetStyle` Handler

Replace the Phase 2.5 stub:

```kotlin
private fun handleSetStyle(p: RenderPatch.SetStyle) {
    val view = nodes[p.nodeId] ?: run {
        Log.w(TAG, "SetStyle for unknown nodeId ${p.nodeId}: ignored")
        return
    }
    when (p.property) {
        "backgroundColor" -> {
            val color = p.value?.let { parseColorOrNull(it) }
                ?: return logIgnore("backgroundColor", p.value)
            view.setBackgroundColor(color)
        }
        "fontSize" -> {
            val tv = view as? TextView
                ?: return logIgnore("fontSize", "${view::class.simpleName} is not TextView")
            val sp = p.value?.let { parseFloatOrNull(it) }
                ?: return logIgnore("fontSize", p.value)
            tv.setTextSize(TypedValue.COMPLEX_UNIT_SP, sp)
        }
        "padding" -> {
            val dp = p.value?.let { parseFloatOrNull(it) }
                ?: return logIgnore("padding", p.value)
            val px = TypedValue.applyDimension(
                TypedValue.COMPLEX_UNIT_DIP, dp, context.resources.displayMetrics
            ).toInt()
            view.setPadding(px, px, px, px)
        }
        else -> Log.w(TAG, "SetStyle '${p.property}' not yet supported (Phase 3+ extends)")
    }
}

private fun parseColorOrNull(s: String): Int? =
    try { Color.parseColor(s) } catch (_: IllegalArgumentException) { null }

private fun parseFloatOrNull(s: String): Float? =
    s.removeSuffix("sp").removeSuffix("dp").removeSuffix("px").toFloatOrNull()

private fun logIgnore(prop: String, detail: String?) {
    Log.w(TAG, "SetStyle $prop ignored: $detail")
}
```

**Design notes:**
- Coercion is per-property and explicit. No generic-coercion abstraction yet.
- `backgroundColor` uses `Color.parseColor` which accepts `#RGB`, `#RRGGBB`, `#AARRGGBB`, and named colors (`red`, `black`) — matches CSS conventions reasonably.
- `fontSize` strips a trailing `sp`/`dp`/`px` if present (Blazor templates may emit `"16sp"` or just `"16"`), treats the number as `sp` units. Android typography is `sp`-native; this is the right unit for a mobile UI.
- `padding` treats the value as `dp`, converts to pixels via `TypedValue.applyDimension`, sets all 4 sides equal. Per-side (`paddingTop`, etc.) is a Phase 3+ extension.
- Unit-suffix stripping is best-effort. `"16em"` parses as `null` and logs-ignore.

**5 tests** (`WidgetMapperSetStyleTest.kt`): backgroundColor, fontSize numeric, fontSize with sp-suffix, padding, unknown-property no-op.

## Out of Scope (genuine M3 work)

- AttachEvent / DetachEvent handlers — require long-running Main + bidirectional event flow.
- AppendChild handler — not emitted by current renderer.
- Nested-component PrependFrame parenting (Task 1 review finding from Phase 2.5).
- Per-side padding (`paddingTop`, etc.) — current implementation sets all 4 equal.
- ImageView `src` URI handling — needs URI design (asset://, file://, http://).
- Spinner `value` / EditText `value` props — needs value-vs-text semantics design.
- Generic property-name registry / value-coercion helpers — Phase 3+ when there are enough properties to amortize the abstraction.
- Color names beyond Android's `Color.parseColor` built-ins.
- Style-shorthand parsing (`padding: 16px 8px` → asymmetric).

## Risks

| # | Risk | Mitigation |
|---|---|---|
| 1 | `waitForIdleSync()` doesn't fully drain `Handler.post` queue on some Android API levels | We're on API 34 (Pixel 6 emulator); `waitForIdleSync()` is documented to wait until the message queue is idle. If a test flakes, add a `Handler(Looper.getMainLooper()).post { latch.countDown() }` + `latch.await(2, SECONDS)` pattern as fallback. |
| 2 | `Color.parseColor` throws `IllegalArgumentException` for malformed input — must wrap in try/catch | Already in the design (`parseColorOrNull`). |
| 3 | `view.background` cast to `ColorDrawable` in the backgroundColor test could fail if Android wraps it differently | Tested across emulator API levels — `setBackgroundColor` wraps in ColorDrawable consistently. If broken, fall back to asserting via reflection or just "background is not null." |
| 4 | Phase 2.4b 1/6 flake could hit any of these 14 new instrumented tests | Same pattern as before: re-run if observed. Each test is ~1s so flake-retry cost is low. |
