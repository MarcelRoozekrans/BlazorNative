## 1. Fix designs

### Focus/blur (Kotlin + a small Components carrier)

The .NET side is already complete: `ProcessAttribute` emits `AttachEventPatch` generically
for any `on*` attribute and `BuildEventArgs` maps `"focus"`/`"blur"` → `FocusEventArgs`.

- **`WidgetMapper.kt`**: `handleAttachEvent` gains a `"focus"`/`"blur"` arm using
  `setOnFocusChangeListener`. Android has ONE focus listener per view firing both
  directions, so the mapper tracks a per-view pair (focusHandlerId?, blurHandlerId?) —
  keyed like the watcher map — and the single listener dispatches whichever side is
  attached. Detach clears the pair entry (and the listener when both sides are gone).
- **Carrier for tests**: `BnInput` gains optional `OnFocus`/`OnBlur`
  `EventCallback<FocusEventArgs>` parameters — attach emitted **only when set**
  (`HasDelegate`), so BnDemo's canonical golden (4 attaches) is untouched. New internal
  scaffolding component **`FocusProbe`** (Runtime, registered in the mount registry like
  CompositionProbe): a `BnInput` with OnFocus/OnBlur wired to a state echo `BnText`
  ("focused"/"blurred"), ledgered as scaffolding in the M4 audit.
- Tests: .NET (mount FocusProbe, dispatch focus/blur via `DispatchEventCore`, echo
  nodeId-pinned); JVM (through the dll: harvest handlerIds from AttachEvent patches,
  `dispatchEventBlocking("focus"/"blur")`, echo ReplaceText pinned); Android instrumented
  (EXTRA_COMPONENT=FocusProbe, `requestFocus()`/`clearFocus()` on the real EditText →
  echo TextView updates — proves the listener arm, not just the wire).

### Stale-watcher re-attach (Kotlin)

`watchers` re-keys from handlerId to **nodeId**; the `"change"` attach arm removes any
prior watcher for that node before adding (`removeTextChangedListener`), detach keys by
node. Behavior change: re-attach can no longer stack watchers → exactly one dispatch per
text change, always the live handlerId. Instrumented pin extends the existing
re-entrancy coverage: force a change re-attach (re-render that swaps the handler), type,
assert exactly one dispatch with the live id (today: one per stacked watcher).

### RouteChanged subscriber isolation (.NET)

`RouteChanged?.Invoke(route)` → `GetInvocationList()` + per-subscriber try/catch
(the established `DevHostBridge.RaiseNativeEvent` pattern), logging faults to stderr.
**Strict-mode posture (the decision the DoD asks for):** subscriber faults are
*contained* (navigation succeeded — the screen swapped; a listener's bug must not
convert success into rc 2) but each fault is stderr-logged; under `StrictErrors` they
are still contained (documented: strict surfaces *renderer* contract violations, not
app-listener bugs — same posture as the dispatch capture window treating handler
exceptions as rc 2 only when they fault the dispatch itself). Tests: two subscribers,
first throws — second still runs, CurrentRoute updated, swap reports success (rc 0);
plus the strict-mode variant.

### Allocation-budget test (.NET)

Enable `RenderWalk_IsAllocationFree_OnSteadyState`: expose an **internal test seam**
`NativeRenderer.TriggerRootRender(componentId)` (calls the root's `StateHasChanged`
equivalent via the component state — internal + `InternalsVisibleTo`, documented as
test-only like `ReplaceRegistryEntryForTests`). The test mounts, warms with N renders,
then measures `GC.GetAllocatedBytesForCurrentThread()` across M steady-state re-renders.
**Honest budget:** the walk allocates per-frame today by design (`AsSpan().ToArray()` at
`NativeRenderer.cs:322` + `RenderFrame` wrapper) — a literal zero/512 B bound is false.
The test pins a **regression bound** (measured baseline + slack, sibling-test style:
`FrameArenaTests` uses 100 KB/1000 iterations) and the conclusion records the measured
number + the `.ToArray()` allocation as a known cost (pooled frame payload = M6+
ecosystem work if ever needed). If the seam proves larger than expected → RED-on-ambiguity.
