### 3. Focus/blur unwired — FIXED

- **Was:** `WidgetMapper.kt`'s attach/detach arms else-skipped `"focus"`/`"blur"`
  (the .NET side was already complete: `ProcessAttribute` emits
  `AttachEventPatch` for any `on*`; `BuildEventArgs` maps `"focus"`/`"blur"` →
  `FocusEventArgs`). No component carried the parameters.
- **Now:** `src/BlazorNative.Components/BnInput.cs:47-60` — optional
  `OnFocus`/`OnBlur` `EventCallback<FocusEventArgs>`, attach emitted only when
  `HasDelegate` (BnDemo's canonical golden — 4 attaches — unchanged).
  `src/BlazorNative.Runtime/FocusProbe.cs` — scaffolding proof app (registry
  `"FocusProbe"`; see the scaffolding ledger in the
  [conclusion](2026-07-11-phase-4.2-conclusion.md)).
  `WidgetMapper.kt:82-97,196-213,246-260` — per-view `FocusEntry` pair behind a
  single `setOnFocusChangeListener` (Android has ONE focus-listener slot per
  view firing both directions); detach clears a side, the listener goes when
  both sides are gone.
- **Tests:** .NET `FocusProbeTests` (mount shape: exactly focus+blur+change;
  dispatch focus/blur → echo "focused"/"blurred" nodeId-pinned); JVM
  `FocusBlurTest.kt` (same pins through the published dll); Android
  `FocusBlurAndroidTest.kt` (`focus_blur_round_trip_on_screen` — real
  `requestFocus()`/park-focus on the EditText → echo TextView;
  `focus_detach_sides_and_listener_removal` — the single-listener semantics).
- **Commits:** `1d0158f` (.NET carriers) · `02939bd` (JVM) · `3b00f56` (Kotlin arms).

### 7. RouteChanged subscriber isolation — FIXED

- **Was:** `RouteChanged?.Invoke(route)` — a throwing subscriber faulted the
  swap unit AFTER the screen swapped (deferred path → export rc 2 despite the
  navigation succeeding) and starved later subscribers. Documented POC posture.
- **Now:** `src/BlazorNative.Runtime/NativeNavigationManager.cs:104-132`
  (`RaiseRouteChanged`) — `GetInvocationList()` + per-subscriber try/catch (the
  `DevHostBridge.RaiseNativeEvent` pattern), faults stderr-logged.
  **The strict-posture decision the DoD asked for:** faults are contained under
  `StrictErrors` too — strict surfaces *renderer contract violations*, not
  app-listener bugs (the same line the dispatch capture window draws: handler
  exceptions are rc 2 only when they fault the dispatch itself; a listener
  firing after a successful swap is not that).
- **Tests:** `NavigationTests.RouteChanged_ThrowingSubscriber_SecondStillRuns_SwapReportsSuccess`
  (non-strict: rc 0, second subscriber runs, CurrentRoute tracks the screen) +
  `RouteChanged_ThrowingSubscriber_StrictMode_StillContained_AndLogged`
  (strict: contained AND the fault text asserted on stderr).
- **Commit:** `1d0158f`.

### 4. Stale-watcher re-attach — FIXED

- **Was:** `WidgetMapper.kt` keyed `watchers` by handlerId, so a change
  re-attach (re-render swapping the handler) STACKED TextWatchers — one text
  change dispatched once per stacked watcher, stale ids included.
- **Now:** `WidgetMapper.kt:70-78,176-192,241-244` — `watchers` re-keyed
  handlerId → **nodeId**; the change attach arm removes the node's prior
  watcher before adding (last wins), detach keys by node. Exactly one dispatch
  per text change, always the live handlerId; the rc-0 at-most-once stale
  contract is unchanged.
- **Tests:** `EventRoundTripAndroidTest.change_reattach_dispatches_exactly_once_with_live_handler`
  (genuinely discriminates pre-fix behavior: forces a handler-swapping
  re-attach, types, asserts exactly ONE dispatch with the live id — pre-fix
  this saw one per stacked watcher).
- **Commit:** `3b00f56`.

## Re-ledgered (5) — rationale + revisit trigger

### 1. Async-handler capture window — RE-LEDGERED

- **State:** `src/BlazorNative.Renderer/NativeRenderer.cs:950-953` (the
  `_uiEventDispatchException` doc): the dispatch capture window's guarantees
  assume SYNCHRONOUS handlers; an `await` in an event handler moves
  continuations off the dispatch thread and out of the window — a late fault
  is stderr-logged but cannot fault the export's return code.
- **Rationale:** size L, entangled with the synchronous-dispatch contract
  pinned at three layers (`Exports.DispatchEventCore` sync contract, the
  InlineDispatcher, Kotlin's `dispatchEventBlocking`). No consumer writes
  async handlers today — every Bn* component and probe uses sync handlers.
  Redesigning the window without a real async consumer risks inventing the
  wrong shape.
- **Revisit trigger:** the first real async `@onclick` consumer (an app or
  Bn* component that must await inside a handler). That work owns: capture
  continuity across continuations, completion signaling to the host, and
  item 2's async-offload story — they are the same design.

### 2. Dispatch-lane starvation — RE-LEDGERED

- **State:** `src/BlazorNative.Jni/src/main/kotlin/io/blazornative/jni/BlazorNativeRuntime.kt:59-81`
  (`dispatchLane` KDoc): the single `BlazorNative-Dispatch` lane serializes all
  post-boot .NET entry; a slow handler blocks later events.
- **Rationale:** the single lane IS the design — renderer affinity (the
  renderer's instance fields assume single-threaded post-boot access) and
  StrictMode hygiene both depend on it. The real fix for slow handlers is not
  more lanes, it is item 1's async-offload (handler awaits, lane frees). A
  second lane would buy races, not throughput.
- **Revisit trigger:** item 1's revisit trigger fires (they are one design),
  OR measured on-device jank attributable to a specific slow handler blocking
  the lane (none observed through M4's instrumented suites).

### 5. RemoveComponent bucket scan — RE-LEDGERED

- **State:** `src/BlazorNative.Renderer/NativeWidgetTree.cs:360-364`:
  `RemoveComponent` scans ALL live slot-list buckets
  (`Keys.Where(k => k.ComponentId == componentId)`) — O(total buckets) per
  disposal, quadratic-ish when N disposals × B buckets in one frame.
- **Rationale:** unmeasured, and POC-scale fine — the demo apps dispose a
  handful of buckets per navigation/keyed-list mutation; the Phase 4.2
  allocation baseline (328 B/frame steady-state) shows no disposal pressure.
  Indexing buckets by componentId is mechanical when justified by a number.
- **Revisit trigger:** a keyed Bn* list benchmark showing measurable disposal
  cost (e.g. a list teardown frame where the bucket scan is visible in a
  profile, or >1 ms disposal time at realistic list sizes). Likely surfaces
  with M6's real component-library work if ever.

### 6. TranslateToViewIndex memoization — RE-LEDGERED

- **State:** `src/BlazorNative.Renderer/NativeWidgetTree.cs:171-175`:
  O(subtree slots) per mid-list insert; its own doc already says "memoize
  per-diff if Bn* list sizes ever make this measurable".
- **Rationale:** premature optimization by its own KDoc — called once per
  MID-LIST insert only (appends bypass it), and the expansion is bounded by
  component-subtree size. No measurement has ever shown it.
- **Revisit trigger:** a profile of a mid-list-insert-heavy frame (large keyed
  list, insert-at-front churn) where this walk is measurable. Same benchmark
  family as item 5 — evaluate the two together when that benchmark exists.

### 9. NativeEvents redesign — RE-LEDGERED → M5

- **State:** `src/BlazorNative.Runtime/NativeShellBridge.cs:304-308`:
  `IMobileBridge.NativeEvents` is a documented no-op stub on the production
  bridge — subscribing is legal, nothing ever fires.
- **Rationale:** a FEATURE FORK (native lifecycle ingress: app pause/resume,
  back button, deep links — host-INITIATED events), not a bug in existing
  behavior. UI events took the `blazornative_dispatch_event` path instead
  (3.2). Deleting the surface would orphan the just-hardened BN0014 analyzer
  rule (which guards the surviving sync-handler contract) and Home.razor's
  DevHost subscription (where DevHostBridge DOES fire events — the dev-host
  path is real).
- **Revisit trigger:** M5 host-initiated lifecycle events (the milestone that
  owns back button / deep links / lifecycle per MILESTONE.md's out-of-scope
  list). That design decides: typed event ingress export vs. the existing
  dispatch lane, and the DevHost/production parity story.
