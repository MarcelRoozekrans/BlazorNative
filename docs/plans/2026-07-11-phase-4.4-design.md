## 1. InspectorHost (Kotlin, main/kotlin ‚Äî PreviewHost's sibling)

`main(args)`: component (default `BnDemo`), port (default `5199`). Boot
`BlazorNativeRuntime(onFrame = ...)`, mount, then serve until Ctrl+C:

- **State**: the 4.3 `TreeSnapshot` (tree of record) + a bounded **patch ring buffer**
  (seq, timestamp, patch summary ‚Äî last ~500) + a bounded **event log** (dispatches with
  rc, frame deliveries, onError faults).
- **Threading contract (the risky part):** frame callbacks arrive on the dispatch lane;
  HTTP handlers run on server threads. One lock guards snapshot+ring+log ‚Äî writers
  (frame callback, dispatch results) and readers (JSON renders) both take it; renders
  are quick (string building). `dispatchEvent` already marshals through the single lane ‚Äî
  the POST handler calls the existing blocking dispatch and returns its rc. Documented
  against the `BlazorNativeRuntime` threading KDoc.
- **Endpoints:**
  - `GET /` ‚Äî the inspector page (one self-contained HTML string; vanilla JS;
    `<details>`-based collapsible tree; patch tail; event log; dispatch buttons).
  - `GET /api/tree` ‚Äî TreeSnapshot as JSON (id/type/text/props/styles/events/children).
  - `GET /api/patches?since=seq` ‚Äî ring buffer tail.
  - `GET /api/events` ‚Äî event log.
  - `GET /sse` ‚Äî SSE: pushes `tree-changed` (with seq) + `event-logged` notifications;
    per-client queue, slow clients dropped (documented). The page re-fetches
    `/api/tree` on `tree-changed` (pull model keeps SSE payloads trivial).
  - `POST /api/dispatch` ‚Äî `{handlerId, eventName, payload?}` ‚Üí the blocking dispatch ‚Üí
    `{rc}`; logged. Malformed ‚Üí 400. Unknown handler rides the runtime's own rc contract.
- **JSON**: a small hand-rolled writer (escape per JSON spec) ‚Äî the Kotlin twin of the
  FlatJson posture; tests pin escaping.
- **Gradle**: `runInspectorHost` JavaExec (the `runPreviewHost` shape + `-Pport=`).
  Makefile: `inspect` target. Session is fast-restart like everything else (restart to
  pick up a new dll ‚Äî the 4.3 constraint, restated in the page footer honestly).

## 2. Tests (JVM ‚Äî the inspector is testable end-to-end without a browser)

- JSON writer escaping pins; TreeSnapshot‚ÜíJSON shape pins (reuse synthetic-frame
  patterns from `TreeSnapshotTest`).
- Ring buffer/log bounding pins.
- **End-to-end HTTP test** (the load-bearing one): start InspectorHost's server on an
  ephemeral port against a real booted session (harness like `BnDemoTest`), then:
  `GET /api/tree` ‚Üí BnDemo shape (input/buttons present); `POST /api/dispatch` with the
  Settings button's handlerId (harvested from the tree JSON) ‚Üí rc 0 ‚Üí `GET /api/tree`
  shows the settings page (the 3.5 navigation as the proof!); event log contains the
  dispatch; SSE delivers a `tree-changed` within a timeout. Single-session constraints
  respected (one JVM, sequential).

## Risks

| Risk | Mitigation |
|---|---|
| SSE with com.sun.net.httpserver (chunked keep-alive) is fiddly | Pull-model SSE (notify + re-fetch) keeps payloads trivial; the e2e test pins one delivery; slow-client drop documented |
| Lock contention/deadlock between dispatch lane and HTTP threads | One coarse lock, no nested lock acquisition, dispatch POST does NOT hold the state lock while dispatching (lock only for the log write after) |
| Page JS scope creep | Vanilla JS, one file, no build step; collapsible = `<details>`; the DoD asks for visible, not beautiful |
| Port collisions | `-Pport=` override + bind-failure message names the flag |
