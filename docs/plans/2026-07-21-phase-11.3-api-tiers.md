# BlazorNative — public API tier table

**Phase 11.3, Gate A.** Every public type a consumer can reach across the seven shipped
packages, in **exactly one** tier, with a reason.

This file stands alone. It does not require the Phase 11.3 design document, and it deliberately
predates the `PublicAPI.*.txt` baselines (Gate B) — a baseline generated before this review would
acquire authority it has not earned.

---

## 0. The tiers

| Tier | Meaning | Consequence today (`0.x`) | Consequence at 1.0 |
|---|---|---|---|
| **STABLE** | The surface we intend to freeze. A consumer is *meant* to bind to it. | A break must be deliberate, acknowledged in the baseline diff, and carry a changelog line. | Breaking it is a **major**-version conversation. |
| **PROVISIONAL** | Real consumer surface, but its shape is expected to move. Baselined so changes are visible; **not** advertised as stable. | Same as STABLE. | May change in a **minor** with a changelog line. |
| **NOT-API** | Public for a mechanical reason — cross-assembly composition, the C ABI, or a source generator. Binding to it is unsupported. | Baselined (an assembly's public surface must be recorded honestly), never advertised. | May change in a **minor**. Marked `[EditorBrowsable(Never)]` at Gate C. |

**NOT-API is not a synonym for unfinished.** Most of it is public *by mechanical necessity* and has
not moved in five milestones. That is why the mark for it is `[EditorBrowsable(Never)]` + xmldoc
and **not** `[Experimental]` — "this may change" would be the wrong claim about `NativeRenderer`;
"this is not for you" is the right one.

## 1. Method — how these counts were derived

Two independent enumerations, reconciled. **Reflection is the source of truth**, because
`GetExportedTypes()` answers exactly the question the tier table asks: *what can a consumer reach?*

1. **Reflection (authoritative).** `dotnet build BlazorNative.sln -c Release`, then each
   `src/BlazorNative.<Pkg>/bin/Release/**/BlazorNative.<Pkg>.dll` loaded into a
   `MetadataLoadContext` and enumerated with `GetExportedTypes()`. `BlazorNative.Analyzers`
   (`netstandard2.0`) enumerates fine this way — it is only *runtime* loading that fails, which is
   why `PackagePurityTests` reads it off the PE.
   `[Parameter]` counted as public instance properties carrying `ParameterAttribute`,
   `DeclaredOnly`, over `BlazorNative.Components.dll` — i.e. **after** the Razor generator has run,
   so `.razor`-declared parameters are included by construction rather than by hope.
2. **Grep (cross-check).** `^\s*public (sealed|abstract|static|partial|readonly|unsafe)* (class|interface|record|record struct|struct|enum|delegate) `
   over `src/**/*.cs` + `src/**/*.razor`, and `\[Parameter` (**not** `\[Parameter\]`) over
   `src/BlazorNative.Components`.

### The reconciliation — and why grep is wrong in both directions

| | Grep (source declarations) | Reflection (exported) | Δ |
|---|---:|---:|---:|
| `BlazorNative.Components` | 25 | **26** | +1 |
| `BlazorNative.Core` | 21 | **21** | — |
| `BlazorNative.Runtime` | 14 | **14** | — |
| `BlazorNative.Renderer` | 16 | **15** | −1 |
| `BlazorNative.Device` | 6 | **6** | — |

| `BlazorNative.Analyzers` | 3 | **3** | — |
| **Total** | **87** | **88** | **+1** |

Grep **over**-counts by 2 and **under**-counts by 3. Both errors are structural, not sloppy:

- **−2 · `public` nested in `internal`.** `BnArrayRange<T>.Enumerator`
  (`src/BlazorNative.Renderer/BlazorInterop.cs:153`) and `BnArrayBuilderSegment<T>.Enumerator`
  (`:178`) are `public struct` inside `internal struct` (`:142`, `:167`). Reflection confirms
  neither is exported. A `grep '^public'` baseline would have invented two public types.
- **+3 · types no `.cs` file declares.** All three are generated and all three are genuinely
  exported:
  - `Microsoft.Extensions.DependencyInjection.BlazorNativeRendererServicesServiceCollectionExtensions`
    — emitted by the ZeroAlloc.Inject generator; the target of
    `services.AddBlazorNativeRendererServices()` (`src/BlazorNative.Renderer/RendererServices.cs:21`,
    `src/BlazorNative.Runtime/HostSession.cs:333`).
  - `Microsoft.Extensions.DependencyInjection.BlazorNativeHttpServicesServiceCollectionExtensions`
    — same generator; called at `src/BlazorNative.Http/HttpServices.cs:31`.
  - `BlazorNative.Components._Imports` — the Razor compiler's class for
    `src/BlazorNative.Components/_Imports.razor`.

**Verdict: 88 effective public types, not 85.** Anyone working from a source grep will get 87 and
will get the wrong 87.

## 4. The contentious calls, decided

### 4.1 `NativeRenderer` (+ the 11 renderer records) = **NOT-API**

**Decision: NOT-API. Marked, not moved, in this phase.**

The evidence is structural, and it was re-verified against the code rather than repeated:

- **Its consumer is an `internal` type in a different assembly.**
  `internal static unsafe class HostSession` (`src/BlazorNative.Runtime/HostSession.cs:31`) holds
  `private static NativeRenderer? s_renderer` (`:34`), sets `renderer.StrictErrors` (`:127`),
  calls `renderer.RunAfterDispatch` (`:277`) and keys its registry on
  `Func<NativeRenderer, int>` (`:63`). `NativeRenderer` is public **so that an internal type in
  another assembly can drive it** — the textbook shape of accidental publicity. Verified, not
  assumed.
- **Its surface is renderer internals in the open:** `FrameSink`, `StrictErrors`,
  `RunAfterDispatch`, `DispatchUiEventAsync`, the `Frames` event, and an overridden `Dispatcher`
  returning a `private sealed class InlineDispatcher`. There is no scenario in which an app author
  sets `StrictErrors`.
- **The one place a consumer's code meets the type is already `internal`:**
  `BlazorNativePage.Routed<T>`'s mount thunk is `Func<NativeRenderer, int>`, and it is `internal`
  (`BlazorNativeApp.cs:46`). The layering comment at `BlazorNativeApp.cs:80`–`:84` explains that
  this thunk's type is the whole reason `BlazorNativeApp` lives in Runtime rather than Core.

**The correct fix is `internal` + `InternalsVisibleTo`, and it is a breaking change this phase
must not make.** Recorded as 1.0 criterion S3, classed DOCUMENTED: the marking is sufficient, and
if the move happens at all it is better spent inside `0.x`. **Marking beats moving in a
docs-and-pins phase** — but the residual risk is stated plainly: a consumer *can* bind to
`NativeRenderer` today, and a tier table does not stop them. Gate C's
`[EditorBrowsable(Never)]` removes it from IntelliSense, which is as far as a non-breaking change
reaches.

### 4.2 `DevHostBridge` = **PROVISIONAL**

**Decision: PROVISIONAL — baselined so changes are visible, excluded from the advertised stable
core.**

It is public for a good reason (tests and dev harnesses across assemblies construct it, and it is
genuinely useful to a consumer writing tests) and it is the most-used non-shipping type in the
repo. But its ~70 public members are **mock-shaped**: seeding hooks and canned results, not a
contract designed for strangers. Freezing it at 1.0 freezes a test double at whatever shape the
last capability happened to leave it in.

The decisive argument is forward-looking: `DevHostBridge` is the **natural home of the consumer
test harness** ([#25](https://github.com/MarcelRoozekrans/BlazorNative/issues/25)). That work will
reshape it — that is the point of it. Freezing the shape now would either block #25 or force it to
ship beside a frozen predecessor. PROVISIONAL says exactly the true thing: *use it, we will tell
you when it moves, we are not promising it will not.*

### 4.3 `INavigationManager`'s location — **stays in `BlazorNative.Core`**

**Decision: the type is STABLE; its location is `BlazorNative.Core` and is not moving before 1.0.**

The design flagged this as a "PROVISIONAL location", which would have put one type in two tiers.
Resolved: **the tier is a property of the type, not of its assembly name.** `INavigationManager` is
STABLE. Its location is a separate, recorded fact:

- The lift to a dedicated `BlazorNative.Navigation` package is
  [#23](https://github.com/MarcelRoozekrans/BlazorNative/issues/23), explicitly scoped out of M11
  as *growth, not readiness* (`docs/planning/MILESTONE.md:153`).
- **A package move is not a source break if the old package ships a `[TypeForwardedTo]`.** That is
  recorded as 1.0 criterion S4 and written into the interface's own `<remarks>`, so the mitigation
  travels with the type rather than living only in a plan.

**The stale comment is fixed.** `src/BlazorNative.Core/INavigationManager.cs:7` read *"M6 lifts it
into a dedicated BlazorNative.Navigation package."* M6 closed five milestones ago and the lift did
not happen — a reader of the shipped source was being told a relocation already occurred, in the
header of a type this very phase classifies. It now records what actually happened and points at
#23 and at this file.

### 4.4 A4 — the interface-additions policy: **consume-only, written into the xmldoc**

**Decision: policy (a) — `IMobileBridge` and `INavigationManager` are consume-only contracts.
Adding a member is non-breaking by declaration. Implementing them outside BlazorNative is
unsupported.** Written into the `<remarks>` of both interfaces (`src/BlazorNative.Core/IMobileBridge.cs`,
`src/BlazorNative.Core/INavigationManager.cs`) — this is criterion **A4**, a 1.0 **BLOCKER** owned
by Gate A, and it is closed by that edit rather than by this paragraph.

Why (a) and not (b) — `virtual` default implementations:

- **The asymmetry is real and it favours callers.** Adding a member to an interface is invisible to
  every caller and fatal to every implementer. The supported consumers here are all callers: the
  two real implementations are `NativeShellBridge` (Runtime) and `DevHostBridge` (Core), both ours.
- **`IMobileBridge` grows once per capability**, by construction — five capabilities have added to
  it so far and #18's haptics/purchases/background-tasks will add more. A policy that makes each
  addition a major bump would make the framework unable to grow; a policy that makes it free for
  callers costs nothing anyone is paying.
- **(b) would put method bodies in the contracts package**, whose entire purpose is to hold none.
- **A consumer who needs a test double has better options**, and the xmldoc names them: the
  `BlazorNative.Device` façades (2–5 members each, designed to be mocked) or `DevHostBridge`
  itself. Nobody should be hand-implementing 27 members.

**The condition that makes the policy honest: it had to be written before the first post-1.0
addition, not after someone had already implemented the interface.** That is why A4 is a Gate A
blocker and not a Gate D tidy-up.

## 5. Two notes the tiers do not capture

- **Enum additions are source-compatible and behaviourally breaking.** Adding a member to
  `GeolocationStatus` compiles fine for a consumer and silently falls through their exhaustive
  `switch`. A `PublicAPI.*.txt` baseline records enum *members*, so Gate B will make such an
  addition **visible** — which is right. The policy for the visible line: **adding a status member
  is a minor-version change with a changelog line, not a free edit.**
- **A baseline records the surface; it does not judge it.** `PublicAPI.Shipped.txt` will happily
  freeze a badly-named parameter forever. This is the strongest reason Gate A precedes Gate B: if
  this review surfaces a rename worth doing, `0.x` is the only time it is cheap, and it must land
  **before** the baseline, not after. This review surfaced none — the 55 STABLE types read as
  intended surface — which is itself the reviewed finding, not an omission.

## 6. What Gates B–D inherit from this table

- **B** — six `PublicAPI.Shipped.txt` / `.Unshipped.txt` pairs (all packages except Analyzers,
  §3.7), `BnEnforcePublicApi` gating `RS0016`/`RS0017`/`RS0037` as **errors** in
  `src/Directory.Build.targets` (**not** `.props`), plus the analyzer diagnostic-ID roster pin.
  Proven by mutation — rename `BnButton.Label` → `BnButton.Text`, expect `error RS0016` **and**
  `error RS0017`, exit 1 — never by a green build. Plus the three falsifiable checks in §1.
- **C** — `[EditorBrowsable(EditorBrowsableState.Never)]` + an xmldoc *"Not part of the supported
  public API"* line on all **31** NOT-API types; the `[Experimental]` policy written down with
  `BN1xxx` reserved and the finding recorded that the current surface warrants **zero** uses.
- **D** — the README's stable-core paragraph (naming the tiers, stating **no** number, since
  `ReadmeDriftTests` pins every count the README prints), this table published as a docs-site
  API-stability page, and the compat statement.
