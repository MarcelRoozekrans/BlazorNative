# Phase 5.1 Design — Host-Initiated Events (Lifecycle + Back + Deep Links)

**Date:** 2026-07-12
**Milestone:** M5 — P4: Full Platform Coverage (closes DoD #5)
**Predecessor:** Phase 5.0 GREEN (PR #47, merge `16a637a`) — iOS feasibility retired
**Owns:** the `NativeEvents` redesign the 4.2 triage routed here — a real host→.NET
event ingress, Android lifecycle wired into it, predictive back → navigation-back,
deep link → startup route. Runs independently of the iOS shell work.

---

## Decisions locked (2026-07-12 brainstorm, survey-grounded)

| Decision | Choice |
|---|---|
| Ingress | **9th C-ABI export `blazornative_host_event(name, payload)`** → fires the real `NativeShellBridge.NativeEvents` (replacing the no-op stub), mirroring `DevHostBridge.RaiseNativeEvent`. JVM-testable like `dispatch_event`. The ABI grows 8→9 — done NOW, before the Swift shell (5.2/5.3), is cheaper than after two shells exist. |
| Back | **Previous-route slot** in the nav manager: it remembers the prior route (`from→to`); `NavigateBack` swaps `to→from`. At root (no prior), back returns a "not handled" signal and the shell falls through to default Android behavior (finish). Extends to a stack later if needed. |
| Deep link | **Launch-time only**: a `VIEW` intent-filter + `intent.data` parsed at `onCreate` feeds the **existing** `QueryStartupRoute`/`GetCurrentRoute` channel — the app starts on the linked route via the 3.5 startup-route-honor mechanism. `onNewIntent` (already-running) is out of scope. |

## 1. The 9th export + real NativeEvents (Gate 1, .NET/ABI)

- **`Exports.cs`**: `blazornative_host_event(IntPtr nameUtf8, IntPtr payloadUtf8) → int`
  (`[UnmanagedCallersOnly(EntryPoint = "blazornative_host_event", CallConvs = cdecl)]`
  — BN0021-conforming, top-level try/catch — BN0020-conforming). Splits into a
  managed-callable **`DispatchHostEventCore(name, payload)`** (the Exports.DispatchEventCore
  pattern — unit-tested without the ABI crossing). rc contract mirrors dispatch_event:
  `0` delivered / `2` a subscriber/re-render faulted (captured) / `3` malformed/NULL.
  Header comment → "Nine exports"; the deletion-rationale block stays historical.
- **`NativeShellBridge.cs`**: replace `NativeEvents { add{} remove{} }` with a real
  multicast + `RaiseNativeEvent` (GetInvocationList + per-subscriber try/catch, the
  established isolation pattern; strict-posture doc: subscriber faults contained, the
  host event itself still "delivered"). `DispatchHostEventCore` resolves the session's
  bridge and raises. BN0014 premise unchanged (sync callback window) — the analyzer
  survives, verified by its tests staying green.
- **ABI ripple** (all in this gate): bump the export-count assertion 8→9 in `ci.yml`
  (×3 RID steps), `android-instrumented.yml`, `ios-spike.yml` (the iOS symbol list —
  the export is platform-neutral, the iOS artifact carries it automatically; the
  dispatch-only spike workflow is not re-run, just kept honest); add the JNA binding
  in `NativeBindings.kt`; grep current-state docs (README architecture) for
  "eight-export" → "nine-export" (historical conclusion docs stay as-written).

## 2. Nav manager: previous-route + NavigateBack (Gate 1, .NET)

- `NativeNavigationManager`: a `_previousRoute` field set on each successful
  `NavigateToAsync` (record `from` before the swap). New
  **`ValueTask<bool> NavigateBackAsync()`**: if `_previousRoute` is set → `NavigateToAsync(previous)`
  (which itself updates the slot — back-from-settings→/ then a further back has no
  prior → false), return `true` (handled); else return `false` (at root, shell
  finishes). `INavigationManager` (Core) gains `NavigateBackAsync`. Off-lane safety:
  `SwapRoot`'s `RunAfterDispatch` must behave when there's no open dispatch batch
  (host-initiated, not in-click) — VERIFY this at Gate 1 (RED-on-ambiguity; the 3.5
  KDoc frames it around the in-click case, so a drain-with-no-batch test is the pin).

## 3. HostEventProbe (the consumer that proves it lands)

New internal scaffolding component (`FocusProbe` precedent, registered "HostEventProbe"):
`[Inject] IMobileBridge`, subscribes to `NativeEvents` in `OnInitialized` (sync handler
— BN0014-clean), renders a `BnText` showing the last event name + a count, unsubscribes
in `Dispose`. Proves lifecycle events reach a mounted component and re-render on-device.
Ledgered as scaffolding in the M5 audit.

## 4. Android wiring (Gate 3)

- **Lifecycle**: `MainActivity` gains `onPause`/`onResume`/`onDestroy` → each marshals
  `runtime.dispatchHostEvent("onPause"/…)` onto the dispatch lane (NEW
  `BlazorNativeRuntime.dispatchHostEvent(name, payload?)` paralleling `dispatchEvent`,
  + its JNA binding). **The onDestroy trap** (BlazorNativeRuntime KDoc): fire the event
  but do NOT call `shutdown` (Activity recreation re-runs `start`) — documented at the
  site.
- **Predictive back**: `OnBackInvokedCallback` (API 33+, the registered dispatcher) with
  the `onBackPressedDispatcher` fallback for lower APIs → `runtime.dispatchHostEvent`
  routes to `NavigateBackAsync`; a `false` return (at root) lets the default back
  proceed (finish). Manifest: `android:enableOnBackInvokedCallback="true"`.
- **Deep link**: `AndroidManifest` gains a `VIEW` intent-filter
  (`blazornative://` custom scheme — no domain verification, simplest honest proof;
  `<data android:scheme="blazornative"/>`). `MainActivity.onCreate` parses `intent.data`
  (`blazornative://settings` → route `/settings`) and feeds it into the
  `AndroidShellBridge` route slot BEFORE boot, so `QueryStartupRoute` returns it and the
  3.5 startup-honor mounts `BnSettingsPage`. EXTRA_COMPONENT precedence documented.

## 5. Tests + baselines

- **.NET** (Gate 1): `DispatchHostEventCore` rc contract; `NativeShellBridge` firing +
  isolation; `NavigateBackAsync` (previous-route, at-root false, the no-open-batch
  drain); HostEventProbe mount + a `host_event` dispatch → probe BnText updates
  (nodeId-pinned).
- **JVM** (Gate 2): `blazornative_host_event` through the dll (a `dispatchHostEvent`
  seam) → HostEventProbe frames show the update; NavigateBack through the dll; the
  9-symbol export check (the JVM boot smoke or a dedicated assertion).
- **Android** (Gate 3): `ActivityScenario.moveToState(STARTED)` → HostEventProbe shows
  "onPause" on-screen (the new lifecycle instrumented pattern); a `VIEW`-intent launch
  (`blazornative://settings`) → BnSettingsPage at startup; predictive back from
  `/settings` → BnDemo returns (fresh), back from `/` → activity finishes.
- Version → `1.3.0-phase-5.1` (+ both Kotlin assertions). Counts grow on all three
  surfaces; ci.yml/android-instrumented assertions updated. Merge via PR.

## Risks

| Risk | Mitigation |
|---|---|
| 9-export ABI change misses an assertion → red CI | Gate 1 enumerates all five assertion sites + NativeBindings; the PR's CI is the backstop |
| `RunAfterDispatch` misbehaves off the dispatch lane (no open batch) | Verify FIRST at Gate 1 (RED-on-ambiguity); the no-open-batch drain is an explicit pin |
| onDestroy→shutdown trap | Documented at the site; the lifecycle event fires without shutdown; an Activity-recreation instrumented check if cheap |
| Predictive back API split (33+ vs lower) | OnBackInvokedCallback + onBackPressedDispatcher fallback; test on the API-34 AVD (the registered path) |
| Deep-link custom scheme "not a real app link" | Honest: `blazornative://` custom scheme proves the mechanism; https App Links (domain verification) = later/M7; recorded |
| iOS symbol assertion drift (spike workflow) | Bump ios-spike.yml to 9 for honesty; not re-run (dispatch-only, 5.2 builds the real lane) |
