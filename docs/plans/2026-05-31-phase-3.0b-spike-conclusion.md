### Diagnostic iterations during Gate 1

Three plan-spec gaps surfaced during execution and were resolved with minimal additive fixes (no plan content changed):

1. **`AllowUnsafeBlocks` missing from csproj.** Plan's `Exports.cs` declares `Init` with `unsafe BlazorNativeInitOptions* opts` (pointer param) but the csproj did not enable unsafe. Added `<AllowUnsafeBlocks>true</AllowUnsafeBlocks>` to the `PropertyGroup` with a comment explaining the [UnmanagedCallersOnly] pointer-param rationale. Safe — unsafe is standard for the C-ABI export idiom.

2. **`BlazorInterop` is `internal` to BlazorNative.Renderer.** Plan's `Exports.cs` calls `BlazorNative.Renderer.BlazorInterop.EnsureInitialized()`. The type is `internal static class BlazorInterop`. Resolved by adding `<InternalsVisibleTo Include="BlazorNative.NativeHost" />` to `BlazorNative.Renderer.csproj` — preserves the Renderer's public API surface; does not require changing BlazorInterop's accessibility.

3. **`vswhere.exe` not on PATH at publish time.** NativeAOT's `findvcvarsall.bat` calls `vswhere.exe` via `%ProgramFiles(x86)%\Microsoft Visual Studio\Installer\vswhere.exe`. The unquoted `%ProgramFiles(x86)%` expansion fails in CMD parser context (parenthesis closes a block prematurely), so vswhere emits `'vswhere.exe' is not recognized` to stdout. That error text gets captured into `_FindVCVarsallOutput` and becomes part of `_CppToolsDirectory`, corrupting the linker command line. Workaround: prepend `C:\Program Files (x86)\Microsoft Visual Studio\Installer` to `$env:PATH` before running `dotnet publish`. Did not modify any project files for this — it's an environment quirk and Gate 4 (WSL) won't see it. **Carryover into Phase 3.0c:** decide whether to bake this PATH prefix into a `build-native.ps1` helper or document as a setup step.

### Diagnostic iterations during Gate 3

1. **Rootfs URL drift.** Plan's URL `https://cloud-images.ubuntu.com/wsl/jammy/current/ubuntu-jammy-wsl-amd64-wsl.rootfs.tar.gz` returned 404. Directory listing showed Canonical renamed the rootfs to `ubuntu-jammy-wsl-amd64-ubuntu22.04lts.rootfs.tar.gz` (last published 2025-03-18). Updated `$rootfsUrl` in setup.ps1 to the new filename. This is exactly Risk R3 from the plan ("WSL rootfs URL drift") materializing; resolved with the in-place rename, no need to pin to a dated URL.

2. **Heredoc CRLF mangling on bash stdin.** Plan's bootstrap used a PowerShell here-string (`@'...'@`) piped into `wsl -d $wslDistro -e bash`. First run worked because the failing line was the LAST line and earlier lines completed before failing. Second run (idempotent path) had `dotnet --version` as its only command and failed immediately. Resolved by rewriting `$bootstrapScript` as a single-line `bash -c "..."` invocation — no CRLF in the bash args. Documented inline in setup.ps1 so the next maintainer doesn't reintroduce the heredoc form.

## Gate 4 — Cross-compile to linux-bionic-arm64 + linux-bionic-x64
- **Status:** RED
- **Date:** 2026-05-31
- **Time:** ~0.5h (Task 4.1 first publish + 1 diagnostic iteration to characterize the failure mode)
- **Result:** `dotnet publish -c Release -r linux-bionic-x64` fails inside `blazornative-ubuntu` WSL with .NET SDK 10.0.300 / ILCompiler 10.0.8. **Risk R4 materialized in its most severe form:** the failure is NOT "NDK sysroot missing" (which would have been fixable with an `apt-get install android-ndk` plus retry); it is **"the `linux-bionic-*` RID is not supported by the .NET 10 NativeAOT toolchain at all."** Neither the runtime pack nor the cross-link path exists in the released SDK.

### Failure mode (Task 4.1)

`dotnet publish -c Release -r linux-bionic-x64` for `BlazorNative.NativeHost.csproj` proceeds through restore + managed compile (the `linux-bionic-x64/BlazorNative.NativeHost.dll` IL assembly produces successfully), then fails at the NativeAOT pre-link step:

```
/root/.nuget/packages/microsoft.dotnet.ilcompiler/10.0.8/build/Microsoft.NETCore.Native.Publish.targets(70,5): error : The PrivateSdkAssemblies ItemGroup is required for _ComputeAssembliesToCompileToNative [src/BlazorNative.NativeHost/BlazorNative.NativeHost.csproj]
```

The error is opaque on its surface but root-causes cleanly:

1. **`PrivateSdkAssemblies` is populated from the RID-specific ILCompiler runtime pack.** The MSBuild target at line 70 of `Microsoft.NETCore.Native.Publish.targets` reads it from items contributed by `runtime.<RID>.Microsoft.DotNet.ILCompiler`.

2. **`runtime.linux-bionic-x64.Microsoft.DotNet.ILCompiler` is not in the runtime.json graph.** Inspecting `/root/.nuget/packages/microsoft.dotnet.ilcompiler/10.0.8/runtime.json` shows only these RIDs: `freebsd-arm64`, `freebsd-x64`, `linux-arm`, `linux-arm64`, `linux-musl-{arm,arm64,x64}`, `linux-x64`, `osx-{arm64,x64}`, `win-{arm64,x64,x86}`. **No `linux-bionic-*` entry.** So the restore graph contributes no `runtime.linux-bionic-*.Microsoft.DotNet.ILCompiler` pack and `PrivateSdkAssemblies` stays empty.

3. **The package was never published for .NET 10.** Direct attempt to add `runtime.linux-bionic-x64.Microsoft.DotNet.ILCompiler` as a PackageReference yields:

   ```
   error NU1102: Unable to find package runtime.linux-bionic-x64.Microsoft.DotNet.ILCompiler with version (>= 10.0.8)
   error NU1102:   - Found 1 version(s) in nuget.org [ Nearest version: 8.0.0-preview.6.23329.7 ]
   ```

   Only one historic preview from June 2023 exists on nuget.org. No 8.0.0 GA, no 9.0.x, no 10.0.x.

4. **Reproduced in a minimal scaffold** — same error on a 3-line empty class library with just `<PublishAot>true</PublishAot>` and `-r linux-bionic-x64`. The issue is intrinsic to the toolchain, not specific to BlazorNative.NativeHost.

### Why Briefing 1's "it just works" outcome was impossible

The Gate 4 plan acknowledged three outcomes — (1) NuGet pulls in the toolchain → just works, (2) sysroot missing → install NDK + retry, (3) other error → diagnose once. The plan's Risk R4 anticipated path (2). What actually materialized is a hidden path (0): **the toolchain does not exist at any version Microsoft publishes for .NET 10**, so paths (1)/(2)/(3) are all unreachable. Installing the NDK would not have helped — the failure is at MSBuild-target time, well upstream of any linker invocation.

### State of `dotnet/runtime` upstream

The `linux-bionic-*` RID descriptors exist in the SDK's RID graph (so the SDK *accepts* `-r linux-bionic-x64` without complaint and produces a managed IL `.dll` under `bin/Release/net10.0/linux-bionic-x64/`), but the NativeAOT pipeline has no implementation behind that RID. This is consistent with the upstream `dotnet/runtime` status quo for .NET 9/10: `linux-bionic-*` is reserved for the **Mono-AOT + .NET Android (MAUI/net10.0-android TFM) path**, not for the standalone NativeAOT toolchain. The 8.0.0-preview.6 package on nuget.org was an experimental probe that did not graduate.

### No artifacts to revert

Tasks 4.2 (arm64), 4.3 (readelf), 4.4 (copy to jniLibs), and 4.5 (Gradle wiring) were not attempted — they all presuppose Task 4.1's `.so` output exists. No `.so` files were written into `src/BlazorNative.Jni/src/androidMain/jniLibs/`. No `build.gradle.kts` edit was made. The only side effects of Gate 4 are:
- `src/BlazorNative.NativeHost/bin/Release/net10.0/linux-bionic-x64/` — managed IL `.dll` (not a published NativeAOT artifact). Build output, gitignored.
- WSL distro's `~/.nuget/packages/` — restore cache (`Microsoft.DotNet.ILCompiler`, `microsoft.netcore.app.runtime.nativeaot.*` runtime packs for x64/arm64/etc.). Cache, not tracked.

The existing `vendor/wasmtime/jniLibs/<abi>/libwasmtime.so` files remain intact and untouched (coexistence rule trivially satisfied since nothing new was added).

## Carryover into Phase 3.0c (architectural pivot required)

Gate 4 RED invalidates the "NativeAOT-direct" architecture branch. Phase 3.0c re-planning must choose ONE of:

1. **Mono-AOT via `net10.0-android` TFM (`dotnet/android` workload).** Switch `BlazorNative.NativeHost.csproj` from `PublishAot=true` + `RuntimeIdentifiers=linux-bionic-*` to a `<TargetFramework>net10.0-android</TargetFramework>` project (which uses Mono-AOT under the hood, has full bionic support, and produces an `.aar` or `.so` consumable from the existing Gradle module). Cost: Phase 3.0a's IL2072 ACCEPT decisions need re-validation under Mono-AOT trim semantics (different from NativeAOT trim), but the IL2026 path Phase 3.0a hardened against is the same. Likely smallest-delta path.

2. **NativeAOT only for JVM-desktop; Android-side stays on the current wasmtime/.wasm path.** Phase 3.0c becomes "wire protocol on JVM-desktop NativeAOT" and Android NativeAOT is deferred indefinitely (or until Microsoft ships `linux-bionic-*` ILCompiler). Cost: Phase 3.0c's atomic cleanup of wasmtime/.wasm is no longer possible — coexistence becomes permanent. Worst outcome for the umbrella Phase 3.0 ("collapse the WASM-era stack").

3. **Mono-AOT for Android + NativeAOT for desktop (dual runtime).** Two `BlazorNative.*.Host` projects with shared `Exports.cs`/`Imports.cs`. Cost: two C-ABI surfaces to keep in sync (Mono p/invoke vs NativeAOT `[UnmanagedCallersOnly]` — same C signature, different declaration); the wire protocol still works because it's byte-level. Highest delivery cost but cleanest end state.

The choice belongs to the next planning session. Gate 1+2's JVM-desktop NativeAOT artifacts (`BlazorNative.NativeHost.dll` on `win-x64`) and the JNA + BootSmokeNativeTest plumbing are reusable under all three options — they are NOT wasted work even if option (3) lands.
