### Diagnostic iterations during Gate 1

Three plan-spec gaps surfaced during execution and were resolved with minimal additive fixes (no plan content changed):

1. **`AllowUnsafeBlocks` missing from csproj.** Plan's `Exports.cs` declares `Init` with `unsafe BlazorNativeInitOptions* opts` (pointer param) but the csproj did not enable unsafe. Added `<AllowUnsafeBlocks>true</AllowUnsafeBlocks>` to the `PropertyGroup` with a comment explaining the [UnmanagedCallersOnly] pointer-param rationale. Safe — unsafe is standard for the C-ABI export idiom.

2. **`BlazorInterop` is `internal` to BlazorNative.Renderer.** Plan's `Exports.cs` calls `BlazorNative.Renderer.BlazorInterop.EnsureInitialized()`. The type is `internal static class BlazorInterop`. Resolved by adding `<InternalsVisibleTo Include="BlazorNative.NativeHost" />` to `BlazorNative.Renderer.csproj` — preserves the Renderer's public API surface; does not require changing BlazorInterop's accessibility.

3. **`vswhere.exe` not on PATH at publish time.** NativeAOT's `findvcvarsall.bat` calls `vswhere.exe` via `%ProgramFiles(x86)%\Microsoft Visual Studio\Installer\vswhere.exe`. The unquoted `%ProgramFiles(x86)%` expansion fails in CMD parser context (parenthesis closes a block prematurely), so vswhere emits `'vswhere.exe' is not recognized` to stdout. That error text gets captured into `_FindVCVarsallOutput` and becomes part of `_CppToolsDirectory`, corrupting the linker command line. Workaround: prepend `C:\Program Files (x86)\Microsoft Visual Studio\Installer` to `$env:PATH` before running `dotnet publish`. Did not modify any project files for this — it's an environment quirk and Gate 4 (WSL) won't see it. **Carryover into Phase 3.0c:** decide whether to bake this PATH prefix into a `build-native.ps1` helper or document as a setup step.

### Diagnostic iterations during Gate 2

1. **CascadingProbe failure on first host-CLR run.** The plan's `CascadingTestHost` set `CascadingValue<string>.Name = "Theme"`, but the `[CascadingParameter]` on `CascadingProbe` had no `Name` argument — making it a TYPE-keyed consumer. Named cascades don't satisfy type-keyed consumers, so `Theme` was null and the rendered text came through as `theme=`. Fix: drop `Name="Theme"` from `CascadingTestHost` so the cascade is type-keyed too. Probe shape unchanged; passes after fix. (Resolution: probe shape itself was correct; the test host was misconfigured.)

2. **Gradle JVM resolution picked up JRE 1.8.** `JAVA_HOME` pointed at `C:\Program Files (x86)\Java\jre1.8.0_461` (system default); Gradle 8.11.1 demands JDK 17+. Set `$env:JAVA_HOME = "C:\Program Files\Eclipse Adoptium\jdk-21.0.11.10-hotspot"` + prepend its `bin/` to PATH. **Carryover into Phase 3.0c:** decide whether to either bake JDK 21 selection into a `build-jvm.ps1` helper, or document the JAVA_HOME requirement in setup.ps1.

3. **Kotlin script needed `import java.io.File`.** `java.io.File.pathSeparator` fully-qualified fails inside `build.gradle.kts` (the script DSL doesn't expose the `java` package symbol by default). Added `import java.io.File` to the top of the build script and used `File.pathSeparator`.

### Diagnostic iterations during Gate 3

1. **Rootfs URL drift.** Plan's URL `https://cloud-images.ubuntu.com/wsl/jammy/current/ubuntu-jammy-wsl-amd64-wsl.rootfs.tar.gz` returned 404. Directory listing showed Canonical renamed the rootfs to `ubuntu-jammy-wsl-amd64-ubuntu22.04lts.rootfs.tar.gz` (last published 2025-03-18). Updated `$rootfsUrl` in setup.ps1 to the new filename. This is exactly Risk R3 from the plan ("WSL rootfs URL drift") materializing; resolved with the in-place rename, no need to pin to a dated URL.

2. **Heredoc CRLF mangling on bash stdin.** Plan's bootstrap used a PowerShell here-string (`@'...'@`) piped into `wsl -d $wslDistro -e bash`. PowerShell here-strings preserve CRLF line endings; bash interprets the trailing `\r` as a literal character so `dotnet --version\r` is argv-split into `dotnet---version` (the `\r` collides with the `--`). First run worked because the failing line was the LAST line and earlier lines completed before failing. Second run (idempotent path) had `dotnet --version` as its only command and failed immediately. Resolved by rewriting `$bootstrapScript` as a single-line `bash -c "..."` invocation — no CRLF in the bash args. Documented inline in setup.ps1 so the next maintainer doesn't reintroduce the heredoc form.
