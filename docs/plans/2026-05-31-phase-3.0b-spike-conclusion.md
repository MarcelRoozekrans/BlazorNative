### Diagnostic iterations during Gate 1

Three plan-spec gaps surfaced during execution and were resolved with minimal additive fixes (no plan content changed):

1. **`AllowUnsafeBlocks` missing from csproj.** Plan's `Exports.cs` declares `Init` with `unsafe BlazorNativeInitOptions* opts` (pointer param) but the csproj did not enable unsafe. Added `<AllowUnsafeBlocks>true</AllowUnsafeBlocks>` to the `PropertyGroup` with a comment explaining the [UnmanagedCallersOnly] pointer-param rationale. Safe ‚Äî unsafe is standard for the C-ABI export idiom.

2. **`BlazorInterop` is `internal` to BlazorNative.Renderer.** Plan's `Exports.cs` calls `BlazorNative.Renderer.BlazorInterop.EnsureInitialized()`. The type is `internal static class BlazorInterop`. Resolved by adding `<InternalsVisibleTo Include="BlazorNative.NativeHost" />` to `BlazorNative.Renderer.csproj` ‚Äî preserves the Renderer's public API surface; does not require changing BlazorInterop's accessibility.

3. **`vswhere.exe` not on PATH at publish time.** NativeAOT's `findvcvarsall.bat` calls `vswhere.exe` via `%ProgramFiles(x86)%\Microsoft Visual Studio\Installer\vswhere.exe`. The unquoted `%ProgramFiles(x86)%` expansion fails in CMD parser context (parenthesis closes a block prematurely), so vswhere emits `'vswhere.exe' is not recognized` to stdout. That error text gets captured into `_FindVCVarsallOutput` and becomes part of `_CppToolsDirectory`, corrupting the linker command line. Workaround: prepend `C:\Program Files (x86)\Microsoft Visual Studio\Installer` to `$env:PATH` before running `dotnet publish`. Did not modify any project files for this ‚Äî it's an environment quirk and Gate 4 (WSL) won't see it. **Carryover into Phase 3.0c:** decide whether to bake this PATH prefix into a `build-native.ps1` helper or document as a setup step.
