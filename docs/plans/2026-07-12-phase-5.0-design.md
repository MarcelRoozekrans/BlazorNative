# Phase 5.0 Design — iOS Feasibility Spike

**Date:** 2026-07-12
**Milestone:** M5 — P4: Full Platform Coverage (closes DoD #1 — with EITHER verdict)
**Predecessor:** M4 complete (`v4.0`, merge `dd0970b`)
**Owns:** the named risk, verified first: can .NET 10 NativeAOT produce a linkable
**iOS-simulator** artifact carrying the eight-export `blazornative_*` C-ABI, built on
a free public-repo macOS runner? A RED reshapes the milestone (fallback decision
documented); a GREEN unlocks Phases 5.2/5.3.

---

## Framing (locked at milestone-open)

- **Simulator RID is the primary target** (`iossimulator-arm64`) — that is what
  Phases 5.2/5.3 actually run on the CI simulator. Device `ios-arm64` is probed
  secondarily (evidence for the audit; not load-bearing this milestone).
- **No Mac hardware**: everything runs on `macos-latest` (Apple Silicon, Xcode
  preinstalled, free for public repos). Iteration discipline: artifact-upload heavy
  (publish logs, `nm`/`otool` dumps), every red run diagnosed in the fixing commit —
  the 4.0 CI-iteration rules, tightened because each cycle costs minutes.
- **Time-box:** ~1–2 days of runner iteration. Past the box with no artifact →
  RED verdict + fallback decision (see below), not more digging.

## The ladder (attempted in order; first GREEN wins)

1. **Plain-RID NativeAOT publish** — `dotnet publish src/BlazorNative.Runtime -r
   iossimulator-arm64 -p:PublishAot=true` with `NativeLib` set appropriately
   (**static** is the expected iOS shape; the csproj gains RID-conditional
   properties, behavior-neutral for existing RIDs — existing CI asserts that).
   This mirrors the 3.0c linux-bionic posture: no workload, ILC + Apple toolchain.
   Expected friction: ILC RID support, linker invocation needing Xcode paths.
2. **The official mobile workload path** — `dotnet workload install ios` (or
   maccatalyst as a diagnostic), `net10.0-ios` TFM on a thin wrapper project around
   Runtime, `PublishAot=true` library/static-lib mode. Heavier (TFM split), but the
   supported lane.
3. **Runtime-pack bypass adaptation** — if packs exist for `iossimulator-arm64`
   (`Microsoft.NETCore.App.Runtime.NativeAOT.iossimulator-arm64`?),
   `PublishAotUsingRuntimePack=true` + a vendored targets file (the 3.0c trick,
   Apple-toolchain edition).

Any rung that produces an artifact proceeds to **verification**; a rung that REDs is
documented (exact error, one diagnostic iteration) and the ladder descends.

## Verification (the GREEN bar)

On the produced artifact (`.a` expected, `.dylib` acceptable for simulator):

- `nm -gU` (or `llvm-nm`) lists **all eight** `blazornative_*` symbols (underscore
  prefix on Apple platforms — `_blazornative_init` etc.; the check normalizes).
- File type sanity: `lipo -info` / `file` shows arm64 for the simulator slice.
- **Link probe** (the honest bar — symbols in an archive aren't proof they link):
  a ~30-line C or Swift stub that declares `blazornative_init`/`blazornative_version`,
  links against the artifact with the simulator SDK (`xcrun -sdk iphonesimulator
  clang ...`), builds an executable. If the simulator can also RUN it (a `xcrun simctl
  spawn` of the stub calling init + version and printing the version string) — that's
  a bonus rung, attempt but don't gate on it this phase (5.2 owns runtime proof).
- Record: artifact size, ILC/link wall time, the exact property set used (the
  pinned-combo documentation duty, like 3.0c's).

## Deliverables

- `.github/workflows/ios-spike.yml` — `workflow_dispatch` only (a spike, not a lane;
  5.2 builds the real lane). Steps: SDK per global.json, the ladder rung(s), the
  verification, artifact uploads. May be deleted or evolved by 5.2.
- Any RID-conditional csproj/targets changes needed (behavior-neutral elsewhere —
  full existing CI green proves it).
- **`docs/plans/2026-07-12-phase-5.0-spike-conclusion.md`** — the verdict doc (the
  3.0b/3.0c conclusion shape): GREEN (the working combo, pinned, + implications for
  5.2) or RED (what failed at each rung, exact errors, and the FALLBACK DECISION).

## Fallback ladder on RED (decided now so a RED doesn't stall)

1. **Mono-AOT via the iOS workload** (`net10.0-ios`, the officially-blessed mobile
   runtime) — same C-ABI surface via `[UnmanagedCallersOnly]`; heavier runtime,
   different perf posture; acceptable for the M5 parity bar, revisit-NativeAOT
   ledgered.
2. If even that REDs on a runner: **milestone re-scope** — M5 drops to Android
   host-events + APIs (5.1 + 5.4), iOS becomes its own milestone pending .NET 11
   (~Nov 2026, official Android NativeAOT era) — controller escalates to the user.

## Gates

| Gate | GREEN bar |
|---|---|
| **1 — Workflow + ladder** | ios-spike.yml runs on macos-latest; each attempted rung's outcome captured with logs/artifacts; an artifact exists (or all rungs documented RED). |
| **2 — Verification + verdict** | Eight symbols verified + link probe green (or RED verdict); spike conclusion doc committed; existing CI fully green (no behavior change to other RIDs); PR merged. |

## Risks

| Risk | Mitigation |
|---|---|
| Runner iteration cost (minutes/cycle, blind debugging) | Validate everything scriptable locally first (workflow lint, property sets); upload EVERYTHING; batch diagnostics per run |
| ILC lacks iossimulator RID support entirely | That's rung 1 RED — expected-possible; rungs 2/3 exist for exactly this |
| Xcode version drift on macos-latest | Pin via `xcode-select` to a named version in the workflow; record it |
| The spike "succeeds" with symbols but the archive can't link | The link probe IS the bar, not nm output |
