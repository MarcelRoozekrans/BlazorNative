## The pinned integration recipe

The whole recipe is **two csproj lines and one `@using`**. On `BlazorNative.Components.csproj`:

```xml
<Project Sdk="Microsoft.NET.Sdk.Razor">          <!-- was Microsoft.NET.Sdk -->
  ...
  <StaticWebAssetsEnabled>false</StaticWebAssetsEnabled>
```

- **`Microsoft.NET.Sdk.Razor`** wires the Razor source generator over `**/*.razor`
  automatically; the generated class is `public partial class X : ComponentBase` in
  `RootNamespace` (+ folder), i.e. exactly where the hand-written classes live. No package
  reference changes: the generator ships in the .NET SDK, and the compilation's existing
  `Microsoft.AspNetCore.Components` (+`.Web`) references are all it needs.
- **`StaticWebAssetsEnabled=false`** is the one web-machinery switch required. Evidence it
  worked: the clean win-x64 publish log contains **zero** matches for
  `staticwebassets|StaticWebAsset|scopedcss|ScopedCss`, and the Components `obj/` holds no
  manifest/bundle artifacts. Scoped CSS needed no extra switch — it rides static web assets
  and there are no `.razor.css` files (7.1 must not add any; CSS is out of scope through M7).
- Everything else in the csproj (pack metadata, `InternalsVisibleTo`, the analyzers
  ProjectReference, `IsAotCompatible`) carries over unchanged; the BN analyzers run over the
  generated code and are clean.

In the `.razor` file (7.1: hoist into `_Imports.razor`):

```razor
@using Microsoft.AspNetCore.Components.Web
```

This line is **load-bearing twice** — see the footgun below.

### The `@bind` decision (design risk #1): Components.Web's built-ins; own attributes are impossible with the stock compiler and own-namespace attributes

The design asked: own `[BindElementAttribute]`s, or reference `Components.Web`? The answer
is sharper than either option — established empirically (four scratch compilations) and
confirmed in the compiler source (`BindTagHelperProducer`, dotnet/roslyn
`src/Razor/src/Compiler/.../TagHelpers/Producers/BindTagHelperProducer.cs`):

1. **`BindElementAttribute` lives in the CORE `Microsoft.AspNetCore.Components` assembly**
   (so does `EventHandlerAttribute`), but `BindInputElementAttribute` lives in
   **`Components.Web`** — and the compiler's bind provider refuses to read ANY bind
   metadata unless **both** attribute types resolve in the compilation:
   ```csharp
   public override bool SupportsTypes
       => _bindElementAttributeType is not null && _bindInputElementAttributeType is not null;
   ```
   Own bind attributes in our own namespace with only the core package therefore **cannot
   work** — verified: the build *succeeds* and `@bind` silently stays literal markup. (The
   provider resolves the attribute types by metadata NAME, so squatting the
   `Microsoft.AspNetCore.Components` namespace with own declarations could theoretically
   satisfy the check — noted for precision, not as an option anyone should take.)
2. **A custom bind-metadata carrier must be named exactly `BindAttributes`** (public):
   ```csharp
   public override bool IsCandidateType(INamedTypeSymbol type)
       => type.DeclaredAccessibility == Accessibility.Public && type.Name == "BindAttributes";
   ```
   With Components.Web referenced AND a correctly-named own carrier, extending `@bind` to a
   custom element **does** work (verified with a `myinput` element) — relevant if M7's form
   controls ever want `@bind` on a non-HTML element name.
3. **Decision: declare nothing.** `Components.Web` has been our pinned reference since 3.4
   (`MouseEventArgs` is on `BnButton`'s public surface) and is already proven trim-benign by
   the standing 4-IL2072/9-export gate. Its built-in `BindAttributes` declare exactly our
   wire: `@bind` on `input` → `value` + `onchange` via
   `EventCallback.Factory.CreateBinder` — precisely the pair `BnInput` hand-wires. The
   "smallest dependency footprint" option turned out to be *the dependency we already have*.

### The footgun: an out-of-scope `@bind`/`@onclick` compiles SILENTLY to literal markup

Bind and event tag helpers are **namespace-scoped like components**: without
`@using Microsoft.AspNetCore.Components.Web` in scope (file or `_Imports.razor`), the
compiler emits `AddMarkupContent(n, "<input @bind=\"Text\">")` — **zero diagnostics, build
green, binding dead**. On a native shell that renders as *nothing at all* (see the Markup
rule below), which at least makes it visible on first mount; in tests the golden-vs-twin
comparison reddens immediately. 7.1's `_Imports.razor` makes this structurally impossible
per page, but reviewers should know the failure mode exists.

### The renderer finding: whitespace Markup frames needed a slot (fixed, red-first)

The Razor compiler **preserves inter-element whitespace as Markup frames**
(`__builder.AddMarkupContent(n, "\n    ")`) — its .NET 5+ whitespace trimming only removes
whitespace leading/trailing within an element and around C# blocks. Hand-written
`BuildRenderTree` never emitted a Markup frame, so the walk's fall-through dropped them
**without a sibling slot**, while Blazor's diff `SiblingIndex` counts them as siblings:
every edit addressing a sibling *after* a markup frame (the echo span in any
naturally-authored page) resolved one slot short — poisoned cursor under strict, the wrong
node otherwise. Armed the moment the first `.razor` compiled; found red-first
(`MarkupFrameTests`: 3/5 failed before the fix).

The pinned contract (`SlotKind.Markup`):

- a **whitespace-only** Markup frame occupies a sibling slot (diff indices stay aligned) but
  emits **no patch** and contributes **zero host views** to insert-index translation — the
  wire never sees whitespace, which is exactly why golden-vs-twin can hold byte-identical
  against a twin that has no markup frames at all;
- a **non-whitespace** Markup frame is raw HTML — native shells have no `innerHTML` — so it
  is a contract violation: strict throws, non-strict logs-and-drops, and the slot is taken
  *first* either way so tolerated drops cannot shift later indices.

.NET-only: no new patch kind, no ABI change, no shell change.

### What the generated code actually looks like (inspected, not assumed)

`dotnet build -p:EmitCompilerGeneratedFiles=true` on `SpikeRazor.razor` shows:

- plain `BuildRenderTree` calls on `ComponentBase` — the exact shape we hand-write;
- **attribute case preserved as authored** — `backgroundColor` arrives camelCase (the style
  wire's requirement; nothing lowercases it);
- `@bind="_text"` → `AddAttribute("value", BindConverter.FormatValue(_text))` +
  `AddAttribute("onchange", EventCallback.Factory.CreateBinder(this, __value => _text = __value, _text))` +
  `SetUpdatesAttributeName("value")`;
- `@onclick="Clear"` → `EventCallback.Factory.Create<MouseEventArgs>(this, Clear)`;
- static text via `AddContent` (a Text frame), whitespace via `AddMarkupContent` (the Markup
  rule above); sequence numbers are per-source-position (diff keys only, never wire data).

**Warning:** `EmitCompilerGeneratedFiles` + `CompilerGeneratedFilesOutputPath` propagate to
referenced projects and write `.cs` files into *their* source trees, which the default glob
then compiles → duplicate-member build breaks. Use it only for one-off inspection and delete
the `generated/` dirs after.

## Implications for Phase 7.1

1. **The recipe is already in place** — `BlazorNative.Components` compiles `.razor` today;
   7.1 adds pages, not plumbing. Add `_Imports.razor` with
   `@using Microsoft.AspNetCore.Components.Web` (+ `@using BlazorNative.Components`) as its
   first act.
2. **Parity method proven**: golden-vs-twin (mount + dispatches, record equality, exclude
   `CommitFramePatch`) is the rewrite-verification tool — each page's existing
   `BuildRenderTree` class is the twin until parity, then retires (the milestone's DoD #2
   mechanism, now demonstrated end to end).
3. **`@bind-Value` on components** (BnInput etc.) is the compiler's case #4 —
   `Value`/`ValueChanged` pairs are inferred from component parameters, no element bind
   metadata involved — so DoD #2's `@bind-Value` syntax goal needs nothing beyond what
   already ships. (`@bind` on raw elements uses Components.Web's built-ins, proven here.)
4. **Whitespace is wire-invisible** — author pages naturally; the Markup arm owns the rest.
   Raw HTML content in markup (`<b>…</b>`, `&nbsp;`, `@((MarkupString)…)`) is a strict-mode
   violation by design: the supported subset is elements-our-renderer-maps + text +
   components. 7.1's five pages fit; anything else is a deliberate decision, not a surprise.
5. **No `.razor.css`, no `@page`** — scoped CSS would re-arm the static-web-asset machinery;
   routes stay in the registry (`@page` remains unsupported, per the design's out-of-scope
   list).
