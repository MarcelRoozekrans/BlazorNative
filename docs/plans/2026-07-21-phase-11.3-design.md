# Phase 11.3 — API stability + the path to 1.0 (M11 DoD #4)

**Milestone:** 11 — Production Readiness. **Phase:** 11.3. **Closes:** DoD #4 — the public API
surface of the seven shipped packages is *reviewed* and its stable core *identified*; the
unstable surface is *marked*; a **baseline** makes an unacknowledged public-API change red a PR;
**concrete 1.0 criteria** are written down; and the README's pre-1.0 caveat is re-cut against the
marked-stable surface.

This document does the review. The mechanism (a `PublicAPI.*.txt` baseline) is the cheap half —
generating one is an afternoon. The expensive half is **deciding what the baseline is allowed to
freeze**, and that decision cannot be made by a tool. So the substance below is the surface
walk, package by package, with a STABLE / PROVISIONAL / NOT-API call and a reason for each.

---

## 0. The distinction that must not blur: two contracts, two freeze stories

The repo already says "frozen" about something, loudly and often, and it is **not** the managed
public API.

| | **The C-ABI export surface** | **The managed public API** |
|---|---|---|
| What it is | 10 `[UnmanagedCallersOnly]` entry points (`src/BlazorNative.Runtime/Exports.cs:106`–`:475`) + an 80-byte / 10-slot `BlazorNativeBridgeCallbacks` struct (`src/BlazorNative.Runtime/BridgeProtocolNative.cs:80`) + the patch/frame structs | ~85 public types across 7 NuGet packages |
| Who consumes it | The two **shells** (Kotlin/JNA, Swift/ObjC++) — code *in this repo* | **App authors**, out of repo, via `dotnet add package` |
| How it is pinned | `Marshal.SizeOf == 80` and `OffsetOf(HostCallBegin) == 72` in `BridgeProtocolNativeTests`, re-asserted per capability (`CameraAbiUnchangedTests.cs:34`, `:46`; `NotificationsAbiUnchangedTests`; `SecureBiometricsAbiUnchangedTests`) **plus** a symbol-count gate on every published binary (`ci.yml:1465`, `:1514`, `:1547`) | **Nothing.** No pin exists today. |
| Freeze status | **Frozen and proven-frozen four times** — geolocation, notifications, biometrics+secure storage and camera each landed with *zero* struct growth and *zero* new exports | **Not frozen.** The README says so (`README.md:9`). |
| What a break costs | A shell rebuild in lockstep — both sides ship together, so a break is *loud and internal* | A consumer's build, silently, at `dotnet restore` of the next minor |

**A reader must not conclude "the ABI is frozen, therefore the API is."** They are opposite in
almost every respect: the ABI is internal-but-pinned, the API is external-but-unpinned. The
irony is exactly the gap this phase closes — the contract with *strangers* on the other end is
the one with no tripwire. The ABI is out of scope here (§11) and stays where it is.

## 1. What already constrains the surface — so this phase does not rebuild it

Four mechanisms already touch the shipped surface. None of them is a public-API baseline, and
knowing precisely what each does *not* cover is what keeps the new pin from being redundant.

- **`PackagePurityTests`** (`tests/BlazorNative.Runtime.Tests/PackagePurityTests.cs`) — pins the
  **shipped set** (`:61`–`:68`: Core, Renderer, Http, Components, Runtime, Analyzers, Device)
  against the `src/*.csproj` enumeration and against three other copies of that list, and
  asserts no app-shaped type (`*Demo`, `*Probe`, `SpikeRazor*`) ever lands in a shipped
  assembly (`:74`). **It constrains *which assemblies* ship and that no demo leaks in. It says
  nothing about the members inside them.**
- **`ComponentReferenceDriftTests`** (`:132`, `GeneratedReference_DocumentsExactlyThePublicTypes`)
  — the closest thing to a public-API pin that exists. It reflects `PublicTypes()` off
  `BlazorNative.Components.dll` (`:104`) and asserts the generated docs pages equal that set
  **in both directions**. Adding *or* removing a public type in Components reds this test today.
  **But: it is type-level, one package, and it enforces documentation coverage, not
  compatibility.** Renaming `BnButton.Label` to `BnButton.Text` is a hard break and stays green.
- **The parity / drift family** — `ShellStyleTableDriftTests`, `ShellFrameTableDriftTests`,
  `RouteTableDriftTests`, `TemplateDriftTests`, `ReadmeDriftTests`, `PackageVersionPinTests`.
  These pin *cross-file agreement* (the style allow-list vs both shells' mirrors, the README's
  counts vs the gates that assert them). They constrain **behaviour and consistency**, never
  **signature shape**.
- **The analyzers** — `BN0004` / `BN0010` / `BN0011` / `BN0013` (`MobilePolicyAnalyzer.cs:45`,
  `:54`, `:63`, `:72`), `BN0014` (`BridgeAsyncHandlerAnalyzer.cs:33`), `BN0020` / `BN0021`
  (`InteropBoundaryAnalyzer.cs:43`, `:52`). These constrain what *consumers may write*, not what
  the library may expose.

**Conclusion:** there is exactly one type-level guard, on one of seven packages, and it exists
for a docs reason. A member-level compatibility baseline is genuinely new work, not a duplicate.

**One inherited pin the design must respect:** `PackageVersionPinTests` walks every shipped
csproj and reds on a per-project `<Version>` (tooth 2). Adding a `PackageReference` is fine; the
baseline work must not introduce a per-project version literal.

**Classification: STABLE.** Three grounded reasons, not sentiment:

1. **It is already parity-contract-pinned.** Every style name a component can emit is held
   against both shells' hand-written mirrors (`ShellStyleTableDriftTests`), and the frame tables
   are asserted identical on the Android emulator and the iOS simulator. A parameter rename does
   not just break a consumer — it breaks a three-way contract with two shells and a golden.
2. **It is already doc-coverage-enforced.** `BnEnforceDocCoverage` is `true` for this package
   alone (`src/BlazorNative.Components/BlazorNative.Components.csproj:98`), and
   `src/Directory.Build.targets:65` escalates CS1591 to an **error** for it. Every public member
   here already carries a `<summary>`. A `PublicAPI.Shipped.txt` for Components will be the
   *cheapest* of the seven to review, because every line already has prose next to it.
3. **It is already type-drift-guarded.** `ComponentReferenceDriftTests` reds on an added or
   removed public type. The baseline extends that guard from types to **members**, which is
   precisely the missing tooth.

**The one caveat:** `BnList<TItem>`, `BnPicker`, `BnSlider`, `BnSwitch`, `BnCheckbox` and
`BnModal` declare their parameters inside `.razor` files. The Razor source generator produces
the partial that carries them, so `PublicApiAnalyzers` sees them **through generated code**.
This is the single implementation unknown of §4 and is called out again in §8.

### 2.2 `BlazorNative.Core` — 21 types — the contracts package

| Group | Types |
|---|---|
| Bridge contract | `IMobileBridge` (`IMobileBridge.cs:13`) — **27 members**: 25 methods, 1 property (`PlatformInfo`, `:125`), 1 event (`NativeEvents`, `:129`) |
| Bridge implementation | `DevHostBridge` (`DevHostBridge.cs:17`) — the in-process mock, 44 public member lines |
| Navigation contract | `INavigationManager` (`INavigationManager.cs:15`) — 4 members: `CurrentRoute`, `NavigateToAsync`, `NavigateBackAsync`, `RouteChanged` |
| HTTP value types | `BridgeHttpRequest` (`:148`), `BridgeHttpResponse` (`:154`) |
| Platform | `PlatformInfo` (`:159`), `PlatformKind` (`:169`) |
| Events | `NativeEvent` (`:165`), `BnImageErrorEventArgs`, `BnScrollEventArgs` |
| Capability results | `GeolocationStatus` (`:192`), `GeolocationResult` (`:205`), `GeolocationPosition` (`:210`), `NotificationStatus` (`:240`), `NotificationSpec` (`:256`), `BiometricStatus` (`:288`), `SecureStorageStatus` (`:313`), `SecretResult` (`:331`), `CameraStatus` (`:367`), `CaptureOptions` (`:384`), `PhotoResult` (`:400`) |

**Classification, split three ways:**

- **STABLE — the capability result records and status enums (11 types).** These are the strongest
  freeze candidates in the whole repo. Each was designed under an explicit milestone law —
  *"denial is DATA, never an exception and never a hang"* (`IMobileBridge.cs`, stated four times
  over) — each is `readonly record struct` or `enum`, each has been exercised on both shells
  through goldens and demos, and each is already **wire-mirrored** into the two shells. Changing
  `GeolocationStatus`'s members is not a C# edit; it is a three-language edit.
  **One caveat worth writing into the baseline review, not deferring:** an `enum` is a
  source-compatible-but-behaviourally-breaking surface — adding a member is legal for a compiler
  and lethal for a consumer's exhaustive `switch`. The baseline records enum *members*; it will
  therefore make an addition **visible**, which is exactly right, and the review policy should be
  "adding a status member is a minor-version change with a changelog line", not "adding a status
  member is free".
- **STABLE — `IMobileBridge` and `INavigationManager` as *contracts*, with a caveat.** They are
  the layering spine (Components→Core, Renderer→Core, Runtime→Core). `IMobileBridge` at 27
  members is wide, but every member has a shell implementation on both platforms; it is not
  speculative surface. **The caveat:** these are interfaces without a default implementation, so
  *adding a member is a breaking change for any consumer who implemented them.* Almost nobody
  will — the two real implementations are `NativeShellBridge` (Runtime) and `DevHostBridge`
  (Core) — but a 1.0 freeze must say out loud which of the two policies applies: (a) the
  interface is *consume-only*, additions are non-breaking by declaration, or (b) additions use
  `virtual` default implementations. **Recommendation: (a), stated in xmldoc**, because the
  alternative litters the contracts package with bodies it has no business holding. Gate A must
  make this call explicitly.
- **PROVISIONAL — `INavigationManager`'s *location*.** Its own header says *"M6 lifts it into a
  dedicated `BlazorNative.Navigation` package"* (`src/BlazorNative.Core/INavigationManager.cs:7`).
  M6 came and went; the lift is live as [#23](https://github.com/MarcelRoozekrans/BlazorNative/issues/23),
  which M11 explicitly scopes out as *growth, not readiness*
  (`docs/planning/MILESTONE.md:153`). So a public type in the contracts package is under a
  **planned relocation**. This is not a reason to leave it unmarked — it is a reason to record
  the mitigation now: **a package move is not a source break if the old package ships a
  `[TypeForwardedTo]`**, and the 1.0 criteria must say the lift lands with one. The comment
  itself is a grounding inconsistency (§12.1).
- **PROVISIONAL — `DevHostBridge`.** It is public because tests and dev harnesses across
  assemblies construct it, and it is genuinely useful to a consumer writing tests. But its 44
  public member lines are *mock-shaped*: seeding hooks and canned results, not a contract anyone
  designed for strangers. Freezing it at 1.0 freezes a test double. **Recommendation: baseline
  it (so changes are visible) but exclude it from the "stable core" the README advertises**, and
  say so in its xmldoc. It is the natural home of a future consumer test harness
  ([#25](https://github.com/MarcelRoozekrans/BlazorNative/issues/25)), which is itself an
  argument against freezing its shape now.

### 2.4 `BlazorNative.Runtime` — 14 types — a split package

| Genuinely consumer-facing | ABI plumbing, public only for interop |
|---|---|
| `BlazorNativeApp` (`BlazorNativeApp.cs:85`) — `DefaultRoute` const (`:87`), `RegisterPages` (`:97`), **`ConfigureServices` (`:113`, new in 0.4.0 via #159)** | `Exports` (`Exports.cs:84`) — the 10 `[UnmanagedCallersOnly]` entry points |
| `BlazorNativePage` (`:42`) — `Routed<T>` (`:57`), `Named<T>` (`:70`), `Route`, `Name` | `BlazorNativeBridgeCallbacks` (`BridgeProtocolNative.cs:80`), `BlazorNativeFetchRequest` (`:105`), `BlazorNativeFetchResponse` (`:117`) |
| | `BlazorNativeInitOptions` (`Exports.cs:60`), `BlazorNativeInitResult` (`:77`) |
| | `BlazorNativePatchKind` (`PatchProtocolNative.cs:29`), `BlazorNativeNodeType` (`:50`), `BlazorNativePatch` (`:57`), `BlazorNativeFrame` (`:71`) |
| | `NativeShellBridge` (`NativeShellBridge.cs:38`), `NativeNavigationManager` (`NativeNavigationManager.cs:37`) |

**Classification: split, and the split is the point.**

- **STABLE — `BlazorNativeApp` + `BlazorNativePage` (2 types, 5 members).** This is the app
  author's entire startup contract, and it is deliberately tiny. `BlazorNativePage`'s
  constructor is `private` (`:48`) with the two static factories as the *only* way in, precisely
  so the trim-law shape (`DynamicallyAccessedMembers(All)` flowed to a concrete call site) is
  not constructible any other way (`:29`–`:37`). That is a surface designed to be frozen.
  `Mount` is `internal` (`:46`) — correctly.
  **`ConfigureServices` (0.4.0, #159) is the newest member of the stable core and the one with
  the least mileage**: it was proven at runtime by 11.1's ABI harness against the *published*
  binary, but it is ~1 day old at the time of writing. Its signature —
  `Action<IServiceCollection>` — is the smallest possible shape and matches the ASP.NET Core
  idiom exactly, so the risk is not the signature, it is the *semantics* (call-once, additive,
  last-wins, exactly one provider, documented at `:100`–`:112`). **Mark it STABLE**: a narrower
  mark would be theatre, since the alternative shapes were considered and rejected in 11.1 and
  there is no plausible v2 of `Action<IServiceCollection>`.
- **NOT-API — the 10 interop types.** `Exports`, the four native structs, the two protocol enums
  and the two patch structs are public **only** because the C-ABI export mechanism and the
  shells' JNA/ObjC++ mirrors require it. No app author calls `Exports.Init`. Freezing them under
  a *managed*-API contract would be false advertising in both directions: they are already
  frozen harder than any baseline can express (§0), and their *managed* shape is an
  implementation artefact of that freeze.
  **They still get baseline lines** — a `PublicAPI.Shipped.txt` records what is public, and
  omitting them would be a lie about the assembly — but they are marked NOT-API in xmldoc and
  excluded from the README's stable-core claim.
- **NOT-API — `NativeShellBridge`, `NativeNavigationManager`.** Public because `HostSession`
  (which is `internal static`, `HostSession.cs:31`) composes them across the Runtime/Core
  boundary and tests reach them. A consumer never constructs either. Same treatment.

### 2.5 `BlazorNative.Renderer` — 16 declarations / 14 effective — mostly not consumer API

**The direct answer to the question posed: `NativeRenderer` is *not* API a consumer should bind
to. It is an implementation detail that happens to be public.**

The evidence is structural, not aesthetic:

- Its consumer is `HostSession` — which is `internal static unsafe`
  (`src/BlazorNative.Runtime/HostSession.cs:31`). The renderer is public so that an **internal**
  type in a **different assembly** can drive it. That is the textbook shape of accidental
  publicity.
- Its surface is a renderer's internals in the open: `FrameSink` (`NativeRenderer.cs:70`),
  `StrictErrors` (`:87`), `RunAfterDispatch` (`:250`), `DispatchUiEventAsync` (`:1197`), the
  `Frames` event (`:59`), plus `Mount<T>` / `MountAsync<T>` / `Unmount` (`:152`–`:223`) and an
  overridden `Dispatcher` returning a `private sealed class InlineDispatcher` (`:114`–`:116`).
  There is no scenario in which an app author sets `StrictErrors`.
- The one type here a consumer *legitimately* meets is `BlazorNativePage.Routed<T>`'s thunk type
  — `Func<NativeRenderer,int>` — and even that is `internal` (`BlazorNativeApp.cs:46`). The
  package's own header explains the layering: the mount thunk's type is why `BlazorNativeApp`
  lives in Runtime rather than Core (`BlazorNativeApp.cs:80`–`:84`).

Same call for the **10 patch records** (`RenderPatch` and its 9 derivatives + `RenderFrame` +
`NativeUiEvent`, `PatchProtocol.cs:23`–`:121`): they are the *in-memory* model between renderer
and encoder. The header states plainly that the wire format lives elsewhere — *"This is NO LONGER
a JSON protocol: the wire format is the typed-struct encoding in `BlazorNative.Runtime`"*
(`PatchProtocol.cs:8`–`:10`). A consumer who binds to `SetStylePatch` is coupling to a layer
the framework is free to re-shape.

**Classification: NOT-API for `NativeRenderer`, the 10 patch records, and
`BlazorVersionMismatchException`'s throw sites.** `ServiceCollectionExtensions.AddBlazorNativeRenderer`
(`RendererServices.cs:19`) is **PROVISIONAL** — it is a composition-root call, made once by the
framework's own host session, but a consumer building a custom host would need it.

**A real judgement to record: making these `internal` + `InternalsVisibleTo` would be the
*correct* fix, and it is a breaking change this phase must not make.** It is a 1.0-criteria item
(§6, criterion S3), argued there, not a Phase 11.3 edit. Marking beats moving in a docs-and-pins
phase.

### 2.6 `BlazorNative.Http` — 2 types — small and stable

- `BridgeHttpHandler : HttpMessageHandler` (`BridgeHttpHandler.cs:28`) — one public constructor
  taking `IMobileBridge` (`:32`); everything else is a `protected override`.
- `ServiceCollectionExtensions` (`HttpServices.cs:10`) — three methods: `AddBlazorNativeHttp`
  (`:25`), and two `AddBlazorNativeHttpClient` overloads, generic and named (`:53`, `:76`).

**How `HttpClient` is composed:** the app never news up an `HttpClient` — `BN0011`
(`MobilePolicyAnalyzer.cs:63`) makes `new HttpClient()` a diagnostic, narrowed deliberately to
the parameterless ctor (`:109`). Instead the DI registration wires `BridgeHttpHandler` as the
primary handler over `IMobileBridge`, so every request crosses the native bridge (host-side TLS)
rather than the .NET socket stack — which is also why `BN0010` bans `System.Net.Sockets.*`.
`AddBlazorNativeHttpClient<TClient>` returns `IHttpClientBuilder`, i.e. the standard
`Microsoft.Extensions.Http` idiom, so a consumer's muscle memory transfers.

**Classification: STABLE.** Four members, all mirroring an established BCL idiom, all analyzer-
reinforced. This is the second-cleanest package after Device.

### 2.7 `BlazorNative.Analyzers` — 3 types — a package whose public API is beside the point

`MobilePolicyAnalyzer` (`:33`), `InteropBoundaryAnalyzer` (`:35`), `BridgeAsyncHandlerAnalyzer`
(`:28`) — three `DiagnosticAnalyzer` subclasses, plus their `public static readonly
DiagnosticDescriptor` fields.

**Classification: NOT-API, and this package should be EXCLUDED from the baseline.** The
justification is specific, not a shrug:

1. **A consumer cannot reference it.** It ships as an analyzer asset
   (`analyzers/dotnet/cs`), targets `netstandard2.0`, and its Roslyn dependency is
   `PrivateAssets="all"` (`BlazorNative.Analyzers.csproj:48`–`:49`). Consuming projects get
   *diagnostics*, never a compile-time reference to `MobilePolicyAnalyzer`.
2. **Its real contract is the 7 diagnostic IDs** — `BN0004`, `BN0010`, `BN0011`, `BN0013`,
   `BN0014`, `BN0020`, `BN0021` — because those are what a consumer types into a `NoWarn`, an
   `.editorconfig` severity, or a `#pragma`. Renaming `BN0011` breaks builds; renaming the C#
   class it lives in breaks nothing.
3. **A `PublicAPI.Shipped.txt` cannot express (2).** It would faithfully record three class names
   and some fields, none of which is the thing that must not change. A pin that guards the wrong
   noun is worse than none: it manufactures confidence.

**Instead, the analyzer package gets the pin it actually needs:** a test asserting the shipped
`DiagnosticDescriptor` **ID set** is exactly the 7 above, red in both directions. That is the
real baseline for this package, it is cheap, and it belongs to Gate B alongside the `.txt` files.
(`PackagePurityTests` already reads the Analyzers assembly off the PE precisely because it
cannot be reflection-loaded — `:250`–`:253` — so the ID pin should live in the analyzer's own
test project, `tests/BlazorNative.Analyzers.Tests`, where Roslyn is already referenced.)

## 4. Mechanism — `Microsoft.CodeAnalysis.PublicApiAnalyzers`

### How it works

Two files per project, as `AdditionalFiles`:

- **`PublicAPI.Shipped.txt`** — the API of the last released version. One line per public member,
  in a canonical Roslyn-emitted format (`BlazorNative.Components.BnButton.Label.get -> string?`).
- **`PublicAPI.Unshipped.txt`** — API added since. On release, its contents move into Shipped.

Three diagnostics carry the contract:

| ID | Fires when | Meaning |
|---|---|---|
| **RS0016** | A public member exists in neither file | *You added public API without declaring it.* |
| **RS0017** | A file lists a member the assembly no longer has | *You removed or changed public API.* A rename fires **both** — RS0017 for the old name, RS0016 for the new. |
| RS0037 | (opt-in) nullability annotations absent from the baseline | Records `?` in the signature, so a nullability change is also a diff |

RS0037 should be **on**: the repo is nullable-enabled throughout, and `string` → `string?` on a
return is a real break that an un-annotated baseline would sleep through.

### Which of the seven get baselines

| Package | Baseline? | Why |
|---|---|---|
| `BlazorNative.Components` | **Yes** | The consumer surface. Also the cheapest to review — CS1591 is already an error here (`Directory.Build.targets:65`), so every member already has prose. |
| `BlazorNative.Core` | **Yes** | The contracts. |
| `BlazorNative.Device` | **Yes** | The façades. |
| `BlazorNative.Http` | **Yes** | 4 members. |
| `BlazorNative.Runtime` | **Yes** | Splits STABLE from NOT-API; the file makes the split *legible* — the two stable types sit in a baseline that also records 12 not-API ones, and the xmldoc says which is which. |

Six baselines, one substitute pin. The exclusion is a decision with a reason, not an omission.

### Generating the baseline, and the RS0016 flood

Adding the analyzer to an existing 85-type surface produces **one RS0016 per undeclared public
member** — order of several hundred, dominated by Components' 196 parameters. This is expected
and is handled, not endured:

1. Add the `PackageReference` (`PrivateAssets="all"`, `IncludeAssets` analyzers-only) and two
   empty `AdditionalFiles` per project. The version pins alongside the existing
   `Microsoft.CodeAnalysis.Analyzers` 5.6.0 (`BlazorNative.Analyzers.csproj:49`) — same
   roslyn-analyzers family. Renovate (`renovate.json`) will track it like any other reference.
2. Run the analyzer's shipped **code fix**, *"Add public API to `PublicAPI.Unshipped.txt`"*, with
   its fix-all provider, per project. This is mechanical: it emits the canonical lines.
3. **Then read every line.** This is the step the flood tempts you to skip, and it is where the
   value is. The reviewer's job is not "is this line correct" (the tool wrote it) but *"should
   this be public at all?"* — §2.5's `NativeRenderer` finding came from exactly this reading.
   Anything that surprises the reviewer is a finding, and findings become ledger lines, not
   silent baseline entries.
4. Move `Unshipped` → `Shipped` **once**, at the commit that lands the baseline, since 0.4.1 is
   already published — the current surface *is* shipped. Thereafter `Unshipped` accumulates
   between releases and is drained by the release PR.

**Sequencing decision: the baseline lands AFTER §6's classification review, not before.** A
baseline generated first records whatever is there, and the review that follows is then arguing
against a file that already looks authoritative.

### How it gates

**Lane: `build-test`.** It is the only *required* context (`README.md:240`, `:250`), the only
lane where the whole checkout is visible (the drift-test house rule), and the lane every existing
surface pin lives in. `android-build` and `ios-build` never compile the shipped `.csproj`s in a
way that would see these diagnostics first.

**Warnings must be escalated to errors, explicitly — and the repo already knows why.**
`src/Directory.Build.targets:57` states it flatly: *"No `-warnaserror`. No
`TreatWarningsAsErrors` anywhere in the repo."* `ci.yml:195` builds without `-warnaserror`. So
RS0016/RS0017 left at their default `warning` severity would print and be ignored — precisely
the failure the CS1591 pin already hit once, where a doc-coverage gate *"read as ON and was OFF
twice"* (`Directory.Build.targets:51`–`:63`).

**The mechanism is therefore the CS1591 precedent, verbatim:** a per-package MSBuild property
(`BnEnforcePublicApi`, mirroring `BnEnforceDocCoverage`) gating a `WarningsAsErrors` line in
`src/Directory.Build.targets`, set `true` in each of the six csprojs.

```xml
<!-- src/Directory.Build.targets — beside the CS1591 line at :65 -->
<WarningsAsErrors Condition="'$(BnEnforcePublicApi)' == 'true'">$(WarningsAsErrors);RS0016;RS0017;RS0037</WarningsAsErrors>
```

**And it must live in `.targets`, not `.props`, for the reason `.targets` already documents at
`:28`–`:34`:** `Directory.Build.props` is imported *before* the csproj body, so a condition there
reads the flag before the csproj has set it. That bug cost the CS1591 pin its teeth twice. It
will cost this one the same if the lesson is not re-applied.

**The named mutation that proves the gate:** rename `BnButton.Label` → `BnButton.Text` and build.
Expect `error RS0017` (Label removed) **and** `error RS0016` (Text undeclared), exit 1. If it
prints as a warning and the build is green, the gate is not wired — that is the CS1591 failure
mode reproducing, and Gate B is not passed.

### What the baseline buys before 1.0 vs at 1.0

The repo is `0.x` with `"bump-minor-pre-major": true` (`release-please-config.json:8`), so a
`feat:` commit produces `0.5.0` and **is allowed to break**. That is not a defect to work around;
it is the pre-1.0 contract, and the baseline does not change it.

- **Before 1.0 the baseline makes a break VISIBLE and DELIBERATE, not impossible.** The PR that
  breaks the API reds; the author edits `PublicAPI.Unshipped.txt` to acknowledge it; the diff
  now *contains the break* as reviewable lines, and the changelog can say so. Today an identical
  break is invisible in review — nothing in the diff announces that `BnButton.Label` vanished.
  That shift, from accident to decision, is the entire pre-1.0 value, and it is enough on its own.
- **At 1.0 the same file becomes a freeze.** The tier table (§3) stops being a design document
  and becomes policy: an RS0017 on a STABLE member is a **major-version** conversation; on a
  NOT-API member it is a changelog line. The file does not change; the *rule about who may edit
  it* does.

## 5. `[Experimental]` — where it fits, and why it is used sparingly

`System.Diagnostics.CodeAnalysis.ExperimentalAttribute` (.NET 8+) takes a diagnostic ID
(`[Experimental("BN1001")]`) and makes **every consumer use site a compile ERROR** unless
suppressed. That is a very loud instrument.

**Recommendation: apply it to nothing in the current surface.** The reasoning, which matters more
than the verdict:

- The NOT-API bucket (§3) is not *experimental* — it is *not for you*. `NativeRenderer` is stable
  in the sense that matters (the shells depend on it and it has not moved in five milestones);
  it simply is not a consumer contract. `[Experimental]` would say "this may change", which is
  the wrong claim, and would inflict a build error on anyone touching a package they must
  reference for unrelated reasons.
- The right mark for NOT-API is **`[EditorBrowsable(EditorBrowsableState.Never)]` + an xmldoc
  `<remarks>` line reading "Not part of the supported public API"**, plus its position in the
  tier table. That hides it from IntelliSense, documents the intent, and breaks nobody.
- Nothing in the surface is genuinely half-built. The closest candidate,
  `BlazorNativeApp.ConfigureServices`, is one day old — but it was *designed* in 11.1, proven at
  runtime against the published binary, and has the smallest possible signature. Marking the
  headline feature of 0.4.0 experimental one release after shipping it would be a worse signal
  than leaving it plain.

**Where it *would* be right, and should be reserved for:** the first member of a genuinely
unproven capability — the FCM push surface when
[#16](https://github.com/MarcelRoozekrans/BlazorNative/issues/16) lands without a real Firebase
project behind it, or haptics/purchases/background-tasks from
[#18](https://github.com/MarcelRoozekrans/BlazorNative/issues/18) shipped ahead of device proof.
**Gate C's deliverable is therefore the *policy*, not a decoration**: reserve `BN1xxx` for
experimental IDs, document the rule ("`[Experimental]` means *this signature may change in a
minor*; it is not a synonym for *internal*"), and record that the current surface warrants zero
uses. A gate can pass by proving nothing qualifies — provided it proves it.

## 7. The consequential question: can 1.0 ship with iOS simulator-only?

**Position: yes — and it must, or 1.0 never happens.** The argument, stated so a critic can
attack it:

**For shipping.**

1. **The blocker is administrative, not technical.** Real-device iOS needs an Apple Developer
   account. It is not gated on a missing capability, an unproven design or a known bug; it is
   gated on a purchase and an enrolment. A 1.0 criterion that no amount of engineering can
   satisfy is not a criterion — it is an indefinite hold.
2. **The parity contract is the strongest evidence available on any platform, and it already
   spans both shells.** The Yoga engine owns placement on both, and the frame tables are asserted
   **identical** on the Android emulator and the iOS simulator. Once Phase 11.2 proves the
   Android *device* matches the Android *emulator*, the transitive claim — device-Android ≡
   emulator-Android ≡ simulator-iOS — is not proof of device-iOS, but it is far more than
   "untested".
3. **The iOS simulator is a real ARM64 execution of the real NativeAOT static library**, not an
   emulator of a foreign ISA: `iossimulator-arm64` with `NativeLib=Static`, linked through the
   real `xcodebuild` toolchain, running real Swift/ObjC++ shell code (`ci.yml:2141`, `:2311`).
   What the simulator cannot prove is device-only behaviour — camera hardware, real Face/Touch ID,
   GPS, APNs, code-signing and provisioning, background/thermal behaviour. That is a **specific,
   enumerable** list, and enumerating it is what "documented" means here.
4. **Precedent.** Shipping 1.0 with a named, honest platform caveat is normal. Shipping 1.0 with
   a *hidden* one is not. The distinction is disclosure, and disclosure is free.

**Against, taken seriously.** Half the framework's headline claim is "Android **and** iOS", and
half of that claim has never touched a phone. Someone will ship an iOS app on this and hit
something the simulator cannot reproduce.

**Resolution — the caveat must be specific enough to be actionable, not a hedge.** 1.0 ships
with iOS simulator-only **only if** the README and docs carry a named list, not a shrug:

> **iOS real-device status.** The iOS shell is proven on the **simulator** (`iossimulator-arm64`,
> real NativeAOT static lib, real Swift/ObjC++ shell, frame tables asserted identical to
> Android's) and has **never run on physical iOS hardware**. Untested on device, specifically:
> camera capture from a real sensor, Face ID / Touch ID against the Secure Enclave, real-GPS
> geolocation, APNs and universal links, code-signing and provisioning, and thermal/background
> behaviour. Android **is** device-proven (Phase 11.2). Tracked as
> [#17](https://github.com/MarcelRoozekrans/BlazorNative/issues/17).

**And one hard condition: P1 (real-device Android) stays a BLOCKER.** Shipping 1.0 with *neither*
platform device-proven would be indefensible. Exactly one platform may carry this caveat, and it
must be the one with an administrative barrier — not a convenient one.

## 9. Gates

- **Gate A — the surface review + classification.** §2's per-package walk is verified against a
  fresh enumeration (not this document's memory of it); every type lands in exactly one tier;
  the contentious calls — `NativeRenderer`, `DevHostBridge`, `INavigationManager`'s location,
  the interface-additions policy (A4) — are decided **in writing** with a reason. Deliverable: a
  tier table in the repo, reviewable independently of any `.txt` file. **This gate is the
  phase's substance; B, C and D mechanise its output.**
- **Gate B — the baseline + the CI gate, proven by mutation.** Six `PublicAPI.Shipped.txt` /
  `PublicAPI.Unshipped.txt` pairs; `BnEnforcePublicApi` wired in
  `src/Directory.Build.targets` (**not** `.props` — `:28`–`:34`); RS0016/RS0017/RS0037 as
  **errors**; the analyzer diagnostic-ID roster pin (A2). **The gate passes on a mutation, not a
  green build:** rename `BnButton.Label` → `BnButton.Text`, observe `error RS0016` + `error
  RS0017` and exit 1, revert. A green `build-test` proves nothing here — CS1591 was green and
  toothless twice.
- **Gate C — marking.** `[EditorBrowsable(Never)]` + the xmldoc line on every NOT-API type; the
  `[Experimental]` **policy** written down with `BN1xxx` reserved and the finding recorded that
  the current surface warrants **zero** uses. A gate that ships zero attributes passes only if it
  shows its work.
- **Gate D — the 1.0 criteria + conclusion.** §6's table lands as a durable doc (not only inside
  this design), the README edit (a) is made, the docs-site API-stability page (D2) and the
  compat statement (D4) are written, planning is updated, DoD #4 is marked closed, and a written
  conclusion states what was frozen, what was deliberately left un-frozen, and why.

Gates A→D are strictly sequential; A before B is the load-bearing ordering (§4: a baseline
generated before the review acquires unearned authority).

## 10. Risks / honest unknowns

- **Baselining a surface that is still moving.** `ConfigureServices` is one release old, Phase
  11.4 will touch logging (possibly adding surface), and #23 relocates `INavigationManager`. A
  baseline captures a moving target. **Mitigation:** that is exactly what `PublicAPI.Unshipped.txt`
  is *for* — the pre-1.0 posture (§4) is "visible and deliberate", not "immovable". The risk is
  real only if someone mistakes the baseline for a freeze; §4's last subsection exists to stop
  that.
- **The RS0016 flood, and the reviewer who stops reading.** Several hundred diagnostics, one
  fix-all, one enormous diff of generated `.txt`. The failure mode is not a red build — it is a
  **green one nobody read**. **Mitigation:** §4 step 3 makes the read a gate deliverable, and
  Gate A precedes it so the reviewer already knows what they expect to see.
- **`.razor`-declared parameters and generated code.** Six components declare 98 parameters
  inside `.razor` files. `PublicApiAnalyzers` analyses the compilation, so the Razor generator's
  output should be visible — but the repo has an established gotcha here: generated sources are
  not written to disk without `-p:EmitCompilerGeneratedFiles=true`, and an up-to-date build
  silently skips compilation. **This must be verified empirically at Gate B, not assumed**; the
  falsifiable check is that `BnSlider`'s 23 parameters appear in
  `PublicAPI.Unshipped.txt`. If they do not, the flood is *quietly incomplete* and the pin has a
  hole exactly where the consumer surface is widest.
- **Renderer/Runtime internals leaking publicly, and awkward to freeze.** §2.5 and §2.4 found 25
  types public for mechanical reasons. The *correct* fix (`internal` + `InternalsVisibleTo`) is
  a breaking change; the *cheap* fix (mark and document) leaves 25 types in a baseline nobody
  intends to honour. This design takes the cheap fix deliberately and records the trade as
  criterion S3. **The residual risk is real:** a consumer *can* bind to `NativeRenderer` today,
  and marking it does not stop them.
- **A baseline records the surface; it does not judge it.** `PublicAPI.Shipped.txt` will happily
  freeze a badly-named parameter forever. Nothing in this mechanism improves the API — **a bad
  API frozen is still bad, and now it is bad permanently.** This is the single strongest reason
  Gate A precedes Gate B, and the reason step 3 of §4 asks *"should this be public at all?"*
  rather than *"is this line correct?"*. If the review surfaces a rename worth doing, `0.x` is
  the only time it is cheap — do it **before** the baseline, not after.
- **`IMobileBridge`'s width.** 27 members on one interface, growing once per capability. Every
  future capability (#18's haptics/purchases/background-tasks) adds members, and under the
  consume-only policy (A4) each addition is fine — *provided the policy is written down before
  the first post-1.0 addition*, not after someone has implemented the interface.
- **The version pin.** A new `PackageReference` in six csprojs must not tempt anyone into a
  per-project `<Version>`; `PackageVersionPinTests` tooth 2 reds on that, correctly.
