**Executive summary (2026-07-12).** The shell bridge grew from 6 → 9 callbacks
(clipboard read/write + share) **in lockstep across .NET + JVM + Swift**, and — the real
DoD #6 deliverable — it grew through a **documented, versioned, size-negotiated pattern**
rather than a raw lockstep edit. `blazornative_register_bridge` gained a leading
`structSize`; the runtime **min-copies + zero-fills**, and an unsupplied slot surfaces as
`NotSupportedException`, so the bridge is now forward- **and** backward-compatible (an
old 48-byte shell interoperates with the new 72-byte runtime). Clipboard is real on both
platforms (`ClipboardManager` / `UIPasteboard`); share is real (`ACTION_SEND` /
`UIActivityViewController`), asserted at the **callback-content** bar via a test seam
(the system UI is not assertable). The **export count is unchanged at 9** — a struct grow
+ a one-symbol signature change, not an export grow. Counts: **.NET 220→230, JVM 78→79,
Android 38→40, iOS XCTest 9→12**; version **`1.4.0-phase-5.4`**.

## 1. The size-negotiated bridge as-built (design §1–3)

`blazornative_register_bridge(int structSize, BlazorNativeBridgeCallbacks*)`. The runtime
copies `min(structSize, sizeof)` bytes into a zero-initialised holder and leaves the tail
zero (`NativeShellBridge.Register`). Memory-safety, four layers:

- **min-copy** — `Buffer.MemoryCopy(source, &dest, known, toCopy)` never over-reads a
  host buffer shorter than the runtime's struct.
- **zero-fill** — `dest = default` zeroes all 72 bytes; any un-copied slot stays
  `IntPtr.Zero`.
- **clamp** — `toCopy = Math.Clamp(structSize, 0, known)`: the upper bound truncates a
  newer host's extra tail, the lower bound makes a stray negative a safe no-copy (not an
  `OverflowException`) — a self-contained invariant beside the export's `structSize > 0`
  guard.
- **`RequireSlot`** — an `IntPtr.Zero` slot → `NotSupportedException("… not supported by
  this host …")`, never a null-pointer call.

This is the DoD #6 pattern: **absent callback = capability unsupported**, forward and
backward safe. Pinned by `NativeShellBridgeTests`: the **48-byte old-shell** registration
(clipboard/share unsupported, navigate still works) and the **negative-structSize**
no-copy (everything unsupported, no throw). Documented in
[docs/bridge-extension.md](../bridge-extension.md).

## 2. The 3 callbacks + the .NET surface

`ClipboardRead@48` (`int(char*,int)` — the `CurrentRoute` buffer twin),
`ClipboardWrite@56` + `Share@64` (`int(const char*)` — the `Navigate` one-string twin).
`IMobileBridge` gained `ClipboardReadAsync` (via `InvokeBufferProtocol`),
`ClipboardWriteAsync` + `ShareAsync` (via `InvokeWithOneString`), each `RequireSlot`-
guarded; `DevHostBridge` mocks an in-memory clipboard + logs share. **`ClipboardProbe`**
(registry `"ClipboardProbe"`, the `HostEventProbe` scaffolding shape): `[Inject]
IMobileBridge`, Copy→`ClipboardWriteAsync`, Paste→`ClipboardReadAsync`→echo `BnText`,
Share→`ShareAsync(echo)`. Exception posture unchanged (host throw → -1 → `HostError`).

## 3. The platform impls

- **Android** (`AndroidShellBridge.kt`): clipboard → the system `ClipboardManager`
  (appContext service) — `setPrimaryClip(ClipData.newPlainText)` / `primaryClip.getItemAt(0).text`;
  reads run on the .NET dispatch lane (Binder, no Looper). Share → an `ACTION_SEND`
  `text/plain` Intent built by `buildShareIntent`, wrapped in `createChooser`, launched
  from **appContext** with `FLAG_ACTIVITY_NEW_TASK` (the retention rule forbids retaining
  an Activity; NEW_TASK is required from a non-Activity context). The `shareLaunchHook`
  seam captures the Intent for the test.
- **iOS** (`AppleShellBridge.swift`): clipboard → `UIPasteboard.general.string` (safe
  off-main). Share → a `UIActivityViewController` presented on the **main thread** from
  the key window's root VC (the bridge is off-main → a `DispatchQueue.main` hop); the
  `shareHook` seam captures the `activityItems` for XCTest.

**Share test bar (honest):** the system share sheet is unassertable under instrumentation
— both platforms assert **callback-fired-with-content** via the seam (`EXTRA_TEXT` + MIME
/ the activity items), not the UI.

## 6. Deviations / decisions

- **Android clipboard-read focus gate.** Android 10+ denies `getPrimaryClip` unless the
  app holds window focus. The value round-trip is genuinely assertable on-device once the
  activity has focus, so the test waits on `hasWindowFocus()` and the CI lane sets
  `hide_error_dialogs 1` (a boot ANR would otherwise hold focus). The read path is also
  proven focus-free at the .NET/JVM layers.
- **Share = callback-content bar** (both platforms) — the system sheet is not assertable;
  the seam is the honest pin.
- **Swift clipboard/share supplied as non-nil trampolines through Gates 1–2** (returning
  -1 "unsupported") so the iOS lane stayed green before Gate 3 made them real — the
  size-negotiated register supplies the slots either way.

## Decisions captured

- **Size-as-argument negotiation** (min-copy + zero-fill + clamp + `RequireSlot`) — chosen
  over a version field while three struct sites existed; the low-churn form that makes the
  bridge extensible without an export grow.
- **Absent callback = capability unsupported** — the forward/backward-compat contract all
  future host APIs follow.
- **Share via a capture seam** — the only honest, non-hanging way to assert a UI-affordance
  capability under instrumentation.
