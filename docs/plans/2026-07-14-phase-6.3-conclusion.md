## What shipped

`<BnImage Src="…" />` fetches, decodes, measures and lays out — on Android (**Coil**) and iOS
(**Kingfisher**), with identical computed frames. The `image` NodeType, stubbed since Phase 2.5,
is real: it was a widget that measured **zero** because neither shell had a source-loading path.

**No ABI change**: still 9 exports + the 72-byte bridge. `image` was already NodeType 5, `img`
already mapped to it, and `Src` rides the existing `UpdateProp` wire (it is a *prop*, not a style).
The shell fetches the bytes — there is no binary path on the wire and there does not need to be.

**Counts (all CI-asserted):** .NET **319/0** · JVM **83/0** · Android instrumented **111/111** ·
iOS XCTest **72/72**.

## The model

An image with no declared size measures **0×0** until its bytes arrive. Then the shell sets the
image, **marks the Yoga node dirty** (the 6.1 path), and the tree re-solves — so the image snaps to
its natural size and **its sibling below it moves**. A declared `Width`/`Height` short-circuits
measurement entirely: Yoga never calls the measure func, and neither the bytes nor a failure can
move that frame *even in principle*.

**The sibling is the witness, not the image.** A shell could paint the bytes and never re-solve, and
the image would still *look* right. Only the band below it reveals that no reflow happened — which
is exactly what the `markDirty` mutation produces: the image paints, the node stays 0×0, the page
never moves.

## The parity contract

Two different libraries with two different cache, decode and downsampling policies were the most
plausible way to lose the cross-platform parity this milestone spent four phases proving. So the
line was drawn where it matters and nowhere else:

> **Parity is asserted on FRAMES, not on cache internals.** Coil and Kingfisher may cache, evict and
> prefetch however they like. What must be identical is the **measured size**, **when** it is
> reported (the reflow), and what happens on **failure** and **cancellation**.

**Zero numeric divergence.** Both shells assert the same table:

| node | BEFORE the bytes | AFTER |
|---|---|---|
| fixed image (`200×120` declared) | (0, 0, **200**, **120**) | **identical — the no-reflow proof** |
| its band | y = **120** | **y = 120, unchanged** |
| intrinsic image | (0, 0, **0**, **0**) | (0, 0, **160**, **90**) |
| **its band** | y = **0** | **y = 90 — the reflow proof** |
| failing image | (0, 0, 0, 0) | (0, 0, 0, 0) |
| **its band** | y = **0** *in its parent* | **y = 0 — the failure reserved nothing** |

Plus a 40×40 image inside a `BnScrollDemo` row — **with the 6.2 frame table asserted unchanged**,
because the row's height is definite and the image's size is definite, so the measure func is never
called and neither bytes nor failure can move a frame.

## The unit rule — the divergence that no frame table could have caught

**One file pixel is one dp/pt.** A 160px PNG measures 160.

Android had to *work* for this. An `ImageView`'s intrinsic size is its drawable's size in **pixels**,
so 6.1's generic `view.measure` path reported `px / density` — **61dp** for a 160px fixture at the
AVD's 2.625. iOS could never agree: `UIImage(data:)` has `scale == 1`, so its size is already in
points and Yoga measures **160**. A divergence by a factor that **differs on every device**.

So `image` gets its own measure func on both shells, reading the **decoded pixel buffer's own
dimensions** (`bitmap.width` / `cgImage.width`) — never a density-scaled `intrinsicWidth`, never
`UIImage.size`. And **Kingfisher's documented idiom for crisp images, `.scaleFactor(UIScreen.main.scale)`,
is forbidden**: it would give `160/3 ≈ 53.3pt` against Android's 160dp, and **no single-device test in
either suite could see it**.

**Honest consequence, ledgered:** a 2×/3× asset therefore renders at 2×/3× its intended physical
size — *on both platforms, consistently*, because iOS does the same by default. React Native does
not answer this either: it auto-sizes only *bundled* assets, where the packager's `@2x`/`@3x` suffix
supplies the scale; a **network** image in RN has no intrinsic size at all. So decision 2's "this is
the RN model" is inherited for the **reflow**, not for the **unit**. The unit is un-inherited ground
this phase broke by itself, and a density-aware source set is M7's problem.

## What the reviews caught (and what that says)

Every one of these was found by an adversarial review or a mutation — never by a failing test.

1. **Two of three demo cases assert that *nothing moved*.** So a shell that asserted the AFTER table
   without awaiting completion would pass both **having loaded nothing** — and if cleartext were
   blocked, all three fetches fail, a blocked load is indistinguishable from a 404, and *only* the
   intrinsic case reddens. **A fully green suite was achievable on a device that never loaded a
   byte.** Hence: await the loader's per-node terminal callback for all three nodes, assert the
   outcomes by name, assert `Wi > 0` *positively*, and check the decoded fixture's preconditions
   before reading a single frame.
2. **A memory-cache hit completes SYNCHRONOUSLY**, from inside the patch batch — in *both* libraries.
   The re-solve had to no-op mid-batch (re-entering also cleared the `applyingBatch` guard for the
   rest of the batch, re-enabling the change → re-render loop it exists to prevent), and a spent
   handle must not be recorded. **Every test cleared the cache in setup — which is precisely what
   hid it.** The owed test mounts `/image` twice *without* clearing.
3. **A clear never bumped the generation.** `Src → null` cancels, but a cancel races its own
   completion — and with the bump sitting *after* the early return, that completion arrived, found
   the generation unchanged, and **re-inflated the node the author had just cleared**. The guard was
   designed correctly, reviewed twice, extracted into a pure function and unit-tested for four
   cases — and still had this hole, because **every test that exercised the null path awaited the
   load first**. Nothing was ever in flight when the clear arrived.
4. **`clearIfMine` re-implemented the guard inline**, so the unit test that advertises itself as
   *"the only test in 70 that can see a generation-only guard"* defended **half** the production
   code. Tell-tale: the mutation note said to apply the change *"in two places"*. **A mutation that
   must be applied twice to redden one test is a mutation whose second site is unpinned.**
5. **The unit rule's "second, independent enforcement" did not exist.** Two comments claimed a test
   asserted the shell's `UIImage.scale == 1`; the assertion only ever decoded the fixture's *own*
   bytes. Adding `.scaleFactor` left all 70 tests green.

## Frames are necessary but not sufficient

The uncomfortable result of this phase: **making the re-solve re-enter mid-batch reddens only a
layout-pass counter — every computed frame stays correct.** A real defect (a broken re-entrancy
guard) that *no frame assertion on either platform can see*.

This milestone's proof strategy has been "assert the frames on both devices," and it has been
enormously effective. But the counter exists only because a reviewer reasoned the bug into existence
*before* anyone observed it. Frames catch what the layout computes; they cannot catch **how many
times** it computed, **what re-entered while it did**, or **which handle was left dangling**. Those
need mechanism-level pins — a layout-pass count, an in-flight count, a pure guard with a unit test —
and those pins come from reasoning, not from adding more frame tests.

## Honest boundaries (ledgered)

- **`Placeholder` / `OnError` / `ContentMode` (the API)** — each changes *measurement* (does a
  placeholder measure like the image? does a failure keep the reserved space?), so each deserves its
  own design. **M7.** The `contentMode` **default** was *not* deferred: both shells are aligned to
  aspect-fit (`FIT_CENTER` / `.scaleAspectFit`), because UIKit's default is `.scaleToFill` and the
  divergence is frame-neutral — i.e. no frame table could catch it.
- **Density-aware assets** (`@2x`/`@3x`, `srcset`) — see the unit rule above. **M7.**
- **GIF / SVG / animated formats** — the `naturalSizeOf` fallback deliberately reports 0×0 and logs
  rather than returning a density-scaled number, so the unit rule cannot be violated through the back
  door when these arrive.
- **The fixture server is test-target-only.** The `/image` page in the **demo app** shows three failed
  images in every build type — there is no server in the app. (And the release network-security
  config permits no cleartext, correctly.)
- **`picker`** is still a stubbed placeholder and does not flex its children — the last framework
  container that runs its own layout.
- **Clamping** (`Wi > 300`) is unanswered by design: the fixture contract keeps images within their
  container, so the question is never asked. It is a `ContentMode` question.
