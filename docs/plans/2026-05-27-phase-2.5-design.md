## Goal

Turn the parsed `RenderFrame` stream from Phase 2.4 into real Android `View` mutations. After this phase, the sentinel `_BridgeFrameSelfTest` component renders as a real Android `TextView("frame-self-test")` inside a `LinearLayout` on the device screen ā€” not as a logcat line.

Phase 2.5's job is the **mapper** ā€” the Kotlin `RenderPatch ā†’ View` translation. Other widget types in the DoD #6 mapping table (button, input, image, scroll, picker) are wired into the `CreateNode` switch but exercised only by future components.

## Brainstorm decisions

| # | Question | Decision | Rationale |
|---|---|---|---|
| Q1 | Phase scope? | **A: minimum viable, sentinel-only.** All 7 NodeTypes wired in the create switch (so adding tests later is mechanical) but only the sentinel's path is end-to-end tested. | Phase 2.5's stated job is "Native widget mapper" ā€” the mapper, not the full UI vocabulary. The sentinel exercises the load-bearing translation. Per-type tests earn their cost when real components use them. |
| Q2 | MainActivity container shape? | **B: split-screen 60/40.** Top is `widget_root`, bottom keeps the existing green-on-black console TextView for diagnostics. | The console TextView has been the truth-source for BOOT/FRAME markers through Phase 2.0-2.4 ā€” keeping it visible during 2.5 development costs almost nothing. Phase 2.7's audit can decide full-screen vs split-screen for the polished demo. |
| Q3 | Threading model? | **B: WidgetMapper internalizes UI-thread posting.** `mapper.apply(frame)` is caller-thread-agnostic and posts the batch to the main looper internally. | The mapper already has to hold a parent View / Context, so it's the natural owner of the posting `Handler`. Keeps `AndroidPlatformInfo` and `MainActivity` thread-agnostic. Avoids Activity-leak risk of capturing in static handlers. |
| Q4 | Patch dispatch ā€” atomic or per-patch? | **A: atomic on commit.** Mapper collects patches into a list until `CommitFramePatch`, then posts the whole batch to the UI thread for application in one go. | Matches Blazor's render-batch semantics. Eliminates the in-progress frame flicker that per-patch application would create for larger frames in Phase 3+. Cost is trivial (the patches already arrive together inside one `RenderFrame`). |
| Q5 | Mapper class shape? | **A: single object with `when (patch)` switch.** No strategy pattern, no `WidgetTree` wrapper. | Phase 2.5's scope is small (4 patch types initially, 9 wire types total). Switch fits on one screen. Per-class strategy pays off only when each patch has nontrivial logic ā€” here each is 2-5 lines. Shadow-tree wrappers add bug surface. |
| Q6 | Default `view` widget? | **A: vertical `LinearLayout`.** No orientation knob yet. | Matches typical Blazor `<div>` semantics (stacked children top-to-bottom). Same default as React Native's `<View>` (flex column). YAGNI on orientation ā€” no component needs horizontal yet; Phase 3's component library is the right time. |
| Q7 | Patch-type coverage? | **B: sentinel-3 + RemoveNode.** UpdateProp / SetStyle / AttachEvent / DetachEvent / AppendChild get `Log.w` "TODO Phase 3+". | A is too strict (accidental future emissions crash). C is too speculative (UpdateProp/SetStyle need per-attribute design decisions we don't have requirements for). B threads the needle: sentinel works, future emissions don't crash, deferred handlers are visibly stubbed. |
| Q8 | Test assertion shape? | **A: structural view-tree introspection** via `ActivityScenario`. | The phase's job is the mapper ā€” asserting tree structure tests exactly that. Screenshot diff adds OS/device sensitivity that breaks tests for non-mapper reasons. Phase 2.7's audit is the right place to capture a screenshot as manual evidence. |

## Architecture & Data Flow

```
[.NET in WASM] sentinel ā†’ DispatchFrame ā†’ [FRAME] stdout
       ā†“
[WasiHost.loadAndRun on bg thread] FrameStreamParser ā†’ handlers.onFrame(frame)
       ā†“
[WidgetMapper.apply(frame)]
   collect patches until CommitFramePatch
       ā†“ post to mainLooper
[UI thread] for each patch ā†’ switch ā†’ mutate widget_root
```

## Components

### New: `src/BlazorNative.Jni/src/androidMain/kotlin/io/blazornative/shell/WidgetMapper.kt`

```kotlin
class WidgetMapper(private val context: Context, private val root: ViewGroup) {
    private val nodes = mutableMapOf<Int, View>()
    private val mainHandler = Handler(Looper.getMainLooper())
    private val pending = mutableListOf<RenderPatch>()  // background-thread-only

    fun apply(frame: RenderFrame) {
        // Called on the WasiHost boot thread. Collect patches; on CommitFrame,
        // post the batch to the main looper for atomic application.
        for (patch in frame.patches) {
            pending.add(patch)
            if (patch is RenderPatch.CommitFrame) {
                val batch = pending.toList()
                pending.clear()
                mainHandler.post { applyBatch(batch) }
            }
        }
    }

    private fun applyBatch(patches: List<RenderPatch>) {
        for (patch in patches) when (patch) {
            is RenderPatch.CreateNode  -> handleCreate(patch)
            is RenderPatch.ReplaceText -> handleReplaceText(patch)
            is RenderPatch.RemoveNode  -> handleRemove(patch)
            is RenderPatch.CommitFrame -> { /* boundary marker; no-op here */ }
            is RenderPatch.UpdateProp,
            is RenderPatch.SetStyle,
            is RenderPatch.AttachEvent,
            is RenderPatch.DetachEvent,
            is RenderPatch.AppendChild -> Log.w(TAG, "TODO Phase 3+: $patch")
        }
    }

    private fun handleCreate(p: RenderPatch.CreateNode) {
        val view: View = when (p.nodeType) {
            "view"   -> LinearLayout(context).apply { orientation = LinearLayout.VERTICAL }
            "text"   -> TextView(context)
            "button" -> Button(context)
            "input"  -> EditText(context)
            "image"  -> ImageView(context)
            "scroll" -> ScrollView(context)
            "picker" -> Spinner(context)
            else     -> { Log.w(TAG, "Unknown nodeType ${p.nodeType}"); TextView(context) }
        }
        nodes[p.nodeId] = view
        val parent = p.parentId?.let { nodes[it] as? ViewGroup } ?: root
        parent.addView(view)
    }

    private fun handleReplaceText(p: RenderPatch.ReplaceText) {
        (nodes[p.nodeId] as? TextView)?.text = p.text
    }

    private fun handleRemove(p: RenderPatch.RemoveNode) {
        val v = nodes.remove(p.nodeId) ?: return
        (v.parent as? ViewGroup)?.removeView(v)
    }

    private companion object { const val TAG = "BlazorNative.WidgetMapper" }
}
```

## .NET-Side Change

The renderer currently emits child `CreateNodePatch`es without `ParentId`. If the mapper attached every node to `root`, the sentinel's TextView would render as a sibling of the LinearLayout, not inside it. Two fixes were considered:

1. **(chosen) Fix `NativeRenderer.ProcessFrame` to populate `ParentId`.** Thread `parentNodeId: int?` through the recursion; pass `nodeId` to child calls; pass `null` from the top-level entry in `ProcessRenderTreeDiff`. ~5 lines of change.
2. Detect parent inferentially in the mapper by tracking "most recent container" ā€” a hack.

After the .NET fix, the sentinel emits:
```
CreateNode(nodeId=1, nodeType="view",  parentId=null)
CreateNode(nodeId=2, nodeType="text",  parentId=1)
ReplaceText(nodeId=2, "frame-self-test")
CommitFrame
```

**Files touched:**
- `src/BlazorNative.Renderer/NativeRenderer.cs:174-222` ā€” `ProcessFrame` signature gains `int? parentNodeId`; both `CreateNodePatch` emissions pass it; recursive call passes the current element's `nodeId` as the child's `parentNodeId`.

**Test impact:**
- `FrameSelfTestParsesAsRenderFrame` extended with: `Patches.OfType<CreateNodePatch>().Where(p => p.NodeType == "text").All(p => p.ParentId != null)`.
- BootSmoke regex still matches.
- Existing renderer tests unaffected (`CreateNodePatch.ParentId` already has a `null` default).

## Tests

### New: `src/BlazorNative.Jni/src/androidTest/kotlin/io/blazornative/shell/WidgetMapperTest.kt`

```kotlin
@RunWith(AndroidJUnit4::class)
class WidgetMapperTest {

    @Test
    fun sentinel_renders_as_linearlayout_containing_textview() {
        ActivityScenario.launch(MainActivity::class.java).use { scenario ->
            val widgetRoot = AtomicReference<FrameLayout>()
            val deadline = System.currentTimeMillis() + 10_000
            while (System.currentTimeMillis() < deadline) {
                scenario.onActivity { act ->
                    widgetRoot.set(act.findViewById(R.id.widget_root))
                }
                val root = widgetRoot.get()
                if (root != null && root.childCount > 0) break
                Thread.sleep(100)
            }

            scenario.onActivity { act ->
                val root = act.findViewById<FrameLayout>(R.id.widget_root)
                assertEquals(1, root.childCount)
                val container = root.getChildAt(0) as LinearLayout
                assertEquals(1, container.childCount)
                val text = container.getChildAt(0) as TextView
                assertEquals("frame-self-test", text.text.toString())
            }
        }
    }
}
```

The existing `BootSmokeAndroidTest` (calls `WasiHost.loadAndRun` directly without launching MainActivity) is unchanged ā€” it still asserts the marker round-trip via captured stdout.

`ActivityScenario` lifecycle handling cleans up the activity after each test, avoiding state leaking across runs and giving each test a fresh activity instance.

## Risks

| # | Risk | Mitigation |
|---|---|---|
| 1 | `ActivityScenario.launch` threading quirks vs. our boot-thread pattern | Polling loop with 10s deadline + 100ms sleep is forgiving. If still flaky, fall back to instrumenting the boot thread with a CountDownLatch. |
| 2 | Phase 2.4b's 1/6 flake recurrence on this test too | Same family ā€” heap-layout dependent. Re-run if observed; document if pattern. |
| 3 | `NativeRenderer.ProcessFrame` change breaks existing renderer-tests | The change is purely additive to `CreateNodePatch` (ParentId already exists with default null). Existing tests that don't check ParentId still pass. Will verify via `dotnet test tests/BlazorNative.Renderer.Tests` mid-execution. |
| 4 | Sentinel renders to `<div>` map to `view` NodeType. The mapper's "unknown nodeType" else-arm returns TextView ā€” silent failure mode | Log.w covers it; if a component emits a NodeType we don't handle (e.g., a custom element like `<bn-something>`), we get a logged warning + a blank TextView. Acceptable for Phase 2.5 scope. |
