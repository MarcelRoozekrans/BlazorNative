# Phase 11.2 — Real-device Android validation, all capabilities, recorded (M11 DoD #2)

**Milestone:** 11 — Production Readiness. **Phase:** 11.2 (third, after 11.1). **Closes:**
DoD #2 — the app runs on the owner's **physical Android phone**; camera (real sensor + EXIF),
biometrics (real fingerprint + a hardware-backed AndroidKeyStore), geolocation (real GPS/fused
fix) and notifications (real post + tap-through, cold **and** warm) are each exercised and
**recorded** in a device-proof doc. This also **discharges the standing physical-phone ledger
item** carried since M9.

**The executable artifact of this phase is not this document.** It is
[`2026-07-21-phase-11.2-device-runbook.md`](2026-07-21-phase-11.2-device-runbook.md) — the
thing the owner opens with the phone in hand. This doc explains *why* the runbook is shaped the
way it is, sets the gates, and states the honesty boundaries up front so the runbook can stay
terse.

## The posture: owner-run, recorded, never CI

Three constraints govern everything below, and they are not negotiable:

1. **The owner runs this, by hand, over USB.** There is no automation deliverable. The phase's
   output is *evidence*, not a test suite.
2. **Results are RECORDED, not asserted.** Nothing here becomes a CI check. The milestone's CI
   posture is unchanged and explicit — *"the owner's phone is never a CI dependency"*
   (`docs/planning/MILESTONE.md`, CI-posture bullet). A FAIL on the phone becomes an issue, not
   a red build.
3. **iOS real-device stays DEFERRED.** No Apple Developer account; the milestone lists it under
   *Out of scope* ("Real-device iOS / TestFlight — still gated on the Apple Developer account").
   11.2 is Android-only, and says so rather than quietly omitting it.

## Which app goes on the phone — the sample app, and why

**Decision: the repo's own `samples/BlazorNative.SampleApp` + the `src/BlazorNative.Jni` Android
shell.** Not the 11.1 dogfood apps (`bn-baseline`, `bn-zeroalloc-showcase`).

Three grounded reasons:

- **It is the only vehicle with all four capability demos.** `SampleAppPages.All`
  (`samples/BlazorNative.SampleApp/SampleAppPages.cs:48`–`169`) declares **13 routed pages**,
  including every capability page this DoD names — `BnGeolocationDemo` (`/geolocation`, `:123`),
  `BnNotificationsDemo` (`/notifications`, `:133`), `BnSecureDemo` (`/secure` = biometrics +
  secure storage, `:145`), `BnCameraDemo` (`/camera`, `:158`) — **plus** the interaction-smoke
  set (`/layout`, `/scroll`, `/image`, `/list`, `/form`, `/modal`, `/imagepolish`). The four
  capability demos are **deliberately sample-only**: the source says so in as many words —
  *"Sample-only (the 9.0/9.1/9.2 template-minimal precedent) … BnCameraDemo joins them — NOT
  added to the template"* (`samples/BlazorNative.SampleApp/BnCameraDemo.cs:41`–`43`). A
  `dotnet new` app therefore ships **none** of them, which rules out the template-derived
  dogfood apps as the default vehicle.
- **Minimum confound.** `bn-zeroalloc-showcase` layers **11 ZeroAlloc packages** and rebuilt
  demos on top. If a real sensor misbehaves there, the first question is "framework or
  ecosystem?" — a question this phase must not have to answer. DoD #2 is an *honesty* check on
  the framework's capability layer; the vehicle should be the leanest thing that carries all
  four.
- **It is the artifact CI already exercises.** The instrumented lane, the goldens, the frame
  tables and `RouteTableDriftTests` all describe *this* app. When the phone disagrees with the
  emulator, the comparison is apples-to-apples.

**Optional second pass (explicitly out of the gates):** re-running Gate B on
`bn-zeroalloc-showcase` would additionally prove the ZeroAlloc-backed flows fire real hardware.
It is a *nice-to-have* and must not block DoD #2. If the owner has the appetite, record it as an
appendix, not as a gate.

## Build + install — the sequence, grounded

Real phones are **arm64**, so `linux-bionic-arm64` is the RID that matters. But
`verifyNativeAssets` (`src/BlazorNative.Jni/build.gradle.kts:385`–`409`) demands **both** bionic
`.so`s when `-PciSoDir` is unset (`:375`–`:378`), and `preBuild` depends on `copyRuntimeSo`
(`:425`–`:427`), so the publishes must precede gradle:

```
dotnet publish samples/BlazorNative.SampleApp -c Release -r linux-bionic-x64
dotnet publish samples/BlazorNative.SampleApp -c Release -r linux-bionic-arm64
cd src/BlazorNative.Jni && ./gradlew.bat installDebug
```

Three notes that save a round trip:

- **`cd src/BlazorNative.Jni`, not `cd android`.** The `cd android && ./gradlew` form is the
  **template's** layout (`templates/…/BlazorNative.App/android/`); the repo's gradle root is
  `src/BlazorNative.Jni/` (`gradlew` + `settings.gradle.kts` live there). `make android-build`
  does exactly this (`Makefile:45`).
- **`-PappPubRoot` does not exist in the repo's gradle.** It is **template-only**
  (`templates/…/android/build.gradle.kts:166`–`171`). The repo build hardcodes
  `runtimePubRoot = ../../samples/BlazorNative.SampleApp/bin/Release/net10.0`
  (`src/BlazorNative.Jni/build.gradle.kts:374`). The repo's only property is `-PciSoDir`
  (`:29`–`:30`) — **do not pass it**: it narrows `abiFilters` to `x86_64` (`:133`) and would
  build an APK with **no arm64 `.so`**, i.e. an APK that cannot run on the phone.
- **Env prerequisites, proven in 11.1 Gate A/D**
  (`docs/plans/2026-07-21-phase-11.1-walkthrough.md:31`–`35`): `JAVA_HOME` → a **JDK 21**;
  `ANDROID_NDK_ROOT` → the installed NDK (e.g. `%ANDROID_HOME%\ndk\26.3.11579264`).
  **`vswhere` is NOT needed here** — that prerequisite is for the `win-x64` NativeAOT link step
  only, and 11.2 never publishes `win-x64`.

`installDebug` produces a **debuggable** APK (`buildTypes.debug { isMinifyEnabled = false }`,
`:174`–`:178`), which is what makes `adb shell run-as io.blazornative.shell` viable — and that
is the *only* way to pull the camera capture out of the app's private cache dir (below).

## Reaching the capability pages — deep links are a PREREQUISITE, not a bonus

**Finding (grounding):** the sample app's default page has **no navigation to any capability
page**. `BnDemo.razor` offers exactly three buttons — `Clear`, `Theme`, `Settings →` — and the
only navigation is `NavigateToAsync("/settings")` (`BnDemo.razor:55`–`57`, `:81`);
`BnSettingsPage.razor` only goes back to `/` (`:32`, `:47`). Nothing on screen reaches
`/camera`, `/secure`, `/geolocation`, `/notifications`, `/list`, `/form`, `/modal`, `/scroll`.

So the runbook reaches every page by one of the two out-of-band routes the shell supports:

- **Deep link** — `adb shell am start -a android.intent.action.VIEW -d "blazornative://camera"`.
  `MainActivity.parseDeepLinkRoute` maps `blazornative://<host>` → `/<host>`
  (`MainActivity.kt:433`–`439`), and `loadDeepLinkRoutes()` resolves the route to a mount name
  from the **generated** `res/raw/blazornative_routes.json` (`:411`, `:180`–`:183`).
- **Mount by name** — `am start -n io.blazornative.shell/.MainActivity -e
  io.blazornative.shell.EXTRA_COMPONENT BnCameraDemo` (the `EXTRA_COMPONENT` constant,
  `MainActivity.kt:61`; the form `scripts/devloop.ps1:212`–`213` uses).

This turns the "deep link on hardware" item from a nice extra into **load-bearing
infrastructure**: Phase 11.0's generated route map has only ever been proven on an emulator and
in a build-time drift test. In 11.2 it is the mechanism by which every other capability is even
reached — so if it works, it is proven on hardware by construction, and if it does not, the
phase stops at Gate A. The `EXTRA_COMPONENT` form is the documented fallback precisely so a
deep-link failure is *isolatable* rather than blocking.

## Logging reality — state it plainly, because it will bite here

**The .NET side is silent on Android.** `BlazorNativeRuntime.kt:32`–`33` says it outright:
*"(JVM-only default — Android callers must pass a `Log`-based sink: **stderr goes to
`/dev/null` on Android, not logcat**.)"* The Activity supplies that sink —
`val onError: (String, Throwable) -> Unit = { msg, t -> Log.e(tag, msg, t) }`
(`MainActivity.kt:157`), with `tag = "BlazorNative"` (`:85`).

Consequences the runbook must state, not bury:

- **The only useful filter is `adb logcat -s BlazorNative`.** That tag carries the boot lines
  (`[BOOT] mounted <Component>` — `:202`–`:204`, and the marker `scripts/devloop.ps1:221` polls
  for), deep-link lines (`[deep-link] startup route → …` `:172`; `[deep-link] warm re-route → …`
  `:310`), the routes-resource warnings (`:413`, `:423`), and caught throwables (`:207`, `:314`,
  `:383`).
- **A managed-side failure may leave NO trace at all.** Anything the .NET runtime writes to
  `Console.Error` is discarded by the OS. If a capability misbehaves and `logcat -s
  BlazorNative` is **silent**, *that silence is itself a finding* — record it verbatim in the
  device-proof doc with the exact command and its empty output. It is direct field evidence for
  [#155](https://github.com/MarcelRoozekrans/BlazorNative/issues/155) (logging discipline,
  DoD #6 / Phase 11.4) and [#164](https://github.com/MarcelRoozekrans/BlazorNative/issues/164)
  (rc-0-on-faulted-render). 11.2 is where the blindness bites hardest, and the phase's job is to
  *document* it, not fix it — the fix is 11.4's.
- **Widen only as a fallback.** If `-s BlazorNative` is empty and something is clearly wrong,
  the second command is `adb logcat -d --pid $(adb shell pidof io.blazornative.shell)` to catch
  ART/system-side complaints (permission denials, FileProvider failures, `SIGSEGV`) — those *do*
  reach logcat because the OS writes them.

## The TEE-verification honesty problem — and how 11.2 handles it

DoD #2 asks for *"a TEE-backed AndroidKeyStore, not the AVD's software keystore."* The real
check is `KeyInfo.isInsideSecureHardware()` (API < 31) / `KeyInfo.getSecurityLevel()` (API 31+).
**Grounded finding: the app does not surface either.** `AndroidShellBridge.kt` imports `KeyInfo`
(`:31`) and reads exactly **one** property from it —
`(factory.getKeySpec(secretKey, KeyInfo::class.java) as KeyInfo).isUserAuthenticationRequired`
(`:965`) — and `BnSecureDemo` echoes only `status:<SecureStorageStatus>` / `value:<secret>`
(`BnSecureDemo.cs:60`–`61`, `:105`–`:130`). Nothing anywhere reports `isInsideSecureHardware`,
`getSecurityLevel`, or StrongBox. A repo-wide grep for those identifiers returns **zero** hits
outside the `isUserAuthenticationRequired` line.

**So 11.2 does not claim "TEE-backed" as an app-observed fact.** It records three separately
labelled things instead:

1. **What the device reports** (`adb`-only, no code change):
   `adb shell pm list features | findstr /i keystore` → presence/absence of
   `android.hardware.strongbox_keystore`; plus `adb shell getprop ro.boot.keymaster`,
   `getprop ro.hardware.keystore*`. Recorded as — verbatim wording for the doc —
   **"device reports a hardware-backed keystore (StrongBox feature: yes/no)"**. Not "the key is
   in the TEE".
2. **The negative control.** Run the *same* commands against the AVD used by the instrumented
   lane and put both columns side by side. A visible difference is the honest version of "not
   the AVD's software keystore" — it demonstrates the *device* differs from the emulator, which
   is exactly the DoD's contrast, without pretending to read the key's security level.
3. **What the app genuinely proves — the property that actually matters.** `Set` creates the key
   with `setUserAuthenticationRequired` (API 30+ pinning `AUTH_BIOMETRIC_STRONG` via
   `setUserAuthenticationParameters` — `AndroidShellBridge.kt:924`, `:946`–`:947`), and `Unlock`
   goes through `BiometricPrompt` with `setAllowedAuthenticators(BIOMETRIC_STRONG)` (`:893`).
   If, on the phone, `Set` → `status:Ok` and `Unlock` → **`value:hunter2` only after a real
   fingerprint**, then **the OS refused to decrypt without a fresh Class-3 authentication**.
   That is an *observed, end-to-end* property of OS key binding, and it is the strongest claim
   this phase can make without changing code.

**Proposed smallest honest upgrade (a DECISION for the owner, NOT part of 11.2's gates).** The
shell is already three lines from making claim #1 app-observed: at
`AndroidShellBridge.kt:959`–`965` it already has the `KeyInfo` in hand. Surfacing
`isInsideSecureHardware` / `getSecurityLevel` through the existing secure-storage payload would
turn "device reports" into "the app read the key's security level". That is a **code change**,
so it is out of scope for this docs-only phase — file it as a follow-up issue if the recorded
device report turns out to be unsatisfying. **11.2 closes DoD #2 without it**, on the precisely
worded claim above. Over-claiming here would be worse than the gap.

## Verification depth, per capability — stated before the runbook, so the runbook can be terse

| Capability | Exercised by | What is actually observable | Honest depth |
|---|---|---|---|
| **Camera** | `/camera` → `Take Photo` → the echo `captured:<W>x<H>:<bytes>` (`BnCameraDemo.cs:69`, `:119`) and the captured file rendered into the on-page `BnImage` | dims + byte size on screen; the **file** at `cacheDir/blazornative_captures/capture_<ms>_<n>.jpg` (`AndroidShellBridge.kt:1404`–`:1408`, `CAPTURE_DIR_NAME` `:1504`) | **deep** — pull the file via `run-as` and read its **EXIF on the desktop**. `exiftool` will not exist on the phone; do not try. Real-sensor EXIF has `Make`/`Model`/`DateTimeOriginal`/exposure; an emulated shutter has a synthetic green frame and thin-to-absent EXIF. |
| **Biometrics** | `/secure` → `Authenticate` / `Set` / `Unlock` / `Delete` → `status:` / `value:` echoes | `BiometricStatus` ∈ {Authenticated, Failed, Cancelled, Unavailable, LockedOut, Error} (`IMobileBridge.cs:288`–`296`); `SecureStorageStatus` ∈ {Ok, NotFound, AuthFailed, Unavailable, Error} (`:313`–`319`) | **medium** — real prompt + real finger is fully observable; the TEE claim is labelled per the section above. **`BIOMETRIC_STRONG` only** (`:893`), so a Class-2 face unlock will **not** satisfy it — enrol a **fingerprint**. |
| **Geolocation** | `/geolocation` → `Locate` → `fix:<lat>,<lng>` (`BnGeolocationDemo.cs:39`, `:80`) | lat/long only. **`Accuracy` is carried by `GeolocationPosition` (`IMobileBridge.cs:210`) but is NOT surfaced in the demo's echo** — see the inconsistency note below. | **medium** — record the fix's presence and its **plausibility** (does it match where the owner is, to a few hundred metres?). Accuracy is recorded as "not observable in-app". |
| **Notifications** | `/notifications` → `Show` / `Schedule` / `Cancel`; tap-through **cold** and **warm** | `status:<NotificationStatus>` (`:52`); the landing proof is the mount-time echo `arrived:/notifications` (`:56`, `:64`) | **deep** — the tap PendingIntent is literally the 5.1 deep link (`buildTapIntent` → `blazornative://<route>`, `AndroidShellBridge.kt:1613`–`1616`), so tap-through *also* re-proves the generated route map. |
| **Interaction smoke** | `/list`, `/scroll`, `/form`, `/modal` (+ `/layout`, `/image` if time) | rendered frames, scroll behaviour, typed text echo, overlay | **shallow by design** — "does the renderer + Yoga behave on real hardware", not a golden re-assertion. Goldens stay the emulator's job. |

**A warm-tap subtlety worth pre-empting:** `arrived:/notifications` is the page's *initial*
echo, set at construction (`BnNotificationsDemo.cs:64`). If the app is already sitting on
`/notifications` when the notification is tapped, the marker is indistinguishable from the
pre-existing state. The runbook therefore **navigates away first** (deep-link to `/settings`)
before the warm tap, so the transition to `arrived:/notifications` is real evidence. The warm
path additionally logs `[deep-link] warm re-route → /notifications` (`MainActivity.kt:310`) —
capture that line as corroboration.

## Gates

- **Gate A — hardware boot.** Phone connected and authorized (`adb devices` → `device`), model /
  ABI / Android version captured, both bionic publishes green, `installDebug` succeeds, the app
  launches and `logcat -s BlazorNative` shows `[BOOT] mounted BnDemo`. **Plus the deep-link
  proof** (`blazornative://settings` opens the settings page), because every later gate depends
  on it. Failing Gate A stops the phase.
- **Gate B — the four capabilities.** Camera (EXIF-verified from a pulled file), biometrics
  (real fingerprint, the `Set`→`Unlock` OS-key pairing, plus the recorded keystore-capability
  report + AVD negative control), geolocation (a real fix, plausibility-checked), notifications
  (post + tap-through **cold** and **warm**). Each with a screenshot and/or logcat excerpt.
- **Gate C — interaction smoke + deep link.** `/list` scrolled, `/form` typed into, `/modal`
  opened, `/scroll` scrolled; every capability page reached by `blazornative://<route>` on real
  hardware (which is 11.0's generated map, confirmed off-emulator for the first time).
- **Gate D — the device-proof doc.** The fill-in template in the runbook completed with real
  values, evidence files committed under `docs/plans/evidence/2026-xx-xx-phase-11.2/`, a written
  **conclusion** (`docs/plans/<date>-phase-11.2-conclusion.md`) stating what was proven and what
  was not, planning updated, DoD #2 marked closed, the physical-phone ledger item discharged.

Gates A→D are strictly sequential. Gate B's four capabilities are independent of each other — a
FAIL on one does not block the others; it becomes an issue and the rest continue.

## Risks / honest unknowns

- **Android version gating.** On **13+ (API 33+)** the first `Show` triggers the
  `POST_NOTIFICATIONS` dialog; below 33 it does not, and `status:Granted` arrives with no prompt
  at all. Both are correct — record which happened, and record `ro.build.version.release` /
  `ro.build.version.sdk` so the result is interpretable later. `targetSdk = 34`
  (`build.gradle.kts:95`) means the phone will apply modern behaviour regardless.
- **No enrolled biometric.** With nothing enrolled, `canAuthenticate(BIOMETRIC_STRONG)`
  (`AndroidShellBridge.kt:693`–`695`) yields `Unavailable` and `Set(requireAuth: true)` /
  `Unlock` cannot demonstrate anything. **Enrol a fingerprint before starting** — it is a
  precondition, not a step. Also: **face unlock on most phones is Class 2 (WEAK)** and will not
  satisfy `BIOMETRIC_STRONG`; if the phone has only face, record `Unavailable` as a *device*
  limitation, not a framework FAIL.
- **Location off / indoors.** A cold GPS fix indoors can take minutes or never arrive; the
  demo's `LocateAsync` echoes `status:LocationUnavailable` rather than hanging
  (`BnGeolocationDemo.cs:78`–`82`). Turn location **on**, prefer **outdoors or a window**, and
  give it a warm-up. If only `LocationUnavailable` is reachable, that is a recorded NOTE with the
  environment stated — not a silent pass.
- **A device that refuses debug installs.** `adb devices` showing `unauthorized` (accept the RSA
  dialog, or `adb kill-server` and re-plug), OEM shells requiring *"Install via USB"* to be
  enabled in Developer options (Xiaomi/Oppo/Vivo notably), `INSTALL_FAILED_USER_RESTRICTED`, and
  MDM-managed devices blocking sideload. The runbook lists the recovery for each; if the phone
  is unusable, DoD #2 stalls and the ledger item stays open — it must not be closed on emulator
  evidence.
- **The logging blindness above.** The most likely bad outcome of this phase is not a crash but
  a **silent** misbehaviour with an empty logcat. Pre-committing to "silence is a finding"
  is the mitigation.
- **Storage pressure eats the capture.** Captures live in the app **cache** dir by deliberate
  choice (`file_paths.xml` header: *"Cache, not files/documents: the OS may reclaim it under
  storage pressure"*). Pull the file **immediately** after capture, before anything else.

## Out of scope for 11.2

- **iOS real-device** — deferred, no Apple account (milestone *Out of scope*).
- **FCM push** — still gated on a Firebase project; local notifications only.
- **Any code change**, including the `KeyInfo.getSecurityLevel` surfacing proposed above and
  surfacing `GeolocationPosition.Accuracy` in the demo echo. Both are candidate follow-ups.
- **Fixing the logging blindness** — DoD #6 / Phase 11.4. 11.2 supplies field evidence for it.
- **Re-asserting the golden frame tables on hardware.** The goldens are the emulator's contract;
  the smoke here is behavioural, not numeric.

## Grounding inconsistencies found while writing this

Recorded here rather than fixed (docs-only phase); each is a candidate follow-up:

1. **`GeolocationPosition.Accuracy` is parsed but never shown.** The type carries it
   (`src/BlazorNative.Core/IMobileBridge.cs:210`) and `BnGeolocationDemo.LocateAsync` formats
   **only** `fix:{Latitude},{Longitude}` (`samples/BlazorNative.SampleApp/BnGeolocationDemo.cs:80`).
   DoD #2's "record lat/long presence + accuracy" is therefore **not fully observable** with the
   app as it stands. 11.2 records accuracy as "not surfaced"; a one-line demo change would fix
   it.
2. **No `KeyInfo.isInsideSecureHardware()` / `getSecurityLevel()` anywhere in the repo** — see
   the TEE section. The shell already holds the `KeyInfo`
   (`src/BlazorNative.Jni/src/androidMain/kotlin/io/blazornative/shell/AndroidShellBridge.kt:965`).
3. **`-PappPubRoot` is template-only.** The task-level assumption that the repo build accepts it
   is wrong: `src/BlazorNative.Jni/build.gradle.kts` has only `-PciSoDir` (`:29`); `appPubRoot`
   exists solely at `templates/BlazorNative.Templates/content/BlazorNative.App/android/build.gradle.kts:171`.
   Likewise the gradle root is `src/BlazorNative.Jni/`, not `android/`.
4. **No in-app path to the capability pages.** `BnDemo.razor:55`–`57` links only to
   `/settings`. Every demo page is reachable exclusively by deep link or `EXTRA_COMPONENT` — a
   real usability observation about the sample app, worth a ledger line even though it does not
   block the phase.
