## Gate A — code: #124 precision + #125 cleanups

### #124 — `BnListWindow.Compute` exact arithmetic
`src/BlazorNative.Components/BnListWindow.cs` computes `count * itemHeight` (max-offset) and
`clamped / itemHeight` (floor/ceil row indices) in `float`. `float` is exact for integers only
to 2²⁴; `300k rows × 64dp ≈ 1.9e7` exceeds it, so the window drifts by a row+ past the
threshold — undermining the "exact arithmetic / deterministic frames" the file is built on.
- **Fix:** compute the offending intermediates in `double` (exact to 2⁵³ — beyond any real
  list), keeping the `float` public inputs. `maxOffset`, `clamped`, `firstVisible`,
  `lastVisibleExclusive` in double; the NaN guard and half-open semantics unchanged.
- **Red-first:** a test at a large count (e.g. 300k × 64dp, an offset in the drift zone)
  asserting the window matches the exact integer expectation — **fails in float, passes in
  double**.

**#124 — the example inputs do not expose the drift; real ones do (recorded because it is
counter-intuitive).** The ticket's "300k × 64dp" is a poor witness: itemHeight 64 is a power of
two, so `clamped / itemHeight` divides EXACTLY in float, and `count·itemHeight` (=19.2e6, even)
is float-exact too — the only drift at 64 is the *input* rounding of `offset`, which the fix
cannot recover because the public inputs stay `float` (`(double)offset` widens an
already-rounded value). The double fix only bites where the ARITHMETIC drifts. So the red tests
use a non-power-of-two itemHeight and float-EXACT offsets, isolating arithmetic drift the fix
corrects and matches to exact-integer math:
  · in-range: `Compute(20_000_044f, 400, 65, 2_000_000, 0)` — float `(307693,307700)`,
    double/exact `(307692,307700)` (firstVisible floor drift);
  · past-end clamp: `Compute(40_000_000f, 400, 66, 524_295, 4)` — float `(524285,524295)`,
    double/exact `(524284,524295)` (maxOffset = count·h−viewport not float-exact → bottom
    window off a row). Both failed red against the old code, pass green after.
  Latent, out of scope: `BnListDemoAndroidTest.window()` re-implements the calc in Kotlin
  `Float`. Harmless at the fixed 500×64 demo (power-of-two, float==double), but it would diverge
  from the .NET `double` at a large-count demo — revisit trigger: if the BnList demo ever moves
  off 500×64, port that helper to `Double`.

**#125.1 — FIXED.** `NativeRenderer.NarrowHandlerId(ulong)` fails loud at the same
`> int.MaxValue` boundary `Exports.DispatchEventCore` already rejects (rc 3), replacing the
silent `(int)` truncation at the attach site. Unreachable from real frames (ids count from 1),
so pinned by a direct unit test (`HandlerIdNarrowingTests`) rather than a live path — the
mutation witness the guard needs.

**#125.2 — DELETED.** `BridgeJsonContext.cs` had zero consumers (grep clean; only the file and
historical docs referenced it). Build + full suite green after removal; no test removed.

**#125.3 — FIXED.** `FrameArena.Grow` doubled capacity unbounded; past 2⁶³ the next double
wraps (to a smaller value → heap corruption on the next bump write, or to 0 → infinite loop).
The doubling+guard is extracted to `FrameArena.GrownCapacity(nuint,nuint)` so the guard is
testable without a 2⁶³-byte alloc; it throws `OutOfMemoryException` past the max. Three unit
tests, one of them the guard-bites witness.

**#125.4 — RE-LEDGERED (no source change).** Verified empirically (probe: BnButton mounted with
no `OnClick` emits **0** `AttachEventPatch`) that Blazor's diff ELIDES an `EventCallback`
attribute whose delegate is absent — no handler id is assigned and no attribute frame is
written. So BnButton's unconditional `AddAttribute("onclick", OnClick)` never produces a phantom
attach; the `if (X.HasDelegate)` guard on BnInput/BnScroll is belt-and-suspenders, not a
correctness requirement. Rather than add a redundant guard, the finding is pinned by a
regression test (`BnButton_OnClickUnsubscribed_EmitsNoAttachAtAll_BlazorElidesTheEmptyCallback`).
**Revisit trigger:** that test going red on a Blazor/ASP.NET Core upgrade — the signal that
elision stopped and the guard is now required.

**#125.5 — FIXED (guard) + parity test.** Added the Kotlin `require(!(patchCount > 0 &&
patchesPtr == null))` guard to `NativeFrameAdapter.read`, mirroring Swift `BnFrameAdapter`'s
`nullPatchesPointer` — a positive patchCount with a NULL patches pointer now fails loud with the
count instead of NPE'ing generically in the decode loop. **Template mirror:** yes —
`NativeFrameAdapter.kt` is byte-identity-pinned by `TemplateDriftTests.TemplateShellSources_
AreByteIdenticalToTheRepos`; synced byte-for-byte (`cmp` clean). Added one JVM test
(`read_positivePatchCount_withNullPatchesPointer_throwsDescriptively`, pure synthetic decode, no
dll) → JVM 119→120. **10.1-lesson grep:** no Kotlin/Swift test feeds a positive patchCount with
a NULL pointer expecting success (it would already have NPE'd), so the new throw path breaks
nothing; the Swift twin already guards. CI compiles/runs the Kotlin — it cannot be verified
locally.
