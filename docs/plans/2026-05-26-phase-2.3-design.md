# Phase 2.3 Design — `mobile_bridge` Revival: WIT-Typed `shell-platform-info` Round-Trip

**Date:** 2026-05-26
**Milestone:** M2 — First End-to-End Demo on Android
**Phase:** 2.3 (revives the `mobile_bridge` imports that Mono-AOT trimmed as dead code in Phase 2.1.0)
**Status:** Design — approved 2026-05-26
**Predecessors:** Phase 2.2 complete 2026-05-26 (commit `4ae6bfd`); Phase 2.1.0 spike (`16a8e5f`) identified the trimming + posed this phase's central question.

---

## Goal

Revive the .NET → host bridge in the slimmest way that proves the round-trip works on all three runtimes (wasmtime CLI subprocess, JVM in-process JNA, Android in-process JNA). Replace dead `[DllImport]` declarations with one `[WasmImportLinkage]`-typed partial method; author a minimal `mobile_bridge.wit` containing one interface with one function; satisfy from JVM + Android hosts with real-data `shell_platform_info` implementations; emit a new `[BOOT] bridge-ok platform-info=<json>` marker whose payload **varies across hosts** (proving the actual host responded, not a constant baked into the .wasm).

**Success criterion (GREEN CHECKPOINT):** `make android-test` shows `BootSmokeAndroidTest PASSED` with the `[BOOT] bridge-ok platform-info={"os":"Android",...}` marker present in captured stdout. JVM `testDebugUnitTest` shows the same marker shape with JVM-stub payload. Bridge call works in at least two of three runtimes; the third (.NET via wasmtime CLI) is best-effort per Q4 risk path.

## Why this phase exists

Phase 2.1.0 discovered that Mono-AOT's trimmer stripped the seven `[DllImport("mobile_bridge")]` declarations in `WasiBridge.cs` because nothing reachable from `Main` called any `shell_*` extern. The .wasm's component world had only standard `wasi:*` imports — no `mobile_bridge` surface to satisfy. Phase 2.2 deferred the revival question.

Phase 2.3 closes that loop: pick a revival strategy, make the .wasm declare `mobile_bridge` imports again, register host handlers from the JNA layer, prove the round-trip works.

## Locked decisions and rationale

| ID | Decision | Rationale |
|---|---|---|
| Q1 | Revival strategy = **B (wit-bindgen-generated typed bindings)** | A's "quick" estimate evaporates if wasi-experimental's component-adapter doesn't surface core-shape `[DllImport]` imports through the C-API; B has smaller unknowns and aligns with Phase 2.1 design's JSI-equivalent direction. wit-bindgen csharp is documented; WIT-typed imports are the model we already exercised with `wasmtime_component_linker_add_wasip2`. |
| Q2 | Implementation depth = **C (real platform_info, stubs for the rest — but only platform_info ships in 2.3)** | `shell_platform_info` is trivially real (~10 lines per host); the demonstration value of "marker payload varies across hosts" requires it. Other 6 imports stay deferred. |
| Q3 | Trigger = **A (new `[BOOT] bridge-ok platform-info=` marker in `Main` self-test)** | Mirrors Phase 2.0's `event-ok` pattern; clear failure attribution; test assertions extend trivially with one `assertContains` per BootSmoke variant. |
| Q4 | .NET BootSmoke handling = **A (optimistic; try and see what wasmtime CLI does with unsatisfied custom WIT imports)** | Empirical answer is cheap (~5 min). Don't pre-architect a Rust wrapper binary for a problem that may not exist. Documented fallbacks: drop bridge-ok assertion only, OR Phase 2.3b Rust wrapper, OR `--stub-imports` flag if available. |
| Q5 | Scope = **B (only `shell-platform-info` in Phase 2.3) + BACKLOG documentation of the 6 deferred imports** | YAGNI ruthlessly. Each future phase that needs `shell-navigate` / `shell-storage-*` / `shell-fetch` adds them additively in WIT (non-breaking). |

### Deferred imports — Phase-by-phase carryover

| Deferred import | Lands in | Why deferred |
|---|---|---|
| `shell-navigate` | Phase 2.5 (widget mapper) | Nothing to navigate to until widgets render |
| `shell-current-route` | Phase 2.5 | Pairs with navigate |
| `shell-storage-read/write/delete` | Phase 2.5+ (state persistence) | No UI state to persist yet |
| `shell-fetch` | M4 (P3 production-readiness) or earlier if a demo needs it | No network use case in M2 |

Tracked in `docs/BACKLOG.md`.

**Key flow at .wasm runtime (any host):**

```
WasiHost.loadAndRun(wasmBytes, scratchDir, handlers)
  ↓ engine + store + WASI config (Phase 2.2 unchanged)
  ↓ wasmtime_component_linker_add_wasip2(linker)  ← Phase 2.2 unchanged
  ↓ NEW: MobileBridgeRegistration(handlers).register(linker)
      ↓ wasmtime_component_linker_define_func(linker,
            "blazornative:mobile-bridge/bridge", "shell-platform-info",
            cb_platformInfo)
  ↓ wasmtime_component_linker_instantiate (resolves all imports)
  ↓ wasmtime_component_func_call(_start)
     ↓ Mono runs Main:
         Console.WriteLine("[BOOT] runtime-start")
         Console.WriteLine("[BOOT] di-ok ...")
         DispatchEventCore("self-test", "phase-2.0")
         Console.WriteLine("[BOOT] event-ok ...")
         var info = bridge.PlatformInfo
           ↓ extern call to blazornative:mobile-bridge/bridge.shell-platform-info
           ↓ wasmtime invokes cb_platformInfo on a wasmtime-owned thread:
              info = handlers.platformInfo()    (returns host's real JSON)
              writeStringResult(results, 0, info)
           ↓ string flows back into Mono-WASI managed heap
         Console.WriteLine($"[BOOT] bridge-ok platform-info={info}")
         Console.WriteLine("[BOOT] done")
```

**Per-host expected marker payload (the proof that this works):**

| Host | Expected JSON in `[BOOT] bridge-ok platform-info=` |
|---|---|
| JVM (Defaults.handlers) | `{"os":"jvm-default","note":"stub-host"}` |
| Android (AndroidPlatformInfo.handlers) | `{"os":"Android","sdk":34,"abi":"x86_64","model":"sdk_gphone64_x86_64"}` |
| wasmtime CLI | Depends on Q4 outcome (best case: stub via `--stub-imports`; lazy case: marker absent; worst case: test skipped) |

The marker payload **varying across hosts** is the proof Phase 2.3's bridge round-trip is real.

## Error Handling

Three new failure boundaries on top of Phase 2.2's three.

### Boundary 1: wit-bindgen integration (Phase 2.3.0 spike-shaped task)

| Failure | Fix |
|---|---|
| `dotnet publish` errors with "WIT file not recognized" | Try alternate MSBuild item names: `<Wit>`, `<WitBindgenInput>`, `<WasmComponentImports>` |
| Publish succeeds but no mobile_bridge import in .wasm | Verify `bridge.PlatformInfo` actually invoked from Main (red-phase catches) |
| wit-bindgen csharp generates non-compilable code | Pin known-good wit-bindgen version in setup.ps1; OR hand-author `[WasmImportLinkage]` partials matching the WIT |
| `[WasmImportLinkage]` attribute namespace differs from spec | Adopt whatever shape wit-bindgen csharp emits; the attribute's namespace varies between previews |

**Fallback if wit-bindgen csharp is unusable:** manually-authored `[WasmImportLinkage]` partials. Wit-bindgen is convenience, not hard requirement.

### Boundary 2: Host-side WIT-typed marshaling (the C-API issue we hedged on in Phase 2.1 Q5)

`shell-platform-info: func() -> string` returns only a typed string — the simplest possible WIT-typed return. Chosen deliberately for max C-API likelihood-of-support.

| Failure | Fix |
|---|---|
| `wasmtime_component_linker_define_func` returns "unsupported signature" | Investigate `vendor/wasmtime-src/examples/`. Try `_define_func_typed` vs `_define_func` variants. |
| Callback fires but `writeStringResult` corrupts val struct | Read `vendor/wasmtime-src/crates/c-api/include/wasmtime/component/val.h` for exact layout. Use `wasmtime_component_val_string_new` rather than raw buffer write. |
| Callback fires but .NET side gets garbage | UTF-8 encoding mismatch; add Log.i diagnostics in both directions |
| Crash inside wasmtime when string val tries to free itself | Lifetime mismatch — use proper ownership-transfer form of `wasmtime_component_val_string_new` |

**Fallback if Boundary 2 is genuinely unsolvable:** change WIT to `shell-platform-info: func() -> list<u8>` (byte array — C-API DOES handle these) and decode UTF-8 on .NET side. Trade elegance for shipping the GREEN CHECKPOINT.

### Boundary 3: .NET BootSmoke vs wasmtime CLI (Q4 = A risk path)

| Behavior | Outcome | Fix |
|---|---|---|
| wasmtime CLI lazy-resolves imports | Main runs until PlatformInfo call → wasmtime CLI traps | Wrap PlatformInfo in try/catch in Main, skip bridge-ok marker if fails; BootSmoke.cs drops bridge-ok assertion |
| wasmtime CLI rejects .wasm at instantiation | BootSmoke.cs fails completely | Phase 2.3b BACKLOG carryover: write Rust wrapper OR `--stub-imports` flag investigation. Mark BootSmoke.cs `[Skip("Phase 2.3b carryover")]` |
| wasmtime CLI has `--stub-imports` flag | Best case — add flag to WasmtimeRunner.cs, keep assertion | Done |

## Open risks (carried into implementation)

1. **wit-bindgen csharp maturity in 2026.** The csharp target may be less polished than wit-bindgen's Rust/Go targets. Fallback: hand-author `[WasmImportLinkage]` partials matching the WIT — proven path even if convenience tooling lags.
2. **wasmtime C-API typed-string-return shape.** Phase 2.1 explicitly hedged on this; Phase 2.3 cashes the bet. If `() -> string` fails, fallback to `() -> list<u8>` + UTF-8 decode on .NET side.
3. **wasmtime CLI vs unsatisfied custom WIT imports.** Q4 = A optimistic; three documented fallbacks. May ship with 12 instead of 13 .NET tests passing (with explicit Phase 2.3b carryover).
4. **wasi-experimental's MSBuild property name for WIT input.** Unverified; spike discovers. Falls back to manually-staged generated code if MSBuild integration is finicky.

These are all "discovered during the work" risks; each has a known fallback. Worst case for the whole phase: Phase 2.3b carries the harder bits forward, but `[BOOT] bridge-ok` ships at least in two of three runtimes.
