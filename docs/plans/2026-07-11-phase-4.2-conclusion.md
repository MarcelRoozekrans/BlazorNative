## The measured allocation number (M1 deferral closed)

`RenderWalk_IsAllocationFree_OnSteadyState` (enabled via the test-only
`NativeRenderer.TriggerRootRenderForTests` seam — `GetComponentState` →
`ComponentBase.StateHasChanged` accessor, load-time verified in
`BlazorInterop.VerifyAccessors`):

- **Measured: 295,200 B / 900 steady-state re-renders = 328 B/frame**,
  deterministic across runs (Release, .NET 10, win-x64).
- **Bound: 600 KB / 900 iterations (~2x slack)**, FrameArenaTests sibling style
  (100-iteration warmup, `GC.GetAllocatedBytesForCurrentThread`).
- Honest budget, not the M1 comment's fictional ~512 B/call zero-alloc ideal:
  the `RenderFrame` envelope + `patches.AsSpan().ToArray()` payload copy in
  `UpdateDisplayAsync` allocate per frame BY DESIGN. Pooled frame payload is
  M6+ ecosystem work if a frame-rate budget ever demands it (carryover below).

## The RouteChanged strict-posture decision (the one the DoD asked for)

Subscriber faults are **contained in strict mode too** (per-subscriber
try/catch via `GetInvocationList()`, the `DevHostBridge.RaiseNativeEvent`
pattern; every fault stderr-logged). Rationale: `StrictErrors` surfaces
**renderer contract violations** — an app listener throwing AFTER a successful
swap (screen swapped, CurrentRoute consistent) is an app bug, and converting
that success into export rc 2 would lie to the host. Same line the 3.2
dispatch capture window draws: handler exceptions are rc 2 only when they
fault the dispatch itself. Pinned by both NavigationTests variants (non-strict
rc 0 + strict contained-and-logged).

## Gate 3 review nits — recorded as decisions

1. **Batch-induced blur is suppressed by the `applyingBatch` guard.** A focused
   view removed by a re-render never fires `OnBlur` into .NET: the mapper's
   focus listener checks the batch guard, and focus loss caused by the
   renderer's own tree surgery is not a user event. This matches browser
   semantics for removed elements and keeps the dispatch stream free of
   self-inflicted echo events. Documented decision, not a gap.
2. **Focus/blur detach clears the stored side ignoring `p.handlerId`.** The
   detach arm trusts the (nodeId, eventName) key and drops whatever handlerId
   is stored — mirroring the change arm. Unreachable-wrong under the
   renderer's registry contract: the detach registry resolves the ORIGINAL
   handlerId at emission (`NativeRenderer._eventHandlers`, last-wins
   re-attach), so a detach for a node/event pair always refers to the stored
   side.
