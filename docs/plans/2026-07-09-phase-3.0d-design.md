## Objective

`HelloComponent` renders on the AVD through the NativeAOT `.so` end-to-end:
`blazornative_init` → `blazornative_register_frame_callback` → `blazornative_mount("HelloComponent")`
→ typed struct frames → Kotlin adapter → **unchanged** `WidgetMapper`.

The `[FRAME]` JSON-over-stdout transport stays alive for WasiHost (coexistence rule holds until
3.0e), but `MainActivity` and the instrumented widget tests move to the new stack — shedding the
~36s wasm cold boot and the Phase 2.4b wasmtime flake from the demo pipeline.

## Decisions locked during the brainstorm (2026-07-09)

| Decision | Choice | Rationale |
|---|---|---|
| Kotlin consumption | **Boundary adapter** with direct pointer-offset reads, materializing the existing sealed `RenderPatch` classes. WidgetMapper unchanged. | User criterion: "whichever is fastest." The C-ABI lifetime rule forces a copy during the callback regardless; the copy and deserialization become one step. The real hot cost — JNA reflection-based `Structure` mapping — is avoided by reading fields at computed offsets (`Pointer.getInt/getLong/getPointer`). A full WidgetMapper struct rewrite would still copy, adds churn, re-tests 16 instrumented tests, and is no faster. |
| Demo pipeline | **MainActivity switches to NativeAOT now.** | `WidgetMapperTest` then validates the new stack end-to-end. `BootSmokeAndroidTest` (drives WasiHost directly, no Activity) keeps wasmtime covered until 3.0e. Requires `HelloComponent` duplicated into NativeHost (it is `internal sealed` in WasiHost; dedupe happens at 3.0e when WasiHost dies). |
| ABI breadth | **Full 9-kind ABI, 6 kinds wired.** `blazornative_dispatch_event` ships dormant. | M3 DoD #2 commits to events; a 6-kind ABI would force a mid-milestone breaking struct change in 3.2. Cost: ~15 lines of dormant surface. AttachEvent/DetachEvent/AppendChild get enum values + the `AuxInt` field; adapter/mapper handle only the active 6 until 3.2. |

## C-ABI additions

### `PatchProtocolNative.cs` (new, in BlazorNative.NativeHost)

```csharp
public enum BlazorNativePatchKind : int
{
    CreateNode = 1, AppendChild, RemoveNode, UpdateProp, ReplaceText,
    SetStyle, AttachEvent, DetachEvent, CommitFrame,
}

public enum BlazorNativeNodeType : int
{ None = 0, View = 1, Text, Button, Input, Image, Scroll, Picker }

[StructLayout(LayoutKind.Sequential)]
public struct BlazorNativePatch
{
    public BlazorNativePatchKind Kind;
    public int    NodeId;
    public int    ParentNodeId;   // -1 = none
    public BlazorNativeNodeType NodeType;
    public int    AuxInt;         // AttachEvent/DetachEvent: HandlerId; AppendChild: AtIndex; else 0
    // 4 bytes implicit padding to align the pointers
    public IntPtr Text;           // const char* UTF-8, NULL if unused — callback-scoped
    public IntPtr PropName;
    public IntPtr PropValue;
}

[StructLayout(LayoutKind.Sequential)]
public struct BlazorNativeFrame
{
    public IntPtr Patches;        // BlazorNativePatch* — native-owned
    public int    PatchCount;
    public int    FrameId;
    public long   TimestampMs;
}
```

`NodeType` stays an int enum on the wire; the Kotlin adapter maps it back to the existing
`"view"/"text"/…` strings WidgetMapper keys on. Struct sizes are asserted on BOTH sides
(a .NET `Marshal.SizeOf` test and a Kotlin offset-constants test) — the offset-read adapter
makes layout drift the primary new risk, so it gets explicit drift-catchers.

### New exports (`Exports.cs`)

```csharp
[UnmanagedCallersOnly(EntryPoint = "blazornative_register_frame_callback")]
public static int RegisterFrameCallback(IntPtr fnPtr);   // 0 = ok

[UnmanagedCallersOnly(EntryPoint = "blazornative_mount")]
public static unsafe int Mount(IntPtr componentNameUtf8); // 0 = ok, 1 = unknown component, 2 = mount failed

[UnmanagedCallersOnly(EntryPoint = "blazornative_dispatch_event")]
public static unsafe int DispatchEvent(ulong handlerId, IntPtr argsJsonUtf8); // dormant until 3.2
```

`Mount` resolves the component via a small static registry (`Dictionary<string, Func<...>>`
mapping `"HelloComponent"` → the generic mount call) — statically rooted, no reflection scan.
The trim-probe export `blazornative_run_trim_probes` **stays** until 3.0e: it remains the only
on-device coverage for cascading/`[Inject]` until real components exercise them.
Version string bumps to `BlazorNative.NativeHost 0.4.0-phase-3.0d` (JVM + Android assertions updated).

### String ownership (unchanged rule, restated)

All patch strings and the patch array are native-owned and valid **only for the duration of the
callback**. Kotlin copies synchronously before returning. Per-frame `FrameArena` allocates and
releases them; no reference counting, no free callback.

## .NET side — pluggable sink, not a replaced transport

- `NativeRenderer` gets `public Action<RenderFrame>? FrameSink`. `DispatchFrame` invokes it when
  set; else falls back to today's `[FRAME]` stdout line. **WasiHost is untouched** — the fallback
  (and the JSON path) delete in 3.0e. The `Frames` event (test channel) is unaffected.
- NativeHost sets the sink at `RegisterFrameCallback` time to a marshaller that:
  rents a `FrameArena` (ArrayPool<byte>-backed: one buffer for the patch array, one string pool
  for UTF-8 copies) → encodes the `RenderPatch` list to `BlazorNativePatch[]` → invokes the
  registered `delegate* unmanaged[Cdecl]<BlazorNativeFrame*, void>` → releases the arena on
  return. Fully synchronous.
- Patch encoding covers all 9 record types; `CommitFramePatch` keeps its wire slot so
  WidgetMapper's batch-on-commit contract is byte-for-byte semantically identical to the JSON path.

## Kotlin side — adapter, not rewrite

- **`NativeFrameAdapter.kt`** (new, `io.blazornative.jni`): inside the JNA `Callback`, reads
  `PatchCount`, then walks the array with direct offset reads (offsets as named constants
  mirroring the C# layout, guarded by the drift test) and materializes the existing sealed
  `RenderPatch` classes into a `RenderFrame`. Unknown `Kind` values log-and-skip (forward compat).
- **`BlazorNativeRuntime.kt`** (new, thin): `start(handlers): Session` wrapping
  `Native.load` → `blazornative_init` → `register_frame_callback` → `mount("HelloComponent")`,
  surfacing status/version/error as strings for the console pane.
- **`MainActivity`** swaps `WasiHost.loadAndRun(bytes, cacheDir)` for `BlazorNativeRuntime.start`;
  the green-on-black console pane now shows boot/version/mount status lines. The widget pane and
  `WidgetMapper` wiring are unchanged.
- `WasmtimeBindings.kt` / `WasiHost.kt` / `FrameStreamParser.kt` remain in place, exercised by the
  existing wasmtime tests until 3.0e.

## Gates (timebox ~4–5 days)

| Gate | Scope | GREEN bar |
|---|---|---|
| **1** | Protocol types + `FrameArena` (.NET, TDD) | Unit tests: synthetic `RenderFrame` → struct array → decoded back, all 9 kinds round-trip via an in-process test callback; arena rent/release leak-free across 1000 frames; `Marshal.SizeOf` layout assertions. |
| **2** | Desktop end-to-end (JVM) | `mount(HelloComponent)` inside the win-x64 .dll fires the JNA callback; a golden test asserts the adapter's patch list equals the JSON path's patch list for the same Hello render. Kotlin offset-drift test green. |
| **3** | Android cutover | Both bionic RIDs republished; MainActivity on the new stack; `WidgetMapperTest` + full instrumented suite pass on the AVD. |
| **4** | Regression sweep + docs | All three surfaces green including wasmtime coexistence tests; version-string assertions updated; conclusion doc + ROADMAP/MILESTONE updates. |

RED-on-ambiguity rule continues: one diagnostic iteration per failure, then stop and assess.

## Risks

| Risk | Mitigation | If it fires |
|---|---|---|
| Offset-read layout drift (.NET struct vs Kotlin constants) | Dual drift tests (Marshal.SizeOf + Kotlin offset asserts); padding made explicit in the C# comment | Fix offsets; if recurring, fall back to JNA `Structure` mapping (slower, self-describing) |
| String lifetime SIGSEGV (R5 from the base design) | Ownership rule + adapter copies inside the callback; no pointer escapes | Gate 2/3 crash → audit adapter escape paths |
| Arena overhead (R4) | ArrayPool-backed; measure at Gate 2, log if >2× JSON path — not a RED | Log as 3.0d follow-up work |
| Mount-registry trim behavior | Registry is static lambdas over the annotated `MountAsync<T>` — same rooted path Gate 4 (3.0c) proved | Trim probe pattern: add explicit `[DynamicDependency]` |
| MainActivity cutover breaks instrumented baseline | WidgetMapper untouched; only the frame SOURCE changes; wasmtime boot tests keep their own coverage | One iteration; if structural, keep MainActivity dual-path behind a flag and RED the cutover portion only |
