# Phase 7.2 Design — The `onScroll` Wire + `BnList`

**Date:** 2026-07-15
**Milestone:** M7 — Components + Razor (closes DoD #3)
**Predecessor:** Phase 7.1 — the app is authored in `.razor` (PR #94)
**Owns:** the milestone's second named risk — **the first 60Hz producer on an event wire designed
for taps** — and the component it exists for: `BnList`, the virtualized list.

---

## Decisions (locked at brainstorm, 2026-07-15)

1. **Shell-side conflation on the existing wire.** Scroll position reaches .NET as a normal
   `dispatch_event`, conflated by the shell to **at most one per committed frame** — latest
   offset wins, older ones dropped. No ABI change, no new export.
2. **Windowing in .NET over `BnScroll`** (the RN VirtualizedList model). .NET computes the
   visible window from the conflated offset and renders only window+overscan rows; recycling
   comes free from Blazor's keyed diffing + the shells' existing create/remove. No new NodeType;
   the shells stay dumb.
3. **Fixed `ItemHeight` required in 7.2.** Window math is exact arithmetic; content size is
   `Count × ItemHeight`; every frame is predictable and assertable on both shells. Variable
   heights are **ledgered** with the named design problem (estimate→measure→correct).
4. **7.2 owns `@key` reorders — empirically first.** Step one is to *observe* what Blazor's diff
   emits when a keyed window slides (it may resolve to insert/remove). If permutations appear,
   implement the `PermutationListEntry`/`End` arms; if not, document the finding and keep 7.0's
   loud violation arm. **No guessing — the 6.2 lesson (a mechanism nobody tested is a mechanism
   nobody knows) is the law here.**

## The wire contract (normative, mirrored in both shells)

| Rule | Behaviour |
|---|---|
| **Event** | name `scroll`, payload the content offset in dp/pt (the 6.3 unit rule), on the existing `dispatch_event` wire — attached only when the app subscribed (`OnScroll` present), like every other handler |
| **Conflation** | the shell keeps ONE pending offset per scroll node. A new native scroll sample **replaces** it; it is dispatched at most once per committed frame / dispatch completion. **Never queue** — scroll position is idempotent state, not an event log |
| **Backpressure** | if .NET is mid-dispatch, the sample conflates into the pending slot; the shell dispatches the freshest value when the lane frees. A slow consumer sees **fewer, fresher** events — never a growing queue |
| **Ordering** | a conflated scroll dispatch must not overtake a user-input event already queued for the same node (taps and change events keep their order) |
| **Units** | offsets in dp/pt (Android divides px by density at the source — one conversion site, the 6.1 rule) |
| **Throughput evidence** | the phase commits a measurement: a scripted fling on each platform, recording samples-seen vs events-dispatched — the conflation ratio is the proof the wire held |

## `BnList` (the component)

```razor
<BnList Items="@people" ItemHeight="64" Overscan="4" TItem="Person">
    <RowTemplate Context="p">…</RowTemplate>
</BnList>
```

- Renders a `BnScroll` whose content is: **a leading spacer** (`WindowStart × ItemHeight` tall),
  the window's rows (each `Height = ItemHeight`, keyed by item), **a trailing spacer**
  (`(Count − WindowEnd) × ItemHeight`). Spacers over absolute positioning: two extra views, no
  per-row `position:absolute`, and the content node's height stays `Count × ItemHeight` by
  construction — the same synthetic-content arithmetic 6.2 pinned.
- On a conflated `scroll` event: recompute the window; if it moved, re-render — rows enter/leave
  via keyed diffing. `@key` = the item, so row state (focus, input text) travels with the item.
- The window calc is a **pure static function** (offset, viewport, itemHeight, count, overscan) →
  (start, end) — unit-tested exhaustively; the component consumes it.

## The proof surface: `/list` (`BnListDemo`, authored in `.razor` — everything is now)

**500 rows × 64dp** in a 300×400 viewport, overscan 4:

- **Liveness is a counted assertion, not a feeling**: at any offset, live rows =
  `ceil(400/64) + 2×4 ± 1` (≈15) — asserted from the .NET node count AND the shells' child
  count (the content view's children = spacers + window).
- Content size = `500 × 64 = 32,000dp`; spacer heights exact per offset — the frame tables,
  same discipline as every M6 page.
- **Scroll → window slides**: drive the offset on-device; assert rows entered/left and the
  spacers resized; the wire's conflation ratio recorded.
- **Row state travels**: an input inside a keyed row keeps its text when the window slides
  (the reason `@key`-by-item was chosen).

## Gates

| Gate | GREEN bar |
|---|---|
| **1 — .NET: the window + the empirical `@key` answer** | The pure window function (exhaustive unit tests); `BnList` + `BnListDemo` + `/list` route + goldens; **the empirical PermutationList finding committed** (what Blazor's diff emits when a keyed window slides) and, if permutations appear, the renderer arms implemented red-first with the shells' `move-child` need assessed. `OnScroll` param on `BnScroll` (the wire's .NET end). |
| **2 — Android** | Conflation per the wire contract (`ScrollView.onScrollChanged` → pending slot → one dispatch per lane-free/frame); the `/list` frame + liveness assertions on the AVD; the throughput measurement; a scripted fling. |
| **3 — iOS** | The same via `UIScrollView` (delegate/`contentOffset` observation → conflation) — **same numbers, same liveness counts**; the throughput measurement. Yoga headers stay out of Swift (the 6.0 rule). |
| **4 — Close** | Conclusion (the wire contract as-proven + the conflation ratios + the `@key` finding + the ledger: variable heights, `scrollTo`/offset-restore, horizontal lists); ROADMAP/MILESTONE (DoD #3); counts; PR. |

## Risks

| Risk | Mitigation |
|---|---|
| The dispatch lane saturates anyway (hardening issue #9's territory) | Conflation bounds the rate at one-per-dispatch-completion by construction; the throughput measurement is the evidence; #9 gets re-examined with real numbers, as the milestone promised |
| Window-slide emits permutations the renderer can't handle | Gate 1 answers it empirically before anything is built on it; the 7.0 `default:` arm guarantees it cannot fail silently while we look |
| Re-render per window move is too slow (jank) | The window only changes every `ItemHeight` dp of travel (64dp ≈ every 4–8 frames of fling); overscan absorbs the latency; the fling test is the judge |
| Spacer resize thrashes Yoga | Two `SetStyle(height)` patches per window move — trivial next to the row diffs; the frame assertions catch any drift |
| A scroll event arriving after the list unmounts | The 6.3 stale-callback discipline: the handler is attached/detached on the existing wire; a detached handler is a no-op by construction |
