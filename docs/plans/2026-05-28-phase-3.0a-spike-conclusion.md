   The trimmer suggests `Type.GetType` for string-literal lookups.

#### Library-deep — Microsoft.AspNetCore.Components (Gate 2.6 decision: TrimmerRootDescriptor vs. accept vs. annotate-call-site)

All four are **`IL2072`** (return value of `System.Object.GetType()` not flowing `DynamicallyAccessedMembers.All` into a downstream API requiring it). Origin = `ILLink` (no path into our `src/`). These match the design's "library-deep, decide TrimmerRootDescriptor vs. accept" category.

**Important nuance for Gate 2.6 (code-review finding 2026-05-28):** these IL2072 warnings are a **different failure shape** than the `RenderTreeFrame.*Field` scenario the plan's Task 2.6 decision matrix targeted. The plan reserved TrimmerRootDescriptor for cases where the trimmer strips an internal Blazor field that our wrapper indirectly reads. IL2072 instead is "trimmer can't statically prove that `Object.GetType()`'s return value carries `DynamicallyAccessedMembers.All` annotations." Different mitigation:
- **Annotate at the call site** if the path originates in our code (e.g., we call `someComponent.GetType()` and pass to a `Type` param requiring `[DynamicallyAccessedMembers(All)]`).
- **Accept** if the flow is purely library-internal (Microsoft hasn't annotated their own `SetProperties` path yet; we can't fix it from outside).
- **TrimmerRootDescriptor** only if we determine the underlying Blazor types must remain whole-preserved AND the previous two options don't apply.

Gate 2.6 should treat this as a real decision based on call-site inspection, not a foregone conclusion in either direction.

4. `Microsoft.AspNetCore.Components.ComponentFactory.PerformPropertyInjection(IServiceProvider, IComponent)`

### Task 2.6 decision matrix — per-warning analysis

Briefing 1 framework: for each IL2072, determine call origin. If `Object.GetType()` flow originates in OUR code → annotate. If purely library-internal (Microsoft's path) → accept. TrimmerRootDescriptor only if path is library-internal AND we can articulate a concrete failure scenario root-descriptor prevents that "accept" doesn't.

**All 4 warnings share the same call-origin shape:** the originating frame is a Microsoft.AspNetCore.Components method, and the `Object.GetType()` call is on a `ComponentState` / `IComponent` / `Object` instance held by Blazor's internal pipeline — not a value we hand-construct and pass through with attribute flow. Our annotations on `TComponent` propagate `DynamicallyAccessedMembers.All` through `InstantiateComponent(typeof(TComponent))` into the constructed component, but Blazor's internal callsites re-derive the Type via `someComponent.GetType()` where the trimmer's static flow analysis can no longer prove the original annotation.

**Root-descriptor analysis:** TrimmerRootDescriptor preserves TYPES from being trimmed; it does NOT add `[DynamicallyAccessedMembers]` annotations to method signatures. The IL2072 warnings are about the trimmer being unable to statically prove that a Type-as-data value carries `DynamicallyAccessedMemberTypes.All`. A root descriptor cannot silence IL2072 because the warning isn't about a missing type — it's about missing flow annotations on a Microsoft API. We cannot articulate any concrete failure scenario that root-descriptor would prevent and accept does not. Therefore root-descriptor is the wrong tool.

| # | Warning origin | Source of `Object.GetType()` | Decision | Rationale |
|---|---|---|---|---|
| 1 | `Microsoft.AspNetCore.Components.CascadingParameterState.FindCascadingParameters(ComponentState, Boolean&)` | Internal: `ComponentState.Component.GetType()` (or equivalent) inside Blazor's cascading-parameter resolver | **ACCEPT** | Path is purely library-internal. Our `TComponent` annotation already roots the component's members; Blazor's `FindCascadingParameters` re-derives Type internally — we have no callsite to annotate. Root-descriptor doesn't add flow annotations. |
| 2 | `Microsoft.AspNetCore.Components.ComponentFactory.PerformPropertyInjection(IServiceProvider, IComponent)` | Internal: `component.GetType()` on the `IComponent` arg | **ACCEPT** | Same shape. `PerformPropertyInjection` is called by Blazor's component-instantiation pipeline; the `IComponent` it receives ultimately traces back to `InstantiateComponent(typeof(TComponent))` which IS annotated, but the warning is on Blazor's internal Type re-derivation flow, not ours. |
| 3 | `Microsoft.AspNetCore.Components.Reflection.ComponentProperties.SetProperties(ParameterView&, Object)` → `WritersForType..ctor(Type)` | Internal: `target.GetType()` inside Blazor's parameter-setter cache | **ACCEPT** | Library-internal Type flow. `SetProperties` is invoked by Blazor's render loop; the `Object target` is a `IComponent` we instantiated, but again the static trimmer can't propagate our annotation through the `.GetType()` re-derivation. |
| 4 | `Microsoft.AspNetCore.Components.Reflection.ComponentProperties.SetProperties(ParameterView&, Object)` → `ThrowForUnknownIncomingParameterName(Type, String)` | Same as #3 | **ACCEPT** | Identical shape; just a different downstream callee in the same Blazor method. |

**Summary:** all 4 are library-internal Type-flow warnings that Microsoft hasn't annotated on their own boundaries yet. We cannot fix them from outside the assembly. They are documented runtime risks that Phase 3.0b (NativeAOT) will surface concretely if they matter — at which point the right fix is upstream PRs to dotnet/aspnetcore, not a band-aid root-descriptor. For Phase 3.0a's Mono-AOT scope, accepting is the honest answer.

## Carryover into Phase 3.0b

### Runtime smoke-test checklist for the 4 ACCEPTed IL2072 warnings

The 4 library-deep IL2072s were accepted because root-descriptor is the wrong tool and we have no annotatable boundary. On Mono-AOT they remain warnings; on NativeAOT they MAY throw at runtime if the trimmer drops the property setters the underlying call paths reflect over. Phase 3.0b's `BlazorNative.Runtime.Tests` should specifically exercise these three call paths on a non-trivial component (not just `HelloComponent`):

1. **`ComponentProperties.SetProperties` (warnings #5 + #6 in Gate 1 inventory)** — fires on every component's `[Parameter]` setter binding. Smoke-test: a component with several `[Parameter]` properties of mixed types, mounted via `Mount<T>` with a `ParameterView` containing values for each. Assert the values reach the component.
2. **`CascadingParameterState.FindCascadingParameters` (warning #3)** — fires when a component declares `[CascadingParameter]`. Smoke-test: a parent providing `CascadingValue<T>` wrapping a child with `[CascadingParameter] T Value { get; set; }`. Assert the child receives the value.
3. **`ComponentFactory.PerformPropertyInjection` (warning #4)** — fires when a component uses `[Inject]` for DI services. Smoke-test: a component with `[Inject] IServiceA Service { get; set; }`, mounted from a DI graph that supplies `IServiceA`. Assert `Service` is non-null.

These tests are NOT Phase 3.0a's responsibility — `HelloComponent` plus our 3 mount overloads is sufficient for 3.0a. But Phase 3.0b Gate 2 (NativeAOT runtime works) must add them, because they're the specific call paths Phase 3.0a couldn't fix and Phase 3.0b will validate.

### Suppression status

The `[UnconditionalSuppressMessage("Trimming", "IL2057")]` on `GetEventFieldInfoType` is currently the ONLY trim suppression. Gate 2 originally also added IL2026 defensively; code review confirmed `Type.GetType(string, bool)` doesn't carry `[RequiresUnreferencedCode]` and the IL2026 suppression was therefore dead code. Dropped in a Gate 2 polish commit. If a future Blazor release annotates `EventFieldInfo` with `[RequiresUnreferencedCode]`, IL2026 will surface and we add the suppression then with concrete justification.
