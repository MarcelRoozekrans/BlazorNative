### Categorized warning inventory

#### Renderer-touching (our code — fixable in Gate 2 Tasks 2.1-2.2)

1. **`IL2087`** on `BlazorNative.Renderer.NativeRenderer.Mount<TComponent>(ParameterView)`
   at `src/BlazorNative.Renderer/NativeRenderer.cs:122`
   — `componentType` argument does not satisfy `DynamicallyAccessedMemberTypes.All` in call to
   `Microsoft.AspNetCore.Components.RenderTree.Renderer.InstantiateComponent(Type)`. The generic
   parameter `TComponent` of `Mount<TComponent>` does not have matching annotations.
   **Fix in Gate 2 Task 2.1:** add `[DynamicallyAccessedMembers(DynamicallyAccessedMemberTypes.All)]` to `TComponent`.

2. **`IL2026`** on `BlazorNative.Renderer.BlazorInterop.VerifyAccessors()`
   at `src/BlazorNative.Renderer/BlazorInterop.cs:67`
   — using member `System.Reflection.Assembly.GetType(String)` which has `RequiresUnreferencedCodeAttribute`.
   The trimmer suggests `Type.GetType` for string-literal lookups.
   **Fix in Gate 2 Task 2.2:** hoist type name to a const + switch to `Type.GetType` with `[UnconditionalSuppressMessage]`.

#### Library-deep — Microsoft.AspNetCore.Components (Gate 2.6 decision: TrimmerRootDescriptor vs. accept)

All four are **`IL2072`** (return value of `System.Object.GetType()` not flowing `DynamicallyAccessedMembers.All` into a downstream API requiring it). Origin = `ILLink` (no path into our `src/`). These match the design's "library-deep, decide TrimmerRootDescriptor vs. accept" category.

3. `Microsoft.AspNetCore.Components.CascadingParameterState.FindCascadingParameters(ComponentState, Boolean&)`
   — `componentType` arg into `CascadingParameterState.GetCascadingParameterInfos(Type)`. Source = `Object.GetType()`.

4. `Microsoft.AspNetCore.Components.ComponentFactory.PerformPropertyInjection(IServiceProvider, IComponent)`
   — `componentType` arg into `ComponentFactory.GetComponentTypeInfo(Type)`. Source = `Object.GetType()`.

5. `Microsoft.AspNetCore.Components.Reflection.ComponentProperties.SetProperties(ParameterView&, Object)`
   — `targetType` arg into `WritersForType..ctor(Type)`. Source = `Object.GetType()`.

6. `Microsoft.AspNetCore.Components.Reflection.ComponentProperties.SetProperties(ParameterView&, Object)`
   — `targetType` arg into `ComponentProperties.ThrowForUnknownIncomingParameterName(Type, String)`. Source = `Object.GetType()`.

### Self-review findings

- **Both csproj edits match plan spec exactly.** Diff is clean: comment + property element, inserted before closing `</PropertyGroup>`, conditional on `wasi-wasm` for WasiHost only. No drift.
- **TrimmerSingleWarn=false took effect.** Yesterday's componentize spike saw 1 aggregate IL2104; today we see 6 individual warnings — that's the disaggregation working.
- **Our-code surface area is narrower than expected.** Plan anticipated ≥3 our-code warnings; only 2 emerged (IL2087 on `Mount<T>`, IL2026 on `VerifyAccessors`). The third anticipated warning (IL2104 aggregate at our layer) was actually a library-deep IL2104 that's now fully expanded. Good news: Gate 2 fix surface in our code is small (≤2 method-level changes).
- **Library-deep warnings are all on one well-known pattern** — `Object.GetType()` → property-injection / cascading-parameter setup. These are exactly the places where the `Bn*` wrapper trim-strip risk lives in production. Gate 2.6 decision matrix points strongly toward **TrimmerRootDescriptor** to root component-type reflection metadata, NOT toward "accept", since `ComponentProperties.SetProperties` is on the hot path for every component's parameter binding. The renderer-internal `RenderTreeFrame.*Field` warnings the design feared *didn't appear* — IL2072 instead, which is a slightly different shape (return value of GetType()) but same underlying concern.
- **`MountAsync<T>` plan-question (Task 2.1 Step 4)** — only one method-level Mount<T> warning fired (line 122). The `Select-String` step Task 2.1 calls for will still be valuable but baseline indicates a single fix point.
- **Conditional wasi-wasm guard on WasiHost csproj is correct** but unconditional on Renderer is also fine. Renderer is a library — `TrimmerSingleWarn` only meaningfully applies when the trim pass runs (i.e., when WasiHost publishes against it). No risk either way.
- **No new IL3xxx (NativeAOT-specific) warnings appeared** — expected, since wasi-wasm is Mono-AOT, not NativeAOT. Plan's honest scope caveat remains true: Phase 3.0a doesn't prove NativeAOT trim safety, just Mono-AOT trim-analyzer cleanliness.
