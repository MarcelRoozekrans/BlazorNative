## The headline

**A stranger really can ship — and we now know it because one did, twice, from nuget.org.**

Three findings carry the phase.

### 1. The front door existed only on paper — until 0.4.0

The most severe finding of the whole dogfood was not a bug in the runtime. It was that
`dotnet new install BlazorNative.Templates` **had never worked for anybody**: the template pack was
deferred out of M8 and never published, while the getting-started docs told every reader it was one
command away (friction **F0**). 0.4.0 is the first release to publish it, and the phase verified
that empirically rather than trusting the pipeline — a real install on a clean machine resolved
`BlazorNative.Templates@0.4.0`. The doc claim is now *true*.

### 2. The seam works for a real consumer, proven at runtime

Gate C's unit test proved the seam in-repo. That is not the same claim as "an app author's service
reaches an app author's page in a published NativeAOT binary," so Gate D proved that one separately.
An ABI harness (`C:\Projects\Dogfood\showcase-runproof`) **P/Invoked the published NativeAOT
`BlazorNative.Runtime.dll`** and replayed the Android shell's own sequence — `blazornative_init` →
`register_frame_callback` → `mount` — dumping the first frame's `ReplaceText` patches. Those patches
carried the **app's own service output**: `Hello, world!` from `IGreetingService`, and
`app DI is live — 1 greeting(s) so far` from a *second* service holding a constructor dependency on
the **same singleton** — the count is what proves instance identity rather than two accidental
copies. All **6 pages mounted `rc = 0`**, including a `/fetch` page whose dependency is a **4-deep
chain**: resilience proxy → fetcher → generated REST client → the framework's bridge-backed
`HttpClient`.

Without the seam, those resolves throw inside `ComponentFactory.PerformPropertyInjection` and mount
returns **2**. The delta is the proof.

### 3. The ecosystem layers trim-clean — 11 in, 1 out, and the failure was not a trim failure

The 11.1 design set one yardstick: a stock BlazorNative publish emits **exactly 4 accepted IL2072s**
(Blazor's own reflection internals — `ComponentProperties` ×2, `CascadingParameterState`,
`ComponentFactory`), and the showcase must add **none**. With **11 ZeroAlloc packages** in — Inject,
Collections, AsyncEvents, Rest, Resilience, Serialisation, StateMachine, Validation (+ Generator),
Results, Mediator, Telemetry — the count was **exactly 4 on `win-x64`, `linux-bionic-arm64` and
`linux-bionic-x64`. Zero new trim or AOT warnings.** Both packages the design flagged as *stretch*
and most at risk — **Mediator and Telemetry — passed.**

The two named integration risks were mitigated as designed and verified absent: **no
`Microsoft.CodeAnalysis` diamond** (the three mandatory pins — Inject 1.7.1, Collections 1.1.2,
AsyncEvents 1.1.2 — did their job) and **no duplicate-generator emit** (the private generator
reference kept it off the framework's transitive copy). What remained was analyzer-level only, not
trim: 3× `ZSM0003` informational and one `ZAM008` advising exactly what the app already did.

**One package dropped: `ZeroAlloc.Cache` 1.1.15** — and the honest detail is that it **never reached
the trim gate**. It fails at `csc`, with 14 × CS1061, from two upstream defects (a generator not
gated on `[Cache]`, emitting proxies for every interface in the compilation; and an emitted
registration extension calling `AddTransient`/`GetRequiredService` **unqualified** while every other
symbol in the same file is `global::`-qualified). Reported as
[ZeroAlloc.Cache#87](https://github.com/ZeroAlloc-Net/ZeroAlloc.Cache/issues/87). It said **nothing**
about that package's AOT-cleanliness — the compile never got far enough to ask.

> **Update (2026-07-21): resolved upstream, and the question the drop could not answer is now
> answered.** The maintainer shipped **1.1.16** the same day, and it was re-verified against this
> same app: the generator emits **exactly one** proxy (the `[Cache]`-marked interface alone — the
> three that generated spuriously now generate nothing), the emitted DI extension carries its
> `using` and is `global::`-qualified, and `dotnet publish -r win-x64` reports **exactly the 4
> accepted IL2072s and zero new trim/AOT warnings**. So `ZeroAlloc.Cache` **is** AOT-clean in this
> stack, and the showcase layers **12 of 12** packages with the trim proof unchanged. The phase's
> verdict does not move — Gate D passed on 11 — but the ledger should not be read as a standing
> mark against the package.

The design also required each package to be *used*, not merely referenced: `HeapPooledList<string>`
behind the list store with `AsReadOnlySpan()` rendering and `Dispose()` returning the rented array;
AsyncEvents fanning out "item added" to two awaited subscribers; `[Validate] SignupRequest` driving
a generated validator iterated as `foreach (ref readonly var f in result.Failures)`;
`[ZeroAllocSerializable] AppSettings` through an injected `ISerializer<T>` into `SetSecretAsync`; a
camera `CaptureFlow` state machine (Idle→Requesting→Capturing→Captured/Failed) whose `TryFire`
prevents a double-fire; `[ZeroAllocRestClient] IPostApi` behind a
`[Retry(3, 200ms, jitter)] [Timeout(10s)]` resilience proxy; `[Instrument]` over `IDeviceProbe`.

### The one item that touches product behaviour — F8

A wrong page parameter (`@bind-Value` where the property is `Checked`) threw, the renderer logged to
stderr, **and `mount` still returned `rc = 0`** with a partially-rendered frame. Investigated, and it
is **not a bug**: `NativeRenderer.StrictErrors` defaults false and `HandleException` rethrows only
under strict — the code itself calls this "the deliberate production POC posture"
(`src/BlazorNative.Renderer/NativeRenderer.cs:80`–`87`, `:423`, `:452`). It is a **design gap**, and
it composes badly with a fact from DoD #6's own problem statement: **Android discards the .NET
runtime's `Console.Error`**. So on a real device this is a silently half-broken screen with **no
diagnostic anywhere**, and `rc = 0` is the framework telling the shell everything is fine. Filed as
[#164](https://github.com/MarcelRoozekrans/BlazorNative/issues/164), cross-referencing
[#155](https://github.com/MarcelRoozekrans/BlazorNative/issues/155); the fix belongs to **Phase 11.4
/ DoD #6**, where the shells get one level-gated sink and this posture is decided alongside it.

## Honesty boundaries

Stated plainly, because a dogfood that over-claims is worse than no dogfood.

- **UI events were never dispatched.** Pages **mount and render their initial frame**. Form submit,
  list add, camera capture, fetch button all compile and are wired; **none were fired.**
  `OnInitialized` / `OnAfterRenderAsync` **did** run — that is precisely how the app-service strings
  reached the frame — but nothing past first render was exercised.
- **No real device, and no capability actually fired.** Every capability call ran against the
  **DevHost bridge** (`PlatformKind.DevHost`). The APK was **built but not installed**. Real
  hardware and real camera / biometrics / geolocation / notifications are **Phase 11.2 / DoD #2**;
  this phase makes no claim on them and must not be read as making one.
- **iOS-sim was not attempted** — deferred by an explicit prior decision, not by omission.
- **`rc = 0` is not a health check** — see F8 / #164. A partially-rendered, faulted page returns 0.
- **The showcase's ZeroAlloc results are about *this* composition** — the versions pinned here,
  through this framework's ILC compile. They are strong evidence and not a general warranty.

  DoD #3 is a build that does not.

  file changed, so no `TemplateDriftTests` re-run and no Docusaurus build is triggered by it.
