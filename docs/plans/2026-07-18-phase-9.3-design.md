# Phase 9.3 Design — Camera Photo Capture (the 9.0 ABI's fourth free reuse — and the image crosses as a PATH, not bytes)

**Date:** 2026-07-18
**Milestone:** M9 — Host APIs (Platform Breadth). Owns **DoD #5 (camera photo capture:
capture a photo via the native capture UI, hand the image across the wire, permission
story, emulator's fake feed in CI + the owner's phone as the real proof)** — the **last
M9 capability**.
**Branch:** `phase-9.3-camera` (at main's tip `0bb3865`).
**Predecessor:** Phase 9.2 — biometrics + secure storage
([conclusion](2026-07-18-phase-9.2-conclusion.md); DoD #4 closed, PR #128, the M5
secure-storage deferral retired). 9.0 grew the ABI **once, generically** (op-enum on
`HostCallBegin@72` + `blazornative_host_call_complete`) on the explicit bet that
*9.1/9.2/9.3 add an op constant and touch the ABI at nothing else.* **9.1 (notifications)
held the bet; 9.2 (biometrics + secure storage) held it a second and third time — bridge
still 80 bytes / 10 exports, the iOS struct-grow mutant failing to COMPILE both times.
Phase 9.3 is the fourth draw, and the strong prior is it holds again — with ZERO ABI
change.**

**But camera raises a genuinely new question the first three did not: the RESULT is an
IMAGE — potentially megabytes.** Geolocation returned a tiny `{lat,lng}`; notifications a
status; secure storage a small string under an 8 KB cap. A photo is 1–10 MB of JPEG. **The
headline design decision is therefore NOT the ABI (that is settled: zero change) — it is
HOW THE IMAGE CROSSES THE WIRE: a FILE PATH the shell writes and the app reads, or the
BYTES inline as base64. This is the phase's real design work, and §2 argues it hard.**

This is a **design-only** document. It states the decisions, the normative rules, the gate
structure, and the PROVEN/UNPROVEN split. Implementation is later work.

---

## 0. THE HEADLINE — camera fits with ZERO ABI change, because the image crosses as a PATH

**Stated first and loudly: camera photo capture adds ONE op-enum value (`Camera = 4`) +
one wire-mirrored status enum (`CameraStatus`) + a host handler on both shells + one
`BlazorNative.Device` façade (`ICamera`, the 5th) — and touches the ABI at NOTHING else. No
struct grow (stays 80 bytes / 10 slots), no new export (stays 10), no new binary/buffer
protocol, no drift-pin move, no export-gate arithmetic. The 9.0 "pay once, reuse thrice"
bet pays its fourth draw.**

The whole reason the ABI does not move is **decision 2**: the completion payload carries a
**small file path**, not the megabytes. Walk the shape:

| Operation | Shape | Rides | ABI touch |
|---|---|---|---|
| **capture** (options) | request → (permission) → OS camera UI → user shoots or cancels → shell writes a temp JPEG → path | `HostCallBegin(id, op:Camera, argsJson{action:"capture","maxDim":…,"quality":…})` → `host_call_complete(id, CameraStatus, payloadJson{"path":…,"width":…,"height":…,"bytes":…})` **on Captured** | none — op value + JSON vocabulary |
| **check** (availability) | read-only "is a camera present + usable?" | same, `argsJson{action:"check"}` → `host_call_complete(id, CameraStatus, null)` | none |

**The thing a reader expects to force the ABI — the multi-MB image — and why it does not:**
`host_call_complete` is `int(long requestId, int status, const char* payloadJsonUtf8)`. It
**already** carries an optional flat-JSON payload, and it has since 9.0: geolocation returns
`{"lat":…,"lng":…}`, secure `get` returns `{"value":…}`. Camera returns
`{"path":"file:///…","width":4032,"height":3024,"bytes":218374}` — **a few hundred bytes of
JSON naming a file, NOT the file's contents.** The bytes never touch the wire; the OS wrote
them to disk and .NET is handed their address. The wire has carried a small flat-JSON
payload for three phases; this is the fourth capability to use it, not a new capability of
the wire. **The image being large does not make the *message* large — that is the entire
point of decision 2, and it is why the ABI stays frozen.**

**The reuse proof is made falsifiable, not asserted** (the 9.1/9.2 precedent).
`CameraAbiUnchangedTests` pins `Marshal.SizeOf<BlazorNativeBridgeCallbacks>() == 80`,
`OffsetOf(HostCallBegin) == 72`, and `Camera == 4`, **inside this phase's suite** so the
claim lives next to the code it guards. The named reuse-proof mutation — assert-81-bytes —
reds the .NET pin; the iOS struct-grow mutant is expected to **fail to COMPILE** again (the
Swift type system forbidding a silent grow, 9.1/9.2's strongest proof).

## 1. Decision — one async completion, riding the EXISTING generic begin/complete pair

**Capture rides the EXISTING `HostCallBegin@72` + `blazornative_host_call_complete`, keyed
by requestId, resolved off-lane — verbatim the geolocation/notifications/secure machinery.**
No new mechanism, no new slot, no new export. `NativeShellBridge.CapturePhotoAsync` calls
the existing `InvokeHostCallAsync(HostCallOp.Camera, argsJson, ct)`
(`NativeShellBridge.cs:430`) — the pending registry (`s_pendingHostCalls`), the
`CancellationToken` posture, the unknown-id benignness are all **reused, not re-written**.

**Camera is the most obviously-async capability of M9.** It suspends the app behind a
full-screen system camera UI for an arbitrary user interaction — structurally identical to a
permission dialog (9.0's whole reason for the async completion), only longer-lived. The
completion arrives on a background continuation after the user shoots or cancels; the
awaiting `ValueTask` always resolves with a status (§3's denial-as-data law). There is no
temptation to a synchronous slot here — capturing a photo across a system activity cannot
ride a buffer-read.

## 2. Decision — THE IMAGE HANDOFF: a FILE PATH, not bytes inline (⛔ VETO-FLAGGED — the phase's real decision)

**The shell captures the photo to a temp JPEG file and returns its `file://` PATH in the
completion payload. The app reads the file (or hands the path straight to `BnImage` as
`src` — §6). The bytes NEVER cross the C-ABI. This is RECOMMENDED and is the default;
base64-bytes-inline is REJECTED.** This is the headline decision of the phase; argue it.

### 2a. The two options

- **(a) File path (chosen).** The shell writes the captured JPEG to an app-private temp
  file (Android app **cache dir**; iOS `NSTemporaryDirectory()`), and the payload is
  `{"path":"file:///…","width":…,"height":…,"bytes":…}`. The **file is the handoff**; the
  payload is a small string.
  - **Small message.** A path + three integers is a few hundred bytes — the exact shape the
    wire already carries (geolocation's fix, secure `get`'s value). Zero marshalling of
    megabytes, zero multi-MB `.NET` string, zero multi-MB base64 intermediate.
  - **Composition falls out for free.** `BnImage`'s `Src` is a URL the *shell* fetches
    (Coil on Android, Kingfisher on iOS — both load `file://` and `content://` locals). A
    captured photo's `file://` path IS a valid `BnImage.Src`: the capability the app just
    exercised feeds directly into the M7 image component (§6). Bytes-inline would force the
    app to re-materialise a data-URI or a byte stream a loader can't consume as cleanly.
  - **The honest cost — a path leaks a filesystem location.** The returned path points into
    the app's **private** cache (`getCacheDir()` / `NSTemporaryDirectory()`) — not
    world-readable, not on external storage, gone when the app's sandbox is cleared. It is a
    location, not a secret; the exposure is far smaller than the multi-MB in-memory copies
    bytes-inline would create. And it introduces a **lifetime** question (§2c) bytes do not.

- **(b) Bytes inline as base64 (rejected).** Base64 the JPEG into the payload
  (`{"data":"/9j/4AAQ…"}`). Self-contained (no file, no cleanup, no path leak) — and that is
  its only virtue. Against it:
  - **Multi-MB through a flat-JSON `const char*`.** A 3 MB JPEG is ~4 MB base64; that string
    is marshalled host→native, parsed into an immutable, GC-movable, **non-zeroable** .NET
    `string`, and held in the pending-registry `TaskCompletionSource` until the continuation
    runs — **the 9.2 secret-in-memory hazard, ×1000**, for a photo that isn't even a secret.
    Peak memory holds several full copies (host String, marshalled buffer, .NET string,
    decoded bitmap) simultaneously.
  - **It contradicts the 8 KB precedent.** Secure storage caps a value at
    `SecretResult.MaxValueBytes = 8 * 1024` precisely because "the payload is for small data,
    not blobs." A photo is ~1000× that cap. Routing a blob through the same channel is
    exactly the misuse that cap exists to forbid.
  - **It buys nothing the app wants.** The app's end goal is to *display* or *upload* the
    photo; both are better served by a file path (BnImage loads it; an upload streams it)
    than by a giant base64 string it must decode.

### 2b. Why NOT a new binary export / buffer protocol (the ABI-growing alternative, rejected)

One could imagine a *new* export or a byte-buffer struct slot to hand raw bytes efficiently
(the shape a "real" image pipeline might use). **That WOULD grow the ABI** — a new symbol or
a new slot, the exact event 9.0 paid once and swore off for the rest of M9. The file path
makes it **unnecessary**: the OS already wrote the bytes to disk; a pointer to that file is
all .NET needs. **Rejected on the spot** — a blob export is a deliberate, argued future
milestone's work if a bytes-in-memory pipeline is ever needed, never a silent grow snuck in
here. This is why §0 can say "zero ABI change" and mean it: **the image crosses as a path,
so the struct and the export set do not move.**

### 2c. The temp-file lifetime (the one thing the path option owes that bytes would not)

A file handoff introduces a file that must eventually die. The rule, stated once and
normative:

- **Location.** The shell writes to a **dedicated subdirectory of the app's cache dir**
  (Android `File(cacheDir, "blazornative_captures")`; iOS a `blazornative_captures` folder
  under `NSTemporaryDirectory()`). Cache — not files/documents — so the OS may reclaim it
  under storage pressure and it is never backed up.
- **Ownership after handoff.** Once the path crosses to .NET, **the app owns the file.** The
  façade does not auto-delete on return (the app may still be reading it, or handing it to
  BnImage which decodes asynchronously). The `/camera` demo (§6) demonstrates the honest
  lifecycle: display the photo, then the app is free to delete it.
- **The backstop — the shell prunes its own capture dir.** So a crashed or careless app
  cannot leak captures forever, **each `capture` first prunes the capture subdir** to a
  bounded budget (keep the last N, e.g. 3, and/or drop entries older than a TTL). This is a
  shell-side safety net, not a correctness guarantee the app can lean on. Documented as a
  decision.
- **A `Delete(path)` affordance is deferred, not shipped.** The app can delete a
  cache-dir file with ordinary .NET I/O; a façade `DeleteAsync` would be a new op for no new
  capability. If a consumer wants a bridge-mediated delete (e.g. to zero across the boundary)
  it is a labelled follow-up. **For 9.3 the app owns the file and the shell prunes as a
  backstop.**

### 2d. Resolution, downscale, and JPEG quality (a `CaptureOptions` bag)

`CapturePhotoAsync` takes an options bag so the app controls the file's size:

- **`MaxDimension` (default a sensible cap, ~2048 px).** The Intent path (§4) hands control
  of the sensor resolution to the system camera app, which writes a **full-resolution** JPEG
  to the output URI (often 4000×3000+, several MB). The shell then **downsamples** the
  written file to `MaxDimension` on the long edge (decode-bounds → sampled decode → re-encode)
  before returning the path. A `null`/`0` `MaxDimension` means "keep full resolution"
  (explicit opt-in to the big file).
- **`Quality` (default ~85).** The JPEG re-encode quality. Default trades a couple hundred KB
  for visual fidelity a phone screen can't out-resolve.
- **The result reports what was actually written** — `width`/`height`/`bytes` in the payload
  are the *final* file's, after downscale, so the app and BnImage measure reality, not the
  sensor's raw dimensions.

### 2e. EXIF orientation — the shell NORMALIZES (bakes rotation into pixels, resets the tag)

A phone photo is very often stored **rotated with an EXIF orientation tag** (portrait shots
are commonly a landscape buffer + "rotate 90°"). Decision: **the shell normalizes
orientation host-side** — it applies the EXIF rotation to the pixel buffer, writes an
**upright** JPEG, and **resets the orientation tag to 1 (identity)**. Rationale:

- **A raw-file consumer gets an upright image with no EXIF literacy required.** An app that
  reads the bytes to upload doesn't have to parse EXIF; what's on disk is what's upright.
- **It avoids DOUBLE rotation with BnImage's loaders.** Coil and Kingfisher both **honor EXIF
  on decode** (the M7 image path relies on their decoders). If the shell baked rotation into
  the pixels but *left* the tag at "rotate 90°", Coil/Kingfisher would rotate a second time —
  a silent, one-platform-at-a-time bug of exactly the kind the M7 parity contract exists to
  forbid. **Bake pixels AND reset the tag → each layer rotates zero times after the shell.**
  This must be stated in the shell parity note (§6, §9 Gate 2/3).

### ⛔ VETO FLAG — file path vs bytes, and the temp-file lifetime

File-path-with-a-documented-lifetime is **RECOMMENDED and default** (§2a–2c): a small
message, the BnImage composition for free, the multi-MB in-memory hazard avoided, at the cost
of a private-cache path leak and a prune-backstop lifetime. Bytes-inline is a **labelled,
heavier** fallback only if a consumer ever needs a self-contained result with no file — and it
must be named as the blob-through-a-string-channel anti-pattern it is, never shipped as the
default. **Decide this before Gate 1 writes the primitives** — the payload key set
(`path`/`width`/`height`/`bytes` vs `data`), the `PhotoResult` shape (§5), the temp-dir
convention, and the prune backstop all follow from it. It is the reason §0's headline holds.

## 3. Decision — the status enum: `CameraStatus`, wire-mirrored, denial-as-data

**Denial/cancellation is DATA — the milestone law, restated.** Capture's completion
`int status` is a **wire-mirrored `CameraStatus`**, defined byte-identically across .NET /
Kotlin / Swift (pinned like the struct). The host maps each platform's native outcome into
it host-side; .NET only ever sees the integer. **A user cancel, a denied permission, no
camera, and a host error are all VALUES, never exceptions and never hangs** — the awaiting
`ValueTask` always resolves.

| status | Name | Meaning | Android (Intent path) | iOS (`UIImagePickerController` / `AVCaptureDevice`) |
|---|---|---|---|---|
| 0 | `Captured` | a photo was taken and written; the payload carries the path | `RESULT_OK` + a written output URI | `didFinishPickingMediaWithInfo` + a written temp file |
| 1 | `Cancelled` | the user backed out of the camera UI | `RESULT_CANCELED` | `imagePickerControllerDidCancel` |
| 2 | `Denied` | the OS refused camera access (permission) | (Intent path: not reached — see §4) | `AVAuthorizationStatus.denied` / `.restricted` |
| 3 | `Unavailable` | no camera hardware / capture UI not available | no camera app resolves the Intent | `!UIImagePickerController.isSourceTypeAvailable(.camera)` — **the honest iOS-simulator result** |
| 4 | `Error` | unexpected host error (a caught Kotlin/Swift throw, a write/encode failure, malformed args) | any other failure / caught throw | any other failure / caught throw |

An out-of-range integer maps to `Error` — still data, never a throw (the
`ToGeolocationStatus` / `ToBiometricStatus` twin). **Five values.** Note the split between
`Cancelled` (the user chose not to shoot — a normal outcome) and `Denied` (the OS refused
access — a permission outcome): the demo shows both distinctly.

## 4. Decision — the capture UI: the system-Intent path (Android) / `UIImagePickerController` (iOS)

### 4a. Android — `ACTION_IMAGE_CAPTURE` to the system camera app + a FileProvider (⛔ VETO-FLAGGED)

**Decision: the `MediaStore.ACTION_IMAGE_CAPTURE` Intent path — hand off to the system
camera app, which writes the full-res JPEG to an output URI the shell supplies via a
FileProvider. This is the honest MVP; the direct Camera2/CameraX path is REJECTED for 9.3.**

Argue it:

- **The Intent path needs NO runtime `CAMERA` permission.** The system camera app owns the
  sensor and the permission; the calling app only receives the result. This is the
  documented Android guidance and it is the *simpler and safer* path — no runtime-permission
  request, no permission-denial branch on Android at all (§5's denial-as-data still holds for
  iOS and for `Unavailable`/`Cancelled`).
- **What it DOES need — a FileProvider for the output URI.** `ACTION_IMAGE_CAPTURE` writes to
  a `content://` URI the app grants the camera app write access to. That requires a
  `<provider android:name="androidx.core.content.FileProvider">` in the manifest plus a
  `res/xml/file_paths.xml` declaring the cache subdir. **This touches the manifest mirror AND
  adds an XML resource — a new drift surface folded into Gate 2 (§9).** (`androidx.core` is
  already on the classpath from prior phases; confirm at Gate 2 whether an explicit
  `androidx.core:core` pin is needed — if so it is a gradle-mirror line like 9.2's
  `androidx.biometric`, drift-enforced the same way.)
- **The direct Camera2/CameraX path is heavier and REJECTED.** It gives an in-app camera UI
  but **requires the runtime `CAMERA` permission**, a preview surface, capture-session
  plumbing, and a much larger dependency (CameraX). For *photo capture* (the DoD's scope; video
  and in-app UI are explicitly out of scope, `MILESTONE.md:120`), the Intent path delivers the
  capability with a fraction of the surface. The in-app path is a deliberate future milestone
  if an app ever needs a custom camera UI.

### ⛔ VETO FLAG — Intent path (no CAMERA permission) vs direct Camera2/CameraX (needs CAMERA)

The Intent path is **RECOMMENDED and default**: no runtime permission, a FileProvider is the
only new manifest surface, and it fully satisfies photo capture. The direct path is a
**labelled, heavier** alternative only if an in-app camera UI is a requirement — which it is
not for M9. **Decide before Gate 2 wires the Android shell**, because the FileProvider
manifest/resource mirror and the absence-of-a-CAMERA-permission story both follow from it.

### 4b. iOS — `UIImagePickerController` with `.camera` source

`UIImagePickerController` with `sourceType = .camera` for photo capture (`PHPicker` is the
*gallery* picker — wrong capability; camera capture is `UIImagePickerController(.camera)`).
The delegate's `didFinishPickingMediaWithInfo` yields a `UIImage`; the shell writes it to the
temp file (normalized per §2e) and returns the path. `imagePickerControllerDidCancel` →
`Cancelled`.

### 4c. The interplay with SafeAreaView / edge-to-edge (the M6/M7 watch item — NOT tripped here)

The 9.0/9.1/9.2 ledgers flagged "the camera capture UI is the likeliest phase to force
SafeAreaView / edge-to-edge." **The Intent + UIImagePickerController choice means it does
NOT.** The capture UI is the **system camera app's own full-screen activity** (Android) / a
**system-presented modal** (iOS) — OS chrome the app does not lay out. The only surface the
app draws is the `/camera` demo page (a button + the resulting `BnImage`), which reuses the
existing routed-page layout. **Recorded so the watch item is explicitly not tripped by 9.3**;
if a future in-app Camera2/CameraX preview lands, edge-to-edge lands there with its named
problem.

## 5. Decision — the .NET surface (`ICamera`, the 5th façade, no 8th package)

Exactly the 9.0/9.1/9.2 split, so `DevHostBridge` mocks headless:

- **`IMobileBridge` (Core)** gains the typed primitive, so the in-process mock can serve it:
  ```csharp
  ValueTask<PhotoResult> CapturePhotoAsync(CaptureOptions options, CancellationToken ct = default);

  public enum CameraStatus { Captured, Cancelled, Denied, Unavailable, Error }

  public readonly record struct CaptureOptions(int MaxDimension = 2048, int Quality = 85);

  public readonly record struct PhotoResult(
      CameraStatus Status,
      string? Path,      // file:// path in the app cache; null unless Captured
      int Width,         // final (post-downscale) pixel dimensions; 0 unless Captured
      int Height,
      long SizeBytes);   // final file size; 0 unless Captured
  ```
  `NativeShellBridge.CapturePhotoAsync` calls the **existing** `InvokeHostCallAsync(op,
  argsJson, ct)` — the pending registry, cancellation posture, unknown-id benignness reused.
  Only the args-JSON builder, the `PhotoResult` parse (the `ParseGeolocationResult` /
  `SecretResult`-parse twin — read `{"path":…,"width":…,"height":…,"bytes":…}` on
  `Captured`), and the `ToCameraStatus` mapping (incl. out-of-range → `Error`) are new .NET.
- **`ICamera` (Device — the SAME 7th package, no 8th)** — the app-facing façade, a thin
  delegate over `IMobileBridge`, sibling to `IGeolocation` / `INotifications` / `IBiometrics`
  / `ISecureStorage`:
  ```csharp
  public interface ICamera {
      ValueTask<PhotoResult> CapturePhotoAsync(CaptureOptions options = default, CancellationToken ct = default);
      ValueTask<CameraStatus> CheckAvailabilityAsync(CancellationToken ct = default);
  }
  ```
  Registered by **`AddBlazorNativeDevice()`** — **one added line**
  (`services.AddSingleton<ICamera, Camera>()`, `DeviceServices.cs:39`). **No new package, no
  new shipped-set pin, no new csproj, no new version mirror** — the M9 device group is the
  whole reason `BlazorNative.Device` exists (its KDoc already says "and the other M9 device
  facades as they land"). **The 7th package is the last package; 9.3 confirms it needs no
  8th, closing the package roster for M9.**

### 5b. `DevHostBridge` mocks it headless — a canned test image

`DevHostBridge` grows a **settable capture result**: a `CameraStatus` and, on `Captured`, the
**path to a canned test image** the harness ships (a tiny fixture JPEG written to a temp path)
with its known `width`/`height`/`bytes`. So the headless lane drives every status —
`Captured` (returns the canned path + dims), `Cancelled`, `Denied`, `Unavailable`, `Error` —
**with no device and no camera.** This is the central proof of denial/cancel-as-data,
asserted at Gate 1 within a bounded await, before any device work. The canned path is what
lets Gate 1 also exercise the **BnImage-consumes-the-capture composition headless** (§6).

## 6. Decision — the proof surface (`/camera` demo) — the capabilities COMPOSE

A **`/camera` demo page** in SampleApp (`BnCameraDemo`, the **13th routed page** — after
`BnSecureDemo` (12th)), the worked example that `[Inject]`s `ICamera`:

- **"Take Photo"** → `CapturePhotoAsync()` → on `Captured`, **display the result in a
  `BnImage`** whose `Src` is the returned `file://` path — the M7 image component consuming
  the captured file, **the capabilities composing** (camera → file → BnImage). Show the
  `Path`, `Width`/`Height`, and `SizeBytes` beside it.
- **The cancel path** (user backs out → `Cancelled`) and the **denied/unavailable paths**
  (echoed as status, never thrown, never hung — the geolocation/notifications/secure
  discipline) are explicit demo affordances (the DoD's "show the failure paths").
- **THE M7 INTERPLAY, made a demo rule (density / ContentMode).** A captured photo has a
  **huge natural pixel size** (even downscaled to `MaxDimension` 2048, that is 2048 dp/pt of
  intrinsic size). The M7 loaders measure at **one file pixel = one dp/pt with NO
  downsampling** (Coil `Size.ORIGINAL`, no Kingfisher `scaleFactor` — `BnImageLoader.swift`).
  So an **intrinsic** BnImage of a captured photo would reflow to thousands of dp and blow the
  layout. **The demo gives the display `BnImage` an explicit `Width`+`Height` (definite) with
  `ContentMode="Contain"`** — the exact M7 sizing contract (definite → never measured → no
  reflow; Contain → aspect-fit inside the box, paint-only), now consumed by a *real* photo
  rather than a fixture. This is the M6/M7 "revisit ContentMode with a real natural-size
  image" ledger item, discharged here as a demo rule and recorded. (The **density-assets**
  ledger item is **NOT** tripped: a captured photo is a *runtime file*, not a bundled `@2x`
  asset — the bundled-asset density story remains ledgered.)
- **Sample-only** (the 9.0/9.1/9.2 template-minimal precedent): `BnGeolocationDemo` /
  `BnNotificationsDemo` / `BnSecureDemo` are sample-only because the template tree is pinned
  minimal; `BnCameraDemo` joins them. Registration touches the **one page-authority**
  (`SampleAppPages.All`) and its **pinned mirror** `DEEP_LINK_COMPONENTS` (`"/camera" →
  "BnCameraDemo"`, both the sample's and — for route-table parity — the template's map), and
  `RouteTableDriftTests` re-greens pair-for-pair. **No new template *page*.**

3. **The temp-file-lifetime note (§2c) and the EXIF-normalization rule (§2e)**, lifted into
   the doc as the two host-side obligations the file-path handoff carries.

## 8. Security / resource note — a file path and a temp file (the new hazards, small)

The image handoff introduces **no secret on the wire** (a photo is not a secret; contrast
9.2). The residual hazards are small and named:

- **A private-cache path crosses the intra-process C-ABI** as a UTF-8 string. It points into
  the app's own sandboxed cache — not world-readable, not external storage. The exposure is a
  location, not content; acceptable and documented.
- **The temp file lives until the app deletes it or the shell prunes** (§2c). The
  prune-on-capture backstop bounds the leak; the app owns the file after handoff.
- **The decoded bitmap in memory** (BnImage's decode) is the real memory cost, and it is the
  M7 image path's existing concern, bounded by the `MaxDimension` downscale (§2d) — a
  2048-px-capped JPEG decodes to a few MB, not tens. Recorded, not new.
- **No zeroable-buffer concern** (there is no secret to zero) — the 9.2 secret-in-memory
  hazard does **not** recur here precisely because the bytes never enter managed memory as a
  marshalled string; they stay in a file the OS wrote. This is a *benefit* of the file-path
  decision worth stating.

## 9. Gates + the honesty split

## 10. Risks (led by the image-handoff, the temp-file lifetime, and the sim-has-no-camera questions)

| Risk | Mitigation |
|---|---|
| **THE IMAGE HANDOFF — a multi-MB photo could force bytes-inline (or a new binary export), growing the ABI** (the phase's real question) | §2: **file path, not bytes.** The shell writes the JPEG to an app-cache temp file and the payload carries the **path** (a few hundred bytes) — the exact small-flat-JSON channel geolocation/secure `get` already use since 9.0. Bytes-inline is rejected (multi-MB through a `const char*`, a non-zeroable .NET string, the 9.2 in-memory hazard ×1000, ~1000× the 8 KB precedent). A new binary/buffer export is rejected as the one ABI grow M9 swore off (§2b). **The image crosses as a path, so the ABI stays 80 bytes / 10 exports** — asserted UNCHANGED at Gate 1. ⛔ Decide before Gate 1 (the payload key set + `PhotoResult` follow). |
| **The temp-file lifetime — who deletes it, and can it leak?** | §2c: written to a dedicated **cache** subdir (OS-reclaimable, not backed up); **the app owns it after handoff** (no auto-delete on return — BnImage decodes async); **the shell prunes the capture subdir on each capture** (keep-last-N / TTL) as a backstop so a crashed app can't leak forever. A bridge-mediated `Delete` is a deferred, labelled follow-up. Documented as a decision. |
| **The iOS simulator has NO camera at all** (a NEW honesty shape — worse than 9.2's no-Enclave) | §9 Gate 3: `UIImagePickerController(.camera)` is genuinely unavailable on the sim, so **`check → Unavailable` is the HONEST simulator result**, asserted as such — not smoothed over. The capture round-trip is **seam-mocked** (a canned image + the delegate seam, the 9.2 construction-seam precedent) and **labeled** simulator-scoped. A real iOS capture is doubly-gated UNPROVEN (no sim camera AND no Apple account). |
| **EXIF orientation → a rotated or double-rotated photo** | §2e: the shell **normalizes** — bakes the rotation into the pixels AND **resets the EXIF tag to identity**, so a raw-file consumer gets an upright image and Coil/Kingfisher (which honor EXIF on decode) do **not** rotate a second time. A named Gate-2 mutation (leave the tag un-reset → double-rotation reds). The real-sensor rotation is physical-phone UNPROVEN (the emulator's synthetic frame is un-rotated). |
| **The M7 intrinsic-size explosion — a captured photo's natural size is thousands of dp** | §6: the M7 loaders measure at one-file-pixel-per-dp with NO downsampling (Coil `Size.ORIGINAL`, no Kingfisher `scaleFactor`), so an **intrinsic** BnImage of a photo reflows huge. The demo displays it in a **definite (Width+Height) BnImage with `ContentMode="Contain"`** — the exact M7 sizing contract (never measured, aspect-fit, paint-only). Discharges the M6/M7 "revisit ContentMode with a real natural-size image" ledger item. The `MaxDimension` downscale (§2d) additionally bounds the decoded bitmap. |
| **The FileProvider drifts the template manifest + adds an XML resource** (a NEW class of drift this phase) | §9 Gate 2's process ledger is **widened**: sync the `<provider>` into BOTH `AndroidManifest.xml` files, the `res/xml/file_paths.xml` into both trees, any `androidx.core` gradle pin (if needed) into both `build.gradle.kts` (9.2's drift-enforced pattern), and **run `TemplateDriftTests` + `RouteTableDriftTests` before pushing**. 9.1 moved the manifest for a permission; a `<provider>` + a resource is more surface. |
| **Does the value-returning `capture` (a payload) force the ABI?** | **No** — same answer as secure `get`. `host_call_complete` already carries an optional flat-JSON payload (geolocation since 9.0, secure `get` since 9.2). Camera returns `{"path",…}` on `Captured`; the wire has carried a payload for three phases. Not a new wire capability. |
| **The op-enum grow is a hidden ABI change** | It is not: adding `Camera = 4` to the value carried in the existing `int op` field is wire vocabulary — the struct stays 80 bytes, the exports stay 10. `CameraAbiUnchangedTests`' `== 80` / offset-72 / `Camera == 4` / 10-export pins are the tripwire, asserted UNCHANGED (the falsifiable headline). |
| **Choosing the direct Camera2/CameraX path would need a runtime `CAMERA` permission + a large dependency** | §4a: **rejected** for 9.3 — the `ACTION_IMAGE_CAPTURE` Intent path needs **no runtime `CAMERA` permission** (the system camera app owns it) and only a FileProvider. It fully satisfies *photo capture* (the DoD's scope; in-app UI / video are out of scope). The in-app path is a deliberate future milestone if ever needed. ⛔ Decide before Gate 2 (the FileProvider + no-permission story follows). |
| **SafeAreaView / edge-to-edge — flagged in 9.0/9.1/9.2 as the camera phase's likely trigger** | §4c: **9.3 does NOT trip it** — the capture UI is the system camera app's own full-screen activity (Android) / a system modal (iOS), OS chrome the app does not lay out; the `/camera` demo reuses the existing routed-page layout. | **The AVD's camera is synthetic — camera is barely emulated** | The PROVEN/UNPROVEN split names it (§9): the emulator returns a synthetic scene (not a lens), un-rotated; the **real sensor, the real camera-app UI, real EXIF off a real capture, and real full-res files** are the **owner's physical Android phone** (documented `adb`-over-USB steps, the honesty check). iOS has no sim camera at all — seam-mocked + labeled. |

## 11. Constraints honored (the standing laws)

- **The ABI does NOT grow — the reuse the 9.0 export event bought (fourth draw).**
  `HostCallOp.Camera` is an op-enum value; `CameraStatus` is wire vocabulary; the
  value-returning `capture` reuses the existing payload channel — **and the image crosses as a
  PATH, not bytes, so no binary/buffer export is added**. **No struct grow (80 bytes), no new
  export (10), no drift-pin move, no export-gate arithmetic** — asserted UNCHANGED at Gate 1
  (§0, §9). The phase's headline, falsifiable; the iOS struct-grow mutant expected to fail to
  COMPILE.
- **Status/denial is DATA**, never an exception/hang (§3, normative — the `CameraStatus`
  matrix within a bounded await; cancel, denied, unavailable, error all as values).
- **Wire vocabulary mirrored 3 ways** — the op value + the `CameraStatus` enum on
  .NET / Kotlin / Swift; the `DEEP_LINK_COMPONENTS` `/camera` route mirror pinned by
  `RouteTableDriftTests`.
- **A FileProvider `<provider>` + XML resource (+ any gradle pin) touches the template mirrors —
  synced in-gate** (§4a, Gate 2): the manifest `<provider>`, the `res/xml/file_paths.xml`, and
  any `androidx.core` line land in both trees, drift-checked before push (a NEW class of
  manifest+resource drift this milestone; the 9.2 gradle-drift discipline widened).
- **Counts with provenance** — baseline **.NET 717 / JVM 115 / instrumented 201 / iOS 218**;
  deltas mutation-verified at close.
- **Five required contexts unchanged** (`build-test`, `android-build`, `ios-build`,
  `pr-title`, `footer-check`); advisory device lanes unchanged; the owner's phone is never a
  CI dependency.
- **No 8th package** — `ICamera` is a sibling of `IGeolocation`/`INotifications`/`IBiometrics`/
  `ISecureStorage` in the existing 7th package `BlazorNative.Device`; `AddBlazorNativeDevice`

- **The capabilities compose** — a captured `file://` path feeds the M7 `BnImage` as `Src`
  (§6); the M6/M7 ContentMode-with-a-real-image ledger item discharges here, and the
  density-assets (bundled `@2x`) item is explicitly NOT tripped (a capture is a runtime file).
