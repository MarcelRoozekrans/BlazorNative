## Rule-by-rule fate

**Retired (6)** — recorded under `### Removed Rules` in `AnalyzerReleases.Shipped.md`
release `4.1.0`; **IDs are never reused**. The WASI premise ("this API throws on WASI
Preview 1") died with the Mono-WASI runtime — NativeAOT has real threads, a real
thread pool, real file I/O:

| ID | Was | Why retired |
|---|---|---|
| BN0001 | `new Thread(...)` forbidden | Real threads exist |
| BN0002 | `Task.Run(...)` forbidden | Thread pool exists; offloading is legal |
| BN0003 | `Parallel.*` forbidden | Multi-core parallelism works |
| BN0005 | Sync primitives flagged | Locks are meaningful again |
| BN0006 | `[ThreadStatic]` flagged | **Now actively FALSE** — `FrameArena` (PatchProtocolNative.cs) uses `[ThreadStatic]` deliberately as its thread-cached arena design |
| BN0012 | `System.IO.File/Directory/Path` flagged | File I/O works in the app sandbox |

**Rescoped survivors (5)** — categories `BlazorNative.WasiCompatibility` →
`BlazorNative.MobilePolicy` / `BlazorNative.Interop`; help links moved off the dead
`github.com/your-org/...` to `docs/analyzers.md#bn00xx` anchors:

| ID | Old → New severity | Old → New home | Rescoped meaning |
|---|---|---|---|
| BN0004 | Error → **Warning** | WasiCompatibility → MobilePolicy (`MobilePolicyAnalyzer`) | `Thread.Sleep` stalls the single Kotlin dispatch lane (advisory, not "will throw") |
| BN0010 | Error → **Warning** | WasiCompatibility → MobilePolicy | Raw `System.Net.Sockets.*` bypasses the host bridge — ride `FetchAsync`/`AddBlazorNativeHttp()` |
| BN0011 | Warning → Warning | WasiCompatibility → MobilePolicy | **Narrowed to the parameterless ctor only** — `new HttpClient(handler)` stays legal (BlazorNative.Http constructs over `BridgeHttpHandler`) |
| BN0013 | Error → Error | WasiCompatibility → MobilePolicy | Process APIs unsupported in Android app sandboxes (correctness) |
| BN0014 | Error → Error | WasiCompatibility → Interop (`BridgeAsyncHandlerAnalyzer`) | Reworded off "Mono-WASI single-threaded scheduler": handlers run synchronously inside a native callback window; an async handler is fire-and-forget whose exceptions vanish. Detection logic unchanged — the 3 Phase 2.0 markup tests (ID-asserting, no message text) passed throughout |

**New (2)** — `InteropBoundaryAnalyzer`, both Error, category `BlazorNative.Interop`:

| ID | Rule |
|---|---|
| BN0020 | Exceptions must not escape `[UnmanagedCallersOnly]` — body must be a single top-level try with a catch-all (bare `catch` / unfiltered `catch (Exception)`), no throw in any catch clause. UB under NativeAOT (process abort on Android); pins the rc-code contract Exports.cs follows |
| BN0021 | Explicit C ABI — both `EntryPoint = "..."` and `CallConvs = new[] { typeof(CallConvCdecl) }` required; the JNA host binds by name to cdecl, so implicit callconv / unnamed exports are silent ABI drift vectors (static + blittability stay compiler-enforced, CS8894 family) |

File shape: `WasiThreadingAnalyzer.cs` + `WasiBclGapsAnalyzer.cs` deleted;
survivors consolidated into `MobilePolicyAnalyzer.cs`; `BridgeAsyncHandlerAnalyzer.cs`
reworded in place; new `InteropBoundaryAnalyzer.cs`.

## Review-driven false-negative fixes (`fc3ad34`, TDD)

1. **Target-typed `new()` evasion:** `MobilePolicyAnalyzer` registered only
   `ObjectCreationExpression`, so `HttpClient c = new();` evaded BN0010/0011/0013
   entirely. Fixed by also registering `ImplicitObjectCreationExpression` (handler
   over `BaseObjectCreationExpressionSyntax`). This fix immediately caught the one
   real hit in src/ at Gate 2 (below).
2. **BN0020 specific-catch rethrow hole:** `catch (IOException) { throw; }` beside a
   catch-all passed the rule, but the rethrow escapes the C-ABI — a catch-all sibling
   does not intercept it. Tightened: a throw in **any** catch clause of the top-level
   try fires the rule.

## Release tracking + the RS2007 naming deviation

`AnalyzerReleases.Shipped.md` + `AnalyzerReleases.Unshipped.md` land as
`AdditionalFiles`; the M1-era RS2008 `<NoWarn>` is deleted — the analyzer project
builds `-warnaserror` clean. **Deviation from the plan's literal release name:**
RS2007 rejects `1.0.0-phase-4.1` — release headers must be `System.Version`-parseable
(no prerelease suffixes). Releases are therefore named by phase number —
**`2.0.0`** (backfill recording the original 11 WASI-era rules as shipped in Phase
2.0; required so the RS validators can check the 6 Removed + 5 Changed entries
against a prior release) and **`4.1.0`** (2 New / 6 Removed / 5 Changed) — with the
scheme documented in the file's `;` comment header.
