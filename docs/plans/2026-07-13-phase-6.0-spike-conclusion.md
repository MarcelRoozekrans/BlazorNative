# Phase 6.0 Conclusion — Yoga-Integration Spike (Both Shells)

**Date:** 2026-07-13
**Milestone:** M6 — Real-UI Foundation (closes DoD #1)
**Design:** [2026-07-13-phase-6.0-design.md](2026-07-13-phase-6.0-design.md)
**Branch:** `phase-6.0-yoga-spike`

## Verdict — **GREEN** (both rungs)

Yoga **3.2.1** links into both shells alongside the NativeAOT runtime artifact, both shells
compute **identical frames from the identical tree**, and the **native measure-callback
round-trip works on both in BOTH channels** — Yoga calls out to the shell to measure an
auto-sized leaf, and the width *and* the height it gets back drive the layout. That is the
load-bearing result — linking was table-stakes; measurement is what makes flexbox usable,
because a text leaf's intrinsic size can only come from platform font metrics.

Phase 6.1 (the flexbox layout core) proceeds as designed: **Yoga in the shells, no C-ABI
change** — flex props ride the existing `SetStyle` wire. The fallback ladder (managed
flexbox in .NET / native-layout mapping) is **not** taken and is now closed.

### The GREEN bar, item by item

| Bar | Android | iOS |
|---|---|---|
| Yoga links | `libyoga.so` ships in the APK (CI verifies the `x86_64` APK — see below) | `libyoga.a` linked into the app binary |
| **Coexists with the runtime lib** | ONE process loads both: the Yoga test lays out through `libyoga.so` (SoLoader) and then calls `blazornative_version()` through JNA — `libBlazorNative.Runtime.so` — and asserts the version | both static archives in one binary; the hosted XCTest runs in the app process — a green run IS the proof |
| **Measure callback fires + BOTH channels of its size are used** | `YogaMeasureFunction` → leaf's width = the measured **80** AND its height = the measured **20** | `YGNodeSetMeasureFunc` → same |
| **Identical frames** | all 12 numbers asserted | the same 12 numbers asserted |
| Test | `YogaSpikeAndroidTest` ×1 (instrumented, on the AVD) | `BnYogaTests` ×1 (hosted XCTest, iOS sim) |

**How frame parity is asserted (not assumed).** Both rungs build the *same canonical tree*
— a row container 300 × 100, direction LTR set **explicitly** on both (never left to a
platform default):

| node | style | expected frame (x, y, w, h) |
|---|---|---|
| `box1` | width 50, height 50 | **0, 0, 50, 50** |
| `box2` | `flexGrow: 1`, height 50 | **50, 0, 170, 50** |
| `text` | no width/height; measure func → 80 × 20; `alignSelf: flex-start` | **220, 0, 80, 20** |

50 + 170 + 80 = 300. Each rung asserts **all four numbers of all three frames** — the same
twelve numbers, in [`YogaSpikeAndroidTest`](../../src/BlazorNative.Jni/src/androidTest/kotlin/io/blazornative/shell/YogaSpikeAndroidTest.kt)
and [`BnYogaTests`](../../src/BlazorNative.Apple/BnHostTests/BnYogaTests.swift) — plus the
left-to-right ordering and the `measureFired` flag. Identical frames from one engine on two
platforms is therefore a *test result*, not a claim; it is the whole reason for choosing
Yoga over two native layout systems, so it is pinned like everything else in this repo.

**Why `alignSelf: flex-start` on the text leaf.** Under Yoga's default `alignItems: stretch`
the leaf's cross axis takes the row's 100 and the measured height (20) is **discarded** —
only the *width* channel of the round-trip would be proven. `flex-start` makes the frame
height the measured 20, so `text.h == 20` proves the measure func's returned **height**
reaches the frame. For 6.1 that is the channel that matters most (text wrapping / multi-line
labels in a column are exactly where measured height drives the frame); re-proving Yoga's own
stretch default would have been worth less.

Test counts: **Android instrumented 40 → 41 · iOS XCTest 13 → 14**. No .NET change, no JVM
change, no ABI change (still 9 exports + the 72-byte 9-callback bridge).

### What CI actually verifies about the Android APK

The instrumented lane's APK is **x86_64-only** (with `-PciSoDir` set, `build.gradle.kts`
narrows `abiFilters` to `x86_64` — the CI emulator is x86_64 and only the x86_64 runtime
`.so` is shipped to that job). So what CI proves is: **on x86_64, `libyoga.so` and
`libBlazorNative.Runtime.so` are both in the APK and both load in one process** (the Yoga
test exercises both). A local/release build takes the other branch of `abiFilters`
(`arm64-v8a` + `x86_64`) and the Maven Yoga artifact ships both ABIs — but *that* is
observed, not asserted, and this doc will not claim it. Phase 6.1 should decide whether an
arm64 rung is worth adding to CI.

## The pinned integration recipe — Android

**Dependency** (`src/BlazorNative.Jni/build.gradle.kts`):

```kotlin
implementation("com.facebook.yoga:yoga:3.2.1")            // the prebuilt JNI bindings RN Android uses
androidTestImplementation("com.facebook.soloader:soloader:0.10.5")
```

- `com.facebook.yoga:yoga` is the **prebuilt JNI binding** (the `YogaNode` Java API over the
  C++ core) — the same artifact React Native Android consumes. No NDK build, no JNA layer
  for Yoga: it brings its own `libyoga.so` per ABI and its own loader. The fallback in the
  design (NDK-built Yoga + JNA) was **not needed**.
- **`implementation`, not `compileOnly`** — that is what puts `libyoga.so` in the APK.
- **The soloader wrinkle.** Yoga declares `com.facebook.soloader:soloader` at *runtime*
  scope: it ships in the APK but is off the compile classpath. Anything that must call
  `SoLoader.init(context)` before touching the first `YogaNode` — i.e. the instrumented test
  — needs soloader added explicitly to its *compile* classpath, pinned to the version yoga
  3.2.1 resolves (0.10.5) so compile and runtime agree. In the app itself, initialisation
  happens once at shell start-up (6.1 wires it).

**Measure callback:** `YogaNode.setMeasureFunction { node, w, wMode, h, hMode -> YogaMeasureOutput.make(w, h) }`
— a Kotlin lambda; the JNI binding handles the native round-trip. No `@convention(c)`
constraint, so it *can* capture (it will capture the `View` to measure in 6.1).

## The pinned integration recipe — iOS

Yoga is C++ with a C API, so it embeds exactly like the runtime archive (the M5 static-`.a`
recipe is the reference). `ios.yml` builds it from source for the simulator:

```bash
curl -sSL https://github.com/facebook/yoga/archive/refs/tags/v3.2.1.tar.gz | tar xz
# compile every yoga/**/*.cpp:
xcrun -sdk iphonesimulator clang++ -arch arm64 -isysroot "$(xcrun --sdk iphonesimulator --show-sdk-path)" \
  -mios-simulator-version-min=13.0 -std=c++20 -O2 -fPIC -I"$yroot" -c "$cpp" -o "$obj"
xcrun libtool -static -o vendor/libyoga.a yoga-obj/*.o
cp -R "$yroot/yoga" vendor/yoga-include/yoga
rm -f vendor/yoga-include/yoga/module.modulemap        # ← see below
```

Link + build settings (`project.yml`, BnHost target):

```yaml
HEADER_SEARCH_PATHS:          [$(SRCROOT)/vendor/yoga-include]
CLANG_CXX_LANGUAGE_STANDARD:  c++20                    # Yoga 3.x headers need it
OTHER_LDFLAGS:                [..., $(SRCROOT)/vendor/libyoga.a, -lc++, ...]
```

- **C++20 is mandatory** — Yoga 3.x does not compile at Xcode's default standard, both for
  its own sources and for any file that includes its headers.
- **`-lc++` was already there** (the runtime archive needs it), so Yoga's C++ runtime came
  for free. **No duplicate symbols** against the NativeAOT archives — the risk the design
  named did not materialise.

### The load-bearing iOS lesson: Yoga's headers must never reach Swift

Six red CI runs, all the same failure:

```
error: bridging header dependency scanning failure:
  BlazorNativeRuntimeC.h:48:10: fatal error: 'yoga/Yoga.h' file not found
```

**Xcode's Swift explicit-module dependency scanner processes the bridging header with its
own, path-less header search. It honours neither `HEADER_SEARCH_PATHS` nor `-Xcc -I` passed
through `OTHER_SWIFT_FLAGS`.** So a `#include <yoga/Yoga.h>` in the bridging header fails
during *scanning* — before any compile — even though the ordinary Clang compile resolves it
perfectly. Two attempts to feed the scanner the path (build setting, then `-Xcc`) both
failed; removing Yoga's bundled `module.modulemap` (which made the scanner try to build a
`yoga` *module*, failing the same way) was necessary but not sufficient.

**The fix — and the rule for 6.1:** all Yoga interop lives in
[BnYogaProbe.mm](../../src/BlazorNative.Apple/BnHost/BnYogaProbe.mm) (Objective-C++), which
is an ordinary Clang compile that `HEADER_SEARCH_PATHS` *does* reach. It exposes only plain
C — a result struct and two functions — declared in its own **shell-owned** header
[BnYogaProbe.h](../../src/BlazorNative.Apple/BnHost/BnYogaProbe.h), which the bridging header
`#include`s and [BnYogaProbe.swift](../../src/BlazorNative.Apple/BnHost/BnYogaProbe.swift)
wraps. Swift never sees a Yoga header. (Only *Yoga's own* headers must stay invisible to
Swift — a plain-C header of ours behind the bridging header is fine.)

**Keep the runtime mirror a mirror.** `BlazorNativeRuntimeC.h` is the Swift-side mirror of
`src/BlazorNative.Runtime/Exports.cs` — one of three pinned mirrors of the C-ABI. The
`bn_yoga_*` symbols are **shell-owned, not runtime exports**, so they live in
`BnYogaProbe.h`, not in the mirror. Phase 6.1's full `bn_yoga_*` node-tree API grows there.

This is not a workaround so much as the discipline the shell already follows: the NativeAOT
runtime is reached through hand-declared plain C too. **Phase 6.1's iOS Yoga layer must be
Objective-C++ behind a plain-C surface** — a `bn_yoga_*` node-tree API (create/insert/set
style/set measure/calculate/read frame), with the measure callback a C function pointer that
receives an opaque context (the `UIView` to measure, passed as `void*` — Yoga's per-node
context slot). Budget that shape into 6.1's iOS work; do not re-litigate it.

## Implications for Phase 6.1

1. **The architecture holds.** Yoga in both shells, `SetStyle` carries flex props, no ABI
   change. The renderer stays thin; placement moves from `LinearLayout`/manual stacking to
   "Yoga computes frames, the shell applies them to a plain container."
2. **Measurement is real, in both channels.** Both shells can measure a leaf natively during
   layout and both the returned width and the returned height reach the frame. 6.1 replaces
   the fixed 80 × 20 stub with `View.measure(...)` / `UILabel.sizeThatFits(...)` driven off
   the node's context.
3. **iOS layout code is Objective-C++, not Swift** (see above), behind a plain-C surface in
   `BnYogaProbe.h` — *not* in the runtime C-ABI mirror. Android layout code is plain Kotlin
   against the `YogaNode` API.
4. **Yoga version pinned to 3.2.1** on both platforms — the Android Maven artifact and the
   iOS source tarball are the same release, so the two shells run the same engine version.
   Bump them together: `ci.yml` (the **required** lane) extracts the version from
   `build.gradle.kts` and from `ios.yml` and **fails the build if they differ**, so the pin
   cannot silently rot.
5. `SoLoader.init(context)` must run once at Android shell start-up before the first
   `YogaNode` (the spike does it in the test; 6.1 does it in the shell).

## Ledger

- The spike's probe surfaces (`YogaSpikeAndroidTest`, `BnYogaProbe.{h,swift,mm}`,
  `BnYogaTests`, the `AppDelegate` warm-up call) are **spike scaffolding**. 6.1 replaces the
  probes with the real layout pass; the *tests* stay (as engine-level pins) unless 6.1's own
  frame assertions subsume them — decide at 6.1 close, don't leave both.
- **The iOS lane rebuilds Yoga from source on every run** — 19 translation units compiled
  with `clang++ -O2`, then `libtool -static`, uncached. It is tolerable while the lane is
  cold (the spike touched it a handful of times), but 6.1 makes the iOS lane *hot*. Add an
  `actions/cache` keyed on `YOGA_VERSION` (+ the Xcode/SDK fingerprint) around
  `vendor/libyoga.a` + `vendor/yoga-include/` before that phase starts.
- **CI's Android coexistence proof is x86_64-only** (the instrumented APK's `abiFilters`
  narrow to x86_64 under `-PciSoDir`). If arm64 coexistence needs to be *asserted* rather
  than assumed, that is a new CI rung — decide in 6.1.
