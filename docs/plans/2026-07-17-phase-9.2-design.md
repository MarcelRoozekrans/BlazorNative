# Phase 9.2 Design — Biometrics + Secure Storage (the 9.0 ABI's third free reuse, and the M5 secure-storage deferral closes)

**Date:** 2026-07-17
**Milestone:** M9 — Host APIs (Platform Breadth). Owns **DoD #4 (biometrics +
secure storage: BiometricPrompt / LocalAuthentication gating a Keystore / Keychain-backed
store — set/get/delete secrets)**, and **closes the M5 secure-storage deferral**.
**Branch:** `phase-9.2-biometrics-storage` (at main's tip `17e6834`).
**Predecessor:** Phase 9.1 — local notifications + tap-through
([conclusion](2026-07-17-phase-9.1-conclusion.md); DoD #3 closed, PR #127). 9.0 grew the
ABI **once, generically** (op-enum on `HostCallBegin@72` + `blazornative_host_call_complete`)
on the explicit bet that *9.1/9.2/9.3 add an op constant and touch the ABI at nothing else.*
9.1 was the first draw and **held the bet exactly** (bridge stayed 80 bytes / 10 exports;
the iOS struct-grow mutant failed to COMPILE). **Phase 9.2 is the second draw — two
capabilities at once (auth + a secret store), one of which returns a payload and one of
which prompts. If anything in M9 forces the ABI, it is likeliest here. It does not.**

This is a **design-only** document. It states the decisions, the normative rules, the gate
structure, and the PROVEN/UNPROVEN split. Implementation is later work.

---

## 0. THE HEADLINE — biometrics AND secure storage fit with ZERO ABI change

**The single most important finding, stated first and loudly: biometrics + secure storage add
TWO op-enum values (`Biometrics = 2`, `SecureStorage = 3`) + two wire-mirrored status enums +
host handlers on both shells + two `BlazorNative.Device` façades — and touch the ABI at NOTHING
else. No struct grow (stays 80 bytes / 10 slots), no new export (stays 10), no drift-pin move,
no export-gate arithmetic. The 9.0 "pay once, reuse thrice" bet pays its third draw.**

Walk the shape and prove it. Every operation is an outbound request the host completes with a
status (denial/failure as data), optionally carrying a payload — the exact
`HostCallBegin`/`host_call_complete` contract geolocation opened:

| Operation | Shape | Rides | ABI touch |
|---|---|---|---|
| **biometric authenticate** (reason) | request → OS prompt → success/fail/unavail/lockout | `HostCallBegin(id, op:Biometrics, argsJson{action:"authenticate","reason":…})` → `host_call_complete(id, BiometricStatus, null)` | none — op value + JSON vocabulary |
| **secure set** (key, value, requireAuth) | request → Keystore/Keychain write → status | `HostCallBegin(id, op:SecureStorage, argsJson{action:"set","key":…,"value":…,"auth":"0\|1"})` → `host_call_complete(id, SecureStorageStatus, null)` | none |
| **secure get** (key) | request → Keychain read → **value or NotFound** | same, `argsJson{action:"get","key":…}` → `host_call_complete(id, status, payloadJson{"value":…})` **on Ok** | none — the **payload rides `host_call_complete` exactly as geolocation's fix does** |
| **secure getWithAuth** (key) — THE PAIRING | request → **OS biometric prompt → OS-key-decrypt → value** | same, `argsJson{action:"getWithAuth","key":…}` → `host_call_complete(id, status, payloadJson{"value":…})` **on Ok** | none |
| **secure delete** (key) | request → Keychain delete → status | same, `argsJson{action:"delete","key":…}` | none |

**Two things a reader might expect to force the ABI, and why neither does:**

1. **Secure `get` returns a value (a payload), unlike notifications' status-only completions.**
   It does not force anything: `host_call_complete` is `int(long requestId, int status, const
   char* payloadJsonUtf8)` — it *already* carries an optional flat-JSON payload, and geolocation
   *already* uses it (the `Granted` fix is `{"lat":…,"lng":…}`). Secure `get` returns
   `{"value":"…"}` on `Ok` and a null payload otherwise. The wire has carried a payload since
   9.0; this is the second capability to use it, not a new capability of the wire.
2. **Secure storage is near-synchronous — surely a *sync* slot (clipboard's shape) is cheaper
   than the async round-trip?** No — and a sync slot would be the *only* thing in M9 that grows
   the ABI. See §1: the async completion is mandatory (the pairing prompts) and correct (the
   Keystore can block), and reusing it is **zero ABI**; a fast sync path would be a struct grow
   for a micro-optimisation the POC does not need. **Rejected on the spot.**

**The reuse proof is made falsifiable, not asserted** (the 9.1 `NotificationsAbiUnchangedTests`
precedent). `SecureBiometricsAbiUnchangedTests` pins `SizeOf<BlazorNativeBridgeCallbacks> == 80`,
`OffsetOf(HostCallBegin) == 72`, and the two op values (`Biometrics == 2`, `SecureStorage == 3`),
**inside this phase's suite** so the claim lives next to the code it guards. The named reuse-proof
mutation — assert-81-bytes — reds the .NET pin; the iOS struct-grow mutant is expected to **fail
to COMPILE** again (the Swift type system forbidding a silent grow, 9.1's strongest proof).

## 1. Decision — one async completion for BOTH capabilities; no fast sync path (⛔ VETO-FLAGGED)

**Everything rides the EXISTING generic async begin/complete pair. There is NO synchronous
secure-storage slot, and adding one is a NON-STARTER.** The reasoning, stated once:

- **The pairing MUST be async.** `getWithAuth` shows an OS biometric prompt and suspends the app
  behind it — structurally identical to a permission dialog (9.0's whole reason for the async
  completion). A prompt cannot ride a synchronous buffer-read slot (it would block the calling
  thread across an arbitrary user interaction). So the async completion is *required* for at
  least one operation.
- **Uniformity beats a split.** Once `getWithAuth` is async, running plain `set`/`get`/`delete`
  on a *different* (synchronous) mechanism would mean two code paths, two error models, two
  mirror surfaces — for a store whose operations differ only in whether a prompt fires. One op,
  one path, action-in-JSON (geolocation's `mode` precedent) is simpler and is what the shells
  already know how to do.
- **"Near-synchronous" is a half-truth on device.** First-touch AndroidKeyStore key generation,
  hardware-backed keystore latency, and Keychain access under load can all block for tens of
  milliseconds; the honest model is async even for the plain path.
- **A sync slot is the only ABI grow in M9.** Appending a `SecureStorageRead`/`Write`/`Delete`
  slot trio (clipboard's shape) would push the struct to 104 bytes and move every drift pin and
  gate — for a micro-optimisation. The entire milestone thesis is *pay the ABI once (9.0) and
  never again*. **Rejected.**

### ⛔ VETO FLAG — the async-only decision is the phase's load-bearing shape

If a future consumer profiles the plain `get`/`set` path and finds the async round-trip a real
cost, a synchronous fast path is a *deliberate, argued* ABI grow in a later milestone — never a
silent one snuck in here. **For 9.2 the answer is async-only, zero ABI.** Decide this before
Gate 1 writes the primitives; it is the reason the headline holds.

## 2. Decision — TWO ops, not one (⛔ VETO-FLAGGED)

**`NativeShellBridge.HostCallOp` gains TWO values: `Biometrics = 2` and `SecureStorage = 3`
(the `Geolocation = 0` / `Notifications = 1` siblings).** The shells' mirror enums (Kotlin
`HostCallOp`, Swift `BnHostCallOp`) gain the same two. This is the ONLY op-enum change, and an
enum value is wire vocabulary carried on the existing `int op` field — not a struct grow.

**Why two ops and not one combined `SecureStore`:** the one-op-per-capability precedent is strong
(geolocation, notifications — one op each, actions-in-JSON *within* a capability). Biometrics and
secure storage are **two genuinely different capabilities**: authentication ("prove who is
holding the phone") is a distinct use from a key/value secret store ("keep a token"), and they
map to **two distinct façades** (`IBiometrics`, `ISecureStorage` — §5). One op per capability
keeps each handler cohesive and each status enum small.

**The pairing does NOT need one op.** `getWithAuth` is a *SecureStorage* action; the biometric
prompt it triggers is an implementation detail of the OS-key-bound read (§4), not a cross-call to
the Biometrics op. So the pairing lives entirely under `SecureStorage = 3`, and `Biometrics = 2`
stays a standalone "authenticate, no storage" capability. Two ops, each self-contained.

### ⛔ VETO FLAG — combined `ISecureStore` / one-op alternative

A single `SecureStore = 2` op + one combined façade covering both auth and storage is the
alternative. It is **rejected**: it conflates two capabilities behind one type, forces a wider
status enum (auth outcomes ∪ storage outcomes), and breaks the one-op-per-capability precedent
for no gain. **Decide the op split before Gate 1** — the Kotlin/Swift mirrors must declare the
exact two values, and the façade shape (§5) follows the op split.

## 3. Decision — biometrics (BiometricPrompt / LAContext), and the AndroidX dependency

### 3a. The status enum — `BiometricStatus`, wire-mirrored, denial-as-data

**Denial is DATA — the milestone law, restated.** Authentication's completion `int status` is a
**wire-mirrored `BiometricStatus`**, defined byte-identically across .NET / Kotlin / Swift
(pinned like the struct). The host maps each platform's native outcome into it host-side; .NET
only ever sees the integer. **Failure, cancellation, lockout, and no-hardware are all VALUES,
never exceptions and never hangs** — the awaiting `ValueTask` always resolves.

| status | Name | Meaning | Android (`BiometricPrompt.AuthenticationCallback`) | iOS (`LAContext` / `LAError`) |
|---|---|---|---|---|
| 0 | `Authenticated` | the user proved presence | `onAuthenticationSucceeded` | `evaluatePolicy` success |
| 1 | `Failed` | a biometric was presented and rejected; **retry allowed** | `onAuthenticationFailed` (not terminal on device; terminal here after the prompt closes) | `LAError.authenticationFailed` |
| 2 | `Cancelled` | the user dismissed the prompt (or the app cancelled) | `ERROR_USER_CANCELED` / `ERROR_NEGATIVE_BUTTON` | `LAError.userCancel` / `.appCancel` / `.systemCancel` |
| 3 | `Unavailable` | no hardware, or **none enrolled** | `BiometricManager.canAuthenticate` ≠ `BIOMETRIC_SUCCESS` (`NO_HARDWARE` / `NONE_ENROLLED`) | `LAError.biometryNotAvailable` / `.biometryNotEnrolled` |
| 4 | `LockedOut` | too many failures — temporarily (or permanently) locked | `ERROR_LOCKOUT` / `ERROR_LOCKOUT_PERMANENT` | `LAError.biometryLockout` |
| 5 | `Error` | unexpected host error (a caught Kotlin/Swift throw) | any other error code / caught throw | any other `LAError` / caught throw |

An out-of-range integer maps to `Error` — still data, never a throw (the `ToGeolocationStatus`
twin). **Six values** — biometrics has a richer terminal set than notifications (a prompt can
fail-but-retryable, be cancelled, or lock out), so it earns its own enum rather than reusing
`NotificationStatus`.

### 3b. Android — `BiometricPrompt` via AndroidX Biometric (⛔ the dependency decision, VETO-FLAGGED)

**Decision: ADD `androidx.biometric:biometric` (the modern, correct choice) — and NOTHING for
storage (§4 uses the no-dep raw AndroidKeyStore).** So the phase's net new Android dependency is
**exactly one library**.

The candidates and why AndroidX Biometric wins:

- **`androidx.biometric:biometric`** — the Jetpack `BiometricPrompt`: one consistent API across
  API 23+, handles the framework fragmentation (framework `BiometricPrompt` is API 28+;
  `FingerprintManager` is deprecated and API 23–27), and — decisively — integrates a
  **`CryptoObject`** cleanly, which is exactly what the OS-key-bound `getWithAuth` needs (§4). It
  is the choice Google documents and every current app uses. **Chosen.**
- **No-dep framework path** (`android.hardware.biometrics.BiometricPrompt` + a `KeyguardManager`
  device-credential fallback): avoids the gradle dependency but forces an API-level branch
  (28+ framework vs 23–27 `FingerprintManager`), reimplements the `CryptoObject` plumbing
  AndroidX already provides, and ships the deprecated `FingerprintManager` on older devices. The
  saved dependency is not worth the fragmentation and the deprecated surface. **Rejected.**

**The template-gradle-mirror consequence (folded into Gate 2's bar).** `androidx.biometric` is
added to **`src/BlazorNative.Jni/build.gradle.kts`** (the sample shell's build) **and its mirror
`templates/BlazorNative.Templates/content/BlazorNative.App/android/build.gradle.kts`** (the
`dotnet new` app's build). The two gradle files are not byte-identical (the template omits test
deps and carries different prose), but the `dependencies { … }` block that lists the shell's
runtime libraries **must gain the same `implementation("androidx.biometric:biometric:<ver>")`
line in both** — otherwise a generated app compiles a shell that references `BiometricPrompt` with
no dependency on the classpath. **This is a new class of drift for M9** (9.0/9.1 added no gradle
deps), so Gate 2's process ledger explicitly names the gradle mirror alongside the Kotlin shell
mirrors. Pin an exact version (no floating range — the shell-versions-are-pinned law in the
template's own header).

### ⛔ VETO FLAG — the AndroidX Biometric dependency

Adding a gradle dependency is a real supply-chain and mirror cost. It is **RECOMMENDED IN SCOPE**
(the modern correct API, and the `CryptoObject` path the honest pairing needs). If the owner
vetoes a new Android dependency, the fallback is the no-dep framework path with its API branch —
but the pairing's `CryptoObject` binding is materially harder to get right there. **Decide before
Gate 2 wires the Android shell**, because the gradle mirror and the `BiometricPrompt` call site
both depend on it.

### 3c. iOS — `LAContext`, no dependency

`LAContext.evaluatePolicy(.deviceOwnerAuthenticationWithBiometrics, localizedReason:)` for the
biometric-only prompt (Face ID / Touch ID). `LocalAuthentication` and `Security` are **system
frameworks** — no dependency, no Info.plist runtime-permission entry beyond **`NSFaceIDUsageDescription`**
(the purpose string Face ID requires; Touch ID needs none). Recorded so the one Info.plist key is
a decision, not an omission. Map `LAError` codes into `BiometricStatus` per §3a.

### 3d. What "authenticate" returns to .NET — just the status

**Decision: `AuthenticateAsync` returns a `BiometricStatus`, not a token and not a bool.** A real
app uses `Authenticated` to *gate* something (a storage read, a screen) — and the honest gate is
the OS-key-bound `getWithAuth` (§4), where the OS itself refuses to decrypt without auth, not a
.NET bool an attacker could skip. So standalone `authenticate` is "prove presence, get a status";
it deliberately returns **no secret and no token** (a token would imply the app trusts its own
bool, the app-level anti-pattern §4 argues against). The status is the whole answer.

## 4. Decision — secure storage, and the biometric binding (THE SECURITY CRUX; ⛔ VETO-FLAGGED)

### 4a. The store — encrypted at rest, no `androidx.security` dependency

- **Android — raw AndroidKeyStore (AES-256-GCM), no dependency.** The shell generates/loads an
  app secret key in the hardware-backed `AndroidKeyStore`, encrypts each value (AES/GCM/NoPadding,
  a fresh IV per write), and stores `base64(iv) || base64(ciphertext)` in a plain
  `SharedPreferences` file. **`androidx.security:security-crypto` (EncryptedSharedPreferences) is
  REJECTED**: Jetpack deprecated it in 2024, it drags in Tink (weight), and — decisively — the
  **auth-bound key needs a raw-AndroidKeyStore key with `setUserAuthenticationRequired(true)`
  anyway** (§4c), so one mechanism (AndroidKeyStore) cleanly serves BOTH the plain-encrypted and
  the auth-bound paths. No new storage dependency; `androidx.biometric` (§3b) is the phase's only
  gradle add.
- **iOS — Keychain, no dependency.** `SecItemAdd` / `SecItemCopyMatching` / `SecItemDelete`
  (`Security` system framework). Plain items stored with `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`
  (device-bound, not iCloud-synced, not readable while locked). No dependency.

### 4b. The status enum + the get payload

**`SecureStorageStatus`**, wire-mirrored 3 ways, denial-as-data:

| status | Name | Meaning | Payload |
|---|---|---|---|
| 0 | `Ok` | set/delete succeeded; **get found the value** | `{"value":…}` on get; none on set/delete |
| 1 | `NotFound` | get/getWithAuth of an absent key | none |
| 2 | `AuthFailed` | the biometric gate on `getWithAuth` denied / failed / cancelled / locked out | none |
| 3 | `Unavailable` | no secure hardware / Keystore unusable / (getWithAuth) biometrics not enrolled | none |
| 4 | `Error` | unexpected host error (a caught throw, a decrypt failure, malformed args) | none |

`get`/`getWithAuth` return a typed **`SecretResult(SecureStorageStatus Status, string? Value)`**
(the `GeolocationResult` twin): only `Ok` carries a value; every other status is the status alone
with a null value. **The biometric-gate detail (failed vs cancelled vs lockout) folds into
`AuthFailed`** for storage — the app only needs "couldn't unlock." A consumer wanting the finer
grain uses the standalone `IBiometrics.AuthenticateAsync` (§3), which exposes the full
`BiometricStatus`. This keeps `SecureStorageStatus` small (5 values) without losing information a
caller can get elsewhere.

### 4c. THE PAIRING — `getWithAuth`, bound at the OS-key level (the honest design)

**Decision: `getWithAuth` binds the secret to biometric authentication at the OS-KEY level — the
OS itself refuses to release the plaintext without a fresh auth. This is the honest secure design,
and it is RECOMMENDED. The app-level alternative (prompt, then read an unbound entry) is a bypass
risk and is REJECTED as the default.**

Argue it:

- **OS-key-level (chosen).**
  - **iOS:** the item is stored with a `SecAccessControl` created `.biometryCurrentSet` (+
    `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`). `SecItemCopyMatching` then triggers the
    **OS-owned** Face/Touch ID evaluation automatically; the Secure Enclave will not return the
    bytes without a successful, *fresh* biometric. `.biometryCurrentSet` additionally
    **invalidates the item if the enrolled biometric set changes** (a newly added fingerprint
    makes the old secret unreadable) — the strongest binding iOS offers.
  - **Android:** the AES key is generated in the `AndroidKeyStore` with
    `setUserAuthenticationRequired(true)` (and `setUserAuthenticationParameters(0, BIOMETRIC_STRONG)`
    on API 30+). To decrypt, the shell must present a `BiometricPrompt` with a **`CryptoObject`**
    wrapping a `Cipher` initialised with that key; only the OS-unlocked cipher can decrypt the
    stored ciphertext. The OS enforces auth at the key, not the app.
  - **The security property:** even if an attacker reaches the read code path (a rooted device, a
    logic bug, code injection), the OS still will not decrypt without a live biometric. The secret
    is safe against a bypass of the *app's* control flow.
- **App-level (rejected as default).** The app calls `authenticate` → on `Authenticated` reads an
  ordinary (non-auth-bound) entry. This trusts the app's own bool: anything that skips or forges
  the prompt (or calls the plain `get` directly) yields the plaintext. It is strictly weaker and
  it is exactly the anti-pattern §3d refuses to enable by not returning a token.

**Consequence for the wire (still ZERO ABI):** `getWithAuth` is one `SecureStorage` action; the
biometric prompt happens entirely host-side during the completion, and the plaintext returns in
the same `{"value":…}` payload as plain `get`. The op does not cross-call `Biometrics`.

**Consequence for `set` (the honest pairing needs a matching write):** an auth-bound *read*
requires an auth-bound *key at write time*. So `SetSecretAsync` takes a `requireAuth` flag: a
`requireAuth:true` write provisions the item under `.biometryCurrentSet` /
`setUserAuthenticationRequired(true)`; a `requireAuth:false` write uses the plain encrypted path.
The wire carries `"auth":"0"|"1"`. (`getWithAuth` vs `get` on the read side selects whether to
present the prompt; a `get` of an auth-bound item without the prompt correctly fails
`AuthFailed`/`Unavailable`, which the demo shows.)

### ⛔ VETO FLAG — OS-key binding vs app-level

OS-key binding is **RECOMMENDED and is the honest answer to DoD #4** ("BiometricPrompt /
LocalAuthentication *gating* a Keystore/Keychain store"). It is more implementation work (the
`CryptoObject` cipher dance on Android, the `SecAccessControl` flags on iOS) than an app-level
prompt-then-read. **If Gate 2/3 budget forces it, app-level is a *labelled, weaker* fallback that
still demonstrates the pairing — but it must be named as weaker in the conclusion, never shipped
as if it were the OS-bound design.** Decide before Gate 2 provisions the Android key, because
`setUserAuthenticationRequired` is set at key *generation* and cannot be added to an existing key.

## 5. Decision — the .NET surface (TWO façades, no 8th package)

### 5a. Layering — typed primitives in Core, façades in the EXISTING Device package

Exactly the 9.0/9.1 split, so `DevHostBridge` mocks headless:

- **`IMobileBridge` (Core)** gains the typed primitives, so the in-process mock can serve them:
  ```csharp
  ValueTask<BiometricStatus>      AuthenticateAsync(string reason, CancellationToken ct = default);
  ValueTask<SecureStorageStatus>  SetSecretAsync(string key, string value, bool requireAuth, CancellationToken ct = default);
  ValueTask<SecretResult>         GetSecretAsync(string key, CancellationToken ct = default);
  ValueTask<SecretResult>         GetSecretWithAuthAsync(string key, string reason, CancellationToken ct = default);
  ValueTask<SecureStorageStatus>  DeleteSecretAsync(string key, CancellationToken ct = default);

  public enum BiometricStatus { Authenticated, Failed, Cancelled, Unavailable, LockedOut, Error }
  public enum SecureStorageStatus { Ok, NotFound, AuthFailed, Unavailable, Error }
  public readonly record struct SecretResult(SecureStorageStatus Status, string? Value);
  ```
  Each `NativeShellBridge` impl calls the **existing** `InvokeHostCallAsync(op, argsJson, ct)`
  (`NativeShellBridge.cs:429`) — the pending registry, the cancellation posture, the unknown-id
  benignness are all **reused, not re-written**. Only the args-JSON builders, the
  `SecretResult` parse (the `ParseGeolocationResult` twin — read `{"value":…}` on `Ok`), and the
  two status mappings (`ToBiometricStatus` / `ToSecureStorageStatus`) are new .NET.
- **`IBiometrics` + `ISecureStorage` (Device — the SAME 7th package, no 8th)** — the app-facing
  façades, thin delegates over `IMobileBridge`, siblings to `IGeolocation`/`INotifications`:
  ```csharp
  public interface IBiometrics {
      ValueTask<BiometricStatus> AuthenticateAsync(string reason, CancellationToken ct = default);
  }
  public interface ISecureStorage {
      ValueTask<SecureStorageStatus> SetAsync(string key, string value, bool requireAuth = false, CancellationToken ct = default);
      ValueTask<SecretResult>        GetAsync(string key, CancellationToken ct = default);
      ValueTask<SecretResult>        GetWithAuthAsync(string key, string reason, CancellationToken ct = default);
      ValueTask<SecureStorageStatus> DeleteAsync(string key, CancellationToken ct = default);
  }
  ```
  Registered by **`AddBlazorNativeDevice()`** — **two added lines**
  (`services.AddSingleton<IBiometrics, Biometrics>()` +
  `services.AddSingleton<ISecureStorage, SecureStorage>()`, `DeviceServices.cs:33`). **No new
  package, no new shipped-set pin, no new csproj, no new version mirror** — the M9 device group
  is the whole reason `BlazorNative.Device` exists (its KDoc already says "and the other M9
  device facades as they land"). **The 7th package is the last package; 9.2 confirms it needs no
  8th.**

### 5b. `DevHostBridge` mocks it headless

`DevHostBridge` grows: an **in-memory `Dictionary<string,string>`** for the secret store
(honouring set/get/delete and a per-key `requireAuth` flag), and a **settable auth result**
(`BiometricStatus` for `authenticate`; a bool-ish gate for `getWithAuth`). So the headless lane
drives every status — `Authenticated`/`Failed`/`Cancelled`/`Unavailable`/`LockedOut` and
`Ok`/`NotFound`/`AuthFailed`/`Unavailable`/`Error`, and the pairing (a `getWithAuth` that returns
the value when the mocked auth succeeds and `AuthFailed` when it does not) — **with no device**.
This is the central proof of denial-as-data for both capabilities, asserted at Gate 1 within a
bounded await, before any device work.

### 5c. The M5 secure-storage deferral closes here — and it is DISTINCT from the plain storage slots

The existing `StorageRead`/`StorageWrite`/`StorageDelete` slots (offsets 16/24/32) are the
**plain, unencrypted** key/value store (Android `SharedPreferences` / iOS `UserDefaults`) — used
today by e.g. the notifications demo's "last tap" marker. **Secure storage is a SEPARATE
capability** (encrypted at rest, optionally biometric-bound, riding the async op), and it is what
M5 deferred. This phase adds the secure variant; the plain slots stay exactly as they are. The M5
ledger item ("secure storage — consumed by DoD #4", `MILESTONE.md:114`) is **closed** by this
phase; record the closure in the conclusion.

## 6. Decision — the proof surface (`/secure` demo)

A **`/secure` demo page** in SampleApp (`BnSecureDemo`, the **12th routed page** — after
`BnGeolocationDemo` (10th) and `BnNotificationsDemo` (11th)), the worked example that `[Inject]`s
**both** `IBiometrics` and `ISecureStorage`:

- **Set a secret** (a text field → `SetAsync("demo-secret", value, requireAuth:true)`), **read it
  back gated by a biometric prompt** (`GetWithAuthAsync("demo-secret", "Unlock your secret")` →
  echo the value on `Ok`), **delete it** (`DeleteAsync`). A **standalone Authenticate** button
  (`IBiometrics.AuthenticateAsync`) echoes the `BiometricStatus`.
- Every action echoes its returned **status as data** — a failed auth, a cancelled prompt, a
  no-hardware device, a `NotFound`, an `AuthFailed` all *shown*, never thrown, never hung (the
  geolocation/notifications discipline). The **auth-failed path** and the **no-hardware path** are
  explicit demo affordances (the DoD's "show the failure paths").
- **Sample-only** (the 9.0/9.1 template-minimal precedent): `BnGeolocationDemo` /
  `BnNotificationsDemo` are sample-only because the template tree is pinned minimal; `BnSecureDemo`
  joins them sample-only. Registration touches the **one page-authority** (`SampleAppPages.All`)
  and its **pinned mirror** `DEEP_LINK_COMPONENTS` (`"/secure" → "BnSecureDemo"`, both the
  sample's and — for route-table parity — the template's map), and `RouteTableDriftTests`
  re-greens pair-for-pair. **No new template *page*** (the demo is sample-only); the route mirror
  parity follows the established pattern.

## 7. The bridge-extension doc (bridge-extension.md)

`docs/bridge-extension.md` section (f) is the reusable pattern; 9.2 is its **third worked
capability** (after geolocation (f.7) and notifications (f.8)), so the doc gains:

1. **A one-line status bump** — the pattern now has three worked capabilities; the ABI note stays
   **10 exports / 80 bytes** (unchanged — the proof the bet held a third time).
2. **A worked-example subsection (f.9) — "Biometrics + secure storage (Phase 9.2)"** — the two op
   constants, the action-in-JSON shapes, the `BiometricStatus` / `SecureStorageStatus` tables, and
   **two teachings the pattern did not yet have**:
   - **A completion that returns a value** (secure `get`) — the `host_call_complete` payload,
     first used by geolocation's fix, is *generic*: any capability may return a flat-JSON payload
     on success. Secure storage is the second user; the doc states the payload is not
     geolocation-specific.
   - **A capability that is NOT permission-gated in the OS-dialog sense** — secure storage's plain
     `set`/`get`/`delete` show **no runtime prompt** (unlike location/notifications/biometrics):
     it is a value-in/value-out call that still rides the SAME async begin/complete for uniformity
     (§1). The doc records that "async-begin" does not imply "prompts" — it implies "may suspend
     and completes off-lane." The OS-key-bound `getWithAuth` is where the prompt re-enters, and it
     is a *store* action, not a permission gate.
3. **A security note (§8 below), lifted into the doc** — a secret crossing the intra-process C-ABI
   as a UTF-8 string is worth one paragraph the pattern did not need before.

## 8. Security note — a secret crossing the C-ABI (the one new hazard)

**The wire is intra-process and trusted; the exposure is in-memory lifetime, not interception.**
A secret value crosses the boundary as a UTF-8 string in a flat-JSON `{"value":…}` payload, host
↔ NativeAOT runtime — both halves of the same process. There is no network, no IPC, no other
process on the wire; the encryption is **at rest** (Keystore/Keychain), not on the wire. So the
hazard is not interception — it is **how many copies of the plaintext live in process memory, and
for how long:**

- The .NET `string` (immutable, GC-managed, **not zeroable**, and movable by GC compaction — a
  copy may linger after the reference drops).
- The marshalled UTF-8 buffer (`Marshal.StringToCoTaskMemUTF8` for args on the way in;
  `Marshal.PtrToStringUTF8` for the payload on the way back). The args buffer is freed in
  `BeginHostCall`'s `finally` but **not zeroed before free**.
- The host-side Kotlin/Swift `String` (and, on Android, the base64 intermediate).
- **The 9.0 async-handler-capture-window ledger item is relevant:** the payload lives in the
  pending-registry `TaskCompletionSource<HostCallResult>` until the awaiting continuation runs —
  a brief but real window where the plaintext sits in a managed object.

**For a POC this is ACCEPTABLE and documented, not fixed here.** A hardening pass (M10's
"perf/security hardening" scope) would: marshal secrets through a zeroable `byte[]`/`Span<byte>`
rather than an immutable `string`, zero the native buffer before `FreeCoTaskMem`, and shorten the
TCS capture window. **Binary secrets** (a raw key, not text) are **base64-encoded** to cross the
string→string flat-JSON (`"value"` is always base64 for `requireAuth` items; the demo uses text).
**Size:** `host_call_complete`'s payload is a single unbounded `const char*` (no buffer protocol,
no cap), but secure storage is for **secrets (tokens, keys), not blobs** — recommend a **soft 8 KB
cap** on the value, enforced host-side with an `Error`/`Unavailable` status if exceeded (a large
value is a misuse, not a store). Record the cap as a decision.

## 10. Risks (led by the OS-key-binding-vs-app-level question and the secret-on-the-wire note)

| Risk | Mitigation |
|---|---|
| **OS-key-level binding is the honest design but is more work; an app-level prompt-then-read is a bypass risk** (the security crux) | OS-key binding is **RECOMMENDED and default** (§4c): iOS `SecAccessControl(.biometryCurrentSet)` + Android `setUserAuthenticationRequired(true)` + `CryptoObject` — the OS refuses plaintext without a fresh auth, safe against a control-flow bypass. App-level is a **labelled, weaker** fallback only if Gate 2/3 budget forces it, never shipped as if OS-bound. ⛔ Decide before Gate 2 provisions the key (`setUserAuthenticationRequired` is set at key generation, immutable after). The bypass mutant (drop the flag) is a named Gate-2 mutation. |
| **A secret crosses the intra-process C-ABI as a UTF-8 string** (the new hazard) | §8: the wire is intra-process and trusted (encryption is at rest, not on the wire); the exposure is in-memory *lifetime* — multiple non-zeroable copies (the .NET string, the marshalled buffer, the host String, the TCS capture window). **Acceptable + documented for the POC; a zeroable-buffer hardening pass is M10.** Binary secrets base64; a soft 8 KB cap (secrets, not blobs). |
| **Does the value-returning `get` force the ABI?** (a payload, unlike notifications) | **No.** `host_call_complete` already carries an optional flat-JSON payload — geolocation's fix is one since 9.0. Secure `get` returns `{"value":…}` on `Ok`; the wire has carried a payload for two phases. Not a new wire capability. |
| **Is a synchronous fast path cheaper than the async round-trip for plain get/set?** | Rejected (§1): the pairing MUST be async (it prompts), uniformity wins, the Keystore can block, and a sync slot is the ONLY thing in M9 that would grow the ABI. Async-only, zero ABI. ⛔ A future profiled sync path is a *deliberate, argued* later-milestone grow, never a silent one here. |
| **The op-enum grow is a hidden ABI change** | It is not: adding `Biometrics = 2` / `SecureStorage = 3` to the value carried in the existing `int op` field is wire vocabulary — the struct stays 80 bytes, the exports stay 10. `SecureBiometricsAbiUnchangedTests`' `== 80` / offset-72 / op-value / 10-export pins are the tripwire, asserted UNCHANGED (the falsifiable headline). |
| **The AndroidX Biometric dependency drifts the template gradle mirror** (a NEW class of drift this milestone) | Gate 2's process ledger is **widened**: sync `androidx.biometric` into BOTH `src/BlazorNative.Jni/build.gradle.kts` and the template's `android/build.gradle.kts` dependency block, and **run `TemplateDriftTests` + `RouteTableDriftTests` before pushing**. | **`androidx.security:security-crypto` (EncryptedSharedPreferences) looks like the obvious storage dep** | Rejected (§4a): deprecated (2024), drags in Tink, and the auth-bound key needs raw AndroidKeyStore anyway — one mechanism serves both plain-encrypted and auth-bound storage. Net new gradle dep is `androidx.biometric` ALONE. |
| **The emulator's fingerprint is fake — biometrics is least like reality on CI** | The PROVEN/UNPROVEN split names it (§9): `adb emu finger touch` injects a synthetic event and the emulator's keystore is software; the **real sensor, the real BiometricPrompt sheet, the TEE-enforced auth-bound decrypt, a real lockout, and `.biometryCurrentSet` invalidation** are the **owner's physical Android phone** (documented `adb`-over-USB steps, the honesty check). iOS Face ID is a simulator menu toggle, labeled. |
| **`authenticate` returning a bool/token would tempt an app-level bypass** | §3d: `authenticate` returns a **status only** — no token, no secret. The honest gate is OS-key-bound `getWithAuth` where the OS enforces auth, not a .NET bool. |
