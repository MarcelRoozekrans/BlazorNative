# Phase 6.2 Design ‚Äî Real Scrolling

**Date:** 2026-07-14
**Milestone:** M6 ‚Äî Real-UI Foundation (closes DoD #4)
**Predecessor:** Phase 6.1 ‚Äî Yoga owns all placement on both shells
([conclusion](2026-07-13-phase-6.1-conclusion.md), PR #58)
**Owns:** the `scroll` NodeType, stubbed since Phase 2.5 and explicitly ledgered as an honest
boundary by 6.1: *"`ScrollView` and `Spinner` are framework `ViewGroup`s that run their own
layout over their children ‚Äî Yoga does not get the final word inside them."* This phase gives
Yoga the final word.

---

## Decisions (locked at brainstorm, 2026-07-14)

1. **Yoga owns the content size** ‚Äî not a shell-side derivation of it. The synthetic content
   node's *computed height* **is** the content size, read straight out of Yoga, including the
   off-screen part. (Two shells independently unioning child frames is exactly where Android and
   iOS drift apart, and it is the class of bug the 6.1 drift tests exist to prevent.)
   *(Amended at the Gate 1 review: this decision used to name `overflow: scroll` as the mechanism
   that keeps the content from being clamped. It is not ‚Äî see "Why the content node computes to
   800" below. The decision is the invariant, not the flag.)*
2. **Vertical only.** DoD #4 is "content taller than the viewport scrolls". Android's
   `ScrollView` is vertical-*only* ‚Äî horizontal is a different widget class, chosen at
   `CreateNode` time from a `flexDirection` that arrives in a *later* `SetStyle` patch, which is
   an ordering problem for an axis no demo needs. Horizontal is ledgered.
3. **A new `/scroll` page** (`BnScrollDemo`). `BnLayoutDemo`'s 22-number frame table is the
   cross-platform parity contract; wrapping it in a scroll view would rewrite that table in the
   same phase that introduces the scroll engine, mixing engine bugs with golden churn.
4. **No scroll events.** `onScroll` fires at 60Hz and would be the first high-frequency producer
   on a wire designed for taps ‚Äî it deserves its own design (coalescing, throttling), and its
   real customer is M7's virtualized list. Ledgered with `scrollTo` and offset-restore-across-nav.

### Added at the Gate 1 review (2026-07-14)

5. **A `BnScroll` is a flex ITEM, not a flex CONTAINER.** The container-layout family ‚Äî
   `Direction`, `Justify`, `Align`, `Wrap`, `Gap`, `Padding` ‚Äî is **absent by construction**, the
   way `BnRow`'s `Direction` is. Every one of them would style the *scroll* node, whose only Yoga
   child is the synthetic content node, and every one of them fails silently and bafflingly (see
   "The component" below). The author who wants to lay the content out **composes a `BnColumn`
   inside** ‚Äî RN's `contentContainerStyle` escape hatch, without a second style surface.
6. **The shells IGNORE-AND-LOG container styles on a `scroll` node.** Closing the component
   surface is not enough: `YogaStyleAttributes` is a global, name-keyed allow-list and `scroll` is
   a mappable element name. See "Container styles on a scroll node" ‚Äî it is **normative** for
   Gates 2/3.

## The model: a synthetic content node (the React Native shape)

A scroll view is a **viewport** with **content** that may exceed it. Yoga expresses that with two
nodes, and so do we:

```
scroll node          ‚Äî the VIEWPORT. Its frame comes from flex, like any node.
                       Needs a DEFINITE height (see below).
  ‚îî‚îÄ content node    ‚Äî SYNTHETIC (not a wire node). height: auto, width: 100%,
                       flexDirection: column, and flexShrink NEVER SET.
                       Its COMPUTED HEIGHT *is* the content size.
       ‚îú‚îÄ child 0    ‚Äî the wire children, laid out normally
       ‚îú‚îÄ child 1
       ‚îî‚îÄ ‚Ä¶
```

The content node is created by the shell whenever a `scroll` node is created, and the scroll
node's **wire children are inserted into the content node**, never into the scroll node itself.
Yoga computes the content node's height from its children (`height: auto`), and it keeps that
height even though it overflows the viewport. So `contentSize.height` is read straight out of
Yoga ‚Äî no shell derivation, nothing for the two platforms to disagree about.

### Why the content node computes to 800 and not 200 (**corrected at the Gate 1 review**)

The original text of this design said `overflow: scroll` is what stops Yoga clamping the content
to the viewport. **That is wrong**, and it was the phase's load-bearing sentence, so it is worth
stating the truth precisely:

> **Yoga's `flexShrink` default is `0`.** (CSS's is `1` ‚Äî this is one of the handful of places
> Yoga deliberately diverges.) The content node is the 200-high scroll node's only child and
> carries 800 of children, so the free space is `200 ‚àí 800 = ‚àí600`. Negative free space is
> distributed by the **shrink** pass, in proportion to `flexShrink` ‚Äî which is **0**. Nothing
> shrinks. The content node keeps its 800.

The same 800 comes out with `overflow: visible`. **`overflow: scroll` is not what produces it.**
What Yoga's `overflow` actually affects is how a node sizes **itself** under fit-content /
auto-height (and, in `hidden`/`scroll`, whether its children contribute to its own scrollable
extent) ‚Äî it is *not* a "do not clamp my overflowing child" switch.

Two consequences, both **normative**:

- **THE CONTENT NODE MUST NEVER SET `flexShrink`.** Yoga's default `0` is load-bearing. A Gate 2/3
  implementer reaching for CSS instincts and writing `flexShrink: 1` on the content node silently
  collapses 800 ‚Üí 200, and the page stops scrolling with no error anywhere. Do not set it; do not
  "normalise" it; do not copy a `flexShrink: 1` out of a React Native style sheet.
- **Gate 2's Yoga-only unit test settles the `overflow` question empirically** (see the Gates
  table). It must assert **both mechanisms explicitly**: that the content node computes to 800
  *with* `overflow: scroll`, and what it computes to *without* it. We keep setting
  `overflow: scroll` if it is semantically right (it is what RN does, and it is what a scrolling
  viewport *means*) ‚Äî but the contract must not claim it is what produces the 800, because a shell
  written against that claim will "fix" the wrong thing when a frame comes out wrong.

### ANSWERED at Gate 2 (`YogaScrollNodeAndroidTest`, Yoga 3.2.1, AVD)

The correction above is **confirmed on the device**, and the two questions this design left open
are now closed. Nothing here amends the model; it records what the engine actually does.

1. **The content node computes to 800 with `overflow: visible`, to the dp.** The overflow flag is
   not what produces it. `flexShrink: 0` is. Both shells still *set* `overflow: scroll` on the
   viewport, for the semantic reason only.
2. **`flexShrink: 1` collapses the content node 800 ‚Üí 200** ‚Äî as predicted. What was *not*
   predicted is **where the damage is not**: the **rows keep their 80** (they carry Yoga's
   `flexShrink: 0` themselves) and simply *overflow* the collapsed content node ‚Äî row 9 still sits
   at y = 720, 520dp outside a node that now claims to be 200 tall. So the mutation breaks
   **exactly one number: the one the shells read as `contentSize`.** Every frame stays correct, a
   screenshot of the first 200dp looks perfect, and the only symptom is 600dp of content that can
   never be reached. That is a sharper reason to pin it in a test than the design had.
3. **An `auto`-height scroll node hugs its content** (800 == 800, scrollable range zero), and
   **`overflow: scroll` does not change that** ‚Äî it does not rescue an auto-height viewport. So the
   definite-height warning fires on exactly that shape: the declared height is neither a point nor
   a percent, *and* the node computed out **exactly** as tall as its content. Both conditions, because
   a flex-sized viewport (`Grow="1" Basis="0"` in a bounded parent) also declares no height and is a
   perfectly good shape.
   *(Gate 2 review: this used to say `Grow="1"` **alone** was that shape. It is not ‚Äî see
   "`Grow="1"` alone does not bound a viewport" below.)*

The view tree mirrors it exactly:

| | Android | iOS |
|---|---|---|
| viewport | `ScrollView` (vertical) | `UIScrollView` |
| content | `BnYogaFrameLayout` (the single child `ScrollView` demands) | a plain `UIView` |
| contentSize | the content view's laid-out height | `scrollView.contentSize` ‚Üê the same |

This is the second index-mapping rule in the shells, alongside the text-collapse invariant, and
it is the same *kind* of rule: **the Yoga tree and the view tree mirror each other; neither
mirrors the patch tree.** A scroll node's child at wire index *i* is the content node's child at
index *i* ‚Äî in both trees, on both platforms.

### Why this works on Android ‚Äî **ANDROID-SPECIFIC, and corrected at the Gate 2 review**

`ScrollView` measures its single child with an **`UNSPECIFIED` height spec** ‚Äî "tell me how tall
you want to be" ‚Äî and `BnYogaFrameLayout.onMeasure` answers with **the last size Yoga applied**
(6.1's review put that fallback in for a boundary we were then only *documenting*).

> **This section used to say that fallback "is what makes the page scroll at all", and that
> reverting it would make "the content vanish". THAT IS WRONG**, and the Gate 2 implementer's own
> mutation run disproved it ‚Äî the demo still scrolled ‚Äî and he reported it honestly. It is the
> same shape of error as the `overflow: scroll` claim corrected above: **a second false causal
> story**, and it is corrected here before Gate 3 inherits it.

**Why the content lays out correctly anyway:** `applyFrames` walks **parent-first** and
`applyFrame` does a direct `measure(EXACTLY) + layout()` on every view. **Yoga is the last word on
the content view's frame**, whatever `onMeasure` answered on the way there.

**What the fallback ACTUALLY protects is the scroll OFFSET:**

> `ScrollView.onLayout` ends with **`scrollTo(mScrollX, mScrollY)`**, which **re-clamps the offset
> against the content child's just-laid-out height ‚Äî on every layout the ScrollView takes part in**,
> i.e. on every commit that dirties the scroll subtree (a row appended, a label re-texted; M7's
> virtualized list does it constantly). With a broken fallback the content is **0-tall at that
> moment**, `mScrollY` is clamped to **0**, and **any re-render while the user is scrolled snaps the
> page back to the top.** Real, user-visible, and completely invisible in a frame table ‚Äî every
> frame is still correct afterwards, because `applyFrame` re-lays the content.

`WidgetMapperScrollTest.a_commit_that_relayouts_the_viewport_does_not_snap_a_scrolled_page_to_the_top`
is the pin, and it reddens under exactly that mutation.

**GATE 3 MUST NOT LOOK FOR THIS MECHANISM ON iOS.** `UIScrollView` neither re-measures its content
view nor re-clamps `contentOffset` on layout ‚Äî there is **no equivalent of the `onMeasure`
fallback to mirror**, and going looking for one is a wasted gate. What iOS owes *instead* is
handling a **shrinking `contentSize` under a live `contentOffset`** itself: setting a smaller
`contentSize` does not move `contentOffset`, so the content can be left scrolled past its own end.
Android's `ScrollView` does that clamp for us; `UIScrollView` does not.

### Clipping: the VIEWPORT clips, and it is the ONE container in the shell that does

Every `BnYogaFrameLayout` sets `clipChildren = false` **to match iOS** (`UIView.clipsToBounds` is
`NO`; Yoga's `overflow` default is `visible`; Android's `ViewGroup.clipChildren` defaults to
`true`, so aligning them costs one line). **The viewport is not one of those.** It is a stock
`ScrollView` and it **keeps `clipChildren = true`** ‚Äî which is correct, and which is what matches
`UIScrollView.clipsToBounds == YES`.

**Stated because a Gate 3 implementer mirroring "our containers don't clip" onto the `UIScrollView`
draws 800dp of content over the whole screen.** The rule is: the shell's own *containers* do not
clip; a *viewport* does. (Android pins it in `WidgetMapperScrollTest`; iOS should assert
`clipsToBounds` on its `UIScrollView` the same way.)

### The constraint this introduces: a scroll node needs a definite height

An `auto`-height scroll node takes its height **from** its content: viewport height == content
height, so there is nothing to scroll. So a `BnScroll` must get its height from somewhere
definite. **Stated, tested, and logged**: a scroll node that ends up as tall as its content gets
one warning, because the symptom (a page that simply doesn't scroll) is otherwise baffling.

#### `Grow="1"` ALONE DOES NOT BOUND A VIEWPORT ‚Äî **corrected at the Gate 2 review**

This section, `BnScroll`'s XML doc, `BnScrollDemo`'s header and the warning's own message all said
*"an explicit `Height`, or `Grow="1"` inside a bounded parent (the common case)"*. **The second half
is wrong**, and it was found on the AVD while writing the tests C1 asked for. The reason is **this
phase's own mechanism, one level up ‚Äî nobody looked at the viewport**:

> `Grow="1"` leaves **`flexBasis: auto`**, so the scroll node's flex **basis is its CONTENT's
> height** (800). Against a 200-high parent the free space is `200 ‚àí 800 = ‚àí600` ‚Äî **NEGATIVE**.
> **`flexGrow` only ever distributes POSITIVE free space**, so it never gets a say; negative free
> space goes to the **SHRINK** pass, in proportion to `flexShrink` ‚Äî **which Yoga defaults to 0**.
> Nothing shrinks. **The viewport keeps its 800, spills out of its 200-high parent, and equals its
> own content: nothing scrolls.**

That is the *same sentence* this design already prints in bold about the **content** node (Yoga's
`flexShrink: 0` is why *it* keeps its 800 ‚Äî see "Why the content node computes to 800"). It is just
as true of the **viewport**, and the recommendation was written without checking. It is also exactly
why CSS's `flex: 1` shorthand expands to `flexGrow: 1; flexShrink: 1; flexBasis: 0` and not to
`basis: auto`.

**The shapes that DO bound a viewport** (normative, both shells' docs):

| shape | why it works |
|---|---|
| `Height="200"` | a POINT height. What `BnScrollDemo` uses. |
| `Grow="1" Basis="0"` | basis 0 ‚Üí free space is `200 ‚àí 0` = **positive** ‚Üí grow takes it. CSS's `flex: 1`. |
| `Grow="1" Shrink="1"` | basis 800, free space ‚àí600, and now the shrink pass takes it back off. |
| `Grow="1"` **alone** | **DOES NOT WORK.** Warned about, correctly, by the definite-height diagnostic. |

The diagnostic **fires on the broken shape** ‚Äî it is doing its job ‚Äî and
`WidgetMapperScrollTest.a_Grow_ONLY_scroll_node_does_NOT_get_a_definite_height_and_is_warned_about`
pins that it does. **Gate 3 mirrors that test**, and the iOS docs must not re-introduce the
recommendation.

*(Gate 1 review: this used to read "sizes itself to its content and never scrolls", which is the
same claim but invited the `overflow`-does-the-clamping reading corrected above. The exact
behaviour of an auto-height scroll node under `overflow: scroll` ‚Äî does it hug its content, or is
its own auto-size affected by the overflow flag? ‚Äî is one of the things **Gate 2's Yoga-only test
must observe and record**, because the definite-height warning is written against it.)*

*(**Answered at Gate 2**: it **hugs its content** ‚Äî 800 over 800, scrollable range zero ‚Äî and the
overflow flag makes no difference. So the shells' warning fires on the conjunction of two
conditions: the declared height is neither a point nor a percent, **and** the node computed out
exactly as tall as its content. Both are needed ‚Äî a flex-sized viewport declares no height either,
and it is a perfectly good shape.)*

#### The second condition is an EQUALITY, not "at least" ‚Äî **NORMATIVE** (Gate 2 review, the blocker)

Gate 2 shipped it as `scroll.layoutHeight >= content.layoutHeight` and the review caught it: that
**warns on a flex-sized viewport whose content is SHORTER than it** ‚Äî a list still loading, a page
with three items on it, M7's virtualized list on its first under-full frame. **That is not a
mistake.** It is a viewport with nothing to scroll *yet*, and it scrolls the moment the content
grows past it. The message even stated a falsehood ("it computed to 200.0dp, which is exactly its
content's height" ‚Äî it was twice it) and then prescribed a fix the author had already applied.
**A diagnostic that cries wolf on an ordinary shape is worse than no diagnostic.**

> **Both shells: `abs(scroll.layoutHeight ‚àí content.layoutHeight) > EPSILON` ‚Üí NO WARNING.**
> A viewport TALLER than its content is not a mistake. Gate 3 mirrors the two negative tests ‚Äî
> `Grow="1" Basis="0"` in a bounded parent over 800 of content, and over 100 of content:
> **neither warns** ‚Äî and the third, `Height="800"` over exactly 800 of content, which pins the
> FIRST condition (the author declared a height; there is nothing to tell them).

**KNOWN FALSE NEGATIVE, ledgered for the Gate 4 conclusion** (spec review): `Height="100%"` against
a parent with **no definite height** resolves to `auto` in Yoga ‚Äî the node hugs its content and dies
exactly as silently as an `auto`-height one ‚Äî but its *declared* unit is `PERCENT`, so it exits at
the **first** condition and is never warned about. The conjunction is deliberate and this is its
price; closing it would mean asking Yoga whether the percent actually RESOLVED, which it does not
expose. Not fixed in 6.2. Recorded so it is a known boundary rather than a surprise.

## The component

```csharp
<BnScroll Height="200">        // or Grow="1" Basis="0" in a bounded parent (CSS's `flex: 1`).
                               // NOT Grow="1" alone ‚Äî see "Grow=1 alone does not
                               // bound a viewport" above.
    <BnColumn Gap="8" Padding="16">   // ‚Üê the CONTENT's layout lives here
        ‚Ä¶rows‚Ä¶
    </BnColumn>
</BnScroll>
```

`BnScroll` emits the `scroll` element (already mapped: `MapElementToNodeType` sends
`scroll`/`overflow` ‚Üí NodeType 6). No new NodeType, no new patch kind, **no ABI change** (still 9
exports + the 72-byte bridge).

**Its surface is `BnView`'s ITEM parameters and none of its CONTAINER ones** (decision 5):

| kept ‚Äî how the viewport is placed IN ITS PARENT | **absent by construction** |
|---|---|
| `Grow` `Shrink` `Basis` `AlignSelf` | `Direction` |
| `Width` `Height` `Min*` `Max*` | `Justify` `Align` |
| `Margin` `Position` `Top`/`Right`/`Bottom`/`Left` | `Wrap` `Gap` |
| `BackgroundColor` | `Padding` |

A `BnScroll` **is a flex item that scrolls its content; how that content lays out is the content
node's job** ‚Äî and the content node is the shell's node, not the author's. Each excluded
parameter would land on the *scroll* node, whose only Yoga child is the content node, and each
fails silently:

- **`Direction`** ‚Äî `row` lays the content node out across the cross axis and stretches it to the
  viewport height: the page just stops scrolling. (This is the original decision 2, and the other
  five are the same argument; the review found we had applied it to `Direction` and then not to
  its siblings.)
- **`Gap`** ‚Äî spaces the scroll node's *one* child against nothing. The author meant "8dp between
  my rows"; the rows are children of the **content** node and never see it.
- **`Justify` / `Align`** ‚Äî free space on a scrolling viewport is **negative** (200 ‚àí 800 = ‚àí600),
  so `Center` offsets the content node to **y = ‚àí300** and `FlexEnd` to **‚àí600**. A scroll view
  cannot scroll above offset 0: the top of the content becomes **permanently unreachable**.
- **`Padding`** ‚Äî insets the content node, moving every frame in the parity table and raising
  "does `contentSize` include the padding?" ‚Äî a question two shells would answer differently.

**Ledgered, not stubbed:** a first-class `ContentPadding` / `ContentGap` (RN's
`contentContainerStyle`, set on the synthetic node) is a real, later feature. Until then,
composition is the answer, and it already works.

## Container styles on a scroll node ‚Äî **NORMATIVE** (decision 6)

Closing the component surface is not enough. `YogaStyleAttributes` is a **global, name-keyed
allow-list**, and `scroll` is a mappable element name, so this still reaches the wire:

```csharp
b.OpenElement(0, "scroll");
b.AddAttribute(1, "flexDirection", "row");   // ‚Üí SetStyle(flexDirection=row) on a scroll node
```

**The rule both shells implement in Gates 2/3:**

> On a node of NodeType `scroll`, the style names **`flexDirection`, `justifyContent`,
> `alignItems`, `flexWrap`, `gap` and `padding` are IGNORED and LOGGED (one warning, naming the
> node and the style).** The scroll node's container layout is owned by the shell: it has exactly
> one child ‚Äî the synthetic content node ‚Äî whose styles are fixed. **Item** styles (`flexGrow`,
> `flexShrink`, `flexBasis`, `alignSelf`, the box, `margin`, `position`, the offsets) and
> `backgroundColor` apply **normally**.

One rule covers both doors: the component surface (decision 5) and the raw-element hatch. It joins
the **definite-height warning** as the scroll node's second runtime diagnostic. `.NET` deliberately
does *not* filter these names ‚Äî the wire says exactly what the author said (the un-styled
invariant) ‚Äî and `BnComponentTests.RawScrollElement_StillPutsContainerStylesOnTheWire_‚Ä¶` pins that
the hatch is open, so that if it ever closes, the shells' rule is *known* to be dead code rather
than silently becoming so.

## The proof surface: `BnScrollDemo` at `/scroll`

Every number predictable, so both shells assert the same ones (the 6.1 discipline). The four
inputs are `internal const` on `BnScrollDemo` (`RowCount`, `RowHeightDp`, `ViewportWidthDp`,
`ViewportHeightDp`) and the derived ones (`ContentHeightDp`, `ScrollRangeDp`) are **computed**, so
the content size is arithmetic the contract owns rather than a number restated in three file
headers. `BnScrollDemoTests` asserts the products:

- a **viewport** 300 wide √ó 200 tall
- **ten rows**, each `Height="80"` ‚Üí `RowCount √ó RowHeightDp` = content node computes to **800**
- therefore: `contentSize.height == 800`, viewport `200`, **scrollable range 600**
- **initial offset 0** ‚Äî the visible window is content y `0..200`: rows 0 and 1 fully inside it,
  row 2 straddling the bottom edge (160..240 against a 200 cut), rows 3-9 entirely below it
- **at offset 600** (the maximum, driven programmatically in the shell tests) ‚Äî the visible window
  is content y `600..800`: **row 7 (560..640) is PARTIALLY visible ‚Äî its bottom 40dp only** ‚Äî and
  rows 8 (640..720) and 9 (720..800) are fully visible. The content cannot scroll further.
  *(Gate 1 review: this used to say "rows 7, 8 and 9 fill the viewport", which is wrong and would
  have become an `assertFullyVisible(row7)` in two shells.)*
- the demo's `BnScroll` carries **`Width` and `Height` and nothing else** ‚Äî no `Gap`, no
  `Padding`, no `Justify`: decision 5, visible in the golden's style table

The load-bearing assertion is **`contentSize > viewport`, from Yoga** ‚Äî that is the whole phase.
The scroll-and-assert step proves the viewport actually moves over the content, not merely that
the numbers add up.

## Verification (the GREEN bar)

1. `BnScrollDemo` computes **the same content size and the same row frames** on the AVD and the
   iOS simulator, asserted number-for-number (the 6.1 parity discipline).
2. Content taller than the viewport **actually scrolls** on both ‚Äî asserted by driving the scroll
   position and observing the rows move relative to the viewport.
3. **An un-scrolled tree is unchanged**: `BnDemo`, `BnSettingsPage` and `BnLayoutDemo` keep their
   goldens and their frames on all four surfaces. The scroll work must not perturb the 6.1
   parity table.
4. A `scroll` node nested *inside* flex (and flex nested inside a `scroll`) both lay out ‚Äî
   the demo does both.
5. CI counts bumped and asserted on every surface.

### Gate 2's Yoga-only unit test owns the mechanism question

It runs **before any widget exists**, and it must assert, explicitly:

1. scroll(height 200) ‚Üí content(`height: auto`, ten 80-tall children) ‚Üí **content computes to 800**
   ‚Äî the phase's whole premise, with no `ScrollView` in the picture to blame.
2. **The 800 survives without `overflow: scroll`** (`overflow: visible`) ‚Äî recording that the
   mechanism is Yoga's `flexShrink` default of `0`, not the overflow flag.
3. **Setting `flexShrink: 1` on the content node collapses it to 200** ‚Äî the mutation an
   implementer with CSS instincts will write, made visible once, in a test, instead of on a device.
4. What an **auto-height scroll node** computes to (the definite-height warning is written against
   that answer).

Whatever (2) and (4) return, the shells keep `overflow: scroll` if it is semantically right ‚Äî but
the *contract* records what actually produces the 800.

## Risks

| Risk | Mitigation |
|---|---|
| The synthetic content node desyncs the two trees' child indices (the 6.1 text-collapse class of bug) | One rule, stated once, mirrored in both shells: a scroll node's wire children go to the CONTENT node in *both* trees. Pinned by an insert-at-index test on both platforms (the .NET side pins that `InsertIndex` reaches the wire at all ‚Äî a scroll node's only child arrives as `-1` = append, and a shell must apply that to the CONTENT node's children, never to the scroll node's) |
| `ScrollView` re-measures/re-lays-out its child behind Yoga's back (the 6.1 `FrameLayout` lesson) | The content view is a `BnYogaFrameLayout` (layout suppressed); `ScrollView` gets its height from `onMeasure`'s Yoga fallback. Asserted by frames, not by types |
| **A shell sets `flexShrink` on the content node** and silently collapses 800 ‚Üí 200 | Yoga's default `0` is load-bearing and the design says so in bold. Gate 2's Yoga-only test asserts the collapse *happens* under `flexShrink: 1`, so the trap is documented by a passing test rather than discovered on a device |
