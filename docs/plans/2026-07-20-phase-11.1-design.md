# Phase 11.1 — Consumer dogfooding + the ZeroAlloc showcase (M11 DoD #3)

**Milestone:** 11 — Production Readiness. **Phase:** 11.1 (second, after 11.0). **Closes:**
DoD #3 — a stranger `dotnet new`s the **published** packages, builds both shells, runs, and
every friction point is fixed or ledgered; the result is a written, reproducible
"zero-to-running" walk that never touches the repo sources. This phase also **grows one new
public API** (the app-service DI seam) because dogfooding surfaced a real production gap.

## Goal + the reframing finding

DoD #3 as written is "walk the newcomer path on the published 0.2.0 and fix the friction." The
seed finding for this phase reframes it: **BlazorNative is already a ZeroAlloc consumer, and the
composition root is already the one thing an app cannot reach.** So the dogfooding target is not
"can a stranger use the packages" in the abstract — it is "can a stranger drive the *same*
three-source-generator ILC compile the framework itself drives, and register their *own*
services the way the framework registers its own." Everything below builds on two grounded facts:

- **Already a ZeroAlloc consumer, in production.** `src/BlazorNative.Renderer/BlazorNative.Renderer.csproj`
  (lines 57–63) pins `ZeroAlloc.AsyncEvents 1.1.2`, `ZeroAlloc.Collections 1.1.2`,
  `ZeroAlloc.Inject 1.7.1`, and the `ZeroAlloc.Inject.Generator 1.7.1` (analyzers/build, private
  assets). `NativeRenderer` is `[Singleton]` (`src/BlazorNative.Renderer/NativeRenderer.cs:23`),
  and `HostSession.EnsureSession()` composes the generated
  `AddBlazorNativeRendererServices()` / `AddBlazorNativeHttpServices()` extension methods
  (`src/BlazorNative.Runtime/HostSession.cs:313`, `:319`) before `BuildServiceProvider()`
  (`:335`). So **three source-generator ecosystems already run through one ILC compile** — Razor,
  `BlazorNative.RouteGen` (Phase 11.0), and `ZeroAlloc.Inject.Generator` — and they survive the
  trim gate today. The showcase does not prove a new pattern; it **extends a proven one**.
  *Consequence:* the showcase app **must pin `ZeroAlloc.Inject` / `.Collections` / `.AsyncEvents`
  to the framework's exact versions** (1.7.1 / 1.1.2 / 1.1.2), or a second copy of
  `Microsoft.CodeAnalysis` enters the compile and the generator diamond-conflicts.

- **The composition root is sealed.** `HostSession.EnsureSession()`
  (`src/BlazorNative.Runtime/HostSession.cs:297`–`359`) news a `ServiceCollection`, registers the
  framework services, and `BuildServiceProvider()`s **inside `BlazorNative.Runtime`, with no app
  hook**. The app's *only* registration surface today is `BlazorNativeApp.RegisterPages`
  (`src/BlazorNative.Runtime/BlazorNativeApp.cs:96`) — pages, not services. So an app author who
  writes a `[Singleton]` service, gets `ZeroAlloc.Inject.Generator` to emit its
  `AddMyAppServices()` registration method, and `[Inject]`s it into a page **has nowhere to call
  that method**. This is the gap decision #1 closes. Note the DI question is **not two rival
  containers**: `ZeroAlloc.Inject` source-generates `IServiceCollection` registration methods
  *into* `Microsoft.Extensions.DependencyInjection` — it does not build its own container. The
  only missing piece is the app's call site into the framework's composition.

## The two apps (outside the repo, published packages only)

Two apps, both created via `dotnet new blazornative` from the **published** packages, **never**
by `ProjectReference` — that is the whole point of DoD #3 (the "a stranger ships" path). They
live outside `c:\Projects\Prive\BlazorNative` (e.g. a sibling `bn-dogfood/` dir); the repo only
gains this doc, the walkthrough, and the friction ledger.

### `bn-baseline` — the clean zero-to-running proof

The stock generated app, unmodified. **Package adds beyond the template: none.** The template
already declares exactly the **three** it needs (`MyBlazorNativeApp.csproj:149`–`151`):
`BlazorNative.Runtime` (brings Core + Renderer + Http + Device transitively),
`BlazorNative.Components`, `BlazorNative.Analyzers`. This app exists to walk Gate A (build + run,
both shells) and to expose the docs-vs-reality friction (the docs' "seven `dotnet add package`"
line, below) with zero confound.

### `bn-zeroalloc-showcase` — the full-ecosystem app

The generated app, then the ZeroAlloc set added on top. The **three mandatory version pins** are
non-negotiable (they must equal the framework's, per the diamond consequence above):

```
dotnet add package ZeroAlloc.Inject      --version 1.7.1
dotnet add package ZeroAlloc.Collections --version 1.1.2
dotnet add package ZeroAlloc.AsyncEvents --version 1.1.2
```

The remaining ecosystem packages (Rest, Resilience, Serialisation, StateMachine, Validation,
Results, Mediator, Telemetry, Cache) are added at their own latest, but **each must survive the
trim-clean-publish gate or be dropped with a written reason** (see the package map + risks). The
`ZeroAlloc.Inject.Generator` is added privately (`PrivateAssets=all; IncludeAssets=analyzers;build`),
mirroring the Renderer csproj, so the app's `[Singleton]`/`[Scoped]`/`[Transient]` types get
their registration method generated.

### Build + run, per shell

Both apps publish per-RID exactly as the template documents (`MyBlazorNativeApp.csproj:44`):

- **Android** — `dotnet publish -c Release -r linux-bionic-arm64` (real devices) and
  `-r linux-bionic-x64` (emulator ABI); the shell's `gradlew installDebug` stages the
  canonicalized `BlazorNative.Runtime.so` and installs. This is where Phase 11.0's generated
  `blazornative_routes.json` gets exercised on a real added page (Gate B).
- **Desktop dev loop** — `dotnet publish -c Release -r win-x64`; the JNA host `Native.load`s the
  canonicalized `BlazorNative.Runtime.dll` and runs the render loop without a device.
- **iOS-sim — DEFERRED.** Consistent with the milestone's iOS posture (MILESTONE.md scoping
  decision #3: iOS stays simulator-scoped, no real-device proof); Phase 11.1 labels the iOS-sim
  walk **deferred**, not attempted, and records it in the ledger. Android + the desktop loop are
  the two shells this phase actually proves.

## The `ConfigureServices` seam (decision #1 — a framework `feat`)

The gap is the sealed root; the fix is a public app hook consumed **before**
`BuildServiceProvider()`. Shape (final name settled in-code, but the contract is):

```csharp
public static class BlazorNativeApp
{
    // captured at module-init, alongside RegisterPages
    public static void ConfigureServices(Action<IServiceCollection> configure);
}
```

- **Where the app calls it:** from its `[ModuleInitializer]` (the same hook that already calls
  `RegisterPages` — `BlazorNativeApp.cs` file header), so app services register with zero managed
  `Main`, exactly like pages do.
- **Where the framework consumes it:** `HostSession.EnsureSession()` invokes the captured
  delegate **after** the framework's own registrations (`AddBlazorNativeRendererServices()` …
  `AddSingleton<INavigationManager, …>()`) and **before** `services.BuildServiceProvider()`
  (`HostSession.cs:335`). One `Volatile`-guarded static field, captured once, consumed once —
  matching the existing `s_renderer`/`s_navigation` idiom in that file. Framework registrations
  come first so an app override is a deliberate last-write, and the composition stays a single
  `ServiceProvider` (no child scope, no second container).
- **The proof (unit test):** an app registers a marker service (e.g. via a generated
  `AddShowcaseServices()` from `ZeroAlloc.Inject`, or a bare `services.AddSingleton<IFoo,Foo>()`),
  a page `[Inject]`s `IFoo`, and the test asserts the page resolves the app-registered instance
  through the framework's provider — i.e. app `[Inject]` reaches app services identically to
  framework `[Inject]`. Red-first: without the seam the resolve throws; with it, it resolves.
- **Versioning:** this is a **new public API** → a `feat` → bumps the minor to **0.4.0**. It
  cross-refs DoD #4 / Phase 11.3 (the `PublicAPI.*.txt` baseline will capture this new surface
  when it lands; 11.1 adds the API, 11.3 freezes it).

Phase 11.1 spans **two** owner-authorized publishes because decision #1 is a framework change,

2. **Build the `ConfigureServices` seam in-repo** (the `feat` above; test; docs/PublicAPI

   told until 0.4.0 is live because the seam is the thing being dogfooded.

## Package → feature map

Each ZeroAlloc package is bound to a concrete demo page or code path — the showcase **rebuilds
the existing demos on ZeroAlloc backing** (decision #2), it does not merely add pages. The demos
being re-implemented are the routed sample pages: `BnListDemo` (`/list`), `BnFormDemo` (`/form`),
`BnCameraDemo` (`/camera`), `BnSecureDemo` (`/secure`), `BnGeolocationDemo` (`/geolocation`),
`BnNotificationsDemo` (`/notifications`), plus the modal/image pages. **Every package must pass
the trim-clean-publish gate or be dropped with a written reason.**

| Package | Backs (page / code path) | Honest vs stretch |
|---|---|---|
| `ZeroAlloc.Collections` | the list store behind the rebuilt `/list` demo | **honest** — already a framework dep |
| `ZeroAlloc.Inject` | app services registered via the new `ConfigureServices` seam (0.4.0) | **honest** — already a framework dep; the seam is the enabler |
| `ZeroAlloc.AsyncEvents` | UI event fan-out / the camera-flow notifications | **honest** — already a framework dep |
| `ZeroAlloc.Rest` | the fetch page **over `BlazorNative.Http`** (the injected `HttpClient`) | **honest** — must respect the analyzer (below) |
| `ZeroAlloc.Resilience` | retry/backoff wrapped around the same fetch page | **honest** |
| `ZeroAlloc.Serialisation` | secure-storage payloads (`/secure`) — serialize before store | **honest** |
| `ZeroAlloc.StateMachine` | the camera capture flow (`/camera`): idle→request→capture→result | **honest** |
| `ZeroAlloc.Validation` | the rebuilt form (`/form`) — field validation | **honest** |
| `ZeroAlloc.Results` | capability outcomes (permission grant/deny, capture success/fail) as `Result<T>` | **honest** |
| `ZeroAlloc.Mediator` | in-app request/handler dispatch behind a demo action | **stretch** — fit + trim gate must both pass |
| `ZeroAlloc.Telemetry` | timing/counters around capability calls (no network sink) | **stretch** — fit + trim gate must both pass |
| `ZeroAlloc.Cache` | memoized fetch/geolocation results | **stretch** — fit + trim gate must both pass |

**The `HttpClient` constraint is load-bearing.** `ZeroAlloc.Rest` / `.Resilience` must go over
the framework's injected `HttpClient` (the one `AddBlazorNativeHttp()` wires through
`BridgeHttpHandler` — `HostSession.cs:319`), **not** a raw `new HttpClient()`. The
`MobilePolicyAnalyzer` ships **BN0011** (parameterless `HttpClient` ctor bypasses the bridge
handler — `src/BlazorNative.Analyzers/MobilePolicyAnalyzer.cs:62`) and **BN0010** (raw sockets —
`:56`) exactly to catch this; the showcase honoring them **is** part of the dogfood signal (the
analyzer ships to consumers, so the showcase is the first real consumer test of it). If a
ZeroAlloc HTTP package can only be constructed over its own client, that is a friction item to
ledger — or a drop-with-reason.

## The integration proof

The dogfood succeeds when the showcase **publishes clean** — that clean publish *is* the
integration proof, and it has one exact yardstick:

- **The 4-IL2072 yardstick.** A stock BlazorNative app's `dotnet publish -c Release -r
  linux-bionic-arm64` emits **exactly 4 accepted IL2072s**, all in Blazor's own reflection
  internals (ComponentProperties / CascadingParameterState / ComponentFactory), documented in
  `templates/BlazorNative.Templates/content/BlazorNative.App/MyBlazorNativeApp.csproj:37`–`42`
  (and `TrimmerSingleWarn=false` there surfaces them individually, not aggregated). The showcase
  publish — arm64 **and** `win-x64` — must emit **no NEW trim/AOT warning beyond those 4**.
  Every ZeroAlloc package that clears this is proven AOT-clean-through-BlazorNative; every one
  that adds a warning is dropped-with-reason.
- **The `Microsoft.CodeAnalysis` diamond risk + mitigation.** `ZeroAlloc.Inject.Generator` and
  `BlazorNative.RouteGen` and the Razor generator all load `Microsoft.CodeAnalysis`. If the
  showcase pulls a *different* `ZeroAlloc.Inject` version than the framework's 1.7.1, two
  generator versions want two `CodeAnalysis` versions in one compile → diamond conflict. **The
  version pins (1.7.1 / 1.1.2 / 1.1.2) are the mitigation** and are mandatory, not advisory.
- **Duplicate-generator risk.** Both the framework (transitively, via Renderer) and the app
  (directly) reference `ZeroAlloc.Inject.Generator`. The app's direct reference must be private
  (`PrivateAssets=all`) so it only runs the generator against the app's own compile, and the
  transitive one does not double-emit. Verify no `ZAIxxxx` duplicate-registration warning appears.
- **Pass criterion:** both apps build; both shells run (Android + desktop loop); the showcase
  publishes with exactly the 4 IL2072s on both proven RIDs; the `ConfigureServices` unit test is
  green; and the walk is reproducible from the published packages alone.

## Risks / honest unknowns

- **A ZeroAlloc package is not AOT-clean on bionic arm64.** Any package that adds a trim/AOT
  warning past the 4 is **droppable per-package** with a written reason — the showcase's job is
  to find out, not to force all 12. Mediator/Telemetry/Cache are the flagged stretch fits.
- **The `Microsoft.CodeAnalysis` diamond** (above) — mitigated by the mandatory version pins;
  the risk is a *newer* ZeroAlloc release requiring a `CodeAnalysis` the framework's 1.7.1 does
  not, which would force the framework to bump too (out of scope here — ledger it if it bites).
- **Capabilities only build/mount in 11.1 — real behavior is Phase 11.2.** The camera/secure/
  geolocation/notification demos *compile and mount* on the showcase here; **firing them on real
  hardware is DoD #2 / Phase 11.2.** Keep the boundary explicit — 11.1 does not claim the sensor
  fired, only that the ZeroAlloc-backed flow builds, mounts, and publishes clean. Do not
  over-claim.
- **Both publishes gate on owner go.** 0.3.0 and 0.4.0 are live nuget.org releases; the phase
  stalls at Gate 0 and Gate C until the owner authorizes each. This is by design.
