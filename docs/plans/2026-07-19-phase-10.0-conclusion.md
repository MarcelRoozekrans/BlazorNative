## Gate A — #123: the test channel can no longer lie green *(DoD #2)*

`NativeRenderer` committed each frame with `_ = _frames.InvokeAsync(frame, default)`,
**discarding** the task through which `AsyncEventHandler` reports subscriber faults. A throw
inside a `Frames` subscriber — the documented **test channel** — was unobserved: never
reached `HandleException`, `StrictErrors` couldn't surface it. A test could be **green while
actually failing**.

- **Red-first (the finding IS false-green):** `FramesFaultSurfacingTests` registers a
  throwing `Frames` subscriber and asserts the fault surfaces under `StrictErrors`. It
  **FAILED on unfixed `main`** ("No exception was thrown") — proving the drop — and passes
  after.
- **Fix:** observe the task — completed inline → `GetResult()` rethrows into the existing
  `catch` → `HandleException`; still-running → a fault continuation routes to
  `HandleException`. Success-path `DispatchFrame` unchanged; no double-report.
- Pure `.NET`, no ABI. Count 754 → **755**.

## Gate B — #121: iOS reports its own platform *(DoD #1)*

`GetPlatformInfoAsync` / the `PlatformInfo` JSON hardcoded `PlatformKind.Android`; the same
NativeAOT runtime links into the iOS `.a`, so **an iOS app reported `Android`** through the
public `IMobileBridge.PlatformInfo` surface. The `os` string already came from the shell —
the `kind` did not.

- **Fix — the kind comes from the shell, explicitly** (no fragile os-string parsing): a new
  `int PlatformInfoKind` on `BlazorNativeInitOptions` (the init-**input** struct, 24 → 32
  bytes), mapped by enum ordinal via `Exports.ToPlatformKind` and served by
  `PlatformInfo` / `GetPlatformInfoAsync`. Both shells pass their own: Android →
  `Android`, iOS → `iOS`. An **unset kind maps to `DevHost`, not `Android`** — an
  un-updated shell can't lie.
- **Cross-shell + mirrors:** `Exports.cs`, `NativeShellBridge.cs`, the C header
  (`BlazorNativeRuntimeC.h`, `int32_t platformInfoKind @24`), `BnRuntime.swift`
  (`BnPlatformKind.iOS`), `NativeBindings.kt` + `BlazorNativeRuntime.kt` + `MainActivity.kt`
  (`BnPlatformKind.ANDROID`), the **template mirrors synced byte-identical** (template-drift
  green), the JNA boot-smoke size assertion 24 → 32.
- **Red-first (.NET):** drive a non-Android kind → assert served kind is `iOS` / unset →
  `DevHost`. Both **FAILED on the hardcoded code** ("Expected iOS, Actual Android"), pass
  after. Plus `ToPlatformKind` ordinal-mapping table.
- **iOS XCTest** (`BnPlatformInfoTests`): asserts the iOS shell's ABI contribution (32-byte
  struct, kind @24, iOS ordinal ≠ Android) and that the **real linked runtime boots with an
  iOS-kind init** (status 0). *Honest boundary:* platform-info has no host-facing export, so
  Swift cannot directly read the served kind without adding an export (forbidden) — the
  **serve** is proven by the .NET tests, the **shell's contribution + acceptance** by the
  XCTest. Proven on the `ios.yml` simulator lane.
- **The frozen bridge is UNTOUCHED** — 80 bytes / offset-72 / 10 exports; `BridgeProtocol
  NativeTests` + the three `*AbiUnchangedTests` green. This was the init-input struct alone.
- Counts: `.NET` 755 → **766**; iOS XCTest 233 → **235**. Every gate ↔ README row reconciled.
