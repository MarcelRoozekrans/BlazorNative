## THE HEADLINE — the `v8.0` hazard: the phase that builds the pipeline creates its own worst input

**DoD #6 tags `v8.0` to close M8.** The `release` event carries **no tag filter** — `types:` is
its only selector — so *every* published Release fires this workflow, including one published on
a milestone tag. An AdoNet.Async-shaped `VERSION="${TAG#v}"` would have turned that milestone
announcement into **six packages pushed at version "8.0"** — a version nobody decided, jumping
past `1.0.0-preview.1`, **permanently**: nuget.org has **no hard delete**, only unlist.

This repo has **zero** GitHub Releases today. The most natural first Release anyone would ever
publish here is an M8 announcement on `v8.0`. **The phase that builds the release pipeline is
the phase that creates its worst input** — and it was found at design time, not at review.

**Double-guarded, and the two guards are independent by construction:**

| Guard | Reads | On `v8.0` |
|---|---|---|
| The classifier (decision 1) | the tag's **shape** | `milestone` → **SKIP**, exit 0, announced |
| The tag↔props assertion (decision 2) | the tag's **content** vs the props | "8.0" ≠ `1.0.0-preview.1` → **RED** |

**Either alone stops it.** The reviewer proved that rather than accepting it: with the milestone
regex neutered, `v8.0` falls through to **unrecognized → RED**. The hazard survives removal of
either guard — verified, not reasoned.

**Milestone → SKIP-and-say-so, not RED, and that is the deliberate call.** A milestone
announcement Release is a legitimate thing the owner may well do at M8's close. **Reddening a
legitimate action trains the owner to ignore reds on release runs** — the last place that habit
should exist. The loudness that matters lives on the **unrecognized** arm; that is the typo net,
and it is red.

**The review constructed the adversarial set, and there is no path from a milestone Release to a
push** — `v8.0`, `v8.0.1`, `pkg/8.0`, `pkg/8.0.0`, `pkg/<semver>+build`, `refs/tags/…`,
leading/trailing whitespace, and a bare `pkg/`. Every one classifies non-publishing; the only
shape that reaches `verdict=publish` is `pkg/<valid semver>` matching the props.

**`v8.0` publishes nothing — and the classifier says so in the required lane.** The M8 close will
exercise this for real.

## The decisions as proven

- **`release: types: [published]` is the only publishing trigger.** Settled by the owner,
  verbatim from the reference implementation. Publishing the Release IS the go.
- **Two disjoint namespaces.** `v<N>.<M>` = milestone (7 tags exist, all Release-less) — **never
  publishes**. `pkg/<semver>` = package release — **the only shape that does**. `git tag -l 'v*'`
  and `git tag -l 'pkg/*'` partition the tag space with an empty intersection: the disjointness
  is an assertion you can run, not a convention you must remember. `release/<semver>` was rejected
  on a **concrete** ambiguity (the owner's own `GitVersion.yml` spends `release/*` on **branches**
  — a tag and a branch of the same name make `git rev-parse` genuinely ambiguous), and bare semver
  because it disambiguates by one character and gives no glob that enumerates package releases.
- **Props-wins, tag-asserts — the tag NEVER feeds the pack.** The version lives in
  `src/Directory.Build.props`; the tag `pkg/X` is a **claim** that the tree at that commit says
  `X`. The workflow verifies the claim and never overrides it. The tag-overrides shape (`-p:Version=`)
  is rejected because it makes the props literal **lie**: packages on nuget.org would not be
  reproducible from the commit they name, and the nuspec's own `repository@commit` would point at
  a tree that disagrees with the package it is stamped into. **`consumer-smoke.ps1` reads the
  props**, so a shape-valid tag cannot push a wrong version — the assertion is a second tooth, not
  the only one. The reference implementation can afford the override because GitVersion computes
  its version anyway; **this repo has a literal, on purpose**, so the predictable drift gets an
  xunit pin rather than a comment.
- **validate (never sees the key) → artifact → push (no checkout at all).** The split is the
  design, and one fact forces it: **the six-package push is not atomic, cannot be made atomic, and
  has no undo.** So everything provable happens **before the key exists in the run**, and `push`
  downloads the bytes `validate` packed and proved. **`push` re-packs nothing and has no checkout
  step** — *"it pushes only validated bytes"* is **structural**, not promised.
- **`--skip-duplicate`: YES — and it is only safe because the preflight is loud.** The pair is the
  design. `--skip-duplicate` owns **recovery** (three pushed, three failed → re-run the job; the
  three already up must not 409 it — otherwise the owner is left hand-pushing from a laptop with
  the production key, the worst end state for a phase whose law is "one door"). The **preflight**
  owns *"you forgot to bump"* — the failure `--skip-duplicate` would otherwise mask by pushing
  nothing and going green. **Neither is sound alone.**
- **No ordering across the six — considered, rejected, reasoning kept.** nuget.org accepts a
  package whose dependencies are not yet published; validation and indexing are **asynchronous**
  and take minutes. Dependency-order pushing would buy **seconds** off a **minutes-wide** window —
  noise, not mitigation — and would cost **six package names in `release.yml`**: a **fifth
  unpinned copy of the shipped set**, which is exactly 8.1's I-2 finding. The push **enumerates the
  feed**; the workflow names no packages. A theatre-grade ordering is not worth a real drift surface.

## `dotnet nuget push` has no `--dry-run` — so the DoD's dry-run lane became a nuget.org-state preflight

**Verified against the live CLI, not recalled.** The full option set is `--force-english-output`,
`-s|--source`, `-ss|--symbol-source`, `-t|--timeout`, `-k|--api-key`, `-sk|--symbol-api-key`,
`-d|--disable-buffering`, `-n|--no-symbols`, `--no-service-endpoint`, `--interactive`,
`--skip-duplicate`, `--configfile`. **There is no push simulation to run.** DoD #3's "dry-run
validation lane" therefore had to be defined by *what a real push can reject that no local step
can know* — and that set has exactly one member: **the state of nuget.org.** Everything else about
these packages is a local fact the smoke already owns on every PR.

Two riders came out of the same read: `<package-paths>...` is **variadic** (one invocation carries
all six), and **`-n|--no-symbols` is an opt-OUT** — the CLI's own evidence that an adjacent
`.snupkg` rides its `.nupkg` by default.

**THE VACUITY TRAP, named at design time because it is 8.1's headline recurring.** All six IDs
return **404 today** — so **every current answer to "is this id free?" is "absent"**, and a typo'd
URL, a swallowed exception, or an offline runner all produce exactly the same green as a genuinely
free id. This is `Get-TypeNames` returning `@()` wearing a different coat, against the house rule
this repo states in its own code: *a pin that cannot see its subject must never pass vacuously.*

**Closed at birth with a two-arm positive control, itself mutation-proven:**

- **Arm 1 — the scanner can SEE:** `newtonsoft.json` → 200 with a non-empty `versions[]`.
- **Arm 2 — the scanner can DISTINGUISH:** that control at a known-present version (`13.0.3`) must
  classify **published**; at a known-absent one (`0.0.0-does-not-exist`) must classify **free**.

Both arms run **before** the six, and only `200` and `404` are accepted answers — nothing is
swallowed. The mutations:

- **Pointed at the control** (`13.0.3`) → **RED, "already published"**, with the immutability
  sentence. The tooth bites.
- **Endpoint typo'd** (`/v3-flatcontainerTYPO`) → **RED**: *"nuget.org returned HTTP 400 … expected
  200 (registered) or 404 (free). An unexpected status is RED: it is not evidence the id is free."*
  The wrong URL dies **while querying the control for arm 1** — one layer earlier than the design
  predicted, and never as a cheerful "six clear".
- **The offline arm proved itself for real**, not by simulation: the sandbox's DNS failed mid-gate
  (twice, correct URL) → *"nuget.org query FAILED … This is NOT 'the id is free' — the scanner
  could not see, so no absence claim below it is worth anything."*

**The script names no packages.** The ids are **enumerated from `src/*/*.csproj`** — a literal list
would be the shipped set's **fifth copy** (8.1 normative rule 2, the I-2 finding).

## release-please: OUT — the re-evaluation reversed 8.1's REASON, not its verdict

8.1 rejected it *"for now"*, partly because *"the payoff starts once a release workflow exists"*.
The workflow exists now, so the re-evaluation was owed. **The verdict is unchanged — but 8.1's
reason turns out to be the weak one.** It is not too early. It is **structurally wrong-shaped for
this repo, and would be at any date**:

1. **Its payoff mechanism is verbatim what DoD #3 forbids.** release-please's model is: merge the
   release PR → **it cuts the tag and publishes the GitHub Release**. That is its value. But here
   **publishing the Release *is* the push** — so release-please in its native mode means *merging a
   PR publishes six packages to nuget.org*, which is exactly what DoD #3 prohibits. It can be
   configured to cut a **draft** Release the owner publishes by hand — manual-go preserved — but at
   that setting its remaining value is changelog + a bump PR, and both are the next two problems.
2. **Its unit is the commit; this repo's unit is the phase.** The reference repo's `.commitlintrc.yml`
   enumerates **component** scopes (`core`, `dataset`, `adapters`…). This repo's log reads
   `fix(phase-7.5): the dispatch table's ROW ORDER is normative`. The scope is **when**, not
   **what** — bump inference over that history mis-bumps **silently**.
3. **A generated CHANGELOG would be worse than the record that already exists.** The real changelog
   material is the phase conclusions and milestone audits in `docs/plans/`, written for humans.
4. **The `tag-prefix: v` collision (8.1's named input) is the least of it** — it is configurable,
   and `pkg/` is decision 1's answer. But a tool you must reconfigure away from its default, run in
   a non-default draft mode to preserve the owner's law, and feed a commit convention the repo does
   not use, is a tool being **fought**, not used.

**GitVersion stays out for 8.1's reason, undisturbed.**

**Reversal conditions, named so this is a decision and not a grudge — all three must hold:** the
commit convention moves from phase scopes to component scopes; **and** releases become frequent
enough that hand-writing the Release body is a chore; **and** draft-Release mode is confirmed to
preserve manual-go (release-please can *prepare* the go without *being* it).

**No CHANGELOG.md — decided here.** The **GitHub Release body IS the changelog**: written at the
moment of go, by the person deciding to go; it is what a consumer following the nuget.org project
link lands on; and it cannot drift out of sync with the thing it describes. **Trigger for
revisiting: the second public release** — the first moment anyone can want to *diff* two of them.
(`PackageReleaseNotes` rides the same trigger.)

## The false-belief sweep — four instances, one root cause

**"build-test uploads `.so` artifacts that the shell lanes consume"** — a mental model that was
**never true**, found in **four** places and corrected in all of them: `release.yml`, the design,
`docs/GITHUB-SETUP.md` (**×2** — the pre-existing M-7: `android-build` described as
*"(ubuntu-latest, `needs: build-test`)"* compiling against a `.so` build-test uploads; **it has
never been any of those things**), and `ci.yml`.

**Root cause, recorded so the next reader inherits the mechanism rather than the myth:**
**`download-artifact` cannot read another WORKFLOW's run.** That is why `android-instrumented`
publishes its own `.so` in its own `publish-so` job. **Nothing consumes build-test's uploads** —
they are **diagnostics**, there for a human to download, not for a job. The three required jobs are
**independent**; `ci.yml` declares **no `needs:` at all**.

**This is the mental model that nearly underwrote `release.yml`'s artifact hand-off with a
precedent that doesn't exist** (I-3). The `ci.yml` instance was flagged during the fix pass and
initially left alone, correctly respecting the design's *"ci.yml is zero-diff except its count
literal"* bar — then corrected in `3238e53` on the reasoning that **comments are not a lane**: the
bar exists so the required lane's **behaviour** stays untouched, a comment cannot move a gate, and
**a false comment is exactly what produced I-3**. Behaviour: zero change — no step, job, condition
or literal touched.
