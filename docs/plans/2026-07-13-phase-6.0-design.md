# Phase 6.0 Design — Yoga-Integration Spike (Both Shells)

**Date:** 2026-07-13
**Milestone:** M6 — Real-UI Foundation (closes DoD #1 — with EITHER verdict)
**Predecessor:** M5 complete (`v5.0`, merge `259c727`)
**Owns:** the named risk, verified first: can **Yoga** (Facebook's C++ flexbox engine)
link into BOTH native shells alongside the NativeAOT runtime, and does the **native
measure-callback** round-trip work? A RED reshapes the milestone (fallback documented);
a GREEN unlocks Phase 6.1 (the layout engine).

---

## Framing (locked at milestone-open)

- **Yoga runs in the shells** (the RN model) — not in .NET. So this is a SHELL-side
  spike: no .NET/renderer change. Prove Yoga on each shell independently.
- **The measure callback is the crux.** Flexbox lays out by measuring leaf content; a
  text leaf's intrinsic size comes from native font metrics. Yoga's designed extension
  point is `YGNodeSetMeasureFunc` — the shell registers a callback Yoga invokes to
  measure a leaf. Proving this round-trip (Yoga → shell measures text natively → Yoga
  uses the size) is the load-bearing part; linking is table-stakes.
- **Coexistence with the runtime native lib.** Each shell already links a NativeAOT
  artifact (Android bionic `.so`, iOS static `.a`). Yoga is a *second* native dep — the
  spike proves they coexist in the APK / the Xcode app without symbol/link conflict.
- **iOS is CI-only** (macOS runner); Android has the local AVD. Batch diagnostics, upload
  everything (the M5 CI-iteration discipline).

## The two rungs (independent; both must GREEN for a full GREEN)

### Android
- **Integration**: add Yoga to the Jni Gradle module. Preferred: the prebuilt
  **`com.facebook.yoga:yoga`** JNI bindings (what RN Android uses) — a Maven dep giving
  the `YogaNode` Java API over the C++ core; falls back to building Yoga via the NDK +
  JNA if the Maven artifact is unavailable/incompatible. Verify it coexists with
  `libBlazorNative.Runtime.so` in the debug APK (both native libs load).
- **Proof** (instrumented or JVM test): build a Yoga tree — a `row` container with two
  child boxes (one `flexGrow:1`) and a `text` leaf with a registered measure func that
  returns a fixed/native-measured size; `calculateLayout`; assert the computed frames
  (the grow child fills remaining width; the text leaf gets its measured size; children
  are laid out left-to-right). If a real `TextView.measure` drives the measure func,
  even better — but a deterministic stub measure func is an acceptable spike proof
  (6.1 wires real measurement).

### iOS
- **Integration**: link Yoga for `iossimulator-arm64` into the Xcode app + test target.
  Build Yoga's static lib from source via the toolchain (the M5 static-`.a` recipe is
  the reference — Yoga is C++/C-API, links like any static lib) OR via SPM/CocoaPods if
  clean. Call Yoga's **C-API** (`yoga/Yoga.h`: `YGNodeNew`, `YGNodeStyleSetFlex*`,
  `YGNodeSetMeasureFunc`, `YGNodeCalculateLayout`, `YGNodeLayoutGet*`) directly from
  Swift (the same C-interop the shell uses for the runtime). Verify it coexists with the
  runtime `.a` (no duplicate-symbol / link conflict).
- **Proof** (hosted XCTest on the sim): the same minimal flex tree + measure callback +
  `YGNodeCalculateLayout` → assert the frames. `@convention(c)` measure func (no capture,
  the runtime-callback pattern).

## Verification (the GREEN bar)

Both shells: Yoga links + coexists with the runtime lib + a `YGNodeSetMeasureFunc`
callback fires and its returned size drives layout + a minimal flex row computes correct
frames (grow, left-to-right placement), asserted in a test. Record the **integration
recipe** per platform (the Maven coords / the iOS Yoga build + link flags — the analog
of the 5.0 static-`.a` recipe, so 6.1 can build on it) + the Yoga version pinned.

## Deliverables

- Android: the Gradle Yoga dep + a passing Yoga-layout test (JVM or instrumented).
- iOS: the Yoga link config + a passing Yoga XCTest on the sim (via `ios.yml` — bump the
  count; the temporary-bootstrap-trigger quirk if needed).
- **`docs/plans/2026-07-13-phase-6.0-spike-conclusion.md`** — the verdict (GREEN: the
  pinned integration recipe per platform + the measure-callback proof + implications for
  6.1; or RED: what failed at each rung + the FALLBACK DECISION).
- No .NET change (Yoga is shell-side). No C-ABI change. Existing CI stays green (the only
  changes are the Jni gradle dep + iOS link + tests).

## Fallback ladder on RED (decided now)

1. **Managed flexbox in .NET** — write/port a flexbox engine in the portable core; the
   shells report leaf measurements back via a new measure round-trip (a small bridge
   callback or a wire request), the renderer computes frames and emits them. Heavier,
   novel, chatty — but keeps shells thin and layout identical-by-construction. Revisit-Yoga
   ledgered.
2. If measurement round-trip proves untenable: **map flex → native layout** (Android
   flexbox-layout lib / iOS Auto Layout) — the two-mappings-must-agree risk, accepted as
   the last resort. Controller escalates to the user on a full RED.

## Risks

| Risk | Mitigation |
|---|---|
| Yoga's Maven artifact incompatible with the shell's Kotlin/AGP | Fall back to NDK-built Yoga + JNA (the shell already uses JNA); the C-API is stable |
| Yoga static lib won't link alongside the runtime `.a` on iOS (duplicate symbols / C++ runtime) | One diagnostic run captures the linker error; the M5 link recipe (force_load, framework set) is the reference; `-lc++` already present |
| Measure callback semantics (Yoga's mode/size contract) subtle | A deterministic stub measure func first; real native measurement is 6.1 |
| CI-only iOS iteration cost | Batch diagnostics; the spike is small; Android proves the model locally first |
