# Phase 8.2 Design — The Release Pipeline, Manual Go (M8 DoD #3)

**Date:** 2026-07-16
**Milestone:** M8 — Developer Ecosystem (owns **DoD #3**: "a release workflow that packs,
validates, and pushes to nuget.org **triggered by a GitHub Release being published**; a
dry-run validation lane runs on CI without the key; `NUGET_API_KEY` documented as the one
secret the owner adds; nothing publishes automatically from merges or tags")
**Predecessor:** Phase 8.1 — publish-ready packages
([conclusion](2026-07-16-phase-8.1-conclusion.md), PR #103; DoD #2 closed at `210b8d7`)
**Owns:** `release.yml` and its trigger; the package-tag namespace; the version-reconciliation
rule; the release-please re-evaluation (8.1's named input); the dry-run lane; the push step's
shape and its partial-failure posture; the icon disposal; the `NUGET_API_KEY` documentation.
**Design only** — implementation follows in this phase's gates.

## The one sentence the whole phase hangs on

**8.2 builds the mechanism that makes the six packages public, and the mechanism cannot be
tested by using it** — there is no key, no throwaway registry, and (by the owner's own
decision) no publish until the owner publishes a Release. So the phase's real deliverable is
not a workflow; it is **a workflow whose every provable arrow is proven, and whose unprovable
arrows are named, minimized, and pointed at a failure mode that is safe and self-announcing.**

8.1 made the artifacts the truth. 8.2 gives them exactly one door, puts every check in front
of it, and hands the owner the key.

**The DoD #2 / DoD #3 boundary, stated so 8.2 does not overreach:** 8.1 = the packages pack
clean and prove themselves on every PR. 8.2 = the door (`release.yml`, the tag rule, the
dry-run lane, the one secret). This phase **publishes nothing, creates no tag, and adds no
required check** — the three required gates (`build-test`, `android-build`, `ios-build`)
keep their names and contexts untouched (branch protection).

Current bars (the standing law: counts with provenance): .NET **557/0** (ci.yml build-test) ·
JVM **106/0** (ci.yml) · Android instrumented **184/0** (android-instrumented.yml) · iOS
XCTest **154/0** (ios.yml). Device lanes are **untouched this phase** — no shell, wire, or
publish-head work exists here; 184 and 154 stand on 8.0's provenance, and the ios lane's
dispatch + push-main posture is unchanged.

## The release reality as it actually is (verified this phase, not recalled)

   `on: release: types: [published]` → ubuntu-latest → checkout (`fetch-depth: 0`) →
   setup-dotnet 10.0.x → `dotnet tool restore` → **extract version from the tag**
   (`VERSION="${TAG#v}"`) → restore → `build -p:Version=$VERSION` → test → `pack ×6
   -p:PackageVersion=$VERSION -o ./artifacts` → `dotnet nuget push ./artifacts/*.nupkg
   --api-key ${{ secrets.NUGET_API_KEY }} --source https://api.nuget.org/v3/index.json`.
   **No `--skip-duplicate`, no validate/push split, one job.** Its machinery: `GitVersion.yml`
   (ContinuousDelivery, `tag-prefix: v`, `feat:`→minor / `fix:`→patch / `!`→major; main=alpha,
   `release/*`=rc), `release-please-config.json` (release-type `simple`,
   `bump-minor-pre-major`), `.release-please-manifest.json` (`{".": "1.3.3"}`), `.commitlintrc.yml`
   (config-conventional; **scope-enum = `core|dataset|adapters|benchmarks|validation|ci|deps`**
   — component scopes). Its tags are `v1.3.3`… and **each carries a published GitHub Release**
   (`v1.3.3`, Latest, 2026-07-13). That repo's shape is coherent: component-scoped conventional
   commits → release-please → a `v`-tag → a Release → a push.
2. **This repo has ZERO GitHub Releases.** `gh release list` returns empty. `v1.0`…`v7.0` are
   **tags only** — seven milestone tags, no Release ever published, so `release: published` has
   never fired here. The `v*` namespace holds milestones and nothing else. **`v8.0` is DoD #6's
   close, and it does not exist yet.**
3. **`dotnet nuget push` has NO `--dry-run`.** Verified from the live CLI, not recalled — the
   full option set is `--force-english-output`, `-s|--source`, `-ss|--symbol-source`,

4. **All six package IDs are unregistered on nuget.org today.**
   `https://api.nuget.org/v3-flatcontainer/blazornative.{core,renderer,http,components,runtime,analyzers}/index.json`

5. **The smoke is the natural dry-run body, and it is Windows-shaped.** `scripts/consumer-smoke.ps1`
   already packs the six at the props version and asserts: zero pack warnings, 6 nupkg + 5 snupkg,

   `Join-Path $repoRoot "src\Directory.Build.props"` (and its siblings) carry literal backslashes,
   which are filename characters on Linux. This decides the release runner's OS (decision 5).
6. **The props holds the one version.** `src/Directory.Build.props` →
   `<Version>1.0.0-preview.1</Version>`, pinned by `PackageVersionPinTests` (exactly one
   `<Version>`; no shipped csproj overrides it) and by the smoke's nupkg-filename drift check.
   **8.1's normative rule 1: one version truth.** Bumps are a one-line edit.
7. **8.1's normative rule 2 is a live constraint on this phase:** *the shipped set is pinned

8. **`docs/GITHUB-SETUP.md` is the repo-setup home** — prerequisites, one-command setup, labels,

## Decisions (recorded at phase-open; ⚠ marks the veto-flagged judgment calls)

### 1. The trigger is `release: types: [published]`; package releases live in a `pkg/` tag namespace ⚠

*The call to veto: the tag namespace — (a) `pkg/<semver>` vs (b) `release/<semver>` vs (c) bare
semver `1.0.0-preview.1` with milestones keeping `vN.0`.*

**The trigger is settled by the owner** and matches the reference implementation verbatim:
`on: release: types: [published]`. Publishing the Release IS the go. Nothing else publishes.

**The collision is not theoretical — this milestone creates it.** DoD #6 tags **`v8.0`**. The
`release` event carries no tag filter (`types:` is the only selector), so **every** published
Release fires this workflow, including one published on a milestone tag. An AdoNet.Async-shaped
`VERSION="${TAG#v}"` would turn a Release published on `v8.0` into **six packages pushed at
version "8.0"** — a version nobody decided, jumping past `1.0.0-preview.1`, **permanently**
(nuget.org has no hard delete; see decision 5). The repo has zero Releases today, and the most
natural first Release anyone would publish here is an announcement of M8 on `v8.0`. **The phase
that builds the release pipeline is the phase that creates its worst input.**

**(a) — `pkg/<semver>`.** Two disjoint namespaces, each enumerable with one glob:

| Namespace | Shape | Meaning | Publishes? |
|---|---|---|---|
| `v<N>.<M>` | `v1.0` … `v7.0`, `v8.0` | **milestone** tag (7 exist, all Release-less) | **never** |
| `pkg/<semver>` | `pkg/1.0.0-preview.1` | **package release** | **only shape that does** |

`git tag -l 'v*'` and `git tag -l 'pkg/*'` partition the tag space with an empty intersection —
the disjointness is an assertion you can actually run, not a convention you have to remember.

**(b) rejected on a concrete ambiguity, not taste:** the owner's own `GitVersion.yml` reserves
`release/*` for **branches** (`release: regex: ^release/.*$`). A tag `release/1.0.0` alongside a
branch `release/1.0.0` makes `git rev-parse release/1.0.0` genuinely ambiguous ("refname is
ambiguous"). Choosing the one prefix the owner's convention already spends on branches would
import a real ref collision to avoid an imagined one. **(c) rejected** because it disambiguates
by a single character (`8.0` vs `v8.0`), gives no glob that enumerates package releases (the
rule becomes "everything not matching `v*`" — a negative, unassertable definition), and leaves
GitHub's create-release UI one keystroke from the hazard.

**The three-way classification, and what a milestone Release does.** The workflow classifies the
tag **before anything else**, and the verdicts are:

| Tag | Class | Verdict |
|---|---|---|
| `pkg/<valid semver>` | package release | proceed to validate → push |
| `v<N>.<M>` | milestone | **SKIP — exit 0, announced** in the log banner and `$GITHUB_STEP_SUMMARY`: *"milestone Release — nothing was published; package releases use `pkg/<semver>`"* |
| anything else | unrecognized | **RED**, naming the tag and both namespaces |

**Milestone → skip-and-say-so, not red** — and this is the deliberate call. A milestone
announcement Release is a *legitimate thing the owner may well do at M8's close*; reddening a
legitimate action trains the owner to ignore reds on release runs, which is the last place that
habit should exist. The DoD's requirement is that nothing publishes **by surprise**; a run that
publishes nothing and says, in the summary, that it published nothing and why, satisfies it. The
loudness that matters is on the **unrecognized** arm — that is the typo net, and it is red.

### 2. Version reconciliation: the props WINS; the tag is a claim the workflow ASSERTS ⚠

*The call to veto: (i) assert `tag == props` and fail loud on mismatch vs (ii) the tag overrides
via `-p:Version=` (the AdoNet.Async shape).*

**(i).** The rule: **the version lives in `src/Directory.Build.props`; the tag `pkg/X` is a
claim that the tree at that commit says `X`. The workflow verifies the claim and never
overrides it.** A mismatch is RED, naming both values and the commit.

**(ii) is rejected because it makes the props literal lie.** 8.1's normative rule 1 is *one
version truth*, pinned two ways. Under (ii) the pushed version comes from a tag while the tree
says something else — so the packages on nuget.org are **not reproducible from the commit they
name**: `dotnet pack` at that SHA yields `1.0.0-preview.1` while nuget.org serves
`1.0.0-preview.2`, and the nuspec's own `repository@commit` (which 8.1 asserts is stamped)
points at a tree that disagrees with the package it is stamped into. It also silently defeats
both of 8.1's version pins by routing around the property they guard. The reference
implementation can afford (ii) because GitVersion computes its version anyway — there is no
literal to contradict. **This repo has a literal, on purpose.**

The ritual (documented in decision 7): **bump the props → merge → tag the merge commit
`pkg/<version>` → publish the Release.** The assertion catches every way that goes wrong: tag
ahead of props, props ahead of tag, tag on the wrong commit.

**And it is a second, independent tooth on decision 1's hazard.** A Release on `v8.0` extracts
"8.0", which ≠ `1.0.0-preview.1` → RED — so even with the classifier removed entirely, the
surprise publish does not happen. The two guards are independent by construction: one reads the
tag's *shape*, the other compares its *content* to the tree. Neither is load-bearing alone.

**Consequence, pinned:** no `-p:Version=` or `-p:PackageVersion=` may appear anywhere in the
release path. That is the exact drift a future contributor introduces by copying the reference
implementation, so it is an xunit pin (decision 6), not a comment.

### 3. release-please: **OUT** — and the honest re-evaluation reverses 8.1's reason, not its verdict

8.1 rejected it "for now", partly because *"the payoff starts once a release workflow exists"*.
The workflow exists after this phase. The re-evaluation is therefore owed, and this is it —
**the verdict is unchanged, but 8.1's reason turns out to be the weak one.** It is not too
early. It is **structurally wrong-shaped for this repo**, and would be at any date:

1. **Its payoff mechanism is the thing DoD #3 forbids.** release-please's model is: conventional
   commits land → it maintains a release PR → **merging that PR cuts the tag and publishes the
   GitHub Release**. That is its value. But here, publishing the Release *is* the push — so
   release-please in its native mode means **merging a PR publishes six packages to nuget.org**,
   which is verbatim what DoD #3 prohibits ("nothing publishes automatically from merges or
   tags"). It can be configured to cut a **draft** Release that the owner publishes by hand —
   manual-go preserved — but at that setting its remaining value is *changelog + a bump PR*, and
   both of those are the next two problems.
2. **Its unit is the commit; this repo's unit is the phase.** `.commitlintrc.yml` in the
   reference repo enumerates **component** scopes (`core`, `dataset`, `adapters`…). This repo's
   log reads `fix(phase-7.5): the dispatch table's ROW ORDER is normative`,
   `ci(phase-7.5): the iOS bar moves 132 -> 152`, `test(phase-7.5): the mapper-level suite`.
   The scope is *when*, not *what*. Bump inference over that history mis-bumps **silently** —
   and on a `1.0.0-preview.N` line, release-please's prerelease handling would want to walk
   patches (`1.0.1`, `1.0.2`) against a version the owner hand-decides at milestone cadence.
3. **A generated CHANGELOG would be worse than the record that already exists.** Rendered from
   this history it reads *"fix(phase-7.5): the dispatch table's ROW ORDER is normative"* — a
   sentence that means everything to this repo and nothing to a consumer. The real changelog
   material is the phase conclusions and milestone audits in `docs/plans/`, which are *written
   for humans* and which no generator can produce.
4. **The collision (8.1's named input) is the least of it.** `tag-prefix: v` collides with the
   milestone namespace; it is configurable, and `pkg/` is decision 1's answer. But a tool you
   must reconfigure away from its default, run in a non-default draft mode to preserve the
   owner's law, and feed a commit convention the repo does not use, is a tool being **fought**,
   not used. That is the verdict's real basis; the collision merely happens to also be true.

**GitVersion stays out for 8.1's reason, undisturbed:** height-based version-per-merge is the
right tool when merges flow toward releases. Here nothing flows anywhere until the owner
publishes, and it would replace one named literal with a computed answer to "what version is
this?".

**What replaces it:** the hand-bumped props literal (8.1 decision 4, unchanged) plus decision 2's
assertion. That is the entire versioning mechanism, and it is two lines of ritual.

**Does M8 need a CHANGELOG.md? — NO, decided here.** The **GitHub Release body is the
changelog**: it is written at the moment of go, by the person deciding to go; it is what a
consumer following the nuget.org project link actually lands on; and it is not a file that can
drift out of sync with the thing it describes. A `CHANGELOG.md` with one entry is a maintenance
surface that restates the Release body. **Trigger for revisiting: the second public release** —
the first moment anyone can want to *diff* two of them. (`PackageReleaseNotes` in the props is
the same question wearing a different hat; it rides the same trigger.)

**What would change the release-please verdict later** — named, so it is a decision and not a
grudge. All three must hold, not any:

- the commit convention moves from phase scopes to component scopes (plausible post-M8, when
  phases stop being the unit of work and outside contributions arrive), **and**
- releases become frequent enough that hand-writing the Release body is a chore (M8 expects a
  handful of previews), **and**
- draft-Release mode is confirmed to preserve manual-go — i.e. release-please can be made to
  *prepare* the go without *being* it.

### 4. The dry-run lane: it is a **nuget.org-state preflight**, because a push dry-run does not exist ⚠

*The call to veto: the dry-run's content and its home.*

Fact 3 removes the obvious answer: **there is no `dotnet nuget push --dry-run`.** And fact 5
removes the other one: the smoke already packs the six and validates them more thoroughly than a
push would. So the lane must be defined by **what a real push can reject that neither the smoke
nor any local step can know** — and that set has exactly one member:

> **the state of nuget.org.** Everything else about these packages is a local fact the smoke
> already owns.

**The lane's content, in two parts:**

**(a) The nuget.org preflight (the only genuinely new assertion).** One key-free GET per package
against `https://api.nuget.org/v3-flatcontainer/<id>/index.json`:

- **404** → the ID is unregistered; any version is pushable.
- **200** → parse `versions[]`; **if it contains the props version → RED**: *"already published —
  nuget.org versions are immutable; bump `src/Directory.Build.props`."*

This catches the **one unrecoverable mistake** (decision 5: a published version can never be
replaced, only unlisted) and the most likely real one (forgetting the bump). It is also the
license for `--skip-duplicate` — see decision 5.

**⚠ THE VACUITY TRAP, named at design time because it is 8.1's headline recurring.** Fact 4:
**all six IDs return 404 today.** The preflight is an *absence assertion over a network scanner
whose every current answer is "absent"* — so **a typo'd URL, a wrong endpoint, a swallowed
exception, or an offline runner all produce exactly the same green as a genuinely-free ID.**
This is `Get-TypeNames` returning `@()` with a different coat on: *"a pin that cannot see its
subject must never pass vacuously"* (`PackagePurityTests.TypeNamesOf`, the house rule). It is
therefore designed with a **two-arm positive control from the start**, not after a reviewer
finds it:

- **Arm 1 — the scanner can see:** a known-published control (`newtonsoft.json`) must return
  200 **and** a non-empty `versions[]`. A wrong URL fails here.
- **Arm 2 — the scanner can distinguish:** that control at a **known-present** version
  (`13.0.3`) must be classified **published**, and at a **known-absent** version
  (`0.0.0-does-not-exist`) must be classified **free**. A scanner that answers "free" to
  everything fails here.

Both arms run **before** the six are consulted, and a failing arm is RED — the six are never
reported clean by a scanner that has not proven it can see.

**(b) The workflow's own guards, self-tested.** Decisions 1 and 2 are logic — a classifier and a
comparison — and logic in YAML is logic no lane can exercise. So **the logic does not live in
YAML** (decision 6): it lives in `scripts/release-preflight.ps1`, which carries a `-SelfTest`
table that runs the classifier and the version assertion over a fixed case set including the
**negative** rows:

| Case | Expect |
|---|---|
| `pkg/1.0.0-preview.1` + props `1.0.0-preview.1` | PUBLISH |
| `pkg/9.9.9` + props `1.0.0-preview.1` | **RED** — mismatch, naming both |
| `v8.0` | SKIP (milestone), exit 0 |
| `v1.0` | SKIP (milestone) |
| `1.0.0-preview.1` (bare) | **RED** — unrecognized |
| `release/1.0.0` | **RED** — unrecognized |
| `pkg/not-a-version` | **RED** — malformed |
| `` (empty) | **RED** — unrecognized |

The self-test is what makes the mismatch assertion **provable without a Release**: on a PR there
is no tag, so a workflow that "tested" the assertion against a synthetic tag derived from the
props would compare the props to itself — **green, vacuous, and exactly the sin 8.1's headline
was about**. The table supplies real negative inputs instead.

**Where it runs — a job in `release.yml` itself, not a new required check.** The workflow gets
**two jobs**:

- **`validate`** — no key, ever. Triggers: `release: published` **and** `pull_request` **and**
  `push: branches: [main]`, the latter two **paths-filtered** to `.github/workflows/release.yml`,
  `scripts/release-preflight.ps1`, and `src/Directory.Build.props`.
- **`push`** — `needs: validate`, `if: github.event_name == 'release'`, and the **only**
  reference to `secrets.NUGET_API_KEY` in the repository.

Three properties fall out, and each is a reason:

1. **The lane is self-proving.** A workflow's syntax is only validated when it runs, so a
   `release.yml` that has never fired is a file nobody has confirmed parses. The `pull_request`
   path trigger means **`release.yml`'s first proof is its own PR** — the file that adds it runs
   it. No `actionlint`, no `act`, no faith.
2. **The paths filter aims the lane at the moment it matters.** It runs on PRs that touch the
   release machinery **or bump the props** — i.e. on the release-preparation PR itself, which is
   precisely when "is this version still free?" is a question with an answer worth having. It
   does not run on the other 95% of PRs, which have nothing to do with releasing.
3. **`push: branches: [main]`** re-validates on the merge commit — **the commit the tag will
   name**. The validate that matters is the one on the commit being released.

**It is advisory, and stays out of the required set.** Making it required would put nuget.org's
availability on the critical path of every PR that touches the props — a nuget.org outage would
red a required gate on a change that has nothing to do with nuget.org. That is the same posture
the device lanes already hold, and the constraint is explicit: the three required gates keep
their names and contexts.

**What the dry-run deliberately does NOT do:** re-pack, re-interrogate, re-restore, or re-run
ConsumerSmoke as a separate exercise. The smoke owns all of that on **every** PR in the required
lane. The dry-run's unique claim is one sentence: **"nuget.org will accept this version, and the
door's own logic behaves."**

**One smoke gap found while scoping this, and fixed at its own layer (not duplicated here).**
The smoke asserts `6 nupkg` and `5 snupkg` — **counts, not pairing.** Flip Analyzers to
`IncludeSymbols=true` and Core to `false` and the feed still holds 6 and 5: **the counts stay
green while `BlazorNative.Core` ships with no symbols at all.** This matters directly to decision
5 (the push carries symbols *by adjacency*), but its home is the **smoke**, not the dry-run: it
is a pack-truth claim, it belongs beside the counts it strengthens, and there it rides the
required lane on every PR instead of only release-machinery PRs. The smoke gains a **pairing**
assertion (each of the five library nupkgs has its `.snupkg` sibling; Analyzers has none) — and
the mutation above is the proof that pairing ≠ counting.

### 5. The push step: validated bytes, `--skip-duplicate`, no ordering, and no atomicity to pretend about

**The posture first, because it determines the shape:**

> **The six-package push is not atomic, cannot be made atomic, and has no undo.** nuget.org has
> **no hard delete** — a published version can only be **unlisted**. There is no transaction and
> no rollback. Therefore: **everything provable happens before the key**, and the push step is
> the smallest, dumbest thing that can work.

That single fact is why `validate` and `push` are separate jobs rather than one job's steps.

**The shape:**

- **Runner: `windows-latest`** — a deviation from the reference implementation (ubuntu-latest),
  for a stated reason: it lets the release lane **reuse `scripts/consumer-smoke.ps1` verbatim**
  as its pack + validation body (fact 5 — the script is Windows-shaped, and rewriting it
  cross-platform to save runner minutes on a **public** repo, where they are free, for a workflow
  that runs **once per Release**, is cost with no payer). The alternative — a second pack loop in
  the release workflow — is a pack path nobody's PR ever exercises, drifting against the one that
  is proven every day. **The release lane packs with the same script the required lane proves.**
- **Pack once, validate those bytes, push those bytes.** `validate` runs the smoke → the feed at
  `artifacts/packages` → uploaded as a workflow artifact → `push` downloads it. **`push` re-packs
  nothing.** If it re-packed, it would be pushing bytes nobody validated — and it mirrors the
  `.so` artifact hand-off `build-test → android-build` already uses.
- **`dotnet nuget push <the six paths> --api-key ${{ secrets.NUGET_API_KEY }} --source
  https://api.nuget.org/v3/index.json --skip-duplicate`**, over paths **enumerated from the
  feed** (`Get-ChildItem *.nupkg`), not from a hardcoded name list.
- **A named secret-presence check first.** `if [ -z "$KEY" ]` → RED with *"`NUGET_API_KEY` is
  unset or misnamed — see `docs/GITHUB-SETUP.md`"*. An absent or mistyped secret otherwise
  surfaces as a bare 401 from a CLI, at the one moment the owner is least able to debug it.

**`--skip-duplicate`: YES — and it is only safe because the preflight is loud.** The pair is the
design:

- `--skip-duplicate` owns **recovery**: three pushed, three failed (network, runner death) → the
  fix is **re-run the failed job in the Actions UI**, and the three already up must not 409 the
  re-run. Without it, the recovery path is red on the first package and the owner is left hand-
  pushing from a laptop with the production key — the worst possible end state for a phase whose
  law is "one door".
- The **preflight** (decision 4) owns **"you forgot to bump"** — the failure `--skip-duplicate`
  would otherwise mask by pushing nothing and going green. It reds **before the push job starts**.

Neither is sound alone; together they cover both directions. **Recovery is a re-run, not a
Release action** — worth documenting because it is non-obvious: editing or re-publishing an
existing Release fires `edited`, **not** `published`, so the workflow does not re-fire. The owner
recovers from the Actions UI.

**Ordering across the six: NO — considered and rejected, with the reasoning kept.** The
tempting rule is "Core first, dependency order". It buys nothing:

- nuget.org **accepts** a package whose dependencies are not yet published; validation and
  indexing are **asynchronous** and take minutes.
- So dependency-order pushing narrows a **minutes-wide** indexing window by the **seconds** the
  pushes take. That is noise, not a mitigation.
- And the cost is real: an ordered loop means **six package names in `release.yml`** — a **fifth
  copy of the shipped set** (fact 7), which 8.1's normative rule 2 would then require pinning,
  after 8.1 spent a finding (I-2) proving that unpinned copies are how a seventh package gets
  packed by nothing and guarded by nothing.

**Enumerating the feed keeps the count at four copies and the release lane out of the rule
entirely.** A theatre-grade ordering is not worth a real drift surface.

**Symbols ride by adjacency.** The `.snupkg` files sit beside their `.nupkg` in the same
artifact, and `dotnet nuget push` carries an adjacent symbol package automatically (fact 3: the
existence of `-n|--no-symbols` as an **opt-out** is the CLI's own evidence of the default). The
glob is `*.nupkg`, which does **not** match `*.snupkg` — no double push. The smoke's new pairing
tooth (decision 4) is what makes adjacency a fact rather than an assumption. **That symbols
actually land on the symbol server is UNPROVEN until the first Release** — recorded, not
claimed.

**Partial failure, stated plainly:** three pushed + three failed is **recoverable** (re-run;
`--skip-duplicate` makes the first three no-ops). Three pushed **wrong** is **not** — the
recovery is unlist + bump + release again, never "fix and re-push the same version". This is the
whole reason validation is front-loaded.

### 6. Where the logic lives: a script the lane self-tests; the YAML holds only wiring

**The design principle this phase turns on:** *the untestable surface must be as small as
possible.* YAML expressions, `if:` guards, and shell one-liners inside a workflow that fires
once per Release are **the least testable code in the repository**. So:

- **`scripts/release-preflight.ps1`** owns the tag classifier (decision 1), the tag↔props
  assertion (decision 2), the nuget.org preflight with its two positive-control arms (decision 4),
  and `-SelfTest`.
- **`release.yml`** owns: the trigger, the two jobs, the artifact hand-off, the `if:` guard on
  the push job, and the `dotnet nuget push` invocation. Nothing that decides anything.

**Two xunit pins** (build-test — the one lane where every file is checkout-visible; the
`RouteTableDriftTests` rule), both **non-vacuity-asserted** (an empty or missing read is RED, not
a pass — 8.1's I-2/I-3 lesson applied at design time rather than at review):

1. **Secret containment** — `${{ secrets.NUGET_API_KEY }}` appears **exactly once** across
   `.github/workflows/*.yml`, in `release.yml`. This is normative rule 1 made unfalsifiable: it
   stops a push step being added to `ci.yml`, or a second reference drifting into the validate
   job.
2. **No version override** — no `-p:Version=` / `-p:PackageVersion=` anywhere in `release.yml`.
   This is decision 2's exact drift: the shape a contributor copies straight out of the reference
   implementation.

Expected .NET **559** (557 + 2); counted exactly at gate, ci.yml literal moved with provenance.

### 7. `NUGET_API_KEY` + the release ritual: documented in `docs/GITHUB-SETUP.md`

The one secret the owner adds, in the file that already owns "here is what you must configure on
the repo and why" (fact 8). A new section under **Recommended repo settings**, in that file's
voice:

- **`### Secrets — NUGET_API_KEY, the one secret`** — how to mint it on nuget.org (scoped to
  **Push new packages and package versions**, glob `BlazorNative.*`), where it goes (repo
  → Settings → Secrets and variables → Actions), the exact name, and the standing law: it is
  referenced by **one job in one workflow**, pinned by the containment test.
- **`### Publishing a release (the manual go)`** — the ritual, numbered: bump
  `src/Directory.Build.props` → PR → merge → `git tag pkg/<version> <merge sha>` → push the tag →
  **GitHub → Releases → Draft a new release → pick the `pkg/<version>` tag → write the body (it
  is the changelog, decision 3) → Publish**. Plus the three things a reader needs before they
  need them: the two namespaces and what a Release on `v8.0` does (nothing, loudly); that a
  mismatch between tag and props reds; and that **recovery from a partial push is an Actions
  re-run, not a Release edit** (decision 5).

**⚠ Secret scoping, flagged and deliberately not taken:** a GitHub **environment** (`nuget`) with
a branch restriction would scope the secret so only the push job in that environment can read it,
and could add a reviewer gate. **Rejected:** the owner's decision is *one secret, and the Release
is the go* — an environment gate is a **second** go, and a second manual approval on a manual
action is ceremony. The residual is stated rather than papered over: a repo-level secret is
readable by any workflow run on a branch by an actor with write access. **The mitigations that
cost nothing are taken** — exactly one reference, in a job guarded on
`github.event_name == 'release'`, pinned by a test. Flagged because it is an owner risk call.

### 8. The package icon: **OUT** — ledgered to 8.4, with the "frozen forever" objection answered

8.1 deferred it ("no icon asset exists; inventing branding inside a packaging phase is scope
creep; it rides alongside the docs-site identity"). Nothing in *this* phase changes that
calculus: an icon is not a pack requirement, not a push requirement, and not a release-pipeline
concern. **Trigger: the docs-site identity, Phase 8.4 (DoD #5)** — the icon is a derivative of
that identity, and choosing it here means choosing it twice.

**The one real counter-argument, taken seriously and answered:** `1.0.0-preview.1` is about to
become the **first public version**, and a package icon is **per-version metadata** — you cannot
retro-fit one onto a published version, so preview.1 is iconless on nuget.org **forever**. That
is true. It is also **free**: nuget.org's package page renders the **latest** version's icon, so
the moment any later version carries one, preview.1's placeholder is invisible to everyone who
is not deliberately browsing version history of a superseded preview. **A permanent cost of zero
is not a reason to make a branding decision in a release phase.** ⚠ Flagged: it is an owner-taste
call, and the veto is cheap either way (an asset + `<PackageIcon>` + a `None` item in the props).

## The normative rules, stated once

1. **Nothing publishes except a published GitHub Release.** `secrets.NUGET_API_KEY` is referenced
   in **exactly one job, in exactly one workflow**, guarded on `github.event_name == 'release'`.
   **Merges publish nothing. Tags publish nothing.** Pinned by the containment test (decision 6).
2. **Two disjoint tag namespaces.** `v<N>.<M>` = **milestone** — never publishes; a Release on one
   skips with an announcement. `pkg/<semver>` = **package release** — the only shape that
   publishes. **Anything else is an error, loudly.**
3. **The props is the version; the tag is a claim.** The release flow **asserts** `tag == props`
   and **never overrides** — no `-p:Version=` / `-p:PackageVersion=` in the release path (pinned).
   This is 8.1's normative rule 1, carried into the workflow layer.
4. **Everything provable happens before the key.** `validate` holds every check and no secret;
   `push` receives **validated bytes as an artifact**, re-packs nothing, and does nothing that
   can fail a check. There is no atomicity and no undo — front-loading is the only mitigation.
5. **The untestable surface is minimized, not trusted.** Decisions live in a script with a
   self-test table; the YAML holds wiring. Every absence assertion over a scanner (the nuget.org
   preflight above all) carries a **positive control** — *a pin that cannot see its subject must
   never pass vacuously.*

## Verification honesty — what is proven, and what is not

**This is the phase's hard problem and its honesty risk.** The real push cannot be tested: no
key, no publish, by the owner's decision. The analog is stated so it cannot be forgotten: *a
mechanism nobody tested is a mechanism nobody knows.* The design's answer is to **shrink the
unknown until what remains is one arrow whose failure mode is safe and self-announcing** — and
then to write that arrow down instead of implying it works.

**PROVEN at Gate 1 (no key, no Release, no publish):**

| # | Claim | How |
|---|---|---|
| P1 | `release.yml` parses and runs | Its own PR runs it — the `pull_request` paths trigger (decision 4). A workflow's syntax is only checked when it fires; this one fires on the PR that adds it. |
| P2 | The tag classifier is correct **and its rows are load-bearing** | `-SelfTest`'s 8-row table, negative rows included; mutations flip the rows and the table must red. |
| P3 | The tag↔props assertion **bites** | The `pkg/9.9.9` row — a real mismatch, not a props-vs-props tautology. Neutering the assertion must red the self-test. |
| P4 | A milestone Release publishes nothing | The `v8.0` / `v1.0` rows → SKIP, exit 0, announced. |
| P5 | The nuget.org preflight **can see and can distinguish** | The two-arm control (`newtonsoft.json` @ `13.0.3` = published; @ `0.0.0-does-not-exist` = free) — the arms are the reason "six clear" is not vacuous when all six 404 (fact 4). |
| P6 | The version-already-published tooth reds | Run the preflight against the control id/version → RED. |
| P7 | The secret is contained | The containment pin + its mutation (add the reference to `ci.yml` → red naming the file). |
| P8 | The release path cannot override the version | The no-override pin + its mutation. |
| P9 | The pushed bytes are the validated bytes | `push` has no pack step; it downloads the artifact `validate` produced. Structural. |
| P10 | Symbols are adjacent to their packages | The smoke's new pairing tooth + the counts-stay-green mutation (decision 4). |

**RECORDED UNPROVEN — until the owner's first Release. This list goes in the conclusion
verbatim, and DoD #3 closes as *publish-READY with these arrows named*, not as "a working
pipeline":**

| # | Unproven | Failure mode if wrong |
|---|---|---|
| U1 | `release: types: [published]` actually fires this workflow | **Nothing happens.** Safe (nothing publishes) and self-announcing (the owner has just clicked Publish and is watching for a run that does not appear). |
| U2 | `secrets.NUGET_API_KEY` resolves — the owner's secret is present and correctly named | The named presence check (decision 5) converts a bare 401 into *"`NUGET_API_KEY` is unset or misnamed"*. The check itself is unproven until a run reads a real secret. |
| U3 | nuget.org **accepts** these exact nupkgs — no reserved-prefix 403, no async validation rejection | 403/validation-failure on first push. Mitigated by fact 4 (all six IDs free today) but a reserved-prefix answer is **not obtainable without a key**. |
| U4 | The adjacent `.snupkg` reaches the symbol server | Packages publish; symbols silently do not. Recoverable (symbols can be pushed later). |
| U5 | `--skip-duplicate` behaves as reasoned on a real 409 | A re-run reds instead of no-op'ing. Recoverable by hand. |
| U6 | Six pushes across a minutes-wide async indexing window behave as reasoned | A consumer restoring during the window sees an unindexed dependency. Self-healing. |

**Why the unproven list is acceptable, argued rather than asserted:**

- **U1's failure mode is the safe direction.** A misspelled trigger (`releases:`) publishes
  **nothing** — it cannot mis-publish. And it announces itself instantly to the one person
  standing there. Every arrow that could fail *toward* publishing (the classifier, the version
  assertion, the preflight, the pack, the containment) is on the **proven** list.
- **The first Release is itself the test, and it is a preview by construction.** The blast radius
  of `1.0.0-preview.1` misfiring is one unlistable preview version of a package nobody depends
  on — which is the cheapest possible first firing of this mechanism, and it is cheap **because**
  8.1 reset the version off `1.2.0-phase-4.5`.

**Two verification vehicles considered and REJECTED, with reasons (so they are decisions, not
omissions):**

- **A scratch Release on a throwaway tag with a bogus key** — to prove U1 (trigger → job start →
  auth failure). **Rejected on three grounds.** (i) **It cannot gate the PR:** `release` events
  run the workflow from the **default branch**, so this is only possible *after* the merge it was
  meant to protect — the proof arrives after the risk. (ii) **It costs public noise to buy the
  least-likely failure:** a real Release + a real tag on a **public** repo (watcher
  notifications, a garbage tag in the namespace decision 1 just defined, both needing cleanup) to
  test an arrow copied **verbatim from a working reference implementation the owner runs today**
  (fact 1) — while everything actually novel here is already proven without it. (iii) **Its
  failure mode is benign** (U1 above). ⚠ Flagged: if the owner wants the trigger proven before
  trusting it, this is the vehicle, and the cost is one tag and one Release to delete — the
  design has no objection, only a cost/benefit verdict.
- **`act`-style local simulation** — **rejected**: `act` would exercise *`act`'s model* of the
  `release` event, not GitHub's. The one thing worth proving (U1: does GitHub fire this?) is
  precisely the thing `act` simulates rather than tests, and everything it *could* honestly prove
  (the script's logic) is already proven by `-SelfTest`, faster and in the required lane. A green
  run against a simulator is the kind of evidence that reads like proof and is not.
