# Phase 3.0a Implementation Plan — Renderer Trim Safety

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. RED-on-ambiguity rule from Phase 3.0 design applies at every gate boundary — when in doubt, roll back rather than extend.

**Goal:** Zero trim warnings on `dotnet publish src/BlazorNative.WasiHost -c Release -r wasi-wasm`, with no behavioral regression on Mono-AOT.

**Architecture:** Three sequential gates on a feature branch, each independently revertable. Gate 1 captures the current warning set as baseline. Gate 2 fixes renderer-touching warnings via `[DynamicallyAccessedMembers(All)]` annotations + `BlazorInterop.VerifyAccessors` refactor; if library-deep warnings remain, decide TrimmerRootDescriptor vs. accept. Gate 3 lands a regression test that locks in the fix at the .NET layer.

**Tech Stack:** .NET 10, Mono-AOT under wasi-experimental (unchanged), Microsoft.AspNetCore.Components 10.0.*, xUnit. No NativeAOT, no runtime swap — that's Phase 3.0b.

**Reference design:** [docs/plans/2026-05-28-phase-3.0-design.md](2026-05-28-phase-3.0-design.md) (commit `d5befd1`).

---

## Honest scope caveat

Phase 3.0a runs entirely on Mono-AOT under the existing wasi-experimental toolchain. The runtime null bug from today's componentize-dotnet spike (where `RenderTreeFrame.ElementName` returned null) only manifests under NativeAOT-LLVM's trim engine. **Phase 3.0a does not prove trim safety end-to-end.** It eliminates everything the static analyzer can tell us about, on the toolchain we currently run. The actual NativeAOT runtime validation happens in Phase 3.0b Gate 2. If 3.0a's annotations don't hold up under real NativeAOT in 3.0b, that's a different diagnosis — possibly Microsoft.AspNetCore.Components internal field trim that no annotation at our layer can fix, in which case we'd need TrimmerRootDescriptor or the whole NativeAOT direction reconsiders.

Phase 3.0a's job is the cheap, defensive, "fix what the static analyzer flags" pass that gives 3.0b the best shot at succeeding.

### Task 2.2: Refactor `BlazorInterop.VerifyAccessors` to eliminate IL2026

**Files:** Modify: `src/BlazorNative.Renderer/BlazorInterop.cs` lines 59-88.

**Step 1:** The current `VerifyAccessors` uses `Assembly.GetType("Microsoft.AspNetCore.Components.RenderTree.EventFieldInfo")` (line 67) — `EventFieldInfo` is internal so we can't `typeof()` it directly. But the `[UnsafeAccessorType]` attribute on `RefAccessors.DispatchEventAsync` (line 221) already names this type as a compile-time string. We can hoist that to a private const:

Add near the top of `BlazorInterop`:

```csharp
private const string EventFieldInfoTypeName =
    "Microsoft.AspNetCore.Components.RenderTree.EventFieldInfo, Microsoft.AspNetCore.Components";
```

**Step 2:** Replace lines 67-68:

```csharp
var eventFieldInfoType = typeof(RenderTreeFrameType).Assembly
    .GetType("Microsoft.AspNetCore.Components.RenderTree.EventFieldInfo");
```

with `Type.GetType(string, throwOnError: false)`:

```csharp
// Annotated to tell the trimmer that EventFieldInfo's members must stay reachable.
// Without this, IL2026 fires on the lookup. The type is rooted by the
// [UnsafeAccessorType(...)] reference on RefAccessors.DispatchEventAsync below,
// so the trimmer will keep it; this annotation just silences the warning at the lookup site.
[UnconditionalSuppressMessage("Trimming", "IL2026",
    Justification = "EventFieldInfo is kept alive by [UnsafeAccessorType] on RefAccessors.DispatchEventAsync. " +
                    "Lookup at runtime is for drift detection only.")]
[UnconditionalSuppressMessage("Trimming", "IL2057",
    Justification = "Same as above — type referenced via UnsafeAccessorType, not subject to trimming.")]
static Type? GetEventFieldInfoType() =>
    Type.GetType(EventFieldInfoTypeName, throwOnError: false);

var eventFieldInfoType = GetEventFieldInfoType();
```

```csharp
using BlazorNative.Renderer;

namespace BlazorNative.Renderer.Tests;

/// <summary>
/// Phase 3.0a regression test — locks in the trim-safety annotation pass.
///
/// HISTORY: componentize-dotnet spike on 2026-05-28 RED'd because
/// RenderTreeFrame.ElementName returned null under NativeAOT-LLVM trim.
/// Phase 3.0a's annotations + (possibly) TrimmerRootDescriptor on
/// Microsoft.AspNetCore.Components root the backing fields.
///
/// This test runs on Mono-AOT (current toolchain). It does NOT prove
/// NativeAOT trim safety — that's Phase 3.0b Gate 2. It DOES prove
/// the annotations don't break behavior on the current toolchain and
/// catches any future regression where ElementName becomes null on
/// Mono-AOT (which would be unexpected).
/// </summary>
[Collection("WasiBridgeTests")]  // shares WasiBridge.Current singleton with other tests
public class TrimSafetyTests
{
    [Fact]
    public void Mount_HelloComponent_FirstElementNameIsNotNull()
    {
        // Build the same DI graph as production WasiHost.
        var services = new ServiceCollection();
        services.AddBlazorNativeCoreServices();
        services.AddBlazorNativeRendererServices();
        services.AddBlazorNativeHttpServices();
        var provider = services.BuildServiceProvider();

        var renderer = provider.GetRequiredService<NativeRenderer>();

        var emittedFrames = new List<string>();
        // Subscribe to the frame channel — see existing test for the pattern.
        // (Adjust per actual NativeRenderer test hook; if no hook exists, capture
        // stdout via Console.SetOut.)
        var sw = new StringWriter();
        var originalOut = Console.Out;
        Console.SetOut(sw);
        try
        {
            renderer.Mount<HelloComponent>();
        }
        finally
        {
            Console.SetOut(originalOut);
        }

        var stdout = sw.ToString();
        Assert.Contains("[FRAME]", stdout);

        // Find the first [FRAME] line and verify it contains a CreateNode patch
        // with a non-null elementName / nodeType field. The exact JSON shape is
        // defined by PatchProtocol.cs's [JsonPolymorphic] discriminator.
        var frameLine = stdout
            .Split('\n')
            .First(l => l.Contains("[FRAME]"));
        Assert.DoesNotContain("\"elementName\":null", frameLine);
        Assert.DoesNotContain("\"nodeType\":null", frameLine);
    }
}
```

**Adjustment note:** the exact way to subscribe to `NativeRenderer`'s frame output depends on existing test patterns. If `MountSyncTests.cs` shows a cleaner hook than `Console.SetOut`, use that instead. The test's intent is clear; the plumbing matches the existing test conventions.
