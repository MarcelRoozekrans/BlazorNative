# Phase 3.2 Design — Bidirectional Event Flow

**Date:** 2026-07-09
**Milestone:** M3 — P2: Real Apps Can Be Built (closes DoD #2; partial DoD #9)
**Predecessor:** Phase 3.1 GREEN (merged `c67cdcb`) — see [conclusion](2026-07-09-phase-3.1-conclusion.md)
**Owns:** `blazornative_dispatch_event` wiring, AttachEvent/DetachEvent end-to-end,
WasiBridge deletion (3.1 carryover), the two documented threading hazards.

---

## Objective

Tap "Tap" on the AVD → the button's `@onclick` handler runs inside the trimmed `.so` →
component state mutates → re-render → the counter text updates on screen. The demo becomes
interactive; M3 DoD #2 closes.

## What already exists (this phase is wiring, not invention)

- Renderer emits `AttachEventPatch(nodeId, eventName, handlerId)` for `on*` attributes
  (`NativeRenderer.cs:338`) and has `DispatchUiEventAsync(NativeUiEvent)` with
  `BuildEventArgs`: click→`MouseEventArgs`, change→`ChangeEventArgs{Value=payload}`,
  focus/blur→`FocusEventArgs` (`NativeRenderer.cs:390-413`).
- The Kotlin adapter already decodes AttachEvent/DetachEvent patches (3.0d full-ABI decision).
- `blazornative_dispatch_event(ulong, IntPtr)` is declared, dormant (returns -1).
- The FlatJson writer/parser pair is drift-tested on both sides (3.1).

Missing: the export body, WidgetMapper listeners, thread discipline, an interactive component.

## Decisions locked during the brainstorm (2026-07-09)

| Decision | Choice |
|---|---|
| Event scope | **Click + change.** Button `setOnClickListener` AND EditText `TextWatcher` (text as payload) — change is direct `@bind` (DoD #5) groundwork, ~20 extra lines against machinery built anyway. Focus/blur stay .NET-mapped but unwired. |
| Demo | **HelloComponent becomes interactive** — `@onclick` counter on the existing Tap button, rendered into the text. Both hand-maintained golden lists update via the documented procedure. |
| Dispatch execution | **Synchronous inside the export** (InlineDispatcher): handler → re-render → FrameSink → frame callback all complete before `dispatch_event` returns. No new async machinery; frames still fire only inside host calls (mount OR dispatch), containing the 3.0d trampoline hazard. |
| Threading | **One dispatch lane**: `BlazorNativeRuntime` gains a single-thread daemon executor (`BlazorNative-Dispatch`) + `dispatchEvent(handlerId, name, payload)`. UI listeners submit to it; the ABI is never called from the UI thread. Resolves BOTH documented hazards: renderer keeps single-threaded access post-boot, and handler-triggered bridge ops (SharedPreferences `commit()`) stay off the main thread — no StrictMode violation. |
| DoD #9 bundle | **Partial.** Handler exceptions become visible via the export's return code 2 + stderr detail + loud Kotlin log. Full renderer `HandleException` strict-mode remains its own M3 item. |
| WasiBridge | **Deleted** (with `BridgeEventTests` + `WasiBridgeTestCollection` — their multicast subject is gone). `IMobileBridge.NativeEvents` STAYS as the DevHost-facing contract; `DevHostBridge` keeps its impl; BN0014 stays valid. `HostSession`'s "last-wins over WasiBridge" comment → "sole registration". |

## Export contract

```csharp
[UnmanagedCallersOnly(EntryPoint = "blazornative_dispatch_event")]
public static int DispatchEvent(ulong handlerId, IntPtr argsJsonUtf8);
```

- Args: flat JSON (3.1 FlatJson pair), e.g. `{"name":"click"}` /
  `{"name":"change","payload":"user text"}`. NULL args → treated as `{"name":"click"}`?
  NO — NULL args → return 3 (malformed); the Kotlin side always sends the object.
- Returns: **0** dispatched (incl. stale-handler case — see below) · **1** no session /
  nothing mounted · **2** handler threw (detail `ex.ToString()` to stderr; Kotlin logs
  loudly) · **3** malformed/NULL args.
- Stale handler (`ArgumentException` from Blazor's handler table after a re-render): already
  caught + logged by `DispatchUiEventAsync`; the export returns 0 — delivery is
  at-most-once, a stale tap is not an error. Documented.
- Synchronous: any re-render's frame callback completes before the export returns.

## Kotlin side

- `BlazorNativeRuntime`: `dispatchEvent(handlerId: Int, eventName: String, payload: String?)`
  submits to the new single-thread daemon executor; non-zero rc → onError (rc 2 loudly).
  Threading contract documented here AND in `Exports.cs`: all post-boot .NET entry goes
  through this lane.
- `WidgetMapper` gains constructor param
  `onUiEvent: (handlerId: Int, eventName: String, payload: String?) -> Unit` and implements
  the two stubs:
  - **AttachEvent**: `"click"` → `view.setOnClickListener { onUiEvent(handlerId, "click", null) }`;
    `"change"` → `TextWatcher` on EditText firing `onUiEvent(handlerId, "change", text)`.
    Re-entrancy guard: the watcher does NOT fire when the text change came from a
    programmatic `setText` (track via a per-view "applying remote patch" flag set by the
    mapper during batch application). Watcher stored in the view's tag keyed by handlerId
    so DetachEvent can remove it.
  - **DetachEvent**: clears the click listener / removes the watcher by handlerId.
  - Unknown event names: log + skip (forward compat).
- `MainActivity`: `WidgetMapper(this, widgetRoot, onUiEvent = runtime::dispatchEvent)`.

## Risks

| Risk | Mitigation |
|---|---|
| TextWatcher re-entrancy loop (programmatic setText → change → dispatch → re-render → setText…) | "applying remote patch" flag in the mapper; dedicated instrumented test |
| Golden churn drift | Documented hand-maintenance procedure; spec review verifies transcription against the actual first frame |
| Stale handler after re-render | Existing ArgumentException catch; contract says at-most-once, rc 0 |
| Dispatch lane starvation (slow handler blocks later events) | Single lane is deliberate (renderer affinity); handlers are app code — document; revisit if M3 components need concurrency |
| Handler mutates state → NEW AttachEvent handlerIds each render | WidgetMapper replaces listeners on re-attach (same view, new handlerId) — covered by the round-trip test tapping twice |
