# Phase 2.2 Design — Android Port: libwasmtime.so + Android Shell

**Date:** 2026-05-26
**Milestone:** M2 — First End-to-End Demo on Android
**Phase:** 2.2 (Android port of the runtime layer proven on JVM desktop in Phase 2.1)
**Status:** Design — approved 2026-05-26
**Predecessor:** Phase 2.1 complete 2026-05-26 (commit `57458e3`)

---

## Goal

Port the JVM/JNA + libwasmtime runtime layer proven in Phase 2.1 to Android. The same `WasmtimeBindings.kt` / `WasiHost.kt` / `WasmtimeException.kt` source files run on both JVM desktop (Phase 2.1) and Android (Phase 2.2) — only the libwasmtime build target + a thin `MainActivity` wrapper differ.

**Success criterion (GREEN CHECKPOINT):** `./gradlew connectedAndroidTest` reports `BootSmokeAndroidTest > boots_and_emits_markers_on_android PASSED` against a running Android emulator. This validates DoD #2 ("Android Kotlin shell scaffold ... installs to an emulator or device") and DoD #3 (JVM ↔ libwasmtime integration on Android, the revised JNA-via-cargo path).

## Why this phase exists

Phase 2.1 proved Strategy G on the JVM desktop. Phase 2.2 ports the runtime layer to Android by cross-compiling `libwasmtime.so` for `arm64-v8a` and `x86_64` via NDK + cargo-ndk, then loading the same .wasm through the same JNA bindings from an Android `MainActivity`. The .NET-side `BlazorNative.WasiHost.wasm` is ABI-portable (one artifact); the only platform-specific code added is the `MainActivity` shell + Android Gradle plugin wiring.

This phase is intentionally narrow — Android scaffolding + first successful load. Real `mobile_bridge` symbol implementations (Phase 2.3), render-frame consumer (Phase 2.4), widget mapper (Phase 2.5), `BlazorNativeHostElement` (Phase 2.6), and the end-to-end demo (Phase 2.7) all land in later phases.

## Locked decisions and rationale

| ID | Decision | Rationale |
|---|---|---|
| Q1 | Project layout = **extend existing `src/BlazorNative.Jni/`** with Android Gradle plugin (single module, shared `src/main/kotlin/` + Android-specific `src/androidMain/` + `src/androidTest/`) | Reuse is literal-same source file; one Gradle config to maintain; mirrors RN/Hermes layout. |
| Q2 | Target ABIs = **arm64-v8a + x86_64** (modern device + emulator) | M2 DoD requires emulator install (so x86_64); arm64-v8a covers 99% of real devices; armeabi-v7a is dead weight for a POC. |
| Q3 | SDK levels = **minSdk 24, targetSdk 34, compileSdk 34** | minSdk 24 is the de-facto modern Android floor (~98% device coverage). targetSdk 34 is current Play Store floor and matches setup.ps1's installed `platforms;android-34`. |
| Q4 | Cross-compile = **NDK 26.3 + `cargo-ndk` wrapper crate** | cargo-ndk handles per-ABI env vars in one command; NDK 26.3 is documented-known-good with wasmtime; one cargo install vs 8-12 lines of manual CC/AR exports. |
| Q5 | .wasm bundling = **Gradle `copyWasm` task pulls from .NET publish output into `src/androidMain/assets/`** | Automation prevents stale-.wasm bugs (#1 source of "works on desktop, fails on Android" false positives); `AssetManager.open` is idiomatic. |
| Q6 | Stdout capture = **`wasi_config_set_stdout_file` → `Context.cacheDir/wasm-stdout.txt` → read back → `Log.i("BlazorNative", line)` per line** | Reuses Phase 2.1's proven mechanism; `cacheDir` is the right Android temp scope; Log.i bridges to logcat (the standard Android observation tool). |
| Q7 | Verification = **Instrumented `BootSmokeAndroidTest` via `./gradlew connectedAndroidTest`** + minimal `MainActivity` with `TextView` for visual confirmation | Mirrors Phase 2.1's `BootSmokeTest` shape exactly; standard Android verification gate; future regression catch when Phase 2.3 lands. |
| Q8 | Scope = **AVD setup automated in `setup.ps1`** (sdkmanager system-image + avdmanager create avd). [USER] gate: start the emulator before `connectedAndroidTest`. | MILESTONE DoD #2 explicitly says "installs to an emulator"; punting verification to a sub-phase is lazy framing. AVD setup is mostly automatable. |

### Limitations carried forward

1. **Linux/macOS `setup.ps1` parity** — still backlog (Phase 2.1 deferral). Phase 2.2 inherits the Windows-only constraint.
2. **`mobile_bridge` imports still absent** from the .wasm — Mono-AOT-trimmed dead code from Phase 2.1.0 finding. Phase 2.3 picks `[DynamicDependency]` vs wit-bindgen migration.
3. **JNA crossing cost** — same per-call boundary tax as Phase 2.1; acceptable for boot path, revisit if Phase 2.4+ render loop shows it as a bottleneck.
4. **No physical device verification** — emulator suffices for M2 DoD. Marcel can manually test on a physical device if he has one; not a phase gate.
5. **No APK shrinking / R8** — debug build, no minification. M4 hardens.

**Key flow at app launch:**

```
MainActivity.onCreate
  ↓ spawns background thread (UI-blocking work avoided)
WasiHost.loadAndRun(wasmBytes, context.cacheDir)
  ↓ context.assets.open("BlazorNative.WasiHost.wasm").readBytes()
  ↓ JNA loads libwasmtime.so from /data/app/.../lib/<abi>/libwasmtime.so
  ↓ SAME WasmtimeBindings calls as JVM (Phase 2.1)
  ↓ wasmtime config + linker_add_wasip2 + component_new + instantiate + func_call
  ↓ .wasm Main → emits [BOOT] markers to wasi-preview2 stdout
  ↓ stdout captured to context.cacheDir/wasm-stdout-${UUID}.txt → read back
captured stdout returned as String

1. stdout.lines().forEach { Log.i("BlazorNative", line) }  ← logcat
2. runOnUiThread { textView.text = stdout }                ← UI
```

## Components

### `setup.ps1` extension (~50 lines added across two sections)

**Section 7 (Android SDK) — extend sdkmanager install list:**

```powershell
echo "y" | & $sdkmanager `
    "platform-tools" `
    "build-tools;34.0.0" `
    "platforms;android-34" `
    "ndk;26.3.11579264" `                                            # NEW
    "system-images;android-34;google_apis;x86_64"                    # NEW (~600 MB)
```

Then create the AVD if it doesn't exist:

```powershell
$avdmanager = "$cmdToolsDest\latest\bin\avdmanager.bat"
$avdName = "blazornative-pixel6-x86_64"
$existingAvds = & $avdmanager list avd -c 2>&1
if ($existingAvds -notcontains $avdName) {
    Write-Step "Creating AVD $avdName..."
    echo "no" | & $avdmanager create avd `
        -n $avdName `
        -k "system-images;android-34;google_apis;x86_64" `
        -d "pixel_6"
    Write-OK "AVD $avdName created"
}
```

Set `ANDROID_NDK_HOME` env var (cargo-ndk derives linker paths from it):

```powershell
$ndkRoot = "$androidHome\ndk\26.3.11579264"
[Environment]::SetEnvironmentVariable("ANDROID_NDK_HOME", $ndkRoot, "User")
$env:ANDROID_NDK_HOME = $ndkRoot
```

**Section 8c (new) — Android libwasmtime cross-compile:**

```powershell
Write-Header "8c · libwasmtime for Android ABIs (cargo-ndk + NDK 26.3)"

if (-not (Command-Exists "cargo-ndk")) {
    Write-Step "Installing cargo-ndk..."
    cargo install cargo-ndk
}

Write-Step "Adding Rust targets for Android..."
rustup target add aarch64-linux-android x86_64-linux-android

$jniLibsDir = Join-Path $PSScriptRoot "vendor\wasmtime\jniLibs"
if ((Test-Path "$jniLibsDir\arm64-v8a\libwasmtime.so") -and (Test-Path "$jniLibsDir\x86_64\libwasmtime.so")) {
    Write-OK "Android libwasmtime.so already built (both ABIs)"
} else {
    Push-Location "$PSScriptRoot\vendor\wasmtime-src"
    Write-Step "Cross-compiling libwasmtime for arm64-v8a + x86_64 (~10-15 min)..."
    cargo ndk -t arm64-v8a -t x86_64 -o $jniLibsDir build -p wasmtime-c-api --release
    Pop-Location
    if ((Test-Path "$jniLibsDir\arm64-v8a\libwasmtime.so") -and (Test-Path "$jniLibsDir\x86_64\libwasmtime.so")) {
        Write-OK "Android libwasmtime.so built (both ABIs)"
    } else {
        Write-Fail "cargo ndk build did not produce expected .so files"
    }
}
```

### `src/BlazorNative.Jni/build.gradle.kts` (significantly modified)

Add the Android Gradle Plugin + `kotlin("android")` to the existing `plugins` block. New `android { ... }` block configures namespace, compileSdk, defaultConfig (minSdk + ndk.abiFilters), sourceSets, compileOptions. New `dependencies` entries for `androidx.test.*` (androidTest). Two new `tasks.register<Copy>` blocks for `copyWasm` and `copyJniLibs`, both wired as `preBuild.dependsOn`.

Full file structure documented in design Section 2 above; ~100 lines added.

### `src/BlazorNative.Jni/src/main/kotlin/io/blazornative/jni/WasiHost.kt` (refactored)

The Phase 2.1 implementation takes `wasmPath: Path` and writes stdout to `Files.createTempFile`. Phase 2.2 needs a `tempDir: File` parameter (Android passes `context.cacheDir`, JVM passes `Files.createTempDirectory().toFile()`).

New shape: `loadAndRun(wasmBytes: ByteArray, tempDir: File): String`. The Phase 2.1 `loadAndRun(wasmPath: Path)` signature is preserved as a thin delegate (so JVM `BootSmokeTest` keeps working unchanged).

### `src/BlazorNative.Jni/src/androidMain/kotlin/io/blazornative/shell/MainActivity.kt`

~50-line Activity. Inflates a single-`TextView` layout. On `onCreate`, spawns a background thread that reads the .wasm bytes from assets, calls `WasiHost.loadAndRun(bytes, cacheDir)`, emits captured stdout to logcat (`Log.i("BlazorNative", line)` per line), updates the TextView via `runOnUiThread`. Catches all throwables and shows `"FAIL: ${t.message}"` in the TextView + `Log.e` for diagnosis.

### `src/BlazorNative.Jni/src/androidMain/AndroidManifest.xml`

Minimal manifest: `<application>` with single `<activity android:name=".shell.MainActivity" android:exported="true">` declaring LAUNCHER intent filter. No permissions (assets are part of the APK, cacheDir is app-private by default).

### `src/BlazorNative.Jni/src/androidMain/res/layout/main.xml`

Single `<ScrollView>` wrapping a monospace `<TextView android:id="@+id/markers">`. Background black, text green — looks like a console for the demo.

### `src/BlazorNative.Jni/src/androidTest/kotlin/io/blazornative/jni/BootSmokeAndroidTest.kt`

~40 lines. JUnit 4 + `AndroidJUnit4` runner. Uses `InstrumentationRegistry.getInstrumentation().targetContext` to access assets + cacheDir. Calls `WasiHost.loadAndRun(wasmBytes, context.cacheDir)`. Asserts the 4 `[BOOT]` markers via `assertTrue(stdout.contains(...))`. Failure messages include the full captured stdout for diagnosis.

## Data Flow

### Build pipeline (one-time + per-iteration)

**ONE-TIME (setup.ps1, ~30 min first run):**
- sdkmanager: install ndk;26.3 + system-images;android-34
- avdmanager create avd → "blazornative-pixel6-x86_64"
- cargo install cargo-ndk
- rustup target add aarch64-linux-android x86_64-linux-android
- cargo ndk build → vendor/wasmtime/jniLibs/{arm64-v8a, x86_64}/libwasmtime.so (~25 MB each)

**PER-ITERATION (./gradlew assembleDebug, ~30-60s):**
- preBuild: `copyWasm` + `copyJniLibs` Gradle tasks
  - `BlazorNative.WasiHost.wasm` → `src/androidMain/assets/`
  - `libwasmtime.so` (per ABI) → `src/androidMain/jniLibs/<abi>/`
- compileKotlin (JVM-shared sources) + compileAndroidKotlin (androidMain sources)
- packageDebug → `app/build/outputs/apk/debug/BlazorNative.Jni-debug.apk`

### Runtime flow on Android (after `adb install` + app launch)

```
1. Android launches MainActivity (LAUNCHER intent in manifest)
2. MainActivity.onCreate
     - inflates layout (TextView)
     - spawns background thread
3. Background thread:
     - context.assets.open("BlazorNative.WasiHost.wasm").readBytes()  (~13 MB from APK)
4. WasiHost.loadAndRun(wasmBytes, context.cacheDir)
5. JNA loads libwasmtime.so
     - NativeLibraryLoader picks /data/app/.../lib/<abi>/libwasmtime.so
     - First call cost: ~50-200 ms (mmap + symbol resolution)
6. WasmtimeBindings.INSTANCE → same C-API sequence as JVM (Phase 2.1)
     - context_set_wasi: stdout file = File(cacheDir, "wasm-stdout-${UUID}.txt")
                         preopen_dir = cacheDir (writable space for the .wasm)
                         argv = ["BlazorNative.WasiHost.wasm"]
7. wasmtime_component_linker_instantiate → wasmtime_component_func_call
     - Cranelift JIT compiles WASM → native code (ARM64 or x86_64)
     - First-call cost: ~300-800 ms (cold JIT compile)
     - Runs Mono-AOT'd .NET in-WASM → emits 4 [BOOT] markers to stdout
8. Read back stdout file → returns captured String
9. MainActivity (background thread):
     - stdout.lines().forEach { Log.i("BlazorNative", it) }
     - runOnUiThread { textView.text = stdout }
```

### Test data flow (`./gradlew connectedAndroidTest` — the GREEN CHECKPOINT)

1. Gradle detects running adb device.
2. Assembles main APK + test APK; adb installs both.
3. `adb shell am instrument` invokes AndroidJUnit4 runner.
4. `BootSmokeAndroidTest.boots_and_emits_markers_on_android` runs on device — same code path as MainActivity.
5. `assertTrue` calls on captured stdout report PASS/FAIL back via adb to Gradle.
6. Captured stdout appears in the test failure message if any marker is missing (diagnostic IS the stdout).

## Error Handling

### Boundary 1: Toolchain / build failures (setup.ps1 + cargo-ndk)

Fail-fast. setup.ps1 reports the failing step + exit code and stops. Common failures + diagnostics documented inline; Marcel diagnoses and reruns. No retry logic — too many distinct failure modes.

| Failure | Fix |
|---|---|
| `cargo install cargo-ndk` network error | Re-run; rustup auto-recovers |
| `cargo ndk` "linker not found" | Set `$env:ANDROID_NDK_HOME` to the NDK 26.3 install dir |
| `libwasmtime.so` huge (>100 MB) | Verify `--release` flag |
| sdkmanager license fail | Manual `sdkmanager --licenses` accept |

### Boundary 2: APK build / packaging

| Failure | Fix |
|---|---|
| "Could not find libwasmtime.so" | Run setup.ps1 to cross-compile; then `./gradlew clean assembleDebug` |
| APK missing `BlazorNative.WasiHost.wasm` | Run `dotnet publish ...` first; copyWasm depends on output being current |
| "Manifest merger failed" | Minimize manifest; rely on defaultConfig for sdk versions |
| "androidMain source set unrecognized" | Explicitly declare in sourceSets block (already in build.gradle.kts spec) |

### Boundary 3: Runtime failures on device (most interesting category)

| Failure | Diagnosis | Fix |
|---|---|---|
| `UnsatisfiedLinkError: dlopen libwasmtime.so` | .so missing for current ABI, or API level mismatch | `unzip -l app.apk \| grep libwasmtime`; check cargo-ndk API level ≤ 24 |
| TextView stays "Loading..."; logcat shows trap | wasmtime trap on `wasi:cli/run` | Read trap message; most likely `preopen_dir` or `argv` issue |
| Partial markers (e.g. only `runtime-start`) | Mono-AOT crashes mid-startup | Examine Mono crash trace in logcat |
| `BootSmokeAndroidTest` fails: stdout empty | UUID mismatch in stdout file path | Use stable shared variable in `WasiHost.kt` |
| Markers mangled (lines interleaved) | Log.i flush not atomic per-line | Buffer full stdout, split on `\n`, emit each Log.i separately |

### What we deliberately don't handle

- Emulator not running → `connectedAndroidTest` fails with "No connected devices." Self-explanatory.
- Network failures during sdkmanager → Marcel reruns.
- Multi-device disambiguation → assumes one connected device.
- APK signing for release → debug builds only.
- OOM during cargo-ndk → standard dev box handles 4 GB peak; if not, ramp Cargo `-j` down.

## Open risks (carried into implementation)

1. **NDK 26.3 / wasmtime v45 ABI compatibility.** Documented-known-good per upstream issues, but `cargo ndk build` could surface unexpected linker errors. Fallback: try NDK 27 if 26.3 fails; if both fail, drop to Phase 2.2.0 spike to identify the right combo.
2. **JNA on Android may need additional `.so`s for itself.** JNA ships its own native code (`libjnidispatch.so`). The Maven artifact `net.java.dev.jna:jna:5.14.0@aar` (or the `-aar` classifier variant) bundles them automatically for Android. If using the plain `.jar` variant from Phase 2.1, Phase 2.2 must switch to the `aar` for Android source set.
3. **AVD x86_64 emulator performance on Marcel's box.** Modern x86_64 emulators need HAXM / Hyper-V acceleration. If neither is available, emulator startup may take 60s+ or fail entirely. Fallback: arm64 emulator (requires Windows on ARM or extremely recent x86 hosts).
4. **`wasmtime_context_set_wasi` preopen_dir behavior on Android.** Android's cacheDir is on /data, which has SELinux constraints. wasmtime may need additional perms or path translation. If preopen fails, fall back to passing `null` to set_wasi (no filesystem) — but then the .wasm can't write stdout to a file. Fallback strategy: use a memory-mapped stdout buffer via wasmtime hostfunc rather than wasi file.

These are all "discovered during the work" risks, not "design choices we punted on." Each has a documented fallback.
