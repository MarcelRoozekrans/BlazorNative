## Verdict

**GREEN — DoD #1 CLOSED, for the repo AND for consumers, arch-independent.** The deep-link
route→component map — the one map that was read by Kotlin at Intent-parse time *before* .NET loads,
and so the one thing that could not derive from the app's page registry the way everything else does
— is now **generated from source at build time** and **shipped to consumers in the Runtime package**.
An app author who adds a routed page writes one row in `AppPages.All` and the deep link works; there
is no shell file to hand-edit and no silent-wrong-screen footgun. The consumer-footgun audit
(Gate C) then confirmed that map was genuinely the *last* page-keyed hand-edit, enumerated every
other surface a real app author touches, and derived or documented each.

## As-builts

### Gate A — codegen over the app's *source* (the arm64 pivot)

The first Gate-A cut **loaded the built app assembly** (isolated `AssemblyLoadContext`, ran the
`[ModuleInitializer]`, reflected `PageManifest`) — and it could not survive CI. The app is published
**per-RID**, and a `linux-bionic-arm64` managed dll cannot load into the x64 build host: *"The
assembly architecture is not compatible with the current process architecture."* Real Android devices
are arm64, so the generator has to work for arm64. Gate A took the design's named fallback: **Roslyn
source analysis.** `BlazorNative.RouteGen` now parses the app's C# source for
`BlazorNativePage.Routed<T>(route, name)` invocations and emits `(route, name)`. It loads
**nothing** — no assembly, no RID, no arch — so its output is **byte-identical for win-x64,
linux-bionic-x64 and linux-bionic-arm64.** The MSBuild target's input moved from `$(TargetPath)` (the
RID dll) to `@(Compile)` (the source files), so it no longer depends on `IlcCompile` producing a
loadable dll. Matching is syntactic (method `Routed`, one generic arg, exactly two args); `Named<T>`
is excluded by shape. The one framework-owned non-literal, `BlazorNativeApp.DefaultRoute` (the
`const "/"`), the generator resolves; any *other* non-literal errors with file+line rather than
silently dropping a row.

### Gate B — shipped to consumers

RouteGen rides inside the `BlazorNative.Runtime` NuGet package, and `build/BionicNativeAot.targets`
(auto-imported into every consumer app) invokes it: a `dotnet new blazornative` app runs the same
source-analysis codegen against *its own* `AppPages.cs` and gets *its own* `res/raw/blazornative_routes.json`.
The template-smoke lane proves it end-to-end — a generated app with an added routed page compiles and
its `routes.json` contains the added page. The resource is a build artifact (gitignored); the drift
test regenerates and compares, so there is no checked-in copy to drift.

### The drift test, reshaped (not retired)

`RouteTableDriftTests` flipped from *"the hand-written Kotlin `mapOf` mirrors `SampleAppPages`"* to
*"the generated resource reproduces `SampleAppPages`'s routed rows."* The pair-for-pair Kotlin-text
pin is **retired** — there is no hand-written map left to drift.
`GeneratedRoutesJson_ReproducesTheManifestsRoutedRows_PairForPair` runs the real RouteGen extractor
against the real SampleApp source, serializes it the way the target does, parses it back the way
`MainActivity` does, and holds it pair-for-pair against `SampleAppPages.All` (the `/` default row
included) — two independent derivations of the one source. Kept verbatim: the
`AndroidDefaultFallbackLiteral…` pin (the resource-less ultimate fallback must still name the default
component) and the 13-page content baseline (a drift test comparing two surfaces is blind to a row
deleted from both).

### Gate C — the audit's findings

The governing fact: **the Android shell is copied source, not a linked library** — no `.aar`, so no
manifest-merge, so the consumer's app manifest *is* the template manifest. That is why every
capability's Android surface is **DERIVED (template-supplied)**: `INTERNET` / `POST_NOTIFICATIONS` /
`USE_BIOMETRIC`, the camera `ACTION_IMAGE_CAPTURE` `<queries>`, the `FileProvider` `<provider>` +
`file_paths.xml`, and the notification `<receiver>` are all pre-declared in the manifest the consumer
receives — nothing to hand-add to use a capability. The FileProvider authority
(`${applicationId}.fileprovider`) derives cleanly because AGP substitutes the consumer's `applicationId`
at build and the shell reads `packageName` at runtime.

What remains is genuinely un-derivable and is **DOCUMENTED**: app identity (`namespace`/`applicationId`
— it *is* the input), the deep-link URI scheme (a per-app choice, and two `blazornative://` apps
collide on one device), the iOS usage-description **strings** (the keys copy for free; the sentence is
app-specific copy the App Store scrutinizes), the iOS root-component source edits (no iOS template or
generator — "on iOS you are the drift test"), and the iOS csproj/link recipe (Android's
`BionicNativeAot.targets` has no iOS equivalent yet). A grep of the Android shell confirmed **not one
`when`/`switch`/map is keyed per page** — every one is generic (node-type, patch-type, style property,
bridge op). The deep-link map really was the last of its kind.

Three consumer docs carried the *old* footgun and were fixed: `getting-started/quick-start.md` and
`shells/android.md` both still told readers to hand-edit `DEEP_LINK_COMPONENTS` (now: the map is
generated, one row in `AppPages.All`), and `shells/ios.md` gained an explicit note that the
usage-description strings are app-specific copy to rewrite, not just keys to keep. `shells/android.md`
also gained a **Capabilities and permissions** table (what is pre-declared, and the trim-before-ship
note).

## The headline

The seed finding is closed. The deep-link single-source-of-truth violation is gone for **the repo
and consumers**, and it is **arch-independent** — the generator reads source, so the same output falls
out for the emulator's x64 and a real device's arm64. Adding a routed page is one row; every derived
view of that row, the Android deep-link map now included, follows for free. The audit confirms no
other page-keyed hand-edit survives, and the capability surfaces a consumer cannot avoid are the ones
that are inputs, not outputs — documented where a consumer will look.

## Verification (CI-verified vs local)

- **.NET (`dotnet test BlazorNative.sln -c Release`):** run locally on this branch — see the count
  reported in the phase's commit message / the final audit. Gates A/B's suites (RouteGen unit tests,
  the reshaped `RouteTableDriftTests`, `TemplateDriftTests`) are **CI-verified** on their merged PRs.
- **arm64 + the instrumented lane:** CI-only — they cannot run on the dev host. The controller re-runs
  all three RIDs; Gate A's whole point is that the source-analysis generator makes arm64 pass by
  construction (no dll load).
- **Docusaurus build (`website && npm run build`):** the Gate-C doc fixes touch `website/docs/**`, so
  the build is run locally on this branch and is the gate that this prose still compiles.
- **This gate (C/D) is docs-only** — no shell source, manifest, or template file changed, so no
  `TemplateDriftTests` re-run is triggered by it (Gates A/B already synced the mirror).

## Carryovers (unchanged by this phase, noted for continuity)

- **The `.aar` / SwiftPM packaging** — the day the Android shell ships as a library, the
  template-supplied manifest entries become *merged-from-library* entries (still DERIVED; the
  mechanism changes). Backlog.
- **The iOS recipe into `BlazorNative.Runtime`** — moving the iOS `PropertyGroup` + link recipe into
  the package (auto-imported for both platforms, as Android's targets already are) is the right
  architecture and stays backlog; it costs a shipped package's inventory shape.
- **iOS deep links** — iOS mounts by name and has no route surface; the day it grows a deep-link
  story it gains its own resource + a `[Fact]` in `RouteTableDriftTests`. A decision, not an accident.
