## Objective

Blazor code on Android reaches navigation, storage, and HTTP through `IMobileBridge` ŌĆö
implemented by a new Runtime-side `NativeShellBridge` that invokes host-registered JNA
callbacks. `BridgeHttpHandler` (and therefore plain injected `HttpClient`) works on Android
for the first time. DevHost keeps `DevHostBridge` untouched.

Direction note: all prior exports are hostŌåÆ.NET calls. The bridge is the reverse ŌĆö .NET
calls INTO the host. Mechanism: the host registers a struct of function pointers once at
boot (`blazornative_register_bridge`), exactly like the frame callback but ├Ś6.

## Decisions locked during the brainstorm (2026-07-09)

| Decision | Choice |
|---|---|
| Fetch shape | **Async completion pattern.** `FetchBegin(requestId, request*)` returns immediately; host performs the HTTP call on its own dispatcher and calls `blazornative_fetch_complete(requestId, response*)`. No blocked threads; builds the hostŌåÆ.NET completion machinery Phase 3.2's events need. |
| String returns | **Caller-allocated buffers** (4 KB default; `-needed` retry protocol). No host-allocated returns ŌåÆ no cross-ABI ownership question. |
| Storage backing | Android `SharedPreferences("blazornative")`. |
| Navigation semantics in 3.1 | Round-trip only (in-memory current-route var + log). Real navigation UI is M3 DoD #7, a later phase. |
| WasiBridge fate | **Deferred to 3.2.** Its remaining live surface is the `NativeEvents` multicast + `BridgeEventTests` ŌĆö exactly what 3.2 redesigns. Deleting now would orphan those tests twice. `IMobileBridge` (contract) and `DevHostBridge` stay. |
| Fetch HTTP client (Android) | `HttpURLConnection` on a background executor ŌĆö no new dependency. |

## C-ABI

### Registration struct (`BridgeProtocolNative.cs`, layout drift-tested both sides)

```csharp
[StructLayout(LayoutKind.Sequential)]
public struct BlazorNativeBridgeCallbacks       // 6 ├Ś IntPtr = 48 bytes
{
    public IntPtr Navigate;        // int (const char* routeUtf8)
    public IntPtr CurrentRoute;    // int (char* buf, int cap)
    public IntPtr StorageRead;     // int (const char* keyUtf8, char* buf, int cap)
    public IntPtr StorageWrite;    // int (const char* keyUtf8, const char* valueUtf8)
    public IntPtr StorageDelete;   // int (const char* keyUtf8)
    public IntPtr FetchBegin;      // int (long requestId, BlazorNativeFetchRequest* req)
}
```

Return-code protocol (sync calls): `>= 0` success (for buffer calls: bytes written incl.
NUL); `-needed` buffer too small (caller retries once with exact size); `-1` host error;
`-2` key absent (`StorageRead` ŌåÆ maps to `null`). A throw inside a Kotlin callback is
caught host-side and surfaces as `-1` ŌĆö never crosses the ABI.

### Fetch structs + completion export

```csharp
[StructLayout(LayoutKind.Sequential)]
public struct BlazorNativeFetchRequest   // .NET-owned, valid ONLY during FetchBegin
{ public IntPtr Url; public IntPtr Method; public IntPtr Body; /* NULL ok */ public IntPtr HeadersJson; /* flat {"k":"v"} */ }

[StructLayout(LayoutKind.Sequential)]
public struct BlazorNativeFetchResponse  // host-owned, valid ONLY during fetch_complete
{ public int StatusCode; public int Ok; /* 0 = transport error */ public IntPtr BodyUtf8; public IntPtr ErrorMessage; public IntPtr HeadersJson; }

[UnmanagedCallersOnly(EntryPoint = "blazornative_fetch_complete")]
public static unsafe int FetchComplete(long requestId, BlazorNativeFetchResponse* resp);
// 0 = delivered; 1 = unknown/already-completed id (logged, ignored ŌĆö cancellation race)

[UnmanagedCallersOnly(EntryPoint = "blazornative_register_bridge")]
public static unsafe int RegisterBridge(BlazorNativeBridgeCallbacks* callbacks); // copies the struct
```

Lifetime rules mirror the frame protocol: each side copies inside the call it receives.
Headers cross as a flat JSON object string (test-project serialization only; the Runtime
side hand-writes/hand-parses the flat object ŌĆö ~30 lines, no JsonSerializerContext revival).

## .NET side

- **`NativeShellBridge : IMobileBridge`** (BlazorNative.Runtime): sync methods wrap the
  callbacks via small `delegate* unmanaged[Cdecl]` invokers + `ValueTask.FromResult`;
  buffer calls use a rented 4 KB buffer with the one-retry protocol. `FetchAsync`:
  `Interlocked` id + `ConcurrentDictionary<long, TaskCompletionSource<BridgeHttpResponse>>`
  (`RunContinuationsAsynchronously` ŌĆö completion arrives on a Kotlin thread); ct
  registration cancels the TCS (late completions hit the unknown-id path). `PlatformInfo`:
  `Init` starts STORING its options (it already receives them); the bridge serves them.
  `NativeEvents`: documented no-op add/remove until 3.2.
- **DI**: `HostSession` registers `NativeShellBridge` as `IMobileBridge` ŌåÆ `BridgeHttpHandler`
  resolves on Android. Unregistered-host calls throw
  `InvalidOperationException("shell bridge not registered ŌĆö call blazornative_register_bridge before mount")`.
- **`blazornative_run_bridge_probes`** ŌĆö temporary diagnostic export (same pattern + fate
  as trim probes; both delete at M3 close): navigateŌåÆcurrent-route round-trip, storage
  write/read/delete (+absent-key `null`), one fetch. Status = failed count, detail via the
  usual InitResult reuse. The fetch probe target URL arrives as a probe argument
  (`blazornative_run_bridge_probes(const char* fetchUrlUtf8)`) so each host points it at
  its own test server.

## Kotlin side

- **`ShellBridgeHandlers`** (main source set, interface): `navigate(route)`,
  `currentRoute(): String`, `storageRead(key): String?`, `storageWrite`, `storageDelete`,
  `fetchBegin(requestId, request)` + a `BridgeRegistrar` that builds the JNA callback
  struct, HOLDS IT STRONGLY (frame-callback GC rule), wraps every callback body in
  try/catch ŌåÆ `-1` + onError (JNA-swallow posture), and implements the buffer-write
  protocol helper once.
- **`AndroidShellBridge`** (androidMain): route = `@Volatile` var + log; storage =
  `SharedPreferences("blazornative")`; fetch = `HttpURLConnection` on a single background
  executor, completing via `blazornative_fetch_complete` (response memory allocated as JNA
  `Memory`, kept alive across the call).
- **`BlazorNativeRuntime.start()`** grows an optional `bridge: ShellBridgeHandlers?`
  parameter ŌĆö registered (before mount) when provided; MainActivity passes
  `AndroidShellBridge`.
- **JVM tests**: in-memory `ShellBridgeHandlers` impl + JDK `com.sun.net.httpserver` for
  real-HTTP fetch coverage. **Instrumented tests**: `AndroidShellBridge` + a ~30-line
  `ServerSocket` responder thread (no test dependency; localhost is cleartext-exempt).

## Risks

| Risk | Mitigation |
|---|---|
| Completion-thread reentrancy (fetch completes during mount) | TCS with RunContinuationsAsynchronously decouples; no renderer state touched on the completion path |
| Buffer-retry protocol drift between the two sides | Gate 1 fake-callback tests pin the .NET half; Gate 2 real-host tests pin the Kotlin half; return-code table documented in BOTH BridgeProtocolNative.cs and ShellBridgeHandlers.kt |
| Callback struct layout drift | Same dual drift-test discipline as the patch structs (48-byte size + per-field offsets) |
| HttpURLConnection quirks on API 34 | Cleartext test traffic stays on localhost (exempt); probe URL is parameterized |
| JNA `Memory` lifetime on fetch_complete | Kotlin keeps the Memory refs in locals across the export call; rule documented at the struct |
