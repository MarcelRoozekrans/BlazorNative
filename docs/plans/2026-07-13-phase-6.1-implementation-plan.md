# Phase 6.1 Implementation Plan — Flexbox Layout Core

> **For Claude:** REQUIRED SUB-SKILL: `superpowers:subagent-driven-development` — fresh implementer subagent per task, spec review then code-quality review after each.

**Goal:** Yoga stops being a probe and becomes the engine: every node lives in a Yoga tree,
containers are plain `FrameLayout`/`UIView`, every child is placed at a computed frame, and
leaves are measured natively — identically on Android and iOS.

**Design:** [2026-07-13-phase-6.1-design.md](2026-07-13-phase-6.1-design.md) (read it — the four
locked decisions and the layout model are there and are not up for re-litigation).

**Architecture:** Typed C# flex params → stringified onto the **existing** `SetStyle` wire → each
shell maps the string to a Yoga style setter → one `calculateLayout` per committed frame → frames
applied to views. **No ABI change** (9 exports + the 72-byte bridge, untouched).

**Tech stack:** .NET 10 / NativeAOT · Kotlin + `com.facebook.yoga:yoga:3.2.1` (JNI bindings) ·
Swift + Objective-C++ over Yoga's C-API (`libyoga.a`, built from source in `ios.yml`).

**Branch:** `phase-6.1-flexbox-core` (already created; the 6.0 close-out docs + this design ride on it).

## Non-negotiables (violating any of these fails spec review)

1. **No C-ABI / wire change.** No new export, no patch-struct change, no bridge-struct change.
   Flex props ride `SetStyle` (kind 6). CI asserts 9 exports on 3 RIDs — keep it that way.
2. **iOS: Yoga's headers must never be visible to Swift.** All Yoga interop lives in
   Objective-C++ (`.mm`) behind a plain-C surface. Xcode's Swift explicit-module dependency
   scanner reads the bridging header with a path-less search honouring neither
   `HEADER_SEARCH_PATHS` nor `-Xcc -I`. This cost six red CI runs in 6.0 — do not re-learn it.
3. **The text-collapse invariant.** A `text` node whose parent is a text-bearing non-container
   gets no view *and must get no Yoga node* — otherwise the two trees' child indices skew.
4. **An un-styled tree must emit no new patches and still stack vertically.** All new `BnView`
   params default to null → no attribute → no patch. `BnDemoTests.cs` / `BnDemoTest.kt` goldens
   must not churn.
5. **Yoga computes in density-independent units on both platforms.** iOS points 1:1; Android
   multiplies by `density` at frame-apply time only. One conversion site. The corollary is the
   **value grammar**, which has exactly ONE normative statement —
   [design §"Style value grammar (normative)"](2026-07-13-phase-6.1-design.md#style-value-grammar-normative).
   Both shell parsers are written from it; nothing restates it. Headline: bare numbers are dp/pt,
   `N%` and `auto` are the only other length forms, `flexGrow`/`flexShrink` are unitless floats,
   null resets, **and there are NO unit suffixes** (`px` is not dp; `sp` is font-scaled; neither
   exists on iOS). A shell may defensively strip a trailing `dp`/`px`/`sp` **and log** — the
   existing `fontSize`/`padding` handlers do, and must keep doing so for the legacy visual props —
   but nothing in .NET emits one.

6. **The measure function attaches BY NODETYPE — `text` / `button` / `input` / `image`.** NOT by
   "this node has no children". They are different sets: `BnLayoutDemo`'s row is three *childless*
   `view`s, and giving them a measure func lets the native view's intrinsic size speak over Yoga,
   destroying the `Grow=1` box's computed width and with it every number the parity assertion
   rests on. A childless container is a container.

7. **Yoga's `alignContent` default is `flex-start` — it DEVIATES from CSS** (where it is
   `stretch`). The wrap row's second line sits at `y = 40` (the line-1 cross size) *because* of
   that. A shell that applies a CSS-ish default silently shifts the frame, and since nothing sets
   `alignContent` (it is not even on the allow-list) there is no patch to catch it. Do not
   "correct" Yoga here.

8. **The SetStyle allow-list is PARTITIONED, and the partition is the shells' routing table.**
   `YogaStyleAttributes` → the node's Yoga node; `VisualStyleAttributes` → the View/UIView. Every
   style name goes to exactly one. In particular **`padding`, `margin`, `width` and `height` are
   LAYOUT** — Yoga's, not the view's (see the Gate 2 note: Android must stop calling `setPadding`).
   Matching is **ordinal/case-sensitive**, mirroring both shells.

## Gate 2 — Android: Yoga owns placement

### Task 2.1 — The Yoga layout tree (Kotlin)

**Files:** create `src/BlazorNative.Jni/src/androidMain/kotlin/io/blazornative/shell/YogaLayout.kt`;
modify `WidgetMapper.kt`; instrumented test
`src/BlazorNative.Jni/src/androidTest/kotlin/io/blazornative/shell/YogaLayoutAndroidTest.kt`.

- `YogaLayout` owns `nodeId -> YogaNode`, mirrors create/insert/remove against the view tree
  (respecting the text collapse — non-negotiable #3), maps style name+value → Yoga setter, and
  runs `calculateLayout` + applies frames (`view.layout(l, t, r, b)` in **px** = dp × density).
- `SoLoader.init(context)` once at shell start (the 6.0 conclusion's Android note).
- Container for `view` becomes a plain `FrameLayout` (children are absolutely placed by Yoga).
- **Style value grammar: implement it FROM
  [design §"Style value grammar (normative)"](2026-07-13-phase-6.1-design.md#style-value-grammar-normative).**
  Do not re-derive it from this plan, and do not restate it in a comment — that section is the
  single source both this parser and the `.mm` are built from. A **null value resets** the
  property to its Yoga default (non-negotiable: the 1.2 fix's other half).
- **Route by the PARTITION, and delete the double-apply (non-negotiable #8).** A `SetStyle` name
  in `NativeRenderer.YogaStyleAttributes` goes to the **Yoga node**; one in
  `VisualStyleAttributes` goes to the **View**. Concretely: `WidgetMapper` currently sends
  `padding` to `view.setPadding(...)` — **that call must go.** Yoga places a container's children
  *inside* its padding box, so once the Yoga node owns `padding`, a surviving `setPadding` applies
  it a second time and every child in that container is off by the padding. Same for
  `width`/`height`/`margin`: they are LAYOUT and belong to Yoga alone.

TDD: instrumented test asserts the canonical row frames (the 6.0 spike's numbers are the template),
then a `null` reset returns a node to default.

### Task 2.2 — Native measurement (Android)

Measure func on the measurable NODETYPES — `text`, `button`, `input`, `image` (non-negotiable #6:
**by nodetype, never by "has no children"**): `view.measure(MeasureSpec.makeMeasureSpec(...))` with
Yoga's mode mapping (`Exactly→EXACTLY`, `AtMost→AT_MOST`, `Undefined→UNSPECIFIED`), returning
measured size ÷ density. Instrumented test: a long `BnText` in a narrow row **wraps** and its
measured height drives the row — *and* the `BnLayoutDemo` row's childless `view` boxes keep their
Yoga-computed widths (the regression that "leaf = no children" would cause).

### Task 2.3 — Relayout on host resize

`onLayout`/`onConfigurationChanged` on the host root re-runs calculate+apply with the new bounds.
Instrumented test: change the root's size, assert frames recompute.

### Task 2.4 — Rewrite the churned instrumented tests

`WidgetMapperTest`, `BnDemoAndroidTest`, `CompositionAndroidTest`, `WidgetMapperSetStyleTest`: swap
type-pins (`LinearLayout`, `orientation == VERTICAL`) for **frame pins** (child count + each child's
computed frame). `WidgetMapperSetStyleTest`'s "unknown property → still a LinearLayout" premise is
inverted by this phase — replace it with "unknown property → logged and ignored, frames unchanged".
Keep `WidgetMapperTextChildOnButtonTest` (the collapse invariant) passing untouched.

### Task 2.5 — `BnLayoutDemo` frames on the AVD + Gate 2 close

Instrumented test asserting **every** frame of the layout demo. Bump + assert the instrumented

## Gate 4 — Close

### Task 4.0 — The frame table must not be two hand-retyped copies

Today the expected frames live in an **ASCII comment** in `BnLayoutDemo.cs`, and Gates 2 and 3
will hand-retype them into a Kotlin test and a Swift test. That is the one thing this repo's house
style does not permit: contracts mirrored across .NET/Kotlin/Swift are held together by a **drift
test** (the `PatchProtocol`/`FrameEncoder`/export-count precedents), never by a human re-reading a
comment. A typo'd `40` in one of the two files is a *silent* parity failure — it fails as
"Android and iOS disagree", i.e. as the exact bug DoD #2 exists to catch, and the engine gets
blamed for a fat finger.

**Recommendation (adopt this): a Gate 4 drift test asserting the Kotlin and Swift expectation
tables are identical.** Not a shared data file. Reasons: (a) both shells' test runners already
read only source (no resource-loading plumbing exists on either lane, and adding it to the iOS
XCTest bundle *and* the Android instrumented APK is more moving parts than the problem is worth);
(b) the tables are ~25 numbers, so the drift test — parse both expectation literals, assert equal
sets — is a dozen lines that runs in the required `build-test` lane, where *both* files are
checkout-visible even though neither shell lane can see the other; (c) it mirrors the Yoga-version
parity check already living in `ci.yml` for exactly this reason.

Keep `BnLayoutDemo.cs`'s header table as the human-readable canonical statement, and have the
drift test name it as the source.

> **The pattern landed early, in Gate 2** (review finding I1): `tests/BlazorNative.Renderer.Tests/ShellStyleTableDriftTests.cs`
> pins the shells' **style-NAME** tables the same way — it parses `YogaLayout.kt`'s `YOGA_STYLES`
> literal out of the checkout and asserts set-equality with `NativeRenderer.YogaStyleAttributes`
> plus disjointness from `VisualStyleAttributes`. (A name in .NET and missing from a shell is not
> a loud failure: the shell's router drops it into the visual branch, logs "not yet supported" and
> **silently discards the style**.) Gate 3 **extends that file** with `BnYogaLayout.mm`'s name
> table — a third mirror, pinned, not copied — and Gate 4 adds the frame-table arm to the same
> file rather than inventing a second mechanism.

- Cross-platform parity stated as a **test result**: the same frame table asserted on both shells
  (the 6.0 discipline — say *how* it is asserted, not that it is true).
- `docs/plans/2026-07-13-phase-6.1-conclusion.md`: what shipped, the frame-parity proof, honest
  boundaries (scroll/picker still un-flexed → 6.2; percent/auto grammar coverage; anything the
  measure func can't do yet), **plus the two boundaries the Gate 2 review surfaced and Gate 2
  half-closed:**
  - **`scroll` / `picker` children.** `BnYogaFrameLayout.onMeasure` now answers a non-EXACTLY
    spec with the last size Yoga applied (it used to answer 0, so a `view` inside a `scroll` —
    which is handed `UNSPECIFIED` — measured zero tall and its content vanished). That makes the
    common case work, but a framework `ViewGroup` still runs its own `onLayout` over its
    children: **Yoga does not get the final word INSIDE a scroll/picker.** 6.2 owns it, and owns
    asserting it.
  - **Overflow clipping.** `ViewGroup.clipChildren` defaults **true**, `UIView.clipsToBounds`
    defaults **false**, and Yoga's `overflow` default is `visible`. Gate 2 sets
    `clipChildren = false` on `BnYogaFrameLayout` so the two shells agree with Yoga's default.
    Not a frame-number divergence — a "renders identically" one — and it is the kind that no
    frame table can catch. Say so, and say it is now aligned.
- The ledger for 6.2/6.3 — **which must carry the style names 6.1
  deliberately did not accept: `alignContent`, `rowGap`, `columnGap` (and, if anything ever wants
  them, `display` / `flex`). They were dropped from the allow-list because no typed `BnView` param
  produces them, and an accepted name is a name TWO hand-written shell parsers must implement.
  Re-adding one means adding the typed param and the parser arm in the same phase.**

  **`BnStack` was consciously dropped** as a `BnColumn` synonym).
