**Executive summary (2026-07-12).** Host-originated events now reach .NET and
land on the screen, proven on all three surfaces. A **9th C-ABI export**
`blazornative_host_event(name, payload)` fires the *real*
`NativeShellBridge.NativeEvents` multicast (replacing the 3.2 no-op stub);
Android lifecycle (`onPause`/`onResume`/`onDestroy`) marshals into it; the
predictive-back gesture routes through it to a new
`NativeNavigationManager.NavigateBackAsync` (a previous-route slot); and a
launch-time deep link (`blazornative://<route>`) starts the app on the linked
page. The ABI grew **eight exports → nine** in one gate — deliberately before
the Swift shell exists (5.2/5.3), when the ripple is cheapest. DoD #5 is closed
**on the AVD**: lifecycle → events, predictive back → NavigateBack, deep link →
startup route, all live on `blazornative-pixel6-x86_64` (API 34).

Counts grew on all three surfaces and are asserted: **.NET 220** (was 203),
**JVM 78** (was 73), **Android instrumented 38** (was 35). Version →

## 1. The 9th export + real NativeEvents (the ABI decision)

`Exports.HostEvent(IntPtr nameUtf8, IntPtr payloadUtf8) → int`
(`[UnmanagedCallersOnly(EntryPoint = "blazornative_host_event", CallConvs = cdecl)]`,
top-level try/catch — BN0020/21 clean), splitting to a managed-callable
`DispatchHostEventCore(name, payload)` (the `DispatchEventCore` split — unit-
tested without the ABI crossing). `NativeShellBridge`'s `NativeEvents { add{} remove{} }`
no-op became a real static multicast + `RaiseNativeEvent` (GetInvocationList +
per-subscriber try/catch isolation, mirroring `DevHostBridge.RaiseNativeEvent`).

**rc contract** (mirrors `dispatch_event`): `0` delivered / `2` a subscriber (or
the re-render its `StateHasChanged` drove, when strict rethrows) faulted —
CONTAINED but surfaced / `3` malformed (NULL/empty name; NULL payload is legal).
`BN0014` survives unchanged — it now guards a *real* `NativeEvents` (the sync-
callback-window premise holds), verified by its tests staying green.

**The eight-→-nine ripple, all sites** (the ABI-change gate enumerated them):
`ci.yml` (×3 RID dumpbin/readelf `$expected` arrays + throw messages + the
header comment), `android-instrumented.yml`, `ios-spike.yml` +
`scripts/ios-spike-verify.sh` (the iOS symbol list — the export is platform-

conclusion docs stay as-written). Win-x64 + both bionic RIDs verified at exactly
**9** `blazornative_*` exports, **4 IL2072** each.

## 2. Back — the reserved "back" host event → NavigateBack, in .NET

The **back→NavigateBack mapping lives in .NET**, in `DispatchHostEventCore`,
keyed on a reserved event name (`Exports.BackEventName = "back"`), reachable via
the *existing* `blazornative_host_event` export — **no 10th export**. `"back"` is
**intercepted before the multicast** (it is a navigation command, not a lifecycle
notification — it never reaches `NativeEvents` subscribers) and routed to
`HostSession.CurrentNavigationManager.NavigateBackAsync`.

**Why .NET, not Kotlin:** one ingress means Android's predictive-back
(`OnBackInvokedCallback`, Gate 3) and the JVM test drive *identical* semantics; a
Kotlin-side mapping would fork Android from the headless/JVM path, and the only
alternative (a dedicated export) is exactly the ABI growth Gate 1 closed.

**rc extension for the back route:** `0` handled (navigated to the previous
route) / `1` **not handled** — at the origin (no previous route) or no session:
the shell falls through to default back (Android finishes). `rc 1` occurs ONLY on
the back route (the multicast path never returns it — a "nothing to act on"
parallel to `dispatch_event`'s rc 1). The Kotlin side reads this via a blocking
seam, `BlazorNativeRuntime.dispatchHostEventAndWait("back")` (the async
`dispatchHostEvent` is fire-and-forget for lifecycle; back needs the return
value).

### Previous-route slot semantics (+ the explicit-null-after-back deviation)

`NativeNavigationManager` gained a single `_previousRoute` slot: each successful
`NavigateToAsync` records the `from` (captured before the swap, inside
`afterSwap`, so a failed swap leaves the slot untouched — it tracks the SCREEN).
`NavigateBackAsync()` swaps to the slot and returns `true`; at the origin (null)
returns `false`.

**Deviation (documented):** a symmetric `NavigateToAsync` that always re-records
`from` would make back-then-back a 2-cycle (`/ ⇄ /settings` forever). The design's
stated behavior is "back → `/`, a further back → false (slot exhausted)". So
`NavigateBackAsync` **explicitly clears the slot AFTER** the nested
`NavigateToAsync` (which re-records it) — a back consumes the trail and leaves no
auto-re-back; a fresh *forward* navigation re-arms it. This yields exactly the
spec's observable behavior; the design's "NavigateToAsync itself updates the slot"
wording was imprecise about the cycle. A back-**stack** beyond one slot is
as-needed later work.

## 3. HostEventProbe — the consumer that proves it lands

`HostEventProbe` (internal sealed, registered `"HostEventProbe"` — the `FocusProbe`
scaffolding precedent): `[Inject] IMobileBridge`, subscribes `NativeEvents` in
`OnInitialized` with a **sync** handler (BN0014-clean), renders a nodeId-pinned
echo `BnText` (`"<name> (<count>)"`), unsubscribes in `Dispose`. It proves a host
event reaches a *mounted* component and re-renders it synchronously. **Ledgered as
scaffolding** in the M5 audit, alongside `CompositionProbe`/`FocusProbe`. The
BnDemo golden was verified UNCHANGED (a new registry entry does not touch it).

**Named-risk verdict (pinned FIRST, Gate 1):** `SwapRoot`'s `RunAfterDispatch`
drains correctly OFF the dispatch lane with no open batch — at
`_uiEventDispatchDepth == 0` it runs the action immediately (no queue). NavigateBack
(host-initiated, between clicks) rides this; the pin
`SwapRoot_OffDispatchLane_NoOpenBatch_DrainsImmediately` guards it.

## 4. Android wiring (Gate 3, on the AVD)

- **Lifecycle** (`MainActivity`): `onPause`/`onResume`/`onDestroy` →
  `runtime.dispatchHostEvent(...)` (async), each guarded by a `booted` flag so
  nothing enters the dispatch lane before the mount (the concurrent-entry
  hazard; the first pre-boot `onResume` is correctly skipped). **The onDestroy
  trap handled:** onDestroy fires "onDestroy" but does **NOT** `shutdown()` —
  Activity recreation re-runs `start()` against the process-global session;
  shutting down would kill the session the recreated Activity expects
  (documented at the site + the `BlazorNativeRuntime.start` KDoc).
- **Predictive back:** `OnBackInvokedCallback` (API 33+, `PRIORITY_DEFAULT`)
  delegating to a single `onBackPressed()` entry (the <API 33 fallback — plain
  `Activity` has no `onBackPressedDispatcher`, so the deprecated override is the
  honest fallback; the AVD is API 34, so the callback path is the tested one) →
  `handleBack()` → `dispatchHostEventAndWait("back")` → NavigateBack; `rc 0`
  consumes, `rc != 0` → finish. Manifest: `enableOnBackInvokedCallback="true"`.
- **Deep link:** a `VIEW` intent-filter (`blazornative://` custom scheme) +
  `onCreate` parses `intent.data` → seeds `AndroidShellBridge(initialRoute=...)`
  and resolves the mount component (see Deviation 1).

### Deviation 1 — deep-link component resolved shell-side

The design fed the deep-link route into the AndroidShellBridge slot and relied on
the 3.5 startup-route-honor to mount the page. That honor fires **only on a
session's FIRST mount** (`s_currentRootComponentId < 0`), which never holds under
**Activity recreation** OR the **shared-process instrumented session** (all
instrumented classes share one .NET session; the deep-link test is never the
first mount). So the shell resolves the deep-link's target component directly
(`DEEP_LINK_COMPONENTS`, still mounting **by name** per the design) while seeding
the route slot for `CurrentRoute` consistency — robust in all three cases. On a
genuine cold launch the two mechanisms are **mutually exclusive and agree**
(BnSettingsPage on screen either way). The .NET first-mount honor stays valid for
host-restored-route startup (covered by the `.NET StartupRoute` tests). The
route→component map now lives in *both* .NET and the shell — **unify at M6** (nav
lifts into a package).

### Deviation 2 — the instrumented back driver

Both an injected `KEYCODE_BACK` and `UiAutomation.GLOBAL_ACTION_BACK` start a
predictive-back *gesture* that CANCELS under instrumentation
(`sendCancelIfRunning: isInProgress=false` in logcat) without committing to
`onBackInvoked`. So the test drives the single `onBackPressed()` entry directly —
the **identical** production logic the registered `OnBackInvokedCallback`
delegates to (its registration is confirmed in logcat). Back-at-root → finish
(rc 1) is not asserted on-device (finish() is awkward through `ActivityScenario`);
it is covered by the JVM back-at-root rc-1 pin + the `.NET HostBackEvent_AtRoot`
test — the same .NET routing all three drive. (Micro-gap ledgered: the
`OnBackInvokedCallback` *registration* runs under API 34 but is not `assertNotNull`-
asserted — not worth a re-run.)

## Decisions captured

- **9th export `blazornative_host_event`**, platform-neutral, done before the
  Swift shell (cheapest ripple). The ABI is now nine exports.
- **back→NavigateBack maps in .NET** (reserved `"back"` name, rc 1 = not-handled),
  one ingress for both shells; the blocking `dispatchHostEventAndWait` seam reads
  the finish decision.
- **Previous-route back** = one slot, cleared after a back (no ping-pong); a
  forward re-arms it.
- **Deep link** = launch-time only, custom scheme, shell-side component
  resolution (Deviation 1).
- **onDestroy fires but never `shutdown`s** (the recreation trap).
