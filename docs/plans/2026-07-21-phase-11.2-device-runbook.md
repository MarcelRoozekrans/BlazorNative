# Phase 11.2 ‚Äî Android device runbook (M11 DoD #2)

**Open this with the phone in your hand.** Everything you need is here; nothing has to be
re-derived. Rationale lives in the
[design doc](2026-07-21-phase-11.2-design.md) ‚Äî you do not need it to execute this.

**App under test:** `samples/BlazorNative.SampleApp` + the `src/BlazorNative.Jni` Android shell.
**App id:** `io.blazornative.shell` ¬∑ **Activity:** `io.blazornative.shell/.MainActivity`
**Logcat tag:** `BlazorNative` ¬∑ **Deep-link scheme:** `blazornative://<route>`

**You run this. Results are RECORDED, not CI-asserted ‚Äî the phone is never a CI dependency.**
**iOS real-device is deferred** (no Apple Developer account).

Shell syntax below is PowerShell (the repo's primary shell). Bash equivalents are noted where
they differ.

## 0. Preconditions ‚Äî do these BEFORE you start

| # | Precondition | How |
|---|---|---|
| P1 | **Enrol a FINGERPRINT on the phone** | Settings ‚Üí Security ‚Üí Fingerprint. **Required**: the shell asks for `BIOMETRIC_STRONG` (Class 3). Face unlock on most phones is Class 2 and will **not** satisfy it. |
| P2 | **Set a screen lock (PIN/pattern/password)** | Prerequisite for the AndroidKeyStore auth-bound key. |
| P3 | **Location ON**, high accuracy | Settings ‚Üí Location. Plan to stand **outdoors or at a window** for Gate B3. |
| P4 | **Developer options + USB debugging ON** | Settings ‚Üí About phone ‚Üí tap *Build number* √ó7, then Developer options ‚Üí USB debugging. On Xiaomi/Oppo/Vivo also enable **"Install via USB"** / disable MIUI optimizations, or `installDebug` fails. |
| P5 | **`JAVA_HOME` ‚Üí a JDK 21** | a JDK, not a JRE; not an older JDK. Otherwise gradle fails. |
| P6 | **`ANDROID_NDK_ROOT` ‚Üí your NDK** | e.g. `%ANDROID_HOME%\ndk\26.3.11579264`. Otherwise the bionic cross-compile cannot run. |
| P7 | `adb` on PATH | it lives at `$env:ANDROID_HOME\platform-tools\adb.exe`. |

> which the OS may reclaim under pressure.

## Gate A ‚Äî build, install, boot on hardware

### A1. Publish both bionic RIDs (both are required)

```powershell
dotnet publish samples/BlazorNative.SampleApp -c Release -r linux-bionic-x64
dotnet publish samples/BlazorNative.SampleApp -c Release -r linux-bionic-arm64
```

**arm64 is the one the phone runs**, but gradle's `verifyNativeAssets` demands **both** and
fails fast otherwise (it prints the exact missing `dotnet publish` command). Expect **exactly 4
`IL2072` warnings** and nothing else ‚Äî anything more is a finding.

### A2. Build + install the APK

```powershell
cd src/BlazorNative.Jni
./gradlew.bat installDebug
cd ../..
```

Bash / macOS / Linux: `./gradlew installDebug`.

- The gradle root is **`src/BlazorNative.Jni/`** ‚Äî *not* `android/` (that is the template's
  layout).
- **Do NOT pass `-PciSoDir`.** It narrows the APK to `x86_64` and the phone would get no arm64
  `.so`. There is no `-PappPubRoot` in this build (template-only); the publish path is hardcoded.
- Alternative if you want the APK file first:
  `./gradlew.bat assembleDebug` ‚Üí `build/outputs/apk/debug/*.apk`, then
  `adb install -r <apk>`.

| Install error | Fix |
|---|---|
| `INSTALL_FAILED_USER_RESTRICTED` | Enable **"Install via USB"** in Developer options (OEM shells). |
| `INSTALL_FAILED_UPDATE_INCOMPATIBLE` / signature mismatch | `adb uninstall io.blazornative.shell` then re-run. |
| `INSTALL_FAILED_NO_MATCHING_ABIS` | The arm64 publish did not happen, or `-PciSoDir` was set. Redo A1. |
| `Native build outputs missing:` from gradle | The message names the exact publish to run. Redo A1. |

adb shell am start -n io.blazornative.shell/.MainActivity

## Gate B ‚Äî the four capabilities

### B1. Camera ‚Äî real sensor + EXIF

```powershell
adb shell am start -a android.intent.action.VIEW -d "blazornative://camera"
```

**Steps on the phone:**
1. Tap **Check** ‚Üí the echo text should read `status:Captured` (that status doubles as
   "a camera app is present"). Any other value ‚Üí record and continue.
2. Tap **Take Photo** ‚Üí the **system camera app** opens. *(No CAMERA permission dialog from our
   app is expected ‚Äî the system camera owns the sensor. The system camera app may ask for its
   own permissions on first use.)*
3. **Point at something recognisable and real** (your desk, out of the window) and take the
   shot, then confirm/accept it.
4. Back in the app: the echo reads `captured:<W>x<H>:<bytes>` and the 240√ó320 image box shows
   **your photo**.

**PASS looks like:** echo `captured:4080x3060:2841923` (your numbers will differ; the point is
non-zero dimensions matching the phone's camera resolution and a byte count in the MB range),
and a recognisable photo on screen ‚Äî **not** a synthetic green/checkerboard frame.

**Evidence ‚Äî capture the screen, then PULL THE FILE AND READ ITS EXIF:**

```powershell
adb exec-out screencap -p > "$EV/B1-camera-screen.png"

# The capture lives in the app's PRIVATE cache dir ‚Äî list it via run-as (debug APK):
adb shell run-as io.blazornative.shell ls -l /data/data/io.blazornative.shell/cache/blazornative_captures

# Pull it out (run-as + cat is the reliable route; plain `adb pull` cannot read the private dir):
$NAME = "capture_<paste the filename from the ls above>.jpg"
adb shell run-as io.blazornative.shell cat "/data/data/io.blazornative.shell/cache/blazornative_captures/$NAME" > "$EV/B1-capture.jpg"
```

> PowerShell note: `>` uses UTF-16 by default for text but binary passthrough from `adb
> exec-out`/`run-as cat` works because the bytes are piped raw. If the JPEG comes out corrupt,
> use `cmd /c "adb shell run-as ... cat ... > file.jpg"` or
> `adb exec-out run-as io.blazornative.shell cat <path> > file.jpg`.

**Read the EXIF on the DESKTOP** (there is no `exiftool` on the phone ‚Äî do not try):

```powershell
# Option 1 ‚Äî Windows, no install:
(Get-Item "$EV/B1-capture.jpg").VersionInfo | Out-Null
$img = [System.Drawing.Image]::FromFile((Resolve-Path "$EV/B1-capture.jpg"))
$img.PropertyIdList   # non-empty = EXIF present
# Option 2 ‚Äî if you have it: exiftool "$EV/B1-capture.jpg"
# Option 3 ‚Äî right-click the file ‚Üí Properties ‚Üí Details tab
```

**What proves a real sensor:** EXIF carries `Make` / `Model` (your phone), `DateTimeOriginal`
(now), and exposure fields (`FNumber`, `ISOSpeedRatings`, `ExposureTime`, focal length). An
emulated AVD shutter produces a synthetic scene with thin or absent camera EXIF. **Record the
`Make`, `Model` and `DateTimeOriginal` values verbatim in the proof table** ‚Äî those three are
the DoD's "real sensor + EXIF" evidence.

**Do this immediately after capture** ‚Äî the cache dir is reclaimable by the OS.

### B2. Biometrics + the auth-bound keystore

```powershell
adb shell am start -a android.intent.action.VIEW -d "blazornative://secure"
```

**Steps on the phone (in this order):**
1. Tap **Authenticate** ‚Üí the OS biometric prompt appears with *"Prove it's you"*. Present your
   **enrolled finger**. ‚Üí echo `status:Authenticated`.
2. Tap **Authenticate** again and **Cancel** the prompt ‚Üí echo `status:Cancelled`. *(Proves
   denial arrives as data, never as a hang.)*
3. Tap **Set** ‚Üí echo `status:Ok`. This creates the auth-bound key (`requireAuth: true`).
4. Tap **Unlock** ‚Üí the prompt appears **again**; present the finger ‚Üí echo `value:hunter2`.
5. Tap **Unlock** once more and **Cancel** ‚Üí echo `status:AuthFailed`. *(The OS refused to
   decrypt without fresh auth ‚Äî this is the real proof of OS key binding.)*
6. Tap **Delete** ‚Üí echo `status:Ok`.

**PASS looks like:** the exact echo sequence
`Authenticated` ‚Üí `Cancelled` ‚Üí `Ok` ‚Üí `value:hunter2` ‚Üí `AuthFailed` ‚Üí `Ok`.

**If you get `status:Unavailable`:** no Class-3 biometric is enrolled/usable. If the phone only
has face unlock, that is almost certainly Class 2 (WEAK) ‚Äî the shell requires
`BIOMETRIC_STRONG`. Record as a **device limitation NOTE**, not a framework FAIL, and say so.

**The TEE claim ‚Äî record it precisely, do not over-claim.** The app does **not** expose
`KeyInfo.isInsideSecureHardware()` / `getSecurityLevel()`, so you cannot observe the key's
security level from the UI. Record these three things instead:

```powershell
# (a) what the DEVICE reports
adb shell pm list features | Select-String -Pattern "keystore|strongbox|biometric|fingerprint"
adb shell getprop ro.boot.keymaster
adb shell getprop ro.hardware.keystore
adb shell getprop ro.hardware.keystore_desede

# (b) the NEGATIVE CONTROL ‚Äî same commands on the AVD, for the contrast
#     (boot the emulator, then re-run the block above against it)
adb -s emulator-5554 shell pm list features | Select-String -Pattern "keystore|strongbox"
```

Write the claim in the proof table **exactly** as:
> *"Device reports a hardware-backed keystore (`android.hardware.strongbox_keystore`:
> present / absent). The app does not surface `KeyInfo.getSecurityLevel()`, so 'TEE-backed' is a
> device report, not an app observation. What the app DID prove end-to-end: the OS refused to
> decrypt the auth-bound key without a fresh Class-3 (BIOMETRIC_STRONG) authentication."*

**(c)** the observed `Set` ‚Üí `Unlock` ‚Üí `AuthFailed-on-cancel` sequence from step 3‚Äì5 **is** the
end-to-end proof; note it alongside.

**Evidence:**
```powershell
adb exec-out screencap -p > "$EV/B2-secure-unlocked.png"
adb shell pm list features > "$EV/B2-device-features.txt"
adb logcat -d -s BlazorNative > "$EV/B2-secure.log"
```

### B3. Geolocation ‚Äî a real fix

**Go outdoors or to a window first.** Then:

```powershell
adb shell am start -a android.intent.action.VIEW -d "blazornative://geolocation"
```

**Steps on the phone:**
1. Tap **Check** ‚Üí echo `status:<Granted|Denied|‚Ä¶>` *without* prompting (read-only path).
2. Tap **Locate** ‚Üí the **location permission dialog** appears on first use (FINE/COARSE). Tap
   **While using the app**.
3. Wait for the fix. ‚Üí echo `fix:<lat>,<lng>`.
4. Optionally tap **Locate** again ‚Äî a second, faster fix (the provider is warm).

**PASS looks like:** `fix:52.3702,4.8952` ‚Äî a real decimal pair that **matches where you are

**Accuracy is NOT observable.** `GeolocationPosition` carries an `Accuracy` field but the demo's
echo prints **only** latitude and longitude. Record accuracy as **"not surfaced by the app"** ‚Äî

Move outside, wait 60‚Äì90 s, retry. If it still fails, record it as a **NOTE with the environment
described** ‚Äî never as a silent pass. `status:Denied` / `status:DeniedPermanently` means you
declined the dialog; clear with
`adb shell pm reset-permissions io.blazornative.shell` and retry.

**Evidence:**

## Logging ‚Äî what to run, and what silence means

**The only filter worth running:**

```powershell
adb logcat -d -s BlazorNative
```

That tag carries the boot marker, the deep-link lines, the routes-resource warnings and every
caught throwable.

**IMPORTANT ‚Äî a managed-side failure may leave NO trace.** The .NET runtime's `Console.Error`
goes to process stderr, which **Android discards**. Only the shell's explicit `Log.e`/`Log.i`
sink reaches logcat. So:

> **If something misbehaves and `adb logcat -d -s BlazorNative` is EMPTY, that silence is
> itself a finding.** Record the exact command and its empty output in the proof doc. It is
> field evidence for issue #155 (logging discipline, DoD #6 / Phase 11.4) and #164.

**Fallback when the tag is silent but something is clearly wrong** ‚Äî this catches OS-side
complaints (permission denials, FileProvider failures, native crashes), which *do* reach logcat:

```powershell
$PID = (adb shell pidof io.blazornative.shell).Trim()
adb logcat -d --pid $PID > "$EV/Zz-fullpid.log"
# or, for a crash:
adb logcat -d -b crash > "$EV/Zz-crash.log"
```

**Save every excerpt you quote** into `$EV/` and name the file in the proof table.
