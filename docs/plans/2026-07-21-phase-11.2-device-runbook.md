> ### ⚠ `[amended 2026-07-22]` P11 — ADB INPUT INJECTION MAY BE BLOCKED. Plan for a manual session.
>
> On the 2026-07-22 session (**MIUI / Xiaomi**), `adb shell input tap` and `adb shell input
> keyevent` were **silently inert** — they returned success and produced **no `MotionEvent` at
> all** in the app's process. The device could not even be woken with `keyevent 224` / `26`.
> Nothing in the output tells you this is happening; you discover it because the app does not
> react.
>
> **What still works:** `am start`, `am kill`, `dumpsys`, `screencap`, `run-as`, `logcat`,
> `pm list features`, `getprop`. That is enough for every *navigation* step in this runbook —
> which is why the phase completed — but **not** for any *tap*.
>
> **The fix, if you can get it:** MIUI gates injection behind **Developer options → *USB
> debugging (Security settings)***, which requires signing in to a **Mi account** and typically a
> waiting period. Other OEM skins have equivalents.
>
> **If you cannot get it — and that is the normal outcome — accept it and say so in the proof.**
> Every button press below is then performed by **your finger on the screen**, in person, while
> `adb` is used only to navigate (`am start`) and to observe (`logcat`, `dumpsys`, `screencap`).
> A human finger on a real screen is *better* evidence than a synthetic `MotionEvent`; the cost is
> that **nothing can be scripted or replayed**, so budget time and expect the optional
> interaction-smoke steps (Gate C) to be the first thing you drop. The 2026-07-22 session did
> exactly that — Gate C came out **PARTIAL**, deliberately and on the record.

> **`vswhere` is NOT needed for this phase.** That prerequisite applies to the `win-x64`
> NativeAOT link step only; 11.2 never publishes `win-x64`.

> **Free up phone storage** before the camera step. Captures land in the app **cache** dir,
> which the OS may reclaim under pressure.

## Gate A — build, install, boot on hardware

### A1. Publish both bionic RIDs (both are required)

```powershell
dotnet publish samples/BlazorNative.SampleApp -c Release -r linux-bionic-x64
dotnet publish samples/BlazorNative.SampleApp -c Release -r linux-bionic-arm64
```

**arm64 is the one the phone runs**, but gradle's `verifyNativeAssets` demands **both** and
fails fast otherwise (it prints the exact missing `dotnet publish` command). Expect **exactly 4
`IL2072` warnings** and nothing else — anything more is a finding.

### A2. Build + install the APK

```powershell
cd src/BlazorNative.Jni
./gradlew.bat installDebug
cd ../..
```

Bash / macOS / Linux: `./gradlew installDebug`.

- The gradle root is **`src/BlazorNative.Jni/`** — *not* `android/` (that is the template's
  layout).
- **Do NOT pass `-PciSoDir`.** It narrows the APK to `x86_64` and the phone would get no arm64
  `.so`. There is no `-PappPubRoot` in this build (template-only); the publish path is hardcoded.
- Alternative if you want the APK file first:
  `./gradlew.bat assembleDebug` → `build/outputs/apk/debug/*.apk`, then
  `adb install -r <apk>`.

> ### ⚠ `[amended 2026-07-22]` `installDebug` FAILED on MIUI — use `assembleDebug` + `adb install -r`
>
> On the real session `./gradlew.bat installDebug` failed with:
>
> ```
> INSTALL_FAILED_USER_RESTRICTED: Install canceled by user
> ```
>
> — despite USB debugging being on and the RSA dialog accepted. It is MIUI refusing the
> *gradle-driven* install specifically. **The working sequence:**
>
> 1. Enable **Developer options → *Install via USB*** (a distinct toggle from USB debugging; it
>    may require a Mi account and a SIM).
> 2. ```powershell
>    cd src/BlazorNative.Jni
>    ./gradlew.bat assembleDebug
>    cd ../..
>    adb install -r src/BlazorNative.Jni/build/outputs/apk/debug/<apk>
>    ```
>    → `Success`.
>
> **Do not lose time debugging gradle for this** — the APK it built was fine. Confirm the payload
> before you install: the 2026-07-22 APK carried
> `lib/arm64-v8a/libBlazorNative.Runtime.so` (5.4 MB) **and** `lib/x86_64/…` (5.2 MB), which is
> what "dual-ABI, `-PciSoDir` not passed" looks like.

| Install error | Fix |
|---|---|
| `INSTALL_FAILED_USER_RESTRICTED` | Enable **"Install via USB"** in Developer options (OEM shells) — **and expect `installDebug` to fail anyway; fall back to `assembleDebug` + `adb install -r`, see the box above `[amended 2026-07-22]`**. |
| `INSTALL_FAILED_UPDATE_INCOMPATIBLE` / signature mismatch | `adb uninstall io.blazornative.shell` then re-run. |
| `INSTALL_FAILED_NO_MATCHING_ABIS` | The arm64 publish did not happen, or `-PciSoDir` was set. Redo A1. |
| `Native build outputs missing:` from gradle | The message names the exact publish to run. Redo A1. |

## Gate B — the four capabilities

### B1. Camera — real sensor + EXIF

> ### ⚠ `[amended 2026-07-22]` THERE IS NO EXIF, AND THAT IS CORRECT — do not hunt for it
>
> Everything in this section about reading `Make` / `Model` / `DateTimeOriginal` **cannot
> succeed**, and the reason is a deliberate framework behaviour: the shell **normalises
> orientation** by baking the EXIF rotation into the pixels *and resetting the tag*, then
> re-encodes the JPEG — so that Coil/Kingfisher, which honour EXIF on decode, do not rotate a
> second time (`AndroidShellBridge.kt:1152-1156`, `:1300-1307`). A re-encoded JPEG carries **no
> camera EXIF**. The 2026-07-22 capture reported `EXIF present: False`, `JFIF present: True`.
>
> **DoD #2's "real sensor + EXIF" clause was therefore amended** — see
> [MILESTONE.md](../planning/MILESTONE.md) and
> [the device proof](2026-07-22-phase-11.2-device-proof.md). **What to record instead:**
>
> - the **dimensions** from the echo (2026-07-22: `3072x4096` — a genuine 12.6 MP sensor
>   resolution, which no AVD shutter produces),
> - that the scene is **real and recognisable**, not synthetic,
> - that it arrived via **`ACTION_IMAGE_CAPTURE`** — the system camera app owning the sensor.
>
> **Still pull the file and dump its structure** — that is how
> **[#178](https://github.com/MarcelRoozekrans/BlazorNative/issues/178)** was found: the DQT
> quantisation tables came back **all 255** (quality 1) at a 4096 px long edge, because
> `CapturePhotoAsync()`'s `options = default` zero-initialises `CaptureOptions` and bypasses its
> documented `MaxDimension = 2048, Quality = 85` defaults. Record the marker list, the DQT
> tables and the `SOF0` dimensions; ignore the EXIF-reading recipes below.

```powershell
adb shell am start -a android.intent.action.VIEW -d "blazornative://camera"
```

**Steps on the phone:**
1. Tap **Check** → the echo text should read `status:Captured` (that status doubles as
   "a camera app is present"). Any other value → record and continue.
2. Tap **Take Photo** → the **system camera app** opens. *(No CAMERA permission dialog from our
   app is expected — the system camera owns the sensor. The system camera app may ask for its
   own permissions on first use.)*
3. **Point at something recognisable and real** (your desk, out of the window) and take the
   shot, then confirm/accept it.
4. Back in the app: the echo reads `captured:<W>x<H>:<bytes>` and the 240×320 image box shows
   **your photo**.

**PASS looks like:** echo `captured:4080x3060:2841923` (your numbers will differ; the point is
non-zero dimensions matching the phone's camera resolution and a byte count in the MB range),
and a recognisable photo on screen — **not** a synthetic green/checkerboard frame.

**Evidence — capture the screen, then PULL THE FILE AND READ ITS EXIF:**

```powershell
adb exec-out screencap -p > "$EV/B1-camera-screen.png"

# The capture lives in the app's PRIVATE cache dir — list it via run-as (debug APK):
adb shell run-as io.blazornative.shell ls -l /data/data/io.blazornative.shell/cache/blazornative_captures

# Pull it out (run-as + cat is the reliable route; plain `adb pull` cannot read the private dir):
$NAME = "capture_<paste the filename from the ls above>.jpg"
adb shell run-as io.blazornative.shell cat "/data/data/io.blazornative.shell/cache/blazornative_captures/$NAME" > "$EV/B1-capture.jpg"
```

> PowerShell note: `>` uses UTF-16 by default for text but binary passthrough from `adb
> exec-out`/`run-as cat` works because the bytes are piped raw. If the JPEG comes out corrupt,
> use `cmd /c "adb shell run-as ... cat ... > file.jpg"` or
> `adb exec-out run-as io.blazornative.shell cat <path> > file.jpg`.

**Read the EXIF on the DESKTOP** (there is no `exiftool` on the phone — do not try):

```powershell
# Option 1 — Windows, no install:
(Get-Item "$EV/B1-capture.jpg").VersionInfo | Out-Null
$img = [System.Drawing.Image]::FromFile((Resolve-Path "$EV/B1-capture.jpg"))
$img.PropertyIdList   # non-empty = EXIF present
# Option 2 — if you have it: exiftool "$EV/B1-capture.jpg"
# Option 3 — right-click the file → Properties → Details tab
```

**What proves a real sensor:** EXIF carries `Make` / `Model` (your phone), `DateTimeOriginal`
(now), and exposure fields (`FNumber`, `ISOSpeedRatings`, `ExposureTime`, focal length). An
emulated AVD shutter produces a synthetic scene with thin or absent camera EXIF. **Record the
`Make`, `Model` and `DateTimeOriginal` values verbatim in the proof table** — those three are
the DoD's "real sensor + EXIF" evidence.

**Do this immediately after capture** — the cache dir is reclaimable by the OS.

### B2. Biometrics + the auth-bound keystore

```powershell
adb shell am start -a android.intent.action.VIEW -d "blazornative://secure"
```

**Steps on the phone (in this order):**
1. Tap **Authenticate** → the OS biometric prompt appears with *"Prove it's you"*. Present your
   **enrolled finger**. → echo `status:Authenticated`.
2. Tap **Authenticate** again and **Cancel** the prompt → echo `status:Cancelled`. *(Proves
   denial arrives as data, never as a hang.)*
3. Tap **Set** → echo `status:Ok`. This creates the auth-bound key (`requireAuth: true`).
4. Tap **Unlock** → the prompt appears **again**; present the finger → echo `value:hunter2`.
5. Tap **Unlock** once more and **Cancel** → echo `status:AuthFailed`. *(The OS refused to
   decrypt without fresh auth — this is the real proof of OS key binding.)*
6. Tap **Delete** → echo `status:Ok`.

**PASS looks like:** the exact echo sequence
`Authenticated` → `Cancelled` → `Ok` → `value:hunter2` → `AuthFailed` → `Ok`.

**If you get `status:Unavailable`:** no Class-3 biometric is enrolled/usable. If the phone only
has face unlock, that is almost certainly Class 2 (WEAK) — the shell requires
`BIOMETRIC_STRONG`. Record as a **device limitation NOTE**, not a framework FAIL, and say so.

**The TEE claim — record it precisely, do not over-claim.** The app does **not** expose
`KeyInfo.isInsideSecureHardware()` / `getSecurityLevel()`, so you cannot observe the key's
security level from the UI. Record these three things instead:

```powershell
# (a) what the DEVICE reports
adb shell pm list features | Select-String -Pattern "keystore|strongbox|biometric|fingerprint"
adb shell getprop ro.boot.keymaster
adb shell getprop ro.hardware.keystore
adb shell getprop ro.hardware.keystore_desede

# (b) the NEGATIVE CONTROL — same commands on the AVD, for the contrast
#     (boot the emulator, then re-run the block above against it)
adb -s emulator-5554 shell pm list features | Select-String -Pattern "keystore|strongbox"
```

Write the claim in the proof table **exactly** as:
> *"Device reports a hardware-backed keystore (`android.hardware.strongbox_keystore`:
> present / absent). The app does not surface `KeyInfo.getSecurityLevel()`, so 'TEE-backed' is a
> device report, not an app observation. What the app DID prove end-to-end: the OS refused to
> decrypt the auth-bound key without a fresh Class-3 (BIOMETRIC_STRONG) authentication."*

**(c)** the observed `Set` → `Unlock` → `AuthFailed-on-cancel` sequence from step 3–5 **is** the
end-to-end proof; note it alongside.

**Evidence:**

#### B4c — COLD tap-through (process killed)

> ### ⚠ `[amended 2026-07-22]` `am force-stop` DESTROYS the notification you are about to tap
>
> The original sequence below **cannot work**: `am force-stop` cancels the package's posted
> notifications along with the process, so the shade is empty when you go to tap. **Use `am kill`
> after backgrounding instead** — it reclaims the process without cancelling what it posted.
>
> **The sequence that worked on hardware:**
>
> ```powershell
> # 1. On the phone: open /notifications and tap Show (post a fresh one).
> # 2. On the phone: press HOME — background the app, do NOT force-stop it.
> adb shell am kill io.blazornative.shell        # kills a BACKGROUNDED process only
> adb shell pidof io.blazornative.shell          # must print NOTHING — that is the proof it died
> adb shell dumpsys notification --noredact | Select-String "records:"   # the notification SURVIVED
> adb logcat -c
> # 3. On the phone: pull down the shade and tap the notification.
> adb shell pidof io.blazornative.shell          # a NEW pid — that is the proof it cold-started
> ```
>
> **Record both pids.** "Old pid gone, new pid after the tap" is what makes this a *cold* start
> rather than a resume, and it is the single most load-bearing observation in Gate B4. On
> 2026-07-22 the new pid was **16535** and the boot log showed
> `[BOOT] mounted BnNotificationsDemo` + `arrived:/notifications`.
>
> `am kill` is a no-op on a **foreground** process — background it first or nothing happens and
> you will think the command is broken.

## Logging — what to run, and what silence means

**The only filter worth running:**

```powershell
adb logcat -d -s BlazorNative
```

That tag carries the boot marker, the deep-link lines, the routes-resource warnings and every
caught throwable.

**IMPORTANT — a managed-side failure may leave NO trace.** The .NET runtime's `Console.Error`
goes to process stderr, which **Android discards**. Only the shell's explicit `Log.e`/`Log.i`
sink reaches logcat. So:

> **If something misbehaves and `adb logcat -d -s BlazorNative` is EMPTY, that silence is
> itself a finding.** Record the exact command and its empty output in the proof doc. It is
> field evidence for issue #155 (logging discipline, DoD #6 / Phase 11.4) and #164.

**Fallback when the tag is silent but something is clearly wrong** — this catches OS-side
complaints (permission denials, FileProvider failures, native crashes), which *do* reach logcat:

```powershell
$PID = (adb shell pidof io.blazornative.shell).Trim()
adb logcat -d --pid $PID > "$EV/Zz-fullpid.log"
# or, for a crash:
adb logcat -d -b crash > "$EV/Zz-crash.log"
```

**Save every excerpt you quote** into `$EV/` and name the file in the proof table.
