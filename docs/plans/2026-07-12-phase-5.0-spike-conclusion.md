- **Spike status:** **GREEN**

- **Milestone:** M5 â€” P4: Full Platform Coverage (closes DoD #1; **GREEN unlocks

**Executive summary (2026-07-12).** .NET 10 NativeAOT **can** produce a linkable
iOS-**simulator** artifact carrying the eight-export `blazornative_*` C-ABI, on a
free public-repo `macos-latest` runner. The named risk is **retired GREEN**. The
winning recipe is the **runtime-pack bypass** â€” the *same* mechanism Phase 3.0c
used for `linux-bionic` â€” because the .NET 10 SDK gates `PublishAot` on a
hardcoded AOT-RID allow-list (`error NETSDK1203`) even though the runtime packs
exist on nuget.org. Two facts made the bypass clean where bionic needed a
vendored NDK linker shim: (1) the RID-specific ILCompiler
`runtime.iossimulator-arm64.microsoft.dotnet.ilcompiler` **does** exist at
`10.0.9` (bionic's did not â€” that was 3.0b's Gate 4 blocker), and (2) the Apple
toolchain (`clang`/`ld` via `xcrun`) is found **natively** on the macOS runner,
so no shim/PATH hackery is required. Both simulator (`iossimulator-arm64`, the
load-bearing target) and device (`ios-arm64`, secondary audit probe) publish,
export all 8 symbols, and link-probe green.

The consequence for M5: the pre-decided **fallback ladder is NOT triggered**
(no Mono-AOT, no milestone re-scope). Phases 5.2/5.3 build the real Swift-shell
lane on top of this artifact.

### Rung-3 iteration ledger (2 distinct reds, both diagnosed, of â‰¤6 budget)

1. **`NETSDK1207` on the netstandard2.0 analyzer** (first attempt, bypass passed
   via `-p:` on the CLI). Root cause: CLI `-p:` properties become **global**
   MSBuild properties inherited by every P2P reference; the bypass therefore
   propagated onto `BlazorNative.Analyzers` (netstandard2.0), where
   `PublishAot`/AOT-for-TFM has no meaning â†’
   `error NETSDK1207: Ahead-of-time compilation is not supported for the target
   framework`. Note the bypass **did** clear `NETSDK1203` on the real Runtime â€”
   proving the mechanism. Fix (commit `aea32cf`): move the bypass into an
   **RID-conditional `PropertyGroup` inside `BlazorNative.Runtime.csproj`** (the
   3.0c bionic pattern) so it applies only to Runtime and never propagates; also
   `UndefineProperties="RuntimeIdentifier;PublishAot;PublishAotUsingRuntimePack"`
   on the analyzer `ProjectReference` so the analyzer is shielded regardless of
   how any publish is invoked.
2. **dSYM companions counted as link-probe FAILs** (cosmetic, run 2). The
   verifier's `*.dylib` glob matched the DWARF companion inside `*.dSYM/â€¦`, which
   `ld` rejects (`unsupported mach-o filetype`). The **real** `native/` and
   `publish/` dylibs both PASSED, so the verdict was already GREEN; fix
   (`72968d4`) filters `*.dSYM/*` and adds `-Wl,-rpath` for the bonus run.

The pre-run hypothesis "NativeAOT is unsupported on the iOS **simulator**
(device-only)" was **disproven** â€” the nuget existence probe (rung 3 step) found
`Microsoft.NETCore.App.Runtime.NativeAOT.iossimulator-arm64` through 10.0.9 and a
dedicated RID ILCompiler at 10.0.9; only the SDK's front-door gate blocked it.

### csproj changes (behavior-neutral for existing RIDs)

1. An **ios-conditional `PropertyGroup`**
   (`Condition="$(RuntimeIdentifier.StartsWith('iossimulator')) Or
   $(RuntimeIdentifier.StartsWith('ios-'))"`) setting
   `PublishAotUsingRuntimePack=true` + `DisableUnsupportedError=true` +
   `RuntimeFrameworkVersion=10.0.9` â€” a direct mirror of the existing
   `linux-bionic` PropertyGroup. The RID condition never matches win-x64 or
   linux-bionic.
2. `UndefineProperties="RuntimeIdentifier;PublishAot;PublishAotUsingRuntimePack"`
   on the `BlazorNative.Analyzers` `ProjectReference` â€” shields the RID-agnostic
   netstandard2.0 analyzer (`ReferenceOutputAssembly=false`) from any AOT/RID
   global property. `UndefineProperties` only strips **global** properties; the
   CI publishes never pass these globally, so it is a no-op for them.

**No-regression proof (local, Windows host):** win-x64 NativeAOT publish still
emits **exactly the 4 accepted IL2072s** and links `BlazorNative.Runtime.dll`
clean (VS Installer on PATH). `RuntimeIdentifiers` was deliberately **not**
extended (adding `iossimulator-arm64` there would break `dotnet restore` for
every RID, which tries to restore ios packs on a plain restore). The PR's CI run
(`build-test`: 203/0 tests, 4Ă—IL2072, 8-export set-equality on all three existing
RIDs) is the authoritative regression gate.

## Implications for Phase 5.2 / 5.3

1. **The recipe is proven and thin.** All iOS-specific MSBuild is the one
   RID-gated PropertyGroup in the Runtime csproj (+ the analyzer shield). No
   vendored targets, no linker shim â€” the macOS runner's Xcode toolchain does the
   linking. Swapping to a first-class dotnet/iOS NativeAOT path (if .NET 11 lifts
   the `NETSDK1203` gate natively) is a contained change.
2. **Shape decision deferred to 5.2:** the spike ships a `.dylib` (shared). A
   production iOS app embed typically wants a **static `.a`** (`NativeLib=Static`)
   linked into the app binary, or an `.xcframework`. That is a packaging choice
   for the real lane, orthogonal to feasibility.
3. **Runtime proof is 5.2's:** symbols + link are proven; actually *booting* the
   runtime on a simulator (init â†’ version string â†’ mount) is the 5.2 bar (the
   bonus `simctl` rung is scaffolded in `verify.sh`).
4. **CI lane (5.2):** the spike workflow is dispatch-only and single-job; the real
   lane pins Xcode explicitly (26.5 today, but `macos-latest` drifts â€” record and
   assert), asserts the 4 IL2072s + 8 exports like the bionic legs, and produces
   the app-embed artifact.
5. **Fallback ladder retired** for this milestone â€” Mono-AOT and re-scope are not
   needed; NativeAOT is the single runtime on iOS as on every other platform
   (preserving the 3.0c "single NativeAOT runtime everywhere" decision).

## Decisions captured

- **iOS = runtime-pack bypass, pinned to 10.0.9**, exactly like linux-bionic; the
  pin lives in the Runtime csproj so a servicing release can't shift the toolchain.
- **`RuntimeIdentifiers` stays `win-x64;linux-bionic-arm64;linux-bionic-x64`** â€”
  ios RIDs are publish-time `-r` targets only (adding them breaks plain restore).
- **The `spikes/ios-aot-probe` wrapper is not the path** â€” kept only as documented
  evidence; 5.2 works directly off `src/BlazorNative.Runtime` with the bypass.
