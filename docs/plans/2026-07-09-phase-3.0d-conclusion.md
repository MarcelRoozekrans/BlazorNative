# Phase 3.0d Conclusion — Native Wire Protocol + Renderer Cutover

- **Status:** **GREEN — all 4 gates**
- **Branch:** `phase-3.0d-wire-protocol`
- **Design:** [2026-07-09-phase-3.0d-design.md](2026-07-09-phase-3.0d-design.md)
- **Implementation plan:** [2026-07-09-phase-3.0d-implementation-plan.md](2026-07-09-phase-3.0d-implementation-plan.md)

**Executive summary (2026-07-09).** The typed-struct wire protocol replaces
JSON-over-stdout as the live frame transport: `HelloComponent` renders on the
AVD through the NativeAOT `.so` via init → register frame callback → mount →
`FrameEncoder` → cdecl callback → Kotlin offset-read adapter → unchanged
`WidgetMapper`. The golden fixture proves the struct path and the JSON path
produce the same patch stream (zero diff after canonical node-ID/timestamp
normalization); the Android demo boot is ~22× faster
than the historical wasm cold boot. The `[FRAME]` stdout transport survives
only as the WasiHost fallback until Phase 3.0e deletes the WASM era.

---

## Per-gate summary

### Gate 1 — Protocol types + FrameArena + FrameEncoder (.NET, TDD)

- `src/BlazorNative.NativeHost/PatchProtocolNative.cs`: the typed **9-kind
  ABI** — `BlazorNativePatchKind` (all 9 `RenderPatch` types have wire ids),
  `BlazorNativePatch` (48 bytes) + `BlazorNativeFrame` (24 bytes), layouts
  drift-tested on both sides (`PatchProtocolNativeTests.cs` /
  `NativeFrameAdapterTest.kt`).
- `FrameArena` — ArrayPool-backed native scratch for frame marshaling
  (patch array + UTF-8 cstrings), valid only for the duration of the
  callback.
- `FrameEncoder` — `RenderFrame` → `BlazorNativePatch[]`, 9-kind round-trip
  covered. **16 TDD tests at gate close** (21 after the review follow-ups
  added offset-pinning, growth-path, and alloc-bound cases). 6 kinds wired
  live today;
  AttachEvent/DetachEvent/AppendChild are ABI-reserved (M3 DoD #2/#10) but
  dormant until 3.2/M3.

### Gate 2 — Desktop end-to-end + golden fixture

- `NativeRenderer.FrameSink` — pluggable frame transport. NativeHost installs
  the struct marshaller; WasiHost leaves it null → the `[FRAME]` stdout
  fallback (deletes with the WASM era in 3.0e).
- Exports: `blazornative_register_frame_callback` (last-wins re-register),
  `blazornative_mount` (status 0/1/2/3), dormant `blazornative_dispatch_event`
  (always −1 — 3.2 wires it).
- Kotlin `NativeFrameAdapter` reads structs by pointer offsets and
  materializes the existing sealed `RenderPatch` classes.
- **Golden-fixture proof: struct == JSON.** Mounting Hello through the
  win-x64 NativeAOT .dll yields **13 patches with zero diff** against the
  JSON-path fixture (compared after canonical node-ID renumbering +
  CommitFrame frameId/timestamp zeroing — the runtime's node-ID counter is
  process-global) — both transports locked to the same wire shape from both
  ends.

### Gate 3 — Android cutover

- `MainActivity` boots the NativeAOT pipeline via the new
  `BlazorNativeRuntime` lifecycle wrapper (init → register → mount); the
  wasmtime/WasiHost boot path remains only inside the pre-existing wasmtime
  tests until 3.0e.
- **`WidgetMapperTest` sees widgets in ~1.6 s vs ~36 s historical wasm
  cold-boot — ≈22× faster demo boot.**
- Full instrumented suite **22/22**; **zero cutover iterations, zero
  wasmtime flakes** during the Gate 3 runs.

## Carryover into Phase 3.0e / 3.2

### (a) Phase 3.0e deletion-list additions

1. `NativeRenderer`'s `[FRAME]` stdout fallback + `FrameSinkTests`'
   `WhenNull` test (the fallback's regression cover) — both die with
   WasiHost.
2. WasiHost `HelloComponent` dedupe — NativeHost carries its own copy since
   Gate 2; the WasiHost twin deletes with the project.
3. Kotlin kotlinx-serialization JSON layer: `FrameStreamParser.kt` + the
   `RenderFrame` sealed-class JSON annotations (`@Serializable` /
   discriminator wiring). The Kotlin classes themselves STAY — they are the
   adapter's output model; only the serialization dies.
4. Golden-fixture regeneration note: the fixture was produced by the JSON
   path; once that emitter is gone, update the fixture's provenance comment
   (or regenerate via the struct path) so it doesn't reference deleted code.

### (b) Phase 3.2 items

1. **Async frames break the sync-only-frames invariant.** The
   Activity-recreation stale-trampoline window is safe today ONLY because
   frames fire exclusively inside `mount()`. Any async frame source must
   re-register the new callback before it starts. Hazard documented in
   `BlazorNativeRuntime` — 3.2 must design for it, not rediscover it.
2. **`blazornative_mount` needs an out-error-message like init.** Mount
   status 2 detail currently goes to native stderr, which Android discards —
   unreachable on-device. Give mount the init-style
   `ErrorMessage` out-param so failures surface through the ABI.
3. **`blazornative_dispatch_event` wiring** + AttachEvent/DetachEvent
   handling in `NativeFrameAdapter`/`WidgetMapper` (ABI slots + adapter
   mapping exist, dormant).
4. **NUL-termination helper consolidation** — 3 copies of the
   append-trailing-0 UTF-8 cstring pattern exist across the Kotlin side;
   consolidate when 3.2 touches this code.
5. **`normalize()` promotion** — hoist the golden-test normalization helper
   to a shared test util when a second golden test lands.

### (c) Deliberate non-wiring

- `MainActivity` has **no shutdown/onDestroy wiring — deliberate**: the
  native session is process-lifetime; Activity recreation re-runs `start()`
  against the same process-global session (see the recreation contract in
  `BlazorNativeRuntime`). `shutdown()` is reserved for genuine process-exit
  paths.
