# Design — `[UnmanagedCallersOnly]` Export Verification (Phase 1.3)

*Brainstormed: 2026-05-24 — Milestone 1, Phase 1.3*

## Problem

MILESTONE.md DoD #4 requires verifying that `WasiBridge.DispatchEvent`'s `[UnmanagedCallersOnly(EntryPoint = "blazornative_dispatch_event")]` attribute survives Mono-AOT compilation and is exposed in the .wasm module's export table — callable by the native shell at M2 time.

The original wording prescribed `wasm-tools dump artifacts/wasi/BlazorNative.Core.wasm | grep blazornative_dispatch_event`. Two reasons to update:
- Path is stale post-Phase-1.2 (architecture pivot: `BlazorNative.WasiHost.wasm`, not `BlazorNative.Core.wasm`).
- `wasm-tools` isn't installed and adds another external dependency. **`wasmtime` is already installed (Phase 1.2)** and ships with `--invoke FUNCTION module.wasm args...` — calls a specific export directly. If the export doesn't exist, wasmtime errors with `function 'X' not found`. If it exists, it runs and returns the function's exit code.

## Decisions

| # | Decision | Chosen | Rejected alternatives |
|---|---|---|---|
| 1 | Scope | **Visibility + invocation in the no-subscriber path** — one test that calls `wasmtime --invoke blazornative_dispatch_event ... 0 0 0 0`, asserts exit 0. Proves the export resolves AND the function is callable. Stops there. | A: visibility-only framing (same code, weaker assertion); C: also fix the D1 async trap inline (mixes phase scopes; trap fix needs its own brainstorm — queue-and-drain has ordering/drop-policy subtleties). |
| 2 | Tool / fallback path | **`wasmtime --invoke` as primary; lazy-install `wasm-tools` only if `--invoke` doesn't work on WASI components** | A: install wasm-tools upfront as hard prereq (more deps, slower setup); C: no fallback at all (loses recoverability if --invoke fails). |
| 3 | DoD #4 wording update | Bundle with Phase 1.5 housekeeping commit (alongside DoD #1c / DoD #3 wording fixes flagged in 2026-05-24 audit) | Update now (creates churn in MILESTONE.md mid-phase) |

## Architecture

```
                            tests/BlazorNative.Wasi.Tests/
                            ├── WasiPublishFixture.cs       (existing — reused)
                            ├── BootSmoke.cs                (existing — 2 passing tests; helpers extracted to WasmtimeRunner)
                            ├── WasmtimeRunner.cs           ← NEW (extracted shared helpers)
                            └── ExportSmoke.cs              ← NEW (1 new test)
                                          │
                                          ▼
                  Export_BlazorNativeDispatchEvent_IsCallable()
                                          │
                                          │  fixture.WasmPath → AOT'd BlazorNative.WasiHost.wasm
                                          ▼
                       wasmtime run -Shttp --invoke blazornative_dispatch_event
                                --dir=.
                                BlazorNative.WasiHost.wasm
                                0 0 0 0          ← (nameptr, namelen, payloadptr, payloadlen)
                                          │
                                          ▼
                       Mono runtime loads, [UnmanagedCallersOnly] export
                       resolved, function called with all-zero args:
                         var name    = Encoding.UTF8.GetString(null, 0);   → ""
                         var payload = null;                                (payloadLen==0)
                         if (Current is { } c) …                            ← false: --invoke
                                                                              skips Main, so
                                                                              WasiBridge.Current
                                                                              is never set
                                          │
                                          ▼
                       Function returns cleanly → wasmtime exits 0
```

**Why this works without tripping D1 (the Mono-WASI async trap):** `--invoke` skips `_start`/`Main` entirely. `WasiBridge.Current` stays `null`. The dispatch path never reaches `_events.InvokeAsync(...).AsTask().GetAwaiter().GetResult()` — the trap is dormant. Export callability is verified; subscriber-driven invocation is M2's problem and tracked in the existing BACKLOG "Mono-WASI async trap" bullet.

## Components

### New: `tests/BlazorNative.Wasi.Tests/WasmtimeRunner.cs`

Extract from `BootSmoke.cs`'s private methods:
- `ResolveWasmtime()` — locates `wasmtime.exe` on PATH / user-PATH-env / standard install dirs
- `Run(WasiPublishFixture fixture, string[] additionalArgs, TimeSpan timeout)` — wraps `Process.Start` with the env-strip (`DOTNET_*` / `MONO_*` / `ASPNETCORE_*`) + WorkingDirectory + WASI flags (`-Shttp` `--dir=.`) common to both tests

`internal static class WasmtimeRunner` in the same namespace. No behavior change vs. the existing `BootSmoke.RunWasmtime`; same signature, broader visibility.

### New: `tests/BlazorNative.Wasi.Tests/ExportSmoke.cs`

```csharp
[Trait("Category", "Integration")]
public sealed class ExportSmoke : IClassFixture<WasiPublishFixture>
{
    private readonly WasiPublishFixture _fixture;
    public ExportSmoke(WasiPublishFixture f) => _fixture = f;

    [Fact]
    public async Task Export_BlazorNativeDispatchEvent_IsCallable()
    {
        var (exitCode, _, stderr) = await WasmtimeRunner.Run(
            _fixture,
            extraArgsBeforeWasm: new[] { "--invoke", "blazornative_dispatch_event" },
            programArgs: new[] { "0", "0", "0", "0" },
            timeout: TimeSpan.FromSeconds(10));

        Assert.True(exitCode == 0, BuildFailureMessage(exitCode, stderr));
    }

    private static string BuildFailureMessage(int exitCode, string stderr) =>
        $"wasmtime --invoke exited {exitCode}.\nSTDERR:\n{stderr}\n\n" +
        $"If stderr mentions 'function not found' or 'no export', the [UnmanagedCallersOnly] " +
        $"attribute on WasiBridge.DispatchEvent did not survive Mono-AOT trimming. " +
        $"Fix candidates: add WasiBridge to a TrimmerRoots.xml referenced by " +
        $"BlazorNative.WasiHost.csproj, or add [DynamicDependency(...)] to keep " +
        $"DispatchEvent rooted.\n\n" +
        $"If stderr mentions '--invoke is not supported for components', install wasm-tools " +
        $"(https://github.com/bytecodealliance/wasm-tools/releases) and rewrite this test " +
        $"to shell out to `wasm-tools dump | grep blazornative_dispatch_event`.";
}
```

### Refactor: `tests/BlazorNative.Wasi.Tests/BootSmoke.cs`

Remove the private `RunWasmtime` + `ResolveWasmtime` methods. Update both existing tests to call `WasmtimeRunner.Run(fixture, extraArgsBeforeWasm: Array.Empty<string>(), programArgs: new[] { fixture.AppAssemblyName }, ...)`. Same behavior; one indirection.

```
[4] wasmtime loads the .wasm, locates the export by name

```

## Error handling

| Failure | Cause | What the test does |
|---|---|---|
| `function not found` / `no export` | Mono-AOT trimmer removed `WasiBridge.DispatchEvent` because nothing in the IL graph references it (the attribute alone isn't a trim root in all SDK versions) | Assert fails with message naming `TrimmerRoots.xml` / `[DynamicDependency]` fix candidates |
| `--invoke is not supported for components` | wasmtime v45 doesn't expose `--invoke` on WASI component worlds | Assert fails with message pointing at the `wasm-tools` fallback path |
| Function throws / traps | Should not happen with all-zero args — but if it does, the trap goes through `MonoError` and wasmtime exits non-0 | Assert fails; investigate stderr for trap details |
| Timeout (>10s) | `--invoke` deadlocked somehow (shouldn't happen for no-op path) | `TimeoutException` from `WasmtimeRunner.Run`; process killed |

## Out of scope (deferred)

- **D1 async-trap fix** — moved to M2 brainstorm. The existing BACKLOG "Mono-WASI async trap will fire on first real bridge event" bullet tracks three remediation options.
- **DoD #4 wording update in MILESTONE.md** — bundle with Phase 1.5 housekeeping (alongside DoD #1c "BlazorNative.Core.wasm" → "BlazorNative.WasiHost.wasm" and DoD #3 ".NET 9 cooperative scheduler" → ".NET 10 Mono-WASI runtime") flagged in the 2026-05-24 audit.
- **`setup.ps1` `wasm-tools` install path** — only added if Phase 1.3 hits the fallback (i.e. `--invoke` doesn't work on components).
- **Negative-case test** (assert function ISN'T present under wrong name) — overkill for a binary visibility check.

## Risks

1. **`--invoke` semantics on WASI components in wasmtime v45.** Wasmtime added component-model `--invoke` support during the v20-v30 era; v45 should have it. Fallback documented.
2. **Mono-AOT trimming of `[UnmanagedCallersOnly]` methods.** The .NET 10 Mono-AOT trimmer SHOULD treat `[UnmanagedCallersOnly]` as an implicit trim root, but if it doesn't, the test will fail with a clear message pointing at the fix.
3. **`WasmtimeRunner` refactor regression** — small risk that the extracted helpers change behavior subtly. Mitigated by re-running the existing BootSmoke tests.
