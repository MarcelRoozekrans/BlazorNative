## THE HEADLINE — the generator lied during design, and three pins read as armed while inert

**This phase's whole subject is stale copies, and it kept catching itself.** Not as a rhetorical
flourish — four times, in the machinery built to prevent exactly this, each caught only because
something counted.

### 1. `Generation: 10 succeeded, 0 failed` — exit 0, green, and zero components

xmldoc2md run against `src/BlazorNative.Components/bin/Release/net10.0/`:

```
Generation: 10 succeeded, 0 failed
```

Ten files: `bntheme`, six flex enums, `flexstylevalues`, `imagecontentmodes`, `_imports`. **Not one
component.** No `BnView`, no `BnText`, no `BnButton`. The generated `index.md` — the reference's front
page — listed ten types and no components, and **the tool exited 0 calling it success.**

The cause sits next to the assembly: `bin/` holds only `BlazorNative.Components.dll` +
`BlazorNative.Core.dll`. **`Microsoft.AspNetCore.Components.dll` is not there**, so `ComponentBase`
will not resolve, so **every derived type drops SILENTLY**. Against a `dotnet publish` output, where
the dependency is present: **`Generation: 26 succeeded`**.

**10 versus 26. Same tool, same exit code, same `0 failed`, same reassuring green.** The difference is
**invisible to anything that does not count**. This is `return ,$names` (8.1) and the blind nupkg
scanner (8.1 I-3) and the roster that did not know its own size (8.3 I-1), arriving for the fourth
time by a new door. **It is not a risk that was mitigated. It is a defect that already happened, on
this machine, during design — and the only reason it is in a document instead of on the site is that
something counted.**

### 2. Pin 2 read as ON and was OFF — twice

**First:** `src/Directory.Build.props` imports **before** the project body, so the csproj's
`<BnEnforceDocCoverage>true</BnEnforceDocCoverage>` was set *after* the `NoWarn` condition had already
been evaluated. The result: `{BnEnforceDocCoverage: true, NoWarn: …;CS1591}` — **true and suppressed
anyway.** Fixed by moving the switch to `src/Directory.Build.targets`, which imports *after*.

**Then it still was not a pin.** `ci.yml` builds without `-warnaserror` and nothing set
`TreatWarningsAsErrors`, so CS1591 would have been **8 warnings in a lane that exits 0**.

***A pin's costume on a pin's absence*** — twice, in the phase whose subject is things that read as
true and are not.

### 3. Pin 2's guarantee covered exactly HALF the parameter surface

The review's S1-1, reproduced and confirmed. The taxonomy claimed *"every public component member is
documented | PINNED | CS1591 as an error"*. **That is false for `.razor`, and structurally so:** the
Razor generator emits `#pragma warning disable 1591` at **line 3 of every `*_razor.g.cs`**, and an
`@code`-block `[Parameter]` is declared **inside that generated file**. CS1591 is structurally blind
to it. The hole was visible in Pin 2's **own recorded mutation** — proven with `BnButton.Label`, and
BnButton is `.cs`.

The mutation: delete `BnSlider.Value`'s `<summary>` — the property `@bind-Value` targets.

```
BEFORE:  0 Warning(s), 0 Error(s)   — 4/4 pins green, and
         P:BlazorNative.Components.BnSlider.Value GONE from the shipped XML.
         The consumer's tooltip empties; nothing reds.
AFTER:   1 of 196 [Parameter] properties have NO <summary> and no <inheritdoc>
         in the shipped XML: BnSlider.Value
```

Closed by `EveryParameter_CarriesADocComment` — subjects reflected, expectation a non-empty
`<summary>` **or** an `<inheritdoc>`.

### 4. `onBrokenAnchors` defaults to `'warn'`

`onBrokenLinks: 'throw'` was adopted *"as a PIN, not decoration"* — but it decides **pages, not
fragments**, and **anchors are half the internal-link surface**. This site is the case that needs it:
the reference is **GENERATED** from `<see cref>`, which xmldoc2md renders as **intra-page anchor
links** — so the largest body of anchors here is machine-written from C# doc comments nobody
proofreads. **Gate 2 fixed five broken anchors at the source cref and nothing kept them fixed.**

Docusaurus agrees in its own default — `configValidation.js:123` reads
`onBrokenAnchors: 'warn', // TODO Docusaurus v4: change to throw`. **v4's behaviour adopted early
rather than inherited late.**

## THE COUNT THAT WAS TRUE OF SOMETHING ELSE — the phase's emblem

**"21" was true.** At design time, 21 types in `BlazorNative.Components` carried a doc comment —
because the six `.razor` components had **no type summary** (the headless pages Gate 2 fixed). 27
documented types − 6 headless = **21 documented types**. A true, narrow, correctly-measured number.

**It was then transplanted into three files as "21 components"** — `generate-reference.ps1`,
`sidebars.js`, and the design — **where it was never true of anything.** Measured: **15 concrete
`ComponentBase` types** (9 `.cs` + 6 `.razor`; `BnFlexPreset` is abstract), **26 public types**, **196
`[Parameter]` properties**. `ci.yml`'s own recorded Pin 1 mutation says *"MISSING 15 OF 15
COMPONENTS"*, which is how we know.

**The original measurement did not rot. It was copied out of its meaning** — which is the failure mode
this entire design is built around, landing on the design itself.

**And "192" was wrong too.** It came from `grep '[Parameter]'`, which **cannot see
`[Parameter, EditorRequired]`** — `BnList.razor` declares four (`Items`, `ItemHeight`, `Height`,
`RowTemplate`), so the pattern read **BnList as 2 where the assembly has 6**. The review reproduced
the design's 192 and **reconciled to it, inheriting the same blind pattern**. Decompiling
(`ilspycmd` → 196), reflecting (the new pin's own count → 196) and `grep -cE '\[Parameter[],]'`
(98 `.cs` + 98 `.razor` = 196) agree.

**The fix pass's pin derived 196 on its first run, because it reflects instead of grepping.** The
blind half was **exactly 50% — 98/98**, not the 51% the review computed. *A number is not measured
until the thing that reads it can see every form it takes.* The **claim** was always true — all 196
are documented. Only the count was wrong. **A pin that DERIVES its count is the only reason the
artifact surfaced instead of being copied forward a fourth time.**

## The DocFX answer, as the owner asked for it

The owner asked *"any advantages over DocFX?"* and pointed at their AdoNet.Async setup. The honest
reckoning, because the answer **changed shape when it was measured**:

**What DocFX genuinely wins, and it is not nothing.** It is **native .NET**. It ingests your
assemblies and their XML docs and emits an API reference **for free** — no second generator, no glue,
no dependency-resolution surprises. It cross-links your own types automatically. **For a repo whose
reference surface IS .NET types, that is the most relevant single feature either tool has, and
Docusaurus does not have it at any price.**

**What Docusaurus genuinely wins.** **You already run it** — one toolchain, one CI shape, one thing to
upgrade, one thing to debug at 11pm. Its prose, navigation, MDX and theming beat DocFX's by a wide
margin, **and a docs site is 80% prose**.

**What measuring changed — and it is why this is not close:**

- **The XML docs were already COMPLETE** (8 gaps in Components, all `BuildRenderTree`). **Nothing was
  traded away.** 8.1's ledger recorded *"doc coverage is 8.4's editorial work"* — **that premise was
  wrong.** Coverage was done; **VOICE was the job.**
- **Their content was repo-voiced.** `BnTheme` cited *"Phase 3.4 design §4"*. `BnView` said *"since
  Phase 3.4 Gate 1"*, *"the BnDemo goldens stay byte-identical"*, and *"Razor awaits .razor
  compilation (M6)"* — **which was not merely dated but FALSE**: `<BnInput @bind-Value="_text" />` is
  live in **`BnDemo.razor:51`**. Four component pages would have told strangers the framework cannot
  do the thing the sample app does on line 51. **DocFX would have published that phase history exactly
  as faithfully.** A generator is a printing press.
- DocFX would hit the **same `ComponentBase` resolution wall** from the same direction, and would need
  **the same counting pin** to prove it hadn't.

**Both tools needed the same editorial pass on the same XML. DocFX saves the plumbing, not the work** —
and against a toolchain the owner already knows, **plumbing is the cheaper thing to give up.**

## The one-home rule, as shipped

> **NORMATIVE RULE — one home per fact.** Every fact the site states is **GENERATED**, **LINKED**, or
> **OWNED**. There is no fourth option. ***"Kept in sync" is not a home.***

**The site NEVER includes — it points.** Docusaurus *can* read outside its tree (`docs.path`), so
INCLUDE was available and was **rejected**: it makes the site's build depend on the shape of
contributor docs, and produces a second *rendering* of a doc written to be read in the repo. It buys
nothing a link does not.

**The precedent is 8.3's, generalized: prose a gate does not keep true should POINT, not COPY.**
`docs/plans/` links **as a class**. The parity page **states the claim and links** the normative 6.3
section + `ShellFrameTableDriftTests` — it does not reproduce the 10-row table.

**Reviewer-audited: the site states no number a gate owns.** No counts, no versions, **no Yoga
literal**. *A number on a page nobody re-runs is a number that is already wrong.*

## Gate 1's best catch — a permanent 404 inside a package

**The design named ONE inbound link to the moved docs. There were eight.**

The other seven are **`helpLinkUri` values in the analyzer sources** — *the string a consumer's IDE
opens from a BN squiggle.* Deleting `docs/analyzers.md` per the fates table would have shipped **a
permanent 404 to every user of every BN rule, at the moment they are already confused.**

Re-pointed at Gate 1 — and **pinned at Gate 4** (`AnalyzerHelpLinkDriftTests`, the review's S2-2).
They were correct; **nothing kept them correct** (`grep -rln helpLinkUri tests/` returned **nothing**).
**They ship INSIDE a nupkg and are absolute external URLs: the one link class a consumer's IDE
resolves that no site build can ever see** — `onBrokenLinks` and `onBrokenAnchors` check the links the
*site contains*, and these are pointed *at* it from a released package. **A wrong one is a permanent
404 for everyone who installed that version; redeploying the site cannot fix it.**

**Both sides derived:** subjects reflected from `SupportedDiagnostics`; anchors parsed from
`website/docs/analyzers.md` **with code fences skipped** (the page opens with
`#pragma warning disable BN0004` inside one, which a naive `/^#/` parser mints an anchor for); the base
composed from the site's own `url` + `baseUrl` + `routeBasePath` + the file's own name — **not one
segment of the address is transcribed.**

```
MUTATION 1: BN0004's link → #bn0099
  1 of 7 shipped analyzer help link(s) point at an anchor that does not exist.
MUTATION 2: baseUrl → '/BlazorNativeMoved/'
  7 of 7 shipped analyzer help link(s) are NOT built on this site's url + baseUrl.
```

**That second red IS U2 seen from the package side** — move the site and all seven rot at once, with
no other witness anywhere.

## The design-corrections (the review judged 15 of 16 right — the corrections are the record)

- **No `prism-razor.js`** — it is `prism-cshtml.js`; razor is an alias.
- **Gate 1 cannot own the sidebar** — the autogenerated category needs Gate 2's generated directory.
- **`generate-reference.ps1` has exactly TWO callers** — the site's `prebuild` and Pin 1. ***Had the
  lane and the pin run different pipelines, the pin would be green while the lane went blind.*** The
  review called this **the strongest architectural call in the phase**.
- **The script asserts NOTHING about completeness**, so Pin 1's mutation proves **Pin 1**, not a guard.
- **GitHub Actions does not support YAML anchors** — the push and PR filters are duplicated by
  construction and drift by hand. Kept identical.
- **`format: 'detect'`** — MDX rejects the generator's `<br>`.
- **The paths filter must be wider than `website/**`** — `+ src/Directory.Build.props` / `.targets`:
  ***they decide whether the XML exists at all.***
- **Pin 3 scans the PUBLIC SURFACE, not the file.** The shipped XML also documents **internal** types
  — `BnItemsJson`'s wire grammar, `BnListWindow.Compute`'s arithmetic, `BnPicker.Clamp`'s guard — that
  the generator drops (`--member-accessibility-level public`) and no consumer can reach. **A
  whole-file regex would red on maintainer docs doing their job and *teach the next author to delete
  engineering truth to go green*.**
- **Pin 3's own mutation corrected the pin:** `/\bgolden\b/` **cannot match "goldens"** — the trailing
  boundary fails against the plural, and *"the BnDemo goldens"* is the exact phrase that was in the
  file. 3 of 4 fired. The boundary is gone; all 4 fire. ***This is why one runs them.***

**The 16th: the design asked for an IMPOSSIBLE mutation.** *"Rename a component ⇒ Pin 1 red both
ways"* — **STRUCK**, with the reason kept rather than deleted. **Pin 1 derives BOTH sides**, so a
rename moves the type and its expectation together: **green, correctly, and by design**, exactly as
its own doc comment claims. Satisfying that row would require **the hand-maintained roster 8.3's I-1
forbids**. ***The mutation was mis-specified, not the pin.***

## Three prescriptions the fix pass REFUSED, correctly

**Worth recording: the reviewer is strong and was still wrong three times.**

1. **Reusing `PublicSurfaceDocs()` for S1-1** would have **reddened 124 correctly-documented
   properties.** It returns `member.Value`, and an **inheritdoc-only member's Value is the empty
   string** — indistinguishable from undocumented. **124 of the 196 are documented that way**
   (BnFlexPreset 23; the four `.razor` form controls 17 each; BnScroll 16; BnImage 15; BnModal 2),
   overwhelmingly BnView's flex/box vocabulary re-exposed — **where one home for the sentence is the
   right answer.** Demanding a hand-written summary each would demand **the 124 copies this repo
   exists to refuse.** The new pin reads the *elements*.
2. **`navbar.logo.to` is invalid** — the build refuses it: *`"navbar.logo.to" is not allowed`*. `to` is
   for navbar **items**; the logo schema takes `href` only. **The diagnosis was right, the remedy was
   `href: '/'`** — and **the build is the pin that said so.**
3. **The "192" reconciliation** — above.

## The seventh package (Gate 3's find — no ledger had noticed)

8.1 declared `src/Directory.Build.props` **the ONE metadata home**. 8.3 then created
`templates/BlazorNative.Templates` **OUTSIDE `src/` deliberately** — a seventh csproj inside
**un-licenses the props** (the shipped-set pin is what makes that home legal). **So it carries its own
`PackageProjectUrl` by necessity, and it was still pointing at the repo while the six moved to the
site.** ***Shipping 6-of-7 is not a rule, it is a coincidence waiting to be noticed by a user.***

Split the same way Gate 3 split the six: **`RepositoryUrl` = where the source is; `PackageProjectUrl`
= where the docs are.** Verified **in the PACKED nuspec**, since **the pack is what a user gets**:

```xml
<projectUrl>https://marcelroozekrans.github.io/BlazorNative/</projectUrl>
<repository type="git" url="…/BlazorNative" commit="e2bc3be…" />
```

**Also Gate 3's:** the footer label **`Source: <repo>`** re-pointed at a docs site would read
***"Source: a site that is not the source"*** — **a fact made wrong by the act of fixing it**, which
is this phase's own failure class arriving through the door marked "mechanical". Now **`Docs:`**;
**nothing is lost — nuget.org renders Source from the nuspec's `<repository>`**, which the smoke
already interrogates as `repository@commit`.

## Found only because something finally looked

**The social card's tagline overflowed its own 1200px viewBox** — clipping the final "s" from
"views". SVG text does not wrap, and **nothing had ever rendered the file**, so nothing had ever shown
it. **Invisible for as long as nothing rasterized it.**

**And `og:image` as an SVG renders NO preview on any social platform** — X, Facebook, LinkedIn and
Slack **every one require a raster**. A link posted as a bare URL would have unfurled *nothing*, with
the tag present and the file resolving 200. **Decision 6 had already learned this for NuGet
("JPEG/PNG only, SVG unsupported") and it did not transfer.** Both fixed; **the card is a PNG**, the
SVG stays as the vector source.
