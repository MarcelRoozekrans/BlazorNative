**ZERO library code, and the wire NOT AT ALL — the packages became the truth.** Still 9
exports + the 72-byte bridge, thirteen NodeTypes on three mirrors, every golden
byte-identical, the publish head zero-diff. What changed is what a stranger can *hold*:

nothing left to fix when it does.

## The six packages, as shipped-ready

**The set is `Core, Renderer, Http, Components, Runtime, Analyzers`** — the purity pin's
literal, verbatim. Two calls inside that, both recorded:

- **Http ships, and the DoD's list is corrected.** MILESTONE.md DoD #2 named five
  ("Core, Runtime, Renderer, Components, Analyzers"). That list was **shorthand drift,
  not a scoping decision**: Http has packed and shipped through the consumer smoke since
  4.5, `ConsumerSmoke.csproj` references it, Runtime `ProjectReference`s it (so a
  Runtime package would carry a dangling dependency if Http went internal), and the
  purity pin's shipped-set literal — the deliberate, mutation-guarded declaration of what
  ships — says six. Making Http internal-only would have been a *regression dressed as a
  reading of the DoD*. **MILESTONE.md's DoD #2 list is corrected to the six** as part of
  this close.
- **Runtime is the genuinely new package** — packable only because **8.0 evicted
  `PublishAot`** into `samples/BlazorNative.SampleApp`. In 4.5 it was the publish head,
  unpackaged by design; its own csproj header already said "a plain packable library
  (8.1 ships it)". It gained a Description/Tags and the shared metadata; its build
  surface (`IsAotCompatible`, the analyzer shielding, the IVTs) is **byte-identical**.
  Its first-ever pack produced **zero warnings**.

**`src/Directory.Build.props` is the one metadata home** (`6a73386`) — Version, Authors,
MIT expression, RepositoryUrl/PublishRepositoryUrl, PackageProjectUrl, snupkg symbols +
`EmbedUntrackedSources` + `ContinuousIntegrationBuild` (GITHUB_ACTIONS-gated),
`PackageReadmeFile` + the Exists-guarded `None` item, XML docs on with CS1591 suppressed
by decision. The six csprojs were trimmed to per-package voice; the redundant
`PackageId`/`Version`/`Authors`/license/`RepositoryUrl` lines and their stale 4.5
comments are **deleted**.

**The finding that licensed the move: the 4.5 "per-csproj by design" rationale was
obsolete BY CONSTRUCTION.** Its stated reason — *"a scoped props under src/ would leak
Version/pack metadata onto the non-packed projects (Runtime, Blazor)"* — was true when
written and false now: `PackagePurityTests.TheShippedSet_IsExactlyTheSrcCsprojs` **proves
src/ holds exactly the six shipped csprojs**, so a src-scoped props rides precisely the
packed set and leaks onto nothing. `tests/`, `samples/` (the publish head *and* the
ConsumerSmoke blank consumer), and the shells are siblings and inherit nothing. The pin
did not merely permit the decision — it *is* the decision's proof, and a stale recorded
decision is worse than none, so the comments went with it.

**The M4 `PackageReadmeFile` prerequisite is CLOSED** — six READMEs, one per project
(what it is, stack position, an honest snippet, the preview disclaimer, the repo link).
`PackageReadmeFile` without content is a mechanism without a deliverable. **The Http one
was corrected against `AndroidShellBridge.kt` reality** rather than against an imagined
design: the shell owns the sockets — `HttpURLConnection` on a single background executor
— and the README says so. The Analyzers README quotes BN0011's **real** `messageFormat`
read out of `MobilePolicyAnalyzer.cs`.

## The versioning verdict (DoD #2's "decided and recorded" lands here)

**One `<Version>1.0.0-preview.1</Version>` in `src/Directory.Build.props`, hand-bumped.**
Pinned by `PackageVersionPinTests` (the props carries exactly one `<Version>`; no shipped
csproj overrides it) and by the smoke's nupkg-filename drift check — so drift back to
per-csproj literals is a **red, not a review catch**. The churn surface the 4.5
conclusion recorded (7 files) **collapses to 1**: the script parses the props and rides
`-p:BlazorNativeVersion` through every restore/build; `ConsumerSmoke.csproj` carries no
in-file fallback (restoring outside the script fails loudly **by design** — the script is
the documented entry point, said in the csproj comment).

**The reset from `1.2.0-phase-4.5` is a downgrade with no observer.** That version was
phase-tracking noise, never published anywhere public — local feed + per-run temp caches
with `--no-cache`, provenance-asserted since 4.5. Nothing on any machine pins the old
identity, so the semver "downgrade" cannot be observed by anyone. `1.0.0` itself is a
milestone decision, not this phase's.

**GitVersion + release-please: REJECTED FOR NOW** — the AdoNet.Async precedent is real
and works *for that repo's shape*; this repo differs on every axis that matters. The
honest reasons, recorded:

- **Height-based version-per-merge contradicts DoD #3.** GitVersion's ContinuousDelivery
  mode mints a version per merge distance — right when merges flow toward releases. M8's
  law is the opposite: *nothing is public until the owner publishes a Release*.
  Version-per-merge buys nothing on a local-feed-only CI and replaces ONE named literal
  with a computed, non-obvious answer to "what version is this?".
- **The payoff starts once 8.2's workflow exists.** release-please automates changelog +
  release-PR cadence; M8 expects a handful of hand-decided previews. Whether
  release-please **cuts the GitHub Release that triggers 8.2's `release.yml`** is a
  genuine 8.2 design question — recorded as a **named 8.2 input**, not pre-decided here.
- **Conventional-commit bump inference doesn't map onto phase-structured history.**
  `feat:`→minor / `fix:`→patch meets a log written as `docs(phase-8.1):` /
  `test(phase-7.5):` / `ci(...)`. Automated inference over this history would mis-bump
  **silently**.
- **A concrete collision, named for 8.2.** AdoNet.Async's `tag-prefix: v` cannot be
  copied blindly — this repo's `v*` namespace is already occupied by **milestone** tags
  (`v4.0`…`v7.0`, and `v8.0` is DoD #6's close). 8.2 must disambiguate package-release
  tags from milestone tags *before* any tag-triggered machinery exists.

Consistency with the owner's other repo is worth having **at the release-workflow layer**,
where the owner interacts with it — not at the version-literal layer, where this repo's
needs are simpler.

## THE HEADLINE — the review demonstrated a vacuous pass, and the fix has a positive control

Gate 1 shipped a nupkg type scanner that **passed GREEN while blind.** The reviewer
neutered `Get-TypeNames`' return to `@()` and the **entire smoke stayed green** —
`"0 types clean"` ×6, **exit 0**. Every type-purity assertion in the interrogation was an
*absence* claim with nothing behind it: zero app-shaped names and zero moved-roster types
are both trivially true over an empty list.

That violates a house rule **this repo already states in its own code** —
`PackagePurityTests.TypeNamesOf`: *"a pin that cannot see its subject must never pass
vacuously."* And it is not hypothetical: the in-code comment at `Get-TypeNames` records
that a wrapping bug **already made the scan report "1 type" once**, caught by eye. The
same failure mode, the same function, twice — the second time with a scanner that would
have shipped it.

**Fixed with a two-arm positive control per package** (`1c3d18f`), and the arms are
proven **non-redundant** by re-running the historical bug rather than reasoning about it:

- **Arm 1 — total blindness:** `$typeNames.Count -gt 0`.
- **Arm 2 — a real sentinel type per package**, read off each dll's actual packed
  surface, not guessed: Core/`IMobileBridge`, Renderer/`NativeRenderer`,
  Http/`BridgeHttpHandler`, Components/`BnView`, Runtime/`BlazorNativeApp`,
  Analyzers/`MobilePolicyAnalyzer` (the analyzer that owns BN0011 — the sentinel and the
  live trip tooth name the same type).

**MUTATION I-3a** (the reviewer's exact vehicle, `return @()`) — was "0 types clean" ×6 +
exit 0. Now it stops at the **first** package, emits **zero** "types clean" lines, exit 1:

```
✗  BlazorNative.Core: the type scan read ZERO types out of BlazorNative.Core.dll -
   every purity assertion below would pass VACUOUSLY over a blind scanner. This is
   Get-TypeNames failing to see its subject, not a clean package (Gate 1 review, I-3).
```

**MUTATION I-3b** (the **historical** bug the comment records — `return ,$names`, the
wrapping that reports "1 type") — **count=1 sails straight past Arm 1.** This is the case
only the sentinel can catch, and it is why two arms exist:

```
✗  BlazorNative.Core: the sentinel type 'IMobileBridge' is NOT among the 1 types the
   scan read from BlazorNative.Core.dll - either the scanner is misreading the surface
   (the purity assertions below are then vacuous) or the package genuinely lost a
   load-bearing type. Both are stop-and-analyze; do not 'fix' this by changing the
   sentinel without reading the dll (Gate 1 review, I-3).
```

Both reverted; the script carries no mutation residue. **The lesson was then applied to
its own siblings in the same PR** — every new pin I-2 added is non-vacuity-asserted
(`SrcCsprojNames()`, `ShippedCsprojs()`, and both foreign-file parsers red on an empty
read rather than greening over one).

## The mutation-4 correction — proven with an equivalent vehicle, NOT unprovable

Gate 1 recorded design mutation 4 (*"a `ProjectReference` to SampleApp added to a shipped
project → the nuspec dependency scan reds"*) as **unprovable by construction**. Two things
were wrong with that, and both are corrected:

**The vehicle IS structurally impossible — and that part is a genuine finding.** SampleApp
transitively depends on all six shipped projects, so pointing a shipped project back at it
is a cycle MSBuild refuses before any pack happens. Re-confirmed this gate, quoted:

```
error MSB4006: There is a circular dependency in the target dependency graph involving
target "_GenerateRestoreProjectPathWalk".
[src/BlazorNative.Runtime/BlazorNative.Runtime.csproj]
```

**But the ASSERTION is provable, and Gate 1 had exercised the wrong branch.** The nuspec
dependency scan has **two** branches; Gate 1's Newtonsoft.Json substitute tripped
**branch 2** (the third-party allowlist) — not branch 1, the
`BlazorNative.*`-and-not-shipped tooth that mutation 4 actually targets. Re-run with the
reviewer's non-cyclic vehicle (a packable `BlazorNative.*`-named stub under `samples/`,
referencing nothing, `ProjectReference`'d from Core), exit 1:

```
✗  BlazorNative.Core: nuspec dependency 'BlazorNative.FakeDep' is outside the shipped six
   - app/sample types must never ride a package's dependency group
```

Confirmed it was **branch 1 and not the allowlist**: zero "not a known third party" lines
in the run. Vehicle reverted — `samples/` is back to its two projects and nothing named
FakeDep survives in the tree.

**The record therefore reads: the SampleApp vehicle is structurally impossible (MSB4006,
quoted); the tooth is proven with an equivalent non-cyclic stub.** A cycle in one vehicle
is a fact about that vehicle, not about the tooth — and "unprovable by construction" is
the kind of sentence that retires a test nobody re-examines.

## The four-copies finding (I-2) — a docstring that promised a pin that did not exist

Normative rule 2 says *the shipped set is pinned everywhere it appears.* **It was not.**
Only `TheShippedSet_IsExactlyTheSrcCsprojs` enumerated `src/`; **three copies rode along
unpinned** — `PackageVersionPinTests`' `ShippedProjects` roster, `consumer-smoke.ps1`'s
`$packages`, and ConsumerSmoke's six `PackageReference`s.

Worse than unpinned: the roster's docstring **claimed** *"purity tooth 3 keeps this equal
to the src/ enumeration"* — false **and unfalsifiable** (tooth 3 pins its own literal
against src/ and had never read that file). A docstring asserting a pin that does not
exist is worse than no docstring: it tells a reviewer the drift is already handled.

**The failure was reachable, not theoretical:** add `src/BlazorNative.Seven`, tooth 3
reds, a dev adds `"BlazorNative.Seven"` to `ShippedAssemblies` to green it — and Seven is
now **packed by nothing, smoked by nothing, version-guarded by nothing**, with every test
green.

The fix distinguishes copies that must be **pinned** from copies that should not **exist**
(`eeb04b0`):

- **The roster is DELETED.** `PackageVersionPinTests` enumerates `src/` itself
  (`ShippedCsprojs()`, full paths — a name list would have to be re-joined into
  `src/{name}/{name}.csproj`, assuming the flat layout *and* making the `File.Exists`
  check a tautology). Nothing left to drift; a seventh project is version-guarded the day
  it appears. The false docstring is replaced by what is actually true.
- **The two copies that CANNOT be removed** — a `.ps1` and an out-of-solution csproj — are
  pinned against **the same enumeration**, in `PackagePurityTests`, whose tooth 3 widens
  from *"the literal is pinned"* to *"the set is pinned everywhere it appears"*. Both
  foreign files are read out of the checkout: build-test is the one required lane where
  every file is visible (`RouteTableDriftTests`' rule). The script is parsed by regex
  anchored at line start (so the comment above `$packages`, which names it, cannot match);
  the csproj by `XDocument` — it is structured, `PackageVersionPinTests` already
  `XDocument.Load`s csprojs, and the "as text" of the drift-test rule is about the **lane**,
  not about refusing a real parser.

**MUTATION (the finding's own story, end to end)** — `src/BlazorNative.Seven` added
carrying `<Version>9.9.9-unguarded</Version>`, then "greened" the pre-fix way by rostering
it. Tooth 3 does go green — and **five pins red**:

- `TheConsumerSmokeScript_PacksExactlyTheShippedSet` — never packed
- `TheConsumerSmokeProject_ReferencesExactlyTheShippedSet` — never consumed
- `NoShippedCsproj_OverridesTheSharedVersion` — *"Offenders: BlazorNative.Seven:
  `<Version>9.9.9-unguarded</Version>`"* — the version pin seeing a seventh project for
  the **first time**; pre-fix it walked a roster of six and was structurally blind to it
- `TheMovedRoster_IsAbsentFromEveryShippedAssembly` + `NoAppShapedTypeName_...` —
  `ResolveAssemblyPath` reds loudly on Seven's absent dll rather than skipping it
  (`TypeNamesOf`'s own rule, working)

**Pre-fix that same end state was ONE red, greened by editing a literal.** A fake seventh
package now reds five pins instead of one greenable literal. All vehicles reverted; `src/`
and `samples/` are back to six and two.

## ConsumerSmoke is the 8.0 API's first out-of-repo consumer

`RegisterPages` had never been called by anything but this repo's sample app (via
`ProjectReference`). The smoke's `Program.cs` is its **first package-only caller**, and it
asserts the surface's **observable laws only** — nothing internal is reached for:

- **Duplicate-route `ArgumentException` naming the row** — sequenced **first**, because
  register-once means only the first call's argument set is ever validated. This proves
  the validation *strings* shipped, not just the happy path.
- **The valid registration succeeds** — `Routed<SmokeRoot>("/")` + `Named<SmokeProbePage>`,
  DAM(All) factories with a concrete **consumer-owned** `T` from the shipped package:
  the params surface and the trim annotations compile and run from the nupkg.
- **A second call throws** — the register-once law, observable from outside.

The existing renderer-harness mount stays alongside (frames=1 patches=10). **Deliberately
NOT claimed:** an AOT/ILC pass over a package consumer — the repo's own publish gates own
that today, and 8.3's template owns it end to end. A recorded boundary, not an oversight.

## The review trail

**Gate 1 review: PASS — three Important (I-1/I-2/I-3) + four Minor (M-1..M-4); every
finding applied, accepted, or ledgered, none carried silently.** I-1/I-2/I-3 and M-1 were
fixed (`1c3d18f`); M-4 (the dev-note's home) was accepted as-placed; M-2 (the dependency
allowlist's maintained-prefix churn) and M-3 (the pattern-net rename gap) are accepted
limitations, recorded in the ledger below with their reasons rather than fixed.

- **I-1** — mutation 4 was mis-recorded. The MSB4006 cycle is genuine, but the conclusion
  it licensed ("unprovable by construction") was wrong; the tooth is now proven with an
  equivalent non-cyclic vehicle (the section above). `1c3d18f`.
- **I-2** — the four-copies finding: one of four shipped-set copies was pinned, and the
  roster's docstring claimed a pin that did not exist. Roster deleted, the two
  irremovable copies pinned against the same enumeration. `eeb04b0`.
- **I-3** — the demonstrated vacuous pass: the type scanner passed green while blind.
  Two-arm positive control, both arms proven load-bearing. `1c3d18f`. **The headline.**
- **M-1 — DECISION: add the ancestor `Import` NOW, do not ledger it.** The finding offered
  either. MSBuild imports only the **nearest** `Directory.Build.props`, so the day a root
  one appears (8.2/8.3 are the likely authors), `tests/` and `samples/` would inherit it
  while **the six shipped projects silently stopped seeing it** — no error, no warning.
  Ledgering costs a named line that must survive 8.2/8.3 review and be acted on by
  someone who was not here; **a ledger line is a promise someone else has to remember.**
  The import costs ~3 lines and removes the trap instead of documenting it.
  **Proven three ways rather than assumed** — the fix is otherwise unverifiable, since the
  condition is false today (the same vacuity trap as I-3):
  1. **No root props (today):** `_AncestorDirectoryBuildProps` is `""`, the probe property
     is empty, the Import is skipped — a true no-op.
  2. **A temporary root props carrying `<M1ProbeFromRoot>`:** Core evaluates it to
     `root-props-was-inherited` — **the chain works** — while `Version` stays src/'s own
     `1.0.0-preview.1`, so the local file still wins (placement correct).
  3. **Same root props, Import disabled:** the probe goes empty again — the exact M-1 bug,
     confirmed **reachable**, and the Import is what closes it.

  The temporary root props was removed. The path goes through a property rather than
  inline in the `Condition`: the quoted property-function arguments get re-parsed by the
  condition grammar and error (**MSB4092**, *"unexpected token Directory"*) — observed
  here, not feared.
- **M-4 — accepted, no action.** The 8.0 review's **M-2 dev-note lives in
  `docs/GITHUB-SETUP.md`**, not in `docs/planning` as the design said, because
  **docs/planning has no dev-notes file** and inventing one to host a single line is
  worse than the deviation. The note is where a developer staring at a local `0 IL2072`
  will actually find it (next to the publish-gate documentation). Recorded here as the
  design deviation it is.

**The 8.0 ledger items owned here, both closed:** **M-3** — `ResolveAssemblyPath`'s
no-load fallback hardcoded `netstandard2.0` for the Analyzers reference path; it now reads
the TFM from the csproj it already knows how to find, with a loud assert if a single
