# Phase 8.1 Design — Publish-Ready Packages + Consumer Smoke on the Real Set (M8 DoD #2)

**Date:** 2026-07-16
**Milestone:** M8 — Developer Ecosystem (owns **DoD #2**: "publish-ready packages" —
the shipped set packs clean with `PackageReadmeFile`, license, symbols + SourceLink,
deterministic build; pack + local-feed consumer smoke asserted on CI every PR; the
versioning scheme decided and recorded)
**Predecessor:** Phase 8.0 — the samples/library separation + registration inversion
([conclusion](2026-07-16-phase-8.0-conclusion.md), PR #102; DoD #1 closed at `8c8238b`)
**Owns:** the package set and its metadata home, the versioning verdict, the
consumer-smoke extension to the real set (including the first out-of-repo consumer of
`RegisterPages`), nupkg-level purity, and the six per-package READMEs. **Design only** —
implementation follows in this phase's gates.

---

## The one sentence the whole phase hangs on

**Phase 8.1 makes the packages the truth, without touching what ships inside them** —
zero library code, zero wire, zero shell, zero publish-head change. The 4.5-era
packaging (five packages at a phase-stamped version, per-csproj metadata, a smoke that
predates the registration API) becomes the M8 shape: **six** packages sharing one
metadata home and ONE version literal, each carrying a README + license + symbols +
SourceLink, packed deterministically, and proven on every PR by a consumer smoke that
now exercises the 8.0 registration API from packages alone and interrogates the packed
nupkgs themselves. 8.2 then builds the release workflow that pushes these exact
artifacts; 8.1's job is that there is nothing left to fix when it does.

**The DoD #2 / DoD #3 boundary, stated so 8.1 does not overreach:** 8.1 = the packages
pack clean and prove themselves on CI every PR (pack + local feed + smoke + purity).
8.2 = the release pipeline (a `release.yml` triggered by a published GitHub Release,
the dry-run validation lane, `NUGET_API_KEY` as the one documented secret). This phase
creates **no workflow file, no secret, no tag, and no nuget.org interaction**.

Current bars (the standing law: counts with provenance): .NET **553/0** (ci.yml
build-test) · JVM **106/0** (ci.yml) · Android instrumented **184/0**
(android-instrumented.yml) · iOS XCTest **154/0** (ios.yml, run 29527121729). Device
lanes are **untouched this phase** — no shell work exists to re-verify. Publish gates:
**4 IL2072 + 9 exports**, all four RIDs — packaging must not move either number.

   and a scoped one under src/ would leak Version/pack metadata onto the non-packed
   projects (Runtime, Blazor)". **`BlazorNative.Runtime` has NO pack metadata** — in
   4.5 it was the publish head, unpackaged by design; 8.0 evicted `PublishAot` into

5. **The consumer smoke is the M4 shape:** `scripts/consumer-smoke.ps1` hardcodes

6. **The 8.0 registration API** (`BlazorNative.Runtime/BlazorNativeApp.cs`):

## Decisions (recorded at phase-open; ⚠ marks the veto-flagged judgment calls)

### 1. The package set: SIX — the purity pin's set, verbatim; Http ships

`Core, Renderer, Http, Components, Runtime, Analyzers`. Two calls inside this:

- **Http ships as a package.** MILESTONE.md DoD #2 names five ("Core, Runtime,
  Renderer, Components, Analyzers") — that list is shorthand drift, not a scoping
  decision: Http has packed and shipped through the consumer smoke since 4.5,
  `ConsumerSmoke.csproj` references it today, Runtime `ProjectReference`s it (so the
  Runtime package would carry a dangling dependency if Http went internal), and the
  purity pin's shipped-set literal — the deliberate, mutation-guarded declaration of
  what ships — says six. Making Http internal-only would be a *regression* dressed as
  a reading of the DoD. The conclusion doc will note the DoD list as corrected.
- **Runtime becomes the sixth package** — the one genuinely NEW package. It gains the
  shared metadata (decision 2) + a Description/Tags of its own. Its csproj changes are
  metadata-only: `IsAotCompatible`, the analyzer shielding, the IVTs, and everything
  else 8.0 left there stay byte-identical. The publish head
  (`samples/BlazorNative.SampleApp.csproj`) is **zero-diff this phase** — the standing
  law that packaging must not touch the publish head is a normative rule below.

The set is pinned in four places after this phase, all of which must agree: the purity
pin's literal, the `src/` csproj enumeration (same test), the smoke script's pack loop,
and `ConsumerSmoke.csproj`'s references. The smoke asserts `6` nupkgs, not `5`.

### 2. The metadata home: a new `src/Directory.Build.props` — shared truth, per-package voice ⚠

*The call to veto: (a) a src-scoped `Directory.Build.props` vs (b) keep per-csproj
(the 4.5 recorded design) vs (c) a repo-root props with `IsPackable` gymnastics.*

**(a).** The 4.5 rationale for per-csproj — "a scoped props under src/ would leak onto
the non-packed projects (Runtime, Blazor)" — was true then and is **false now by
construction**: the purity pin's third tooth proves src/ contains exactly the six
shipped csprojs. A `src/Directory.Build.props` therefore scopes onto precisely the
packed set; tests/, samples/, and the shells are siblings and inherit nothing. (c) is
rejected because a repo-root props would ride onto SampleApp, ConsumerSmoke, and every
test project — the exact leak 4.5 refused, plus it would reach ConsumerSmoke whose
whole identity is "a blank consumer with no repo inheritance". (b) is rejected because
DoD #2's new surface (readme/symbols/SourceLink/CIB/version) would multiply the
existing 5-file churn into ~10 identical stanzas — the churn the 4.5 conclusion already
called out as a carryover.

**What the shared props carries (identical for all six):**

- `<Version>` — **the one source of version truth** (decision 4).
- `<Authors>Marcel Roozekrans</Authors>`, `<PackageLicenseExpression>MIT`
  (the repo's `LICENSE` is MIT; the SPDX expression stays — nuget.org renders it and
  no file needs packing), `<RepositoryUrl>` + `<PublishRepositoryUrl>true>` (the
  SourceLink-facing pair), `<PackageProjectUrl>` = the repo (8.4 re-points to the docs
  site — a named 8.4 input).
- `<PackageReadmeFile>README.md</PackageReadmeFile>` + the
  `<None Include="README.md" Pack="true" PackagePath="\" />` item — each project
  supplies its own `README.md` file (decision 7); **pack fails loudly (NU5039) if one
  is missing**, which is a tooth, not a hazard.
- Symbols + SourceLink + determinism: `<IncludeSymbols>true` +
  `<SymbolPackageFormat>snupkg`, `<EmbedUntrackedSources>true`,
  `<ContinuousIntegrationBuild Condition="'$(GITHUB_ACTIONS)' == 'true'">true` (CI
  builds normalize paths; local builds stay debuggable). Determinism itself is the SDK
  default — asserted, not configured. SourceLink: the .NET SDK (8+) includes
  `Microsoft.SourceLink.GitHub` implicitly for git checkouts; Gate 1 **verifies** the
  packed nuspec carries `<repository ... commit="...">` rather than trusting the
  implicit inclusion — if the SDK band doesn't fire it, the explicit
  `PackageReference` goes into the shared props (a named fallback, applied only on
  evidence).
- `<GenerateDocumentationFile>true` + `<NoWarn>$(NoWarn);CS1591</NoWarn>` — the XML
  docs ride into the packages (and are the raw material for 8.4's xmldoc2md option).
  CS1591 (missing doc comment) is suppressed **by decision, recorded here**: doc
  coverage is 8.4's editorial work, and holding 8.1's zero-warning bar hostage to it
  would force either doc-stub spam or a silent bar break. Not a warning suppression
  smell — a sequencing decision.

**What stays per-csproj:** `Description`, `PackageTags` (each package speaks for
itself), the Analyzers' special shape (decision 3), and every non-packaging property.
The now-redundant per-csproj `PackageId`/`Version`/`Authors`/`PackageLicenseExpression`/
`RepositoryUrl` lines are **deleted**, along with the obsolete "per-csproj by design"
comments (replaced by one line pointing at the shared props and this design).
`PackageId` defaults to the assembly name for all six — the explicit lines go.

**Package icon: deferred, ledgered.** No icon asset exists; an icon is not a
pack-clean requirement and nuget.org renders without one. Inventing branding inside a
packaging phase is scope creep — it rides the ledger for 8.2/8.4 to pick up alongside
the docs-site identity.

**The 8.0 ledger's "shared publish .props":** that input was about the PUBLISH head
props (`PublishAot`, runtime-pack bypasses) for the 8.3 template — **not needed by
8.1** (this phase touches no publish machinery) and stays ledgered for 8.3, exactly
where 8.0 left it. The packaging props above is a different artifact; the two must not
be conflated into one file that mixes pack metadata with publish-head switches.

### 3. The Analyzers package shape: already correct — the delta is symbols + README

The 4.5 shape stands verified (analyzers/dotnet/cs via `None`, no `lib/`,
`DevelopmentDependency`, `SuppressDependenciesWhenPacking`, pinned private Roslyn
refs) and is NOT redesigned. Two deltas:

- **Symbols:** the shared props' `IncludeSymbols=true` would produce an empty/invalid
  snupkg for a package with no `lib/`. The Analyzers csproj overrides:
  `<IncludeSymbols>false</IncludeSymbols>` + `<DebugType>embedded</DebugType>` — the
  pdb travels inside the analyzer dll (the established pattern for analyzer packages;
  SourceLink still applies to the embedded pdb). The five library packages ship
  `.snupkg`; the smoke asserts **5 snupkgs + 6 nupkgs** in the feed.
- **README** rides the shared props mechanism unchanged — the `None` item packs
  `README.md` to the package root regardless of `IncludeBuildOutput`.

**The proof stays behavioral:** the consumer-smoke's analyzer trip (BN0011 surfaces in
a consumer build from the packaged analyzer, zero BN diagnostics without the trip) is
retained verbatim — it is the assertion that the analyzers/dotnet/cs path actually
loads, and it must still bite after every change above (a named mutation re-proves it).

### 4. Versioning: ONE `<Version>` literal in `src/Directory.Build.props`, manually bumped; `1.0.0-preview.N` for M8 ⚠

*The call to veto: (a) a single hand-bumped Version vs (b) GitVersion +
release-please, the AdoNet.Async precedent — and separately, the reset from
`1.2.0-phase-4.5` to `1.0.0-preview.1`.*

**(a), and the reset.** The scheme, recorded (DoD #2's "versioning scheme decided and
recorded" lands HERE):

- **Source of truth:** the `<Version>` element in `src/Directory.Build.props`. Named,
  singular, and pinned by a new build-test xunit fact (the six csprojs contain no
  `<Version>` override; the props contains exactly one) so drift back to per-csproj
  literals is a red, not a review catch.
- **Scheme:** `1.0.0-preview.N` — starting at `1.0.0-preview.1`. The current
  `1.2.0-phase-4.5` was phase-tracking noise, never published anywhere public (local
  feed + temp caches only, `--no-cache`), so the semver "downgrade" has no observer:
  nothing on any machine pins the old identity, and the smoke's per-run temp cache
  cannot collide. `preview.N` bumps are manual (a one-line props edit) until 8.2's
  release flow takes over the bump-at-release question; `1.0.0` itself is a milestone
  decision, not this phase's.
- **The churn surface collapses to 1:** `consumer-smoke.ps1` stops hardcoding
  `$version` — it **parses the props** (fails loudly if the element is missing or
  ambiguous) and passes `-p:BlazorNativeVersion=$version` to the ConsumerSmoke
  restore/builds; `ConsumerSmoke.csproj`'s six `PackageReference`s become
  `Version="$(BlazorNativeVersion)"` with **no in-file fallback** — restoring outside
  the script fails loudly by design (the script IS the documented entry point, said in
  the csproj comment). The script additionally asserts every packed nupkg filename
  carries exactly the props version — the version-drift tooth (a csproj that grows a
  `<Version>` override reds here AND in the xunit pin).

**(b), weighed honestly — rejected for NOW, and most of it re-addressed to 8.2 where
it actually belongs.** The AdoNet.Async machinery is real and works *for that repo's
shape*; this repo's shape is different on every axis that matters:

- **GitVersion (ContinuousDelivery, height-based)** mints a new version per merge
  distance — the right tool when merges flow toward releases. M8's law is the
  opposite: *"no package is public until the owner publishes a Release"* (DoD #3).
  Version-per-merge buys nothing on a local-feed-only CI and adds a computed,
  non-obvious answer to "what version is this?" — the exact opposite of ONE named
  literal.
- **release-please** automates changelog + release-PR cadence. M8 expects a handful of
  hand-decided previews; the automation's payoff starts after the release workflow
  exists — which is **8.2's deliverable**. Whether release-please CUTS the GitHub
  Release that triggers 8.2's `release.yml` is a genuine 8.2 design question, and it
  is recorded as a **named 8.2 input** rather than pre-decided here.
- **The bump rules don't map:** AdoNet.Async's conventional-commit inference
  (`feat:`→minor, `fix:`→patch) meets a history written as `docs(phase-8.1):` /
  `test(phase-7.5):` / `ci(...)` — phase-structured, not release-structured.
  Automated inference over this history would mis-bump silently.
- **A concrete collision, named for 8.2:** AdoNet.Async's `tag-prefix: v` cannot be
  copied blindly — this repo's `v*` namespace is already occupied by **milestone**
  tags (`v4.0`…`v7.0`, and `v8.0` is DoD #6's close). 8.2's release workflow must
  disambiguate package-release tags from milestone tags before any tag-triggered
  machinery exists.

Consistency with the owner's other repo is worth having **at the release-workflow
layer**, where the owner interacts with it — not at the version-literal layer, where
this repo's needs are simpler. That is the honest split, and it is veto-flagged
because it is an owner-preference call.

### 5. The consumer-smoke extension: six packages, and the first out-of-repo `RegisterPages`

The script's five-step shape survives; every existing tooth is retained verbatim
(pack-warning zero bar, provenance from `.nupkg.metadata`, BN0011 trip, zero-BN clean
build, harness mount + patch-set assertions). The deltas:

1. **Pack loop → six** (Runtime joins), feed assertion → 6 nupkgs + 5 snupkgs, version
   parsed from the props (decision 4), provenance loop → six ids.
2. **A new interrogation step** between pack and restore — decision 6.
3. **`ConsumerSmoke.csproj` adds `BlazorNative.Runtime`** (explicit reference — the
   package under test, not a transitive) at `$(BlazorNativeVersion)`.
4. **`Program.cs` exercises the 8.0 registration API from packages alone** — the
   API's first consumer that is not this repo's sample app, asserting the surface's
   observable laws (nothing internal is reached for):
   - `BlazorNativeApp.RegisterPages(BlazorNativePage.Routed<SmokeRoot>("/", nameof(SmokeRoot)), BlazorNativePage.Named<...>(...))`
     succeeds — proving the DAM(All) factories, the params surface, and the trim
     annotations **compile and run from the shipped package** with a concrete
     consumer-owned `T`.
   - A deliberately invalid registration (duplicate route, fresh AppDomain-free
     sequencing: validated on the FIRST call's argument set) throws
     `ArgumentException` **and the message names the offending row** — proving the
     validation strings shipped, not just the happy path.
   - A second `RegisterPages` call throws — the register-once law, observable from
     outside.
   The existing renderer-harness mount stays as-is alongside (it proves the
   renderer/components surface; the registration block proves Runtime's). What this
   deliberately does NOT claim: an AOT/ILC pass over a package consumer — that proof
   belongs to the repo's own publish gates today and to 8.3's template end-to-end
   (recorded boundary, not an oversight).
5. **The ci.yml step** updates name + header comment to the six-package/purity scope
   (provenance discipline: the step name says what the gate asserts).

### 6. Nupkg-level purity: interrogated in the smoke script, because only the script owns nupkgs

**Where it runs — decided by job topology, not preference:** in the build-test job the
xunit suite runs *before* the smoke step packs, so no test assembly can see a nupkg;
the smoke script is the only actor that owns the packed artifacts on every PR. The
interrogation is therefore a **script step (1.5 "interrogate")** over
`artifacts/packages/*.nupkg`, and the existing assembly-level `PackagePurityTests`
**remains the stronger daily guard** (metadata-level, runs earlier, mutation-proven) —
the nupkg step is the packaging-truth tooth layered on top, exactly as the 8.0 ledger
framed it.

**The assertion set, per nupkg (unzipped to a temp dir):**

- **Type-level purity:** the shipped dll (`lib/net10.0/*.dll` for the five;
  `analyzers/dotnet/cs/*.dll` for Analyzers) is scanned with
  `System.Reflection.Metadata` from pwsh (the same names-off-the-PE approach the xunit
  pin uses — no loading, no dependency closure): zero types matching the pattern net
  (`Demo$|Probe$|^SpikeRazor`) and zero types from the 16-row moved roster. If pwsh
  API loading proves brittle at implementation, the **named fallback** is an
  `--interrogate <feed>` mode on ConsumerSmoke.exe (C#, same assertions) — decided
  only on evidence, not both.
- **Nuspec truth:** package id + version match the props version; no dependency group
  in ANY nuspec names `BlazorNative.SampleApp` (or anything outside the six + known
  third-party ids); `<license type="expression">MIT</license>` present; the readme
  entry present AND `README.md` exists in the package root; `<repository ...>` carries
  the commit attribute (the SourceLink verification from decision 2).
- **Inventory shape:** the five library packages carry exactly one assembly (+ its XML
  doc) under `lib/net10.0/` — no stray content items; Analyzers carries **no `lib/`
  at all** and has its dll under `analyzers/dotnet/cs`.

Any red names the package, the file, and the offending entry — the loud-failure house
rule.

### 7. The READMEs: six short files, written in this phase, scoped honestly

`PackageReadmeFile` without content is a mechanism without a deliverable — the writing
is **in scope** (it is small and it is the M4-recorded prerequisite this DoD closes).
Six `README.md` files, one per project directory, each 15–30 lines: what the package
is, where it sits in the stack (one line: app → Components → Renderer → Runtime →
shells; Core = contracts, Http = the bridge handler, Analyzers = the BN rules), a
minimal honest snippet (Components: a Bn* markup fragment; Runtime: the
`RegisterPages` call; Analyzers: what BN0011 looks like), a preview-status disclaimer
(this is a POC heading to `1.0.0-preview.N`; the ABI and component surface may move),
and a link to the GitHub repo. **The links re-point to the docs site in 8.4** — a
named 8.4 input, so the site build includes a six-file link sweep. The root README is
not reused as package content: it tells the repo's story, not any one package's.

### 8. Hygiene riders (the 8.0 ledger, closed here)

- **M-3:** `PackagePurityTests.ResolveAssemblyPath`'s fallback hardcodes

- **M-2:** the dev-docs line lands (docs/planning dev notes): an *incremental* local

- The obsolete 4.5 "per-csproj by design" comments deleted with the migration

## The normative rules, stated once

1. **One version truth:** the `<Version>` in `src/Directory.Build.props`. No shipped
   csproj may carry a `<Version>`; the xunit pin and the smoke's drift check both
   enforce it.
2. **The shipped set is one set, pinned everywhere it appears:** the purity literal ==
   the src/ csproj enumeration == the smoke's pack loop == ConsumerSmoke's references
   == six nupkgs in the feed. A new package joins all five places deliberately.
3. **Packaging never touches the publish head:** `samples/BlazorNative.SampleApp.csproj`
   and both shells are zero-diff this phase; the 4 IL2072 + 9 exports gates run
   same-PR on all three ci.yml RIDs and must not move. Drift = stop and analyze —
   never re-baseline silently (the rule that caught the 8.0 trim finding).
4. **Every packaging claim is a CI assertion, not a property setting:** readme →
   NU5039 + the nupkg entry check; SourceLink → the nuspec commit attribute; symbols →
   the snupkg count; determinism/CIB → present in the props AND the pack runs under
   the zero-warning bar. A setting nobody asserts is a hope.
