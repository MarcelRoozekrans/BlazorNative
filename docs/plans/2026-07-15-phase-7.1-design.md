# Phase 7.1 Design — `.razor` Authoring End-to-End

**Date:** 2026-07-15
**Milestone:** M7 — Components + Razor (closes DoD #2 + the typed-props half of #8)
**Predecessor:** Phase 7.0 — the Razor-compilation spike, **GREEN**
([conclusion](2026-07-15-phase-7.0-spike-conclusion.md), PR #93)
**Owns:** converting the app. The five demo pages (`BnDemo`, `BnSettingsPage`, `BnLayoutDemo`,
`BnScrollDemo`, `BnImageDemo`) become `.razor` files with **parity proven per page**, then the
hand-written versions retire. After this phase, everything new is authored in `.razor`.

---

## The method (spike-proven; not up for re-litigation)

**Golden-vs-twin, per page, then swap.** For each page:

1. Author `<Page>.razor` under a transitional name (the `.razor` class and the existing
   `BuildRenderTree` class must be mountable side by side).
2. **Twin test**: mount both; assert the patch streams are record-identical across mount and the
   page's interactions (the 7.0 method — `SpikeRazorTests` is the template).
3. **Swap**: the `.razor` class takes the canonical name; the registry entry
   (`HostSession`/`NativeNavigationManager`) is untouched (it maps a *name* to a type); the old
   class retires. **Every existing golden and every shell frame table must pass UNCHANGED** —
   they are the parity proof, and none of them may be edited to make a conversion fit.
4. The twin test retires with the twin (its job is done; the goldens take over).

**Conversion order — hardest-risk-first after a warm-up:** `BnSettingsPage` (smallest — proves
the pipeline), then **`BnDemo`** (the hard one: `@bind`, cascading `BnTheme`, four event
handlers, `BnThemedPanel` chaining — if `.razor` can express BnDemo, it can express anything we
have), then `BnLayoutDemo`, `BnScrollDemo`, `BnImageDemo` (mostly declarative — bulk work).

## Decisions recorded now

1. **Typed `FontSize`/`Padding` ride along** (the M4-ledger stragglers; the milestone assigns
   them here). `string? FontSize` → `float? FontSize`, `string? Padding` → `float? Padding`,
   stringified invariantly like `Grow` (the 6.1 pattern; culture pinned by the existing tests).
   **The wire does not change** — `"24"`/`"16"` as before; every golden must hold. Pre-1.0
   breaking API change: recorded in the conclusion, no compat shim.
2. **`@key` reorders (PermutationList) stay unimplemented in 7.1** — none of the five pages
   reorders keyed items, and 7.2's `BnList` recycling is the real customer. The 7.0 `default:`
   arm keeps it a *loud* violation. Deferred with a named customer, not dropped.
3. **The supported `.razor` subset is documented** (a section in the conclusion, ledgered for a
   future analyzer): no `@page` (routes live in the registry), no `.razor.css`, no
   `MarkupString`/raw HTML (strict violation by design), no JS interop, `@key` insert/remove only.
   The load-bearing `@using Microsoft.AspNetCore.Components.Web` lives in **`_Imports.razor`**
   (created once; its absence makes `@bind` silently dead — the spike's recorded footgun).
4. **The shells are untouched.** Parity means the Kotlin/Swift frame tables and instrumented/XCTest
   suites pass without a single edit. If a shell test needs editing, the conversion is wrong.

## Verification (the GREEN bar)

1. All five pages authored in `.razor`; the hand-written versions and twin tests retired.
2. **Every pre-existing golden and frame table passes unchanged** (.NET, JVM, Android
   instrumented, iOS XCTest — the device lanes re-run green with zero shell edits).
3. `FontSize`/`Padding` typed, wire-identical, goldens untouched.
4. Publish gates hold: 4 IL2072 ×3 RIDs, 9 exports, clean-rebuild zero warnings, consumer smoke
   PASS (the `.razor` pages ship in the Components package).
5. Counts bumped + asserted; conclusion documents the supported subset + the authoring recipe.

## Risks

| Risk | Mitigation |
|---|---|
| A page's `.razor` form emits subtly different frames (whitespace, attribute order) | The twin test catches it record-for-record before the swap; the 7.0 markup-slot work already absorbed the whitespace class |
| Cascading values / component chaining compile differently from hand-written trees | BnDemo is conversion #2, deliberately — fail fast on the hard case |
| Typed params change the wire by accident | They stringify to the identical strings; the goldens are the tripwire |
| The Components package shape changes (`.razor` compile artifacts) | Consumer smoke is in the bar |
