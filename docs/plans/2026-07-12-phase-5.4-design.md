# Phase 5.4 Design — Clipboard + Share + the Bridge-Extension Pattern

**Date:** 2026-07-12
**Milestone:** M5 — P4: Full Platform Coverage (closes DoD #6)
**Predecessor:** Phase 5.3 GREEN (PR #50, `3996e9e`) — iOS at v3.0 parity.
**Owns:** the **documented, versioned bridge-extension pattern** (DoD #6's real
deliverable) — proven by adding clipboard + share as the first bridge growth now that
there are TWO shells to keep in lockstep.

---

## The decision (2026-07-12 brainstorm): size-negotiated bridge growth

The bridge is a fixed-size 6-callback struct copied whole at `sizeof` — no version
field (survey §1). Growing it 6→9 in lockstep is *safe today* (monorepo, drift tests),
but the milestone wants "a specified, extensible pattern," so we add the versioning
**now, while there are 3 struct sites**, via the **low-churn size-as-arg form**:

- **`blazornative_register_bridge(int32 structSize, BlazorNativeBridgeCallbacks*)`** —
  the export gains a leading `structSize`. The runtime copies
  `min(structSize, sizeof(its known struct))` bytes into its holder and **zero-fills the
  rest**. Existing 6 field offsets (0…40) are UNCHANGED — the 3 new callbacks append at
  48/56/64. (Export *count* stays 9; only `register_bridge`'s signature grows.)
- **Absent callback = capability unsupported.** A null slot (old shell that didn't set
  it, or a runtime older than the shell) means the managed side returns a graceful
  "not supported" (see §2), never a crash. Forward AND backward safe: old-shell(48)→
  new-runtime copies 48, clipboard/share null → unsupported; new-shell(72)→old-runtime
  copies 48, ignores the extra. This is the pattern all future host APIs (camera, geo…)
  follow.
- This is documented as `docs/bridge-extension.md` — DoD #6's "how a new host API joins
  the C-ABI: struct slot, size negotiation, per-platform impl, drift-test offsets."

## 1. The 3 new callbacks (append to the struct)

| Slot | Callback | Signature | Twin of |
|---|---|---|---|
| 48 | `ClipboardRead` | `int(char* buf, int cap)` — the -needed buffer protocol; empty → "" | `StorageRead`/`CurrentRoute` |
| 56 | `ClipboardWrite` | `int(const char* textUtf8)` | `Navigate`/`StorageDelete` (one-string) |
| 64 | `Share` | `int(const char* textUtf8)` — fire-and-forget (opens the share sheet) | `Navigate` (one-string) |

Struct 48→**72 bytes**. Each new callback reuses an existing `NativeShellBridge`
invoker shape (survey §3): ClipboardRead → `InvokeBufferProtocol`; ClipboardWrite/Share
→ `InvokeWithOneString`. Exception posture unchanged (host throw → -1 → `HostError`).

## 2. .NET surface (Gate 1)

- `IMobileBridge` (Core): `ValueTask<string> ClipboardReadAsync()`,
  `ValueTask ClipboardWriteAsync(string text)`, `ValueTask ShareAsync(string text)`.
- `NativeShellBridge`: implement each via the invoker; **the null-slot guard** — if the
  registered callback pointer is `IntPtr.Zero`, throw `NotSupportedException("clipboard/
  share not supported by this host")` (the forward-compat "unsupported" behavior that
  makes the size-negotiation real; a dedicated test drives a 48-byte registration and
  asserts the graceful unsupported).
- `DevHostBridge` (the dev-host mock): in-memory clipboard string; `ShareAsync` logs.
- **`ClipboardProbe`** (Runtime scaffolding, `HostEventProbe` shape): `[Inject]
  IMobileBridge`; a "Copy" `BnButton` → `ClipboardWriteAsync(state)`, a "Paste" button →
  `ClipboardReadAsync()` → echo `BnText`, a "Share" button → `ShareAsync(state)`.
  Registered "ClipboardProbe" in the mount registry; ledgered as scaffolding.

## 3. The lockstep struct grow (Gate 1 — .NET + JVM together)

The struct is mirrored on 3 sides + the drift tests pin it; ALL update together (survey
§2, §6):
- **.NET**: `BridgeProtocolNative.cs` struct (48→72, +3 fields); `register_bridge`
  signature (+structSize); `NativeShellBridge.Register` (min-copy + zero-fill); the
  `BridgeProtocolNativeTests` size (48→72) + the new offset assertions (48/56/64).
- **JVM**: `NativeBindings.kt` `@Structure.FieldOrder` mirror + the 3 new callback
  interfaces + the `register_bridge` JNA sig; `ShellBridge.kt` `ShellBridgeHandlers`
  interface gains `clipboardRead/Write` + `share`; **all 6 `ShellBridgeHandlers` impls**
  (AndroidShellBridge, InspectorHostBridge, RecordingHandlers, InertHost, InertBridge,
  ShellBridgeTest anon) add the methods; the `ShellBridgeTest` struct-size drift
  (48→72). The BridgeRegistrar builds + passes structSize.
- **Swift**: `BlazorNativeRuntimeC.h` (the 3 `bn_*_cb` typedefs + struct fields +
  `register_bridge` sig); `AppleShellBridge.swift` (3 trampolines); `BnRuntime` passes
  structSize.
- **.NET tests**: `NativeShellBridgeTests.FakeShellHost` gains the 3 fake callbacks +
  round-trip + the null-slot/unsupported assertions.
- **JVM test**: clipboard round-trip through the dll via an in-memory `ShellBridgeHandlers`.

## 4. Platform impls (Gates 2, 3)

- **Android** (`AndroidShellBridge.kt`, storage-impl template): clipboard →
  `ClipboardManager` (getSystemService, appContext); share → `Intent.ACTION_SEND` +
  `EXTRA_TEXT`, `FLAG_ACTIVITY_NEW_TASK` from appContext (respects the "no Activity
  retained" rule) — or a seam routing to MainActivity. Test bar: clipboard
  write-then-read-back against the REAL `ClipboardManager` (instrumented); share = the
  Intent built with the right EXTRA_TEXT/type (assert via a seam, NOT the system sheet).
- **iOS** (`AppleShellBridge.swift`, storage-stub template): clipboard →
  `UIPasteboard.general.string`; share → `UIActivityViewController` (main-thread hop +
  a presenting VC seam — the bridge is off-main/data-lane). Test bar: clipboard
  round-trip against REAL `UIPasteboard` (XCTest); share = the activity controller
  constructed with the content (assert via a seam, NOT the sheet).

## Risks

| Risk | Mitigation |
|---|---|
| Size-negotiation min-copy/zero-fill wrong → torn or over-read struct | The 48-byte-registration unsupported test + the drift tests pin it; `Buffer.MemoryCopy(min)` + explicit zero-fill; atomic holder swap unchanged |
| Stale mirror (one of 3 struct sides / 6 handler impls missed) | The .NET + JVM struct-size/offset drift tests fail loudly; the survey enumerates every site |
| Android share needs an Activity (retention rule) | FLAG_ACTIVITY_NEW_TASK from appContext, or a MainActivity seam; test asserts the Intent, not the sheet |
| iOS share off-main / needs a presenting VC | Main-thread hop + a presenter seam; test asserts the controller constructed, not the sheet |
| Share not end-to-end assertable | Honest bar = callback-fired-with-content (both platforms); documented |
