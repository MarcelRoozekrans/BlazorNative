## Verdict

**GREEN — DoD #2 CLOSED.** The sample app was built for `linux-bionic-arm64`, installed on a real
Xiaomi phone running **Android 16 (SDK 36)**, and **all four capabilities were exercised against
real hardware**: the system camera's sensor, a real enrolled fingerprint against the **OS
keystore**, a real fused/network location fix cross-checked against the platform's own record, and
a real notification posted, surviving a process kill, and tapped through to a **cold** start on the
right page. The generated deep-link route map resolved **all 13 routes** on hardware with no
crash, including the headline case: a **cold** `blazornative://notifications` mounted
`BnNotificationsDemo` — the route was resolved at Intent-parse time **before the .NET runtime
loaded**, on a device, exactly as Phase 11.0 claimed it would.

Two findings came out of it. **Finding A** is a real consumer-facing defect the emulator could
never have shown ([#178](https://github.com/MarcelRoozekrans/BlazorNative/issues/178) — the
documented `CaptureOptions` defaults never apply). **Finding B** is not a defect: it is a DoD
clause that asks the framework to stop doing something it does on purpose, and the honest
resolution is to **amend the clause**, not to fail the gate or fake a pass.

## The constraint that shaped the whole session — MIUI blocks input injection

**`adb shell input tap` / `input keyevent` produced no `MotionEvent` at all** in the app's process,
and the device could not even be woken with `keyevent 224` / `26`. MIUI gates ADB input injection
behind *USB debugging (Security settings)*, which is Mi-account gated. Only `am start`, `am kill`,
`dumpsys` and `screencap` worked.

**Consequence: every tap in this session was performed manually by the owner.** That is not a
degradation of the evidence — a human finger on a real screen is *stronger* proof than a synthetic
`MotionEvent` — but it is a hard fact about reproducibility, and the runbook now says so up front.
It also means the session could not be scripted, which is why the interaction smoke is partial
(below).

### B3 — Camera — PASS functionally, with two findings

**No permission prompt appeared, and that is correct.** `ACTION_IMAGE_CAPTURE` hands off to the
**system camera app**, which owns the sensor and its permission; the shell has no runtime `CAMERA`
branch on Android at all. This is exactly what the Phase 11.0 footgun audit documented and what
`AndroidShellBridge.kt:1145-1150` says in prose — confirmed on hardware.

App echo:

```
captured:3072x4096:93670
```

and the photo **rendered back into `BnImage`** — the full round trip: system camera → FileProvider
→ path across the C-ABI → .NET facade → native widget. File at
`cache/blazornative_captures/capture_1784705853828_1.jpg`, **93,670 bytes**.

JPEG structure analysis of the pulled file:

```
markers: JFIF, ICC_PROFILE, DQT ×2, SOF0, DHT …
EXIF present: False      JFIF present: True
DQT table 0 -> all 255   (min 255, max 255, mean 255.0)
DQT table 1 -> all 255
SOF0: 3072x4096, 3 components
```

#### Finding A — the documented `CaptureOptions` defaults never apply ([#178](https://github.com/MarcelRoozekrans/BlazorNative/issues/178))

Those all-255 quantisation tables are the tell. A JPEG whose DQT is saturated at 255 was encoded
at the **lowest possible quality**, and the visible result is posterisation.

The chain, all four links verified in this repo:

1. `src/BlazorNative.Device/Camera.cs:19` declares
   `CapturePhotoAsync(CaptureOptions options = default, …)`.
2. `default` **zero-initialises the struct**. `CaptureOptions` is a
   `readonly record struct` (`src/BlazorNative.Core/IMobileBridge.cs:384`) whose defaults —
   `MaxDimension = 2048, Quality = 85` — live on its **primary constructor**, and `default` does
   not run a constructor. So the call sends `Quality = 0, MaxDimension = 0`.
3. `src/BlazorNative.Jni/.../AndroidShellBridge.kt:1330` does
   `options.quality.coerceIn(1, 100)` → **1**.
4. `MaxDimension = 0` is a *documented opt-in* meaning "keep full resolution"
   (`IMobileBridge.cs:380-382`) — so no downscale happens and the long edge stays at **4096**.

The XML doc at `IMobileBridge.cs:376-383` promises "a ~2048 px long edge and JPEG quality 85 (a
couple hundred KB)". The device produced **4096 px at quality 1, 93 KB**. Every consumer calling
`CapturePhotoAsync()` without explicit options gets this — including the repo's own
`samples/BlazorNative.SampleApp/BnCameraDemo.cs:114`.

**This is exactly the class of bug a device session exists to find.** The DevHost bridge returns a
canned test image and never touches the encoder, so no amount of headless testing could surface it.

#### Finding B — EXIF is absent **by design**, so the DoD clause is unsatisfiable as written

`EXIF present: False` is **not** a failure and **not** evidence against a real sensor. The Android
shell deliberately **normalises orientation**: it bakes the EXIF rotation into the pixels *and
resets the tag to identity*, then re-encodes the JPEG, precisely so that EXIF-honouring decoders
(Coil on Android, Kingfisher on iOS) do not rotate the image a second time. That is stated in the
shell's own comments at `AndroidShellBridge.kt:1152-1156` and `:1300-1307`. A re-encoded JPEG
carries no camera EXIF at all.

So DoD #2's *"camera (real sensor + EXIF, no emulated shutter)"* clause asks the framework to stop
doing something it does on purpose. It **cannot be satisfied without a regression**, and the
correct disposition is to **amend the clause** — done in
[`docs/planning/MILESTONE.md`](../planning/MILESTONE.md) on this branch — not to mark it FAIL and
not to mark it pass on evidence that does not exist.

**What carries the real-sensor claim instead**, and it is not weak:

- **3072×4096** — a genuine 12.6 MP sensor resolution. No AVD shutter produces that.
- A **real photographed scene**, taken by the owner, recognisable on screen — not a synthetic
  green/checkerboard frame.
- Arrival via **`ACTION_IMAGE_CAPTURE`**, i.e. through the system camera app that owns the physical
  sensor, not through any in-app or emulated path.

### B4 — Biometrics + secure storage — PASS, a complete positive/negative pair

**Three distinct keystore challenges**, each demanding fresh authentication:

```
Set → status:Ok
  keystore2 get_auth_tokens(challenge=-8145652936997659278, authType=0x2, …)
  BiometricStatusRepositoryImpl: Succeeded(biometricSourceType=FINGERPRINT, isStrongBiometric=true, …)
Unlock + CANCEL → status:AuthFailed
  keystore2 get_auth_tokens(challenge=6398616799262488752, authType=0x2, …)
  Error(biometricSourceType=FINGERPRINT, errString=Vingerafdrukbewerking geannuleerd, errCode=5)
Unlock + FINGERPRINT → value:hunter2
  Succeeded(biometricSourceType=FINGERPRINT, isStrongBiometric=true, …)
  keystore2 get_auth_tokens(challenge=-5562603365139634622, authType=0x2, …)
```

> **One label in the raw session log was wrong, and resolving it against the code changed what
> this gate claims — so it is written down rather than quietly fixed.** The log rendered the first
> line as `Authenticate → status:Ok`. That cannot be an `Authenticate` result: `BiometricStatus`
> has **no `Ok` member** (`IMobileBridge.cs:288-296` — `Authenticated | Failed | Cancelled |
> Unavailable | LockedOut | Error`), and `BnSecureDemo.AuthenticateAsync` echoes
> `status:{BiometricStatus}`, which would render **`status:Authenticated`**. The handler that
> *does* render `status:Ok` is **`SetAsync`**, which echoes `status:{SecureStorageStatus}`. So the
> observed line was **`Set`**, and the `Succeeded(… isStrongBiometric=true)` beneath it was
> **`Set`'s own authentication** — `SetAsync` passes `requireAuth: true`, so creating the
> auth-bound key is itself biometric-gated and raises the prompt.
>
> **Therefore: the standalone `Authenticate` button was never separately observed in this
> session.** What the evidence shows is **Set (auth-bound write, biometric-gated) → Unlock+cancel
> → Unlock+fingerprint** — three operations, three challenges, which is exactly the three
> `keystore2` lines above. The gate is unaffected and arguably stronger for it: every one of those
> three is the **OS keystore** acting, not an app-level status echo. `IBiometrics.AuthenticateAsync`
> is carried as NOT EXERCISED below.

**Why this is the strongest of the four results.** The three `keystore2 get_auth_tokens` lines carry
**three different challenge values** — meaning the OS issued a fresh challenge per operation and
would not accept a stale token. And the pair that matters is `Unlock + CANCEL → AuthFailed` against
`Unlock + FINGERPRINT → value:hunter2`: **the OS refused to decrypt the auth-bound key without
fresh Class-3 (`BIOMETRIC_STRONG`) authentication, and permitted it with.** That is the key binding
being enforced by Android, not the app choosing what to display. `isStrongBiometric=true` confirms
the Class-3 path.

**This exceeds the runbook's plan.** The runbook expected to have to settle for *"the device
reports a hardware-backed keystore"* plus an AVD negative control, because nothing in the repo
surfaces `KeyInfo.isInsideSecureHardware()` / `getSecurityLevel()`. The session got a real
positive/negative pair **on a device reporting `android.hardware.strongbox_keystore=300`**, which
is a genuine end-to-end enforcement proof rather than a device self-report. The over-claim guard
still applies and is stated plainly below.

## What this proves — and what it does not

Stated in full, because a device proof that over-claims is worse than no device proof.

### Proven on real hardware

- The NativeAOT arm64 runtime **boots and mounts** on Android 16 / SDK 36 — a newer OS than any CI
  lane covers.
- The **generated deep-link route map** resolves at Intent-parse time, before the .NET runtime
  loads, for **all 13 routes**, cold and warm.
- The **touch → JNI → .NET → re-render → patch → widget** loop closes on a physical screen.
- A **real GPS/fused fix**, bit-identical to the platform's own record.
- A **real notification**, a **real runtime permission grant** (`false` → `true`), survival across
  a process kill, and a **cold tap-through** to the correct page in a new process.
- A **real system-camera capture** at sensor resolution, round-tripped into a native image widget.
- The **OS keystore enforcing** auth-bound decryption — refused on cancel, permitted on
  fingerprint, across three fresh challenges, on a StrongBox-reporting device.

### NOT proven — the honest residue

- **Warm notification tap-through was NOT exercised.** What *was* proven on the same hardware is
  the warm re-route path itself (`blazornative://geolocation` re-routed the live app), and the
  notification's `contentIntent` **is** that same deep link — so the untested composition is
  narrow: the shade delivering an already-running process an Intent. Narrow is not zero. It is a
  residual, carried below.
- **The location permission prompt was NOT exercised** — `ACCESS_FINE_LOCATION` /
  `ACCESS_COARSE_LOCATION` were already granted on this device. The *notification* runtime prompt
  was genuinely exercised, so the grant mechanism is not wholly untested, but the location
  grant-flow specifically is unproven on hardware.
- **Interaction smoke is PARTIAL** — one text field driven; list/scroll/form/modal mounted only.
- **`IBiometrics.AuthenticateAsync` — the standalone `Authenticate` path — was NOT EXERCISED.**
  The session's first logged line was mislabelled `Authenticate` and is in fact `Set` (see the note
  under B4). The **biometric gate itself is thoroughly proven** — three OS-issued challenges, a
  cancel and a success, all through the secure-storage path — so what is untested is narrow: the
  façade call that performs a *bare* authentication with no keystore operation behind it. Also
  untested for the same reason: `Authenticate` + **Cancel** → `status:Cancelled` (the runbook's
  B2.2), though the equivalent cancel-as-data behaviour *was* observed on the `Unlock` path
  (`status:AuthFailed`, `errCode=5`).
- **The key's security level was not read from the app.** The app does not surface
  `KeyInfo.getSecurityLevel()`. *"StrongBox / TEE-backed"* remains a **device report**
  (`android.hardware.strongbox_keystore=300`); what the app proved end-to-end is the **enforcement**
  (refuse-on-cancel / permit-on-finger), which is the claim that actually matters.
- **Geolocation accuracy is still not surfaced by the app** ([#169](https://github.com/MarcelRoozekrans/BlazorNative/issues/169)).
- **One device, one OS version, one OEM skin.** Xiaomi/MIUI on Android 16. Nothing here is a
  cross-device warranty; two of the session's four biggest obstacles (install restriction, input
  injection) were MIUI-specific.
- **iOS real-device remains deferred** — no Apple Developer account. Unchanged since M5.
- **None of this is CI-asserted**, by construction. The phone is not a CI dependency; this document
  is the artifact.
- **Screenshots and logcat dumps live in the device session's scratchpad, outside the repo.** They
  were not copied into `docs/plans/evidence/`, and that directory does not exist. The verbatim
  excerpts quoted above are the in-repo record — no file paths into the repo are claimed for them.

  field evidence** for the logging-discipline issues. Phase 11.4's case rests where it did.

## Conclusion

The emulator honesty gap is closed. For ten milestones the camera and the keystore were the two
capabilities whose green ticks rested on an AVD pretending — and the first time a real sensor and a
real fingerprint were put behind them, one produced a genuine consumer-facing defect (#178) and the
other produced a *stronger* proof than the design dared plan for (OS-enforced auth-bound
decryption, three fresh challenges, on StrongBox hardware).

The session also did the thing a good honesty check is supposed to do to its own paperwork: it
found a **DoD clause that was wrong**. "Real sensor + EXIF" sounded rigorous when it was written
and turned out to demand that the shell abandon a deliberate correctness behaviour. Amending it —
with the reason written down and the replacement evidence named — is the only disposition that
leaves the record true.

What remains unproven is small, named, and written above rather than rounded off: one warm tap, one
permission dialog, four demo pages mounted but not driven. On one phone, one OEM skin, one OS
version. That is the shape of an honest device proof.
