# Phase 8.6 Design — release automation: 8.2 set a three-part test for this, and this phase passes two of it

**Status:** design (2026-07-17). Branch `phase-8.6-release-automation`, cut from `a3fe234`
(main's real tip — PR #107, the M8 final audit, merged after this branch was first named).

**What this phase is:** the owner has asked for *"as much automation as possible"* in the
release story and answered three scoping questions. Two of them **reverse decisions M8
shipped and its final audit passed**. This design records the reversal honestly, and it is
the last thing this repo does under the old commit convention — it designs the new one; it
does not get to use it.

---

## The one sentence the whole phase hangs on

> **release-please computes the version and writes the changelog, but a DRAFT Release fires
> no workflow — so every arrow this phase automates stops one click short of nuget.org,
> which is exactly where M8 DoD #3's law already put the owner's hand.**

The law does not bend. `release.yml` publishes on `release: types: [published]`; release-please
cuts the Release as a **draft**; GitHub does not fire `published` for a draft. The owner's
click is what converts the draft into the event. **Everything before the click becomes
automatic; the click itself is untouched.**

---

## 8.2 wrote the test this phase has to pass, and it names its own failure

This is not a phase reversing an unexamined decision. **8.2 decision 3 rejected release-please
and enumerated, in advance, exactly what would change its verdict** — "named, so it is a
decision and not a grudge. **All three must hold, not any**":

| 3. "draft-Release mode is confirmed to preserve manual-go" | ✅ **holds — confirmed this phase** | `draft: true` in the live schema + GitHub's documented "Workflows are **not** triggered for the `created`, `edited`, or `deleted` activity types for draft releases." **Two of three. 8.2 said all three.** So the honest sentence is:

> **This phase does not pass 8.2's test. The owner waives precondition 2, and that waiver is
> an owner's valuation, not a fact this design discovered.**

8.2 and 8.6 do not disagree about a single fact. 8.2 itself named the exact configuration
this phase adopts — *"It can be configured to cut a **draft** Release that the owner publishes
by hand — manual-go preserved — but at that setting its remaining value is changelog + a bump
PR"* — and judged that remaining value **insufficient**. The owner now judges it **sufficient**.
**That is a re-weighting, and the owner owns it.** Recording it as anything else — as a
discovery, as new information, as 8.2 having been wrong — would be flattery.

⚠ **Veto-flagged and stated plainly:** if the owner wants the three-part test honoured as
written, the veto is to wait until a Release has actually been published by hand a few times
and precondition 2 becomes a fact. The cost of proceeding now is that this repo adopts a tool
whose headline value (relieving a chore) relieves a chore **nobody has ever performed**. The
cost of waiting is that the first few Releases are hand-written — which is what M8 designed
for and documented in `docs/GITHUB-SETUP.md`.

### One thing 8.2 got measurably wrong, and this phase is where it is corrected

8.2 decision 3, reason 2, claimed: *"on a `1.0.0-preview.N` line, release-please's prerelease
handling would want to walk patches (`1.0.1`, `1.0.2`) against a version the owner hand-decides
at milestone cadence."*

**Measured against the actual source (`versioning-strategies/prerelease.ts`), that is false.**
With `versioning: prerelease` it walks the **prerelease counter**, not patches:
`1.0.0-preview.1 → 1.0.0-preview.2`. 8.2 assumed the behaviour; decision 2 below traces it.
The rejection did not rest on this reason, so the verdict was not built on the error — but the
reason was shipped in a design doc and is wrong, and this is the note that says so.

## The reality as it actually is (verified this phase, not recalled)

### release-please's schema — read from the live schema, not assumed

`https://raw.githubusercontent.com/googleapis/release-please/main/schemas/config.json`:

| Option | The schema's own words | Matters because |
|---|---|---|
| `draft` | "Create the GitHub release in **draft** mode. Defaults to `false`." | **The whole phase.** This is the option that preserves the owner's click. |
| `prerelease` | "Create the GitHub release as prerelease. Defaults to `false`." | **Overloaded — see below.** It feeds the GH Release flag *and* the versioning strategy. |
| `prerelease-type` | "…if prerelease strategy used and type set, will set the prerelease part of the version…" | Only bites when `versioning: prerelease`. |
| `versioning` | "Versioning strategy. Defaults to `default`" | Valid values (from `factories/versioning-strategy-factory.ts`, not from docs): `default`, `always-bump-patch`, `always-bump-minor`, `always-bump-major`, `service-pack`, **`prerelease`**. |
| `extra-files` | generic (annotation) / `json`+jsonpath / `xml`+xpath / `toml` / `yaml` / `pom` | The version-mirror mechanism (decision 3). |
| `bootstrap-sha` | root-level | Draws the line under 580 non-conventional commits (decision 1). |

**The `prerelease` overload, named because it is genuinely confusing:** the schema describes
`prerelease` as a *GitHub Release flag*, but `manifest.ts:1397-1408` passes the same key into
the **versioning strategy** (`prerelease: config.prerelease`). One key, two jobs. And the
Release flag is conditioned:

```ts
prerelease: config.prerelease &&
  (!!release.tag.version.preRelease || release.tag.version.major === 0)
```

→ **self-correcting**: at `1.0.0` final, with `prerelease: true` still in the config, the
Release is *not* flagged prerelease. One less thing to remember at graduation.

### The prerelease ladder — traced through the source, not guessed

`isPreMajor` is **`major < 1`** (`version.ts:86-88`). Our version is `1.0.0-preview.1` →
major = 1 → **`isPreMajor` is FALSE**. Therefore:

> ⚠ **AdoNet.Async's `bump-minor-pre-major` and `bump-patch-for-minor-pre-major` are DEAD
> OPTIONS for us — and for them.** Both are gated on `isPreMajor`. At `1.0.0-preview.1` (major 1)
> and at their `1.3.3` (major 1) neither can ever fire. **Mirroring the reference config
> verbatim would import two settings that do nothing.** We omit them and say why.

Tracing `PrereleaseVersioningStrategy.determineReleaseType` from `1.0.0-preview.1` with
`prerelease: true`, `prerelease-type: preview`:

| Commit | Path through the source | Result |
|---|---|---|
| `fix:` | `PrereleasePatchVersionUpdate` → `preRelease` present → `bumpPrerelease("preview.1")` | **`1.0.0-preview.2`** |
| `feat:` | `isPreMajor` false → `PrereleaseMinorVersionUpdate` → `preRelease` present **&& `patch === 0`** → `bumpPrerelease` | **`1.0.0-preview.2`** |
| `feat!:` / `BREAKING CHANGE:` | `isPreMajor` false → `PrereleaseMajorVersionUpdate` → `preRelease` present **&& `patch === 0` && `minor === 0`** → `bumpPrerelease` | **`1.0.0-preview.2`** |

> ### **THE FINDING: all three produce the same version.**
>
> While we sit at `1.0.0-preview.N` with `minor === 0` and `patch === 0`, **the commit type
> does not move the version at all.** `feat`, `fix` and a breaking change are
> version-identical. **Conventional commits do not compute our version — they compute our
> changelog.** The version is a **counter**.

This is the single most important fact for setting expectations: a contributor who writes
`feat!:` expecting a major bump gets `preview.N+1`, and that is correct behaviour, not a
misconfiguration. It also means **the blast radius of a mis-typed subject is a changelog
section, never a wrong version** — which is what makes the commitlint gate safely advisory
(decision 5).

`bumpPrerelease` uses `PRERELEASE_NUMBER = /(?<number>\d+)(?=\D*$)/` and pads to the original
number's width. We start at `.1` (width 1), so `preview.9 → preview.10` — no zero-padding trap.

### GitHub's release event — quoted, because the law rests on it

From GitHub's Actions "Events that trigger workflows" → `release`, verbatim:

> **"Workflows are not triggered for the `created`, `edited`, or `deleted` activity types for
> draft releases."**

> **"The `prereleased` type will not trigger for pre-releases published from draft releases,
> but the `published` type will trigger. If you want a workflow to run when stable *and*
> pre-releases publish, subscribe to `published` instead of `released` and `prereleased`."**

**Two things fall out, and the second is a gift 8.2 did not know it was giving:**

1. **A draft Release fires nothing.** release-please creating the draft is silent. The owner's
   click is what fires `published`. **The law holds structurally, not by promise.**
2. **`release: types: [published]` — already shipped by 8.2 — is the *only* trigger that works
   here.** Our packages are pre-releases (`1.0.0-preview.N`), published *from a draft*. GitHub
   documents that `prereleased` **will not fire** for exactly that path. Had 8.2 chosen
   `released` or `prereleased`, this phase would have had to change it. It chose `published`
   for its own reasons and landed on the one correct answer. **Verified, not assumed —
   `release.yml:100-101` reads `release: types: [published]` today.**

### The merge settings — measured, and one of them breaks the contract

```
{"merge":true,"rebase":true,"squash":true,
 "squash_title":"COMMIT_OR_PR_TITLE","squash_msg":"COMMIT_MESSAGES"}
```

> ### ⚠ **`squash_title` is `COMMIT_OR_PR_TITLE`, NOT `PR_TITLE`.**
>
> GitHub's `COMMIT_OR_PR_TITLE` means: **if the PR has exactly one commit, that commit's
> subject becomes main's subject — the PR title is ignored.** Only multi-commit PRs use the
> title.
>
> **A PR-title lint is therefore NOT sufficient on this repo as configured.** A single-commit
> PR titled `feat(runtime): …` whose commit says `wip` lands `wip` on main — passing the lint
> and arriving at release-please as an unparseable subject. **This is the quiet failure that
> would make the whole commit contract decorative.** Decision 5 fixes it with a repo setting.

## Decisions (⚠ marks the veto-flagged judgment calls)

### 1. release-please is **IN**, in draft mode, with `bootstrap-sha` drawing a line under history ⚠

*The call to veto: adopting the tool at all, against 8.2's own three-part test (two of three).
Argued at the top of this document; not repeated here.*

**The config, and every value is justified:**

```jsonc
{
  "$schema": "https://raw.githubusercontent.com/googleapis/release-please/main/schemas/config.json",
  "bootstrap-sha": "<main's tip at the 8.6 implementation merge>",
  "packages": {
    ".": {
      "release-type": "simple",          // no dotnet type exists; simple = changelog + extra-files
      "versioning": "prerelease",        // the counter ladder (decision 2)
      "prerelease": true,                // BOTH the GH flag AND the strategy — the overload
      "prerelease-type": "preview",      // only consulted when the version has no prerelease part
      "draft": true,                     // ← THE LAW. The owner's click is the go.
      "extra-files": [ /* decision 3 */ ]
    }
  }
}
```

Omitted deliberately: **`bump-minor-pre-major` / `bump-patch-for-minor-pre-major`** — dead at
major ≥ 1 (`isPreMajor = major < 1`). The reference sets both; here they would be config that
cannot fire, and config that cannot fire is a comment pretending to be a decision.

**`bootstrap-sha` is the answer to 8.1's third rejection reason** ("conventional-commit
inference mis-bumps phase-structured history"). It draws a line: release-please **never reads
a single pre-8.6 commit**. Without it, release-please searches history for a last release,
finds none (zero Releases, and no `v<semver>` tag will exist), and walks all 580 commits — of
which the only conventional ones are Renovate's and `a3fe234`. Harmless but wrong-shaped;
`bootstrap-sha` makes it *correct*, not merely survivable.

**`release-please.yml`: one job, no publish, no key.** The workflow is the action and nothing
else — `permissions: contents: write, pull-requests: write`, `on: push: branches: [main]`.
**No `publish` job.** See the reference-mirror warning above; this is the one place the mirror
is refused outright.

### 2. The prerelease ladder: the version is a **counter**, and reaching `1.0.0` needs a commit **and a config change** ⚠

*The call to veto: `versioning: prerelease` (the counter) vs `versioning: default` (the
reference's implicit setting) vs `always-bump-patch`.*

`versioning: default` is what mirroring AdoNet.Async verbatim gives you. **Traced through
`versioning-strategy.ts`, the default updaters PRESERVE the prerelease component verbatim:**

| From `1.0.0-preview.1`, `versioning: default` | Produces |
|---|---|
| `fix:` → `PatchVersionUpdate` (keeps `version.preRelease`) | **`1.0.1-preview.1`** |
| `feat:` → `MinorVersionUpdate` (keeps `version.preRelease`) | **`1.1.0-preview.1`** |
| `feat!:` → `MajorVersionUpdate` (keeps `version.preRelease`) | **`2.0.0-preview.1`** |

**`1.1.0-preview.1` is a nonsense version** — the `-preview.1` is glued on forever and never
increments. **This is the trap, and it is what "mirror the reference" produces.** Rejected.

**Adopted: `versioning: prerelease` + `prerelease: true` + `prerelease-type: preview`.**

**THE LADDER, stated as a ladder:**

| Rung | From | Action | To |
|---|---|---|---|
| 1 | `1.0.0-preview.1` | **any** conventional commit — `fix:`, `feat:`, or `feat!:` | `1.0.0-preview.2` |
| 2 | `1.0.0-preview.N` | any conventional commit | `1.0.0-preview.N+1` |
| 3 | `1.0.0-preview.N` | a commit whose **body** carries `Release-As: 1.0.0` | **`1.0.0`** |
| 4 | `1.0.0` | **a config change** — drop `versioning`/`prerelease`/`prerelease-type` → back to `versioning: default` | — |
| 5 | `1.0.0` (post-config-change) | `fix:` / `feat:` / `feat!:` | `1.0.1` / `1.1.0` / `2.0.0` |

**Rung 3 is verified in the source, and it short-circuits everything:** `determineReleaseType`
scans `commit.notes` for a `RELEASE AS` note **first**, before any breaking/feature counting,
and returns `new CustomVersionUpdate(Version.parse(releaseAs.text))`. It works *inside* the
prerelease strategy. Graduation is a commit, exactly as the brief suspected.

> ### ⚠ **RUNG 4 IS NOT OPTIONAL, AND IT IS THE TRAP IN THE LADDER.**
>
> Reaching `1.0.0` with the prerelease config still in place, the **next** `fix:` computes:
> `PrereleasePatchVersionUpdate` → `version.preRelease` is now **empty** → the `else` branch →
> `new Version(major, minor, patch + 1, this.prereleaseType)` → **`1.0.1-preview`**.
>
> **Graduating to 1.0.0 and stopping there silently re-enters preview on the next commit.**
> Rung 4 must land in the same PR as the `Release-As:` commit, or immediately after and before
> any other merge. **Named here because nothing in the tooling will warn about it** — and
> because the classifier's tag↔props assertion (decision 4) would *not* catch it either: the
> tag and the props would agree on `1.0.1-preview`. The only guard is this paragraph.
>
> *(A subtler alternative exists: keep `versioning: prerelease` and set `prerelease: false` —
> the strategy's tail then strips the prerelease part (`if (!this.prerelease)` →
> `CustomVersionUpdate("${major}.${minor}.${patch}")`). It reaches the same place. Rung 4 as
> written is simpler and is what M9+ should do.)*

### 3. The version authority: **one AUTHOR, six pinned MIRRORS** — and the pins now audit a machine ⚠

*The call to veto: (i) the manifest authors and the props mirrors vs (ii) the props stays the
author and release-please is told to read it (impossible — release-please's manifest is not
optional) vs (iii) drop the props entirely and let the build read the manifest (rejected: the
manifest is JSON with no MSBuild meaning; every csproj would need a custom target, and the
build would gain a dependency on a release tool).*

> ### **8.6 NORMATIVE RULE — the version's authority (supersedes 8.1's "ONE literal")**
>
> **`.release-please-manifest.json` is the version's AUTHOR** — the only file into which a
> version is ever *decided*. **`src/Directory.Build.props`'s `<Version>` is its FIRST MIRROR
> and remains the build's single source of truth** — nothing in the build, the pack, the
> smoke, or the classifier ever reads the manifest. **Every other version literal mirrors the
> props.** **No version literal is ever hand-edited again**; release-please writes them all
> via `extra-files`, and **the pins assert agreement on release-please's own PR.**

**8.1's rule was "ONE literal". The true rule is "ONE AUTHOR, N PINNED MIRRORS" — and N is 6.**

**The mechanism, per file. One updater kind — the Generic (annotation) updater — everywhere:**

| File | Literals | Annotation |
|---|---|---|
| `src/Directory.Build.props` | 1 | `<Version>1.0.0-preview.1</Version> <!-- x-release-please-version -->` |
| `templates/…/BlazorNative.Templates.csproj` | 1 | same |
| `templates/…/MyBlazorNativeApp.csproj` | 3 | one per `BlazorNative.*` `PackageReference` line |
| `templates/…/.template.config/template.json` | 2 | `// x-release-please-version` on the `defaultValue` and `replaces` lines |

**Why Generic and not `xml`+xpath, which the schema also offers:** `BaseXml.updateContent`
does a full `DOMParser` → `XMLSerializer` **round-trip of the whole document**.
`src/Directory.Build.props` is a heavily-commented, carefully-formatted file that *is* the
repo's version truth; handing it to a third-party serializer to be re-emitted on every release
is a formatting risk taken for no gain. **The Generic updater is a line-scoped regex replace —
it touches the annotated line and nothing else.** Same reasoning for `template.json` vs
`json`+jsonpath (`JSON.parse`/`stringify` would reformat it).

⚠ **Flagged:** the `template.json` annotations introduce **two `//` comments into a file that
today has zero** (measured: `grep -c "^\s*//"` → `0`). This is safe — `dotnet new` supports
JSONC officially, and `TemplateDriftTests` already parses it with
`JsonCommentHandling.Skip` + `AllowTrailingCommas`, which is direct evidence the repo already
treats it as JSONC — but it is a real change to a shipped file and the veto (accept the
`json`+jsonpath reformat instead) is cheap.

**An `xml`+xpath option worth recording even though it is not taken:** `GenericXml` runs
`xpath.select(…)` and sets `node.textContent`, which reaches **attribute** nodes. So a single
entry — `//PackageReference[starts-with(@Include,'BlazorNative.')]/@Version` — would cover all
three template references **without a roster**, honouring 8.1 normative rule 2 automatically,
and would keep covering a fourth reference nobody remembered to annotate. **That is genuinely
better on drift and worse on formatting.** It is the strongest veto candidate in this decision.
The annotation approach's answer to the same drift is *the pin*, below.

> ### **THE PART THAT MATTERS: the pins bite HARDER than before, and they bite on the machine.**
>
> The existing pins **enumerate** rather than roster (8.1's I-2 finding, and the reason the
> roster was deleted). So:
>
> - Add a fourth `BlazorNative.*` `PackageReference` to the template and forget its annotation
>   → release-please leaves it at the old version → **`TemplateDriftTests` (a) reds**, because
>   it enumerates *every* `BlazorNative.*` reference and asserts each equals the props.
> - release-please's `extra-files` misses the props entirely → props stays behind the manifest
>   → the mirrors still agree with the props → **the drift pins stay GREEN** (they compare
>   mirrors *to the props*, not the props to the manifest).
>
> **That second case is a real gap, and it is new.** The manifest↔props arrow is the one arrow
> no existing pin watches. Two independent guards close it — the 8.2 shape/content pattern,
> applied to the automation:
>
> 1. **A NEW PIN — `PackageVersionPinTests.TheManifest_AgreesWithTheProps`**: parse `"."` out
>    of `.release-please-manifest.json`, assert it equals the props' `<Version>`. **Sound as an
>    always-true invariant on main**: release-please writes the manifest *and* the props in the
>    same release PR, so they agree at every commit on main — between releases both sit at the
>    last released version. **This runs in the required `build-test` lane, so it reds

### 5. The commit contract: **the PR title IS the commit subject** — and one repo setting has to change first ⚠

*The call to veto: (i) lint the PR title only (adopted) vs (ii) lint branch commits too vs (iii) both.*

**The repo squash-merges. Under squash, the branch's commit subjects are DISCARDED** — they
never reach main, never reach release-please. Linting them would enforce a contract on text
that is thrown away and would red PRs for honest WIP commits. **(ii) and (iii) are rejected on
that ground.**

But **(i) is only sound if the PR title actually wins**, and today it does not:

> ### ⚠ **OWNER ACTION, AND THE CONTRACT DOES NOT HOLD WITHOUT IT**
>
> **`squash_merge_commit_title` must change from `COMMIT_OR_PR_TITLE` to `PR_TITLE`.**
>
> As configured, a **single-commit PR** uses the *commit's* subject and ignores the title
> entirely. The lint would pass on a good title while `wip` lands on main. This is a
> repository setting (Settings → General → Pull Requests → *Default commit message*), and it
> is the **first** step of the implementation sequence — before the lint exists, so the lint is
> never sound-looking-but-not-sound.
>
> `squash_merge_commit_message` **stays `COMMIT_MESSAGES`** — the branch commits' WHY bodies
> concatenate into main's body, which is exactly where this phase sends the prose (below), and
> which is where release-please reads `Release-As:` and `BREAKING CHANGE:` footers from.

> ### **8.6 NORMATIVE RULE — the commit contract**
>
> **The PR title is the commit subject.** It is the only text that becomes a subject on main,
> the only text release-please ever reads, and the only text a consumer's changelog is
> rendered from. It is **conventional**: `type(scope): subject`, **header ≤ 100 chars**
> (mirroring `.commitlintrc.yml`'s `header-max-length: 100` exactly, per the owner).
>
> **The house's finding-in-the-headline style MOVES TO THE BODY.** `Phase 8.4 — the docs site:
> the generator said "10 succeeded, 0 failed" and shipped zero components` stops being a
> subject and becomes the **first line of the PR description / commit body**. The subject
> becomes `feat(docs): the Docusaurus docs site`.
>
> **The ~40 phase conclusions in `docs/plans` remain the real record.** The generated
> CHANGELOG.md is a consumer's index, not this repo's memory — 8.2 decision 3's reason 3 was
> *right about this and stays right*: "the real changelog material is the phase conclusions and
> milestone audits in `docs/plans/`, which are written for humans and which no generator can
> produce." **Nothing about that changes. The changelog is added beside the record, not
> instead of it.**

**Enforcement — `.commitlintrc.yml` is the ONE authority.** A `commitlint.yml` workflow on
`pull_request: [opened, edited, synchronize, reopened]` pipes **the PR title** into commitlint
using the same `.commitlintrc.yml`. **Deliberately not `amannn/action-semantic-pull-request`**:
its `types:`/`scopes:` inputs would be a **second copy of the enums** — 8.1 normative rule 2's
exact sin, and the one this repo deleted a roster to avoid.

**ADVISORY, not a fourth required gate** — following 8.2's own precedent for `release.yml`
("ADVISORY BY DESIGN — this is NOT a fourth required check"). **The three required gates keep
their names and contexts: `build-test`, `android-build`, `ios-build`.** The honest consequence,
stated rather than hidden: **a mis-titled PR can merge, and release-please will silently ignore
it** — no changelog line, no bump contribution. **That failure is a missing changelog entry,
never a wrong publish** (decision 2: the type cannot move the version at `1.0.0-preview.N`
anyway). Promoting it to required is the owner's call and a branch-protection change; it is
**not** taken here.

### 6. The scope-enum: consumer-facing, and **`phase-N` is exactly what it excludes** ⚠

*The call to veto: packages vs areas vs both.*

The reference's scope-enum is **areas** (`core`, `dataset`, `adapters`, `benchmarks`,
`validation`, `ci`, `deps`). **Ours cannot be a copy — their areas are not our areas.** And
the brief is right that this is the interesting one: **a consumer's changelog wants
`components`, not `phase-8.4`.** `fix(phase-7.5): the dispatch table's ROW ORDER is normative`
means everything in this repo and nothing on nuget.org.
