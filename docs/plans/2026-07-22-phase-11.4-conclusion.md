## Verdict

**GREEN — DoD #6 CLOSED.** Before this phase the framework's own diagnostics reached **nobody on
either shipping platform**: Android sends process stderr to `/dev/null` (the repo says so in its own
comments and *depended* on it), and iOS emitted the opposite failure — 78 unconditional, always-public
`NSLog` sites that shipped in Release. There was exactly one throttle in the entire framework, nine
characters long and not configurable. After 11.4 there is **one seam, one threshold, one line format,
and a real destination on both platforms** — plus the half of #164 that a log alone could never fix:
a parameter-binding fault now **aborts the mount** instead of returning `rc = 0` over a half-rendered
screen.

The headline in one sentence: **the runtime's diagnostics are now visible on both platforms, and a
binding fault no longer reports success.**

## As-builts

### Gate A — one seam, and the field that cost nothing

`BnLog` lives in `BlazorNative.Core` because it must be reachable from Core, Renderer **and**
Runtime — three assemblies, so `internal` was not expressible without an `InternalsVisibleTo` web.
It is public, and the file argues that is a *second* benefit rather than a concession: raising the
level is a thing a consumer needs to do.

**The default is `Warn`, and deliberately not `#if DEBUG`** (`BnLog.cs:78`). A build-configuration
switch cannot be opened by a consumer already shipping a Release build who needs one verbose
session, and it makes the two configurations' code paths differ — which is how a logging bug
survives to production.

**The level rides `BlazorNativeInitOptions` at offset 28, and `SizeOf` is still 32.** That struct is
explicitly *not* the frozen 80-byte callbacks bridge; it had already grown once, in M10. The new
`int` lands in the tail padding that alignment-8 had already reserved. The pin was made **stronger,
not merely edited** (`NativeShellBridgeTests.cs:687`, `:707`): it asserts `SizeOf == 32` **and**
`OffsetOf(LogLevel) == 28`, and the comment at `:696`–`:701` states why neither alone suffices —
size alone would still hold if the field had displaced an existing offset; offset alone would still
hold if the struct had grown to 40. **Only the pair distinguishes "free" from "smuggled in at a
cost", and it was measured before/after rather than deduced from the alignment rule.**

The `Console.Error` census was **exactly 31** at Gate A's parent (`b329851^`): `Exports.cs` 15,
`NativeRenderer.cs` 7, `NativeShellBridge.cs` 3, `HostSession.cs` 2, `RendererServices.cs` 2,
`NativeNavigationManager.cs` 1, `DevHostBridge.cs` 1. All 31 migrated; today the only `Console.Error`
in `src/**/*.cs` is `BnLog`'s own default sink. `RendererServices`' hard-coded
`logLevel >= LogLevel.Warning` now delegates to the seam, which is what makes Blazor's own `ILogger`
and the framework's diagnostics share **one** throttle.

`ConsoleErrorDriftTests` pins it, and — the part worth copying — **asserts its own non-vacuity**: the
scan must find >20 files, must contain `BnLog.cs` / `Exports.cs` / `DevHostBridge.cs`, and the regex
must still match in `BnLog.cs`, the one file where a match is known to exist. A pin whose every
assertion is "found nothing" is indistinguishable from a broken scan without that counterweight. The
`DevHostBridge.cs` exemption is asserted in **both** directions: its 18 `Console.Write*` sites must
still be there (or the exemption is stale prose), and its `Console.Error` must not have crept back —
the stdout exemption buys no stderr exemption.

New public API was declared into `PublicAPI.Unshipped.txt` — **two files**, `BlazorNative.Core`
(27 lines, the whole `BnLog`/`BnLogLevel` surface) and `BlazorNative.Runtime`
(`BlazorNativeInitOptions.LogLevel -> int`). Phase 11.3 Gate B's baseline red-flagged it, on its
first live encounter, which is exactly the workflow that baseline exists to enforce.

### Gate B — the transport nobody had built

Design §1.3 established that **nothing in the repo redirected stderr**, so Option 1 was genuinely
unbuilt rather than merely unused. It turned out to be ~150 lines of pure Kotlin in `androidMain`:
`android.system.Os.pipe()` and `Os.dup2(FileDescriptor, int)` are both API 21 and the module floor
is `minSdk 24`, so no NDK and no JNI.

**The install ordering is load-bearing.** `BnStderrLogcatPump.install()` is the first statement of
`MainActivity.onCreate` (`MainActivity.kt:182`) — before SoLoader, before the `dlopen` that
`NativeBindings.INSTANCE` triggers, long before the boot thread calls `runtime.start()`. That is
what catches `blazornative_init`'s own failure path: the one place the framework deliberately emits a
full `ex.ToString()`, because for the real NativeAOT trim failures (`TypeLoadException`,
`MissingMethodException`) the `Message` alone hides the offending type. Catching that line is one of
the two reasons this transport beat routing logs through the bridge.

**R1 was the phase's sharpest risk and it was pinned in both directions.** The line format
`[BN|<E|W|I|D|V>|<category>] <message>` is two constants in two languages, and a one-character drift
would silently downgrade every framework line to the unprefixed `Log.w` fallback **with nothing
looking broken**. `BnStderrPumpTest` round-trips format→parse within Kotlin; `BnLogFormatDriftTests`
reads **both sources** — the live `BnLog` and the checked-out Kotlin — and runs the Kotlin parser's
rules over C#'s real output.

The Kotlin `BlazorNativeInitOptions` mirror gained `logLevel` at offset 28 (`NativeBindings.kt:348`),
and Gate B also shipped the **consumer-facing knob** the design only sketched: a manifest
`<meta-data android:name="io.blazornative.logLevel">` plus an `EXTRA_LOG_LEVEL` Intent extra
(`MainActivity.kt:93`, `:117`, `:507`–`:524`), with manifest meta-data winning and every failure
mode falling through to UNSET.

And the docs half, which is larger than it sounds: **12 KDoc rc contracts** that pointed readers at
a stderr Android destroys — eight in `NativeBindings.kt`, four in `ShellBridge.kt` — became true for
the first time. The two now-false *"reproduce on the desktop JVM to see it"* strings were corrected
together with the assertions that pinned them (`DispatchEventTest.kt`, `HostEventTest.kt`) — a
self-inflicted, self-pinned update, which is the good case.

### Gate C — `os_log` for the disclosure half, not the noise half

The deciding argument for `os.Logger` over a cheap level-checked `NSLog` wrapper was **not**
verbosity. It was that **`Logger`'s string interpolation is private by default** while `NSLog` is
always public — every exception description, keychain key, image URL and file path it is handed is
readable in any log collected off the device. That is the half of #155 that level gating *cannot*
fix, because an `Error` ships in Release by design. The sweep is the same 78 edits either way; only
the body of `BnLog` differs.

**Deployment-target gotcha, found while writing the file and worth carrying forward:** `os.Logger` —
the struct whose interpolation carries the privacy specifiers — is **iOS 14+**, and the shell's floor
is **13.0**. `BnLog.swift:317` uses `Logger` where available and falls back to `os_log` on the same
`OSLog` object below it, with explicit `%{private}@` / `%{public}@` specifiers. The fallback is not a
privacy downgrade: those specifiers are the same mechanism `Logger`'s interpolation compiles into.

All **78** shipped `NSLog` sites under `BnHost/` were swept, levelled **by class** per design §4.3 —
boot narration → `Info` (suppressed in Release), the mapper's `ignored`/`skipped` → `Warn` (they
**ship** in Release; each records a wire the app author asked for being silently dropped), faults →
`Error`. `BnWidgetMapper`'s existing `diagnose(nodeId:kind:)` funnel kept its dedupe and swapped its
`NSLog` for `BnLog.warn`; the other sites in that file were **not** folded into it, because the funnel
warns once per key and appends to `diagnostics`, which three test classes assert on.
`BnYogaLayout.mm`'s 4 Objective-C++ sites go through the `@_cdecl BnLogC` shim, formatting with
`vsnprintf` into a stack buffer rather than `NSString` — the target declares
`CLANG_ENABLE_OBJC_ARC` neither way, so `alloc` would leak under MRR and `autorelease` would not
compile under ARC.

Verified: **zero bare `NSLog` calls remain under `BnHost/`** — the 9 surviving occurrences of the
string are all comments explaining the sweep. `BnHostTests/**`'s sites deliberately stay: XCTest
output is not shipped output, and routing it through a level gate would only mean a failing test
could print nothing about why.

`NSLogDriftTests` carries a **deletion guard**, and this is the design decision most worth
re-reading. *"No bare `NSLog`"* is trivially satisfiable by deleting every diagnostic in the shell,
which would be strictly worse than the state Gate C found. So each swept file must still reach the
seam (`TheSweptFiles_StillRouteThroughTheSeam`). The header also states plainly why this pin does
more work than its Android twin: **the Swift half of this framework is built by exactly one thing —
`ios.yml`, on a macOS runner.** For anyone without a Mac, a source-scanning .NET test is the *only*
feedback that arrives before CI.

Gate C also gave the runtime's stderr a destination on iOS (`BnStderrPump` — `pipe` + `dup2` + a
`DispatchSource`, so no permanent thread, unlike the Android twin), installed inside
`HostViewController`'s existing XCTest guard. **The pump is not Android-only**; the two platforms are
genuinely symmetric, which was the point.

### Gate D — provenance, not prose

`mount` no longer returns 0 over a half-rendered screen when a component was handed a parameter it
does not declare. **The rc is the already-documented rc 2, and that is load-bearing rather than
convenient:** `BlazorNativeRuntime.kt`'s mount `when` ends in `else -> throw`, so an **unknown rc is
fatal**. A "non-fatal rc 4" would hard-crash every consumer still on an older **copied** shell — the
`StrictErrors`-in-production outcome #164 explicitly rules out, arriving through the back door. Both
shells already handle rc 2 correctly, so **no Kotlin, Swift or template file changed**: the JVM,
Android and iOS baselines all held across Gate D.

**The predicate keys on provenance.** `BlazorInterop.ParameterBindingFrames`
(`BlazorInterop.cs:181`–`:185`) names two Blazor **method identities** —
`Microsoft.AspNetCore.Components.Reflection.ComponentProperties.SetProperties` and its public caller
`ParameterView.SetParameterProperties` — and the classifier scans the recorded `StackTrace` plus the
inner chain, depth-capped at 8, with `AggregateException` unwrapped. Message matching was rejected in
writing: `ex.Message.Contains("does not have a property matching")` is localizable
(satellite-assembly material), version-fragile, and its failure mode is **silent** — it stops
matching, mount goes back to 0, nothing looks broken. Method identities are not localized and not
reworded for clarity, and when Blazor renames one the pin reddens on the upgrade PR, which is the
moment a human is already reading the diff. `TheClassifier_IgnoresMessageText_AndKeysOnTheThrowSite`
proves it by feeding an **impostor carrying #164's message verbatim** and asserting it is *not*
classified.

**The fail direction is safe.** An unrecognised stack classifies as *not* a binding fault, so the
renderer keeps log-and-continue. The failure mode of this predicate is "#164 is not fixed for that
fault", never "a recoverable fault crashed the app."

**It is deliberately narrower than the design.** `ComponentState.SupplyCombinedParameters` and
`ComponentBase.SetParametersAsync` are named as *excluded* (`BlazorInterop.cs:175`–`:180`): they
enclose the whole of parameter supply, including a component's own `SetParametersAsync` override —
genuinely app code, and not the "always an author bug, never transient" class §6.2 scoped this to.
`TheAllowList_IsTheBindingWriter_NotTheEnclosingSupplyPath` pins the exclusion so widening is a
decision rather than a tweak. The design named `SupplyCombinedParameters` as #164's own throw path;
narrowing to the writer beneath it is the phase's one substantive departure from its own design, and
it was the right one — including it would have made a throwing `OnParametersSet` a boot failure.

Log-and-continue is **scoped, not reversed**: an event handler throwing mid-session, a fault inside a
dispatch window, a `ReportContractViolation` all keep today's posture. And the binding fault is
logged **once per instance** by reference identity (`NativeRenderer.cs:495`) — the rethrow unwinds
through Blazor's own machinery and routes back through `HandleException` three times, and three
identical `ToString()` blobs in logcat for one author bug is exactly the noise #155 is about.

Design §6.2's **(b)** — a host error callback (bridge slot 11) or a frame flag — stays deferred, with
the trigger written down on `Exports.Mount` rather than left as a vague "later": revisit when a
genuinely **recoverable** fault class appears that the host must react to *programmatically* rather
than merely observe.

## What this does and does not prove

Stated plainly, because a logging phase that over-claims is the classic "green test, unread output"
trap the design warned about.

**Proven, by a test that would red:**

- Level gating, the ordinal→level map (including 0 → `Warn` and out-of-range → `Warn`), the
  `SizeOf == 32` **and** `OffsetOf == 28` pair in all three mirrors, and the line format's round
  trip **across C# and Kotlin sources read from the checkout**.
- **Zero** bare `Console.Error` in `src/**/*.cs`, **zero** bare `NSLog` under `BnHost/` — both with
  asserted non-vacuity, and the iOS one with a deletion guard.
- #164's behaviour directly: rc 2 for a bad `@bind`, rc 0 preserved for a throwing handler, the
  classifier ignoring message text, the allow-list still naming real Blazor methods.
- The pump's **forwarding logic**, driven over an in-memory stream against an injected sink.

**Compile/link-verified in CI, not observed:**

- **The iOS sweep is genuinely CI-verified.** No Mac and no Xcode were used to author Gate C — but
  `ios.yml` ran on `71470db` (Gate C's merge to `main`) and **passed**, including its
  `Assert XCTest baseline (242 passed / 0 failed)` step (`ios.yml:353`, `:1083`). The Swift and
  Objective-C++ compiled, linked and ran on a real macOS runner. What CI did *not* do is watch a
  Release-configured build's console.
- **The Android instrumented lane was the phase's one real evidence gap — it has since been run,
  and it FAILED.** ⚠ *Updated 2026-07-22, after this conclusion was first written.* At the time of
  writing, the lane's last execution was the scheduled run on `88e2b1c` at 05:50Z, which **predates
  Gate A** — so every Phase 11.4 Android change had landed without it ever executing. It was then
  manually dispatched on `f53f74a`:

  ```
  211 of 212 passed.
  io.blazornative.shell.BnStderrPumpAndroidTest.stderrWrites_reachTheSink_throughTheRealDup2: FAILED
    java.lang.AssertionError: nothing reached the sink — the dup2 over fd 2 did not take
      expected:<BnLogRecord(priority=6, category=runtime, message=device pump probe)> but was:<null>
  ```

  The single failing test is **the only proof that `Os.pipe()` + `Os.dup2()` actually captures fd 2
  on a real Android runtime** — the one thing a JVM test structurally cannot pin, and the entire
  reason this transport was chosen over the bridge. Everything downstream of the syscall (line
  splitting, partial lines, the format round-trip, the fallback) passes on the JVM lane, because
  those are pure functions.

  **So Gate B's Android transport is UNPROVEN, and may be undelivered.** It is not yet known whether
  this is a *transport* bug (`dup2` does not take) or a *test* bug (the probe writes via
  `Os.write(FileDescriptor.err, …)`, and if ART hands instrumentation a dup rather than literal fd 2
  the probe bypasses the pipe while the pump is fine). **The two have opposite fixes**, so the cause
  must be measured, not guessed. Tracked as
  [#191](https://github.com/MarcelRoozekrans/BlazorNative/issues/191).

  **Read DoD #6's closure with this carve-out:** the seam, the level gating, the .NET migration and
  the iOS transport are delivered and evidenced; the **Android transport's device-level proof is
  currently red**. The lane is advisory, so no required gate was bypassed — the gap was that it had
  simply never run against this work.

**Inspection-only, and NOT done:**

- **"A Release build is actually quiet on a device."** Nobody has installed a Release-configured
  build on hardware or a simulator and watched logcat / Console.app go silent. The *mechanism* is
  proven (default `Warn`, boot narration at `Info`, gated before the write), and the boot narration
  is levelled in source — but the observation itself has not been made. This is the sole open
  clause of #155 (below).
- **That the log is *useful*.** No test asserts that a message helps a developer.
- **Third-party stderr interleaving** under a real crash reporter (design R2 — `dup2` over fd 2 is
  process-global and effectively irreversible; last writer wins). Documented, not tested, and it
  cannot be tested.
- **The consumer-facing docs for the level knob.** The Android `<meta-data>` / Intent-extra knob and
  `BnLog.Level` are documented in KDoc and xmldoc, where IntelliSense finds them — but **nothing
  under `website/docs/**` mentions the log level at all.** Design R6's mitigation asks for the opt-in
  to be documented "where a consumer actually looks, not only in a design doc." That is not yet true,
  and it is carried below rather than rounded off.

**No device and no Mac were used by the author in Gates A–D.** Every Swift and Kotlin decision was
made from source and pinned by a source-scanning .NET test precisely because of that.

## #155 — assess honestly, and the recommendation

The issue is titled *"Logging discipline: level-gated, quiet-in-Release, unified across shells"* and
is **still OPEN**. Clause by clause:

| Clause | Status |
|---|---|
| **Level-gated** | ✅ **Delivered.** One `volatile int` threshold, five levels, three configuration inputs, gated before the write so a suppressed line costs one branch. |
| **Unified across shells** | ✅ **Delivered.** Same levels, same ordinals, same wire format, drift-pinned across C#, Kotlin and Swift; both platforms now have a transport for the runtime's own output. |
| **No internal detail at default Release verbosity** | ✅ **Delivered**, and it was the clause most at risk of being skipped — level gating alone does not achieve it, because an `Error` ships in Release by design. `FormatException` + iOS's private-by-default interpolation close it. |
| **Quiet-in-Release** | ⚠️ **Mechanism delivered, silence not observed.** The default is `Warn`, so boot narration is gated. But **nobody has run a Release build on a device or simulator and watched the log go quiet.** |

**Recommendation: do NOT close #155 yet. Carry the observation as its explicit remainder.**

The reasoning, and it is the repo's own standard rather than caution for its own sake:

1. **The unobserved clause is the one in the title.** "Quiet-in-Release" is not a footnote of #155 —
   it is a third of its name and the whole of its original user-visible complaint (an App Store build
   emitting `native init ok`, `mounted <component>` and the mapper's volume). Closing on a
   source-level argument that those lines are now `Info` would be closing the issue on the *design*,
   not on the *result*.
2. **This repo has been burned by exactly this shape of claim, twice, and wrote it down.**
   `src/Directory.Build.targets:51`–`:63` records CS1591 being *"read as ON and was OFF"* — twice —
   and Phase 11.3 Gate B was wired specifically not to reproduce it, proving its gate **by mutation**
   rather than by a green build. A logging claim verified only by reading the code that produces the
   log is the same class of evidence that failed there.
3. **The observation is cheap and an owner-run device session is a known, repeatable activity.**
   Phase 11.2's runbook already exists and the phone is available. Design §8.2 recommended precisely
   this — a recorded observation in the device runbook, not a CI gate — and 11.2 ran before 11.4
   landed, so the step was never available to that session.
4. **There is a second, smaller remainder in the same issue:** *"verbose diagnostics are opt-in"* is
   delivered in code and in KDoc/xmldoc but **absent from `website/docs/**`**. Design R6 explicitly
   asks for it "where a consumer actually looks." An issue with two named, concrete remainders is not
   a done issue.
5. **Repo discipline.** The 2026-07-21 issue audit found that none of the 16 open issues were fully
   done and that each had real remainder; the standing rule is not to tidy by closing things that do.

**Proposed remainder, to be pasted onto #155 by the owner** (not by this phase — the owner decides):

> Delivered by Phase 11.4 (#185/#187/#188/#189): level-gating ✅, unification across both shells ✅,
> no internal detail at default Release verbosity ✅. **Open:** (a) observe a **Release-configured
> build on a device** emitting no `Info`-level narration — Android via logcat, iOS via Console.app on
> the simulator — and record it, ideally as a step appended to the Phase 11.2 device runbook;
> (b) document the level knob (`BlazorNativeInitOptions.LogLevel`, `BnLog.Level`, the Android
> `<meta-data>` / Intent extra) under `website/docs/**`, where a consumer actually looks.
> **Also worth updating on the issue itself:** its goal text asks to gate *"the normal-path
> boot/`ignored` chatter behind Debug/Verbose."* 11.4 deliberately **split that sentence** — boot
> narration → `Info`, but the mapper's `ignored`/`skipped` diagnostics **stay at `Warn` and ship in
> Release**, because each records a wire the app author asked for being silently dropped and Android
> already levelled them `Log.w`. Demoting them would hide the one class of message that reliably
> indicates a real author bug, in the build where it costs most. The issue and the delivery should
> agree in writing rather than the divergence living only in a design doc (§4.3).

## Where the design was departed from, deliberately

Recorded so the divergences are decisions rather than drift.

1. **The public level setter is `BnLog.Level` in `BlazorNative.Core`, not a property on
   `BlazorNativeApp`.** Design §3.3(2) put it on the 11.1 `ConfigureServices` seam. It landed on the
   seam type itself instead — reachable from a `ConfigureServices` callback all the same, with one
   fewer forwarding property and one fewer thing to keep in sync. `BlazorNativeApp` gained nothing.
2. **Gate D's classifier excludes `SupplyCombinedParameters`** — see Gate D above. The design named
   it; including it would have made a throwing `OnParametersSet` a boot failure.
3. **The mapper `ignored`/`skipped` diagnostics stay at `Warn`**, against #155's literal goal text.
   Design §4.3 argued it; it shipped that way on both platforms; the issue should be updated to
   match (above).
4. **`BnWidgetMapper`'s 53 non-funnelled sites were rewritten individually rather than folded into
   `diagnose`** — folding would change dedupe semantics that three test classes assert on. The design
   already ruled this out; noting it because the funnel looks like the obvious tidy-up and is not.
