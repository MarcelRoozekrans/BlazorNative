## What shipped

**Camera photo capture on both shells, riding the 9.0 ABI at zero further cost — even though a
photo is 1–10 MB.** One op — `Camera` (=4) — flows over the **existing** `HostCallBegin` slot
(offset 72) and `blazornative_host_call_complete` export. The captured image crosses as a **file
PATH, not bytes**: the shell captures to an app-cache/temp JPEG and returns
`{"path":"file://…","width","height","bytes"}` on the **OPTIONAL payload** the completion channel
has carried since 9.0 (geolocation opened it; secure-storage/notifications were the second/third
users; camera is the fourth — the channel is now proven generic across a coordinate, a status, a
secret, and a **file path**). The only NEW surface is **wire vocabulary**: the op value `Camera = 4`
and the `CameraStatus` enum (Captured/Cancelled/Denied/Unavailable/Error). `ICamera` joins
`IGeolocation` / `INotifications` / `IBiometrics` / `ISecureStorage` as the **5th Device façade** in
the existing **7th package** `BlazorNative.Device` (no 8th package); `DevHostBridge` mocks every
status + a canned real `file://` test-image path headless; the sample app grows a `/camera` demo
(`BnCameraDemo`, the **13th routed page**, sample-only per the 9.0/9.1/9.2 precedent).

## THE HEADLINE — the 9.0 ABI bet paid a FOURTH time: camera cost the ABI NOTHING

**Phase 9.0 grew the bridge to 80 bytes / 10 exports ONCE and argued it would be paid once** —
9.1/9.2/9.3 add op constants and touch the ABI at nothing else. Notifications drew first (9.1),
biometrics + secure storage second (9.2); **camera draws fourth, and it is the sharpest test of the
bet yet — because the result is a multi-MB image, exactly the kind of payload that "obviously" needs
a new binary export.** It did not:

- **Bridge stayed 80 bytes.** No new slot. The op rides the existing `HostCallBegin` at offset 72
  (the last slot; nothing appended).
- **Exports stayed 10.** No new symbol. The capture reuses `host_call_complete`; the image rides its
  **optional payload** as a `file://` PATH — the bytes never cross the boundary, so no binary/buffer
  export is added.
- **No drift-pin moved, no gate arithmetic.** The three-mirror struct pin (80 on .NET / Kotlin /
  Swift), the export gate (10), and the offset pins are byte-for-byte what 9.0 left. Camera added
  only **one op value + one status enum**.

Proven on all three RIDs: the win-x64/bionic `dumpbin`/`readelf` gates and the iOS archive `nm` gate
all show the SAME ten `blazornative_*` symbols (`all 10 present`, run **29639956941**). **The reuse
proof is made falsifiable, not asserted.** `CameraAbiUnchangedTests` pins
`Marshal.SizeOf<BlazorNativeBridgeCallbacks>() == 80`, `OffsetOf(HostCallBegin) == 72`, and
`Camera == 4`, *inside the camera suite* so the claim lives next to the code it guards. The named
mutations prove it bites: reuse-proof-81-bytes reds the .NET pin, and the iOS **struct-grow mutant
FAILED TO COMPILE** (`missing argument 'cameraExtra'`, run **29640255862**) — the Swift type system
itself forbids a silent grow. **"Pay once, reuse thrice" is now demonstrated a FOURTH time, by
construction.**

## THE IMAGE HANDOFF — the phase's real decision, made and proven

A photo is 1–10 MB, but **the payload NAMES A FILE; it does not carry the bytes.** The shell
captures to an app-cache/temp JPEG and returns `{"path":"file://…","width","height","bytes"}` on the
optional `host_call_complete` payload. This is the decision the phase turned on:

- **Bytes-inline was rejected.** Marshalling multiple megabytes through a `const char*` and into a
  non-zeroable .NET `string` is ~1000× the secure-storage in-memory hazard (9.2's zeroable-buffer
  note) — for something that **is not even a secret**. A binary export was also rejected: it would
  have grown the ABI (the very thing the bet forbids).
- **The file-path handoff keeps the ABI frozen.** The bytes stay on disk; only a path (and cheap
  metadata) crosses. The completion channel that carried a coordinate (9.0), a status (9.1), and a
  secret (9.2) now carries a **file path** — the fourth proof that the channel is generic, not
  shaped to any one user.
- **The temp-file lifetime is stated as an ownership boundary.** The capture lands in the
  **app-cache dir** (OS-reclaimable). After the handoff the **app owns the file** — `BnImage` decodes
  it asynchronously, so there is no auto-delete race. The shell prunes its own capture dir
  (keep-last-N) **per capture** as a leak backstop; **the .NET side never deletes the file** — the
  ownership boundary is explicit.

## BnImage composition — capabilities composing across milestones, and a ledger discharged

The captured `file://` path is a valid `BnImage.Src` — Coil (Android) and Kingfisher (iOS) both load
local files — so the `/camera` demo displays the photo in a **DEFINITE 240×320 `BnImage` with
`ContentMode="Contain"`**. This is not just a demo: **it DISCHARGES the M6/M7 "revisit ContentMode
with a real natural-size image" ledger item.** M6/M7 shipped ContentMode against synthetic fixtures
and ledgered a real megapixel photo in a definite box; a captured camera image IS that photo —
placed in a definite frame, its natural size never measured, so no reflow. The composition is proven
**on the AVD end-to-end** (camera → file → a measured 240×320 `BnImage`) AND **on iOS via the seam
through `BnImageLoader.naturalPixelSize`** — the exact function the image node's Yoga measure func
calls.

**EXIF / orientation: BOTH shells NORMALIZE** — they bake rotation into upright pixels and reset the
tag, so Coil/Kingfisher never double-rotate. Android does it via `ExifInterface` + `Matrix`; iOS via
`UIImage.imageOrientation` redraw. (The Android EXIF-normalize is mutation-proven — see below.)

## The capture path + permission — a new drift class on Android

- **Android: `ACTION_IMAGE_CAPTURE` Intent to the system camera app — NO runtime `CAMERA`
  permission.** The system camera app owns the sensor, so no dangerous permission is requested; the
  only surface is a **`FileProvider`** for the output URI. This is a **NEW manifest + resource drift
  class**: a `<provider>` (plus a `<queries>` for `IMAGE_CAPTURE`) and a `res/xml/file_paths.xml`
  `<cache-path>`, all mirrored into the template, with `file_paths.xml` added to
  `TemplateDriftTests`' expected manifest — **the pack's inventory moved 32 → 33 files**.
- **iOS: `UIImagePickerController(.camera)` + `NSCameraUsageDescription` + `AVCaptureDevice`
  authorization.** (`NSCameraUsageDescription` must be present or camera use `SIGABRT`s.)
- **Denial / cancel is DATA, never an exception across the boundary, never a hang.**
  `CameraStatus` carries Captured / Cancelled / Denied / Unavailable / Error.

## The emulator / simulator honesty — this phase's sharpest

**Android:** the emulator's fake-scene satisfies `ACTION_IMAGE_CAPTURE`, but the **system camera-app
SHUTTER is not CI-drivable** — so the capture RESULT is **seam-driven** (`cameraCaptureHook`, the
geolocation/biometrics real-UI split). The honesty: the synthetic bytes are written **THROUGH the
real `FileProvider` `content://` URI** (as the camera app writes to `EXTRA_OUTPUT`) and delivered
through the SAME processing path — so the **provider authority is exercised for real**, proven by the
mis-authority mutation that reds the composition (`Couldn't find meta-data for provider`).

**iOS is worse: the simulator has NO camera at all.** `UIImagePickerController(.camera)` is genuinely
unavailable, so **`check → Unavailable` is asserted as the CORRECT sim result** (no override, not a
workaround) — the honest analogue of 9.2's no-Enclave. The capture round-trip is seam-mocked (the
test fires the REAL delegate with a canned `UIImage`), and a real capture is **DOUBLY UNPROVEN — no
sim camera AND no Apple account** (iOS real-device deferred, the M5–M9 trigger).

**This is the milestone's second capability (with biometrics) where the emulator is LEAST like
reality** — recorded honestly rather than smoothed over.
