# Phase 7.0 Design — The Razor-Compilation Spike

**Date:** 2026-07-15
**Milestone:** M7 — Components + Razor (closes DoD #1 — with EITHER verdict)
**Predecessor:** M6 complete (`v6.0`)
**Owns:** the named risk, verified first: **can a `.razor` file compile into a component that
renders through `NativeRenderer` under NativeAOT — no web host, trim-clean?** A GREEN unlocks
M7's authoring story (Phase 7.1 rewrites the five demo pages); a RED re-scopes it, with the
fallback documented.

---

## Framing (locked at milestone-open)

- **The natural route is a Razor Class Library** (`Microsoft.NET.Sdk.Razor`): it compiles
  `.razor` → C# (`BuildRenderTree` calls on `ComponentBase`) via the Razor source generator, and
  `BlazorNative.Components` *already* references `Microsoft.AspNetCore.Components` — the
  generated code's only hard dependency. The open questions are what *else* the SDK drags in.
- **The three specific risks, in likely order of pain:**
  1. **`@bind` on elements.** `@bind-`/`@bind:event` expansion for standard elements is defined
     by `[BindElementAttribute]`s living in **`Microsoft.AspNetCore.Components.Web`** — a
     *web* assembly. Either referencing it is trim-benign under NativeAOT (possible — the bind
     attributes are compile-time metadata), or we declare our **own** `[BindElementAttribute]`s
     for the elements our renderer actually maps (`input` etc.) in our own assembly. The spike
     answers which.
  2. **The SDK's web baggage.** `Microsoft.NET.Sdk.Razor` wires static-web-assets, scoped CSS,
     and browser-refresh targets. All believed build-time-only and switch-off-able
     (`StaticWebAssetsEnabled=false`, no `.razor.css`) — believed, not known. The spike knows.
  3. **NativeAOT/trim.** The generated `BuildRenderTree` bodies are plain calls — the same shape
     we hand-write — so the **4 accepted IL2072s and 9 exports must not move**. Any new trim
     warning is a finding, not an accept.
- **What the spike does NOT do:** no page rewrites, no new components, no `@page` routing
  semantics (our routes live in the registry), no CSS. One component, end to end.

## The GREEN bar

One spike component, `SpikeRazor.razor`, exercising the four things 7.1 needs:
**markup + a `[Parameter]` + an event handler (`@onclick`) + `@bind` on an input** — compiled
from `.razor` by the SDK (not hand-translated), and:

1. It renders through the existing .NET test harness and **emits the same patch stream as a
   hand-written `BuildRenderTree` twin** (golden-vs-golden — the proof it's the same wire).
2. The event handler and the `@bind` round-trip work through `blazornative_dispatch_event`
   (the existing dispatch tests' harness).
3. `dotnet publish -r win-x64` stays at **exactly 4 IL2072 + 9 exports** (the required lane's
   own assertions, run locally).
4. The JVM suite (83) is untouched; if the spike component is cheap to mount on the AVD/sim,
   a device render is a bonus, not the bar (the wire golden already proves the contract).

Record the **integration recipe** (csproj deltas, the bind-attribute decision, anything the SDK
needed disabled) — the analog of 6.0's per-platform recipes, so 7.1 builds on it.

## Fallback ladder on RED (decided now)

1. **Own the generator:** a small source generator translating `.razor` (or a constrained subset)
   to `BuildRenderTree` — we control trim behaviour and bind expansion outright. Heavier, ours.
2. **Stay hand-written, better:** keep `BuildRenderTree` authoring with helper APIs/builders.
   M7 proceeds without DoD #2's syntax goal; the ledger records why.

## Gates

| Gate | GREEN bar |
|---|---|
| **1 — verdict** | The spike component compiles from `.razor`, matches its hand-written twin golden-for-golden, round-trips events + bind, publish counts unmoved; conclusion doc committed with the recipe (or the RED + fallback decision); PR merged. |

## Risks

| Risk | Mitigation |
|---|---|
| `@bind` needs `Components.Web` | Try own `[BindElementAttribute]`s first — smallest dependency footprint; referencing Components.Web is the fallback *if* trim-clean |
| SDK drags web build steps into the required lane | Everything the SDK adds must be visible in the publish/CI logs; the spike diff shows exactly what changed |
| Generated code trips new trim warnings | The 4-IL2072 assertion is the tripwire; a new warning is a RED finding to resolve, never an accepted-count bump without analysis |
| Razor language features we don't support yet (@page, CSS isolation, JS interop) | Out of spike scope; 7.1 defines the supported subset and the analyzers can police it later |
