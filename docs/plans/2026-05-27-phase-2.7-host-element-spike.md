# Phase 2.7 Host-Element Spike — Findings

**Date:** 2026-05-27
**Status:** complete — drives Phase 2.7 real scope
**Predecessor:** Phase 2.6 (widget mapper completeness, three-way GREEN)
**Successor:** Phase 2.7 design (this doc shapes its scope), then Phase 2.8 (Hello demo + final audit)

---

## Why this spike existed

The original `ROADMAP.md` / `MILESTONE.md` Phase 2.7 entry read: *"`BlazorNativeHostElement` stub (renderer-side host element descriptor satisfying Blazor's requirements without a real DOM)"*. The framing was speculative — written before Phase 2.4-2.6's evidence that `AssignRootComponentId`-based mounting works for at least one component (the sentinel).

The user asked: **"Is this the same bUnit does? Then we can maybe use AngleSharp?"** — a sharp reference pointing at bUnit's pattern of `RenderBatch` → AngleSharp DOM. The spike's job: figure out whether Blazor components NEED a host abstraction or whether what we have is sufficient. If sufficient, Phase 2.7 has no host-element work to do. If insufficient, AngleSharp (per the bUnit precedent) is the natural reference architecture.

## Method

Wrote 5 probe components in `tests/BlazorNative.Renderer.Tests/Phase27HostElementSpike.cs`, each escalating in Blazor-feature complexity:

1. **Parameters** — `[Parameter] public string Greeting { get; set; }`
2. **Event handlers** — `<button @onclick="OnClick">` via `EventCallback.Factory.Create`
3. **Nested components** — `<Parent>` containing two `<NestedChild Label="..." />`
4. **DI** — `[Inject] public GreetingService Svc { get; set; }`
5. **CascadingValue** — `<CascadingValue Value="dark">` wrapping a child consuming `[CascadingParameter] string? Theme`

Each test mounts via `NativeRenderer.MountAsync<T>(ParameterView.Empty)`, captures the first emitted `RenderFrame`, and asserts on the patch stream.

Tests run on the .NET host CLR (not Mono-WASI). The Mono-WASI-specific defects from Phase 2.4 Task 4 are already documented and orthogonal to this spike's question (renderer architecture, not runtime).

## Findings — top-level

**All 5 probes PASS.** Concretely, with the captured `[FRAME]` lines:

| Probe | Pattern | Result | Visible patches |
|---|---|---|---|
| 1 | Parameters | ✅ PASS | `create(view) → create(text, parentId=1) → text("Hello from spike") → commit` |
| 2 | Event handlers | ✅ PASS | `create(button) → event(nodeId=1, eventName="click", handlerId=1) → create(text, parentId=1) → text("tap") → commit` |
| 3 | Nested components | ✅ PASS (but with a real renderer bug — see below) | parent + two children's text patches all in one frame |
| 4 | DI via `[Inject]` | ✅ PASS | `create(view) → create(text, parentId=1) → text("Hello, DI!") → commit` |
| 5 | CascadingValue | ✅ PASS | `create(view) → create(text, parentId=1) → text("theme:dark") → commit` |

**Top-level conclusion: no host element abstraction is needed.** Real Blazor components — events, DI, cascading values, nested composition — work with the current `AssignRootComponentId`-based mounting pattern. The original "host element stub" framing turned out to be unnecessary.

**AngleSharp is not needed either.** bUnit uses AngleSharp because *tests* want a DOM to query. Our renderer produces patches for Android widgets; nothing on the .NET side ever queries a DOM. Adding AngleSharp would create a parallel shadow tree that's never read.

## Findings — bugs surfaced

The spike surfaced **two real bugs** that Phase 2.7 should now scope itself around.

### Bug A — `MountAsync<T>` defaults to `default(ParameterView)` which NREs

**Symptom:** calling `renderer.MountAsync<T>()` with no arguments throws `NullReferenceException` deep in Blazor framework code:

```
[BlazorNative.Renderer] System.NullReferenceException
   at Microsoft.AspNetCore.Components.ParameterView.RenderTreeFrameParameterEnumerator..ctor
   at Microsoft.AspNetCore.Components.ParameterView.GetEnumerator()
   at Microsoft.AspNetCore.Components.Reflection.ComponentProperties.SetProperties
   at Microsoft.AspNetCore.Components.ParameterView.SetParameterProperties
   at Microsoft.AspNetCore.Components.ComponentBase.SetParametersAsync
   at Microsoft.AspNetCore.Components.Rendering.ComponentState.SupplyCombinedParameters
```

The NRE is silently swallowed by `NativeRenderer.HandleException` (logs to `Console.Error`, doesn't re-throw), so the mount appears to "succeed" — but no patches are produced. From the caller's perspective: mount succeeds, no frame fires, eventual timeout if waiting on `_frames` event.

**Root cause:** `NativeRenderer.MountAsync<TComponent>(ParameterView parameters = default, ...)` defaults to `default(ParameterView)`. The default `ParameterView` struct has null internal `_frames` array; Blazor's enumerator NREs when iterating it.

**This is the same defect Phase 2.4 Task 4 documented as #3.** Phase 2.4's fix patched `Mount<T>` (the sync variant) to explicitly pass `ParameterView.Empty` in its parameterless overload. The fix was **incomplete** — `MountAsync<T>` still has the latent bug. Also: the bug fires on the .NET host CLR, not just Mono-WASI — it's a Blazor framework bug or renderer-API misuse, not Mono-specific. Recontextualizes the Phase 2.4 Task 4 finding.

**Fix in Phase 2.7:** change `MountAsync<T>` signature to `ParameterView parameters = default` → either always normalize internally to `ParameterView.Empty` when default is detected, or change the default-value-expression. The latter isn't possible (`ParameterView.Empty` isn't a `const`), so normalize internally:

```csharp
public Task<int> MountAsync<TComponent>(
    ParameterView parameters = default,
    CancellationToken ct = default)
    where TComponent : IComponent
{
    var safeParams = parameters.Equals(default(ParameterView)) ? ParameterView.Empty : parameters;
    return Dispatcher.InvokeAsync(() => AddComponentAsync(typeof(TComponent), safeParams));
}
```

(Verify `ParameterView.Equals(default)` works as expected, or use an alternative null-check pattern.)

### Bug B — `ProcessFrame` mis-attributes Component-frame parameters to the parent Element

**Symptom:** Probe 3's parent-with-two-children renders correctly *visually* (both children's text appears), but the patch stream contains erroneous `UpdatePropPatch`es on the **parent div**, not on the child components:

```
create(view, nodeId=1)
prop(nodeId=1, name="Label", value="A")   ← belongs to NestedChild #1, attributed to parent div
prop(nodeId=1, name="Label", value="B")   ← belongs to NestedChild #2, attributed to parent div
create(text, nodeId=2) ...                 ← child A's spans/text (from child's own diff)
create(text, nodeId=4) ...                 ← child B's spans/text (from child's own diff)
commit
```

The widget mapper silently log-ignores unknown `UpdateProp` names (Phase 2.6), so the visible output happens to be correct — but the patch stream is noisy and conceptually wrong.

**Root cause:** `NativeRenderer.ProcessFrame`'s `switch (frame.FrameType)` handles only `Element` and `Text`. When a `Component` frame appears as a child inside an Element's subtree, the recursive call hits no case and emits no patch. Worse, the subtree iteration does NOT advance past the Component's own subtree (no `i += child.ComponentSubtreeLength - 1` like the Element case has) — so the Component's child Attribute frames (which carry the parameter values like `Label="A"`) are seen by the parent's iteration and routed through `ProcessAttribute(parent_nodeId, ...)` as if they were attributes of the parent Element.

**Fix in Phase 2.7:** add a `Component` arm to the `ProcessFrame` switch. Either:
- (a) Emit nothing for Component frames, but advance the parent's iteration past `ComponentSubtreeLength` to skip the misattributable Attribute frames. The Component's own diff in the same `RenderBatch.UpdatedComponents` array handles its own rendering. Simplest fix.
- (b) Emit a logical "component boundary" patch (new type), use it for parenting info downstream. More elaborate; only needed if Phase 3+ component composition wants explicit boundary tracking.

(a) is YAGNI-correct for M2.

## Findings — non-bugs worth noting

1. **`NativeRenderer.MapElementToNodeType` maps `<span>` → `"text"`** (per `NativeRenderer.cs:313`). Probe 3 confirmed this: the child component renders `<span>child:A</span>` which becomes `create(text) → create(text, parentId=...) → text("child:A")`. Visually fine — the host's `WidgetMapper.handleCreate("text")` returns a `TextView`, child text replaces its content. But "span containing text" is a slightly weird representation. Phase 3+ may want to revisit the element→nodeType table when real components arrive.

2. **EventCallback `handlerId` is opaque but stable.** Probe 2's emitted `event(handlerId=1)` shows the renderer assigns sequential ids starting from 1. This is what we need for the future Phase 3+ host→.NET click round-trip (`WasiBridge.DispatchEventCore(eventName="click", payload=...)` keyed on `handlerId`).

3. **The renderer's `HandleException` swallowing is a hazard.** Bug A would have been caught in Phase 2.4 if `HandleException` re-threw or set a flag the caller could check. Currently exceptions during render are written to `Console.Error` and silently dropped. Consider a "strict mode" or a public `LastException` property for debugging in M3.

## Recommendation for Phase 2.7's real scope

**Drop the "host element stub" framing entirely.** That work isn't needed.

**Phase 2.7 = "Renderer hardening: bug fixes + spike regression coverage."** Three concrete deliverables:

1. **Fix Bug A:** `MountAsync<T>` normalizes `default(ParameterView)` → `ParameterView.Empty`. Add a renderer test that calls `MountAsync<SimpleComponent>()` with no args and asserts a frame fires.
2. **Fix Bug B:** Add Component-frame handling to `ProcessFrame` (emit nothing + advance iteration past `ComponentSubtreeLength`). Update Probe 3's assertion to verify the parent's patch stream contains NO `UpdatePropPatch`es for Component-parameter names.
3. **Graduate the spike into permanent tests.** Move `Phase27HostElementSpike.cs` → `Phase27RendererCoverage.cs` (or similar). Strengthen assertions where the spike used soft "log-only" reporting. These become the regression suite for the renderer's component-API surface — protecting M2's gains as M3+ adds richer components.

**Estimated effort:** ~half a day. Two small `NativeRenderer.cs` fixes + test cleanup + ROADMAP/MILESTONE update.

**What this means for Phase 2.8 (Hello demo + final audit):** unblocked. The Hello demo can use any of the 5 probe patterns directly — parameters, events, nested components, DI, cascading values all work. No need to wait for richer infrastructure.

## What the spike rules OUT

- **bUnit-style AngleSharp DOM:** not needed. The renderer doesn't need a DOM; the widget mapper doesn't need one either. AngleSharp would be ~2 MB of dead weight.
- **A `BlazorNativeHostElement` class:** not needed. `AssignRootComponentId` is the supported renderer API for headless mounting; no host element selector required.
- **A custom DOM-replacement abstraction:** YAGNI. The patches-to-widgets pipeline is the abstraction; nothing in the .NET-side renderer needs anything more.
