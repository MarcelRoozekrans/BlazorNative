**Milestone:** 10 — Consolidation & Hardening. **Phase:** 10.0 (first).
**Closes:** DoD #1 (#121 iOS-reports-Android) and DoD #2 (#123 Frames false-green).

## Gate A — #123: the `Frames` channel cannot silently swallow a failure

### The defect
`src/BlazorNative.Renderer/NativeRenderer.cs:379`:
```csharp
_ = _frames.InvokeAsync(frame, default);
```
`_frames` is an `AsyncEventHandler<RenderFrame>` (`InvokeMode.Sequential`). It reports
subscriber faults **through the returned task**, which is discarded. The surrounding
`try/catch` guards only the synchronous body, so a throw inside a `Frames` subscriber is
**unobserved**: it never reaches `HandleException`, and `StrictErrors` cannot surface it.
`Frames` is the documented **test channel** — a failing assertion inside a frame subscriber
is dropped, so a test can be **green while actually failing**.

### The fix
Observe the task and route its fault through the same path the synchronous body uses
(`HandleException`, which under `StrictErrors` surfaces rather than logs):

- If the returned task is already completed (the common case — synchronous subscribers
  complete inline), surface its fault immediately so it lands in the surrounding `catch`
  → `HandleException` (`GetAwaiter().GetResult()` inside the try, or an explicit
  `IsFaulted` → `HandleException(task.Exception.InnerException ?? ...)`).
- If it is still running (a genuinely async subscriber), attach a fault continuation that
  routes to `HandleException` so the fault is never dropped.

The frame is still dispatched (`DispatchFrame`) on the success path. The exact shape is the
implementer's call, subject to: **no fault is ever discarded**, faults route through
`HandleException`, and `StrictErrors` surfaces them.

### Red-first (mandatory — the finding IS false-green)
A test that registers a `Frames` subscriber which throws, drives a render/frame, and asserts
the fault is surfaced (under `StrictErrors`) / observed by `HandleException`. It **must FAIL
against current `main`** (proving the drop is real) and pass after the fix. Without the red
run, we cannot claim the green means anything — that is the whole point of the issue.

### Scope
Contained to `BlazorNative.Renderer` + `BlazorNative.Runtime.Tests`. No ABI, no shells, no
drift. `.NET` count moves +1 (the new test) — update the `754` literal in `ci.yml` + README
in the same change.

## Gate B — #121: iOS must not report `PlatformKind.Android`

### The defect
`NativeShellBridge.GetPlatformInfoAsync` (and the `PlatformInfo` JSON builder) hardcode
`PlatformKind.Android` (`src/BlazorNative.Runtime/NativeShellBridge.cs:~912`). The same
NativeAOT runtime is linked into the iOS `.a`, so an **iOS app reports `Android`** through
the public `IMobileBridge.PlatformInfo` / `GetPlatformInfoAsync` surface. The `os` string is
host-supplied (correct), but `kind` is a constant.

### The fix — the kind comes from the shell, like `os` does (no fragile string parsing)
Add an explicit platform-kind to the init path rather than deriving it from the `os` string
(the issue calls derive-from-`os` fragile, and it conflates display-OS with kind):

- Add `PlatformInfoKind` (an `int`) to `BlazorNativeInitOptions` (`Exports.cs:60`), after
  the existing three fields. `Init` passes it to `SetPlatformInfo`, which stores it; the
  `PlatformInfo` JSON + `GetPlatformInfoAsync` serve the stored kind.
- Contract: the int mirrors the `PlatformKind` enum ordinal (`DevHost=0, Android=1, iOS=2,
  Windows=3, Mac=4` — pin the mapping in a test). An unset/0 value is `DevHost`, a safe
  non-lying default for a shell that hasn't been updated.
- **Both shells pass their own kind:** Android's `MainActivity.kt` (currently
  `platformOs = "android"`) adds the kind = Android; the Swift `BnRuntime.swift`
  `blazornative_init(&opts)` sets kind = iOS. The C header (`BlazorNativeRuntimeC.h`) gains
  the field; the template's Android shell mirror stays byte-identical (template drift).

### This IS an init-struct ABI edit — done in lockstep, and it is NOT the frozen bridge
`BlazorNativeInitOptions` is the **init-input** struct, distinct from the frozen 80-byte
callbacks bridge and the 10 exports (those do NOT change — assert it). Both shells ship in
this repo and update together, so appending a field is a lockstep edit, not a flag day. Add
/update a drift assertion for the init-options layout if one exists; if none exists, add a
small pin that the field count + the kind-ordinal mapping are what the shells expect.

### Red-first
An `iOS` XCTest asserts `GetPlatformInfoAsync`/`PlatformInfo` reports **`iOS`**, not
`Android` — failing against current `main` (where it is hardcoded Android) and passing after.
The Android side keeps reporting `Android` (a .NET test + the instrumented lane).

### Scope
`Exports.cs` (init struct + Init), `NativeShellBridge.cs` (store + serve kind),
`IMobileBridge`/PlatformInfo if needed, `BlazorNativeRuntimeC.h`, `MainActivity.kt` (+ its
template mirror), `BnRuntime.swift`, drift + unit + XCTest. Counts move (.NET +, iOS XCTest
+) — reconcile every gate ↔ README row in the same change (the 9.2 lesson: all rows, not
just the one you remember).
