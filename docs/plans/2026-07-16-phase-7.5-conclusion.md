## The three features as proven

### `PlaceholderColor` — paint that never measures

The 4-row state table is **normative on both shells** (in-flight: paints the box; SUCCESS:
cleared, letterbox bars show `BackgroundColor` never the placeholder; ERROR: **stays** — the
error state's visual; `src → null`/absent/`""`: nothing, cleared with the image). Painted on
the in-flight row **before** the load is issued (a warm memory hit completes synchronously
inside the enqueue and the real bytes must be the LAST write — the ordering both shells
share), and it **cannot** measure by construction: neither shell's measure func reads
anything a placeholder path touches. An intrinsic image's placeholder is invisible (a 0 × 0
box paints nothing) and still reflows exactly ONCE when its bytes land — 6.3's reflow proof
re-run against the exact feature 6.3 refused to ship for fear of it. The wire name is
`placeholderColor`, never `placeholder` — the input hint has owned that name since M2, and
the partition pin asserts the two stay distinct props on the prop wire.

**Recorded deviation, judged an improvement:** the design suggested Kingfisher's placeholder
mechanism for iOS; the shipped mechanism is a **bounds-tracking color subview** inside
`BnImageView` (`bnShowPlaceholder`/`bnClearPlaceholder`), rejected-alternatives recorded:
(1) **not Kingfisher's placeholder slot / a color `UIImage`** — a `UIImage` has an intrinsic
size and contentMode letterboxes it; Android's `ColorDrawable` is intrinsic-less and UIKit
has no twin of that; (2) **not `backgroundColor`** — that UIKit slot belongs to the
`backgroundColor` STYLE, and the SUCCESS row says letterbox bars show `BackgroundColor`,
never the placeholder — one property cannot hold two wire values; (3) **the state table is
normative and loader-free** — the placeholder must survive terminal ERROR and clear on
`src → null`, lifetime the TABLE owns, not the loader's placeholder-until-completion
convention. The subview mechanism implements the table directly, per the 7.3 guard lesson
(mechanism per shell, verified never assumed).

### `OnError` — a new word, not a new wire

Event name `error` on the existing wire; **payload = the wire's `src` string, verbatim** —
the URL is the only fact Coil and Kingfisher share about the same failure, so it is the only
payload two shells can dispatch identically (strict-payload posture: a missing OR empty
payload is a shell contract violation → the loud rc-2 fault). Attach is
**iff `HasDelegate`** (the `BnScroll.OnScroll` shape) — an unwired `BnImage`'s patch stream
is byte-identical to pre-7.5. Dispatch discipline, each rule pinned: **at-most-once per
(src, generation)** behind the same liveness guard the paint uses (`isLiveImageRequest` /
its Swift twin — one guard, two consumers); **CANCELLED is not an error** (src change,
`src → null`, node removal all dispatch nothing — structurally: `onImageCancelled` never
asks the question); an unparseable non-empty URL IS a failure and dispatches. And
**failure never changes measurement in either direction**: a declared box holds because it
was DECLARED, an intrinsic node stays 0 × 0 and reserves nothing — the shell paints and
dispatches, never re-measures.

### `ContentMode` — four modes, paint-only

`Contain` (default) / `Cover` / `Stretch` / `Center` — the strict lowercase four-word wire
table, `FIT_CENTER`/`CENTER_CROP`/`FIT_XY`/`CENTER` on Android,
`.scaleAspectFit`/`.scaleAspectFill`/`.scaleToFill`/`.center` on iOS, row for row. **The
Yoga box never moves**: the measure func never consults the mode, asserted as four identical
120 × 60 frames under four modes on both devices (and four byte-identical style tables under
four mode words in the .NET golden). Unknown words are diagnosed loudly and NOT applied;
`null` restores the default (the `Enabled`-null precedent). **The default is `Contain`,
diverging from RN's `cover` — recorded, with the 6.3 reason**: the 6.3 contract row set
aspect-fit explicitly on both shells and said it chose fit "because it is the value an M7
`ContentMode` would default to"; adopting `cover` would silently repaint every existing
image **frame-neutrally**, the class of change no test on either shell can see. The
corollary is iOS's own: **`clipsToBounds = true` at creation, always, pinned as a
property** — `.scaleAspectFill` and `.center` paint past the Yoga box and iOS does not clip
(Android's `ImageView` does, by construction).

## THE HEADLINE — the defer mechanism, twice corrected by review

The design's defer-out-of-batch rule ("the dispatch never runs inside a patch batch —
deferred, never dropped") was analyzed correctly and **built wrong twice**, and both times
the construction error was caught by a gate review, not a test run — the phase's "analysis
says X, construction disagrees" story:

1. **Gate 2 review I-1 — the DEFER arm fired unconditionally with decision-time state.**
   Android's `onImageFailed` closed the DEFER arm over a decision-time verdict
   (`mainHandler.post { dispatchError(handlerId!!, url) }` — an unconditional fire carrying
   state read while the batch was still open). With the batch open, ordinary two-patch
   frames put patches BEHIND the failure in the same batch: `src = <bad>` … `RemoveNode`
   dispatches for a **purged node**; `src = <bad>` … `src = <good>` delivers the **old
   src's error into live user code** — the exact stale-callback class the generation guard
   was built to kill, at the one site it was built for. **Latent on Android** (Coil has no
   deterministic synchronous mid-batch failure — its memory cache proves only synchronous
   SUCCESS; a 404 is never cached), **proven LIVE on iOS** via the nil-URL path
   (`URL(string:) → nil` terminates inside `applyBatch` by construction, `3389a84`). The
   fix is **fire-time re-decision on both shells**: `decideAndDispatchError` posts ITSELF,
   never a verdict — the deferred turn re-enters the decision with fresh
   generation/view/handler facts (iOS first via mid-flight correction; Android mirrored
   back in `8ef64be` — `ImageErrorDispatch.kt` untouched, the pure table always asked for
   fire-time facts; the bug was feeding it decision-time ones). Both adversarial same-batch
   orderings (failure-then-remove → ZERO dispatches; failure-then-new-src → the superseded
   error drops while the new source paints) are pinned end-to-end on iOS — a deferred
   dispatch replaying a decision-time capture passes the defer test and fails both, which
   is why they exist.

2. **Gate 3 review I-1 — the decision TABLE's row order was normative, and wrong.** Both
   pure tables (Swift `BnImageErrorDispatch` / Kotlin `ImageErrorDispatch`, deliberate
   literal twins) consulted `handlerAttached` before `applyingBatch` — so the handler row
   reached a **permanent DROP from mid-batch state**. At mount the wire order is `src`
   (seq 24) → … → attach `error` (seq 27) in ONE batch: iOS's synchronous nil-URL failure
   asked "handler attached?" three patches early and dropped — `OnError` never fired for
   `<BnImage Src="<unparseable>" OnError="…">` at mount — while Android's post-batch
   failure for the same markup DISPATCHED. A cross-shell parity break, design decision 2
   broken by name — and a test comment had recorded the iOS drop as *"matching Android"*:
   the divergence had been transcribed as parity. Fixed by the **row swap in BOTH tables**
   (`20afbcb`): liveness → applyingBatch (DEFER) → handlerAttached (DROP) → dispatchNow.
   Mid-batch failures now always DEFER and the fire-time re-decision (finding 1's
   machinery, already on both shells) reads the SETTLED handler state — attached-by-
   batch-end dispatches exactly once post-batch, never-attached still drops. Only the
   liveness DROP stays permanent inside a batch (identity and generation never
   un-supersede). Pinned by the iOS attach-later-in-batch end-to-end test AND the JVM
   table rows — both lanes.

## The .NET equivalent-mutant finding (honestly recorded)

The design's first .NET mutation — drop the `HasDelegate` guard, attach `onerror` always —
**leaves the suite GREEN: an equivalent mutant.** `RenderTreeBuilder` itself omits
delegate-less element `EventCallback`s, so the unbound-zero-wire-presence invariant is
doubly held and the guard's removal is unobservable. The normative behavior stays pinned
either way by the zero-wire-presence golden
(`BnImage_OnErrorUnbound_EmitsNoAttachAtAll`); the guard is kept because it is the
documented `BnScroll` shape and does not rely on a framework detail. Recorded as what it
is — a mutation expectation the framework satisfies for us — not silently reworded.

## Platform findings

1. **A non-numeric port is what it takes to make `URL(string:)` say nil (Gate 3).**
   Foundation's lenient parsing percent-encodes mere garbage into a valid URL — the defer
   test's unparseable src had to be a **structural** violation (a non-numeric port), and
   the suite asserts the nil-parse as its own named **precondition** so a future Foundation
   change fails the test loudly at the premise, not mysteriously at the assertion.
2. **Defer is observable at DELIVERY, not by ordering (Gate 3).** The main queue's drain
   can run a deferred block before the test regains control — "no dispatch yet, then one"
   is not a reliably observable ordering. `isApplyingBatch` joined the test-only
   accessors: the pin is "the dispatch that arrived saw no open batch".
3. **The slow-window vs download-timeout state-layout deviation (Gate 3, reviewed,
   coverage-equivalent).** iOS's `BnImageLoader.downloadTimeout` is 5 s (a 6.3 decision:
   an uncancelled held request dies on the OUTCOME, not the gate — Android gets 10 s free
   from OkHttp), and `/slow.png`'s idle clock runs from the mount batch. So the iOS demo
   suite's state 2 asserts ONLY what needs the request still open, and every
   slow-independent assertion moves after `releaseSlow` — the proof content is unchanged;
   the held window stays inside the timeout **by construction** instead of by runner luck.
4. **The sync-gate header miscounted seven; the true count is EIGHT (Gate 1 review,
   must-fix `1dee252`).** The `/imagepolish` header's arithmetic didn't even agree with
   itself (6+2=7); the page carries 8 src-bearing nodes — slow(1) + missing(1) + 4× fixed
   + missing(1) + intrinsic(1). Caught **before the Gate 3 device suite transcribed it**
   (a gate awaiting 7 terminal callbacks races the 8th request — the exact flake class the
   counted gate exists to prevent); Gate 2's Android suite had already derived 8 from the
   page itself and recorded the discrepancy in its KDoc.

## The honest ledger (deferred with names, not dropped)

- **Density-aware assets (`@2x`/`srcset`) — ledgered AGAIN**, and the problem has its
  name: the wire has no scale channel; mode is *paint*, density is *measurement*, and the
  two never touch. **Trigger:** the first bundled/local-asset story (M8+ packaging),
  the only place RN itself ever solved this.
- **`OnLoad`** — not in DoD #6; the sync gate stays loader-callback-based. It is
  `OnError`'s shape exactly — a day's work once a customer wants it.
- **`PlaceholderSrc`** (a placeholder IMAGE) — decision 1(b)'s named double-load problem.
- **`repeat` mode / GIF / SVG** — no customer / still the 6.3 decode ledger.
- **Retry-in-shell** — resolved as designed: failure does not retry, and a retry is now
  **app-authorable** (re-set `Src` from `OnError`) — this feature IS that old ledger
  line's resolution.
- **The shared placeholder null-clear/recolor test gap** — neither device suite pins
  `placeholderColor → null` clearing a painting placeholder, nor a recolor while
  in-flight; both shells, so it is a **matched pair** — to be added together when either
  suite is next touched.
- **The `scrollDiagnostics` rename** (Gate 2 review M-1) — the test-only diagnostics
  accessor now carries modal and image-contentMode diagnostics on both shells; renaming it
  is 7.6's hygiene pass.
- **`BnClipboardTests` flake-hardening** — the real-`UIPasteboard` poll's environmental
  sensitivity (the one flake above) goes to the 7.6 audit list.
