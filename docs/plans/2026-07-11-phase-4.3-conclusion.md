a fresh win-x64 NativeAOT publish + a console widget-tree dump over the real

**fast-restart, not hot-reload** — and the numbers below say what a restart

## The measured numbers (the DoD #5 centerpiece)

Dev machine, warm (`devloop.ps1 -Once`, one `BlazorNative.Components` file
touched before each measured run; every number reproduced across ≥2 runs):

| # | Stage | Warm | First/cold run |
|---|---|---|---|
| a | Incremental win-x64 NativeAOT publish | **8.2–9.4 s** | 14.4 s |
| b | PreviewHost boot-to-tree (dll-load 226–323 ms + init ~35 ms + mount ~30 ms) | **~0.3 s** (321 ms) | 412 ms |
| c | Full JVM-lane cycle (publish → tree on screen) | **9.9–11.2 s** | 16.5 s |
| d | Full ADB-lane cycle (publish → install → launch → mounted) on the AVD | **14.1–14.5 s** (publish ~8.8 + install ~3.6–4.4 + launch→mounted ~1.6–2.1) | 16.2 s |

**The NativeAOT publish dominates both lanes** (~85 % of the JVM cycle, ~60 %
of the ADB cycle); everything the tooling owns downstream of it is seconds or
less. The design's gradle-overhead contingency (running PreviewHost via direct
`java -cp` if the daemon startup swamped the cycle) was **not taken**: the
warm gradle leg is 1.7–2.4 s against an 8–9 s publish — evidence said no.

## As-builts (decisions the code now embodies)

- **`TreeSnapshot`** (`src/BlazorNative.Jni/src/main/kotlin/io/blazornative/jni/TreeSnapshot.kt`,
  pure JVM, zero Android imports) mirrors WidgetMapper's placement semantics:
  bucket-local `InsertIndex` (−1 append, 0 valid front, out-of-range **throws**
  — the N3-pinned `addView` strict-placement parity), unknown-parent → root
  bucket, text-child-on-leaf collapse (including the unknown-type TextView
  fallback), last-wins re-attach, identity-purged subtree removal. Deliberate
  divergences, documented in the class KDoc: no CommitFrame batching to a main
  looper (the snapshot is single-threaded — the boundary patch is a no-op),
  and unknown/removed node ids are skipped **silently** where WidgetMapper
  `Log.w`s (the console tree itself is the diagnostic). 13 model-level tests
  pin all of it with synthetic frames — no JNA in the loop.
- **PreviewHost stage timings — the init-probe judgment:**
  `BlazorNativeRuntime.start()` fuses init+register+mount, so the `init` stage
  is measured by an explicit `blazornative_init` probe *before* the canonical
  `start()` boot. Init is idempotent (the documented recreation contract), so
  start()'s internal re-init is a no-op and the `mount` stage is honestly
  register + mount + first-frame delivery. Boot stays on the canonical path.
- **PreviewHost exit-code honesty (review N1):** an `onError` sink flags any
  dropped frame during mount; the dump still prints (marked PARTIAL on
  stderr) but the process exits 1 — the loop never reads a half-applied tree
  as success. Placement: `main/kotlin` beside the runtime classes it drives
  (~1 KB of dead code in the APK, accepted over a second compilation
  arrangement; documented in the file header).
- **Gradle daemon KEPT** (design offered `--no-daemon` or verify-exit):
  `runPreviewHost` is a `JavaExec`, which always forks a fresh java process —
  the daemon never loads `BlazorNative.Runtime.dll`, and PreviewHost exits
  after its dump, so nothing lingers to hold the dll against the next publish.
  The script verifies exit codes instead; the warm daemon is what keeps the
  gradle leg at ~2 s.
- **Watcher obj/bin guard:** the publish itself regenerates `obj/**/*.cs`, so
  the FileSystemWatcher handler drops any path matching `\obj\|\bin\` —
  without it the loop retriggers itself forever. Verified live: one save →
  exactly one re-cycle, then quiet.
- **devloop.ps1 fail-fast surface:** dll-lock hint on publish failure (keyed
  on `used by another process|MSB3026/7|LNK1104`, naming the JavaExec-forks
  fact so nobody chases the daemon), missing-adb hint (`setup.ps1 section 7`),
  no-device hint (points at `test-android.ps1`), `am start` output/exit
  checked for immediate `Error:`/exception instead of waiting out the 90 s
  marker timeout, boot-marker timeout dumps the last logcat lines.
  `ANDROID_NDK_ROOT` is autodetected (the bionic publish reads
  `source.properties` from it — mirrors ci.yml's `NDK_PIN`).

## Honest boundaries

- **Fast-restart, not hot-reload — three stacked reasons, any one sufficient:**
  (1) NativeAOT binaries cannot hot-patch — there is no metadata/IL to swap;
  (2) Windows locks a loaded dll, so a warm process pins the very file the
  next publish must overwrite; (3) JNA's `Native.load` binding is
  process-lifetime in practice. On (3) the design doc overstated slightly:
  `NativeLibrary.dispose()` **does exist** in JNA — but disposing and
  re-loading in a warm JVM would still be blocked by (1) and (2), so restart
  is the correct design regardless, not a workaround. This conclusion corrects
  the design doc's "no unload API" wording; the restart decision stands on
  the other two legs.
- **The ADB lane republishes bionic-x64 only** (the AVD lane, per design). A
  physical arm64 device needs a separate `linux-bionic-arm64` publish or its
  staged `.so` goes stale — the script help says so.
- **Review nit N2 (platform-info note cosmetic) consciously skipped** — the
  `phase-4.3-previewhost` init-probe note string is diagnostic-only.
- The loop measures the **mount path**, not interaction: PreviewHost dumps
  the mounted tree and exits. Driving events through the console host is
  DevTools territory (DoD #6, Phase 4.4).

## Carryovers

- **Desktop widget window = M6.** PreviewHost is deliberately a console
  surface; a real desktop widget host (rendering the tree into actual
  controls) is ecosystem work, not inner-loop work.
- **Any future loop speedup lives in the .NET publish, not the tooling.** The
  measured split is unambiguous: ~8–9 s of every cycle is ILC. Incremental-
  link/ILC-caching investigations belong to the .NET side; the script and
  PreviewHost legs are already sub-2.5 s combined.
- **The dll-lock/no-hot-patch constraint is permanent**, not a TODO: recorded
  here, in the README's Dev experience section, and in devloop.ps1's header —
  with the `NativeLibrary.dispose()` wording correction above as the precise
  statement of record.
