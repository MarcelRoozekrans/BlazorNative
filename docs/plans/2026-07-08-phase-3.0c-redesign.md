## Why the re-plan

Phase 3.0b Gate 4 RED'd because `runtime.linux-bionic-*.Microsoft.DotNet.ILCompiler` does not
exist for .NET 10 (only an abandoned 8.0.0-preview.6 from June 2023). The conclusion doc pivoted
to a dual runtime: Mono-AOT (`dotnet/android` workload) for Android + NativeAOT for desktop.

Fresh research (2026-07-08) overturns both halves of that pivot's premise:

1. **The Gate 4 RED has a documented bypass.** Only the RID-specific ILCompiler *package* is
   missing. Microsoft actively ships NativeAOT **runtime packs** for `linux-bionic-*` and
   `android-*` in .NET 10 — `Microsoft.NETCore.App.Runtime.NativeAOT.linux-bionic-arm64`
   10.0.3–10.0.9 (serviced as recently as 2026-06-09), plus all .NET 11 previews. The
   community-proven recipe ([josephmoresena/NativeAOT-AndroidHelloJniLib](https://github.com/josephmoresena/NativeAOT-AndroidHelloJniLib))
   publishes with `-p:PublishAotUsingRuntimePack=true -p:DisableUnsupportedError=true` plus NDK
   linker properties (`SysRoot`, `CppCompilerAndLinker`, `ObjCopyName`): the **host** ILCompiler
   compiles against the **target** runtime pack. Output is a plain `.so` with
   `[UnmanagedCallersOnly]` exports, loadable from Kotlin — and the recipe works from a Windows
   host directly (no WSL). Caveat: publicly proven on .NET 9; the .NET 10 packs exist but the
   recipe is unconfirmed on 10. That confirmation is exactly what Gate 1 buys.

2. **The Mono-AOT leg is far worse than the pivot assumed.** `net10.0-android` Mono-AOT cannot
   produce a standalone JNA-loadable `.so`: AOT images are data consumed by `libmonosgen-2.0.so`,
   which must be initialized via a custom JNI embedding shim (`monovm_initialize` +
   `mono_jit_init`; see `dotnet/runtime` `src/tasks/AndroidAppBuilder/Templates/monodroid.c`).
   Mono "library mode" — exactly our scenario, lazy init from `[UnmanagedCallersOnly]` exports —
   was designed upstream (dotnet/runtime#79377) but never shipped. The 3.0b conclusion's
   "likely smallest-delta path" framing was wrong: it is heavy custom embedding work that strains
   the C-ABI-parity requirement.

3. **NativeAOT is the strategic direction regardless.** .NET 11 Preview 4 (2026-05-13) made
   CoreCLR the default MAUI Android runtime; Android NativeAOT is "actively underway"; NativeAOT
   bionic/android runtime packs ship in every .NET 11 preview; GA expected November 2026. There is
   still no official standalone-`.so` story from Microsoft on any timeline — the bypass recipe is
   the only path, and it builds on the same officially-serviced runtime packs the eventual
   dotnet/android path uses.

## Decisions locked during this brainstorm (2026-07-08)

| Decision | Choice |
|---|---|
| Phase shape | **Split, risk-first.** 3.0c = Android runtime spike only; 3.0d = wire protocol + renderer; 3.0e = WASM-era collapse. 3.0b's lesson: small gated phases make REDs cheap. |
| Shell purity bar | **C-ABI parity.** Kotlin calls the same `blazornative_*` exports with the same structs as desktop. One idiomatic bootstrap call hidden in a small Kotlin helper is acceptable; app-model entanglement is not. |
| Spike target | **NativeAOT bypass first.** Mono embedding is demoted to escalation-only. |
| RED fallback | **Escalate embedding effort.** One follow-up Mono-embedding spike before conceding; wasmtime-on-Android is the last resort; parking Android is rejected. |

If 3.0c goes GREEN, the dual-runtime pivot is cancelled: one NativeAOT runtime on every platform,
the 3.0b desktop foundation (Exports.cs, NativeBindings.kt, BootSmokeNativeTest) extends to
Android unchanged, and Mono never enters the codebase.

## Phase 3.0c — gate structure

**Timebox: 3–4 days.** RED-on-ambiguity rule from the 3.0b/componentize retrospectives applies
throughout: one diagnostic iteration per failure, then call the gate.

### Gate 1 — Minimal scaffold proves the toolchain (½ day)

A 3-line class library with one `[UnmanagedCallersOnly]` export, published for
`linux-bionic-x64` AND `linux-bionic-arm64` with:

```
dotnet publish -c Release -r linux-bionic-x64 \
  -p:PublishAot=true \
  -p:PublishAotUsingRuntimePack=true \
  -p:DisableUnsupportedError=true \
  <NDK props: SysRoot, CppCompilerAndLinker, ObjCopyName → NDK 26.3.11579264>
```

- **Windows-direct first.** The recipe claims Windows-host support. If the ILC→NDK link fails on
  Windows, one retry inside the existing `blazornative-ubuntu` WSL distro before calling the gate.
- **GREEN:** ELF64 `.so` for both ABIs with the export symbol visible (`llvm-readelf`/`nm`).
- **Purpose:** isolates toolchain risk from our dependency graph — single-variable hypothesis,
  same discipline as 3.0b. Also answers whether WSL stays in the picture.

### Gate 2 — NativeHost publishes for bionic (½–1 day)

Same recipe applied to `BlazorNative.NativeHost.csproj` (full Blazor + Renderer graph).

- **GREEN:** both ABIs produce `.so` files; 3 boot exports present; trim warnings triaged —
  expect the same 4 accepted IL2072s from Phase 3.0a, zero new of-our-code warnings.
- **Runtime-pack version pinned explicitly in the csproj.** `PublishAotUsingRuntimePack` is
  unsupported; pinning defends against servicing drift. The exact working
  SDK + ILCompiler + runtime-pack + NDK combination is documented in setup.ps1.

### Gate 3 — Pure-JNA boot smoke on AVD (1 day)

`.so` files into `jniLibs/x86_64` + `arm64-v8a` (Gradle copy task, same pattern as Phase 2.2's
`copyJniLibs`). New `BootSmokeNativeAndroidTest` instrumented test does
`Native.load("blazornative")` through the **existing, unchanged** `NativeBindings.kt` and asserts
`Init` Status=0 + version string on the AVD.

- **GREEN:** test passes. This is the C-ABI-parity proof — zero Java-side bootstrap, identical to
  the desktop JVM path.

### Gate 4 — Trim probes inside the .so (½–1 day)

The Phase 3.0a carryover checklist (the 4 accepted IL2072 paths: `[Parameter]` binding, cascading
values, `[Inject]` DI) has so far only run on the host CLR — 3.0b Gate 2 explicitly deferred
in-dll validation. Add a **temporary** `blazornative_run_trim_probes` export that mounts the three
existing probe components (`ParameterProbe`, `CascadingProbe`, `InjectProbe` from
`BlazorNative.Runtime.Tests`, relocated or linked into NativeHost behind the export) inside the
NativeAOT `.so` and returns pass/fail + message per probe.

- **GREEN:** all 3 probes pass on-device (AVD x86_64) and on desktop win-x64 (same export via
  the JVM test for cross-validation).
- This is the load-bearing trim validation the whole NativeAOT bet rests on — done **before**
  any 3.0d protocol investment. The export's fate (delete vs. keep as a diagnostics surface) is a
  3.0d decision.

## Escalation ladder (RED path)

1. **Gate 1/2 RED on Windows** → one WSL retry (`blazornative-ubuntu` already provisioned).
2. **Bypass fundamentally RED on .NET 10** (e.g. runtime pack rejects host ILC) → stop, write
   conclusion doc, re-plan a **Mono-embedding spike** as the escalation phase: monodroid.c-style
   JNI shim + `libmonosgen` + BCL + AOT images, bootstrap hidden in `BlazorNativeRuntime.kt` per
   the C-ABI-parity bar's "one idiomatic init call" allowance.
3. **Mono embedding also RED** → wasmtime stays on Android; 3.0d proceeds desktop-only; collapse
   (3.0e) is deferred, not cancelled — revisit at .NET 11 GA (Nov 2026).

## Downstream phases

- **Phase 3.0d — Native wire protocol + renderer + WidgetMapper rewire.** The old 3.0c protocol
  design carries over intact (typed `BlazorNativePatch` structs, `FrameArena`,
  `NativeRenderer.DispatchFrame` rewrite, frame/event callback registration, `blazornative_mount`,
  Kotlin struct reads): built on desktop NativeAOT first, then validated on the Android `.so`.
  Now targets **one** runtime — no dual-surface sync cost.
- **Phase 3.0e — WASM-era collapse.** Deletes `src/BlazorNative.WasiHost/`, `vendor/wasmtime/`,
  `tests/BlazorNative.Wasi.Tests/`, wasmtime Kotlin bindings (`WasmtimeBindings.kt`,
  `WasiHost.kt`, `FrameStreamParser.kt`, sealed-class `RenderFrame.kt`, `MobileBridgeHost.kt`),
  wasi-sdk/wasmtime/cargo setup.ps1 sections, stale `maui-android` refs (Makefile:120, README:73);
  renames `BlazorNative.NativeHost` → `BlazorNative.Runtime`. If Gate 1 proved Windows-direct,
  the WSL distro section of setup.ps1 retires too.
- **ROADMAP/MILESTONE on 3.0c GREEN:** pivot note superseded — "NativeAOT-per-ABI resurrected via
  runtime-pack bypass." Watch item: swap to the official dotnet/android NativeAOT path when
  .NET 11 ships it; the bypass is a bridge, deliberately kept thin (all bypass-specific MSBuild
  lives in one csproj + one setup.ps1 section).

## Risks

| Risk | Mitigation | If it fires |
|---|---|---|
| `PublishAotUsingRuntimePack` undocumented; recipe unconfirmed on .NET 10 | Gate 1: ½-day minimal scaffold before anything else | Escalation ladder step 2 |
| Servicing release breaks the bypass | Pin runtime-pack version; document working combo in setup.ps1 | Re-pin; treat as maintenance, not RED |
| Accepted IL2072s materialize at runtime on-device | Gate 4 probes, before protocol investment | Upstream-facing fix (dotnet/aspnetcore annotations) or targeted rooting; evaluated then |
| NDK 26.3 vs recipe's r26b | Same major version | One Gate 1 diagnostic iteration (NDK prop tweak), not RED |
| Windows-host link path broken for bionic | WSL retry is one command away; distro exists | Keep WSL section in setup.ps1 through 3.0e |
