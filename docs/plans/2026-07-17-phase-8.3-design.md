# Phase 8.3 Design — `dotnet new blazornative` (M8 DoD #4)

**Date:** 2026-07-17
**Milestone:** M8 — Developer Ecosystem (**DoD #4** — the template: the .NET app + the
Android shell, runnable end-to-end; creation → build validated on CI; iOS documented
against the reference shell)
**Predecessor:** Phase 8.2 — the release pipeline, manual go
([conclusion](2026-07-16-phase-8.2-conclusion.md), PR #104; DoD #3 closed at `01984f3`)
**Named inputs (MILESTONE.md DoD #4, from Phase 8.0):** (a) the template's csproj **must**
carry `<TrimmerRootAssembly Include="<AppAssembly>" />`; (b) the `EnsureRegistered`
guard-order flip/note (8.0 Gate 1 review M-1).

---

## The one sentence the whole phase hangs on

**A fresh `dotnet new blazornative` app has EXACTLY the shape nativelib ILC trims
silently — and unlike Phase 8.0, nobody will be watching when it happens.**

8.0 met this failure with a human at the console: the publish head moved, the count went
`4 → 0`, the binary shrank 400KB, the page names vanished, and the **stop-and-analyze
rule** caught it before it shipped. A template user gets none of that. They get a green
build, a green APK, an app that installs, and **rc 1 at first mount** — the first thing
they ever see this framework do is fail, silently, for a reason nothing in their project
names.

So this phase is not "write a template." It is: **make a generated app clear the same bar
the repo's own app clears — 4 IL2072 + 9 exports + the page-presence probe — and prove it
on GENERATED output, in the required lane.** Everything else here is in service of that
sentence, or is the Android shell that makes the app worth generating.

## The reference as it actually is (verified this phase, not recalled)

Ten facts, read out of the tree rather than remembered. Several of them changed decisions
below.

1. **The guard-order bug is real and is exactly as M-1 described it.**
   `samples/BlazorNative.SampleApp/SampleAppPages.cs:147-151`:
   ```csharp
   if (s_registered)
       return;
   s_registered = true;
   BlazorNativeApp.RegisterPages(All);
   ```
   The guard is set **before** the call. A throwing registration leaves the guard set and
   every retry silently no-ops.
2. **The sample names itself the template's source.** `SampleAppPages.cs:29-31`: *"`All`
   is deliberately PUBLIC: it is the drift test's target, **the 8.3 template's worked
   example**, and sample apps are documentation."* And the csproj's trim comment
   (`:117`): *"8.3's template must carry this line."* The reference knows what this phase
   is going to do with it.
3. **The app roots itself by a hardcoded literal** —
   `<TrimmerRootAssembly Include="BlazorNative.SampleApp" />`. A template cannot copy that
   literal; the assembly name is the user's choice.
4. **`$(AssemblyName)` evaluates correctly in a csproj-body `ItemGroup`** — proven here,
   not assumed (`dotnet msbuild -getItem:TrimmerRootAssembly` → `Identity: trimprobe`).
   **And it survives explicit blanking**: setting `<AssemblyName></AssemblyName>` in the
   body still yields `trimprobe`, because MSBuild evaluates *all* properties (including
   the SDK's bottom-import re-default) before *any* item. The spelling is robust.
5. **A bogus root name passes MSBuild evaluation with no error and no warning** — proven
   here (`<TrimmerRootAssembly Include="ThisAssemblyDoesNotExist" />` →
   `Identity: ThisAssemblyDoesNotExist`, clean). MSBuild does not validate the name. **What
   ILC does with an unresolvable root is UNPROVEN** (U3) — and it is the whole game: if ILC
   ignores it silently, a mis-substituted root is the 8.0 failure with a green build.
6. **The publish gates' page probe already names the fix.** `ci.yml:922-934` — three page
   name-strings asserted present in the win-x64 native image, failure message: *"the
   SampleApp module was trimmed (is TrimmerRootAssembly still in the sample csproj?)"*.
   The tripwire this phase must re-aim at generated output already exists, verbatim.
7. **`build-test`'s runner has an Android SDK** (`ANDROID_HOME`, `sdkmanager`, the pinned
   NDK `26.3.11579264`), and `android-build` runs Gradle on `windows-latest` today. Gradle
   on `build-test` is available, not a new capability.
8. **`build-test` already publishes BOTH bionic RIDs.** `ci.yml:936/968`. A generated
   Android project's `verifyNativeAssets` chain wants a `linux-bionic-*/publish/BlazorNative.Runtime.so`
   — and one already exists on the runner. **This is what makes the Android half of the CI
   lane cost zero extra publishes** (decision 8).
9. **The sample's publish head has two dependencies a generated app does NOT inherit:**
   `<Import Project="..\..\build\BionicNativeAot.targets" />` (the vendored NDK
   cross-compile shim — without it, no bionic publish) and the repo-root `global.json`
   (SDK band `10.0.3xx`, which the csproj comment names as the pin the ILC host relies on).
   Neither travels with a `dotnet new` app. Both are silent-ish failures.
10. **The iOS shell's link is a directory, not a path.** `project.yml` references
    `$(SRCROOT)/vendor/...` and **names no publish directory and no SampleApp anywhere**.
    The recipe that populates `vendor/` exists **only as ~90 lines of bash inside two
    workflow steps** — there is no script, no Makefile target, and no doc. This is the
    single fact decision 9 is built on.

## Decisions (⚠ marks the veto-flagged judgment calls)

### 1. Shape and home: a template pack in `templates/`, which does **NOT** join the shipped six ⚠

**The template is a real `dotnet new` template pack** — `templates/BlazorNative.Templates/BlazorNative.Templates.csproj`,
`<PackageType>Template</PackageType>`, `IncludeBuildOutput=false`, `NoDefaultExcludes=true`
(template content legitimately includes dotfiles pack excludes by default).

**It does not live under `src/`, and it does not join the six.** This is not squeamishness;
`src/` membership is load-bearing:

- `PackagePurityTests.TheShippedSet_IsExactlyTheSrcCsprojs` **proves `src/` holds exactly
  the six shipped csprojs** — and that proof is what *licenses* `src/Directory.Build.props`
  to be the packaging home at all (8.1's finding: the 4.5 "per-csproj by design" rule died
  by construction). A seventh csproj under `src/` reds that pin and un-licenses the props.
- **The six's pins are assembly-shaped.** Type purity off the PE, a sentinel type per
  package, snupkg pairing, `ResolveAssemblyPath`. A content-only template pack has no dll,
  no `lib/`, no TFM, and satisfies **none** of them. 8.1's own mutation proved the shape:
  `src/BlazorNative.Seven` made `ResolveAssemblyPath` red *loudly on the absent dll*.
  Joining would mean special-casing it in five pinned places — and **a special case is a
  hole in a pin**, which is precisely the 8.1 four-copies lesson.

**So: `templates/` is a new top-level sibling of `src/`, `samples/`, `tests/`, `build/`.**
The pack goes to a **separate feed** — `artifacts/templates` — so `consumer-smoke.ps1`'s
`6 nupkg + 5 snupkg` stays an **exact** count that means something rather than "6 or 7,
depending."

**It does not publish this phase.** `release.yml`'s `push` enumerates `artifacts/packages`;
the template pack is not in it, so **no release-machinery edit is required and 8.2's
verdict is untouched**. This is consistent, not a gap: **DoD #4 requires "creation → build
validated on CI", not publication**, and M8's own out-of-scope says the push is the owner's
Release. The template is publish-**ready** in the same sense the six are.

> ⚠ **VETO FLAG.** The consequence, stated plainly: **`dotnet new install BlazorNative.Templates`
> does not work for a stranger until the pack publishes** — and it is not in the release
> path, so the M8 Release would not carry it. If the owner wants the template public with
> the first Release, the terms are named in the ledger (a preflight id row, the feed
> decision, and the smoke's shape questions). It is a **decision about the release**, not
> about the template, and it is a small edit — but it is the owner's, not this phase's.
>
> Note the same gate already binds the app itself: **a generated app cannot restore until
> the six publish.** The template's usefulness to a stranger is gated on exactly the Release
> everything else in M8 is gated on. Nothing here is worse off than the packages are.

### 2. The short name is `blazornative`; the identity is `BlazorNative.App`

Verified locally: **nothing named `blazornative` resolves** (`dotnet new blazornative` →
*"No templates or subcommands found"*), and the neighbourhood is clear — `blazor` (Blazor
Web App) and `blazorwasm` are Microsoft's, `maui-blazor` is MAUI's. No collision with
anything installed.

**UNPROVEN (U1): nuget.org.** `dotnet new search` failed in this environment — DNS is
unavailable (`Host is onbekend. (go.microsoft.com:443)`), the same sandbox condition
8.2's offline arm hit mid-gate. Verify with `dotnet new search blazornative` at
implementation.

**Why U1 is acceptable:** a short-name collision is **loud, not silent** — two installed
templates sharing a short name make `dotnet new blazornative` an ambiguity error naming
both packages, with the full identity available as the disambiguator. It cannot silently
generate the wrong thing.

### 3. The version problem: the template's literal is a **pinned mirror**, not a second source ⚠

The generated csproj must name a version for the packages it references. Three candidates,
and the repo has already decided this class of question twice.

- **A wildcard (`1.0.0-preview.*`) — REJECTED, and it is the dangerous option.** It still
  carries `1.0.0-preview` (a partial copy that drifts at the `1.0.0` boundary), it makes a
  consumer's build change under them on a preview line, and it needs the network to
  resolve. But the disqualifying reason is sharper: **the CI check restores from a local
  feed holding exactly one version, so the wildcard always resolves and the drift is
  invisible on CI.** That is a vacuous pass — a check that is green because it cannot see.
  This repo has that lesson written in its own code (`TypeNamesOf`: *"a pin that cannot see
  its subject must never pass vacuously"*) and it is 8.1's headline. Not twice.
- **A pack-time rewrite of `template.json`'s `defaultValue` from the props — REJECTED.**
  It works, but it buys removal-of-a-literal at the cost of a generation step, and it makes
  the repo's checked-in template content *not* the thing that ships. Machinery to avoid a
  test the repo writes in its sleep.
- **A literal + a drift pin — TAKEN.** The template content carries `1.0.0-preview.1`; a
  test in `build-test` parses it out of the checkout and asserts equality with
  `src/Directory.Build.props`. This is the house pattern, three times over:
  `RouteTableDriftTests` parses Kotlin, `ShellStyleTableDriftTests` parses Kotlin *and*
  `.mm`, and `PackagePurityTests` tooth 3 parses `consumer-smoke.ps1` *and*
  `ConsumerSmoke.csproj`. `build-test` is the one required lane where every file is
  checkout-visible — the standing rule.

**The prompt's objection — "it lives inside a nupkg, so it drifts silently" — is exactly
what the pin answers.** The pack is built **from** the pinned content, so pinning the
content pins the nupkg. There is no third place for the literal to hide.

**THE NORMATIVE FRAMING, and it matters:** 8.1's law is *ONE version literal*. This does
not break it, because **the template's literal is a MIRROR, not a source** — the same
relationship `MainActivity.kt`'s `DEEP_LINK_COMPONENTS` has to the page manifest: a copy
that exists because it *cannot* be a derived view, held equal by a pin that reds. The props
remains the one source. A bump stays a one-line edit **plus a red test that tells you the
second place**, which is the entire point of a mirror over a silent copy.

**The pin's shape** (`TemplateDriftTests`, Runtime.Tests):

- Enumerate **every** BlazorNative version literal in the template tree — the generated
  csproj's `PackageReference Version=`s, `template.json`'s `defaultValue`, and the template
  **pack's own `<Version>`** — and assert each equals the props.
- **The pack's own version needs this pin specifically**: it lives outside `src/`, and
  `PackageVersionPinTests.ShippedCsprojs()` enumerates `src/` **only**. This is the 8.1
  "Seven" trap exactly — a packable project whose version is guarded by nothing. Named, and
  closed here.
- **Non-vacuity, asserted**: zero literals found → **RED**, not a pass over an empty read.
  (`ShippedCsprojs()`'s own rule, verbatim.)

**The referenced set:** the template's csproj mirrors the sample's three references —
**Runtime, Components, Analyzers** (Core/Renderer/Http arrive transitively via Runtime;
the sample proves the shape). The pin asserts the referenced ids are a **subset of the
shipped six** (enumerated from `src/`, never rostered), so a typo'd or foreign id reds.

> ⚠ **Flagged, lightly:** a `template.json` symbol lets a user override
> (`--BlazorNativeVersion 1.0.0-preview.2`). It adds a second literal *inside the template*
> — both are covered by the "every literal" pin above, so the cost is one more line the pin
> reads. Included; drop it if the reviewer prefers the smaller surface.

### 4. THE HEADLINE — the trim line, and what actually PROVES it ⚠

**Where it lives:** the generated csproj, in the same `ItemGroup` as
`UnmanagedEntryPointsAssembly`, carrying a distilled version of the sample's comment (which
is excellent and explains *why*: exports live in the referenced Runtime assembly, so
nothing in the image references app code, so ILC trims the whole app, module initializer
included).

**The spelling: `<TrimmerRootAssembly Include="$(AssemblyName)" />`** — not a
sourceName-substituted literal.

Reasoning from the facts, not from taste:

- Fact 4: `$(AssemblyName)` evaluates correctly in the csproj body, and **survives even
  explicit blanking**. The spelling is robust.
- **It removes the substitution dependency entirely.** The literal form is correct *iff*
  `template.json`'s `sourceName` is configured right and substitution fires. `$(AssemblyName)`
  is correct for **any** `-n` the user picks, including names substitution might mangle.
- Fact 5 is why this matters: **a wrong root name passes MSBuild with no error at all.** So
  the literal form's failure mode is "the csproj looks right, the root names nothing, ILC
  trims the app, rc 1 at first mount." That is the 8.0 failure, arrived at from a new
  direction.

> ⚠ **VETO FLAG — this DIVERGES from the sample's hardcoded literal.** Decision 5 is where
> that divergence gets resolved; the cheap option (make the sample use `$(AssemblyName)`
> too) is available and is a 1-line publish-head change the gates re-arbitrate anyway. The
> design's preference is to leave the sample's literal alone (it is correct, it is
> readable, and its comment is load-bearing documentation) and to state the template's
> different spelling **with its reason in the comment** — the template's assembly name is
> not knowable when the file is written, and that is a real difference, not an
> inconsistency.

**Now the real question: what PROVES a generated app still has it?**

**A text pin on the template content is NOT sufficient, and the reason is the whole phase.**
A text pin proves *the line is present*. It cannot see:

- substitution failure (irrelevant once `$(AssemblyName)` wins — but the pin cannot know
  that);
- **ILC silently ignoring an unresolvable root** (U3 — unproven, and unprovable without a
  publish);
- the template's csproj drifting from the publish head some **other** way that
  re-introduces the trim — a dropped `UnmanagedEntryPointsAssembly`, a broken RID
  condition, a missing `PublishAot`;
- the generated app failing to register for a reason that has nothing to do with the line.

**And, decisively: the line's absence in 8.0 was found by a PUBLISH, not by a read.** The
failure is silent by construction — 0 IL2072, green build, smaller binary, rc-1 on a
device. *"The line is in the file"* and *"the app survives ILC"* are different claims, and
only the second one is the one that matters.

**DECISION: the CI lane generates an app, publishes it, and asserts the 8.0 tripwire trio
on GENERATED output:**

| Tripwire | On the generated app |
|---|---|
| **Exactly 4 IL2072** | The trim law holds — the app is rooted and the libraries are still trimmable. A whole-module trim drives this to **0**. |
| **Exactly the 9 `blazornative_*` exports** | The exports still ride out of the referenced Runtime assembly, and `CanonicalizeNativeArtifactName` produced `BlazorNative.Runtime.dll` from `<AppName>.dll` — the frozen shell contract, proven for a name that is not `BlazorNative.SampleApp`. |
| **The page-presence probe** | **The app SURVIVED.** The starter page's name-string is in the native image. |

**One RID: `win-x64`.** It is the cheapest (no NDK, no bionic bypass), and it is the RID
the page probe already runs on in `ci.yml`. The trim law is a fact about the app's shape,
not about the RID — 8.0 verified the hypothesis on all four and got the same answer.

**The probe must be MUTATION-PROVEN, not assumed.** Its vehicle: **delete
`TrimmerRootAssembly` from the template → the generated publish must go `4 → 0` IL2072 and
the probe must RED.** This is a gate requirement, not a nice-to-have — the probe asserts the
*presence of a string in a binary*, which is exactly the shape that passes vacuously if the
string happens to appear in framework code. **If the starter page's name is not distinctive
enough to red under that mutation, the name changes until it is.**

### 5. The guard order: flip it in the template **and** in the sample ⚠

**The flip:**
```csharp
if (s_registered)
    return;
BlazorNativeApp.RegisterPages(All);
s_registered = true;   // only after the call SUCCEEDS
```
A throw now leaves the guard clear, so a retry **re-throws** — loud and repeatable —
instead of silently no-op'ing into an empty registry. It is safe in both directions:
`RegisterPages` validates before it registers, so a validation throw registers nothing; and
if it ever threw *after* registering, the retry meets the register-once law and throws
again. **Every path is loud. That is the improvement.**

**The sample gets the same fix, in this phase.** Arguments, weighed:

- **The sample IS the reference the template distills, and it says so about itself** (fact
  2). Divergence between reference and template **on the exact line a review flagged** is a
  trap that outlives the phase — a reader comparing them finds two answers and no way to
  tell which is normative. 8.4 is about to publish one of them as the worked example.
- 8.0's ledger says *"flip the order or note it **when 8.3 copies the pattern**"* — this is
  that moment. The flip is this phase's to make.
- **Cost is three lines**, and the change is **behavior-identical on every green path**
  (the order only matters when `RegisterPages` throws, which the sample's manifest never
  does).

> ⚠ **Flagged:** it touches the **publish head**, so the publish gates re-arbitrate — 4
> IL2072 + 9 exports + the page probe must be re-quoted at the gate. They run on every PR
> anyway; this is a bar to re-quote, not a risk to run.

**The honest problem: the flip is not testable in place.** `EnsureRegistered` is a static
once-guard over a static array; there is no seam to inject a throwing manifest. A
behavioural test would require a manifest that throws, which neither copy has.

**So the pin is a SOURCE-ORDER pin** (`TemplateDriftTests`): parse `EnsureRegistered`'s
body out of **both** copies — the sample's and the template's — and assert the
`s_registered = true` assignment appears **after** the `RegisterPages(` call.
Non-vacuity asserted (body not found → RED).

This is the house pattern, and there is direct precedent:
`AndroidSetStyleDispatch_HasAnArmForEveryYogaStyle` pins Kotlin **source shape** for the
same reason — the behaviour has no observable rc to demand.

**Stated honestly: a source-order pin proves the ORDER, not the SEMANTICS.** Here the
semantics *is* the order, and the alternative is nothing. It also does the thing this
decision is actually about: **it covers both copies in one test**, so the reference and the
template cannot drift apart on this line again.

### 6. The Android shell: verbatim-and-pinned, or parameterized — and nothing in between

**The problem, named first.** A template that copies ~14 Kotlin files creates a **second
copy of the shell that nothing compiles**. That is the AGP 9 incident's exact shape — *an
assertion can be green and structurally unreachable* — and it would rot silently, in a
nupkg, on a schedule nobody watches.

**The real fix is a published `.aar`, and it does not exist.** There is no Maven
publication, no feed, no version story. Building one is its own deliverable with its own
"manual go" question. **Ledgered to M9 with its terms named** — and the decision below is
deliberately shaped so that migration is cheap when it comes.

**So: copy, and PIN THE COPY.** The template's Kotlin is a **pinned mirror** of
`src/BlazorNative.Jni`'s — the same relationship `BnDemoFrameTables.kt` and
`BnDemoFrameTables.swift` have to each other, held by `ShellFrameTableDriftTests`.

**The trick that makes the pin EXACT: the shell's Kotlin package stays `io.blazornative.shell`.**
AGP's `namespace` (the R/BuildConfig package) does **not** have to match a source package,
and `applicationId` is a separate identity again. So:

- **Kotlin sources are byte-identical to the repo's** → the pin is a file comparison, not a
  normalized one.
- `build.gradle.kts` parameterizes `namespace` + `applicationId`.
- The manifest names the activity **fully qualified** (`io.blazornative.shell.MainActivity`)
  rather than `.MainActivity`, since the relative form resolves against `namespace`.

A user's shell classes living in `io.blazornative.shell` is correct — it is library code
they are not meant to edit — and it makes the eventual `.aar` migration a deletion.

**The split:**

| Fate | Files | Pin |
|---|---|---|
| **VERBATIM** (byte-identity) | `jni/`: `BlazorNativeRuntime`, `NativeBindings`, `NativeFrameAdapter`, `RenderFrame`, `ShellBridge`, `ItemsJson` · `shell/`: `WidgetMapper`, `YogaLayout`, `AndroidShellBridge`, `BnSpinner`, `ImageContentModeTable`, `ImageErrorDispatch`, `ImageRequestGuard` · `res/layout/main.xml`, `res/xml/network_security_config.xml` | **Byte-identical** to `src/BlazorNative.Jni`'s original. A shell change that skips the template REDS in `build-test`. |
| **TEMPLATE-OWNED** | `MainActivity.kt`, `build.gradle.kts`, `settings.gradle.kts`, `gradle.properties`, `AndroidManifest.xml` | Separately pinned — see below. |
| **OUT** | `PreviewHost`, `InspectorHost`/`InspectorServer`/`InspectorPage`/`InspectorState`/`InspectorJson`/`TreeSnapshot`, the 106 JVM tests, the 184 instrumented tests, the `jvmHost` compile machinery, `ciSoDir` | The repo's dev loop and test surface. A generated app gets none of it, by decision. |

> **Gate 1 review correction — the wrapper was in the wrong row.** `gradlew`,
> `gradlew.bat`, `gradle-wrapper.jar` and `gradle-wrapper.properties` are **byte-identical**
> to the repo's; they were listed as TEMPLATE-OWNED above and they are not. The
> mis-classification is not cosmetic — **it is why the byte-identity pin's roster never
> listed them, so nothing compared them at all.** A wrong row in this table became a hole in
> a pin, which is the argument against rosters restated: the roster inherited the table's
> error and had no way to notice. The set is now derived from the tree (review I-1), so the
> table no longer feeds the pin — it only describes it.

**`MainActivity` is the one genuinely divergent file, and it diverges in exactly three
places:** `DEEP_LINK_COMPONENTS` (the template's is **empty** — a one-page app has no
non-`/` routes), the `?: "BnDemo"` fallback (the template's is the starter page's name), and
**`import <namespace>.R`** — template-only. The first two are precisely what
`RouteTableDriftTests` already parses — it pins the map pair-for-pair *and* the fallback
literal.

> **The third arrived at Gate 2, and this section said "two" until then.** AGP generates `R`
> into the `namespace` package while Kotlin resolves a bare `R` against the **file's**
> package. The repo's two match *by coincidence* (both `io.blazornative.shell`); the
> template's differ *by design* — that split is exactly what lets the shell sources stay
> byte-identical while `namespace` is the user's app id — so only the template needs the
> import. **This design asserted that AGP's namespace "does not have to match a source
> package" and concluded the sources could be byte-identical. That is true for every shell
> file that never names `R` — and `MainActivity` names it three times.** `assembleDebug` on
> the GENERATED tree is what disproved it ("Unresolved reference 'R'", `MainActivity.kt:133`),
> which is precisely why decision 8 chose a real compile over `gradlew tasks`. The excision
> is asserted in BOTH directions: the template must carry it, the repo must NOT.

So:

- **Pin A — modulo the map:** the template's `MainActivity` ≡ the repo's, with the map block,
  the fallback literal and the R import excised. ⚠ Brittle (an excision regex). Ledgered: the
  clean fix is extracting the map to its own file, which is a shell change and therefore a
  184-test re-run this phase is not paying for.
- **Pin B — the template's own pair:** the template's map + fallback vs the template's
  `AppPages.All`. **The template ships the same contract the repo has** — *a page is
  declared once; the Kotlin map is the one pinned mirror* — and the user inherits a pin that
  will insist when they add page two.

**The seams a generated app's layout needs** (the repo's are repo-shaped and cannot travel):

| Seam | Repo | Template |
|---|---|---|
| `runtimePubRoot` | `rootProject.projectDir.resolve("../../samples/BlazorNative.SampleApp/bin/Release/net10.0")` | `../bin/Release/net10.0` (gradle root is `<App>/android/`, the app is its parent), **overridable via `-PappPubRoot`** — the `ciSoDir` pattern, kept as the one override |
| `winX64PublishPath` | same escape | same, derived from `appPubRoot` |
| `ciSoDir` + the `abiFilters` narrowing | CI's cross-job `.so` hand-off | **DROPPED** — a generated app has no analog. Both ABIs staged, both publishes expected. Simpler than the repo's. |
| `namespace` / `applicationId` | `io.blazornative.shell` | template symbols (default `com.example.<appname>`) |
| The artifact **name** | frozen `BlazorNative.Runtime.so` | **unchanged, and free** — `CanonicalizeNativeArtifactName` renames `<AppName>.so` → `BlazorNative.Runtime.so` inside the publish folder. **8.0's canonicalization is what lets a generated app reuse the shell's `Native.load("BlazorNative.Runtime")` and `jniLibs` layout with zero shell edits.** Worth saying out loud: that target was built for a directory re-point and it pays for the template. |

**The pins are pinned, not inherited.** AGP `9.3.0`, KGP `2.2.10`, gradle `9.6.1`, JNA
`5.19.1`, Coil `2.7.0`, soloader `0.12.1`, Yoga `3.2.1`, `compileSdk`/`targetSdk` 34,
`minSdk` 24, NDK `26.3.11579264`. A floating pin would break a generated app
unpredictably — but each pinned literal is **a copy of the repo's**, so each is drift.
`TemplateDriftTests` parses both `build.gradle.kts` files and asserts them equal.

> **A concrete find: the template's `build.gradle.kts` is a FOURTH Yoga pin.** `ci.yml`'s
> very first step asserts Yoga parity across **three** files (`build.gradle.kts`,
> `ios.yml`, `ci.yml`'s own `YOGA_VERSION`) because *"one engine, three files — bump them
> together"* and *"two different engine versions would erode [identical frames] silently."*
> **That step must learn about the fourth**, or a generated app lays out differently from
> both shells, silently, which is the exact failure the step exists to prevent. Extending
> it is ~2 lines and it is not optional.

**No `minSdk` symbol.** The prompt asks; the answer is no. Below 24 breaks the NDK/Coil
assumptions; above 24 is the user's one-line edit. **A symbol invites a value nothing
tests.** The symbol surface stays: app name (`sourceName`), `applicationId`/`namespace`,
and the version. Three.

### 7. The build machinery a generated app does **not** get for free ⚠

Fact 9, made actionable. Two files the sample depends on that no `dotnet new` app inherits:

- **`build/BionicNativeAot.targets`** — the vendored NDK cross-compile shim
  (`PublishAotUsingRuntimePack` bypass, the Windows `.cmd` shims + the PATH-prepend local
  deviation, the `_init`/`_fini` section removal for NDK ≥ 26). **Without it there is no
  bionic publish, hence no `.so`, hence no APK.**
- **`global.json`** — SDK band `10.0.3xx`. The csproj's own comment names it as the pin the
  host ILCompiler relies on. Without it a generated app publishes against whatever band the
  user has, and the bypass is band-sensitive.

**DECISION: the template ships both, verbatim, pinned** (`TemplateDriftTests` — same
byte-identity treatment as the Kotlin).

> ⚠ **VETO FLAG — and the better answer, named rather than quietly skipped.**
> `BionicNativeAot.targets` **belongs inside the `BlazorNative.Runtime` package**, as
> `build/BlazorNative.Runtime.targets` — NuGet auto-imports that path for consumers, so
> every consumer would get the bionic recipe **for free and forever**, and the template
> would get thinner by a file. It is the right architecture, **it is the same fix the iOS
> side needs** (decision 9), and I am **not** taking it here because: it changes a shipped
> package's contents, which reds the smoke's inventory-shape assertion (*"five libs:
> exactly one dll + its XML doc under `lib/net10.0`"*), and re-opening DoD #2's closed pack
> shape inside the template phase is scope the phase cannot carry honestly. **Ledgered with
> its cost named.** If the owner wants it now, it is a real improvement and the design has
> no objection — only a sequencing verdict.

### 8. The CI lane: **inside `build-test`**, one publish, and zero for the Android half ⚠

**Where: a step inside `build-test`, after the consumer smoke.** Not a new job.

- **A new required job = a fourth required check** = a branch-protection change, and *"the
  required gates keep their names and contexts"* is a standing constraint. Out.
- **A new advisory job** would duplicate the toolchain (SDK, NDK, Gradle cache) and the
  pack, to prove something the required lane can prove in-place.
- **`build-test` already owns everything this needs**: the packed local feed (the smoke
  just built it), the SDK, an Android SDK, a warm Gradle cache, **and both bionic
  publishes**.

**Logic lives in `scripts/template-smoke.ps1`; the YAML holds wiring** — 8.2's rule 6,
verbatim, and the same reason (a step that exists only in YAML is the least testable code
in the repo).

**The lane:**

| # | Step | Cost |
|---|---|---|
| 1 | Pack the template → `artifacts/templates` | seconds |
| 2 | `dotnet new install` the nupkg | seconds |
| 3 | `dotnet new blazornative -n <App> -o <temp>` — **creation succeeds** | seconds |
| 4 | Restore the generated app from the **local feed** into a temp cache, `--no-cache`, **provenance asserted** (the three came from `artifacts/packages`, not nuget.org) — the ConsumerSmoke pattern verbatim | ~30s |
| 5 | **`dotnet publish -r win-x64` → assert 4 IL2072 + 9 exports + the page probe** | **~1–3 min** — the phase's point |
| 6 | **`gradlew assembleDebug -PappPubRoot=<the SAMPLE's publish tree>`** on the generated Android project | **~1–2 min, ZERO extra publishes** |

**Step 6 is the decision worth explaining.** The generated project's gradle needs a
`linux-bionic-*/publish/BlazorNative.Runtime.so` to satisfy `verifyNativeAssets` →
`copyRuntimeSo` → `preBuild`. **`build-test` already published both bionic RIDs for the
sample** (fact 8), and the `.so` is a **build input staged into `jniLibs`** — its *contents*
are irrelevant to a Kotlin compile and an APK assembly. So `-PappPubRoot` points the
generated project at the sample's tree and **a real APK assembles for the price of a gradle
run.**

**Why this is worth doing rather than a `gradlew tasks` configuration check:** the AGP 9
incident was a **`sourceSets` misconfiguration that silently stopped compiling the entire
shell** while everything stayed green. `gradlew tasks` would not have caught it. The
template's `build.gradle.kts` declares those same `kotlin.srcDirs` — **the template could
ship that exact bug, reborn, and only a real compile sees it.** `assembleDebug` compiles
`src/androidMain/kotlin` in the template's own tree. That is the assertion.

> ⚠ **HONESTY FLAG — the APK carries the SAMPLE's `.so`.** So step 6 proves: the gradle tree
> is coherent, the parameterized files are valid, the shell Kotlin **compiles in the
> template's tree**, and the APK assembles. It does **not** prove the generated app's own
> `.so` pairs with the generated APK (U2). The residual is small — step 5 proves the
> generated app's own native image canonicalizes to `BlazorNative.Runtime.dll`, carries the
> 9 exports, and contains its page; the `.so` is the same target's other extension — but it
> is a **seam, and it is named rather than papered over.** The veto option is one bionic
> publish of the generated app (~2–4 min) to close it; the design has no objection, only a
> cost verdict.

**Total: ~3–6 min** on a job with a 45-minute timeout. **No new required check; no context
name changes; no second pack; no extra publish for Android.** Weighed against the posture
that cost `ios.yml` its PR trigger: this is the cheapest shape that proves the sentence the
phase hangs on, and the expensive half (step 5) **is** that sentence.

### 9. The iOS documentation: `docs/ios-shell-setup.md`, and its honesty is that it refuses to transcribe ⚠

**Where.** *Not* `docs/GITHUB-SETUP.md` — that is **owner**-facing (labels, branch
protection, the secret, the release ritual). This is **consumer**-facing. `docs/` already
holds exactly this shape (`docs/analyzers.md`, `docs/bridge-extension.md`), so
**`docs/ios-shell-setup.md`** fits without inventing a home. **8.4's DoD already names
"both shell setup guides"** as site content — so 8.4 *consumes* this file rather than
re-authoring it, and 8.3 is not blocked on Docusaurus. The generated app's own `README.md`
carries a pointer, not a copy.

**What makes it honest.** The survey's structural fact decides the whole document:
**`project.yml` references `$(SRCROOT)/vendor/...` and names no publish directory and no
SampleApp anywhere** — the link between "your .NET app" and "the iOS shell" is a directory
of five frozen-name files, and **the only thing that populates it today is CI.** The recipe
is ~90 lines of bash inside two workflow steps. There is no script and no Makefile target.

So the doc **must not transcribe the recipe** — a transcription is a fourth copy of a
recipe that already exists twice in YAML, and it rots the moment `ios.yml` moves. **It
points at the live, CI-compiled files and names what the reader must change.** `ios-build`
is a **required** check, so the referenced material is kept true by a gate rather than by
prose discipline.

**What it says a user actually does** — and it says it in this order, starting with the
sentence that respects their time:

1. **There is no iOS template, and this is a manual procedure against a reference
   implementation.** It is more work than the Android path. **Simulator only** — real
   device is M9.
2. **Copy `src/BlazorNative.Apple/BnHost/` into your tree** (19 files; `BnWidgetMapper.swift`
   alone is 3,498 lines). The shell is not packaged.
3. **What to delete**, named exactly: the whole `BnHostTests/` target (30 files) and its
   scheme action; and — the one a reader would not guess — **`Info.plist`'s
   `NSAllowsLocalNetworking` ATS exemption exists only so the test fixture server works.**
   A new app inherits an exemption it never asked for. Say so.
4. **What to edit**, named exactly: **`HostViewController.swift:60`'s
   `try runtime.start(component: "BnDemo", …)`** and **`BnRuntime.swift:184`'s
   `component: String = "BnDemo"` default** → their root component. Plus the bundle
   id/name in `Info.plist` and `project.yml`. **These are source edits in the shell, not
   configuration** — that is the honest word for it.
5. **What is a stub, not a feature:** **`AppleShellBridge.swift:102` — fetch fails
   synchronously** because "BnDemo does no fetch." **A real app must implement it.** A
   setup guide that let a reader discover this at runtime would be a bad guide.
6. **The staging**, described as the shape it is (publish `iossimulator-arm64`; the `.a`
   lands under `publish/` **or** `native/`; `libbootstrapperdll.o` out of the NuGet runtime
   pack; `libtool -static` merge minus exactly three GC/eventpipe variants) with the
   load-bearing warning: **`bootstrapperdll.o` must be a DIRECT object, not `-force_load`'d
   — get it wrong and you get a SIGSEGV in `ThreadStore::AttachCurrentThread`, not a link
   error.** Then a pointer to `ci.yml`'s `ios-build` step as **the executable truth**.
7. **Yoga from source** (C++20, simulator SDK) and **delete its `module.modulemap`** or
   Clang's module scanner tries to build a `yoga` module and fails.
8. **The csproj's iOS `PropertyGroup`** — and here the doc earns its keep by pointing at
   the asymmetry: **Android gets `build/BionicNativeAot.targets`; iOS gets nothing.** The
   RID condition, `NativeLib=Static`, `RuntimeFrameworkVersion`, `UnmanagedEntryPointsAssembly`,
   `TrimmerRootAssembly` and `CanonicalizeNativeArtifactName` are all hand-copied. That is
   decision 7's ledger item, seen from the iOS side.

> ⚠ **Flagged — U4: no lane executes this procedure.** It is prose, and prose rots. The
> mitigation is structural, not diligence: **it references live files that `ios-build`
> compiles on every PR** instead of copying their contents. What remains unproven is the
> *procedure's* completeness — that a reader following it arrives at a running app. The
> veto option is an iOS template (the owner scoped it out at milestone-open) or a staging
> script (`scripts/stage-ios.ps1`) that would at least make the recipe executable and
> testable — **genuinely tempting, and named in the ledger**, because it would delete the
> "~90 lines of bash in YAML" problem for the repo *and* the reader at once.

## The normative rules, stated once

1. **A generated app clears the same publish bar as the repo's app** — **4 IL2072 + the 9
   exports + the page-presence probe**, asserted on GENERATED output in the required lane.
   A template that produces an app the gates would reject is a broken template, and the
   only way to know is to generate one and publish it.
2. **The template's version literal is a MIRROR, not a source.** `src/Directory.Build.props`
   remains the ONE version. Every BlazorNative version literal in the template tree — the
   generated csproj's references, `template.json`'s default, and the pack's own `<Version>`
   — is pinned equal to it, non-vacuously. Drift is a **red, not a review catch**.
3. **The template's shell Kotlin is byte-identical to `src/BlazorNative.Jni`'s.** The
   package stays `io.blazornative.shell` precisely so this can be a file comparison. Files
   that *cannot* be identical (`MainActivity`, the gradle tree) are pinned by their own
   parsers, and each carries a stated reason for diverging.
4. **The shipped six is a set of shipped LIBRARY ASSEMBLIES.** The template pack is a
   seventh artifact of a different kind, in `templates/`, on its own feed, outside the
   release path. It does not join the six, and the six's pins gain no special cases.
5. **One engine, now FOUR files.** Yoga's pin is asserted equal across `build.gradle.kts`,
   `ios.yml`, `ci.yml`'s `YOGA_VERSION` **and the template's `build.gradle.kts`**. Bumping
   Yoga means bumping all four in one commit.
