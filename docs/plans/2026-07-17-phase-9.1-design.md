# Phase 9.1 Design — Local Notifications + Tap-Through (the first reuse of the 9.0 generic ABI)

**Date:** 2026-07-17
**Milestone:** M9 — Host APIs (Platform Breadth). Owns **DoD #3 (local notifications:
schedule / show / cancel + tap-through, permission story, device tests)**.
**Branch:** `phase-9.1-notifications` (at main's tip `3866410`).
**Predecessor:** Phase 9.0 — the permission pattern + geolocation
([conclusion](2026-07-17-phase-9.0-conclusion.md); DoD #1+#2 closed, PR #118). 9.0 grew
the ABI **once, generically** (op-enum on `HostCallBegin@72` + `blazornative_host_call_complete`)
on the explicit bet that *9.1/9.2/9.3 add an op constant and touch the ABI at nothing else.*
**Phase 9.1 is the first reuse — it either validates that bet or exposes its first gap.**

This is a **design-only** document. It states the decisions, the normative rules, the gate
structure, and the PROVEN/UNPROVEN split. Implementation is later work.

---

## 0. THE HEADLINE — local notifications fit with ZERO ABI change, tap-through included

**The single most important finding, stated first and loudly: notifications add
`HostCallOp.Notifications` (one op-enum value) + a wire-mirrored status enum + host handlers
on both shells + an `INotifications` façade — and touch the ABI at NOTHING else. No struct
grow (stays 80 bytes / 10 slots), no new export (stays 10), no drift-pin move, no
export-gate arithmetic. The 9.0 "pay once, reuse thrice" bet pays off on its first draw.**

Walk the shape and prove it:

| Operation | Shape | Rides | ABI touch |
|---|---|---|---|
| **schedule** (id, title, body, when, route) | request → (permission) → complete | `HostCallBegin(id, op:Notifications, argsJson{action:"schedule",…})` → `host_call_complete(id, status, null)` | none — op-enum value + JSON vocabulary |
| **show** (immediate) | request → (permission) → complete | same, `argsJson{action:"show",…}` | none |
| **cancel** (by id) | request → complete | same, `argsJson{action:"cancel","id":…}` | none |
| **permission** (request / check) | request → prompt → complete | same, `argsJson{action:"request"|"check"}` — geolocation's `mode` precedent verbatim | none |
| **tap-through** | **inbound event, no in-flight .NET call** | the EXISTING inbound path — see §1 | **none** — but it is NOT a `host_call_complete` |

The first four are pure geolocation-shaped host calls: each is an outbound request that the
host completes with a tri-state status. They fit the generic `HostCallBegin`/`host_call_complete`
pair with room to spare — the completion carries no payload at all for schedule/show/cancel
(a status is the whole answer), which is *less* than geolocation asked of it.

**The fifth — tap-through — is the genuine finding, and it is NOT a host-call completion.**
Tapping a notification is an **unsolicited inbound event**: the OS wakes the app because the
user tapped, with **no in-flight .NET `ValueTask` to resolve**. It cannot ride
`host_call_complete` (there is no `requestId` — the app may have been dead when the
notification posted). But it does **not force an ABI change either**, because the inbound
host→.NET channel *already exists* and tap-through is exactly the kind of traffic it was
built for. §1 is the design's hardest part and proves this.

## 1. Decision — tap-through as an inbound event (the design's hardest part; ⛔ VETO-FLAGGED)

**A notification carries a route. Tapping it opens/foregrounds the app AND navigates to that
route, via the SAME deep-link machinery `blazornative://<route>` uses today — with ZERO ABI
change. Two cases, two existing channels:**

### 1a. Cold start (app was killed) — the existing launch-time deep link, verbatim

When the app is dead, a tap is a **cold launch**. The notification's tap action is a
`PendingIntent` that reproduces the **exact launch-time deep-link Intent** 5.1 already handles:
a `VIEW` Intent with `data = blazornative://<route>`. `MainActivity.onCreate` calls
`parseDeepLinkRoute(intent)` → resolves the mount component via `DEEP_LINK_COMPONENTS[route]`
→ mounts by name (`MainActivity.kt:228–239`). **This path is untouched by 9.1 — the
notification simply constructs a `PendingIntent` that mints the same Intent the OS mints for
`blazornative://<route>` from a browser.** No new mechanism, no channel, no ABI. The route is
resolved *before the `.so` loads*, exactly as 5.1 designed (the reason `DEEP_LINK_COMPONENTS`
is a hand-written pinned mirror and not a derived view).

iOS cold launch: the route rides the notification's `userInfo["route"]`; on launch the shell's
`UNUserNotificationCenterDelegate.userNotificationCenter(_:didReceive:)` reads it and seeds
the initial mount the same way (iOS has no Intent, so `userInfo` is the carrier).

### 1b. Warm re-route (app alive) — the EXISTING `host_event` export, a NEW event name

When the app is alive, the tap does **not** re-run `onCreate` — it delivers to
`Activity.onNewIntent`, which `MainActivity` **explicitly does not handle today** (5.1 scoped
itself to launch-time only — `MainActivity.kt:392–393`). And there is **no host→.NET
"navigate to route X" command today** — the only host-initiated navigation is the reserved
`"back"` event (`Exports.cs:475`, `DispatchHostEventCore` → `NavigateBackAsync`). **So warm
re-route IS the real work of this phase.** It rides existing exports:

1. **Shell side (Kotlin, no ABI):** `MainActivity` gains `onNewIntent(intent)` + a `singleTop`
   launch mode (manifest), parses the same `blazornative://<route>`, and — instead of a mount
   (the session is live) — dispatches an **inbound navigation command**.
2. **The command channel is the EXISTING `blazornative_host_event(name, payload)` export.** It
   already routes on the event NAME: `"back"` is intercepted before the `NativeEvents`
   multicast and mapped to `NavigateBackAsync`. **9.1 adds a second reserved name —
   `"navigate"` — whose payload is the route, mapped in `DispatchHostEventCore` to
   `nav.NavigateToAsync(payload)`.** This is **wire vocabulary + a .NET branch, NOT an ABI
   change** — the precise shape 5.1 used to add `"back"` without touching the export set. The
   `NativeNavigationManager.NavigateToAsync` swap (route → `SwapRoot` → `RouteChanged`,
   `NativeNavigationManager.cs:129`) already re-routes a live session; the shell has simply
   never had a way to *ask* it to. Now it does, over the existing wire.

**iOS warm re-route** rides the same existing export. Note the 9.0 milestone-first: *the iOS
shell had never called `host_event` — it is "undeclared" on iOS* (9.0 conclusion). The
`blazornative_host_event` export nonetheless **exists in the shared runtime** (`Exports.cs:417`);
9.1's iOS shell simply **starts calling it** — exactly as 9.0's iOS shell started calling the
pre-existing `host_call_complete` for the first time. Still zero ABI change: the export was
there all along.

### 1c. What .NET sees, and when

- **Cold:** .NET sees a **normal first mount** of the routed page (the 5.1 startup-honor path).
  There is no distinct "I was tapped" signal — the arrival *is* the mount. `CurrentRoute`
  agrees because the shell seeds `initialRoute` from the deep link (`AndroidShellBridge`
  `initialRoute`, `MainActivity.kt:234`).
- **Warm:** .NET sees a `host_event("navigate", route)` → `NavigateToAsync` → a **`RouteChanged`
  raise** to the target route. A mounted component subscribed to `INavigationManager.RouteChanged`
  observes the re-route; the target page mounts fresh. This is the *existing* navigation event,
  not a new one.

### ⛔ VETO FLAG — warm re-route is the severable half; decide scope before Gate 2

**Cold-start tap-through alone closes DoD #3** ("tap-through opens the app to a route — the
deep-link machinery from 5.1 is the landing path"): a killed app is the common notification
case, and cold launch reuses 5.1 with zero new code beyond the `PendingIntent`. **Warm
re-route (`onNewIntent` + the `"navigate"` host-event name) is the honest, complete answer
and is RECOMMENDED IN SCOPE** — it is cheap (rides existing exports) and it is what a user
expects when they tap a notification over a running app. **But it is severable:** if Gate 2's
budget tightens, warm re-route extracts to 9.4 (or a follow-up) with DoD #3 still closed by the
cold path, and the risk proof is unchanged. **Decide before Gate 2 wires the shell.** The
begin/complete ABI is untouched either way — this veto is about *scope*, never about the wire.

### ⛔ VETO FLAG — the reserved event name + shape

`"navigate"` with the route as the bare `payload` string is the recommendation (symmetry with
`"back"`, minimal). The alternative — a structured deep-link event name carrying a flat-JSON
payload — buys nothing for a single route string and is rejected as premature. **Decide the
literal + payload shape before Gate 1 builds `DispatchHostEventCore`'s branch**, because the
Kotlin/Swift shells must use the exact literal (the `BackEventName` discipline: the mapping
lives in .NET so every shell gets identical semantics — `Exports.cs:467`).

## 2. Decision — the three operations + the payload shape

**One op constant, the action inside the flat JSON** (geolocation's `mode` precedent — one op,
many sub-actions, no second slot). `NativeShellBridge.HostCallOp` gains **`Notifications = 1`**
(the `Geolocation = 0` sibling); the shells' mirror enums (`HostCallOp` Kotlin, `BnHostCallOp`
Swift) gain the same value. **This is the ONLY change to the op-enum, and adding an enum value
is wire vocabulary — not a struct grow, not an export, not a drift-pin move.**

The args cross as the existing flat string→string JSON (`WriteFlatJsonObject`/`ParseFlatJsonObject`
— no new serializer, the trim-cost decision restated):

| action | args (flat JSON) | completion status | payload |
|---|---|---|---|
| `schedule` | `{"action":"schedule","id":"7","title":"…","body":"…","when":"<unix-ms>","route":"/notifications"}` | `NotificationStatus` | none |
| `show` | `{"action":"show","id":"7","title":"…","body":"…","route":"/notifications"}` (no `when`) | `NotificationStatus` | none |
| `cancel` | `{"action":"cancel","id":"7"}` | `NotificationStatus` (`Ok` — cancel is idempotent) | none |
| `request` | `{"action":"request"}` | `NotificationStatus` | none |
| `check` | `{"action":"check"}` | `NotificationStatus` | none |

**Normative rule (stated once): every field crosses as a string** (numbers string-encoded,
`InvariantCulture`/`Double.toString`/`String(_)` all agree on `'.'` — the geolocation fix-key
discipline). `id` is a decimal int; `when` is Unix epoch **milliseconds**; `route` is a
`DEEP_LINK_COMPONENTS` key or absent.

### 2a. The notification id space — app-chosen 32-bit ints

**Decision: the id is a caller-supplied `int`.** Android `NotificationManager.notify(id, …)` /
`.cancel(id)` and the scheduling `PendingIntent` request code are all `Int`-keyed; iOS
`UNNotificationRequest(identifier:)` is `String`-keyed, so the iOS shell **stringifies the int**
(`id.description`). The app owns the id space (it is what `cancel` targets); collisions replace
(Android `notify` semantics) — which is correct and expected. A stable int → a stable
identity across schedule/show/cancel on both shells.

### 2b. Android specifics — the channel, and how "schedule" wakes the app

- **The channel (Android 8+ requires one).** A single channel created at shell init (idempotent):
  id **`"blazornative_default"`**, importance `IMPORTANCE_DEFAULT`, user-visible name
  `"Notifications"`. Named here so it is one literal, mirrored in the doc. (Per-notification
  channels are out of scope for the POC — one channel, named.)
- **`show`** → build a `Notification` on that channel with `setContentIntent(pendingIntent)`
  (the tap-through `PendingIntent`, §1) → `NotificationManager.notify(id, n)`. Immediate.
- **`schedule`** → the notification cannot post itself later, so the shell uses **`AlarmManager`
  + a `BroadcastReceiver` (`NotificationPublisher`)**: `AlarmManager.set(RTC_WAKEUP, whenMs,
  publishPendingIntent)` where the publish `PendingIntent` carries the id/title/body/route as
  extras; the receiver reconstructs the notification and calls `notify`. **Inexact alarms
  (`set`, not `setExact`)** — a notification does not need exact-alarm timing, so **no
  `SCHEDULE_EXACT_ALARM` permission is required** (the honest no-extra-permission path, the
  `LocationManager`-over-fused precedent). `WorkManager` is the production-grade alternative,
  deferred.
- **`cancel`** → `NotificationManager.cancel(id)` for a shown one AND `AlarmManager.cancel` for
  a scheduled-but-unfired one (both, keyed by the same id).

### 2c. iOS specifics — `UNUserNotificationCenter`

- **`show`/`schedule`** → `UNMutableNotificationContent` (title/body, `userInfo["route"]` for
  tap-through) + a trigger: `nil`/`UNTimeIntervalNotificationTrigger(timeInterval: 0.1)` for
  immediate show, `UNCalendarNotificationTrigger`/`UNTimeIntervalNotificationTrigger` for a
  scheduled `when` → `UNUserNotificationCenter.current().add(request)`. The center schedules
  itself — no alarm plumbing.
- **`cancel`** → `removePendingNotificationRequests(withIdentifiers: [id])` +
  `removeDeliveredNotifications(withIdentifiers: [id])`.

## 3. Decision — the permission (POST_NOTIFICATIONS / UNUserNotificationCenter)

**Denial is DATA — the milestone law, restated.** The completion's `int status` is a
**wire-mirrored `NotificationStatus`**, defined byte-identically across .NET / Kotlin / Swift
(pinned like the struct); the host maps each platform's native outcome into it host-side, and
.NET only ever sees the integer.

| status | Name | Meaning | Android | iOS |
|---|---|---|---|---|
| 0 | `Granted` | permission held; the op ran (posted / scheduled / cancelled) | API<33 implicit, or `POST_NOTIFICATIONS` granted | `.authorized` / `.provisional` |
| 1 | `Denied` | denied **this time**; a later request MAY prompt | not granted ∧ rationale true | (n/a — iOS prompts once) |
| 2 | `DeniedPermanently` | "don't ask again" / iOS `.denied` — only Settings changes it | not granted ∧ rationale false (after a request) | `.denied` |
| 3 | `Restricted` | policy/MDM — the user **cannot** grant it | device-policy restriction | `.restricted` (rare) |
| 4 | `Error` | unexpected host error (a caught throw) | caught Kotlin throw | caught Swift throw |

Five values (geolocation's shape minus `LocationUnavailable`, which has no notification analogue).
An out-of-range integer maps to `Error` — still data, never a throw. **This is `NotificationStatus`,
a NEW wire-mirrored enum, mirrored 3 ways** (it is the phase's one new vocabulary item beyond the
op constant — and vocabulary, not ABI).

### 3a. The already-granted fast path (the geolocation `check` precedent)

**Normative rule (stated once): a request short-circuits when permission is already held —
never a redundant prompt.**

- **Android < 13 (`SDK_INT < 33`): `POST_NOTIFICATIONS` did not exist and is implicitly
  granted.** The host returns `Granted` **without any prompt or manifest permission gate** —
  the fast path is the *majority* of devices. On API 33+, the host checks
  `checkSelfPermission(POST_NOTIFICATIONS)`; if held → `Granted`; if not → `requestPermissions`
  (the geolocation `requestPermissions`/`onRequestPermissionsResult` machinery, **reused
  verbatim** — same static `requestCode→requestId` map, same recreation-survival split, same
  `MainActivity.onRequestPermissionsResult` forward). **9.1 writes no new permission plumbing on
  Android — it points the existing plumbing at a different permission string.**
- **iOS:** `getNotificationSettings().authorizationStatus` — `.authorized`/`.provisional` →
  `Granted` (no prompt); `.notDetermined` → `requestAuthorization(options: [.alert, .sound,
  .badge])` (the one prompt); `.denied` → `DeniedPermanently`. **Decision: `.provisional`
  (quiet notifications, delivered without a prompt) maps to `Granted`** — they post, which is
  what the caller asked. (⛔ minor veto: expose `.provisional` as a distinct status only if a
  consumer needs to distinguish quiet delivery; the POC folds it into `Granted`.)

### 3b. The manifest / Info.plist declarations

- **Android** — `<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />` in
  **both** manifests (sample `src/BlazorNative.Jni/.../AndroidManifest.xml` and the template's),
  next to the 9.0 location lines. (Below API 33 it is inert; above, it gates the runtime
  request.) The `NotificationPublisher` `BroadcastReceiver` is declared in the same manifests.
- **iOS** — **no Info.plist key is required** for local notifications (unlike location's
  purpose string); authorization is purely runtime (`requestAuthorization`). Recorded so the
  absence is a decision, not an omission.

## 4. Decision — the .NET surface (no 8th package)

### 4a. Layering — typed surface in Core, façade in the EXISTING Device package

Exactly the 9.0 split, so `DevHostBridge` mocks the tri-state headless:

- **`IMobileBridge` (Core)** gains the typed primitives, so the in-process mock can serve them:
  ```csharp
  ValueTask<NotificationStatus> ScheduleNotificationAsync(NotificationSpec spec, CancellationToken ct = default);
  ValueTask<NotificationStatus> ShowNotificationAsync(NotificationSpec spec, CancellationToken ct = default);
  ValueTask<NotificationStatus> CancelNotificationAsync(int id, CancellationToken ct = default);
  ValueTask<NotificationStatus> RequestNotificationPermissionAsync(CancellationToken ct = default);
  ValueTask<NotificationStatus> CheckNotificationPermissionAsync(CancellationToken ct = default);

  public readonly record struct NotificationSpec(
      int Id, string Title, string Body, DateTimeOffset? When, string? Route);
  public enum NotificationStatus { Granted, Denied, DeniedPermanently, Restricted, Error }
  ```
  Each `NativeShellBridge` impl calls the **existing** `InvokeHostCallAsync(op:Notifications, argsJson, ct)`
  (`NativeShellBridge.cs:425`) — the pending registry, the cancellation posture, the unknown-id
  benignness are all **reused, not re-written**. Only the args-building and the status-mapping
  (`ToNotificationStatus`, the `ToGeolocationStatus` twin) are new .NET.
- **`INotifications` (Device — the SAME 7th package, no 8th)** — the app-facing façade, a thin
  delegate over `IMobileBridge`, sibling to `IGeolocation`:
  ```csharp
  public interface INotifications {
      ValueTask<NotificationStatus> ScheduleAsync(NotificationSpec spec, CancellationToken ct = default);
      ValueTask<NotificationStatus> ShowAsync(NotificationSpec spec, CancellationToken ct = default);
      ValueTask<NotificationStatus> CancelAsync(int id, CancellationToken ct = default);
      ValueTask<NotificationStatus> RequestPermissionAsync(CancellationToken ct = default);
      ValueTask<NotificationStatus> CheckPermissionAsync(CancellationToken ct = default);
  }
  ```
  Registered by **`AddBlazorNativeDevice()`** — one added line
  (`services.AddSingleton<INotifications, Notifications>()`, `DeviceServices.cs:31`). **No new
  package, no new shipped-set pin, no new csproj, no new version mirror** — verified: the M9
  device group was the whole reason `BlazorNative.Device` exists (its registration KDoc already
  says "and the other M9 device facades as they land", `DeviceServices.cs:18`). **The 7th
  package is the last package; 9.1 confirms it needs no 8th.**

### 4b. `DevHostBridge` mocks it headless

`DevHostBridge` records scheduled/shown notifications in an in-memory list, honours `cancel`
by id, and returns a **configurable `NotificationStatus`** (so the headless lane drives all five
statuses with no device). This is the central proof of denial-as-data for notifications —
asserted at Gate 1, within a bounded await, before any device work.

## 5. Decision — the proof surface (`/notifications` demo)

A **`/notifications` demo page** in SampleApp (`BnNotificationsDemo`, the **11th routed page**),
the worked example that `[Inject]`s `INotifications`:

- **Show** button → `ShowAsync` with `Route = "/notifications"`; **Schedule (in 5s)** →
  `ScheduleAsync(When: now+5s, Route:"/notifications")`; **Cancel** → `CancelAsync(id)`.
- Every button echoes its returned **`NotificationStatus` as data** (a denial is *shown*, never
  thrown, never hung — the geolocation `BnGeolocationDemo` discipline).
- **The tap-through flow:** the notification's route is `/notifications`. Tapping it (cold: cold
  launch to the page; warm: `host_event("navigate","/notifications")` → re-route) lands the app
  ON this page. The arrival IS the proof — the page mounting via the tap is the observable. (A
  belt-and-braces marker: the demo may write a "last tap at <t>" via `IMobileBridge` storage that
  the page reads on mount, so a warm re-route to the *same* page is visibly distinguishable; kept
  optional to avoid golden churn.)

Registration touches the **one page-authority** (`SampleAppPages.All`) + its **pinned mirror**
`DEEP_LINK_COMPONENTS` (`"/notifications" → "BnNotificationsDemo"`, both the sample's and the
template's `AppPages.cs`/map), and `RouteTableDriftTests` re-greens pair-for-pair. **The template
gets the page + the route mirror too** (parity), so a `dotnet new` app has the demo.

### PROVEN on CI vs UNPROVEN (the M5-era simulator-honesty discipline)

- **PROVEN on CI:** all five statuses as data on all three surfaces (mock + AVD + simulator);
  a **real notification posts** on the AVD (`dumpsys notification` inspect) and the simulator;
  **cancel removes it**; the **denial path resolves without a hang**; **tap-through routes to
  the page** — cold via a relaunch Intent, warm via `onNewIntent`/`didReceive` → `NavigateToAsync`;
  the below-33 implicit-grant fast path. The **ABI is unchanged** (80 bytes / 10 exports pins
  green — the headline, made a CI fact).
- **UNPROVEN — the owner's physical Android phone only** (documented `adb`-over-USB steps, the
  honesty check, never a CI node): the **real notification in the shade** with the real channel
  UX; the **real tap-through from a genuinely cold start** (a killed process, not a test relaunch);
  the **real `POST_NOTIFICATIONS` system-dialog gesture** (bypassed in instrumentation, exactly
  as 9.0 bypassed the location dialog); `DeniedPermanently` ("don't ask again") across restarts.
- **UNPROVEN — iOS:** **simulator-scoped + labeled** (no physical iPhone — the M9 deferral,
  trigger = an Apple Developer account); the simulator posts real local notifications but the
  real Lock-Screen/Notification-Center tap UX is device territory.

## 8. Risks (led by the tap-through-as-inbound-event question)

| Risk | Mitigation |
|---|---|
| **Tap-through is an unsolicited inbound event with no in-flight call — does it force the ABI?** (the headline) | **No.** It rides two EXISTING channels with zero ABI change: cold = the 5.1 launch deep-link (`DEEP_LINK_COMPONENTS`, resolved before the `.so` loads); warm = the existing `blazornative_host_event` export with a NEW reserved name `"navigate"` (wire vocabulary + a `DispatchHostEventCore` branch, the exact shape `"back"` took in 5.1). `host_call_complete` is correctly NOT used — there is no `requestId`. Proven at Gate 1 (the `"navigate"` branch) before any device work. |
| **Warm re-route is genuinely new work** (`onNewIntent` unhandled since 5.1; no host→.NET "navigate to route" today) | It is — and it is the phase's real engineering, but it is **cheap and ABI-free**: `NavigateToAsync` already re-routes a live session (`NativeNavigationManager.cs:129`); the shell has simply never had a way to ask. §1c wires it over the existing export. ⛔ Severable (cold path closes DoD #3) if Gate 2 budget tightens. |
| **The op-enum grow is a hidden ABI change** | It is not: adding `Notifications = 1` to a value carried in the existing `int op` field is wire vocabulary — the struct stays 80 bytes, the exports stay 10. Gate 1's `== 80` / 10-export pins are the tripwire, asserted UNCHANGED (the falsifiable headline). |
| **Android "schedule" needs an exact-alarm permission** | It does not: **inexact** `AlarmManager.set` is used (a notification needs no exact timing) → no `SCHEDULE_EXACT_ALARM`. The only new permission is `POST_NOTIFICATIONS` (API 33+; implicit below). |
| **A shell edit drifts the template mirror** (the 9.0 Gate-2 miss) | The PROCESS LEDGER is folded into Gate 2's bar: sync the byte-identical template `MainActivity.kt`/`AndroidShellBridge.kt`/`AndroidManifest.xml` mirrors and **run `TemplateDriftTests` + `RouteTableDriftTests` before pushing**, in the same gate. |
| **iOS has never called `host_event`** (undeclared on iOS) | The export EXISTS in the shared runtime; iOS simply starts calling it — the 9.0 precedent, where iOS called `host_call_complete` for the first time. Zero ABI change. |
| **`.provisional` iOS authorization is a fourth grant-ish state** | Folded into `Granted` (provisional notifications post — what the caller asked). ⛔ Expose as a distinct status only if a consumer needs to distinguish quiet delivery; the POC does not. |
| **The system permission-dialog / real shade UX is not CI-drivable** | Bypassed behind the 9.0 seams (request hook + status override), production paths unsuppressed; the real gesture + the real shade are the **owner's physical Android phone** (the PROVEN/UNPROVEN split names it). |
| **iOS provable only on the simulator** | Simulator-scoped + labeled (the M5→M8 posture); the owner's phone carries the Android real-device honesty; iOS real-device trigger named (Apple Developer account). |

## 9. Constraints honored (the standing laws)

- **The ABI does NOT grow — the reuse the 9.0 export event bought.** `HostCallOp.Notifications`
  is an op-enum value; `NotificationStatus` is wire vocabulary; tap-through rides existing
  channels. **No struct grow (80 bytes), no new export (10), no drift-pin move, no export-gate
  arithmetic** — asserted UNCHANGED at Gate 1 (§0, §7). This is the phase's headline and it is
  falsifiable.
- **Denial is DATA**, never an exception/hang (§3, normative — the five-status matrix within a
  bounded await).
- **Wire vocabulary mirrored 3 ways** — `HostCallOp.Notifications` and `NotificationStatus` on
  .NET / Kotlin / Swift; the `DEEP_LINK_COMPONENTS` route mirror pinned by `RouteTableDriftTests`.
- **Counts with provenance** — baseline **.NET 616 / JVM 106 / instrumented 188 / iOS 169**;
  deltas mutation-verified at close.
- **Five required contexts unchanged** (`build-test`, `android-build`, `ios-build`, `pr-title`,
  `footer-check`); advisory device lanes unchanged; the owner's phone is never a CI dependency.
- **No 8th package** — `INotifications` is a sibling of `IGeolocation` in the existing 7th
  package `BlazorNative.Device`; `AddBlazorNativeDevice` gains one line.
- **The M8/M9 ledger watch:** SafeAreaView / edge-to-edge was flagged as "the camera and
  notification phases are the likeliest to force it" (9.0 conclusion). **9.1 does NOT force it**
  — a notification shade is OS chrome, not app layout; the demo page reuses the existing routed-
  page layout. Recorded so the watch item is explicitly *not* tripped here.
- **Commits land as `feat:`** (the notifications surface + the tap-through wire) so release-please
  walks the 0.x version and writes the changelog; the design doc lands as `docs(phase-9.1): …`.
  Conventional-commit contract respected (≤100-char titles, no line-start footers in bodies).
