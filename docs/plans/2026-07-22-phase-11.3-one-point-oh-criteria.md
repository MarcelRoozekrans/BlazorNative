## 6. Two standing conditions attached to this list

**(1) The iOS caveat must stay specific.** 1.0 may ship iOS-simulator-only **only if** the README
and docs carry a named list, not a shrug: camera capture from a real sensor, Face ID / Touch ID
against the Secure Enclave, real-GPS geolocation, APNs and universal links, code-signing and
provisioning, thermal/background behaviour. And the hard condition: **P1 stays a BLOCKER.**
Shipping with *neither* platform device-proven would be indefensible; exactly one platform may
carry this caveat, and it must be the one with an administrative barrier — not a convenient one.
P1 is now met, so the condition is satisfiable.

**(2) The README states no counts.** `ReadmeDriftTests` pins every number the README prints
(`ReadmeCounts_MatchTheGatesThatAssertThem`, `ReadmeCountTable_PinsEveryRowItLists`). A count in
the status paragraph would either join that roster or drift. The API-stability page is the right
home for "88 public types, 55 STABLE" — it is regenerated from the tier table, and the tier table
is the thing that changes. The same convention removed the *version* literal from the status
paragraph in this gate: `README.md:7` said **"currently v0.4.0"** while 0.4.1 was published. It
now points at nuget.org and names no version, so it cannot re-stale.

## 7. Conclusion — what is frozen, what is not, and why

### What is now frozen

1. **Six package surfaces are recorded, member by member.** `BlazorNative.Components`, `.Core`,
   `.Device`, `.Http`, `.Runtime`, `.Renderer` each carry a `PublicAPI.Shipped.txt` —
   **1 166 baseline lines**. Every public member, every enum member, every nullability
   annotation.
2. **RS0016 / RS0017 / RS0037 are build ERRORS in the required lane.** Not warnings. The repo has
   no `TreatWarningsAsErrors` and never will (`src/Directory.Build.targets:57`), so the gate is a
   per-package `BnEnforcePublicApi` property driving a targeted `WarningsAsErrors` line — in
   `.targets`, because `.props` is imported before the csproj body and that bug cost the CS1591
   pin its teeth **twice**. **The mutation is the proof:** `BnButton.Label` → `BnButton.Text`
   produced RS0016 ×2 + RS0017 ×2 and exit 1. A green build would have proven nothing.
3. **The analyzer contract is pinned as its ID roster**, both directions — the 7 `BN00xx` ids a
   consumer actually types into a `NoWarn`. Not three class names.
4. **The interface-additions policy is in the shipped xmldoc**, so it reaches a consumer through
   IntelliSense rather than through a plan file they will never open.

**What "frozen" means here, precisely: pre-1.0, a break is VISIBLE and DELIBERATE, not
impossible.** `bump-minor-pre-major` is still on; a `feat:` still produces `0.5.0` and is still
allowed to break. What changed is that the breaking PR now **reds**, and its author must write
the break into `PublicAPI.Unshipped.txt` — so the diff *contains* the break as reviewable lines.
Before Gate B, `BnButton.Label` could vanish and nothing in the diff would say so. That shift,
from accident to decision, is the whole pre-1.0 value, and it is enough on its own. At 1.0 the
same files stop being a record and become a freeze: the file does not change, **the rule about
who may edit it does**.

### What was deliberately left un-frozen

1. **The 2 PROVISIONAL types.** `DevHostBridge` — baselined so changes are visible, **excluded**
   from the advertised stable core, because it is a test double whose ~70 members are mock-shaped
   and because it is the natural home of the consumer test harness ([#25](https://github.com/MarcelRoozekrans/BlazorNative/issues/25)),
   work that *will* reshape it. Freezing it now would either block #25 or force it to ship beside
   a frozen predecessor. And `AddBlazorNativeRenderer` — a composition-root call a custom host
   genuinely needs, but a one-line delegate to generated code whose shape is not ours to promise.
2. **The 31 NOT-API types — public for mechanical reasons, and the honest fix is a break.**
   `NativeRenderer` is public **so an `internal` type in another assembly can drive it**; the
   C-ABI structs are public because `[UnmanagedCallersOnly]` and the shells' JNA/ObjC++ mirrors
   require it; the patch records are the renderer's in-memory model. None of these is
   *experimental* — they are *not for you*. The correct fix (`internal` + `InternalsVisibleTo`)
   is a breaking change this phase must not make, recorded as **criterion S3** and classed
   DOCUMENTED. **The residual risk is real and is not hidden: a consumer can bind to
   `NativeRenderer` today, and `[EditorBrowsable(Never)]` does not stop them.**
3. **The 3 unmarkable generated types.** Two ZeroAlloc.Inject registration classes and Razor's
   `_Imports`. An attribute cannot be added to source we do not own; partial re-declaration does
   not compile and would make a generator upgrade a build break. **Stated, not skipped.**
4. **`[Experimental]` on nothing.** Zero uses, argued rather than assumed: nothing in the current
   surface is half-built, `BN1xxx` is reserved for when something is, and putting `[Experimental]`
   on NOT-API would make a **false** claim (`NativeRenderer` has not moved in five milestones)
   while inflicting a compile **error** on people referencing a package for unrelated reasons.
   *A gate can pass by proving nothing qualifies — provided it proves it.*

### The caveat that outlives this phase

**A baseline records a surface; it does not judge it.** `PublicAPI.Shipped.txt` will freeze a
badly-named parameter as faithfully as a good one — a bad API frozen is still bad, and now it is
bad permanently. That is the entire reason Gate A preceded Gate B, and the reason the baseline
read asked *"should this be public at all?"* rather than *"is this line correct?"*.

**The standing proof that the reading was worth doing is
[#181](https://github.com/MarcelRoozekrans/BlazorNative/issues/181).** The baseline emitted one
line nobody wrote —

```
BlazorNative.Runtime.BlazorNativePage.BlazorNativePage() -> void
```

— a public parameterless constructor on a `readonly struct` whose xmldoc claims it is *"created
**ONLY** via the two factories."* C# guarantees that constructor on every struct, so
`default(BlazorNativePage)` compiles and produces a page with a **null mount thunk** — precisely
the shape the private constructor exists to make unconstructible. The trim-law invariant the
design leans on is enforced by convention, not by the type. Its sibling
[#178](https://github.com/MarcelRoozekrans/BlazorNative/issues/178) is the same trap on
`CaptureOptions`.

**Neither was found by a test, a reviewer, or a bug report. Both were found by reading a
generated `.txt` file line by line.** That is the argument for the flood-reading step, made once,
concretely, and it is why criterion **Q5** now exists in a list the design did not have it in.
