# Phase 9.0 Design — The Permission Pattern + Geolocation (the named risk, proven first)

**Date:** 2026-07-17
**Milestone:** M9 — Host APIs (Platform Breadth). Owns **DoD #1 (the permission pattern)**
and **DoD #2 (geolocation)**.
**Branch:** `phase-9.0-permissions-geolocation` (tip `1c72e57`).
**Predecessor:** M8 complete (2026-07-17); milestone 9 opened same day (`1c72e57`).
**Owns the risk:** permissions are "a NEW cross-cutting surface the bridge has never
carried" (MILESTONE.md, *The named risk*). The one prior bridge growth (clipboard/share,
5.4) needed no prompt, no app suspension, no denial. Phase 9.0 proves the permission
machinery on the **simplest permission-gated API — geolocation — before anything heavier
is built**, so a flaw in the async/denial shape is found once, not inherited three times.

This is a **design-only** document. It states the decisions, the normative rules, the
gate structure, and the PROVEN/UNPROVEN split. Implementation is later work.

---

## 0. What clipboard proved, and the two things it never had to

The bridge-extension pattern (`docs/bridge-extension.md`) is the frozen, size-negotiated
C-ABI: **9 exports + a 72-byte / 9-slot callback struct**, mirrored byte-exact on three
sides, pinned by drift tests. Clipboard/share (5.4) is its worked example: three struct
slots appended (48/56/64), each near-**synchronous**, each **ungated**.

Geolocation copies clipboard's *shape* — a host capability behind a struct slot, mocked
in `DevHostBridge`, proven at three layers — and adds exactly the two things clipboard
never needed:

1. **A permission prompt that SUSPENDS the app.** Android `ActivityCompat.requestPermissions`
   pops a system dialog over (a possibly-recreated) Activity; iOS `CLLocationManager`
   raises the system authorization alert. The call is airborne while the app is
   backgrounded.
2. **A denial.** "The user said no" must reach .NET **as data — a status value — never as
   an exception and never as a hang** (MILESTONE.md's stated law).

Neither fits clipboard's synchronous return-code path. Both fit the **fetch** path — the
one existing async precedent — which is where this design anchors.

### The existing async precedent (the model geolocation extends)

`FetchAsync` (`NativeShellBridge.cs:252`) is the bridge's async-completion pattern, and it
already solves "a result arrives back in .NET after an arbitrary delay, keyed to the right
call":

- **Begin (call out):** a struct slot `FetchBegin@40` (`int(long requestId,
  BlazorNativeFetchRequest*)`). .NET assigns an `Interlocked` id, parks a
  `TaskCompletionSource` (`RunContinuationsAsynchronously` — the completion arrives on a
  host/Kotlin thread and must not run continuations inline on it) in a
  `ConcurrentDictionary<long, TCS>` (`s_pendingFetches`), then calls the slot.
- **Complete (result back):** a **dedicated export** `blazornative_fetch_complete`
  (`Exports.cs:381` → `CompleteFetch`, `NativeShellBridge.cs:327`). The host calls it later
  with the `requestId`; `CompleteFetch` removes the id and resolves the TCS. **Cancellation**
  removes the id and `TrySetCanceled`s; a **late/unknown completion** finds nothing and
  takes the unknown-id path (`return 1`, logged, **never a throw**).

Geolocation is fetch with a permission prompt in the middle and a tri-state instead of an
HTTP response. That single observation drives every decision below.

> Note on the M4 hardening ledger, **issue #8 (async-handler capture window)**
> (`docs/plans/2026-07-11-phase-4.2-hardening-triage.md:126`): it is about `await` inside a
> synchronous **event handler** escaping the dispatch-exception window — **not** app
> suspension, and **not** this path. Geolocation resolves a TCS **outside** the dispatch
> window (the fetch model), and the "await a bridge call, then `StateHasChanged`"
> continuation is already proven by `BnDemo`'s fetch + re-render. So #8 stays **contained**
> and 9.0 does not own its redesign. The genuinely-new territory — a permission prompt that
> backgrounds the app mid-call — was not tracked by any M4 item; the nearest prior work was
> the M5 host-initiated lifecycle ingress (`host_event`), which is done and which this
> design leans on for the "app was backgrounded" reality.

## 1. Decision — the permission model on the C-ABI (the headline; ⛔ VETO-FLAGGED)

**A permission-gated call is an async-begin (call out) + a deferred push-completion
(result back), keyed by `requestId`, resolved off-lane — structurally identical to fetch.
Denial is carried in the completion as a tri-state status. The ABI grows by exactly one
struct slot AND one export.**

### 1a. Can we avoid growing the ABI? (argued hard, both ways)

The standing law: *prefer NO ABI change; adding to the frozen 9-export/72-byte bridge is a
genuine event.* So each reuse path was tried and **rejected for cause**:

| Reuse candidate | Why it fails |
|---|---|
| **Reuse `fetch_complete`** | Typed to `BlazorNativeFetchResponse*` (StatusCode/Ok/Body/Headers). A geolocation fix is not an HTTP response — encoding lat/lng into a fake HTTP body is type abuse, and the two id-spaces would have to share `s_pendingFetches`. Rejected: dishonest. |
| **Reuse `blazornative_host_event`** (reserved name, the "back" interception precedent) | `host_event` is contractually **synchronous, raised on the dispatch-lane thread, drives `StateHasChanged` re-renders inline** (`Exports.cs:475`, DispatchHostEventCore). A geolocation completion arrives on a **host background thread after an arbitrary suspension** — routing it through `host_event` violates that threading contract and conflates a lifecycle *notification* with a keyed *request-response*. Rejected: wrong channel. |
| **Poll a buffer-read slot** (`GeolocationPoll(requestId, buf)`) | Polling across an app-suspension burns cycles and re-invents a push the bridge already has. Rejected: the fetch precedent (push completion) is cleaner and proven. |

**Conclusion:** an honest async **permission** completion cannot ride any existing export.
The completion is fetch-shaped but not fetch-typed, and it must arrive on a background
thread — so it earns a **new export**. This is the first export grow since Phase 3.1
(`fetch_complete` itself). **The export grow is UNAVOIDABLE for an honest design** — a
geolocation-specific completion export would grow exports too. There is no no-export path;
the only real choice (§1c) is *generic vs specific*, not *grow vs not*.

### 1b. The grow, exactly

**One new struct slot** (the documented, size-negotiated growth — bridge-extension.md (b)):

| Offset | Slot | Signature | Kind |
|---|---|---|---|
| 72 | `HostCallBegin` | `int(long requestId, int op, const char* argsJsonUtf8)` | async-begin (the `FetchBegin` shape, generalized) |

Struct **72 → 80 bytes**. Forward/backward compatible: an old shell that predates the slot
leaves it `IntPtr.Zero`, and the null-slot guard (`RequireSlot`) surfaces
`NotSupportedException` — never a null call. **Moves the three mirrors + three drift tests
only, NOT the export gates**: `.NET` `BridgeProtocolNative.cs` struct +
`BridgeProtocolNativeTests` (72→80 + offset 72); JVM `NativeBindings.kt`
`@Structure.FieldOrder` + a `BridgeHostCallBeginCallback` interface + `ShellBridgeTest`
(`callbacks_struct_is_72_bytes` → `…is_80_bytes`); Swift `BlazorNativeRuntimeC.h`
(`bn_host_call_begin_cb` typedef + struct field) + `BnDriftTests`. All six
`ShellBridgeHandlers` impls gain the method (production real, the five test doubles inert),
per bridge-extension.md (c) step 5.

**One new export** (the genuine ABI event — the phase headline):

```
blazornative_host_call_complete(long requestId, int status, const char* payloadJsonUtf8) -> int
    // 0 = delivered; 1 = unknown/already-completed id (benign cancellation race,
    // logged, never a throw); 2 = invalid call / internal failure (host logs loudly)
```

The `fetch_complete` twin, generalized. Exports **9 → 10**. This **moves the export-count
gates** — the hard-coded 9-symbol arrays (each with an "expected exactly the 9" throw) in
`ci.yml` (×4 RIDs: win-x64, linux-bionic-x64/arm64, plus the iOS archive-visibility + link
checks), `android-instrumented.yml`, and `ios.yml` — **six-plus arrays and their prose**,
the `Exports.cs` header comment ("Nine exports"), and the two Kotlin version assertions.
This churn is the cost, counted honestly, and it is the reason §1c pays it only once.

### 1c. Generic-shaped, geolocation-wired (⛔ VETO FLAG)

Both new ABI members are **capability-agnostic**: `HostCallBegin` takes an `op` enum + a
flat-JSON arg string; `host_call_complete` returns an `int status` + a flat-JSON payload
string. **9.0 wires exactly one op — `Geolocation` — and nothing else.** But because the
shape is generic, **9.1 (notifications) / 9.2 (biometrics) / 9.3 (camera) add an `op`
constant + a status mapping + a host handler — and ZERO further ABI, ZERO export-gate
churn, ZERO drift-test moves.** The export-grow event is paid **once**, in the named-risk
phase, exactly mirroring the milestone thesis: *prove the permission machinery once, reuse
it three times.*

**⛔ Veto flag — the load-bearing call.** The completion export is the **firm** generic
commitment: `(requestId, int status, JSON payload)` is capability-independent, and even a
photo fits it as a **file path** in the payload (DoD #5 explicitly lets camera hand the
image across as a path, not bytes — the known stress test, and it passes). The **begin**
slot's op-genericity is **provisional**: if a later capability's request cannot express its
arguments as flat JSON, it may add its own typed begin slot (cheap, size-negotiated) while
**still sharing the completion export**. If review judges the generic `op` dispatch to be
the wrong abstraction this early, the fallback is a geolocation-specific
`GeolocationBegin` slot + this same generic completion export — the risk proof is
identical; only the reuse story shrinks. **Decide this before Gate 1 builds the ABI.**

**Normative rule (stated once).** After 9.0, the ABI is **generic and frozen at 10
exports / 80 bytes**; a new host capability is a typed .NET method + a `DevHostBridge` mock
+ an `op` constant + host handlers on both shells + (optionally) a `Device` façade — never
a new export, never a struct grow, never an export-gate edit.

## 2. Decision — the tri-state vocabulary (denial-as-data; wire-mirror discipline)

The completion's `int status` is a **wire-mirrored enum**, defined byte-identically on all
three sides (a `.NET` enum, a Kotlin enum, a Swift enum — pinned like the struct). The
host maps each platform's native permission/outcome into it **host-side**; **.NET only
ever sees the integer + the payload.**

| status | Name | Meaning | Payload |
|---|---|---|---|
| 0 | `Granted` | permission held; a fix was obtained | the fix (flat JSON) |
| 1 | `Denied` | user denied **this time**; a later request MAY prompt again (Android: `shouldShowRequestPermissionRationale` true) | none |
| 2 | `DeniedPermanently` | Android "don't ask again" (rationale false ∧ not granted after a request) / iOS `.denied` after its one prompt — re-request will NOT prompt; only Settings can change it | none |
| 3 | `Restricted` | iOS `.restricted` (parental controls / MDM) / Android device-policy restriction — the **user cannot** grant it | none |
| 4 | `LocationUnavailable` | permission fine, but services OS-disabled / no provider / fix timed out | none |
| 5 | `Error` | unexpected host error (mapped from a caught Kotlin/Swift throw, like the ABI's `-1`) | optional message |

**Normative rule (stated once): denial is a value of `status` (1/2/3), never an exception,
never a hang.** Every terminal outcome — grant, denial, restriction, no-fix, error — is a
completion with a status; the awaiting .NET `ValueTask` **always** resolves (or is
cancelled by the caller's token, §3). The both-shells mapping table
(Android `PackageManager.PERMISSION_*` + `shouldShowRequestPermissionRationale`;
iOS `CLAuthorizationStatus` + `CLError`) lives in the permission doc (§6).

**Payload wire form.** The fix crosses as a **flat JSON object of string→string**, reusing
the existing `WriteFlatJsonObject`/`ParseFlatJsonObject` pair (numbers string-encoded:
`{"lat":"52.3702","lng":"4.8952","accuracy":"12.0","altitude":"3.0","timestamp":"…"}`).
No new serializer — the same trim-cost decision the fetch headers took. `.NET` parses it
into a typed `GeolocationPosition`.

## 3. Decision — the async-suspension flow + the pending registry

A **new pending registry** on `NativeShellBridge`, the `s_pendingFetches` twin:

```
s_pendingHostCalls : ConcurrentDictionary<long, TaskCompletionSource<HostCallResult>>
HostCallResult      : (int Status, string? PayloadJson)
```

`RunContinuationsAsynchronously` (completion lands on a host thread). The flow:

1. `.NET` `GetCurrentPositionAsync` → `Interlocked` id → park the TCS → `HostCallBegin(id,
   op:Geolocation, args)`. Marshalling of the pointer args is a non-async helper (the
   `BeginFetch` split — pointers can't live in an async body).
2. The host runs the **whole permission dance** (check → prompt if undetermined → on grant,
   fetch a fix → on deny/restrict, note the status), then calls
   `blazornative_host_call_complete(id, status, payloadJson)`.
3. `CompleteHostCall(id, status, payload)` removes the id and resolves the TCS. Unknown id →
   `return 1`, logged, never throws (the `CompleteFetch` posture exactly).

**Surviving the suspension — the key insight: the pending state lives in the PROCESS, not
the Activity.** The `.NET` TCS registry is in the NativeAOT runtime (process-scoped),
**untouched by an Android Activity recreation**. Only the host-side `requestCode → requestId`
map must be app-scoped — and the retention law already forbids retaining an Activity
(app-scoped context only, bridge-extension.md (e)). So:

- **Android:** `ActivityCompat.requestPermissions(requestCode)` → the dialog suspends the
  app → `onRequestPermissionsResult` may land on a **recreated** Activity → `MainActivity`
  forwards `(requestCode, grantResults)` to the **app-scoped** `AndroidShellBridge`, which
  looks up the pending op by `requestCode`, obtains the fix (or notes denial), and calls
  `host_call_complete(requestId, …)`. The `.NET` registry never noticed the recreation.
- **iOS:** the `CLLocationManager` is **app-scoped** on `AppleShellBridge`; its delegate
  (`locationManagerDidChangeAuthorization` + `didUpdateLocations` / `didFailWithError`)
  carries the in-flight `requestId` and calls `host_call_complete`.

**If the app is KILLED during the prompt — the honest answer: the completion never
arrives.** So (normative rule, stated once): **the .NET API is cancellable** — a
`CancellationToken` threads through `GetCurrentPositionAsync`, and on cancel the id is
removed and the TCS `TrySetCanceled`d (the `FetchAsync` `ct.Register` pattern). A killed
process drops the registry with it; `ResetForTests`-style drain covers test isolation. **A
never-completing call is the caller's token to abandon, never a leaked pending entry and
never a hang.** One-in-flight-per-op on each shell keeps the host-side map trivial (a queue
is later work if a phase needs concurrent requests).

## 4. Decision — the .NET surface + packaging

### 4a. Layering: typed .NET surface over a generic frozen ABI

- **The ABI plumbing** (the `HostCallBegin` invoker, the pending registry, the wire enum)
  lives in **`BlazorNative.Runtime`** (`NativeShellBridge`) — internal detail.
- **The typed bridge method + result types** live in **`BlazorNative.Core`** on
  `IMobileBridge`, so **`DevHostBridge` (Core) can mock them** — the headless tri-state lane
  (exactly how clipboard put `ClipboardReadAsync` on `IMobileBridge` so the mock could serve
  it):

  ```csharp
  // IMobileBridge (Core)
  ValueTask<GeolocationResult> GetCurrentPositionAsync(CancellationToken ct = default);

  public readonly record struct GeolocationResult(GeolocationStatus Status, GeolocationPosition? Position);
  public readonly record struct GeolocationPosition(double Latitude, double Longitude,
      double Accuracy, double? Altitude, long TimestampUnixMs);
  public enum GeolocationStatus { Granted, Denied, DeniedPermanently, Restricted, LocationUnavailable, Error }
  ```

  `NativeShellBridge.GetCurrentPositionAsync` calls `InvokeHostCall(op:Geolocation, …)`,
  awaits the registry, parses the payload → `GeolocationResult`. `DevHostBridge` returns a
  **configurable position + configurable status** (so tests drive granted / denied /
  permanent / restricted / unavailable with no device).

- **`IGeolocation`** — the app-facing, DI-injectable ergonomic façade — lives in the new
  **`BlazorNative.Device`** package, a thin delegate over `IMobileBridge.GetCurrentPositionAsync`.
  App code injects `IGeolocation`, not the low-level bridge.

**API shape (normative): a SINGLE call that requests-then-fetches** —
`GetCurrentPositionAsync()` does the whole dance; the permission state machine is
**host-side**, .NET stays dumb (matching denial-as-data). A separate `RequestPermissionAsync`
is NOT the primary surface. A read-only `CheckPermissionAsync()` (no prompt) is provided
**only** so the demo can *show* the current state without triggering a dialog.

### 4b. Packaging — a new 7th library package `BlazorNative.Device` (⛔ VETO FLAG)

**Recommendation: create `BlazorNative.Device` now, in 9.0.** It is the natural home for
all four M9 device-API groups (geolocation now; notifications / biometrics / camera into it
in 9.1–9.3). **Nothing is published yet (0 releases)** — creating it now is free, whereas
re-homing public types *after* the first publish is a breaking move. The cost is **bounded
and one-time**, and the single-version-literal design absorbs most of it:

- **No new version mirror.** A `src/` package inherits `<Version>` from
  `src/Directory.Build.props` (the single literal, `x-release-please-version`) automatically;
  `release-please-config.json` (a single `.` component) needs **no edit** for a new library.
- **The bounded pin edits** (the 8.1 shipped-set, cross-pinned four ways): add
  `"BlazorNative.Device"` to `PackagePurityTests.ShippedAssemblies`, to
  `scripts/consumer-smoke.ps1` `$packages`, and to `samples/ConsumerSmoke/ConsumerSmoke.csproj`
  PackageReferences; `TheShippedSet_IsExactlyTheSrcCsprojs` then re-greens against the new
  `src/` count. If the generated template app should reference it, add a fourth annotated
  `<PackageReference … Version="0.0.0" />` to `MyBlazorNativeApp.csproj` (pinned by
  `TemplateDriftTests`).

**⛔ Veto flag — severable from the risk.** The package is orthogonal to the risk proof. If
Gate 1 shows the ABI/permission work consumes the budget, **defer the package**: park
`IGeolocation` in `BlazorNative.Core` (no 7th package, no shipped-set churn) and extract
`BlazorNative.Device` in Phase 9.4 — **still pre-publish, still free**. The non-negotiable
is the ABI + permission + geolocation-on-both-shells proof; the package is the negotiable.

## 5. Decision — geolocation specifics

- **Permissions.** Android: `ACCESS_FINE_LOCATION` + `ACCESS_COARSE_LOCATION` `<uses-permission>`
  lines added to **both** manifests to keep sample/template parity —
  `src/BlazorNative.Jni/src/androidMain/AndroidManifest.xml` (next to the existing
  `INTERNET`) and
  `templates/BlazorNative.Templates/content/BlazorNative.App/android/src/androidMain/AndroidManifest.xml`.
  iOS: `NSLocationWhenInUseUsageDescription` (a purpose string) added to the top `<dict>` of
  `src/BlazorNative.Apple/BnHost/Info.plist` (near the ATS block); **when-in-use**
  authorization only (foreground position — background/geofencing are M-out-of-scope). No
  iOS template tree exists today, so the template gets Android only (recorded as a gap, not
  a regression).
- **Provider choice.** Android uses the **platform `LocationManager`** (`getLastKnownLocation`
  / `requestSingleUpdate` on a `HandlerThread`), **not** `FusedLocationProviderClient` — the
  fused client needs Google Play Services, and the platform manager is fed directly by
  `adb emu geo fix`, so CI needs no Play-Services dep. FusedLocation is the production-grade
  alternative, deferred. iOS uses `CLLocationManager.requestLocation` (one-shot).
- **Mockability (so CI asserts a REAL fix and a REAL denial):**
  - **Android AVD:** `adb emu geo fix <lon> <lat> [alt]` seeds the emulator's location; the
    instrumented test asserts the returned `GeolocationPosition` matches. Denial:
    `adb shell pm revoke … ACCESS_FINE_LOCATION` (or deny the dialog via UiAutomator) →
    assert `Status == Denied`, **within a bounded await** (a hang reddens the test).
  - **iOS simulator:** a custom location (the scheme's default location / `simctl location`);
    when-in-use auth auto-granted (or the springboard alert driven) → assert the fix; the
    denial path asserted too. (Honesty note: a simulator location is a *set coordinate*, not
    a real GPS fix.)
  - **Headless (`.NET` + JVM):** `DevHostBridge` with a configurable position + status —
    where **all six status values** are asserted cheaply, including **`Denied` → no hang**.
    This is the central proof of the named risk.

## 6. Decision — the permission doc (DoD #1)

A **versioned extension of `docs/bridge-extension.md`** — a new top-level section
**(f) "The Permission Pattern — permission-gated async host calls"** (plus a one-line bump
to the doc's Status/version header noting the pattern now covers permission-gated caps and
the ABI is at 10 exports / 80 bytes). It is written as **the pattern 9.1/9.2/9.3 copy**, with
the 5.4 worked-example discipline — **geolocation IS the worked example**, as clipboard was
for the base pattern. Contents:

1. The **request → prompt → result flow** (begin slot → the host permission dance → the app
   suspension → `host_call_complete` → the pending registry → the TCS → the typed `.NET`
   result), with the fetch lineage called out.
2. The **pending-registry rule** — keyed by `requestId`, resolved off-lane, unknown-id
   benign, cancellation-safe, process-scoped (survives Activity recreation).
3. The **denial-as-data rule** — the tri-state enum (§2), stated once, and *why* it is never
   an exception or a hang.
4. The **both-shells vocabulary** — the Android ↔ enum ↔ iOS mapping table.
5. The **manifest / Info.plist declarations** each permission API needs (geolocation's shown;
   the shape the next three follow).
6. An update to bridge-extension.md **(c) the step-list**: "adding a permission-gated async
   cap" = pick an `op` constant + a status mapping + a typed `IMobileBridge` method + a
   `Device` façade — **NOT a new export** (the generic begin/complete are shared), and a note
   that the export grew **once** (9→10) in 9.0 and will not again this milestone.

### PROVEN on CI vs UNPROVEN (the M5-era simulator-honesty discipline)

- **PROVEN on CI:** the **tri-state as data** on all three surfaces (mock + AVD + simulator);
  a **fix from mocked coordinates** (`adb emu geo fix` / simulator location); the **denial
  path resolves without a hang**; **Activity-recreation survival** (AVD config-change); the
  **ABI grow** (drift tests 72→80, the 10-export gates on every RID).
- **UNPROVEN — the owner's physical Android phone only** (documented `adb`-over-USB steps, the
  honesty check, never a CI node): a **real GPS/network fix outdoors**; the **real system
  permission-dialog UX**; **`DeniedPermanently` ("don't ask again") across app restarts** and
  the Settings deep-link re-request.
- **UNPROVEN — iOS:** **simulator-scoped and labeled** (no physical iPhone — the M9 deferral,
  trigger = an Apple Developer account); a simulator location is a set coordinate, not a real
  GPS fix.

## 8. Risks (led by the ABI-vs-tri-state question)

| Risk | Mitigation |
|---|---|
| **The completion path can't carry a tri-state + an async-prompt gap** (the named risk) | It provably can: the fetch model already carries an arbitrary-delay, background-thread, keyed completion; §2 adds a status enum as a first-class field, resolved via the pending registry — denial is a value, asserted end-to-end at Gate 1 on the mock before any device work. |
| **Export grow 9→10 leaves a stale gate array** | ~6 hard-coded arrays + prose are enumerated in §1b; red-first at Gate 1; every RID's "expected exactly the N" throw is the tripwire. |
