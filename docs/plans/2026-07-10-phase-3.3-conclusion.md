## Verdict

**GREEN — M3 DoD #8, #9, #10 CLOSED.** The renderer is composition-grade: a real
component tree (`CompositionProbe` — root div → header → interleaved `ItemComponent`
badge → label → keyed `@foreach` list of `ItemComponent`s → Add/Insert/Remove buttons)
renders and mutates correctly on all three surfaces. On the AVD: the badge lands
**visibly at child index 1** (InsertIndex on-screen), insert-at-front puts the new item
**first** in the list container (`[item-3, item-1, item-2]`), remove-first promotes
item-2, and the badge's own `@onclick` increments its own counter. All five 3.2
`_childOrderMap` carryovers (a)–(e) are closed, nested components parent under their
parent's view (DoD #8), `AppendChildPatch` is deleted in favor of
`CreateNodePatch.InsertIndex` (DoD #10), and `NativeRenderer.StrictErrors` surfaces
what `HandleException` used to swallow (DoD #9). Hello stays the untouched demo; its
goldens moved by exactly one field (`InsertIndex: -1`) — the mount-path regression net
held on both sides. Version string: `BlazorNative.Runtime 0.8.0-phase-3.3`.
**Gate 3 shipped with zero iterations.**

Three first-class findings below: component disposal had been a silent no-op since 3.2,
strict mode flushed **zero** latent errors from the .NET test surface (an honest
positive), and both Kotlin-mirror gates closed with **zero contract mismatches**.

## The slot-list model as-built

`NativeWidgetTree` keys slot lists by `(componentId, parentNodeId)` — one bucket per
container per owning component (`RootParentKey = -1` for a component's root level).
Each `Slot` is a node (owns a view) or a component marker (occupies Blazor's sibling
position, owns no view). The diff cursor is `(ComponentId, Container, EmitParent)`:
`StepIn` descends through node slots into child buckets and through component slots
into that component's root-level bucket; positional edits resolve via `GetSlotAt`.
Disposal (`ProcessDisposedComponent`) emits RemoveNodePatch per root view, trims the
component's sibling slot (no-op when the parent's RemoveFrame already trimmed it), and
purges buckets + the component-parent map entry. One documented host contract: when an
ancestor element's removal and a child component's disposal land in the same batch, the
child's RemoveNodePatch targets views the host already detached — hosts treat unknown
RemoveNode ids as a no-op (WidgetMapper does).

**The one design deviation — `TranslateToHostInsertIndex`.** The design said "translate
slot position → view index by skipping component slots" (`TranslateToViewIndex`). That
is insufficient on its own: a child component's **root-level bucket shares its host
container** with the parent's other slots, so a bucket-local view index is not a host
child index. As-built, `TranslateToHostInsertIndex` = bucket-local
`TranslateToViewIndex` **plus the component-slot-chain base offset**
(`HostBaseOffset` — the component's own start position within the enclosing container,
recursing up the chain to the nearest element bucket or host root), and the −1/append
decision (`HasHostViewsAfter`) must consider host views after the position *anywhere in
the shared container*, not just in the local bucket. This is what keeps the Hello
golden append-shaped (−1) while giving keyed insert-at-front of a child component the
correct absolute front index. `CompositionProbeTests`' badge-at-InsertIndex-1 assertion
is the direct consumer: the badge's diff runs after the parent walk, and without the
base offset a blind bucket-local index would emit 0.

Quiet fallbacks in the translation (trimmed-slot → append, zero offset) report through
`NativeWidgetTree.ContractWarning` — visible on stderr under strict, silently tolerated
otherwise; a warning, not a violation, because the fallback result is applied either
way.

## Finding 1 — component disposal had been a silent no-op

Gate 1a's disposal work uncovered that the pre-3.3 disposal loop read
`_componentMap` — **a map nothing ever populated** — so `DisposedComponentIDs`
processing had been a silent no-op since it was written: no RemoveNodePatch, no
bookkeeping cleanup, ever. Latent because nothing had ever disposed a component
mid-session (Hello never does). The TDD RED case made it undeniable: disposal emitted
NO RemoveNodePatch and left slot buckets behind. The dead 3.2-era members
(`_componentMap`/`RegisterComponent`/`GetNodeId`/`_parentMap`/`SetParent`/
`GetParentId`/`ComponentCount`) were deleted with the fix. This is the third
silent-no-op-on-an-unexercised-path discovery in the project's record (after Bug A and
the 3.2 diff-cursor bug) — and precisely the failure class strict mode + the
adversarial matrix now exist to catch.

## Finding 2 — strict mode flushed ZERO latent .NET errors

The plan budgeted one triage iteration for latent swallowed errors that turning
`StrictErrors = true` across ALL .NET fixtures might flush. **It flushed none** — the
full 150-test suite is green with strict on everywhere, and the instrumented suite ran
green under a strict session on the AVD. An honest positive: after three phases of
paying down silent-swallow debt (Bug A, Bug B, the diff-cursor bug), the .NET test

## Finding 3 — zero contract mismatches across both mirror gates

Gate 2 (Kotlin adapter/golden mirror) and Gate 3 (WidgetMapper on real views) both

## Carryovers

**Phase 3.4 (Bn* + @bind + cascading — MUST-FIX first):**
- **Region frames are not walked** — the big one. `RenderFragment` parameters and
  `CascadingValue` render their ChildContent as a **Region frame**, which
  `ProcessFrame` does not descend into: components inside a region get no slot and no
  parent record, and the subtree is **silently dropped** (pre-3.3 behavior, unchanged;
  documented at `NativeWidgetTree`'s carryover block and on `CompositionProbe`, which
  deliberately avoids both constructs). `CascadingValue.ChildContent` IS a Region root,
  so **DoD #6 (cascading values) cannot close until Region frames are walked** — 3.4's
  first renderer task, before any `Bn*` component with a ChildContent parameter.
- **Strict-ordering one-shot + runner hardening** — the instrumented strict hook reads
  `BLAZORNATIVE_STRICT` once at first-session creation; `Os.setenv` in a `@BeforeClass`
  degrades to a **silent no-op** if a class sorting before "Composition" ever mounts
  first (ordering contract documented on both sides). Hardening: a custom
  `AndroidJUnitRunner` that sets the variable before any test class runs.
- **Trim/bridge probes run non-strict** — `blazornative_run_trim_probes` /
  `run_bridge_probes` build throwaway renderers directly and never consume the env
  read, so a swallowed error inside a probe would not surface under a strict session.
  Verified-benign nuance today; retires with the probe exports at M3 close.
- **RemoveComponent bucket scan** — O(total live buckets) per disposal; index by
  componentId if keyed `Bn*` lists dispose many children per frame.
- **`TranslateToViewIndex` memoization** — O(subtree slots) per mid-list insert
  (recursive over component slots); memoize per-diff if list sizes make it measurable.
- **Out-of-range `addView` crashes on the main thread** — the mapper applies the
  renderer-computed index verbatim; a bad index throws in `applyBatch` on the main
  looper (inherently strict placement, noted at the site). Acceptable while the
  renderer's translation is the only index source; a renderer-limit note, not a bug.
- **Attach-over-attach stale-watcher caveat remains** — change re-attach replaces the
  `watchers` map entry but leaves the previous TextWatcher registered on the EditText;
  the stale watcher dispatches a dead handlerId, absorbed by the rc-0 at-most-once
  contract (benign; waste + log noise under churn). Fix by keying watchers on
  (node, event) when 3.4's `@bind` work touches this path anyway.

**M3 close (ledger):**
- **Test scaffolding ships in the production .so** — `CompositionProbe` +
  `ItemComponent` (mount registry) and the trim/bridge probe components are all
  statically rooted via `HostSession`'s registry and ride along in every publish. The
  M3 final audit deletes both probe exports (`run_trim_probes`, `run_bridge_probes` +
  runners and label cstrings) as already planned — decide `CompositionProbe`'s fate in
