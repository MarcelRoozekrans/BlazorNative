# Phase 7.3 Design — Form Controls + a Real `picker`

**Date:** 2026-07-15
**Milestone:** M7 — Components + Razor (closes DoD #4)
**Predecessor:** Phase 7.2 — the onScroll wire + `BnList` (PR #96)
**Owns:** `BnCheckbox`, `BnSwitch`, `BnSlider`, and making **`picker` real** — the last stubbed
widget (NodeType 7 since Phase 2.5; a `Spinner`/placeholder that does nothing). Two-way bind on
all four.

---

## Decisions (recorded at phase-open; #2 flagged for veto)

1. **Three new NodeTypes: `checkbox = 8`, `switch = 9`, `slider = 10`.** A **wire-vocabulary
   extension, not an ABI change**: the NodeType rides the existing int32 field of the 48-byte
   patch; exports stay 9, the bridge stays 72 bytes. The alternative — `input` +
   a `type` prop — re-creates 6.2's ordering trap (the widget class must be chosen at
   `CreateNode`, but props arrive in later patches). The vocabulary lives in **three mirrors**
   (`FrameEncoder.MapNodeType`, `NativeFrameAdapter.kt`, `BnFrameAdapter.swift`) that must move
   together — `FrameEncoder` throws on unknown, both shells log-and-fallback; the existing
   NodeType tests extend on all three.
2. **`BnCheckbox` maps to `UISwitch` on iOS.** ⚠ *The call to veto:* iOS has **no native
   checkbox** — UIKit's toggle is `UISwitch` (React Native ships only `Switch` for the same
   reason). So `BnCheckbox` and `BnSwitch` are semantically identical on iOS (both `UISwitch`)
   and visually distinct only on Android (`MaterialCheckBox` vs `SwitchMaterial`/`Switch`). This
   is a **recorded "renders natively, not identically" divergence** — same wire, same events,
   same frames (both controls get fixed intrinsic sizes via the measure func), different pixels.
   The alternative (a custom-drawn checkbox on iOS) buys pixel parity at the cost of not being
   native — against the project's thesis.
3. **`picker` is the state-owner precedent.** Unlike input (text lives in .NET), a native
   picker owns its items *and* its selection UI. The wire: `items` rides `UpdateProp` as a
   **flat-JSON string array** (the `BnFlatJson` conventions; invariantly parsed, mirrored
   grammar in both shells — the 6.3 strict-parse discipline applies); `selectedIndex` rides
   `UpdateProp` as an int string; selection changes come back on the **existing `change`
   wire** with the new index as payload. Android `Spinner` + `ArrayAdapter`; iOS
   **`UIPickerView`** with the component as its own dataSource/delegate. Item changes re-bind
   the adapter and clamp/preserve the selection — the clamping rule is normative and mirrored.

## The components

| Component | Element → NodeType | Android | iOS | Bind surface |
|---|---|---|---|---|
| `BnCheckbox` | `checkbox` → 8 | `CheckBox` | `UISwitch` (decision 2) | `Checked`/`CheckedChanged` (`bool`) |
| `BnSwitch` | `switch` → 9 | `Switch` | `UISwitch` | `Checked`/`CheckedChanged` (`bool`) |
| `BnSlider` | `slider` → 10 | `SeekBar` | `UISlider` | `Value`/`ValueChanged` (`float`) + `Min`/`Max`/`Step` |
| `BnPicker` | `select` → 7 (exists) | `Spinner` | `UIPickerView` | `SelectedIndex`/`SelectedIndexChanged` (`int`) + `Items` |

- All four: the flex **item** surface (the `BnScroll` precedent), `Enabled`, and two-way bind via
  the `@bind-` pair convention (7.0 proved component-pair inference needs nothing new).
- **Change events**: checkbox/switch → payload `"true"/"false"`; slider → invariant float (the
  scroll-offset parse discipline); picker → the index. All on the existing `change` wire —
  no new event names.
- **The bind-loop guard**: programmatic value-set must not re-fire change (Android needs the
  `applyingBatch` listener guard — the 4.2 TextWatcher lesson; iOS `UISwitch.setOn`/`UISlider.setValue`
  don't fire `.valueChanged` — the 5.3 finding — but **verify per control, don't assume**;
  `UIPickerView.selectRow` also doesn't call the delegate — verify).
- **Measurement**: all four are leaves with **fixed intrinsic sizes** via the measure func (by
  NodeType — the 6.1 law). The intrinsic sizes are platform-native (a `UISwitch` is 51×31pt, a
  `CheckBox` is whatever Material says) — **frame parity applies to LAYOUT (declared sizes and
  placement), not to intrinsic control sizes**, which are the platform's own. The demo declares
  sizes wherever a number is asserted cross-platform; intrinsic sizing is asserted per-platform
  against the platform's own measurement (the 6.3 oracle method).

## The proof surface: `/form` (`BnFormDemo.razor`)

All four controls, each twice (bound pair + disabled), inside a `BnColumn`; a summary `BnText`
echoing the live state (the multi-control bind proof). Assertions: the wire golden (creates with
the new NodeTypes, props, attach counts); on-device — toggle/slide/pick each control → the bound
state round-trips → the echo re-renders; programmatic set does NOT re-fire change (the loop
guard, per control); picker item-change clamps selection per the normative rule; disabled
controls dispatch nothing.

## Risks

| Risk | Mitigation |
|---|---|
| The items flat-JSON grammar drifts between shells | One normative grammar in this design; strict whole-string parse both sides; a drift-style test if the tables are literal enough to pin |
| A control's programmatic set fires its change event on one platform | The loop-guard is per-control and TESTED per-control (the 4.2/5.3 lessons say the platforms differ here in both directions) |
| `UIPickerView` vs `Spinner` UX differences leak into assertions | Assert wire + state + declared frames, never native chrome |
| NodeType drift across the three mirrors | The existing all-NodeTypes tests extend on all three surfaces; FrameEncoder throws on unknown |
