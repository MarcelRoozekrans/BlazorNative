## Red-run ledger

Zero red CI runs across both workflows — both were green on their first real
attempt (bought by validating every parser locally against real output before
pushing). The full diagnosis ledger for the phase:

1. **`gh workflow run` 404 for a new workflow** — GitHub doesn't register a
   workflow for dispatch until it exists on the default branch or has run once.
   Bootstrapped with a temporary push trigger (`a9dfb7d`), removed after green
   (`5e461fd`); dispatch re-verified post-removal.
2. **Git-bash MSYS path mangling broke the milestone API call** (Gate 4 live
   run): `"/repos/…"` with a leading slash gets rewritten to a Windows path
   before `gh.exe` sees it → all 7 milestone creates 404'd, 25 issues cascaded.
   The Gate-0 failure surfacing caught it exactly as designed (inventory 37/0/0,
   exit 1, per-category failure summary). Fix: `e27e6af` (drop the leading
   slash). Clean re-run: 37/7/25, exit 0.
3. **Branch protection requires Pro on private personal repos** (Gate 4 →
   Gate 5 deviation): protection is free only on public repos, so it lands
   immediately after the public flip instead of before it (`729edd4`).

## Workflow change adopted

From Phase 4.1 on, phases merge **via PR**: `gh pr create` → review checkpoints
as before → CI green → `gh pr merge`. Local `merge --no-ff` is retired (branch
protection forbids direct pushes to `main`, admins included). Required check:
`build-test`.
