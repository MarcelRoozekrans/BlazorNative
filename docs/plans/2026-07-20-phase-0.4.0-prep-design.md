# 0.4.0 preparation — the `ConfigureServices` seam, the template pack goes live, the KDoc sweep

**Milestone:** 11 — Production Readiness. **Phase:** the in-repo work that must ship in the **0.4.0**
release before Phase 11.1's ZeroAlloc showcase can tell its full DI story. **Closes:** three
owner-approved changes that 11.1's dogfooding surfaced, bundled into one minor release because they
share one publish and one owner go:

1. **The `ConfigureServices` app-service DI seam** — a framework `feat` (new public API), the enabler
   for 11.1 Gate D (the showcase upgrades to app-level `[Inject]` on `ZeroAlloc.Inject`-registered
   *app* services).
2. **Publishing `BlazorNative.Templates` to nuget.org** — dogfooding finding: the template pack has
   **never** been published (an M8 deferral), so `dotnet new blazornative` does not work for a
   stranger, and the quick-start doc says it does.
3. **The M8-ledgered KDoc sweep** — its trigger was *"before the first Release that publishes the
   template pack"*, and that Release is this one.

This is a **design doc**: it settles the shape, the call-sites, and the invariant-respecting wiring.
No code, no workflow edits land here.

## Why these three, why now, why 0.4.0

Phase 11.1's design (`docs/plans/2026-07-20-phase-11.1-design.md`) spans **two** owner-authorized
publishes. The props version is **0.3.0** today (`src/Directory.Build.props:112`,
`.release-please-manifest.json`), lined up as release PR #152; 11.1 Gate C then builds the seam
in-repo and publishes **0.4.0**, at which point the showcase can register its own services through
it. The seam is a **new public API**, so semver forces a **minor** bump — `feat` → 0.4.0 (the 0.x
ladder in `.github/workflows/release-please.yml`: `feat:` → minor while `bump-minor-pre-major` is
live). Once we are cutting a 0.4.0 release anyway, the two ledgered items whose trigger is *"the
first Release that publishes the template pack"* fire in the same PR: publishing the pack, and the
KDoc sweep that was gated behind it. One publish, one owner go, three changes.

## Section 1 — the `ConfigureServices` seam

### The gap, grounded

`HostSession.EnsureSession()` (`src/BlazorNative.Runtime/HostSession.cs:297`–`359`) is the
composition root, and it is sealed inside `BlazorNative.Runtime`:

```
:312   var services = new ServiceCollection();
:313   services.AddBlazorNativeRendererServices();     // ZeroAlloc.Inject-generated
:319   services.AddBlazorNativeHttp();                 // over AddBlazorNativeHttpServices()
:325   services.AddSingleton<IMobileBridge, NativeShellBridge>();
:329   services.AddBlazorNativeDevice();
:334   services.AddSingleton<INavigationManager, NativeNavigationManager>();
:335   ServiceProvider provider = services.BuildServiceProvider();
```

There is **no app hook** between the framework's registrations and `BuildServiceProvider()`. The
app's *only* registration surface today is `BlazorNativeApp.RegisterPages(...)`
(`src/BlazorNative.Runtime/BlazorNativeApp.cs:96`) — pages, not services. So an app author who writes
a `[Singleton]` service, gets `ZeroAlloc.Inject.Generator` to emit its `AddMyAppServices()`
registration method, and `[Inject]`s it into a page **has nowhere to call that method**. A page can
`[Inject]` framework services (they went through the block above) but not app-authored ones. This is
friction item **F3** in the 11.1 ledger, and it is the whole reason 0.4.0 is a second publish rather
than a docs pass.

`ConfigureServices` does not exist yet — a source grep for it across `src/` returns nothing.

### The public API shape

Add one static entry point to the existing app-facing surface, alongside `RegisterPages`:

```csharp
public static class BlazorNativeApp   // src/BlazorNative.Runtime/BlazorNativeApp.cs:84
{
    // captured at module-init, exactly like RegisterPages
    public static void ConfigureServices(Action<IServiceCollection> configure);
}
```

- **Capture vs consume.** `RegisterPages` is *captured* at the app's `[ModuleInitializer]` (the file
  header at `BlazorNativeApp.cs:18`–`25` explains why: a NativeLib has no managed `Main`, so
  `[ModuleInitializer]` is the app's one hook that runs eagerly inside `blazornative_init` before it
  returns). `ConfigureServices` is captured the same way and at the same site — the app calls it from
  its module initializer, so app services register with zero managed `Main`, identically to pages.
  It is **consumed once**, later, by `EnsureSession()`.
- **The store.** One `Volatile`-guarded static field, `Action<IServiceCollection>? s_configureServices`,
  matching the existing `s_renderer` / `s_navigation` / `s_lock` idiom in `HostSession`.
  `ConfigureServices` does `Volatile.Write`; `EnsureSession()` does `Volatile.Read` once, under the
  existing `s_lock`.
- **No-op if never called.** The stock generated app never calls it (the template does not); the
  field stays null and `EnsureSession()` skips the invocation. Nothing changes for an app that does
  not use the seam — which is the baseline (`bn-baseline`) app in 11.1.
- **Ordering — the exact call-site.** The captured delegate is invoked **after** the framework's own
  registrations (`:313`–`:334`) and **immediately before** `BuildServiceProvider()` (`:335`):

  ```
  :334   services.AddSingleton<INavigationManager, NativeNavigationManager>();
  ─────  var configure = Volatile.Read(ref s_configureServices);
  ─────  configure?.Invoke(services);        // app services, after framework, before build
  :335   ServiceProvider provider = services.BuildServiceProvider();
  ```

- **Additive, last-wins.** Framework registrations come first *deliberately*, so an app that
  re-registers a framework contract is a conscious last-write (MS.DI resolves the last registration
  for a single service). An app that only adds its own contracts is purely additive. There is **one**
  `ServiceProvider` — no child scope, no second container.

### Why it is AOT-safe

`Action<IServiceCollection>` + `Microsoft.Extensions.DependencyInjection` is reflection-free: the
delegate is an ordinary closure and MS.DI's registration methods are direct calls. There is nothing
here that ILC cannot see. And this is **not a new mechanism** — it is the framework's own mechanism
opened to the app. BlazorNative is **already a `ZeroAlloc.Inject` consumer in production**: the
Renderer csproj pins `ZeroAlloc.Inject 1.7.1` + generator, `NativeRenderer` is `[Singleton]`, and
`EnsureSession()` already composes the generator's output (`AddBlazorNativeRendererServices()` at
`:313`, `AddBlazorNativeHttpServices()` under `AddBlazorNativeHttp()` at `:319`) into MS.DI before
`BuildServiceProvider()`. `ZeroAlloc.Inject` source-generates `IServiceCollection` registration
methods *into* MS.DI — it does not build a rival container. So an app registering its own
`ZeroAlloc.Inject`-generated `AddMyAppServices()` through this seam is running the **same** three-
source-generator ILC compile (Razor + `BlazorNative.RouteGen` + `ZeroAlloc.Inject.Generator`) the
framework already survives the trim gate with. The seam adds a call site, not a code-generation
ecosystem.

### The red-first test

In `tests/BlazorNative.Runtime.Tests`: register a marker app service via `ConfigureServices`
(`services.AddSingleton<IFoo, Foo>()`, or a generated `AddShowcaseServices()`), then assert a page
that `[Inject]`s `IFoo` resolves the app-registered instance through the framework's provider —
i.e. an app `[Inject]` reaches app services identically to how it reaches framework services.
**Red-first:** without the seam the resolve throws (nothing registered `IFoo`); with it, it resolves.
The test serializes via the existing "host-session" collection (it touches the process-wide session
singletons) and restores registration state in a `finally`, using the existing
`BlazorNativeApp.ResetRegistrationForTests()` seam (`BlazorNativeApp.cs:104`) as its precedent; a
companion `ConfigureServices` reset (or a captured-delegate clear) is added the same way.

### Public-API cross-ref

This is new public surface → cross-ref **M11 DoD #4 / Phase 11.3**. 11.1 (and this 0.4.0 prep) *adds*
the `ConfigureServices` surface; **Phase 11.3 freezes it** in the `PublicAPI.*.txt` baseline. The
design constraint here is only that the surface be **minimal and final in shape** now, so 11.3 has a
stable thing to bless: one static method, one `Action<IServiceCollection>` parameter, no overloads
speculatively added.

## Section 2 — publishing the template pack (respecting the M8 invariants)

This is the delicate part. The template pack was built in M8 to be **deliberately un-publishable by
the release machinery** (`templates/BlazorNative.Templates/BlazorNative.Templates.csproj` header,
lines 28–32: *"IT DOES NOT PUBLISH THIS PHASE. release.yml's push enumerates artifacts/packages and
this pack is not in it"*). 0.4.0 wires the publish **around** every invariant that decision
protects, breaking none.

### The binding constraints (state, then design around)

| # | Invariant | Where it lives |
|---|---|---|
| C1 | Template is a **content-only pack** (`PackageType=Template`), on a **separate feed** `artifacts/templates`, never `artifacts/packages` | `BlazorNative.Templates.csproj:50`; `template-smoke.ps1:45`–`47` |
| C2 | The main feed is **exactly `7 nupkg + 6 snupkg`** — a stray nupkg there breaks the count | `consumer-smoke.ps1:148`,`:152` |
| C3 | `PackagePurityTests.TheShippedSet_IsExactlyTheSrcCsprojs` proves `src/` holds exactly the shipped csprojs — the template lives **outside** `src/` and must stay there | `BlazorNative.Templates.csproj:12`; enforced by the src/-enumeration in `release-preflight.ps1` (`Get-ShippedPackageIds`) and `TemplateDriftTests` |
| C4 | Content-only pack has **no snupkg** | `template-smoke.ps1` packs one nupkg, asserts no symbols |
| C5 | `${{ secrets.NUGET_API_KEY }}` appears **exactly once**, only in `release-please.yml` (comments counted) | `ReleaseWorkflowPinTests.TheNugetApiKey_IsReferencedExactlyOnce_AndOnlyInReleasePleaseYml`; the ref is the push job's job-level `env:` (`release-please.yml:379`–`380`) |
| C6 | The release path **never** overrides the version (`-p:Version=` / `-p:PackageVersion=`) | `ReleaseWorkflowPinTests.TheReleaseWorkflow_NeverOverridesTheVersion` |

> ⚠ **Grounding correction — the count is 7+6, not 6+5.** The task brief, and the template csproj's
> own header (`BlazorNative.Templates.csproj:25`–`26`, *"consumer-smoke.ps1 asserts an EXACT `6 nupkg
> + 5 snupkg`"*), both say 6+5. That is **stale**: `BlazorNative.Device` joined as the seventh library
> package in Phase 9.0, and `consumer-smoke.ps1:148`/`:152` now assert **7 nupkg + 6 snupkg** (its
> `$packages` array is Core/Renderer/Http/Components/Device/Runtime/Analyzers = 7).
> `template-smoke.ps1:47` already says *"7 nupkg + 6 snupkg"* correctly. This design targets the
> **live** invariant (7+6). Several "six" / "6+5" references in the template csproj header
> (`:12`, `:25`) predate Device and are stale in the same way — see Gate C notes; correcting them is
> part of shipping this pack, since the header also claims the pack "does not publish".

### The wiring — where each piece goes

The release runs in `release-please.yml` as **validate** (packs + proves, never sees the key) →
artifact hand-off → **push** (transports, the only job that sees the key). The template rides the
**same split**, on its **own feed**, with **no second key**.

1. **Validate — pack + prove the template, on its own feed.** The validate job today runs
   `consumer-smoke.ps1` (packs the 7 → `artifacts/packages`) and uploads that feed as artifact
   `packages` (`release-please.yml:329`–`343`). Add, **after** it, a step running
   `scripts/template-smoke.ps1`. That script already: packs `BlazorNative.Templates` →
   `artifacts/templates` (its **own** feed, C1), then **proves the pack** end-to-end — installs it,
   `dotnet new blazornative`s a fresh app, and publishes it to assert the tripwire trio (4 IL2072 +
   the 10 exports + the page in the image) plus the generated deep-link map. Its precondition is a
   populated `artifacts/packages`, which the preceding `consumer-smoke.ps1` step supplies — the exact
   ci.yml ordering (`template-smoke.ps1:87`–`89`). validate is `windows-latest` with the VS toolchain
   (`release-please.yml:280`), which template-smoke's AOT publish + `dumpbin` export check require
   (`template-smoke.ps1:311`–`338`). So the release lane packs the template with the **same script**
   the required `build-test` lane proves on every PR — the "pack once, validate those bytes, push
   those bytes" rule (`release-please.yml:325`–`332`), now applied to the template too.
   Then add a **second** `upload-artifact` for `artifacts/templates` as artifact `templates`
   (`if-no-files-found: error`, mirroring `:342`).

2. **Preflight — cover the template id.** `release-preflight.ps1 -Preflight` queries nuget.org for
   each shipped id at the props version, so `--skip-duplicate` in the push is safe (it reds on a
   forgotten bump *before* the key exists). But it enumerates ids from **`src/`** (`Get-ShippedPackageIds`,
   `release-preflight.ps1:242`–`252`), and the template lives outside `src/` (C3) — so today the
   preflight **does not see the template id**. Extend `-Preflight` to also query
   **`BlazorNative.Templates`**, with the id read from the pack csproj's `<PackageId>`
   (`BlazorNative.Templates.csproj:51`) — its single home — rather than added as a bare literal. This
   is the same "necessity, not drift" shape the pack csproj already uses for its own metadata
   (`:67`–`78`): the template is the one publishable that `src/` enumeration structurally cannot
   reach, so it is a **named** eighth query, sourced from the one file that owns its identity. The
   preflight's two positive-control arms (`Invoke-ControlArms`) already prove the scanner can see and
   distinguish before any absence claim, so extending the queried set inherits that non-vacuity.

3. **Push — a second `dotnet nuget push`, same key, other feed.** The push job has no checkout;
   it downloads `packages` and pushes `artifacts/packages/*.nupkg` reusing the **job-level**
   `$env:NUGET_API_KEY` (`release-please.yml:426`–`443`). Add:
   - a **second** `download-artifact` for `templates` → `artifacts/templates`;
   - a **second** push step: `dotnet nuget push artifacts/templates/*.nupkg --api-key
     $env:NUGET_API_KEY --source https://api.nuget.org/v3/index.json --skip-duplicate`, with the same
     "assert ≥1 nupkg, error on zero" guard the packages push uses (`:430`).

   This reuses the **existing** `$env:NUGET_API_KEY` shell variable — it does **not** add a second
   `${{ secrets.NUGET_API_KEY }}` expression, so C5 holds (the pin counts expression-form refs, of
   which there stays exactly one, job-level). The glob is `artifacts/templates/*.nupkg` — one file,
   a different directory — so `artifacts/packages` is **never touched** and C2 (7+6) is untouched.
   The template carries no `.snupkg` (C4), so the "symbols ride adjacent" note needs no change on the
   template feed. No `-p:Version` appears anywhere — the template version comes from its own
   `<Version>` (below), packed by template-smoke, never overridden — so C6 holds.

### Version tracking — nothing new

The template's `<Version>` is already `x-release-please-version`-annotated
(`BlazorNative.Templates.csproj:61`, `0.3.0`), rewritten by release-please's `extra-files` on every
release PR and held equal to `src/Directory.Build.props` by
`TemplateDriftTests.EveryTemplateVersionLiteral_EqualsTheSharedProps`. So the pack file that
template-smoke produces is already named `BlazorNative.Templates.<props version>.nupkg`
(`template-smoke.ps1:212`). **No new version wiring** — only the push *path* is new; the version
*tracking* already exists and already agrees with the six mirrors.

### The invariant-preservation proof (what Gate B must show)

- `consumer-smoke.ps1` still asserts **7 nupkg + 6 snupkg** in `artifacts/packages` — green
  (nothing added to that feed).
- template-smoke produces exactly **one** `BlazorNative.Templates.<version>.nupkg` in
  `artifacts/templates`, zero warnings.
- `ReleaseWorkflowPinTests` (both facts) stays green: still one `${{ secrets.NUGET_API_KEY }}`, still
  no `-p:Version=`.
- The preflight now reports the template id alongside the seven, all free at the release version.

## Section 3 — the KDoc sweep (scope from the M8 ledger)

### The ledger item, and what 11.0 already resolved

The M8 final audit (`docs/plans/2026-07-17-milestone-8-final-audit.md:223`–`265`) deferred *"the KDoc
sweep + the map extraction (ONE item)"* to fire *"before the first Release that publishes
`BlazorNative.Templates`"*. It was **one** item because the KDoc fix was **welded to the map
extraction**: fixing the stale KDoc without extracting the inline route map would need a new,
brittle excision in `TemplateDriftTests`, and extracting the map was *"a Kotlin change + a 184-test
device re-run"*.

**Phase 11.0's RouteGen already did the map extraction half.** The inline `DEEP_LINK_COMPONENTS` map
is gone from `MainActivity.kt`; both the repo shell and the template now read the build-time-
generated `res/raw/blazornative_routes.json` through an **identical** `loadDeepLinkRoutes()`
(`MainActivity.kt:401`–`440`). `TemplateDriftTests` records the consequence: the map excision is
**retired**, Pin B is **retired**, and the MainActivity byte-identity pin now excises only **two**
divergences — the fallback literal and the template-only `import <namespace>.R`
(`TemplateDriftTests.cs:82`–`95`, `:850`–`898`). So the "map extraction" half is **done**, and the
device-re-run cost the ledger feared was 11.0's to pay — and 11.0 paid it.

**What remains is the KDoc-correctness half, and it is now decoupled and comments-only.**

### The concrete task list

The stale prose the audit cited still stands, in `MainActivity.kt`, **byte-identical in both copies**
(so pinned identical by `TemplateDriftTests.TemplateMainActivity_EqualsTheRepos_ModuloTheFallbackAndTheImport`),
and therefore wrong in the template:

1. **Class KDoc, line 34** — *"register frame callback → register shell bridge → mount **BnDemo**, or
   the [EXTRA_COMPONENT] Intent-extra override"*. A generated app's default page is **`BnStarterPage`**,
   not `BnDemo` (the code's own fallback is `?: "BnStarterPage"`, `MainActivity.kt:197`, pinned by
   `TemplateDefaultFallbackLiteral_IsTheTemplateManifestsDefaultComponent`). A `dotnet new` user reads
   KDoc naming a page their app does not have, contradicting their own code 163 lines down.
2. **Companion `EXTRA_COMPONENT` KDoc, lines 68–72** — *"Absent → **"BnDemo"** since Phase 3.4 Gate 4
   — the Bn* library demo … IS the launcher experience; tests that pin another component's shape pass
   it explicitly ("HelloComponent", "CompositionProbe")"*. This is **load-bearing engineering truth
   for the repo shell** (whose default *is* `BnDemo`) but false and repo-internal for the template.

**Recommended approach — genericize the two KDoc passages in BOTH copies so they name no specific
default page, keeping byte-identity and adding no excision.** The specific default is now *data* (the
`AppPages` manifest's `DefaultRoute` row + the one already-excised fallback literal), so the KDoc
should describe the *mechanism* ("mounts the default page registered at `BlazorNativeApp.DefaultRoute`,
or the `EXTRA_COMPONENT` override") rather than hard-code `BnDemo`. This resolves the M8 three-way
dilemma cleanly now that the map is out:
- it is **not** option 1 (diverge the prose → a fourth brittle excision — rejected by 8.4's rule);
- it is option 2 (genericize both), and 8.4's objection to it — *"deletes engineering truth from the
  repo's shell to fix the template's copy"* — is answered by relocating the repo-only truth (the
  "BnDemo is the launcher experience / tests pin other shapes explicitly" note) into a **non-KDoc
  code comment** beside the repo's own fallback, which the byte-identity pin does not compare against
  the template (it excises the fallback region). The engineering truth stays in the repo; only the
  byte-pinned KDoc goes generic.

3. **Light completeness scan of the rest of the template's byte-identical Kotlin surface** — the other
   shell public API the template ships (`BlazorNativeRuntime.kt`, `ShellBridge.kt`,
   `AndroidShellBridge.kt`, `WidgetMapper.kt`, and the `jni` binding files, all pinned byte-identical
   by `TemplateShellSources_AreByteIdenticalToTheRepos`) for any *other* KDoc that names a page, demo,
   or repo-internal type a generated app does not have. **Scope honestly:** there is **no Kotlin
   KDoc-coverage gate** (the C# side has `BnEnforceDocCoverage` / CS1591; Kotlin has no equivalent
   here), so this is an **editorial correctness pass over the known stale surface**, not a new
   coverage bar. The MainActivity `BnDemo` prose is the one ledgered, evidence-backed instance; the
   scan confirms there is not a second.

### Cost, restated

Because 11.0 already extracted the map, this is now a **comments-only** change with **no behavioral
delta** — so it does **not** need the "184-test device re-run" the M8 ledger attached to the welded
item (that cost belonged to the map extraction, which is done). It rides the standard PR lanes; the
byte-identity and fallback pins are the proof it stayed coherent. (Instrumented-test counts have
grown since the 184 estimate — ~210 today per the project ledger — but a KDoc-only change touches
none of them.)

## Section 4 — gates + sequencing

- **Gate A — the seam.** Build `BlazorNativeApp.ConfigureServices(Action<IServiceCollection>)`;
  capture at module-init, consume in `EnsureSession()` after `:334` and before
  `BuildServiceProvider()` at `:335`; land the **red-first** test (app service resolves via page
  `[Inject]`). This is the `feat` that makes the release a **minor** → 0.4.0. Coordinate the surface
  with Phase 11.3 (keep it minimal for the PublicAPI baseline to freeze).
- **Gate B — the template-publish wiring + preflight.** Add the validate template-pack/prove step +
  `templates` upload; extend the preflight to the template id; add the push job's second
  download + `dotnet nuget push` reusing `$env:NUGET_API_KEY`. **Prove, by a validate/dry-run, that
  the 7+6 invariant on `artifacts/packages` still holds and exactly one template nupkg is produced on
  `artifacts/templates`** — and that `ReleaseWorkflowPinTests` stays green (one key, no version
  override). Update the now-false pack csproj header (it claims the pack "does not publish" and says
  "6+5"/"six") as part of this gate.
- **Gate C — the KDoc sweep.** Genericize the two `MainActivity.kt` KDoc passages in both copies
  (byte-identity preserved, no new excision), relocate the repo-only engineering truth to a non-KDoc
  comment, and run the light completeness scan. Comments-only; the byte-identity + fallback pins are
  the proof.
- **Gate D — conclusion → the 0.4.0 release PR.** The three changes land; the release PR is cut. **The
  actual 0.4.0 publish is owner-authorized and live on nuget.org** — a gate on owner go, not
  automatic (the release-PR merge is the go; `release_created` gates the publish jobs). On publish:
  the template becomes installable (`dotnet new install BlazorNative.Templates` /
  `dotnet new blazornative` from nuget.org), and the **F0/F2 quick-start doc line is corrected** —
  `website/docs/getting-started/quick-start.md:13`–`14` currently says *"the template is published on
  nuget.org (v0.1.0)"*, which is **false today** (never published) and version-stale; shipping the
  pack is what makes the first clause true, and the version clause is corrected in the same edit.
  This unblocks 11.1 Gate D (the showcase upgrades to app-level DI through the new seam).

## Risks

- **Breaking the 7+6 invariant.** The single largest hazard, and the reason the template rides its
  **own** feed and its **own** push glob. Mitigation: the template nupkg never enters
  `artifacts/packages`; `consumer-smoke.ps1` continues to assert 7+6 unchanged; Gate B's dry-run is
  the explicit check.
- **The seam's public-API surface.** A new public method is permanent once frozen. Mitigation:
  minimal shape (one method, one `Action` parameter, no overloads), coordinated with Phase 11.3's
  `PublicAPI.*.txt` baseline (M11 DoD #4).
- **The single-key pin.** Adding a template push is exactly the kind of change that tempts a second
  `${{ secrets.NUGET_API_KEY }}`. Mitigation: the template push is a **step inside the existing push
  job**, reusing the job-level `$env:NUGET_API_KEY`; `ReleaseWorkflowPinTests` reds the required lane
  if a second expression-form reference ever appears.
- **nuget.org is append-only.** A template published at 0.4.0 cannot be recalled, only unlisted.
  Mitigation: the extended preflight covers the template id before the key exists, and
  `--skip-duplicate` makes a re-run of a partially-failed push safe.
