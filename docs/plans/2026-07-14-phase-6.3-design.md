# Phase 6.3 Design — URL Images

**Date:** 2026-07-14
**Milestone:** M6 — Real-UI Foundation (closes DoD #5 — the last capability criterion)
**Predecessor:** Phase 6.2 — real scrolling ([conclusion](2026-07-14-phase-6.2-conclusion.md), PR #68)
**Owns:** the `image` NodeType — wired into the mapper switch since Phase 2.5, and the last
stubbed leaf. It creates a widget on both platforms today and measures to **zero**, because
neither shell has a source-loading path.

---

## Decisions (locked at brainstorm, 2026-07-14)

1. **Coil (Android) + Kingfisher (iOS)** — the platform-standard image libraries, rather than a
   hand-rolled loader. They bring decode, downsampling, cancellation and caching for free.
   **The parity risk is real and is managed explicitly** — see "The parity contract" below.
2. **Intrinsic sizing with re-layout on load.** An image with no declared size measures **0×0**
   until its bytes arrive; when they do, the shell marks the Yoga node **dirty** and the tree
   re-solves, so the image snaps to its natural size and reflows its siblings. This is the React
   Native model, and it re-uses the `markDirty` path that 6.1 built and 6.2 mutation-proved. An
   explicit `Width`/`Height` short-circuits it entirely (sized immediately, no reflow, ever).
3. **`BnImage` exposes `Src` + the flex *item* surface. Nothing else.** No `Placeholder`, no
   `OnError`, no `ContentMode` — each of those changes *measurement* (does a placeholder measure
   like the image? does a failure keep the reserved space?), so each deserves its own design
   rather than a footnote in the fetch phase. Ledgered for M7.
4. **Proven on a new `/image` page, plus one image inside the scroll demo** — see "The proof
   surface", which is designed so that **`BnScrollDemo`'s 6.2 parity table does not move**.

## No ABI change

`image` is already NodeType **5**, and `MapElementToNodeType` already maps the `img` element to
it. `Src` rides the existing `UpdateProp` wire (it is a *prop*, not a style — it does not belong
in the `YogaStyleAttributes`/`VisualStyleAttributes` partition). Still **9 exports + the 72-byte
bridge**. There is no binary path on the wire and there does not need to be: **the shell fetches
the bytes**, exactly as React Native does.

## The parity contract (the thing two libraries put at risk)

Three phases of this milestone have been spent proving that Android and iOS compute *identical
numbers*. Two different image libraries, with different default cache policies, decode paths and
downsampling rules, are the most plausible way to lose that. So the contract is drawn where it
matters and nowhere else:

> **Parity is asserted on FRAMES, not on cache internals.** Coil and Kingfisher may cache,
> evict, decode and prefetch however they see fit. What must be identical on both platforms is:
> the **measured size** an image reports to Yoga, **when** it reports it (the reflow), and what
> happens on **failure** and **cancellation**.

Normative, and mirrored in both shells:

| Rule | Behaviour |
|---|---|
| **THE UNIT — one file pixel is one dp/pt** | **The measured size of an image is the PIXEL COUNT OF THE DECODED FILE, read directly as dp / points.** A 160px-wide PNG measures **160**, on every device, at every density, on both platforms. See "The unit" below — this row is the one that decides whether the two shells compute the same number at all, and **both platforms hand you a plausible wrong number for free** |
| **Measured size, no `Width`/`Height`** | `0 × 0` before the bytes land; the image's **natural pixel size in dp/pt** (the row above) after |
| **Measured size, `Width`/`Height` set** | exactly those, always — the bytes never move the frame |
| **On load** | set the image, `markDirty` the Yoga node, re-solve. **One** reflow, never two |
| **On failure** | the node keeps measuring `0 × 0` and logs. It does **not** reserve space, and it does **not** retry |
| **On `Src` change** | the in-flight request is cancelled; the node measures `0 × 0` again until the new bytes land |
| **On `Src` → `null` (or `""`)** | **the reflow in the other direction.** The renderer emits `UpdateProp(nodeId, "src", null)` today (a `RemoveAttribute` on a non-style name — `BnButton.Enabled`'s precedent). The shell **cancels**, **clears** the image, `markDirty`s and re-solves: an intrinsic node collapses back to `0 × 0` and **its siblings move back UP**. A definite (`Width`+`Height`) node keeps its frame — only the pixels go. **An EMPTY string takes this path too, and is never fetched** — a shell decision, stated here so it is one decision rather than two (iOS's `URL(string: "")` is `nil`, which is a crash shaped like an NPE if a shell force-unwraps it). Silence here is two shells guessing differently |
| **On node removal** | the in-flight request is cancelled — a completion **must not** paint into a removed or recycled node |
| **The stale-callback guard is GENERATION *and* IDENTITY** | in **every** terminal callback, not only the painting one. Node ids **restart at 1 after a reset**, so a stale callback carrying generation 1 can meet a brand-new node that is also on generation 1: the generations MATCH, and only the view's identity separates them. A guard that asks the generation alone lets the stale callback evict the LIVE request's cancellation handle — leaving a request nothing can cancel, whose completion (on iOS) runs against a **freed `YGNodeRef`** |
| **No downsampling that changes the reported size** | the measured size is the image's natural size, not the decoder's chosen sample size |
| **Content mode: ASPECT-FIT, explicitly, on both shells** | Android `ImageView.ScaleType.FIT_CENTER`; iOS `contentMode = .scaleAspectFit`. The two **framework defaults disagree** (Android fit-centres, `UIImageView` **stretches**), and the divergence is **frame-neutral** — so every number in the tables survives it and no test on either side can see it. What breaks is "renders identically". Aspect-fit because it cannot lie about the pixels, because it is free on the intrinsic path (there the frame IS the natural size), and because it is what an M7 `ContentMode` would default to. **Deferring the `ContentMode` API does not defer the default** — 6.1's `clipChildren = false` precedent: it costs one line to align the two shells |

The middle pair are the classic image bugs (the stale completion painting into a recycled row;
the decoder quietly halving a size). Both get a test on both platforms.

### The unit: one file pixel is one dp/pt (normative)

The contract used to say only *"the image's natural pixel size in dp/pt"*, and that is exactly the
ambiguity Gate 2 had to resolve — it resolved it correctly, in a Kotlin comment, where Gate 3 will
never read it. It is normative, it belongs here, and **each platform's corollary is a trap that
looks like the recommended thing to do**:

- **Android** — read the **decoded bitmap's own `bitmap.width`**. **Never `intrinsicWidth`**: a
  `BitmapDrawable` scales that by `targetDensity / bitmap.density`, so it is device- *and*
  version-dependent. Never the generic `view.measure` path either — an `ImageView`'s intrinsic size
  is its drawable's size in **pixels**, which 6.1's measure func would divide by density (61dp for a
  160px file on the Pixel 6 AVD's 2.625). This is why `image` gets a measure function of its own.
  And ask Coil for `Size.ORIGINAL`, or it sizes the decode to the **display**.
- **iOS** — `UIImage(data:)` has **`scale == 1`**, so its `size` is *already* the file's pixel count
  expressed in points. Gate 3 gets the rule for free — **and must not take it away:**
  - **DO NOT set Kingfisher's `.scaleFactor(UIScreen.main.scale)`.** It is Kingfisher's *own
    documented idiom* for crisp images and it is the first thing an implementer reaches for. It
    divides the reported size by the screen scale: **160/3 ≈ 53.3pt against Android's 160dp.**
  - **DO NOT add a downsampling / resizing processor**, for the same reason and for the
    "no downsampling" row.

  **No single-device test in either suite can catch this.** Each shell stays internally consistent;
  only the two frame tables side by side disagree, and nothing compares them automatically. Which is
  why the fixture's pixel size is now **declared in `BnImageDemo.cs`** (`IntrinsicNaturalWidthPx` =
  160, `…HeightPx` = 90, and the fixed fixture's 64 × 48) and **both shells assert the same
  constants**, pinned by a drift test in the `build-test` lane. Each shell **also** asserts against
  its **decoded bitmap** — that, and only that, is what pins "no downsampling"; a shell asserting
  only its own transcription asserts nothing.

**The honest consequence, and it is a real one.** A "2×" asset (a 320 × 180 file meant to be shown
at 160 × 90) renders at **twice its intended physical size**. So does a 3×. This is the rule working
as specified, and it is **consistent on both platforms** — iOS does exactly this by default for
network images. **It is not inherited from React Native**: RN auto-sizes only *bundled* assets, where
the packager's `@2x`/`@3x` filename suffix supplies the scale; for **network** images RN requires an
explicit size and does not answer this question at all. So decision 2 inherits RN's model for the
**reflow** and *not* for the **unit** — the unit is un-inherited ground this phase broke by itself.
Ledgered for Gate 4: a scale-aware `Src` (an `@2x` convention, or a `Scale` parameter) is an M7
question.

### A memory-cache hit completes SYNCHRONOUSLY, inside the patch batch (normative)

**Both libraries do this, and it is the most surprising thing in the phase.** Coil 2 dispatches on
`Dispatchers.Main.immediate`, so an `enqueue` issued *from* the main thread on a **memory-cache hit**
runs the request — decode lookup, `onSuccess`, everything the shell does in it — **to completion
before `enqueue` returns**. **Kingfisher's `setImage` calls its `completionHandler` synchronously on
a memory hit too.** And the shell issues its request from `UpdateProp("src", …)`, which runs on the
main thread **inside the patch batch**.

So on the **second mount** of any page whose images the process has already fetched — `/image` after
a back-and-forward, `BnScrollDemo`'s row image (same fixture file), *any* real app navigating back to
a screen — the completion runs **inside `applyBatch`**. Two things follow, and a shell that does not
know it gets both wrong:

1. **The completion's re-solve must NOT run inside a batch.** The batch's own `CommitFrame` re-solves
   the whole tree at the end; a re-solve from inside it runs Yoga against a **half-applied tree** and
   then again at commit — **two reflows, where the contract says one**. Worse, the shells guard patch
   application against re-entrant dispatch (a programmatic `setText` firing a change event), and
   **that guard is not re-entrant**: the inner re-solve's `finally` clears it *for the rest of the
   batch*, so every subsequent `setText`/`value`/focus change in the batch dispatches back into .NET —
   the change → re-render → setText loop the guard exists to prevent. The fix is one line: the
   completion's re-solve is a **no-op while a batch is in flight**.
2. **Record the in-flight request only if it is still live.** The completion already ran, so its own
   bookkeeping found nothing to clear; storing the (already-finished) cancellation handle afterwards
   leaks it permanently — the "nothing left in flight" invariant never returns to zero, and a later
   removal "cancels" a request that finished long ago.

**Every existing test clears the image caches before mounting**, which is right for the BEFORE-table
gate and means this whole path is **unexercised** unless a test deliberately **mounts twice with a
warm cache**. Both gates owe that test, and it asserts the frames *and* that nothing is left in
flight.

**Scope of the `Src` → `null` row.** The *wire* emission is pinned in .NET
(`BnComponentTests.BnImage_SrcGoesNull_EmitsUpdatePropNullOnThePropWire`). No demo page flips a
`Src` at run time — adding one would rewrite this phase's frame tables — so Gates 2/3 assert the
*shell* behaviour at the **mapper level**, where they already assert `UpdateProp` behaviour, and
not on `/image`. That is the whole of the 6.3 decision on it: normative in the contract, tested on
both sides, not demoed.

### CI must never touch the public internet — and cleartext HTTP is blocked by default

Tests load from a **loopback HTTP fixture** the shells serve in-process. A suite whose green
depends on a remote host is not a suite. Two things this implies that are *not* obvious, and whose
failure modes are **silent** (see "The proof surface" — a blocked load is indistinguishable from
the 404 the failure case expects, so two of three assertions stay green on a device that never
loaded a byte):

- **Android** — `targetSdk ≥ 28` blocks cleartext. Already **covered by inheritance** for
  instrumented tests: `src/BlazorNative.Jni/src/debug/res/xml/network_security_config.xml` permits
  cleartext to `127.0.0.1`/`localhost`, and instrumented tests run the debug build (Phase 3.1's
  lesson). **The release corollary**: `src/androidMain/res/xml/network_security_config.xml` permits
  none, so a **release** build of the demo shows three *failed* images on `/image`. That is correct
  and expected; the fix is never to weaken the release config.
- **iOS** — **ATS blocks arbitrary HTTP, and Kingfisher fetches through `URLSession`, so ATS
  applies.** `src/BlazorNative.Apple/BnHost/Info.plist` has **no** `NSAppTransportSecurity` key.
  **Gate 3 must add `NSAllowsLocalNetworking = true`** (the narrow loopback exemption — *not*
  `NSAllowsArbitraryLoads`).
- **And `127.0.0.1` is not the same thing on the two platforms.** On the AVD it is the emulated
  device's own loopback. **In the iOS simulator it is the host Mac's** — the simulator shares the
  host's network stack. The offline guarantee holds either way, but the fixture port (**8099**) is
  therefore **host-global on the macOS runner**: Gate 3's server must bind exclusively, fail loudly
  on a taken port, and prove the bytes are *ours* before asserting a frame (the fixture-precondition
  assertions below are that probe).

## The model

`image` is already in `measuredNodeTypes` (6.1 attaches the measure func **by NodeType**), and
6.1's Gate 3 review already made the iOS `image` a real `UIImageView` — so an *empty* image
measures `.zero` on both shells today, by construction rather than by accident. That is exactly
the pre-load state this phase needs.

```
UpdateProp(nodeId, "src", "https://…")
      → shell cancels any in-flight request for this node
      → Coil / Kingfisher fetch + decode (off the main thread)
      → on the MAIN thread: set the drawable/UIImage
                            markDirty(yogaNode)          ← the 6.1 path
                            re-solve + apply frames      ← the 6.2 path
```

The measure func then reports the image's natural size (or the declared `Width`/`Height`), and
the layout absorbs it. **Cancellation on removal is not hygiene — it is the same class of bug as
6.2's dangling `YGNodeRef`:** a completion firing into a purged node would touch a freed Yoga
handle on iOS.

## The proof surface

**A new `/image` page (`BnImageDemo`)** — both measurement paths, so both are pinned:
- **[0]** a **fixed-size** image (`Width="200" Height="120"`) → sized immediately, and **its frame
  must not move when the bytes land** (the "no reflow" assertion);
- **[1]** an **intrinsic-size** image → measures `0 × 0`, then reflows to its natural size on load,
  and its **sibling below it moves down** (the reflow assertion — that sibling's y is the proof);
- **[2]** a **failing URL** → stays `0 × 0`, logs, reserves nothing.

> ### The synchronization gate (normative — and it is the phase's most dangerous failure mode)
>
> **The "after" frame table may only be asserted once ALL THREE requests have terminated.**
>
> Two of the three cases assert **"nothing moved"** — [0]'s frame is definite, and [2]'s failure
> reserves nothing. **Both of them pass on a device that never loaded a byte.** The wire carries no
> completion signal by design (no `OnError`, no `OnLoad` — decision 3), so an implementer who
> asserts the AFTER table straight after mount proves *nothing* and is green on two of three. And a
> **blocked** load (cleartext, above) is indistinguishable from the 404 that [2] *expects*: with
> HTTP blocked, only [1] goes red while [0] and [2] certify a contract that is entirely broken.
>
> So each shell **awaits its own loader's per-node terminal callback for all three nodes** — Coil's
> `ImageRequest.Listener` (`onSuccess`/`onError`), Kingfisher's `completionHandler`
> (`.success`/`.failure`) — counted to three, with a timeout that FAILS. **Not** band I's movement:
> that witnesses only case [1] and says nothing about whether [0]'s or [2]'s request ever finished.
>
> And the assertions must include a **positive** one: `Wi > 0 && Hi > 0` on the intrinsic image's
> computed frame, so "band F did not move" means *"the bytes landed and did not move it"* rather
> than *"no bytes landed"*.

**The fixture's contract** — asserted **in the test, on the decoded fixture, before any frame**:
`0 < Wi ≤ 300` (a section is 300 wide; a wider fixture asks a clamping question this phase does not
answer), `Hi > 0` (Hi *is* the reflow), `(Wfixed, Hfixed) ≠ (200, 120)` (otherwise "it measures
200 × 120" is a coincidence, not a proof that the declared size short-circuits measurement), and
**no size-changing downsampling**. An unasserted fixture constraint is a coincidence waiting to
happen — and these assertions double as the probe that the bytes came from *our* fixture server.

**One image inside `BnScrollDemo`** — images-in-a-scroll-viewport is the most common real usage,
and leaving it unexercised until someone hits it is how you find out the hard way. But
`BnScrollDemo`'s frame table is the **6.2 cross-platform parity contract**, so it must not move.

> **The image goes *inside* an existing row, at a fixed size.** The rows are `Height="80"` and
> the image is smaller, so **every frame in the 6.2 table is unchanged** — the same ten rows at
> the same `y = 80·i`, the same 800dp content, the same 600 scroll range. Only that one row gains
> a child. Images-in-scroll is proven; the parity contract is untouched.

That constraint is deliberate and is a non-negotiable of this phase: **if the scroll demo's
numbers move, the change is wrong.**

## Verification (the GREEN bar)

1. `BnImageDemo` renders on the AVD and the iOS simulator with **the same frames** — the fixed
   image at its declared size, the intrinsic image at its natural size, the failing image at zero.
2. **The reflow is real**: the intrinsic image's sibling *moves* when the bytes land (asserted by
   frame, before and after).
3. **The fixed image does NOT reflow** — its frame is identical before and after the load.
4. **Cancellation**: a `Src` change and a node removal each cancel in flight; a late completion
   paints nothing (and, on iOS, touches no freed node). **The guard is generation AND identity, in
   every callback**, and the reset collision (same id, same generation, different view) is pinned by
   a unit test — it cannot be staged on a device.
5. `BnScrollDemo`'s **6.2 frame table is unchanged**, with an image now inside a row.
6. CI never reaches the public internet.
7. **The page mounts TWICE with a WARM cache and the frames are the same** — the synchronous
   memory-hit path (above), which every cache-clearing test leaves unexercised. Nothing is left in
   flight afterwards.
8. Counts bumped and asserted on every surface.

## Risks

| Risk | Mitigation |
|---|---|
| **Two libraries, two behaviours** — the parity the milestone exists to prove | The contract above is normative and mirrored; parity is asserted on *frames*, and the libraries are configured to it (no size-changing downsampling). If a library will not honour a rule, the rule wins and the library is replaced |
| A late completion paints into a **recycled or freed** node (the classic list bug; on iOS a freed `YGNodeRef`) | Cancel on `Src` change and on removal; pin both with tests. This is 6.2's dangling-pointer lesson in a new costume |
| Two reflows (0×0 → placeholder → natural) | No placeholder in this phase — decision 3. One reflow, asserted |
| A decoder's downsampling changes the reported size, silently, on one platform only | The measured size is the *natural* size by contract, and the demo's intrinsic image has a known pixel size asserted on both |
| CI depending on a remote host | Local fixtures + a loopback server. Never the public internet |
| Touching `BnScrollDemo`'s parity table | The image goes **inside** an existing 80dp row, smaller than it. If a number in the 6.2 table moves, the change is wrong |
