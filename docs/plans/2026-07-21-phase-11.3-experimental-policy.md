# BlazorNative — the `[Experimental]` policy

**Phase 11.3, Gate C.** When `System.Diagnostics.CodeAnalysis.ExperimentalAttribute` is warranted,
what its diagnostic ids are, and the recorded finding that the surface as it stands today warrants
**zero** uses.

This is a policy, not a decoration. Gate C's deliverable was always the rule; the count of
attributes it ships is an output of applying the rule, and this phase's output is zero. A gate that
ships zero attributes passes only if it shows its work, so §4 walks the candidates it rejected.

---

## 1. The reserved range: `BN1xxx`

| Range | Owner | Meaning |
|---|---|---|
| `BN0xxx` | `BlazorNative.Analyzers` | Shipped analyzer rules. **7 in use** — `BN0004`, `BN0010`, `BN0011`, `BN0013`, `BN0014`, `BN0020`, `BN0021` — pinned in both directions by the diagnostic-id roster test in `tests/BlazorNative.Analyzers.Tests` (Gate B, criterion A2). |
| **`BN1xxx`** | **`[Experimental]`** | **Reserved by this document.** Every `[Experimental]` id BlazorNative ever issues comes from here, allocated sequentially from `BN1001`. |

The two ranges must not overlap, and the reason is a consumer-facing one: both ids end up in the
same places — an `.editorconfig` severity line, a `<NoWarn>`, a `#pragma warning disable`. A
consumer suppressing `BN1003` must be suppressing exactly one thing. The ranges being disjoint by
declaration is what makes "suppress this specific experiment" a safe instruction to give.

**Ids are never reused.** When an experimental API graduates or is deleted, its id is retired, not
recycled — a stale `NoWarn` in a consumer's project file must not silently start suppressing a
different, live warning. `BN0004`'s wire-id neighbours already work this way (see
`BlazorNativePatchKind.AppendChild = 2`, reserved-dormant); this is the same discipline applied to
diagnostics.

**Allocating one.** Add the id to the table in this file in the same PR that adds the attribute,
with the type or member it marks and the condition that graduates it. An `[Experimental]` with no
written graduation condition is an experiment nobody can end.

### Ids issued so far

**None.** See §4.

## 2. What `[Experimental]` means here

> **`[Experimental]` means: *this signature may change in a minor release.* It is not a synonym
> for *internal*, and it is not a synonym for *buggy*.**

The consumer consequence is deliberately severe and must be understood before reaching for it:

- Every **use site** in consumer code becomes a **compile ERROR**, not a warning.
- The consumer's only ways forward are `<NoWarn>BN1xxx</NoWarn>`, an `.editorconfig` severity
  downgrade, or a `#pragma warning disable BN1xxx` around each use.
- The error fires on *use*, not on *reference* — but "use" includes touching one member of a
  package the consumer had to reference for entirely unrelated reasons.

That last point is what makes it an instrument of last resort. It buys **informed consent** — a
consumer cannot adopt an experimental API by accident — and it charges for that consent in build
breakage. Only spend it where the consent is genuinely worth having.

---

## 3. Choosing the mark: a three-way decision

The three marks answer three different questions, and picking the wrong one tells the consumer
something false.

| Mark | The claim it makes | Use it when |
|---|---|---|
| **`[Experimental("BN1xxx")]`** | *"This is real consumer surface, we want you to use it, and we expect its **shape** to change."* | The API is **for consumers**, is **shipped**, and is **unproven** — typically the first cut of a capability that has not met its real backend or a real device. |
| **`[EditorBrowsable(EditorBrowsableState.Never)]` + an xmldoc line** | *"This is not for you."* Not unstable — **not addressed to you at all.** | The type is public for a **mechanical** reason: a C-ABI export, cross-assembly composition, a source generator, a Roslyn entry point. The NOT-API tier, in full. |
| **Documentation alone** (the tier table, a `<remarks>`) | *"Use it; we will tell you when it moves."* | PROVISIONAL surface — real, supported, baselined, but not advertised as frozen. |

**The two failure modes this table exists to prevent:**

- **`[Experimental]` on NOT-API.** It would say *"this may change"* about `NativeRenderer`, which
  is false — `NativeRenderer` has not moved in five milestones and both shells depend on it. The
  true statement is *"this is not a consumer contract"*, and it costs nobody a build error.
- **`[EditorBrowsable(Never)]` on something genuinely experimental.** Hiding a half-proven API from
  IntelliSense while leaving it usable is the worst of both: the consumer who finds it anyway gets
  no warning at all.

### `[EditorBrowsable(Never)]` is a signpost, not a barrier

Stated plainly because the marking done at Gate C should not be mistaken for enforcement:

- It hides the member from **IntelliSense completion in *other* assemblies**. Inside the declaring
  assembly (and in the same solution, depending on the IDE) it stays visible.
- It **does not restrict access**. `new SetStylePatch(...)` still compiles. Nothing is `internal`,
  nothing is analyzer-blocked, no build fails.
- Its whole effect is on **discovery**: a consumer no longer stumbles into the type; they must go
  looking for it, past an xmldoc that tells them not to.

That is the honest ceiling of a non-breaking change, and it is why the tier table records the
residual risk rather than claiming the mark closed it. The genuinely closing fix for
`NativeRenderer` — `internal` + `InternalsVisibleTo` — is a **breaking** change, recorded as 1.0
criterion S3.

## 4. The finding: zero uses warranted today

**Applying §3 to all 88 public types yields no `[Experimental]`.** The bucket that would have been
the obvious home for it — NOT-API, 31 types — is the one §3 explicitly routes elsewhere: those
types are *not for you*, not *not yet*. That leaves only STABLE (55, which by definition do not
qualify) and the **2 PROVISIONAL** types, walked below.

### 4.1 `DevHostBridge` (`BlazorNative.Core`) — rejected

**Why it looks like a candidate.** It is the type most likely to change shape. Its ~70 public
members are mock-shaped seeding hooks and canned results, and issue
[#25](https://github.com/MarcelRoozekrans/BlazorNative/issues/25) — the consumer-facing test
harness — will deliberately reshape it. "The shape will move" is exactly what `[Experimental]`
announces.

**Why it is rejected anyway — the consequence lands in the wrong place.** `DevHostBridge`'s callers
are **test projects**. Making every use a compile error means every consumer who writes a test
against BlazorNative starts with a broken build and a `NoWarn` line. The framework would be
charging its most diligent users — the ones writing tests — an entry fee, to warn them about churn
in a **test double**, where churn costs a mechanical test-file edit and never a production
incident. The severity of the instrument is inverted against the severity of the risk.

The second reason is that its instability is **already communicated in the right register**: it is
PROVISIONAL in the tier table, it is baselined so any change is visible in a PR diff, and #25 is
public. A consumer who reads the tier table knows; a consumer who does not read it is not helped by
a build error whose fix is one line of suppression they will paste without reading.

**What would change the verdict.** If #25 ships a *new, deliberately provisional* harness surface —
a redesigned entry point rather than the incremental evolution of the existing one — that surface
is a fair candidate for `BN1001`, precisely because it would be new API a consumer has to opt into
rather than existing API they already depend on. Marking the **predecessor** experimental now
achieves nothing; marking the **successor** experimental at birth is exactly the intended use.

### 4.2 `BlazorNative.Renderer.ServiceCollectionExtensions` (`AddBlazorNativeRenderer`) — rejected

**Why it looks like a candidate.** The tier table's own reason for PROVISIONAL says its shape is
not ours to promise: it is a one-line delegate to ZeroAlloc.Inject-generated code
(`RendererServices.cs:21`). If the generator's output changes, this method's shape changes with it
— an honest "may move" story.

**Why it is rejected — the surface is one method and the smallest possible one.**
`AddBlazorNativeRenderer(this IServiceCollection)` returning `IServiceCollection` is the verbatim
`Microsoft.Extensions.DependencyInjection` idiom. There is no plausible v2 of that signature; what
could change is the *generated registrations behind it*, which are not part of the signature and
which `[Experimental]` would not be warning about. Marking it would warn about the wrong noun.

Second, its audience is the **custom-host** consumer — someone deliberately composing their own
host session instead of using `BlazorNativeApp`. That is already the most advanced path in the
framework, taken on purpose by someone who has read how the framework composes. A build error tells
them nothing the tier table has not, and it would also fire in the framework's own composition root
(`HostSession.cs:333`) and in the ABI harnesses, which would then carry suppressions — noise inside
the repo for a signal aimed outside it.

**What would change the verdict.** A second overload, or any parameterisation (options, a
configuration callback, a renderer-selection argument). The moment this stops being a
zero-argument idiom and starts having a *design*, that design is unproven and `BN1xxx` earns its
place.

### 4.3 The two near-misses outside PROVISIONAL, for completeness

- **`BlazorNativeApp.ConfigureServices`** (STABLE, shipped in 0.4.0 / #159) is the newest API in the
  repo and therefore the least-mileaged. Rejected: its signature is `Action<IServiceCollection>` —
  the smallest possible shape, matching the ASP.NET Core idiom exactly — and it was designed in
  11.1 and proven **at runtime** against the published binary via an ABI harness. Marking the
  headline feature of a release experimental one release after shipping it would be a worse signal
  than leaving it plain: it would read as a retraction of something that was, in fact, verified.
- **The capability result types** (`GeolocationResult`, `CameraStatus`, and the other 9) are the
  strongest freeze candidates in the repo, not the weakest: each is wire-mirrored into Kotlin and
  Swift, and each has been exercised on both shells. Nothing to mark.

### 4.4 Where the first `BN1001` is expected

Reserved, not issued, for the first member of a genuinely unproven capability:

- the **FCM push** surface if [#16](https://github.com/MarcelRoozekrans/BlazorNative/issues/16)
  lands without a real Firebase project behind it, and
- **haptics / in-app purchases / background tasks** from
  [#18](https://github.com/MarcelRoozekrans/BlazorNative/issues/18) if any of them ships ahead of
  device proof.

Both share the property the current surface lacks: **new consumer-facing API whose shape is a
guess**. That is the case `[Experimental]` was built for, and reserving `BN1xxx` now means the
first one can ship with a mark instead of an apology.
