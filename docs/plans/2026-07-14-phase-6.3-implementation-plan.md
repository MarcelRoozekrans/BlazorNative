# Phase 6.3 Implementation Plan — URL Images

> **For Claude:** REQUIRED SUB-SKILL: `superpowers:subagent-driven-development` — fresh implementer
> subagent per gate, spec review then code-quality review after each.

**Goal:** the `image` NodeType stops being a stub. `<BnImage Src="…" />` fetches, decodes, measures
and lays out — identically on Android and iOS.

**Design:** [2026-07-14-phase-6.3-design.md](2026-07-14-phase-6.3-design.md) — the four locked
decisions, **the parity contract**, and the proof surface. Not up for re-litigation.

**Architecture:** the shell fetches the bytes (there is no binary path on the wire and there does
not need to be — this is the RN model). `Src` rides the existing `UpdateProp` wire. On load: set the
image on the main thread → `markDirty` the Yoga node (the 6.1 path) → re-solve and apply frames (the
6.2 path). **No ABI change** (9 exports + the 72-byte bridge; `image` is already NodeType 5 and `img`
already maps to it).

**Branch:** `phase-6.3-images`.

---

## Non-negotiables

1. **No C-ABI / wire change.** No new export, no new NodeType, no new patch kind. `Src` is a
   **prop** (`UpdateProp`), not a style — it does not belong in the
   `YogaStyleAttributes`/`VisualStyleAttributes` partition.
2. **`BnScrollDemo`'s 6.2 frame table must not move.** The image goes **inside** an existing 80dp
   row and is smaller than it, so every frame in that table is unchanged — the same ten rows at
   `y = 80·i`, the same 800dp content, the same 600 scroll range. **If a number in the 6.2 table
   moves, the change is wrong.**
3. **The parity contract** (design §"The parity contract") is normative and mirrored in both shells.
   Parity is asserted on **frames**, not on cache internals. In particular:
   - **THE UNIT — one file pixel is one dp/pt.** The measured size is the **pixel count of the
     decoded file**, read directly as dp/points: a 160px PNG measures **160**, on every device.
     Android reads `bitmap.width` (**never** `intrinsicWidth` — density-scaled); iOS gets it free
     from `UIImage(data:).scale == 1` and **must not set Kingfisher's `scaleFactor`** or any
     downsampling processor. **No single-device test in either suite can catch a breach of this**
     (each shell stays internally consistent) — so the fixture's pixel size is **declared in
     `BnImageDemo.cs`**, both shells assert the same constants, and a drift test pins each shell's
     transcription. Each shell also asserts against its **decoded bitmap** — that is what pins "no
     downsampling".
   - no `Width`/`Height` → measures **0×0** before the bytes, the image's **natural** size after;
   - `Width`/`Height` set → exactly those, always — **the bytes never move the frame**;
   - on load → set + `markDirty` + re-solve. **One** reflow, never two;
   - on failure → stays `0×0`, logs, reserves nothing, does not retry;
   - on `Src` change **and on node removal** → the in-flight request is **cancelled**; a late
     completion paints nothing;
   - on **`Src` → `null`** → cancel, **clear** the image, `markDirty`, re-solve → the node collapses
     to `0×0` and **its siblings move back UP**. The renderer emits this today
     (`UpdateProp(nodeId, "src", null)`); it is a **second reflow direction**, and it is tested at
     the shells' **mapper** level (no demo page flips a `Src` at run time);
   - **no downsampling that changes the reported size** — the measured size is the natural size.
   - **the content mode is ASPECT-FIT, set EXPLICITLY on both shells** (Android `FIT_CENTER`, iOS
     `.scaleAspectFit`). The framework defaults disagree (Android fit-centres, `UIImageView`
     stretches) and the divergence is **frame-neutral**, so no frame table can catch it. Deferring
     the `ContentMode` *API* does not defer the *default*.
4. **Cancellation is memory safety, not hygiene.** A completion firing into a purged node would
   touch a **freed `YGNodeRef`** on iOS — 6.2's dangling-pointer lesson in a new costume. **The
   guard is GENERATION *and* IDENTITY, in every terminal callback** — node ids restart at 1 after a
   reset, so a stale callback carrying generation 1 meets a brand-new node also on generation 1 and
   the generations MATCH. A generation-only guard lets it evict the **live** request's cancellation
   handle, and the un-cancellable request that survives is the freed-`YGNodeRef` bug wearing the
   fix's clothes.
4b. **A MEMORY-CACHE HIT COMPLETES SYNCHRONOUSLY, INSIDE THE PATCH BATCH** — Coil dispatches on
   `Dispatchers.Main.immediate`; **Kingfisher's `setImage` calls its completionHandler synchronously
   on a memory hit too.** So the completion's re-solve must **no-op inside a batch** (the batch's own
   commit re-solves — otherwise Yoga runs twice, against a half-applied tree, and the non-re-entrant
   re-entrancy guard is cleared for the rest of the batch), and the in-flight handle must be
   **recorded only if it is still live**. Every cache-clearing test leaves this path unexercised:
   **both gates owe a "mount twice with a warm cache" test.** Design §"A memory-cache hit completes
   SYNCHRONOUSLY".
5. **CI never touches the public internet.** A loopback HTTP fixture the shells serve in-process. A
   suite whose green depends on a remote host is not a suite. **Cleartext HTTP is blocked by default
   on both platforms** — see the design's "CI must never touch the public internet" section, and the
   per-gate bullets below. The failure mode is **silent**.
6. **The AFTER frames may only be asserted once ALL THREE requests have terminated**, and the suite
   must contain a **positive** assertion (`Wi > 0`, `Hi > 0`). Two of the three cases assert
   "nothing moved", so a suite that skips the gate is green having loaded nothing — design
   §"The synchronization gate". This is the phase's most dangerous failure mode.
7. **Everything 6.1 and 6.2 pinned still holds** — read both conclusions. The measure func attaches
   **by NodeType** (`image` is already in the set); Yoga's own rounding is off; one `RemoveNodePatch`
   means a whole subtree (and the scroll content node is in it); the style value grammar is strict;
   `alignContent` is never set; `BnScroll` is a flex item.

## Gate 1 — .NET: `BnImage`, the demo page, the goldens

**Files:** create `src/BlazorNative.Components/BnImage.cs`, `BnImageDemo.cs`; register the `/image`
route (`HostSession` + `NativeNavigationManager` — **all mirrors move together**); modify
`BnScrollDemo.cs`; test `tests/BlazorNative.Runtime.Tests/BnImageDemoTests.cs` + the existing
`BnScrollDemoTests.cs`.

- **`BnImage`** — emits the `img` element (already mapped to NodeType 5). `Src` (a prop) plus the
  flex **item** surface. **No `Placeholder`, no `OnError`, no `ContentMode`** (design decision 3 —
  each changes *measurement* and deserves its own design). Follow `BnScroll`'s precedent for the
  item-vs-container split, and note that `BnImage` is a **leaf**: no `ChildContent`.
- **`BnImageDemo`** at `/image` — the three cases, all with predictable frames:
  1. a **fixed-size** image (`Width="200" Height="120"`) with a **sibling below it**, so "the frame
     does not move when the bytes land" is assertable;
  2. an **intrinsic-size** image (no `Width`/`Height`) with a **sibling below it** — that sibling's
     `y` is the *proof of the reflow*;
  3. a **failing URL** — stays `0×0`, reserves nothing (the sibling below it does not move).
  Plus a "← Back" button. Document the expected frames in the header, as `BnLayoutDemo` and
  `BnScrollDemo` do — **you are writing the spec the shells will be held to**, so every number must
  be *derived*, not narrated.
- **`BnScrollDemo`** gains **one image inside an existing row** (fixed size, smaller than the 80dp
  row). Its golden's create/style counts move by exactly that one node; **the frame table does
  not.** Say so in the test.
- Gate 1 close: `dotnet test` green (record the new count; bump `ci.yml` with provenance). JVM
  unchanged.

## Gate 2 — Android: Coil

**Files:** `src/BlazorNative.Jni/build.gradle.kts` (the Coil dep), `WidgetMapper.kt`,
`YogaLayout.kt` (if the measure func needs the natural size); instrumented tests.

**Read `BnImageDemo.cs`'s header first — the whole of it.** It is the spec, and the three sections
that are new since the plan was written (the synchronization gate, the cleartext opt-in, the fixture
contract) are the ones a Gate 2 implementer can produce a fully green suite without.

1. Wire **Coil** to the normative contract. `UpdateProp("src", …)` → cancel any in-flight request
   for that node → enqueue → **on the main thread**: set the drawable, `markDirty` the Yoga node,
   re-solve. (`markDirty` already exists and is mutation-proven — do not invent a second path.)
2. **The measure func must report the natural size**, not a decoder-chosen sample size. Configure
   Coil accordingly and *assert the number* against a fixture whose pixel size you know.
3. **Cancel on `Src` change and on node removal.** The removal path must hook the existing purge
   (`handleRemove`) — a completion into a removed node must paint nothing. And handle
   **`src` → `null`** (`UpdatePropPatch.value == null`, which the renderer emits today): cancel,
   `setImageDrawable(null)`, `markDirty`, re-solve — the node collapses to 0×0 and its siblings move
   back **up**. A mapper unit test, not a demo-page test.
4. **CLEARTEXT — already covered, but know why.** `targetSdk ≥ 28` blocks cleartext HTTP, and this
   repo ate that in Phase 3.1. `src/debug/res/xml/network_security_config.xml` already permits
   cleartext to `127.0.0.1`/`localhost`, and **instrumented tests run the debug build** — so Gate 2
   needs no manifest change. **The corollary, which must be recorded in the conclusion:** the
   `androidMain` config permits none, so a **release** build of the demo shows three *failed* images
   on `/image`. Correct and expected; do not "fix" it by weakening the release config.
5. **THE SYNCHRONIZATION GATE (non-negotiable #6).** The AFTER frames may only be read once **all
   three** requests have terminated. Attach an `ImageRequest.Listener` per node and count
   `onSuccess`/`onError` (and treat `onCancel` as a setup failure) to a `CountDownLatch(3)` with a
   timeout that FAILS the test. **Do not** poll until band I moves: that witnesses only case [1], and
   [0]'s and [2]'s "nothing moved" assertions pass on a device that fetched nothing.
6. **THE FIXTURE'S CONTRACT (non-negotiable #6), asserted in the test, on the DECODED fixture,
   before any frame is read:** `0 < Wi ≤ 300`; `Hi > 0`; `(Wfixed, Hfixed) ≠ (200, 120)`; and **no
   size-changing downsampling** (the measured size is the *natural* size). Do **not** pick a
   400 × 300 fixture, and do **not** reuse a 200 × 120 one for the fixed case — either turns the
   no-reflow proof into a coincidence. An unasserted fixture constraint is a coincidence waiting to
   happen.
7. Tests on the AVD, from **local fixtures** (never the network): intrinsic sizing + reflow (band I
   moves — **and** `Wi > 0 && Hi > 0` asserted **positively**, so the negatives mean something);
   fixed sizing + **no** reflow; failure (stays 0×0, nothing reserved); `src` → null (the collapse);
   cancellation (a stale completion paints nothing); the image inside the scroll demo, with **the
   6.2 frame table asserted unchanged**. Also assert the **root's own frame** on `/image`: it FILLS
   the host's WIDTH (default `alignItems: stretch` on the synthetic host root) and **HUGS** in
   HEIGHT — `BnScrollDemoAndroidTest`'s two root assertions are the template.
8. Bump + assert the instrumented count in `android-instrumented.yml` (currently 96), house style.

## Gate 3 — iOS: Kingfisher

**Files:** `src/BlazorNative.Apple/project.yml` (Kingfisher via SPM), **`BnHost/Info.plist` (ATS —
see below)**, `BnWidgetMapper.swift`, possibly `BnYogaLayout.{h,mm}`; XCTests.

**Read `BnImageDemo.cs`'s header first — the whole of it**, and Gate 2's bullets above: the
synchronization gate, the fixture contract and the positive assertion are **the same rules**, and
"the same numbers" is the point of the phase.

- **ATS WILL BLOCK THE FETCH, AND THE SYMPTOM IS A LIE.** App Transport Security blocks arbitrary
  HTTP, and **Kingfisher fetches through `URLSession`, so ATS applies to it.** `Info.plist` today has
  **no** `NSAppTransportSecurity` key. **Add `NSAppTransportSecurity` → `NSAllowsLocalNetworking =
  true`** — the narrow loopback exemption, *not* `NSAllowsArbitraryLoads`. Without it every fetch
  fails, and because a blocked load is indistinguishable from a 404, cases [0] and [2] STILL PASS:
  the only red is case [1], and it reads as a reflow bug.
- **`127.0.0.1` IN THE SIMULATOR IS THE HOST MAC'S LOOPBACK.** The simulator shares the host's
  network stack — it does not have its own (that claim, which the demo header used to make, is true
  of the AVD only). The offline guarantee still holds; two things follow that do not:
  1. **port 8099 is host-global on the CI runner.** Bind exclusively; **fail loudly** if the port is
     taken (never "someone is already listening, good enough"). The fixture-contract assertions are
     the probe that the bytes are *ours*.
  2. ATS applies (above).
- **THE UNIT, AND THE ONE LINE THAT DESTROYS IT.** One file pixel is one **point**.
  `UIImage(data:)` has `scale == 1`, so `image.size` is already the file's pixel count in points and
  the rule is FREE — **so long as Gate 3 does not take it away:**
  - **DO NOT set `.scaleFactor(UIScreen.main.scale)`.** It is Kingfisher's own documented idiom for
    crisp images, it is the first thing you will reach for, and it makes a 160px fixture measure
    **160/3 ≈ 53.3pt** against Android's 160dp.
  - **DO NOT add a downsampling / resizing processor.**
  - **Nothing in either suite can catch this** — each shell is internally consistent, and no test
    compares the two tables. What *does* catch it: `BnImageDemo.cs` declares the fixture's pixel size
    (`IntrinsicNaturalWidthPx` = 160, `…HeightPx` = 90, fixed = 64 × 48), Gate 3's XCTests assert the
    **same constants**, and Gate 3 **adds the iOS half of the drift pin** next to the Android one
    (`BnImageDemoTests.TheAndroidFixtureServer_ServesExactlyBnImageDemosNaturalPixelSizes`) so its
    fixture server's transcription cannot drift either. Assert against the **decoded** `UIImage` as
    well — that is what pins "no downsampling".
- **THE SYNCHRONOUS MEMORY-CACHE HIT** (non-negotiable 4b). Kingfisher's `setImage` calls its
  `completionHandler` **synchronously** on a memory-cache hit, and the shell calls it from
  `UpdateProp("src")` — i.e. **inside the patch batch**. So: the completion's re-solve must **no-op
  while a batch is in flight**, and the `DownloadTask` must be **recorded only if the request is
  still live**. **Add the test**: mount `/image` TWICE **without clearing Kingfisher's caches
  between**, assert the AFTER frames and that nothing is left in flight. Every other test clears the
  cache, so this path is otherwise unexercised.
- **THE STALE-CALLBACK GUARD IS GENERATION *AND* IDENTITY**, in **every** terminal callback — the
  one that paints AND the ones that only clean up. Node ids restart at 1 after a reset, so the old
  node 2's cancelled callback (generation 1) meets a brand-new node 2 also on generation 1 and the
  generations MATCH: a generation-only check evicts the LIVE request's `DownloadTask`, which then
  cannot be cancelled, and **its completion runs against a freed `YGNodeRef`** — non-negotiable #4,
  arrived at through the guard that was supposed to prevent it. Android's shape:
  `InFlight(generation, view, disposable)` in one map, plus a `clearIfMine(nodeId, generation, view)`
  that drops the entry only if it is the callback's own. Pin the reset collision in a **unit test**
  (`ImageRequestGuardTest`, Kotlin's JVM lane) — no single-mount device test can stage it.
- **THE CONTENT MODE**: `contentMode = .scaleAspectFit`, explicitly. `UIImageView`'s default is
  `.scaleToFill` (STRETCH) and Android's is `FIT_CENTER` — frame-neutral, so no frame table catches
  it; what breaks is "renders identically". One line.
- **The 6.0 rule stands: Yoga's headers must never be visible to Swift.** Any new Yoga call goes in
  the `.mm` behind the plain-C surface. (Kingfisher is pure Swift and does not touch Yoga — but the
  *measure* and *dirty* calls do.)
- Kingfisher to **the same contract and the same numbers** as Gate 2. `image` is already a real
  `UIImageView` (6.1's Gate 3 review made it one, so an empty image measures `.zero` by construction
  — that is the pre-load state, already correct).
- **THE SYNCHRONIZATION GATE (non-negotiable #6)**: await Kingfisher's `completionHandler`
  (`.success`/`.failure`) for **all three** nodes — an `XCTestExpectation` with
  `expectedFulfillmentCount = 3` — before reading the AFTER frames. **Not** band I's movement.
- **THE FIXTURE'S CONTRACT (non-negotiable #6)**, asserted on the decoded fixture before any frame:
  `0 < Wi ≤ 300`, `Hi > 0`, `(Wfixed, Hfixed) ≠ (200, 120)`, no size-changing downsampling. Same
  fixture, same numbers, both platforms. Plus the **positive** assertion (`Wi > 0 && Hi > 0`).
- Handle **`src` → `null`** (`UpdateProp` with a null value — the renderer emits it): cancel,
  `.image = nil`, `markDirty`, re-solve → 0×0 and the siblings move back **up**. Mapper-level test.
- **Cancellation must also be proven not to touch a purged node** — on iOS that is a freed
  `YGNodeRef`, i.e. memory safety (non-negotiable #4). Extend the lifecycle tests.
- Adding an SPM dependency to an XcodeGen project is new ground for this repo — expect a CI
  iteration or two, and check it does not disturb the Yoga static-lib link (the 5.2 recipe).
- Bump + assert the XCTest count in `ios.yml` (currently 50).

### The ledger entries this phase's own decisions created (Gate 2 review — carry them into the conclusion)

1. **A 2×/3× ASSET RENDERS AT 2×/3× ITS INTENDED PHYSICAL SIZE.** The unit rule ("one file pixel is
   one dp/pt") means a 320 × 180 file drawn to be shown at 160 × 90 measures **320 × 180**. This is
   the rule working as specified and it is **consistent on both platforms** — iOS does exactly this
   by default for network images — but it is a real, user-facing consequence and it must be written
   down rather than discovered.
   **And it is NOT the React Native model, whatever decision 2 says.** RN auto-sizes only *bundled*
   assets, where the packager's `@2x`/`@3x` filename suffix supplies the scale; for **network**
   images RN requires an explicit size and **does not answer this question at all**. So this phase
   inherits RN's model for the **reflow** and *not* for the **unit** — the unit is un-inherited
   ground 6.3 broke by itself. **A scale-aware `Src`** (an `@2x` convention, a `Scale` parameter, or
   a `ContentMode` that resolves it) is the M7 question, and it is open.
2. **`ContentMode` is deferred; the DEFAULT is not.** Both shells are pinned to **aspect-fit**
   (Android `FIT_CENTER`, iOS `.scaleAspectFit`) — see the parity contract. The M7 `ContentMode`
   design inherits that as its default and owes the `cover`/`fill` cases.
3. **THE DEMO APP SHOWS THREE FAILED IMAGES ON `/image` — IN EVERY BUILD TYPE.** The fixture server
   lives in each platform's **test target**, not in the app, so nothing is listening on 8099 when the
   demo app runs. (The release cleartext block is a *second, independent* reason, and the two were
   previously conflated into "release only".) Correct and expected for a demo whose fixtures are
   loopback HTTP; the fix, if a demo ever must show them, is to **bundle the fixture as an asset** —
   never to weaken the release network config, and never to ship a `ServerSocket` in the app.
4. **GIF / SVG / animated formats.** A drawable with no bitmap now reports **no natural size (0 × 0)
   and logs** rather than falling back to a density-scaled `intrinsicWidth` — i.e. a second,
   contradictory unit rule was removed rather than left waiting to activate. Whatever ships these
   formats must decide their unit deliberately.
5. **The image result log is a bounded ring** (last 64) rather than a growing list. It is a
   diagnostic, not a ledger.
