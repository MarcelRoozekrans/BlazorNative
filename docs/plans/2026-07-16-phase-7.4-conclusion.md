committed.** `BnModal` (show/hide via the `@bind-Visible` pair, `ChildContent`, `ScrimColor`,

## The overlay-in-root decision as proven (design decision 1 ‚Äî flagged for veto, not vetoed)

**A `modal` is ordinary nodes in the existing tree, hosted by a full-root overlay ‚Äî not a
native dialog window.** The rejected option is recorded with its reasons: a `Dialog`'s content
view lives in its own Window, a presented view controller's view outside
`HostViewController.view` ‚Äî Blazor-rendered, Yoga-laid-out content inside either means a
**second Yoga root, a second calculate pass, a second resize hook, a second frame-application
surface** (RN's `Modal` pays exactly this with a `ReactRootView` inside the dialog). Worse than
the cost is the **state-owner violation**: a native dialog OWNS its dismissal ‚Äî back,
outside-touch, and accessibility escape all `dismiss()` the window while .NET still holds
`Visible = true`, the precise desync the 7.3 picker precedent exists to forbid.

The model as built ‚Äî the 6.2 synthetic-node machinery, pointed at the root. A `modal` create
materializes **two shell-side pieces in both trees in the same breath**:

- **The anchor** ‚Äî a 0-sized view at the modal's WIRE slot (Android: a plain `View`,
  deliberately not a `ViewGroup`, so a redirection bug fails loudly; iOS: a named
  `BnModalAnchorView` + the zero-footprint frame pin, because UIKit has no non-container view),
  shell-fixed `position: absolute` / 0 √ó 0 via the one style grammar the wire already uses.
  This is **the third index-mapping rule**, after the text collapse and the scroll content
  node: *the anchor occupies the modal's slot in its wire parent in the view tree AND the Yoga
  tree, so sibling insert indices never skew; the modal's wire child at index i is the
  OVERLAY's child at index i.*
- **The overlay** ‚Äî a full-root container attached **LAST at the host root** (stacking =
  creation order, never re-ordered; re-show = re-create = on top), shell-fixed
  absolute/0/0/100%/100% + justify/align CENTER ‚Äî the design's ((W‚àíw)/2, (H‚àíh)/2) centering IS
  that pair; the wire carries no layout for it. Children redirect into it (`containerFor`, the
  `scrollContents` precedent); `scrimColor` is a **prop** painting it. A top-level wire APPEND
  slots in AHEAD of live overlays ‚Äî **one resolved index feeds both trees** (root children and
  hostRoot mirror 1:1), so the clamp decision stays identical on both sides.

`SetStyle` on a `modal` node is **diagnosed-and-ignored for every name** on both shells (one
site, before the layout/visual routing ‚Äî every style would land on the anchor or the overlay,
neither of which the author owns; `BnModal`'s surface cannot emit one, and the hand-rolled-wire
test pins the rule live). `modal` is deliberately **not measured** ‚Äî anchor and overlay are
containers, and a measure func on a container is the 6.1 law's named violation. Host
resize/rotation while open is free by construction: the overlay is 100%/100% in the ONE tree
the existing resize hooks already re-solve.

## Dismissal is a REQUEST ‚Äî the state-owner law extended to overlays

`BnModal.razor` is internally `@if (Visible)` ‚Äî hide unmounts, `Visible = false` is **zero wire
presence**, and .NET's unmount is **the only close path**. Every dismissal trigger ‚Äî the scrim
tap, the app's own dismiss button, Android back ‚Äî rides the wire as `click` on the modal
element (always attached); the handler invokes `VisibleChanged(false)` and changes nothing
itself. An unbound parent keeps the modal open (pinned); a raced second dismissal is absorbed
(stale handler, at-most-once delivery, no frames). **The shell never self-closes** ‚Äî no
self-closing state exists anywhere in either shell.

**Android's back consults the modal stack BEFORE navigation-back** (design decision 3, flagged
for veto, standing): on back-invoked with ‚â• 1 live overlay, the shell dispatches on the
TOPMOST modal's `click` wire ‚Äî the very wire a scrim tap rides ‚Äî and **consumes the back event
on presence, not on dispatch success**; navigation-back runs only when no overlay is live. The
dispatch is a request: .NET flips `Visible`, the re-render removes the overlay. iOS ships
nothing here (no hardware back; sheet/swipe gestures are out of scope) ‚Äî recorded as the second
"same wire, platform-appropriate trigger" divergence, the checkbox precedent's shape.

The scrim-tap rule is normative on both shells: **a tap dismiss-requests ONLY when the touch
lands on the scrim itself, never on a descendant.** iOS enforces it with a
`UITapGestureRecognizer` whose delegate filter asks `touch.view === the attached view` (the
`click` arm growing past `UIButton` ‚Äî decision 4, the survey's down payment on `Pressable`);
Android ‚Äî where a non-clickable child's tap falls THROUGH to the scrim ‚Äî by the content box
swallowing its own taps (a real no-op `click` dispatch the device counters account for). Same
observable rule, two named mechanisms, verified per shell.

## `BnActivityIndicator` ‚Äî the survey's cheap win (DoD #7)

One NodeType (`activityindicator = 12`), a measured leaf, **no parameters** ‚Äî the wire shape is
one `CreateNodePatch` and nothing else, pinned three ways (the wire-shape golden, the
reflective empty-surface pin, presence-is-`@if` unmount ‚Äî stop == RemoveNode, no start/stop
prop for two shells to keep equal). Android: `ProgressBar` with `isIndeterminate` set
explicitly; iOS: `UIActivityIndicatorView(.medium)` with `startAnimating()` set explicitly
(animating-while-mounted is the contract ‚Äî a stopped spinner hides itself). Intrinsic size is
platform-owned and asserted **by oracle** (a fresh widget measured with the same specs ‚Äî the
6.3 method), in both hosting contexts (page and overlay).

## The RN parity survey (DoD #7)

The survey table is committed in the [design doc](2026-07-16-phase-7.4-design.md#the-react-native-parity-survey-dod-7)
‚Äî RN's core set mapped to `src/BlazorNative.Components/`. The verdict in one line: **11 of 19
"have it"** (View/Text/Image/TextInput/ScrollView/FlatList/Switch/Slider/Picker/Button, plus
tap-on-any-view as `Pressable`'s down payment); **Modal and ActivityIndicator shipped this
phase**; **Image polish (Placeholder/OnError/ContentMode) ships in 7.5** (DoD #6); the rest
**ledgered with reasons, not dropped**:

- **`SafeAreaView`** ‚Äî the named edge-to-edge problem: insets are HOST data arriving
  asynchronously and changing at runtime; on Android the current shell is not edge-to-edge, so
  the framework consumes the insets and the component would provably pad 0. Ships with the
  edge-to-edge/`StatusBar` work (M9-adjacent) as a container whose padding the shell owns.
- **`SectionList`** ‚Äî composition over `BnList`; blocked on variable row heights + sticky
  headers; no customer.
- **`Pressable` pressed/hovered STATE styling** ‚Äî gesture-system work, M8+ per the milestone's
  out-of-scope list.
- **`StatusBar`** ‚Äî host chrome control, the same edge-to-edge posture question.
- **`RefreshControl`** ‚Äî pull gesture + a `refreshing` round-trip; gestures out of scope.
- **`KeyboardAvoidingView`** ‚Äî keyboard insets are the safe-area problem's hard case, plus
  animation.
- **`Alert`** ‚Äî an imperative host API, not a render-tree node; the shell-bridge (clipboard
  precedent) is its natural home; deferred for a customer.

## The phase's empirical findings

1. **A modal hide frame is FIVE removes, not one (Gate 1).** With nested `ChildContent`
   components, hide is the modal's own diff (one remove ‚Äî the unit-pinned shape) **plus the
   nested components' disposal removes** riding ahead of it ‚Äî the 7.2 removes-first shape
   composing, harmless to the shells' subtree purge but wrong to idealize away. Pinned at
   **both altitudes**: the unit test pins the modal's own diff (inline content ‚Üí exactly one
   remove, and the review made the five **distinct**); `BnModalDemoTests` and both device
   suites pin the composite five-remove frame, which the synthetic one-remove tests cannot
   prove.
2. **A zero-attribute element in `.razor` collapses to `AddMarkupContent` (Gate 1).**
   `<activityindicator />` authored as `.razor` is exactly the shape the Razor compiler's
   static-markup optimization collapses to markup ‚Äî which the native renderer **rejects by
   contract** ("non-whitespace markup content is not representable on a native widget tree").
   So `BnActivityIndicator` is hand-written C# ‚Äî the csproj's "leaf primitives stay
   hand-written" rule, now with its structural reason recorded.
3. **The two-subtree fixpoint purge frees each Yoga node exactly once, by
   ownership-disjointness (Gate 3).** `RemoveNode(modal)` must purge the anchor's subtree AND
   the overlay's ‚Äî and a modal can sit inside another modal's overlay, so the purge is a
   fixpoint. On iOS the law is memory safety, not hygiene: a dangling `YGNodeRef` is a crash.
   Exactly-once holds because **overlay nodes all hang directly off `hostRoot`, disjoint from
   every wire subtree** ‚Äî a nested modal's ANCHOR is freed by the outer overlay's subtree free
   and only evicted thereafter. The **nested-modal explicit test** (modal B inside modal A's
   overlay; `RemoveNode(A)` ‚Üí both overlays purged, counts to 0 in both trees, a follow-up
   layout pass proving no dangling handle) was added mid-flight from the Gate 2 review's flag.
4. **The tap-filter mutation expectation, restated in its ACHIEVABLE form (Gate 3).** The
   design's mutation row said "drop iOS's touch-view filter ‚Üí the content-tap-does-not-dismiss
   test red" ‚Äî **unsatisfiable by construction**: no hosted XCTest can synthesize a `UITouch`
   (the recorded 7.3 limitation), so every device tap test fires the recognizer's effect
   (`bnFire`), which bypasses touch delivery and the filter with it. The achievable
   discriminator is the **truth-table pin on the pure decision + the delegate-wiring identity
   pin + the live-recognizer pin** (the Gate 3 review's M1: the decision moved onto the
   instance as `bnShouldReceive(touchView:)`, and one test asks the LIVE mounted scrim
   recognizer ‚Äî content box ‚Üí false, scrim ‚Üí true ‚Äî so a hardcoded-`true` mutant that survived
   all 131 tests now reddens). The untested seam shrank to **the `touch.view` property read
   alone**.

## Recorded divergences (the checkbox-precedent section)

- **The content-box swallow's observable (Gate 3 review M3):** a tap on an interaction-enabled
  non-control descendant inside the content box dispatches the content-box swallow on Android
  (fall-through) but **nothing** on iOS (the filter declines a descendant's touch). The
  **normative rule holds identically** ‚Äî dismiss only on the scrim, never on a descendant ‚Äî
  only the swallow-counter observable differs. Same family as 7.3's checkbox divergence:
  same wire behavior where it matters, different platform mechanism where it doesn't.
- **Android back** consults the modal stack; iOS ships nothing there (decision 3's last line) ‚Äî
  the second "same wire, platform-appropriate trigger" divergence.

## The honest ledger (deferred with names, not dropped)

- **`SafeAreaView`** and the survey's other ledgered rows ‚Äî see the
  [design table](2026-07-16-phase-7.4-design.md#the-react-native-parity-survey-dod-7); each
  carries its reason and (where known) its trigger. `SafeAreaView`'s is named: the Android
  edge-to-edge posture change, M9-adjacent.
- **The Android indexed-insert-at-root synthetic test** ‚Äî the append-lands-before-overlays rule
  is pinned; the *indexed* insert at root is correct by construction (the same one resolved
  index feeds both trees) but unpinned by a dedicated synthetic test (Gate 2 review). Named so
  the asymmetry is a decision, not an oversight.
- **iOS: an out-of-range indexed insert at root lands after the overlay** (Gate 3 review M2) ‚Äî
  reachable by malformed wire only (the .NET writer never emits one), and the consequence is
  stacking cosmetics, not correctness. Ledgered as a clamp-tightening candidate.
- **The `touch.view` property read** ‚Äî the last untested line of the iOS tap filter (the
  `UITouch` no hosted XCTest can construct; finding 4). Everything around it is pinned.
- **No animation system** ‚Äî out of scope by milestone (the modal appears and disappears in one
  frame commit; the M8+ animations item is where show/hide transitions live).
