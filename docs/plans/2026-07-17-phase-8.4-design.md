## The one sentence the whole phase hangs on

**A docs site is a machine for making second copies of facts — and this repo's own front
door, thirty lines under the CI badge, states four test counts and all four are wrong.**

`README.md:227-230` says .NET **333**, JVM **83**, instrumented **111**, iOS **72**. The
truth, asserted by gates on every PR, is **570 / 106 / 184 / 154**. Not one of those four
numbers is within 50% of reality. `README.md:5` says "Milestones 1–6 are complete… Milestone
7 is next" — M7 shipped and was tagged `v7.0` yesterday. `README.md:67` states `Yoga 3.2.1`,
a **fifth** copy of a version the repo pins across four files and which this fifth copy is
not part of. `ci.yml:16` tells its reader the JVM lane is "(92 passed / 0 failed)" while the
gate 1,257 lines below asserts **106**.

None of this was negligence. Every one of those lines was true when written, by people who
had just proven they cared about the number. **This repo has spent eight milestones
demonstrating that "we'll keep it updated" is not a mechanism, and it is about to build the
largest unpinned prose surface it has ever owned.** That is the phase's actual problem. The
Docusaurus config is an afternoon.

## Decisions (⚠ marks the veto-flagged judgment calls)

### 1. Docusaurus vs DocFX — the honest answer to "any advantages over DocFX?"

**Verdict: Docusaurus, per your milestone-open decision.** The reckoning, because the
question deserves one and the answer changed shape when it was measured:

**What DocFX genuinely wins, and it is not nothing.** It is native .NET. It ingests your
assemblies and their XML docs and emits an API reference *for free* — no second generator, no
glue, no dependency-resolution surprises. It cross-links your own types automatically. For a
repo whose reference surface **is** .NET types, that is the most relevant single feature
either tool has, and Docusaurus does not have it at any price.

**What Docusaurus genuinely wins.** You already run it. `AdoNet.Async/website` + `docs.yml` →
Pages is live today at `marcelroozekrans.github.io/AdoNet.Async/`. Adopting it here costs no
new mental model, no new CI shape, no new failure modes: one toolchain, one `npm run build`,
one artifact upload, one `github-pages` environment, one thing to upgrade. Its prose,
navigation, MDX and theming beat DocFX's by a wide margin — and a docs site is mostly prose.
DocFX would be a second toolchain in your world: a second thing to learn, a second thing to
debug at 11pm, for one feature.

**The part that changed when I measured it — and the reason this is not a close call.**
DocFX's headline advantage is worth materially less *here* than it looks, and the evidence is
above rather than asserted:

- The XML docs are **already complete** (8 gaps in Components, all `BuildRenderTree`). Nobody
  is trading away an API reference they'd otherwise get; the raw material exists either way.
- The XML docs' *content* is **repo-voiced** — "since Phase 3.4 Gate 1", "the BnDemo goldens
  stay byte-identical", "awaits M6". **DocFX would publish that phase history exactly as
  faithfully as xmldoc2md does.** A generator is a printing press: it reproduces what the XML
  says, and what the XML says is about us.
- DocFX would hit the same `ComponentBase` resolution wall from the same direction, and would
  need the same counting pin to prove it hadn't.

**Both tools need the same editorial pass on the same XML. DocFX saves the plumbing, not the
work.** That reframes the trade from *"free API docs vs. no API docs"* — which DocFX wins —
to *"free plumbing vs. one bolted-on tool, against one toolchain you already know"*, which it
does not. The plumbing is the cheaper thing to give up.

**The consequence, stated plainly: choosing Docusaurus means decision 3 must answer the
API-reference gap itself.** It does, and the answer is not "hand-write it."

### 2. Structure: `website/` mirroring the reference; the site LINKS, OWNS, or GENERATES — never INCLUDES ⚠

`website/` mirrors AdoNet.Async file-for-file: `package.json` (Docusaurus **3.10.2**, React
19, the `overrides: { webpack }` pin carried over), `docusaurus.config.js`, `sidebars.js`,
`docs/`, `src/css/custom.css`, `src/pages/index`, `static/img/`, `.gitignore`
(`node_modules/ build/ .docusaurus/`).

```js
url: 'https://marcelroozekrans.github.io',
baseUrl: '/BlazorNative/',            // project site — the trailing slash matters
organizationName: 'MarcelRoozekrans',
projectName: 'BlazorNative',
trailingSlash: false,
onBrokenLinks: 'throw',               // adopted as a PIN, not as decoration
```

**Three deliberate divergences from the mirror, each with a reason:**

1. **`blog: false`** — explicit. The reference leaves the key unset; the DoD says no blog.
2. **Prism gains `kotlin`, `swift`, `razor`** — this repo's snippets are not C#-only.
3. **The docs lane's paths filter is wider than `website/**`** — see decision 5. This one is
   not cosmetic; getting it wrong ships a stale site.

**Sidebar** (`docs`, one sidebar, mirroring the reference's single-sidebar shape):

```
intro
Getting Started → installation · quick-start (the template path)
Architecture    → overview · the-wire · layout-and-yoga · parity
Components      → overview · reference/* (GENERATED — autogenerated category)
Shells          → android · ios
analyzers
contributing (→ links into the repo)
```

**The rule that governs what lives where, stated once:**

> **NORMATIVE RULE — one home per fact.** No fact may exist in both the repo and the site.
> Every fact the site states is exactly one of: **GENERATED** from the code at build time and
> never committed; **LINKED** to its one home; or **OWNED** by the site, in which case the
> repo's copy is *deleted*, not left behind. There is no fourth option. **"Kept in sync" is
> not a home.**

**Does the site LINK, INCLUDE, or MOVE the repo's docs?** Docusaurus *can* read outside its
tree (`docs.path: '../docs'`), so INCLUDE is technically available. **It is rejected.**
Including turns files whose audience is this repo into files that must also carry Docusaurus
frontmatter, must satisfy `onBrokenLinks: 'throw'` against a URL space they don't live in,
and must not trip MDX — while remaining readable on GitHub. It makes the site's build depend
on the shape of contributor docs, and it produces a second *rendering* of a doc that was
written to be read in the repo. It buys nothing that a link does not.

**The precedent is already in this repo, and it is 8.3's, and it is exactly on point.**
`docs/ios-shell-setup.md:14-33`:

> **It does not transcribe the recipe.** … So this document **points at the live files** …
> which means the material this guide sends you to is kept true by a gate rather than by
> someone remembering to update a document. **A transcription here would be a fourth copy of
> a recipe that already exists twice in YAML, and it would rot the day `ci.yml` moved.**
>
> **The executable truth is `.github/workflows/ci.yml`, job `ios-build`.** When this prose and
> that job disagree, the job is right.

That is the site's governing principle, generalized: **prose that a gate does not keep true
should point, not copy.**

**The fates, decided:**

| Material | Fate | Why |
|---|---|---|
| `README.md` (257 l.) | **STAYS** + its stale facts fixed (decision 8) | It is the GitHub/NuGet front door; it cannot move. The site's landing page links to it, and states **no counts and no versions** — see decision 4. |
| `docs/ios-shell-setup.md` (276 l.) | **MOVE** → `website/docs/shells/ios.md`; repo copy **deleted** | 8.3: *"consume it; do not re-author it."* Moving is the only fate that leaves one home. |
| `docs/analyzers.md` (192 l.) | **MOVE** → `website/docs/analyzers.md`; repo copy **deleted** | Consumer reference — BN0004/0010/0011/0013/0014/0020/0021 are things a *user's* build emits. |
| `docs/bridge-extension.md` (207 l.) | **LINK** | Self-declared audience: *"a future contributor."* |
| `docs/GITHUB-SETUP.md` (493 l.) | **LINK** + **extend** (decision 6) | Owner-only repo machinery. |
| `docs/BACKLOG.md`, `docs/SESSION-HISTORY.md` | **LINK**, never site | Both open with self-invalidating banners; SESSION-HISTORY describes a deleted architecture. |
| `docs/planning/{MILESTONE,ROADMAP}.md` | **LINK** | Contributor machinery. The site's status prose links here; it does not restate it. |
| `docs/plans/` — **138 files, 39,297 lines** | **LINK as a class** | Not ~45. It is a dated, write-once, phase-numbered decision log for contributors. Docusaurus must not own an append-only log, and 39k lines of "Gate 2 review I-1" is not consumer material. **One carve-out below.** |

**The parity contract is the one that needed thought.** It is not a document. The normative
text is a section *inside a dated phase design doc* —
`docs/plans/2026-07-14-phase-6.3-design.md:39-174` — which **~50 in-source citations** treat
as permanent and normative (`WidgetMapper.kt:82`, `BnImageLoader.swift:4`,
`build.gradle.kts:71`, `BnImage.cs`, three workflows, six demo pages). It is also
image-specific, while DoD #5's "parity contract" plainly means the broader claim.

The site's `architecture/parity` page therefore **states the claim** — *the same numbers on
both platforms; here is what that guarantees you and what it does not* — and **links** to (a)
the normative 6.3 section and (b) `ShellFrameTableDriftTests.cs`, the test that makes it true.
**It does not reproduce the 10-row table.** That is the iOS doc's refusal, applied.

⚠ **Extracting the normative text to a stable `docs/parity-contract.md` is NOT this phase.**
It would re-point ~50 citations, several of which live in shell sources that
`TemplateDriftTests` **byte-compares against the template** — a comment edit in
`WidgetMapper.kt` reds a byte pin unless the template moves with it. That is a template-and-
shell change wearing a docs change's clothes. **Ledgered** with the reason.

### 3. THE HEADLINE — the component reference: GENERATED, never committed, and pinned against the tool that already lied ⚠

**Verdict: (b) xmldoc2md — generated in CI from the XML docs, never committed to the repo —
with an editorial pass on *voice*, and three pins.**

**Why not (a) hand-written MDX.** It would be a second copy of 192 parameter descriptions
that already exist, are already correct, and already sit six inches from the code they
describe. Hand-writing them creates 21 files whose only relationship to the truth is that
someone remembered. **That is this repo's named failure class**, and the README four hundred
words up is what it looks like after eight milestones.

**Why not (c) generate-and-commit with a drift pin.** A committed copy plus a test that
regenerates and diffs *works*, but it is strictly worse than not having the copy: it burns a
test, a regeneration step and a review conversation to defend a file whose only purpose is to
be identical to something else. **If the copy need not exist, the pin for it need not
exist.** Generate at build time and the fact has one home — the `///` next to the code.

**So: `docs.yml` runs `dotnet publish` of `BlazorNative.Components`, runs xmldoc2md against
the publish output, emits MDX into `website/docs/components/reference/`, and that directory
is `.gitignore`d.** The sidebar picks it up as an `autogenerated` category. Nothing is
committed; nothing can drift; there is no second copy to review.

**The three pins — and the first one is not optional:**

**Pin 1 — the reference is complete (the anti-vacuity pin).** The generator produced *ten
files and zero components* while reporting `0 failed`. A count is the only thing that saw it.
The pin reflects the **published assembly** for public, non-abstract `ComponentBase`-derived
types, and asserts the generated file set equals that set, **red in both directions**,
**non-vacuity asserted** (a zero-element expectation fails). It **derives its subjects; it
declares its expectation** — 8.3 I-1's rule, quoted where it will be read:

> The manifest is an **expected value**, not a roster of subjects — *enumerate what you
> MEASURE, declare what you EXPECT.* … *a roster that cannot be shrunk is a roster that does
> not exist.*
>
> *A pin that cannot see its subject must never pass vacuously.*

Mechanism mirrors `TemplateDriftTests.TemplateContentTree_IsExactlyTheExpectedManifest`
(`tests/BlazorNative.Runtime.Tests/TemplateDriftTests.cs:325`) — subjects enumerated from
disk, expectation a named set, `Except` both ways, hosted in **`build-test`**, the one
required lane where every file is checkout-visible.
**Mutation proof at Gate 2: point the generator at `bin/` instead of `publish/` → 10 vs 26 →
RED.** That mutation is not hypothetical. It is what happened first.

**Pin 2 — every public component member is documented (the compiler is the pin).**
**Un-suppress CS1591 for `BlazorNative.Components` only.** The price is **8 `<inheritdoc />`
lines** — the eight `BuildRenderTree` overrides, the entire measured gap. In exchange, *an
undocumented `[Parameter]` stops compiling*, and the reference's raw material cannot silently
thin. Mechanism keeping 8.1's one-metadata-home rule intact: `Directory.Build.props` gains
`<BnEnforceDocCoverage>false</BnEnforceDocCoverage>` and conditions the CS1591 `NoWarn` on
it; `BlazorNative.Components.csproj` sets it `true`. Per-project `NoWarn` is established
precedent — `Http.csproj:14` and `Renderer.csproj:19` already carry their own.

**The other 174 stay suppressed, ledgered, with the number named.** Runtime 87 + Core 52 +
Analyzers 16 + Renderer 13 + Http 6 = **174**. They are the plumbing — exports, bridge
callbacks, `DevHostBridge` — not the component reference. Documenting them is a real
milestone-sized job and it is not DoD #5's. **8.1's ledger said "doc coverage is 8.4's
editorial work"; measurement says coverage of the reference surface is already done, and the
174 that remain are not the reference.** The premise moved; the decision follows the
measurement.

**Pin 3 — the reference is written for strangers (the voice pin).** A regex over the
**shipped** `BlazorNative.Components.xml` reds on `Phase \d`, `Gate \d`, `BnDemo`,
`golden`, `HelloComponent`. It is cheap, it is specific, and it catches *exactly the defect
that is in the file today*. Non-vacuity: the pin asserts it read a non-empty XML with a
known sentinel member first — an absence assertion over an unread file is green for the wrong
reason (8.1 I-3).

**The editorial pass — the work this phase actually owns.** Rewrite the type-level summaries
and remarks of `BlazorNative.Components`' public surface so they address a consumer, and
**add type-level summaries to the six `.razor` components** that have none (`BnCheckbox`,
`BnList`, `BnModal`, `BnPicker`, `BnSlider`, `BnSwitch`) via a `partial class` in a
`.razor.cs` — `@* … *@` never reaches the assembly. ⚠ **Scope fence: `BlazorNative.Components`
only.** The phase history moves *out* of the XML, not into a second document; the `///` stays
the one home. Pin 3 is what makes this stick after the phase ends.

### 4. The drift taxonomy — what is PINNED, what is LINKED, what is prose ⚠

| Fact | Class | Mechanism |
|---|---|---|
| Component signatures + parameter docs | **GENERATED** | xmldoc2md at CI time; not committed; no second copy exists |
| "the reference shows every component" | **PINNED** | Pin 1 — derived from the assembly, red both ways |
| "every public component member is documented" | **PINNED** | Pin 2 — CS1591 as an error in Components |
| "the reference speaks to consumers" | **PINNED** | Pin 3 — voice regex over the shipped XML |
| Internal site links | **PINNED** | `onBrokenLinks: 'throw'` — Docusaurus itself |
| The template's short name (`blazornative`) in getting-started | **PINNED** | equality against `.template.config/template.json`'s `shortName` — renaming the template otherwise breaks the site's *first command* silently |
| **Test counts (570/106/184/154)** | **ABSENT** | **The site states no counts.** It links to the badge and the workflows. `README.md:227-230` is the proof this is the only safe answer. |
| **Version literals (`1.0.0-preview.1`)** | **ABSENT** | Getting-started says `dotnet new install BlazorNative.Templates` with **no version**; the number comes from a NuGet badge that reads nuget.org |
| **Yoga `3.2.1`** | **ABSENT** | Pinned across four files already; the site would be a fifth copy — and `README.md:67` is a fifth copy *today*, unpinned, which is how we know |
| The parity contract's normative table | **LINKED** | The site states the claim, links 6.3 §parity + `ShellFrameTableDriftTests` |
| Shell setup guides | **MOVED** | One home: the repo copy is deleted, not orphaned |
| The architecture story | **PROSE + review trigger** | Narrative, not facts-with-a-source. Trigger: **any ABI or wire change** (`docs/bridge-extension.md`'s existing trigger surface) |
| The icon | **PROSE-class (unpinned, deliberately)** | See below |

**The normative rule, stated once:**

> **NORMATIVE RULE — the site states no number that a gate owns.** Counts, versions, and
> pinned literals are linked or omitted, never transcribed. If the site wants to tell you how
> many tests pass, it links to the badge. **A number on a page nobody re-runs is a number
> that is already wrong.**

**Why the icon is deliberately unpinned, and the principle behind it.** The identity is one
design in two encodings — SVG for the site, PNG for the packages (NuGet accepts JPEG/PNG
only; SVG is not supported). A mismatch between them is *cosmetic and self-evident*: you look
at it and see it. **Pins exist for facts that are invisibly wrong.** A stale count is
invisible — that is the entire reason `README.md` says 333. A mismatched icon is visible to
anyone who opens either page. No pin.

### 5. The deploy lane — `docs.yml`, mirrored, with two divergences that matter ⚠

Mirrors AdoNet.Async: `workflow_dispatch`; `permissions: contents:read, pages:write,
id-token:write`; `concurrency: group "pages", cancel-in-progress: false`; `build` (ubuntu,
`checkout@v7`, `setup-node@v7` node 24, `npm install`, `npm run build`,
`upload-pages-artifact@v5`) → `deploy` (`needs: build`, environment `github-pages`,
`deploy-pages@v5`). `actionlint` clean, as every workflow in this repo is.

**Divergence 1 — the paths filter is wider than `website/**`, and this is not optional.**
Decision 3 makes the site's component reference a function of `src/BlazorNative.Components`'
XML docs. A PR that improves `BnView`'s `<summary>` changes the site. Under the reference's
`paths: ['website/**']`, main would take that commit and **never redeploy** — the site would
silently serve the old reference, and the failure would be invisible for exactly as long as
nobody looked. So:

```yaml
paths:
  - 'website/**'
  - 'src/BlazorNative.Components/**'      # the reference is generated FROM here
  - '.github/workflows/docs.yml'
```

**This is the cost of generate-at-build-time, and it is the right cost** — the alternative
(committing the generated MDX) re-introduces the copy decision 3 exists to avoid. Named here
rather than discovered later.

**Divergence 2 — a PR build check: YES.** AdoNet.Async builds only on main-push; a broken
docs build there is found *after* it lands. This repo's posture is the opposite, and the
owner's CI-cost posture settles it: **ubuntu is cheap and free for public repos.** One node
run per docs PR to catch a broken build before main is an obvious trade. The `build` job runs
on `pull_request` with the same paths filter; **`deploy` is gated on
`if: github.event_name != 'pull_request'`** so a PR never touches Pages, never takes the
`pages` concurrency group, and never needs `id-token`.

**`docs.yml` does NOT join the required checks.** The three required gates keep their names
and contexts — `build-test`, `android-build`, `ios-build`, whose contexts *are* their job ids
verbatim (`GITHUB-SETUP.md:196`). A fourth required check is a milestone-level change to
branch protection and DoD #5 does not ask for one. The docs build is advisory, exactly as
`release.yml`'s `validate` was left in 8.2.

**The build job needs .NET** (`setup-dotnet`) for the reference generation — a divergence
from the reference's node-only build, forced by decision 3.

**What the owner must click — one setting, once:**

> **Settings → Pages → Build and deployment → Source: `GitHub Actions`.**

That is the whole list. The `github-pages` environment is created automatically on first
deploy; no secret, no token, no branch. Today `has_pages: false` and `/pages` 404s; the target
is AdoNet.Async's `build_type: "workflow"`. **Until that click, `deploy` fails** — loudly, on
main, with a legible error. `GITHUB-SETUP.md` gains `### GitHub Pages — enabling the docs

### 6. The icon: the trigger has fired — design it

8.2 pinned the trigger precisely: *"the docs-site identity, Phase 8.4 (DoD #5) — the icon is
a derivative of that identity, and choosing it here means choosing it twice."* **This is that
phase.** The identity is being chosen now; the icon falls out of it rather than being invented
beside it.

- `website/static/img/logo.svg` (navbar) + `favicon.ico` — the site's encoding.
- `src/icon.png` — **128×128 PNG** (NuGet: JPEG/PNG only, ≤1 MB; SVG unsupported), packed to
  the package **root** by `Directory.Build.props` — 8.1's ONE metadata home — via
  `<PackageIcon>icon.png</PackageIcon>` + `<None Include="$(MSBuildThisFileDirectory)icon.png"
  Pack="true" PackagePath="\" />`. Shared across all six (unlike the per-project README).

**Does it break the smoke's counts or the inventory tooth? No — verified, not assumed:**

- **6 nupkg + 5 snupkg** (`consumer-smoke.ps1:147,151`) count *packages in the feed*. An icon
  is a file inside a package. **Unaffected.**
- **The inventory tooth** (`:296-321`) is an exact-set `Compare-Object` **scoped to
  `lib/net10.0`** against `{$id.dll, $id.xml}`. The icon packs to the **root**, next to
  `README.md`. **Unaffected.**
- **The nuspec assertions** (`:245-270`) check id/version/license/readme/repository. They do
  not forbid an `<icon>`. **Unaffected.**

**The tooth comes free, exactly as the README's did.** 8.1's `PackageReadmeFile` + `Exists`
guard makes a missing readme fail pack loudly as NU5039; `PackageIcon` pointing at an
unpacked file fails pack as **NU5046**. The pack *is* the pin. Add the nuspec-`icon`-entry +
file-present assertion to the smoke's per-nupkg interrogation to mirror the readme's, since
that loop already unzips every package and the marginal cost is two lines.

**The "frozen forever" objection stays answered** (8.2): a package icon is per-version
metadata, so `1.0.0-preview.1` is iconless on nuget.org permanently — but **nuget.org renders
the LATEST version's icon**, so preview.1's absence costs nothing the moment any later
version carries one. And nothing publishes without the owner's Release (8.2), so the icon is
inert until then regardless.

⚠ **Scope fence on taste.** This design fixes *where the icon lives, what format, what it must
survive* — 128×128, legible at 32 px in a nuget.org result row, no gradient text, readable on
light and dark. It does not hand-author the art in a design doc. The concept — a mark derived
from the site's identity, not a wordmark — is Gate 1's to produce against that bar.

## The normative rules, stated once

1. **One home per fact.** Every fact the site states is GENERATED, LINKED, or OWNED. Never
   copied. "Kept in sync" is not a home.
2. **The site states no number that a gate owns.** Counts, versions and pinned literals are
   linked or omitted, never transcribed.
3. **Prose a gate does not keep true points; it does not copy.** (8.3's iOS doc, generalized.)
4. **A pin that cannot see its subject must never pass vacuously.** Every pin this phase adds
   asserts a positive control first.
5. **Enumerate what you MEASURE; declare what you EXPECT.** No rosters.
6. **Pins exist for facts that are invisibly wrong.** Visible mismatches (an icon) do not get
   pins; invisible ones (a count) do not get prose.
