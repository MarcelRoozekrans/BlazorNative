## Task 1 — Bug A: `MountAsync<T>` normalizes `default(ParameterView)` → `ParameterView.Empty`

**Files:**
- Modify: `src/BlazorNative.Renderer/NativeRenderer.cs` (the `MountAsync<TComponent>` method)
- Test: extend an existing or add a new test in `tests/BlazorNative.Renderer.Tests/`

**Background (from the spike):** `MountAsync<T>()` defaults `parameters` to `default(ParameterView)`. That struct has a null internal frames array; Blazor's enumerator NREs when iterating it. The NRE is silently swallowed by `NativeRenderer.HandleException`, so the mount appears to succeed but no frame fires. Phase 2.4 Task 4 fixed this for the sync `Mount<T>` but missed the async variant.

        // Regression guard: Phase 2.4 Task 4 found that default(ParameterView)
        // NREs in Blazor's ParameterView enumerator (silently swallowed by
        // HandleException). The sync Mount<T> was fixed; MountAsync<T> still
        // had the latent bug per Phase 2.7 spike. This test calls MountAsync

### Step 1.3: Implement the fix

Edit `src/BlazorNative.Renderer/NativeRenderer.cs`. Find the `MountAsync<TComponent>` method (currently around line 80 after the Phase 2.5 sync `Mount<T>` additions):

Replace with:

```csharp
public Task<int> MountAsync<TComponent>(
    ParameterView parameters = default,
    CancellationToken ct = default)
    where TComponent : IComponent
{
    // Normalize default(ParameterView) → ParameterView.Empty. Blazor's
    // ParameterView.GetEnumerator NREs on the default struct (null internal
    // frames array); HandleException silently swallows the NRE so the mount
    // appears to succeed but no frame is produced. Phase 2.7 Bug A fix
    // (carried over from the Phase 2.4 Task 4 sync-Mount<T> fix that missed
    // this async path). Spike doc: 2026-05-27-phase-2.7-host-element-spike.md.
    var safe = parameters.Equals(default(ParameterView)) ? ParameterView.Empty : parameters;
    return Dispatcher.InvokeAsync(() => AddComponentAsync(typeof(TComponent), safe));
}
```

**Note on the `.Equals(default)` check:** `ParameterView` is a struct so reference equality doesn't apply. `ParameterView.Equals` should do a structural comparison — verify by running the test. If `Equals(default)` doesn't work as expected, fall back to checking an internal field via reflection or use a custom flag (`parameters._frames is null` if accessible — likely not since it's internal).

If `Equals(default)` proves fragile, an alternative pattern:
```csharp
// ParameterView is a struct; the default-constructed instance has no frames.
// We can't access _frames (internal), but we can detect the default via
// a no-allocation probe: try to enumerate; if it throws or returns nothing
// from a default instance, normalize. Or just always normalize when the
// caller used the default arg (which we can't directly detect either).
// Simpler: just always pass Empty if no explicit parameters were supplied
// by making the parameterless overload explicit:
```

So a cleaner approach is to **add an explicit parameterless overload** that always passes `ParameterView.Empty`, matching the Phase 2.4 Task 4 pattern for sync `Mount<T>`:

```csharp
public Task<int> MountAsync<TComponent>(CancellationToken ct = default)
    where TComponent : IComponent
    => MountAsync<TComponent>(ParameterView.Empty, ct);

public Task<int> MountAsync<TComponent>(
    ParameterView parameters,
    CancellationToken ct = default)
    where TComponent : IComponent
    => Dispatcher.InvokeAsync(() => AddComponentAsync(typeof(TComponent), parameters));
```

(No default value on the `parameters` arg in the second overload — callers must pass `ParameterView.Empty` explicitly if they want empty parameters. The first overload handles the no-args call site.)

**Pick the cleaner approach (two overloads).** It mirrors the existing `Mount<T>()` / `Mount<T>(ParameterView)` split from Phase 2.4 Task 4 (NativeRenderer.cs line ~98), so the API is symmetric: both sync and async variants have a parameterless overload that explicitly threads `ParameterView.Empty`.

## Task 2 — Bug B: `ProcessFrame` handles `Component` frames (no mis-attribution)

**Files:**
- Modify: `src/BlazorNative.Renderer/BlazorInterop.cs` (expose `ComponentSubtreeLength` on `BnRenderTreeFrame`)
- Modify: `src/BlazorNative.Renderer/NativeRenderer.cs` (`ProcessFrame` Component-frame handling)
- Modify: `tests/BlazorNative.Renderer.Tests/Phase27HostElementSpike.cs` (`Probe3_NestedComponents_*` test — strengthen assertions)

**Background (from the spike):** `NativeRenderer.ProcessFrame`'s `switch (frame.FrameType)` handles only `Element` and `Text`. When a `Component` frame appears as a child inside an Element's subtree, the recursive call hits no case (no patch emitted), AND the parent's subtree iteration does NOT advance past the Component's `ComponentSubtreeLength` — so the Component's child `Attribute` frames (carrying `Label="A"` etc.) are seen by the parent's loop and routed through `ProcessAttribute(parent_nodeId, ...)` as if they were attributes of the parent Element. Result: parent gets erroneous `UpdatePropPatch`es.

### Step 2.1: Expose `ComponentSubtreeLength` on `BnRenderTreeFrame`

Edit `src/BlazorNative.Renderer/BlazorInterop.cs`. Find the `BnRenderTreeFrame` ref struct (around line 198-210):

    public int                 ElementSubtreeLength => _frame.ElementSubtreeLength;

    public object?             AttributeValue       => _frame.AttributeValue;
    public ulong               AttributeEventHandlerId => _frame.AttributeEventHandlerId;
    public string?             TextContent          => _frame.TextContent;
}

    public string?             ElementName              => _frame.ElementName;
    public int                 ElementSubtreeLength     => _frame.ElementSubtreeLength;
    public int                 ComponentSubtreeLength   => _frame.ComponentSubtreeLength;

(`RenderTreeFrame.ComponentSubtreeLength` is a public property in Blazor — no `UnsafeAccessor` needed.)

Add Component-frame handling: when we see a `Component` frame, skip its entire subtree (avoid attribute mis-attribution). The Component's own diff in the same `RenderBatch.UpdatedComponents` array handles its rendering separately.

Replace the inner loop's body with:

```csharp
                var subtreeLen = frame.ElementSubtreeLength;
                for (var i = 1; i < subtreeLen; i++)
                {
                    var child = new BnRenderTreeFrame(ref frames[frameIndex + i]);
                    if (child.FrameType == RenderTreeFrameType.Attribute)
                    {
                        ProcessAttribute(nodeId, ref child, ref patches);
                    }
                    else if (child.FrameType == RenderTreeFrameType.Component)
                    {
                        // Component frame inside this element's subtree. We do NOT
                        // emit a patch here — the component's own ComponentRenderTreeDiff
                        // is in the same RenderBatch.UpdatedComponents array and handles
                        // its rendering separately. We MUST advance i past the component's
                        // subtree so its child Attribute frames (carrying its parameter
                        // values like Label="A") aren't mis-attributed to THIS element
                        // by the next loop iteration. Phase 2.7 Bug B fix.
                        i += child.ComponentSubtreeLength - 1;
                    }
                    else
                    {
                        // Non-attribute, non-component child frame inside the subtree —
                        // recurse to emit its create/text patches as a child of this element.
                        // Use the child's index as siblingIndex (positions within the
                        // parent's subtree are unique enough for the M1 mapping). Pass
                        // this element's nodeId as the child's parentNodeId so the host-
                        // side widget mapper can attach the child inside this element's
                        // view (Phase 2.5 design).
                        ProcessFrame(componentId, ref frames, frameIndex + i, frameIndex + i, parentNodeId: nodeId, ref patches);
                        // Skip the child's own subtree so we don't double-walk it.
                        if (child.FrameType == RenderTreeFrameType.Element)
                            i += child.ElementSubtreeLength - 1;
                    }
                }
```

The new `else if (child.FrameType == RenderTreeFrameType.Component)` branch is the load-bearing change. The `else` branch's comment also gets updated to reflect that Components are now explicitly handled.
