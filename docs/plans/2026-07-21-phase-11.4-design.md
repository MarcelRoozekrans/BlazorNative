# Phase 11.4 ‚Äî logging discipline (M11 DoD #6)

**Milestone:** 11 ‚Äî Production Readiness. **Phase:** 11.4.
**Closes:** DoD #6 ‚Äî [#155](https://github.com/MarcelRoozekrans/BlazorNative/issues/155) (one
level-gated logging seam, quiet-by-default in Release, unified across both shells) and
[#164](https://github.com/MarcelRoozekrans/BlazorNative/issues/164) (a render error is invisible
in production: stderr goes nowhere on Android and `mount` still returns rc 0).

Two issues, one root cause and one *extra* cause. The shared root is that **the runtime has no
transport to a platform log**: it writes to `Console.Error`, which Android sends to `/dev/null`
and iOS sends nowhere useful in an unattached Release build. The extra cause, which belongs to
#164 alone, is that **`mount` returns rc 0 on a faulted render** ‚Äî so even a perfect log leaves
the host with nothing to act on. ¬ß5 designs the transport; ¬ß6 answers #164 on its own terms and
deliberately does *not* treat it as a logging problem with a logging fix.

This document decides. It does not implement.

---

## 0. The hard constraint, stated up front

**The C-ABI is frozen: 80 bytes / 10 exports.** Everything below is designed inside that.

| | Value | Where it lives | Where it is pinned |
|---|---|---|---|
| Callbacks struct | **80 bytes**, 10 √ó `IntPtr` slots, offsets 0‚Ä¶72 | `src/BlazorNative.Runtime/BridgeProtocolNative.cs:80`‚Äì`:97` | `BridgeProtocolNativeTests.cs:52`‚Äì`:63`, re-asserted per capability (`CameraAbiUnchangedTests.cs:49`, `NotificationsAbiUnchangedTests`, `SecureBiometricsAbiUnchangedTests`) |
| Exports | **10** `blazornative_*` | `Exports.cs:106`, `:176`, `:197`, `:221`, `:243`, `:268`, `:385`, `:413`, `:442`, `:475` | `ci.yml:1514` + `:1547` (`llvm-readelf --dyn-syms`, both bionic RIDs), `ios.yml:168` (`nm -gU` on the `.a`) |

M9 shipped **five capabilities** ‚Äî geolocation, notifications, biometrics, secure storage,
camera ‚Äî with *zero further ABI change* as an explicit, repeatedly-asserted property. Phase 11.4
must not be the phase that spends it on a diagnostic channel.

So the design question is not "what would a nice logging API look like." It is:

> **How does a managed-side log line reach logcat / the iOS unified log without adding an
> export, and without growing the 80-byte struct?**

¬ß5 enumerates the four real answers, grounds each in code, and picks. ¬ß5.5 is the recommendation.

**One thing the constraint does *not* cover, and this is the pivot of the whole design:**
`BlazorNativeInitOptions` (`Exports.cs:60`) is **not** part of the frozen surface. The source
says so in as many words at `Exports.cs:71`‚Äì`:72`: *"this is the init-INPUT struct, **NOT** the
frozen 80-byte callbacks bridge."* It already grew once, in M10. ¬ß5.4 uses that.

### 1.3 Android ‚Äî silent, by construction

The opposite failure. The .NET runtime's `Console.Error` is process stderr, and **Android
discards it**. The repo does not merely know this ‚Äî it *depends* on it and says so repeatedly:

- `ShellBridge.kt:280` ‚Äî *"Android must pass android.util.Log, stderr is `/dev/null` there."*
- `BlazorNativeRuntime.kt:33` ‚Äî *"goes to `/dev/null` on Android, not logcat."*
- `BlazorNativeRuntime.kt:373` ‚Äî a `mount` failure message that literally tells the developer the
  detail *"went to native stderr, which Android discards; reproduce on the desktop JVM to see it."*
- `NativeBindings.kt:50`, `:65`, `:82`, `:117`, `:136`, `:155`, `:165`, `:170` ‚Äî eight KDoc rc
  contracts that point the reader at a stderr that will never exist on the target platform.
- `DispatchEventTest.kt:194` **asserts on that string**, so "reproduce on desktop" is a *pinned*
  behaviour today.

The only channel that reaches logcat is the shell's own `android.util.Log`, injected as a
pluggable `onError` sink because the shared source set cannot import Android classes:

```kotlin
private val onError: (String, Throwable) -> Unit = { msg, t -> System.err.println("$msg: $t") },
```

`BlazorNativeRuntime.kt:32`‚Äì`:34` (the JVM default), wired by the Activity at
`MainActivity.kt:157` as `{ msg, t -> Log.e(tag, msg, t) }`. `BridgeRegistrar` deliberately
refuses a default (`ShellBridge.kt:279`‚Äì`:281`).

Kotlin `Log.*` census ‚Äî **72 sites**, concentrated in `WidgetMapper.kt` (48) and `MainActivity.kt`

**Nothing in the repo redirects stderr.** A search of `src/BlazorNative.Jni/**` for `dup2`,
`Os.pipe`, `ParcelFileDescriptor.createPipe` or any redirection returns only unrelated prose
about HTTP redirects and modal child re-parenting. **Option 1 in ¬ß5 is genuinely unbuilt.**

## 2. Two asymmetries that make "unify the shells" harder than it sounds

Before designing, two facts that a naive "make both platforms do the same thing" would get wrong.

**Asymmetry 1 ‚Äî the shells log *their own* stuff fine; it is the *runtime's* output that is
stranded.** Both `Log.*` (72 sites) and `NSLog` (78 sites) are shell-side Kotlin/Swift, and both
already reach their platform log. iOS's problem is that it reaches it *too much*; Android's is
that it is *correctly levelled but incomplete*. The genuinely missing piece ‚Äî on **both**
platforms ‚Äî is the 31 managed `Console.Error` sites. Framing 11.4 as "align the two shells"
would fix the smaller half.

**Asymmetry 2 ‚Äî Android already levels the mapper diagnostics at Warn; iOS does not level at
all.** #155's goal text asks to *"gate the normal-path boot/`ignored` chatter behind
Debug/Verbose."* Applied literally with a Warn-in-Release default, the boot lines vanish
(correct) and the `ignored` lines vanish too ‚Äî **but on Android they are `Log.w` today**, so
that is a *demotion of an existing warning*, not a quieting of new noise. ¬ß4.3 resolves it.

## 3. The seam

### 3.1 Levels

Five, mapping cleanly onto both platforms and onto `Microsoft.Extensions.Logging.LogLevel` so the
existing `ILogger` plumbing survives:

| `BnLogLevel` | ordinal | `LogLevel` | Android `Log` | iOS `OSLogType` | In Release by default? |
|---|---:|---|---|---|---|
| `Error` | 1 | `Error` / `Critical` | `Log.e` | `.error` / `.fault` | **yes** |
| `Warn` | 2 | `Warning` | `Log.w` | `.default` | **yes** |
| `Info` | 3 | `Information` | `Log.i` | `.info` | no |
| `Debug` | 4 | `Debug` | `Log.d` | `.debug` | no |
| `Verbose` | 5 | `Trace` | `Log.v` | `.debug` | no |

**Ordinal 0 is reserved and means "unset ‚Üí apply the default."** That is not decoration: it is
what makes the init-input carrier back-compatible (¬ß5.4) and it mirrors the precedent set by
`PlatformKind` ordinal 0 ‚Üí `DevHost` (`Exports.cs:171`‚Äì`:176`, *"a safe non-lying default"*).

**Default = `Warn`.** Not `#if DEBUG`. A build-configuration `#if` cannot be changed by a
consumer shipping a Release build who needs one verbose session, and it makes the Debug and
Release code paths differ ‚Äî which is exactly how a logging bug survives to production. A runtime
level with a quiet default gives the same production quietness *and* a way in.

### 3.2 The managed shape

One internal static `BnLog` in `BlazorNative.Core` (it must be reachable from Core, Renderer and
Runtime), holding:

- a `volatile int` current level, read on every call ‚Äî the gate is an integer compare, so a
  suppressed `Debug` costs one branch;
- a **sink delegate** `Action<BnLogLevel, string, string>` (level, category, message) defaulting
  to the level-tagged `Console.Error` writer of ¬ß5.5;
- `Error/Warn/Info/Debug/Verbose(category, message)` plus an `Exception` overload that applies
  ¬ß7's redaction.

`RendererServices`' `ConsoleLogger` (`RendererServices.cs:34`‚Äì`:53`) stops hard-coding
`>= LogLevel.Warning` and delegates both `IsEnabled` and `Log` to `BnLog`. That single change
makes the Blazor-side `ILogger` and the framework's own diagnostics share one throttle, which is
the "one seam" DoD #6 asks for.

**Message-building must be inside the gate.** `BnLog.Debug("mapper", $"node {id} ‚Ä¶")` builds the
interpolated string *before* the call and pays for it even when suppressed. Every call site
either takes a `Func<string>` or is guarded by `if (BnLog.IsEnabled(...))`. On the frame path
(`NativeRenderer`, `WidgetMapper`) this is not pedantry ‚Äî those run per patch.

### 3.3 Where the level is configured

Three inputs, precedence highest-wins-last:

1. **The init input** (`BlazorNativeInitOptions`, ¬ß5.4) ‚Äî the shell's declaration at boot. This
   is the one that matters, because it is set *before* the first managed line is written and it
   is the only one a shell author can reach.
2. **A public managed setter** ‚Äî a `BlazorNativeApp` property settable from the
   `ConfigureServices` seam that Phase 11.1 shipped, for a consumer who wants verbosity for one
   session without touching the shell.
3. **An MSBuild property ‚Üí the shell's own default** ‚Äî e.g. `-p:BnLogLevel=Debug` flowing into the
   Kotlin/Swift constant that the shell passes at (1). Convenience only; it configures nothing the
   shell could not pass by hand.

(2) is **new public API** and therefore lands in Phase 11.3's `PublicAPI.Unshipped.txt` ‚Äî see
¬ß8.3 for the ordering.

## 4. The two shells

### 4.1 Android ‚Äî `Log`, plus the transport of ¬ß5

The `Log.*` sites are already correctly levelled and stay as they are; they gain a level gate by
routing through a thin `BnLog` Kotlin object with the same five levels and the same `volatile`
integer compare. `MainActivity.kt:157`'s `onError` sink stays exactly where it is ‚Äî it is the
right shape and it is what makes `BlazorNativeRuntime` platform-agnostic.

The genuinely new Android work is getting the **runtime's** output to logcat: ¬ß5.

### 4.2 iOS ‚Äî `os_log`, and the migration of the 78 sites

**Target state:** a `BnLog.swift` exposing `BnLog.error/warn/info/debug/verbose(_ category:
_ message:)` over `os.Logger`, with one `Logger` per subsystem `io.blazornative` and a category
per component (`runtime`, `mapper`, `bridge`, `camera`, ‚Ä¶).

Three concrete reasons `os_log` over `NSLog`, all of which matter for #155's stated concerns:

1. **It is OS-filterable.** Levels below `.default` are not even persisted unless the user enables
   them for the subsystem. Quietness becomes the platform's job, not a `#if`'s.
2. **It redacts by default.** In `Logger`'s string interpolation, non-static values are `private`
   unless explicitly marked `%{public}`. `NSLog` is *always* public. This is a free second layer
   under ¬ß7's information-disclosure requirement ‚Äî and it is the single strongest argument
   against a cheaper "wrap `NSLog` in a level check" shim.
3. **It is cheap when disabled.** `os_log` defers formatting; `NSLog` formats then writes.

**The migration ‚Äî a shim, then a sweep, then a pin.** Concretely, in this order:

1. **Add `BnLog.swift`.** New file, no call sites yet.
2. **Mechanically rewrite the 78 `BnHost/**` sites.** `NSLog("[Tag] msg")` ‚Üí
   `BnLog.<level>("<tag>", "msg")`. The tag is already in every message as a bracketed prefix
   (`[BnRuntime]`, `[BnWidgetMapper]`, `[AppleShellBridge]`, `[HostViewController]`), so the
   category is derivable from the existing text ‚Äî the sweep is genuinely mechanical, not a
   judgement call per site. Level assignment is the only judgement, and ¬ß4.3 fixes it by class,
   not by site.
   - `BnYogaLayout.mm`'s 4 sites are Objective-C++ and cannot call a Swift API. They get a
     tiny C-callable `BnLogC(int level, const char *category, const char *message)` declared in
     `BlazorNativeRuntimeC.h`'s neighbourhood and implemented `@_cdecl` in `BnLog.swift`. Four
     sites; not worth a second design.
   - `BnWidgetMapper.swift:1554`'s existing `diagnose(nodeId:kind:message:)` funnel keeps its
     dedupe logic and swaps its `NSLog` (`:1557`) for `BnLog.warn`. The other 53 sites in that
     file are rewritten to `BnLog` directly (folding them all into `diagnose` would change
     dedupe semantics ‚Äî out of scope).
3. **Leave `BnHostTests/**`'s 24 sites alone.** XCTest output is not shipped.
4. **Land the pin in the same PR** ‚Äî ¬ß8.1's "no bare `NSLog` under `BnHost/`" drift test. Without
   it, site 79 arrives next month and the sweep silently rots.

**Not a shim-only approach.** Routing all 78 through a `BnLog` that internally calls `NSLog`
would deliver the level gate for a fraction of the effort ‚Äî and would forfeit reason (2) above,
which is the half of #155 that is about information disclosure rather than noise. The sweep is
the same 78 edits either way; only the body of `BnLog` differs. Do it once, properly.

### 4.3 The finding: the `ignored` diagnostics should stay at **Warn**, on both platforms

#155 asks to gate *"the normal-path boot/`ignored` chatter behind Debug/Verbose."* **Split that
sentence.** The two halves are different things:

- **The normal-path boot lines** (`BnRuntime.swift:223`, `:228`, `:249`, `:254`;
  `MainActivity.kt:172`, `:204`, `:310`) are pure success narration. **‚Üí `Info`.** Suppressed in
  Release. This is the uncontroversial half and it is most of #155's user-visible complaint.
- **The mapper's `‚Ä¶ ignored` / `‚Ä¶ skipped` diagnostics** are not chatter. Each one records that a
  **wire the app author asked for was silently dropped** ‚Äî an event that will not fire, a style
  that will not apply. `BnWidgetMapper.swift:660` and `:1147` say so in their own doc-comments:
  the diagnostic exists precisely because *"`NSLog` is not an assertion surface and every one of
  these failures is [otherwise invisible]."* Android already agrees in code ‚Äî all four of
  `WidgetMapper.kt:778`, `:865`, `:897`, `:926` are `Log.w`. **‚Üí `Warn`, on both.** They stay in
  Release.

Demoting them to Debug would take the one class of message that reliably indicates a real author
bug and hide it in exactly the build where it costs most ‚Äî the same mistake #164 documents, in a
different file. **This is a deliberate divergence from #155's literal goal text**, made because
grounding contradicted it; it is recorded here so the issue can be updated rather than
silently under-delivered.

Net effect on volume, which is what #155 actually cares about: the iOS Release log loses the boot
narration and keeps the (rare, actionable) mapper warnings. The `ignored` lines only fire on a
broken page; a correct app emits **nothing at all** in Release except real faults.

## 5. The transport decision ‚Äî four options, grounded

Restating the constraint: **no new export, no growth of the 80-byte struct.**

### 5.1 Option 1 ‚Äî shell-side stderr redirection (Android)

Create a pipe, `dup2` it over fd 2, read it on a daemon thread, forward each line to
`android.util.Log`.

**Grounding:** nothing in `src/BlazorNative.Jni/**` does this today (¬ß1.3). And it is **cheaper
than it sounds**: it needs no NDK code and no JNI. `android.system.Os` has exposed `pipe()` and
`dup2(FileDescriptor, int)` since **API 21**, and the project floor is **`minSdk = 24`**
(`src/BlazorNative.Jni/build.gradle.kts:125`, mirrored at
`templates/.../android/build.gradle.kts:94`, where the comment calls 24 *"a real floor, not a
preference"*). So the whole pump is ~40 lines of pure Kotlin in the `androidMain` source set.

| | |
|---|---|
| **ABI cost** | **Zero.** The runtime is not modified at all; the shell reads a file descriptor. |
| **Coverage** | **Total.** Every `Console.Error` site ‚Äî including the 31 of ¬ß1.1 ‚Äî plus anything the **BCL or the NativeAOT runtime itself** writes (unhandled-exception dumps, `TypeLoadException` detail, JNA's own swallow-to-stderr handler at `BlazorNativeRuntime.kt:21`). No other option covers that. |
| **Availability** | From the moment the shell installs it, which can be **before `dlopen`** ‚Äî so it covers `blazornative_init`'s failure path (`Exports.cs:157`), the one place the repo deliberately emits a full `ex.ToString()`. |
| **Caveat ‚Äî no level** | The pump sees formatted bytes, not a level. Solved in ¬ß5.5 by a tagged prefix. |
| **Caveat ‚Äî buffering** | Line-oriented. `Console.Error` in .NET is auto-flushing per `WriteLine`, so lines arrive whole in practice; a partial write at a read boundary must be buffered until the newline, and a very long line needs a cap (logcat truncates around 4 KB per entry regardless). |
| **Caveat ‚Äî indiscriminate** | It captures fd 2 process-wide, including any third-party native library. That is mostly a feature; it is a problem if the consumer's app also redirects stderr (Crashlytics/Sentry NDK handlers do) ‚Äî last writer wins. ¬ß8's R2. |
| **Caveat ‚Äî a thread for the process lifetime** | One daemon thread, blocked on `read`, forever. Acceptable; it is what every implementation of this pattern costs. |
| **Caveat ‚Äî one-way** | Installing over fd 2 is effectively irreversible for the process. Must be install-once and guarded. |

### 5.2 Option 2 ‚Äî route logs through the existing bridge

The `HostCallBegin` slot (offset 72, `BridgeProtocolNative.cs:97`) already crosses the boundary
with signature `int (long requestId, int op, const char* argsJsonUtf8)`. The `op` is an
**existing `int`**, so a sixth value (`Log = 5`) alongside `Geolocation‚Ä¶Camera = 0‚Ä¶4`
(`NativeShellBridge.cs:87`) adds **no export and no struct growth**. It genuinely fits the
constraint.

**And it should still be rejected.** Four reasons, in descending weight:

1. **It cannot carry the output that matters most.** The BCL's output, the NativeAOT runtime's
   own diagnostics, and JNA's swallow-to-stderr handler do not call `BnLog`. Neither does
   anything before `register_bridge` ‚Äî including `blazornative_init`'s failure path, which is the
   single highest-value line the framework ever writes. Option 1 gets all of it for free.
2. **The path is async request/response, and logging is fire-and-forget.**
   `InvokeHostCallAsync` (`NativeShellBridge.cs:437`) allocates a `TaskCompletionSource`, parks it
   in `s_pendingHostCalls`, registers a cancellation callback, and awaits
   `blazornative_host_call_complete`. A log line would need a bypass straight to `BeginHostCall`
   (`:477`) ‚Äî i.e. a second, differently-shaped use of a slot whose entire contract is
   "permission-gated capability, awaited."
3. **JSON on the hot path.** `BeginHostCall` marshals a flat-JSON args string. Renderer
   diagnostics fire per patch.
4. **It grows a mirrored enum with pinned twins.** `HostCallOp` exists in three places (C#
   `NativeShellBridge.cs:87`, Kotlin `HostCallOp`, Swift `BnHostCallOp`) and
   `NativeShellBridge.cs:85` records that the shells mirror it. Adding a value costs three edits
   plus their pins ‚Äî for a channel that still misses (1).

**`RequireSlot` (`NativeShellBridge.cs:290`) throws when a slot is unregistered**, so a logging
path built on it would have to special-case its own absence ‚Äî a logger that throws is worse than
a logger that is quiet.

### 5.3 Option 3 ‚Äî a new export

`blazornative_register_log_callback(fnPtr)` would be the clean API. It is also the most expensive
line of code in the repo.

**What it costs, precisely:**

- **Four CI gates go red by design** and must each be hand-edited: `ci.yml:1514` and `:1547`
  (bionic x64 and arm64, `llvm-readelf --dyn-syms`, both messages hard-coding *"exactly the 10
  `blazornative_*` exports"*) and `ios.yml:168`/`:177` (`nm -gU`).
- **Both shells** grow a binding (`NativeBindings.kt`, `BlazorNativeRuntimeC.h`, `BnRuntime.swift`).
- **The template mirror** grows the same binding ‚Äî `NativeBindings.kt` is one of the 14 Kotlin
  files the template ships, held equal by `TemplateDriftTests` (`:21`, and the file-set pin at
  `:381` that asserts *"exactly these 35 files and no others"*).
- **Every consumer's copied shell** ‚Äî the shells are copied source, not a package. An export that
  a new runtime requires and an old copied shell does not call is survivable; the reverse is not.
- **The property itself.** "Five capabilities, zero further ABI change" is a claim the repo has
  earned four times over and states in four ABI tests. Spending it on a diagnostic channel ‚Äî when
  Option 1 delivers *more* coverage for zero ABI ‚Äî would be a bad trade at any price.

**Not warranted.** Recorded as the honest option it is, and declined.

### 5.4 Option 4 ‚Äî managed-side only, with the level carried in the init input

This is not really a transport (managed-only logging still lands on a `Console.Error` that
Android discards) ‚Äî but it **is** the right answer to the *configuration* half, and the grounding
here is the strongest single finding in this document.

`BlazorNativeInitOptions` (`Exports.cs:60`‚Äì`:74`):

```csharp
public IntPtr PlatformInfoOs;        // offset 0
public int    PlatformInfoApiLevel;  // offset 8   (4 bytes interior padding at 12)
public IntPtr PlatformInfoNote;      // offset 16
public int    PlatformInfoKind;      // offset 24  ‚Üê appended by Phase 10.0 (#121)
                                     //            (4 bytes TAIL padding at 28)
```

Three facts:

1. **It is explicitly not the frozen surface.** `Exports.cs:71`‚Äì`:72`: *"this is the init-INPUT
   struct, **NOT** the frozen 80-byte callbacks bridge."*
2. **It has already grown once, in M10**, by exactly this method ‚Äî append at the end, existing
   offsets 0/8/16 unchanged, ordinal 0 = a safe default. Size went 24 ‚Üí 32. The growth is
   mirrored in the C header (`BlazorNativeRuntimeC.h:64`), the Kotlin JNA struct
   (`NativeBindings.kt:309`‚Äì`:316`), and the Swift call site (`BnRuntime.swift:214`), and pinned
   in all three (`NativeShellBridgeTests.cs:687` = 32 bytes; `BnPlatformInfoTests.swift:39` =
   offset 24; `BootSmokeNativeAndroidTest.kt:42` = 32 bytes).
3. **The next `int` is free.** The struct's alignment is 8 (it contains `IntPtr`s), so
   `PlatformInfoKind` at 24 is followed by **4 bytes of tail padding**. Appending
   `public int LogLevel;` lands it at **offset 28 and leaves `Marshal.SizeOf == 32` unchanged.**

That last point is worth stating plainly: **adding the log level to the init input costs zero
bytes.** It is strictly safer than M10's own growth, which did change the size. An old shell
allocating 32 bytes already covers offset 28; it leaves it zero; ordinal 0 means "unset ‚Üí
`Warn`," which is the quiet Release default. Back-compatible by construction, with the precedent
already in the file.

**Adopted for the level. Insufficient alone for the transport** ‚Äî which is why ¬ß5.5 is a pair.

### 5.5 **Decision**

> **Level travels in the init input (Option 4). Transport is the shell-side stdio pump
> (Option 1), on *both* platforms. Options 2 and 3 are rejected.**

| Option | Verdict | One-line reason |
|---|---|---|
| 1 ‚Äî stderr redirection | **ADOPTED** for transport | Zero ABI, total coverage including BCL and pre-`register_bridge` output, ~40 lines of pure Kotlin at `minSdk 24`. |
| 2 ‚Äî bridge `HostCallOp = 5` | rejected | Fits the constraint, but misses BCL/init output, abuses an awaited permission-gated slot, allocates JSON per line, and grows three mirrored enums. |
| 3 ‚Äî new export | rejected | Breaks the frozen-10 property across four CI gates, both shells, the template mirror and every copied consumer shell ‚Äî to deliver *less* than Option 1. |
| 4 ‚Äî init-input level field | **ADOPTED** for configuration | Not the frozen struct, already grown once by this exact method, and the new `int` fits in existing tail padding at offset 28 with `SizeOf` unchanged at 32. |

**The join between them ‚Äî a level-tagged line format.** Option 1's pump sees text, not levels, so
the managed default sink writes the level into the line and the pump parses it back:

```
[BN|<E|W|I|D|V>|<category>] <message>
```

- The pump maps `E‚ÜíLog.e`, `W‚ÜíLog.w`, `I‚ÜíLog.i`, `D‚ÜíLog.d`, `V‚ÜíLog.v`, tag `BlazorNative/<category>`.
- **A line that does not match the prefix is not dropped.** It is BCL / NativeAOT / third-party
  output, which is exactly the output Option 1 exists to rescue. It is emitted at **`Log.w`** with
  tag `BlazorNative/native` ‚Äî chosen over `Log.e` because unstructured runtime chatter is not
  self-evidently a fault, and over `Log.i` because in Release the only thing that reaches stderr
  unannounced is usually a problem.
- **The prefix is one constant on each side and therefore drift-prone.** It is pinned in both
  directions by ¬ß8.1's drift test ‚Äî the same treatment `ShellStyleTableDriftTests` gives the
  style allow-list.
- Both the formatter and the parser are **pure functions**, which is what makes ¬ß8 able to pin
  behaviour rather than assert a claim.

**iOS gets the same pump.** The unified log does not capture stdio either, so the same
`pipe()` + `dup2()` + a `DispatchSource.read` on the read end, forwarding to the same `BnLog`
categories, gives the runtime's 31 sites a home on iOS too ‚Äî and it is the *only* thing that
makes the two platforms genuinely symmetric rather than merely both-configured. Same format, same
parser semantics, one more implementation of a ~40-line function.

**What this leaves un-fixed, honestly:** the level gate is applied *managed-side*, before the
write. A suppressed `Debug` never reaches the pipe, which is correct and cheap. But a consumer
cannot raise the level *without a restart*, because the init input is read once. That is
acceptable ‚Äî the managed setter of ¬ß3.3(2) covers the in-process case ‚Äî and it is stated so
nobody discovers it later.

## 6. #164 ‚Äî this is not a logging bug, and it does not get a logging-only fix

¬ß5 fixes the half of #164 that is about visibility. On Android, a faulted render will land in
logcat at `Log.e`. That is necessary, and **it is not sufficient**: `mount` still returns **rc 0**,
so the *host* ‚Äî the code that could retry, fall back, or report ‚Äî still learns nothing.

### 6.1 The three options the issue lists

| | Option | ABI-expressible? |
|---|---|---|
| (a) | Route renderer errors to a real platform log sink | **Yes** ‚Äî that is ¬ß5, free |
| (b) | Give the host a signal: an error callback, or a distinct non-fatal rc / frame flag | **Partly ‚Äî and the "yes" half is a trap.** See below |
| (c) | Make property-binding errors specifically fatal | **Yes** ‚Äî see ¬ß6.2 |

**Option (b) deserves the ABI analysis the prompt asks for, because it does not come out where
one expects.**

- **A new mount rc** looks free: `blazornative_mount` already returns `int` with 0/1/2/3
  documented (`Exports.cs:239`‚Äì`:241`). Adding **rc 4 = "mounted with errors"** changes no
  signature. **But it is not back-compatible against the shells that already exist.**
  `BlazorNativeRuntime.kt:369`‚Äì`:374` is:

  ```kotlin
  when (val rc = lib.blazornative_mount(...)) {
      0 -> lines += "[BOOT] mounted $componentName"
      1 -> throw IllegalStateException("unknown component '$componentName'")
      else -> throw IllegalStateException("mount(...) failed with status $rc ‚Äî ‚Ä¶")
  }
  ```

  **An unknown rc is fatal.** The shells are *copied source*: a consumer on an older shell copy
  who upgrades the runtime package would get a hard `IllegalStateException` at boot on a page
  that previously half-rendered. A "non-fatal" rc that crashes every un-updated consumer is the
  `StrictErrors` flip the issue explicitly rules out, arriving through the back door.
- **A frame flag** would live in the frame/patch structs (`PatchProtocolNative.cs`), which both
  shells parse by offset. That is an ABI change of the frame protocol ‚Äî the same class of cost as
  ¬ß5.3, without ¬ß5.3's honesty about it.
- **An error callback** needs slot 11 at offset 80, taking the struct to 88 bytes. The struct is
  *designed* to grow this way ‚Äî `BridgeProtocolNative.cs:66`‚Äì`:69` documents append-only growth
  and `register_bridge` does `min(structSize, sizeof)` + zero-fill (`Exports.cs:377`‚Äì`:392`), so
  it is technically safe in both directions. But it spends the frozen-80 property (four ABI
  tests) on a channel nothing has asked for.

### 6.2 **Recommendation: (a) + (c). Defer (b) with a stated trigger.**

**(a) is mandatory and already bought.** ¬ß5's transport turns "silently half-broken" into
"loudly half-broken." That alone removes the M11-blocking property of #164 ‚Äî the failure is no
longer *invisible*.

**(c) is the right second half, expressed through an rc that already exists.**

A parameter-binding failure ‚Äî `InvalidOperationException` out of
`ComponentState.SupplyCombinedParameters`, the path `NativeRenderer.cs:166` names ‚Äî is
**categorically an author bug**. It cannot be transient, it cannot be recovered from, it will
fail identically on every run of every device, and continuing produces a screen that is wrong in
a way the user cannot report. It is the *opposite* of the fault class log-and-continue exists for.

So: **a binding fault during mount is not swallowed by `HandleException`; it aborts the mount.**
And the rc it produces is the **already-documented rc 2 ("mount threw")**, not a new value.

That choice is the whole point:

- **Zero ABI change** ‚Äî no new value, no new slot, no new export.
- **Both shells already handle rc 2 correctly today.** `BlazorNativeRuntime.kt:371`‚Äì`:374` throws
  a clear `mount(...) failed with status 2`, and ‚Äî with ¬ß5 in place ‚Äî the message it currently
  ends with (*"detail went to native stderr, which Android discards; reproduce on the desktop JVM
  to see it"*) becomes **false in the good way**: the detail is now in logcat, so that string gets
  corrected as part of this phase (and `DispatchEventTest.kt:194`, which asserts on it, updates
  with it).
- **Consumer impact is bounded to code that is already broken.** The only apps whose behaviour
  changes are those with a mistyped `@bind` ‚Äî which today render a half-screen and report
  success. They now fail loudly, and they fail in the **dev loop first**, where the desktop host's
  stderr has always been visible. Nobody's *working* app changes behaviour. That is the argument,
  and it is why (c) is defensible where "turn on `StrictErrors`" is not: `StrictErrors` makes
  *every* render fault fatal, including the genuinely recoverable ones.

**What explicitly does not change:** `HandleException`'s log-and-continue stays for recoverable
faults ‚Äî an event handler throwing mid-session, a fault inside a dispatch window
(`NativeRenderer.cs:1158`‚Äì`:1165`), a `ReportContractViolation` (`:453`). Crashing a running app
over one bad click handler *would* be worse, and the issue says so. The posture is not being
reversed; it is being **scoped** to the fault class it was always right for.

**The predicate matters and must not be message-matching.** Selecting the fault class by
`ex.Message.Contains("does not have a property matching")` would be locale- and version-fragile.
It must key on the **call site** ‚Äî the parameter-supply path ‚Äî so the classification survives a
Blazor message change. Determining the exact seam (a wrapper around the parameter-supply call vs.
a flag set across `SupplyCombinedParameters`) is Gate D's first task and its first risk (¬ß8, R3).

**(b) is deferred**, with a written trigger rather than a vague "later": revisit an error callback
(slot 11) when a **genuinely recoverable** fault class appears that the host must react to
*programmatically* ‚Äî retry, fall back, report ‚Äî rather than merely observe in a log. None of the
five shipped capabilities has produced one. Recording the trigger is what keeps this a decision
rather than an omission.

## 7. No internal detail at default Release verbosity

#155 ends with *"no internal exception detail / paths leaked at default Release verbosity."*
**Level gating alone does not achieve this**, and it is important to say why: the exception that
#164 is about is an **Error**, and Errors are emitted in Release by design. Gating changes
*which* messages appear, not *what is inside* the ones that do.

`NativeRenderer.cs:433` and `:444` write `$"[BlazorNative.Renderer] {exception}"` ‚Äî
`Exception.ToString()`, i.e. type, message, **full stack**, and the whole `InnerException` chain.
NativeAOT without PDBs yields poorer frames than a JIT build, but type names, member names and
any path embedded in a message survive.

**So exception text is treated, in addition to level gating:**

- At **`Warn`/`Error`** (the Release default) the seam emits **exception type + message + the top
  managed frame** ‚Äî enough to identify the fault and the component, not a map of the assembly.
- The **full `ToString()`** ‚Äî inner chain and stack ‚Äî requires **`Debug`**.
- **One documented exception stays verbatim:** `blazornative_init`'s failure path
  (`Exports.cs:157`) deliberately uses `ex.ToString()`, and the comment at `:152`‚Äì`:156` gives the
  reason ‚Äî for the real NativeAOT-trim failure modes (`TypeLoadException`,
  `MissingMethodException`) the `Message` alone hides the offending type. That is a one-shot boot
  failure on a binary that is already not going to run; the diagnostic value outweighs the
  disclosure. Keeping it is a decision, recorded here so it does not read as an oversight.
- **iOS gets a second layer for free.** `Logger`'s interpolation is `private` by default (¬ß4.2),
  so even an emitted message is elided in logs collected off-device unless marked `%{public}`.
  `NSLog` has no such notion ‚Äî another reason the sweep targets `os_log` rather than a
  `NSLog` wrapper.

## 8. What CI can honestly assert

Logging is the classic "green test, unread output" trap, and this repo's house style prefers a
pin over a claim. The honest split:

### 8.1 Genuinely pinnable (real tests, no device)

1. **Level gating** ‚Äî .NET unit tests: each level's `IsEnabled` at each threshold; the default
   is `Warn`; a suppressed level invokes no sink; `RendererServices`' `ConsoleLogger` honours
   `BnLog`'s level instead of a constant.
2. **The init-input ordinal ‚Üí level map** ‚Äî parameterised, exactly mirroring
   `NativeShellBridgeTests.cs:669`'s `ToPlatformKind_MapsTheInitOptionsOrdinal_ByEnumValue`,
   including **ordinal 0 ‚Üí `Warn`** and out-of-range ‚Üí `Warn`.
3. **The struct pin gets *stronger*, not merely edited.**
   `NativeShellBridgeTests.cs:678`'s `InitOptionsStruct_Is32Bytes_MirroringTheCHeaderAndKotlinJna`
   keeps `SizeOf == 32` **and gains** `OffsetOf(LogLevel) == 28`. Those two assertions together
   *prove* the claim of ¬ß5.4 ‚Äî that the field was free. Twins update in the same PR:
   `BnPlatformInfoTests.swift:39` and `BootSmokeNativeAndroidTest.kt:42`.
4. **The line format, both directions.** The managed formatter and the Kotlin/Swift parsers are
   pure functions: given `(level, category, message)` the formatter's output parses back to the
   same triple; an unprefixed line maps to `(Warn, "native")`; a partial line is buffered until
   its newline. Plain xUnit + plain JVM unit tests, no emulator.
5. **The pump's forwarding logic** ‚Äî a JVM unit test drives the pump over an in-memory stream and
   asserts the **sink callback** receives the right priority and tag. This is the honest pin:
   asserting against the injected sink is deterministic, whereas shelling out to `logcat -d` and
   grepping is flaky and slow.
6. **The two anti-regression drift tests ‚Äî the strongest pins in the phase.**
   - **No bare `Console.Error` in `src/**/*.cs`** outside the seam and `DevHostBridge.cs`.
     Today's count is **31**; the pin asserts the allow-listed set, so site 32 reds the PR that
     adds it.
   - **No bare `NSLog` under `src/BlazorNative.Apple/BnHost/**`** outside `BnLog.swift`. Today's
     count is **78**; after the sweep it is 0 (plus `BnLog.swift`'s own). `BnHostTests/**`'s 24
     are explicitly exempt.
   Both are text scans of shell/source files from the .NET test project ‚Äî the exact mechanism
   `ShellStyleTableDriftTests` and `RouteTableDriftTests` already use, so there is precedent and
   no new infrastructure.
7. **#164's behaviour, directly.** Three renderer tests, and these are the ones that make Gate D
   real rather than asserted:
   - a component with a mistyped `@bind` parameter mounts ‚Üí **rc 2**, not rc 0;
   - a component whose *event handler* throws still mounts ‚Üí **rc 0** (the recoverable posture is
     preserved, not collaterally broken);
   - the Release-level exception formatter emits type + message + one frame and **no stack**.
8. **The template mirror** ‚Äî `TemplateDriftTests`' file-set (`:381`) and content pins cover the
   pump's landing in the template's copied shell automatically. No new pin needed; it just must
   not be forgotten, and the existing test will say so if it is.

### 8.2 Instrumented / device ‚Äî weaker, and labelled as such

- An **`androidTest`** that installs the pump, writes to fd 2 via
  `android.system.Os.write(FileDescriptor.err, ‚Ä¶)`, and asserts the **sink** observed it. This
  proves the `dup2` install works on a real Android runtime ‚Äî which the JVM test cannot.
- Asserting the line actually appears **in logcat** requires `logcat -d` scraping and is
  timing-sensitive. **Recommended as an observation in Phase 11.2's device runbook, not a CI
  gate** ‚Äî the runbook is already the repo's home for recorded-not-asserted device truth, and
  11.2 has not run yet, so the step can be added before the owner's session.

### 8.3 Inspection-only ‚Äî say so plainly

- **That an App Store / Release iOS build emits nothing at `Info`.** The iOS lanes are
  simulator-scoped and there is no Apple Developer account (M11 scoping decision #3). Verified by
  reading the sweep and by a simulator Console.app observation; not gated.
- **That the log is *useful*.** No test asserts that a message helps a developer.
- **Third-party stderr interleaving** under a real crash reporter (¬ß8's R2). Documented, not
  tested.

## 9. Backwards compatibility

**Log output is not API, but three things around it are.**

1. **Quieting Release output is not a breaking change** in the semver sense ‚Äî no signature moves,
   no behaviour a program can depend on changes. The honest caveat is human: a consumer who
   greps their logs for `[BnRuntime] mounted` in a Release build loses that line. Mitigation is a
   release note plus the documented opt-in (¬ß3.3), not a compatibility shim.
2. **Two pinned strings *are* depended on, inside this repo.** `BlazorNativeRuntime.kt:373`'s
   *"reproduce on the desktop JVM to see it"* is asserted by `DispatchEventTest.kt:194` and
   referenced at `HostEventTest.kt:195`. ¬ß5 makes that advice obsolete, so message and assertion
   change together ‚Äî a self-inflicted, self-pinned update, which is the good case.
3. **The public level setter is new public API** (¬ß3.3(2)) and therefore interacts with **Phase
   11.3**, which introduces the `PublicAPI.*.txt` baseline. Two orderings, and the choice should
   be explicit rather than accidental:
   - **11.4 before 11.3's baseline generation** ‚Äî the setter is simply part of the first
     baseline. Cleanest.
   - **11.4 after** ‚Äî the setter lands in `PublicAPI.Unshipped.txt` in 11.4's own PR, which is
     precisely the workflow the baseline exists to enforce, and is therefore a useful first
     exercise of it.
   Either is fine; **the PR must not be the place this is discovered.** 11.3 already classes #155
   and #164 as 1.0 blockers and scopes their fix out of itself, so the dependency is one-way.
4. **#164's rc-2 change is the one real behaviour change**, and ¬ß6.2 makes the case: it affects
   only apps that are already broken, and it surfaces in the dev loop before the field. It is a
   **`fix`**-shaped change with a release note, not a `feat`.

## 11. Risks

- **R1 ‚Äî the line format is a stringly-typed contract across three languages.** The prefix
  `[BN|L|category]` is written by C# and parsed by Kotlin *and* Swift. A one-character drift
  silently downgrades every framework line to the unprefixed `Warn` fallback ‚Äî **and nothing
  looks broken**, which is the worst failure shape. *Mitigation:* ¬ß8.1 pin 4 asserts round-trip in
  both directions, on both shells, with the constant declared once per language and pinned by the
  drift test.
- **R2 ‚Äî `dup2` over fd 2 is process-global and effectively irreversible.** A consumer app with
  Crashlytics/Sentry NDK, or its own redirection, collides ‚Äî last writer wins, and if theirs runs
  second the framework's output is silently gone again. *Mitigation:* install once behind a guard,
  make it opt-out via the init input, and document the interaction in the shell docs. This cannot
  be tested; it must be written down.
- **R3 ‚Äî classifying "a binding fault" precisely is the hard part of Gate D.** Too narrow and
  #164's exact case slips through; too broad and a recoverable fault becomes a boot failure ‚Äî
  which *is* the `StrictErrors` flip the issue rules out, arriving by accident. *Mitigation:* key
  on the call site, never the message; both directions pinned by ¬ß8.1 pin 7 (rc 2 for binding,
  rc 0 for handler). **If the call-site seam turns out not to be cleanly expressible, Gate D
  ships (a) alone and (c) is re-scoped** ‚Äî that is an acceptable outcome, and better than a
  fragile predicate.
- **R4 ‚Äî the 78-site iOS sweep touches nearly every `BnHost` file.** Enormous diff, near-zero
  semantic content, high merge-conflict cost against any parallel iOS work, and a review that
  invites rubber-stamping. *Mitigation:* one mechanical commit with no semantic edits mixed in,
  the level assignment decided by class in ¬ß4.3 rather than per-site, and the ¬ß8.1 pin 6b drift
  test landing in the same PR so the sweep cannot rot.
- **R5 ‚Äî level gating without ¬ß7's redaction leaves half of #155 open.** Errors ship in Release by
  definition, and today an Error is a full `ToString()`. It is easy to declare #155 done on the
  noise half and quietly skip the disclosure half. *Mitigation:* Gate D's exit criterion names the
  redaction test explicitly.
- **R6 ‚Äî a quiet framework is a framework nobody can debug.** Over-suppression is the mirror-image
  failure of #155, and it will be discovered by a stranger, in the field, at the worst moment.
  *Mitigation:* `Warn`/`Error` genuinely ship in Release, ¬ß4.3 keeps the mapper diagnostics at
  `Warn` against #155's literal text, and the opt-in of ¬ß3.3 is documented where a consumer
  actually looks ‚Äî not only in a design doc.
- **R7 ‚Äî the Phase 11.3 ordering (¬ß9.3).** The public setter is new public API and the baseline is
  being introduced in a neighbouring phase. *Mitigation:* pick the ordering before Gate A, not in
  review.
- **R8 ‚Äî one more daemon thread, forever.** Two, counting iOS's dispatch source. Small, but it is
  a permanent process-lifetime cost added to a framework whose selling point is a small runtime.
  *Mitigation:* accepted explicitly; it is the price of Option 1's total coverage, and Options
  2‚Äì4 do not avoid it while delivering the same coverage.

## 12. Out of scope for 11.4

- **Telemetry and metrics.** No counters, no timings, no aggregation. A log seam is not an
  observability stack.
- **Structured logging sinks** ‚Äî Serilog, OpenTelemetry, JSON-per-line, a `ILoggerProvider`
  ecosystem. The sink delegate of ¬ß3.2 is deliberately the *narrowest* extension point that works;
  a consumer wanting more can build on it later.
- **Crash reporting.** Crashlytics/Sentry integration, symbol upload, minidumps. ¬ß11's R2 notes
  the *interaction* with such a tool; integrating one is not this phase.
- **`DevHostBridge.cs`'s 18 `Console.Write*` calls.** The desktop dev host's stdout *is* its UI
  (¬ß1.1). Quieting it would break the loop it exists to serve.
- **`BnHostTests/**`'s 24 `NSLog` sites** and the Kotlin test sources. Test output is not
  production output.
- **The C-ABI.** Frozen (¬ß0). 11.4 is designed specifically to not touch it, and ¬ß5.4 proves the
  one field it does add is outside it.
- **Option (b) of #164** ‚Äî a host error callback or a new mount rc. Deferred with the written
  trigger in ¬ß6.2.
- **#169** (geolocation `Accuracy` never displayed) ‚Äî a sample-app defect with a known cross-shell
  blast radius; unrelated to logging.
