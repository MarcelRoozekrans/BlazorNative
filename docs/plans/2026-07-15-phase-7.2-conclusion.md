## What shipped

**The first 60Hz producer runs on the existing event wire, and the wire held.** Scroll position
reaches .NET as a normal `dispatch_event` — no ABI change, no new export (the 9-export surface is
untouched on all three RIDs) — because the shells **conflate**: one pending offset per scroll node,
latest wins. On top of it: **`BnList<TItem>`**, the virtualized list, authored in `.razor` from day
one (`@typeparam` rides the Razor source generator into `public partial class BnList<TItem>` — the
7.1 recipe holds for generics; no `.cs` fallback was needed).

The pieces, bottom-up:

- **`BnListWindow.Compute`** — the pure window function
  `(offset, viewport, itemHeight, count, overscan) → [Start, End)`, exact arithmetic, exhaustively
  unit-tested (**35 cases**). Offsets are clamped as runtime DATA (iOS rubber-band negatives, 6.2's
  shrink-past-the-end); author misuse (non-positive viewport/height, NaN offset) throws loud.
- **`OnScroll` on `BnScroll`** — the wire's .NET end: `EventCallback<BnScrollEventArgs>` (the args
  type lives in Core, shared by renderer-construct and component-consume), invariant-culture payload
  parse, and the handler is **attached only when the app subscribed** — attach-on-subscribe, like
  every other handler.
- **`BnList<TItem>`** — a `BnScroll` whose content is a keyed lead spacer, the window's rows (each
  `Height = ItemHeight`, **`@key` = the item**), a keyed trail spacer. Lead + window·ItemHeight +
  trail sums to Count × ItemHeight by construction — 6.2's synthetic-content arithmetic. Recycling
  is nothing but keyed diffing plus the shells' existing create/remove; the shells stay dumb.
- **`BnListDemo` at `/list`** — 500 rows × 64dp in a 300×400 viewport, overscan 4. Liveness is a
  **counted assertion, not a feeling**: 11 rows live at the top, 15 mid-scroll, 11 at the bottom —
  asserted from the .NET node count AND both shells' child counts, with the goldens and frame
  tables pinned like every M6 page.

**Counts (Gate 4 close):** .NET **388/0** · JVM **83/0** · Android instrumented **124/0** ·
iOS XCTest **85/0** (run 29435524820) · no ABI change anywhere.

## The wire contract, as-proven

The design demanded throughput evidence before `BnList` was built on the wire. The contract's rows,
each with its proof:

| Rule | As-proven |
|---|---|
| **Conflation** | ONE pending offset per scroll node; a new native sample **replaces** it — never queues. Scroll position is idempotent state, not an event log. Kotlin `ScrollWire` / Swift `BnScrollWire` are the same object in two languages. |
| **Dispatch edges** | The pending offset is dispatched on the three lane-availability edges (attach, sample-with-free-lane, dispatch-completion) — at most one in-flight dispatch per node, and **the completion always runs** (iOS grew a `dispatchEvent(onComplete:)` Swift-side overload so the completion is structural, one lane by construction). |
| **Backpressure** | A slow consumer sees **fewer, fresher** events. The burst tests are the hard bound: flooding the wire while the lane is busy produces a handful of dispatches and the FINAL offset always arrives. |
| **Ordering** | Free, not implemented: scroll dispatches ride the same single FIFO lane (`BlazorNative-Dispatch`) as taps and change events, so a conflated scroll can never overtake a user-input event already queued for the same node. |
| **Units** | dp/pt at the source — Android divides px by density at the ONE conversion site (6.1 rule); iOS points are already pt. |
| **Throughput evidence** | **Android: a 100-sample burst → 2 dispatches (50:1); a scripted fling: 78 samples → 35 dispatches (2.23:1), final offset always delivered. iOS: the same numbers' mirror on the same `/list` surface.** The fling ratio varies with dispatch latency by design (a fast lane conflates little); the burst test owns the hard bound. |

The conflation was verified **at the mechanism, by mutation, on both shells**:

- **Android** (instrumented, 124/0 green): mutate dispatch-per-sample (drop the conflation slot) →
  **4 tests red**; mutate queue-not-replace → **2 tests red**, with the stale-middle-value smoking
  gun (the consumer observes an offset the finger already left).
- **iOS — mutations run ON CI, not claimed:** dispatch-per-sample → run **29436079300**, **81/4
  red on exactly** burst + busy-lane + detach + purge; queue-not-replace → run **29436083505**,
  **83/2 red on exactly** busy-lane + burst — **the same failure shapes as Android**, and the frame
  tables stayed green throughout (the mutations break the wire, not the layout — exactly the
  isolation the design predicted).

**Row state travels by view IDENTITY, and eviction really destroys.** A surviving row keeps its
node, so an `EditText`/`UITextField`'s text and focus survive the slide (asserted on-device by
identity, not equality); a row that leaves the window is destroyed with its native view, and
re-entering creates it fresh — unbound native state does not survive eviction, only sliding. An app
that wants eviction-proof text binds it (`@bind-Value`), exactly as on the web.

## The `@key` finding (design decision 4 — empirically first)

**A keyed window slide diffs as INSERT/REMOVE ONLY — never permutations.** `KeyedWindowSlideTests`
mirrors BnList's exact content shape (keyed lead spacer, keyed slice, keyed trail spacer) and runs
**strict, with the 7.0 `default:` arm live** — the test is the detector, not an assumption: if a
`PermutationListEntry` ever appeared, the arm would throw and redden the dispatch (and
`StrictModeTests` proves the arm fires by reddening a genuine two-item swap through it today).

Why it holds: Blazor's keyed diff emits permutations only when keys present on BOTH sides **cross**.
A sliding window preserves the relative order of every surviving key — the leaver is one
`RemoveFrame`, the enterer one `PrependFrame` at its sibling index, the survivors structurally
untouched. That untouchedness is what makes recycling free *and* what makes row state travel.

**Consequence:** the `PermutationListEntry`/`End` arms are NOT implemented; the loud arm stays. A
genuine keyed reorder remains a deliberate, named violation. **No wire move-concept is needed for
virtualization.**

**The deferred ABI conversation, with its trigger:** a true keyed reorder (surviving keys crossing —
drag-to-reorder, sort-in-place) could NOT be lowered to remove+insert of the same node id without
destroying native view state; it would need `CreateNode`'s InsertIndex semantics applied to an
EXISTING id — a new patch meaning, i.e. an ABI conversation. That conversation is deferred **until
the first component whose UX requires surviving keys to cross**; the sliding window, the only
customer this milestone has, does not.

## The two renderer bugs — found red-first, in code that shipped in 3.3

Building the first component that removes-at-the-front while inserting-at-the-back **in one batch**
put load on a disposal path no page had ever exercised at that angle. Both bugs were caught by
Gate 1's own tests before any device saw them:

1. **Disposal removes trailed the batch's creates — with InsertIndex translated against trimmed
   state.** The 3.3-era ordering emitted a disposed component's `RemoveNodePatch`es AFTER the
   diffs' creates, but the diffs had already trimmed the disposed children's sibling slots and
   translated later insert indices against the trimmed state — so the host, applying patches in
   order, inserted against a tree that still contained views .NET had already forgotten. The
   window slide is the exact shape: keyed rows leave at the front while new ones insert before the
   trail spacer. **Fixed removes-first**: disposal pass 1 (`EmitDisposedComponentRemoves`) runs
   before the diffs, so removes PRECEDE every create in the frame. (Side effect, documented in the
   host contract: a disposed child's redundant removes now arrive BEFORE its removed ancestor's
   own remove, where they used to trail it — either way unknown ids are a no-op and
   child-before-ancestor is a legal detach order.)

2. **Same-batch re-render + disposal → a zombie view — and the review's "narrow future case" was
   constructible after all.** If a component's re-render was queued before the parent render that
   disposed it, its FINAL diff runs in the same batch and can ADD root-level views after pass 1
   read the bucket — views nothing ever removes on the host. The Gate 1 review flagged the shape
   but judged that **no current page produces it**; the fix agent then **built it through a public
   path** — a child's event handler calling `StateHasChanged()` followed by the parent's remove
   callback, nothing internal touched. That is the finding to name: **"no current page produces
   it" is a claim about the demo pages, not about the public surface — if a shape is constructible
   through public API, it is not a future case, it is a present bug without a current caller.**
   (The 6.2 lesson — a mechanism nobody tested is a mechanism nobody knows — sharpened into its
   review-time form; this milestone-pair keeps re-earning it.) **Fixed with a pass-2 delta
   emission** (`EmitDisposedComponentRemovesDelta`: any disposed root view pass 1 didn't cover is
   removed at the frame's tail — safe there because no InsertIndex translation happens after the
   diffs and RemoveNode is by-id). `SameBatchRerenderDisposalTests` pins it, mutation-proven:
   drop the delta and the zombie is back, red.

The Gate 3 review's discipline note carries: **every guard was RUN, on both shells** — the mutation
evidence above is CI run links, not commit-message assertions.

## The honest ledger

- **Variable row heights** — ledgered with the design problem NAMED: **estimate → measure →
  correct**. An unmeasured row must be given an estimated height for window math, measured on the
  host after creation, and the correction fed back without the scroll position jumping — a
  feedback loop through the wire that fixed `ItemHeight` (design decision 3) deliberately avoids.
  It is its own design section when a customer forces it, not a parameter to bolt on.
- **`scrollTo` / offset restore** — the wire is one-directional (shell → .NET). Programmatic
  scrolling and navigation-time offset restore need the reverse path; ledgered since 6.2, still
  open, unchanged by this phase.
- **Horizontal lists** — `BnScroll` is vertical (6.2 decision 2). A `Horizontal` list variant may
  force horizontal scroll; ledgered, revisit noted in MILESTONE's inheritance section.
- **The true-move ABI conversation** — deferred with its trigger recorded (above): the first
  component whose UX makes surviving keys cross.
- **Hardening issue #9 (dispatch-lane starvation) — re-examined with real numbers, as the
  milestone promised.** The first 60Hz producer **cannot flood the lane by construction**: the
  conflation slot bounds scroll to at most one in-flight dispatch per scroll node, backpressure is
  replace-not-queue (the burst evidence: 100 samples → 2 dispatches while the lane is busy, with
  the freshest value winning), so lane load scales with scroll-node count, never with sample rate.
  What #9 was actually about — one slow **handler** head-of-line-blocking the single FIFO lane —
  is untouched by 7.2 and stays open for M10; but its worst projected aggravator (a high-frequency
  producer growing the queue behind a slow handler) is now structurally impossible for scroll. A
  slow handler under a fling degrades to *fewer, fresher* scroll events — measured, not hoped.
