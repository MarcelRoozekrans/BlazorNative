## Why this ledger exists

DoD #3 does not say "the newcomer path works." It says **every friction point found is fixed or
ledgered with a reason.** So the walk had to be done as a newcomer really does it — install the
published template, `dotnet new`, add packages, publish, gradle — and *every* stumble recorded
with what the docs promised, what actually happened, and who owns the outcome. This file is that
record. It is deliberately complete rather than flattering: the items that turned out **not** to be
defects are kept, with their investigation, so they are not re-raised by the next walker.

**Two apps produced it**, both outside `c:\Projects\Prive\BlazorNative`, both from nuget.org with
**no `ProjectReference`**:

- `C:\Projects\Dogfood\bn-baseline` — the stock generated app on published **0.3.0** (Gates A/B).
- `C:\Projects\Dogfood\bn-zeroalloc-showcase` — the full-ecosystem app on published **0.4.0**
  (Gate D), plus the ABI harness `C:\Projects\Dogfood\showcase-runproof`.

## Dispositions used

| Disposition | Means |
|---|---|
| **FIXED** | a change landed in this repo (PR named), so the next newcomer does not hit it |
| **RESOLVED** | the underlying condition is gone (a release shipped the missing thing), verified live |
| **NOT-A-DEFECT** | investigated, found to be correct behaviour or a local artifact — recorded so it is not re-raised |
| **LEDGERED** | real, not fixed here, owner named (an issue, a later phase, or an upstream repo) |
| **DEFERRED** | out of scope by an explicit prior decision, not by omission |

## Part 1 — BlazorNative-owned friction

These are **our** problems. Every one of them is either closed or has a named owner in this repo.

| # | What a newcomer hit | Evidence | Disposition | Rationale / owner |
|---|---|---|---|---|
| **F0** | **The template pack was never published** — and `getting-started/quick-start.md` told the reader `dotnet new blazornative` would just work. The single most severe finding of the whole dogfood: the documented front door did not exist on the feed. | The M8 deferral, carried in the project ledger as "template not published"; the doc claim in `quick-start.md`. | **RESOLVED (0.4.0)** | 0.4.0 is the **first release ever to publish `BlazorNative.Templates`** (wiring: **#162**). Verified empirically, not assumed: a real `dotnet new install BlazorNative.Templates` on a clean machine resolved **`BlazorNative.Templates@0.4.0`** from nuget.org. The doc's claim is now *true* rather than aspirational. |
| **F-version** | A walk observed every version literal in the repo sitting at **0.2.0** while nuget.org already served 0.3.0 — read at first as a release-automation gap. | Re-checked against real `main`: `templates/.../MyBlazorNativeApp.csproj:149`–`151` carry `Version="0.4.0"` each tagged `<!-- x-release-please-version -->`; `.release-please-manifest.json` = `0.4.0`. | **NOT-A-DEFECT** | It was a **stale local checkout**, not the repo. The template's content pins are release-please-managed and track each release correctly. Recorded here *only* so this observation is not filed again as a bug. |
| **F1** | Docs said a consumer runs **seven** `dotnet add package` commands. Reality: **three direct** (`Runtime`, `Components`, `Analyzers`) — `Core`, `Http`, `Device`, `Renderer` all ride **transitively** via `Runtime`. | `installation.md` "Consuming the packages directly" table vs `MyBlazorNativeApp.csproj:149`–`151`. | **FIXED — #157** | The docs now state three direct + a table naming the four transitive ones, with `dotnet list package --include-transitive` as the reader's own check. Over-stating the dependency count makes the framework look heavier than it is. |
| **F2** | Docs hardcoded **v0.1.0** while the feed had moved on. | `installation.md` version literals. | **FIXED — #157** | Fixed **version-agnostically**, not re-bumped: the install command carries no version, and the page says the current version lives on nuget.org (with `::<version>` shown for pinning). A re-bump would re-stale on the next release; this cannot. |
| **F3** | Docs described a **win-x64 desktop dev-loop run**. The template ships **no runnable desktop host** — the win-x64 publish produces a loadable `BlazorNative.Runtime.dll` and nothing that launches. | `quick-start.md` §3 vs the template contents (no host project). | **FIXED — #157** | The claim was made **honest** rather than backfilled: §3 now says the publish gives you "the loadable `.dll`, not an app you can launch without writing a host of your own." **No desktop host was invented to rescue a doc sentence** — inventing one would have been new framework surface smuggled in under a docs fix. |
| **F4** | The `@using Microsoft.AspNetCore.Components.Web` footgun — without it `@bind`/`@onclick` compile to **literal markup**, green build, zero diagnostics, silently dead page. | The template's `_Imports.razor` already carries the `@using` **and** a load-bearing comment saying exactly this; `quick-start.md` repeats it under "Two rules worth knowing". | **NOT-A-DEFECT (confirm-and-keep)** | The footgun is real but **already defused** for anyone starting from the template. The item is kept as a standing *do-not-remove* marker on that line, because the failure mode is invisible. |
| **F5** | **iOS simulator walk** — not attempted. | MILESTONE.md scoping decision #3; 11.1 design "iOS-sim — DEFERRED". | **DEFERRED (by design)** | Owner decision predating the walk: 11.1 proves **Android + the desktop publish**. iOS remains a manual, non-templated procedure (no iOS template, no iOS generator). Deferred ≠ untested-by-accident; the boundary is stated in the conclusion. |
| **F6** | **Three environment prerequisites** a newcomer has no way to guess: `vswhere` reachable (else the win-x64 NativeAOT **link** fails with `MSB3073`), `JAVA_HOME` on **JDK 21** for gradle, `ANDROID_NDK_ROOT` for the bionic publish. | `installation.md:91`–`96` (the Prerequisites note added by the fix). | **FIXED — #157** | A Prerequisites note now names all three with the failure each one prevents. **It recurred in Gate D** on a second machine — which is the evidence that documenting it was worth doing rather than a one-off environment quirk. |
| **F7** | The template's `AppPages.cs` ships **no `using Microsoft.Extensions.DependencyInjection;`** and no hint the DI seam exists — so a newcomer's first `BlazorNativeApp.ConfigureServices(...)` attempt fails with **CS0246** on `IServiceCollection`, in the one file where they would naturally put it. | `templates/.../AppPages.cs` — the `[ModuleInitializer]` sits at the bottom with only `System.Runtime.CompilerServices` + `BlazorNative.Runtime` in scope. | **FIX IN FLIGHT — PR #165** | Adds the `using` plus a **commented example beside `RegisterPages`**. The seam is 0.4.0's headline feature; shipping it with a template that fails to compile the first honest attempt at using it is exactly the kind of friction DoD #3 exists to catch. |
| **F8** | A wrong page parameter (`@bind-Value` where the property is `Checked`) **threw**, the renderer logged to stderr, **and `mount` still returned `rc = 0`** with a partially-rendered frame. | `src/BlazorNative.Renderer/NativeRenderer.cs:80`–`87` (`StrictErrors`, default **false**, documented as "the deliberate production POC posture"), `:423` (`HandleException` rethrows only under strict), `:452` ("Non-strict: stderr + drop, the POC posture"). | **LEDGERED — [#164](https://github.com/MarcelRoozekrans/BlazorNative/issues/164)**, cross-ref [#155](https://github.com/MarcelRoozekrans/BlazorNative/issues/155); fix belongs to **Phase 11.4 / DoD #6** | Not a bug — the code does exactly what it documents. Logging discipline (DoD #6) is where the shells get one level-gated sink; the rc/strictness posture is decided there with it. |
| **F13** | `gradlew assembleDebug` needs the **linux-bionic-x64** publish too, not only arm64 — publishing just arm64 (the "real device" one, the obvious choice) fails the build. | `templates/.../android/build.gradle.kts:203` — `verifyNativeAssets` requires **both** `.so`s and, notably, **names the exact `dotnet publish` command** for each missing one. | **LEDGERED → documented in the walkthrough** | The tooling behaves **well** here (fail-fast, actionable message, explicit `-PappPubRoot` hint), so there is nothing to fix in code. ## Part 2 — ZeroAlloc-upstream friction

These are **not BlazorNative's problems.** They are recorded because a consumer following the
showcase hits them, and because "the ecosystem layers cleanly" is only an honest claim if the
places it did not layer cleanly are named. Each names its upstream fix.

| # | Package | What a newcomer hit | Disposition |
|---|---|---|---|
| **F9** | `ZeroAlloc.Serialisation` | The `SystemTextJson` format **requires a `JsonSerializerContext` in the same compilation**, enforced as build error **`ZASZ004`**. Nothing in the nupkg says so — there is **no README** — so the first build fails on a diagnostic code with no searchable documentation. | **LEDGERED — good behaviour, bad discoverability.** The check itself is *right*: it converts a runtime AOT failure (reflection-based JSON under ILC) into a **compile error**, which is precisely the trade BlazorNative wants. The fix is documentation. The required ~6 lines are written out in the [walkthrough](2026-07-21-phase-11.1-walkthrough.md) so a consumer is not left bisecting. |
| **F10** | `ZeroAlloc.Rest` | `IRestSerializer`'s members are annotated `[RequiresUnreferencedCode]` + `[RequiresDynamicCode]`, so **any** implementation must repeat both annotations — **IL2046/IL3051 ×4** — even one that is provably AOT-safe (source-generated, no reflection). | **LEDGERED — upstream.** The annotation buys nothing here: the **generated** REST client already suppresses at the call site, so the interface-level requirement only taxes honest implementors. Upstream fix: drop the annotations, or split an `IAotRestSerializer` for implementations that make the AOT-safe promise. |
| **F11** | `ZeroAlloc.Validation` | The README says the **core package contains the generator**. It does not — there is no `analyzers/` folder in 1.5.3. Without also adding `ZeroAlloc.Validation.Generator`, `[Validate]` **silently does nothing**: no validator is emitted, no diagnostic is raised, the build is green. | **LEDGERED — upstream README fix.** Silent no-op on an attribute is the worst failure shape: the app compiles and appears to validate. |
| **F12** | `ZeroAlloc.AsyncEvents` | The documented payload base `SourcedAsyncEventArgs` is **`sealed`** — an app cannot derive from it and must implement `IAsyncEventArgs` directly; and `AsyncEvent<T>`'s real subscriber shape `(TArgs, CancellationToken)` is **undocumented**. Both were found by **compiler-error bisection**, not by reading anything. | **LEDGERED — upstream docs (and arguably an unseal).** |
| **Cache** | `ZeroAlloc.Cache` **1.1.15** | **DROPPED — the one package of 12 that did not make it**, and it never even reached the trim gate: it fails at **`csc`**. Two independent defects. **(1)** Its generator is **not gated on `[Cache]`** — it emitted cache proxies for **every interface in the compilation**, including ones carrying only `[ZeroAllocRestClient]` / `[Retry]` / `[Instrument]`. **(2)** The emitted `Add{X}Cache<TImpl>()` calls `AddTransient` / `GetRequiredService` **unqualified**, with no `using Microsoft.Extensions.DependencyInjection;` and no `global::` prefix — while **every other symbol in the same generated file *is* `global::`-qualified** — which makes it a plain emit oversight. Result: **14 × CS1061**. | **LEDGERED — reported upstream as [ZeroAlloc-Net/ZeroAlloc.Cache#87](https://github.com/ZeroAlloc-Net/ZeroAlloc.Cache/issues/87).** Dropped **with a written reason**, exactly as the 11.1 design required of any package that fails its gate. Note *what this is not*: it is **not** a trim/AOT incompatibility, and it says nothing about whether `ZeroAlloc.Cache` would survive ILC — the compile never got that far. |

**The scoreboard: 11 of 12 in, 1 out.** Inject, Collections, AsyncEvents, Rest, Resilience,
Serialisation, StateMachine, Validation (+ Generator), Results, Mediator, Telemetry all built and
published trim-clean; only Cache was dropped. Mediator and Telemetry — both flagged **stretch** in
the design, both most at risk — **passed**.

## What the ledger says about the framework

Read Part 1 top to bottom and the shape is consistent: **every item is docs, template content, or a
posture decision.** Not one is a defect in the runtime, the renderer, the bridge, the ABI, or the
codegen. The two heaviest findings — F0 (the front door was not on the feed) and F7 (the seam's
own file cannot compile the seam) — are both *packaging and template* problems, and both are
closed or in flight. F8 is the only one that touches product behaviour, and it is a documented
posture with a named owner (Phase 11.4), not a surprise.

That is the honest, unglamorous result a dogfood is supposed to produce: the machine works, and
the paper around the machine was where the newcomer kept tripping.
