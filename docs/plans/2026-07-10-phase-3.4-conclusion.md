## Finding 1 ‚Äî the 3.3 Region diagnosis was wrong; the fix converts an accident into a contract

**The honest reversal:** 3.3's carryover (and its conclusion doc) claimed Region
subtrees were **"silently dropped"** ‚Äî that `RenderFragment`/`CascadingValue`
ChildContent never rendered. That was **empirically wrong.** Gate 1's TDD RED step
refused to go red: **all 6 planned `RegionWalkTests` were GREEN before any
implementation existed.**

What is actually true on Blazor 10.0.x, established by scratch-worktree
reproduction: `RenderTreeDiffBuilder` **decomposes region inserts/removes into
per-child edits** ‚Äî a PrependFrame's reference root is never a Region frame ‚Äî and
the pre-3.4 element walk's no-default `switch` **fell through Region frames by
accident**, coincidentally producing exactly the region-transparent sibling
numbering Blazor expects. Cascading values had been rendering all along.

**Sabotage methodology ‚Äî proving the tests bite:** since the tests passed against
the accident, the obvious worry is that they'd pass against anything. A deliberate
sabotage (skip region subtrees *without* descending) turned **4 of 6 tests RED** ‚Äî
the suite genuinely pins region-transparent rendering; the two survivors
(CascadingValue/RenderFragment basics) are disclosed in-file as pinning the
per-child decomposition assumption itself rather than the Region arm.

So the shipped change is not a bugfix but a **hardening**: an explicit Region arm
(descend, same slot context, no slot for the region, subtree skip in the element
walk) replaces load-bearing fall-through, with the defensive region-root-prepend
path kept for a frame shape Blazor 10 never emits. The ledger in
`NativeWidgetTree.cs`, the 3.3 conclusion, the 3.4 design, and ROADMAP all carry
the correction blockquote ‚Äî the record is honest end to end.

**The meta-lesson** (this is the project's first reversed diagnosis): the 3.3 claim
was written from reading the walk code, not from running the construct. Verify a
diagnosis empirically before building a phase plan on it ‚Äî and when TDD refuses to
go RED, treat that as data, not friction: sabotage-test the tests before trusting
green.

## Finding 2 ‚Äî component-root chaining recorded the wrong parent bucket (`ContractWarning` caught it as designed)

Found by BnDemo's own composition in Gate 2 (`e51d5de`): a component whose ROOT
frame is *another component* ‚Äî the natural Bn* pattern; `BnThemedPanel` renders a
`BnView` ‚Äî has its component slot in the wrapper's root bucket (`parentNodeId`
null), but `ProcessFrame`'s Component arm recorded the component-parent entry with
the **emit parent** (the enclosing host node). Every `IndexOfComponentSlot`
consumer ‚Äî `HasHostViewsAfter`/`HostBaseOffset` (host-index translation) and
disposal's `RemoveComponentSlot` ‚Äî keys the bucket by the recorded ParentNodeId,
looked in the **wrong bucket**, and hit the quiet append fallback: the chained
inner's view (rendered LAST by Blazor's FIFO render queue) landed *after* its later
siblings. Symptom: BnDemo's echo panel appended below the buttons.

**The 3.3 strict infrastructure caught its first real bug.** The quiet-fallback
path reports through `NativeWidgetTree.ContractWarning` ‚Äî built in 3.3 exactly for
"the fallback result is applied either way, but somebody should hear about it" ‚Äî
and the warning named the wrong-bucket lookup directly. Fix: the Component arm
registers the **slot container**; `ResolveComponentEmitParent` walks the
component-parent chain to the nearest element container (replacing the one-hop
read). `ComponentChainTests` pins the shape: div > [span, Wrapper(‚ÜíInner), span] ‚Äî
Inner's span must insert at host index 1 (RED ‚àí1 append, GREEN `InsertIndex 1`).
Goldens, DiffCursor, RegionWalk, SlotList tests all untouched by the fix.

## The @bind posture ‚Äî mechanics, not syntax

`BnInput` ships the `Value` + `ValueChanged` (`EventCallback<string>`) pair ‚Äî
**exactly what Razor's `@bind-Value` compiles to**. The Razor syntax itself awaits
`.razor` compilation (M6, `BlazorNative.Cli`/tooling territory); until then parents
wire the pair by hand (`BnDemo` does). This is documented in `BnInput`'s XML doc
and worded honestly in MILESTONE DoD #5: the *mechanics* are closed; the *syntax
sugar* is deliberately out of M3 scope. The loop as shipped: host change event
(EditText TextWatcher ‚Üí 3.2 dispatch plumbing) ‚Üí `onchange` invokes `ValueChanged`
‚Üí parent state mutates ‚Üí re-render ‚Üí `UpdatePropPatch("value")` ‚Üí the host writes
the widget under its `applyingBatch` guard.

## Complementary write-back assertions ‚Äî the complete DoD #5 proof

The JVM and Android twins deliberately assert **opposite halves** of the same
contract:

- **JVM pins arrival:** `BnDemoTest.kt` asserts the value write-back `UpdateProp`
  *arrives* through the ABI after a change dispatch ‚Äî the echo loop exists.
- **Android pins suppression:** `BnDemoAndroidTest` asserts the EditText is **NOT
  clobbered** after typing ‚Äî the arriving echo takes the inequality-check skip (it
  merely confirms what was typed) and the `applyingBatch` guard would swallow the
  watcher dispatch even if it didn't. Typing produces exactly one change dispatch.

Neither alone proves the bind loop: arrival without suppression is an infinite
echo; suppression without arrival is a dead loop. Together they are the complete
DoD #5 proof. The same complementarity holds for placement: the .NET/JVM tests pin
the echo panel's **`InsertIndex 2`** on the wire; the Android test **materializes
it on-screen** ‚Äî the echo panel is asserted at child position [2] of the form,
between the input and the Clear button, exactly where Blazor's FIFO-late creation
would NOT have put it under an append fallback (Finding 2's symptom, permanently
pinned).

## Carryovers

### PRE-3.5 MUST ‚Äî custom `AndroidJUnitRunner` for strict mode

The `Os.setenv` strict-mode pattern is **at its ceiling**. Current cost: **two**
identical `@BeforeClass` setenv copies (`BnDemoAndroidTest` +
`CompositionAndroidTest`), **three** ordering-contract KDocs, a standing dependence
on the runner's default **alphabetical class ordering** ("Bn" < "Bo" <
"Composition" ‚Äî Gate 4 moved first-mount ownership once already), and a live
**filtered-run gap**: a filtered run of `EventRoundTripAndroidTest` or
`WidgetMapperTest` mounts first with *nobody* setting the variable ‚Äî those classes
silently run **non-strict** today. A custom `AndroidJUnitRunner` subclass
(~10 lines: `onCreate` ‚Üí `Os.setenv("BLAZORNATIVE_STRICT", "1", true)` before any
test class loads) plus one `testInstrumentationRunner` line in `build.gradle.kts`
deletes both copies, all three KDocs, the alphabetical assumption, and the gap.
Do this **before** 3.5 adds its own instrumented classes to the ordering.

### M6 ledger (ships with the packaged component library)

- **Stringly-typed `FontSize`/`Padding`** ‚Äî conscious Gate 2 decision: `int?`/
  `double?` parameters + component-layer formatting would be idiomatic, but the
  props are string pass-through at POC fidelity. **Switching later is
  source-breaking for consumers** ‚Äî decide at M6 packaging, not after.
- **Stale-echo/IME-composition artifact under fast typing** ‚Äî the value write-back
  compares against the EditText's *current* text; an in-flight echo carrying an
  older value can land after further typing, lose the inequality skip, and
  overwrite + jump the cursor. Last-write-wins is the documented posture (Blazor
  state is truth), but distinguishing a stale echo from a genuine programmatic
  overwrite is **sequence-stamping territory** (frame/sequence ids on the wire) ‚Äî
  M6, alongside any IME-composition work.
- **Theme colors literal-pinned in 4 files** ‚Äî `#FFEEAA`/`#334455` appear in
  `BnDemo.cs` (canonical), `BnDemoTests.cs`, `BnDemoTest.kt`, and
  `BnDemoAndroidTest.kt`; a palette change is a 4-file edit. Fine for a demo;
  a shared fixture source when the components package ships.
- **`FrameTestPins`/`FramePins` paired harness extraction** ‚Äî the structural pin

  `BnDemoTest.kt` (third partial copy in the CompositionProbe twins). Extract the
