# Phase 2.1.0 Spike Conclusion — Format Pivot Was a No-Op

**Date:** 2026-05-26
**Outcome:** PASSED trivially. No code changes required.

## Spike's original premise (per Phase 2.1 design)

> "The .wasm currently imports `wasi:http/types`. We need to strip that transitive and author a custom `mobile_bridge.wit` to re-emit a slim component that wasmtime's C-API can fully host."

## What the baseline revealed (Task 2 — `docs/plans/2026-05-26-phase-2.1.0-baseline-wit.txt`)

The Phase 2.0 `.wasm` (May 25, 13.7 MB) already imports a minimal WASIp2 surface — **no `wasi:http/types`** anywhere. The component world is:

```
world root {
  import wasi:cli/* (12 interfaces)
  import wasi:io/{error, poll, streams}
  import wasi:clocks/{monotonic-clock, wall-clock}
  import wasi:filesystem/{types, preopens}
  import wasi:sockets/{network, udp, tcp}
  import wasi:random/random
  export wasi:cli/run@0.2.0
}
```

The brainstorm's premise was based on Phase 1.3 observations (`wasmtime run -Shttp ...` was needed to invoke the .wasm CLI-side). The flag was needed for *wasmtime CLI's* feature gating, not because the .wasm actually imports `wasi:http/types` at the component layer. We conflated two things.

## Second surprise — `mobile_bridge` imports are absent at the component level

Mono-AOT's trimmer strips the `[DllImport("mobile_bridge")] internal static extern void shell_navigate(...)` declarations in `src/BlazorNative.Core/WasiBridge.cs:172-178` because nothing in the reachable graph from `Main` actually calls any `shell_*` extern:

- The Phase 2.0 self-test calls `WasiBridge.DispatchEventCore` (managed-only path; no extern).
- DI composition resolves types but doesn't invoke bridge externs.
- `[DynamicDependency(All, typeof(WasiBridge))]` on `Main` keeps `WasiBridge` itself alive, but transitively-unreachable members (the `Native` class with the externs) can still be stripped.

This is unintended but not broken for Phase 2.1 — the .wasm has no `mobile_bridge` imports for libwasmtime to satisfy, so the JNA-side stubs in `MobileBridgeHost` end up registering against an interface the .wasm doesn't actually import. The stubs will be dead code at Phase 2.1's boot test.

## Carryover question for Phase 2.3 — bridge import revival

Phase 2.3 ("real `mobile_bridge` implementations") was originally scoped as "put real Kotlin code behind the imports the .wasm declares." But the imports aren't declared at the component level today. When Phase 2.3 arrives, we choose between:

- **(a) Force-root the externs.** Add a `[DynamicDependency(All, typeof(Native))]` to keep all `Native.shell_*` reachable, OR add a `[ModuleInitializer]` that takes a dummy reference to each. The trimmer keeps them; they show up as component imports. Quick fix (~30 min); same shape as the existing Phase 1.3 trim-root pattern.
- **(b) Migrate to `wit-bindgen`-generated bindings.** Author `mobile_bridge.wit`, run `wit-bindgen csharp` (or its equivalent — needs research) to produce typed partial methods, replace the `[DllImport]` declarations. Bigger refactor; aligns with the design's "WIT-typed imports = JSI-equivalent" future direction. Phase 2.3 brainstorm picks.

The hybrid WIT contract from Q5 of the Phase 2.1 design (typed primitives for simple imports, JSON-string payloads for `shell-fetch` / `shell-storage-write`) is the *target* shape for option (b). The mobile_bridge.wit file authored as part of that effort lives at `src/BlazorNative.WasiHost/wit/mobile_bridge.wit` (not yet created — Phase 2.3 work).

## Decision recorded

Proceed with the revised Phase 2.1 path: skip Task 3, collapse Task 4 to this conclusion doc, continue with Task 5 (libwasmtime cargo build).
