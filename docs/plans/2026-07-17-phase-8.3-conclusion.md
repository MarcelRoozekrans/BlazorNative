## THE HEADLINE — two defects, both invisible to every pin, both found by doing the real thing

**This is the phase's whole lesson, and it vindicates design decisions 3 and 8.** The template was
green by every measure the phase had built for it — nine .NET pins, byte-identity on fifteen shell
files, six version literals mirrored, the Kotlin compared byte for byte — and it shipped **an app
that did not build** and **an app missing a file git never committed**. Neither defect was reachable
by reading. Both were found by doing the real thing.

### 1. The generated Android app did not compile

Gate 2's first `assembleDebug` on a freshly generated tree:

```
e: .../TemplateSmokeApp/android/src/androidMain/kotlin/io/blazornative/shell/MainActivity.kt:133:24 Unresolved reference 'R'.
FAILURE: Execution failed for task ':compileDebugKotlin'.
```

`MainActivity` used a bare `R.layout.main` with no `import <namespace>.R`. The mechanism, and it
falsifies a design assumption rather than a typo:

- **AGP generates `R` into the `namespace` package.**
- **Kotlin resolves a bare `R` against the FILE's own package.**
- The repo's two match **by coincidence** — `build.gradle.kts` sets `namespace = "io.blazornative.shell"`,
  which happens to equal its source package.
- **The template's differ BY DESIGN** — and that divergence **IS decision 6's byte-identity trick**:
  the shell's Kotlin package stays `io.blazornative.shell` precisely so the pin can be a file
  comparison, while `namespace` becomes the user's app id.

**So the trick that makes the pin exact is what broke the file the pin exempts.** The design had the
first clause right (*"AGP's namespace does not have to match a source package"*) and drew a
conclusion that holds for every shell file that never names `R` — and `MainActivity` names it three
times. **The trick had a hole exactly where the design didn't look.**

**All nine .NET pins were green. Only `assembleDebug` saw it** — which is precisely why decision 8
chose a real compile over `gradlew tasks`. **DoD #4's own sentence — "runnable end-to-end" — was
unmet until this landed.**

**State plainly why the pins were blind:** *the pin was green precisely because the template
faithfully copied a file that is only correct in the repo's namespace.* **Byte-identity to a
reference is not correctness when the reference's correctness depends on a property the copy
deliberately changes.**

**The fix:** `import com.example.starterapp.R`, rewritten at generation by `finalApplicationId`'s
**existing** `replaces` — **no new symbol**. Chosen over the alternative that costs no excision
(`namespace = "io.blazornative.shell"` in the template) **because that does not delete the trap, it
MOVES IT INTO THE USER'S CODE**: their app's `R` and `BuildConfig` would live in a package this very
template tells them is *"library code you are not meant to edit"*, and the day they add their own
Activity in their own package and write `R.layout.foo`, it would not resolve — a baffling error, in
their code, for a reason nothing names. **That is this repo's least favourite failure shape, paid
for by a test's convenience.**

**The third Pin A excision is the STRONGEST of the three, not the weakest — it asserts in BOTH
directions:** the template must **have** the import (`Assert.True` — deleting it breaks the generated
build, so the pin says so before a user does); the repo must **NOT** (`Assert.False` — if the
reference ever grows one, its namespace and package have drifted apart and this excision would be
hiding that).

**The reviewer bounded the blast radius rather than accepting it:** all 15 (now 19) byte-pinned shell
files are bare-`R`-free — `WidgetMapper`'s two hits are `android.R.layout.simple_spinner_item`, the
**framework's** R, fully qualified and namespace-independent. **Decision 6's trick survives intact;
the fix is contained to the file that was already exempt.**

### 2. `.gitignore` was swallowing the template's NDK shim

`templates/.../content/BlazorNative.App/build/BionicNativeAot.targets` — the vendored NDK
cross-compile shim — was matched by the unnegated `build/` rule. **Every generated app would have
shipped with no NDK shim → no bionic publish → no `.so` → no APK.**

**And a byte-identity pin cannot see a file git never committed.** The pin compares what is *there*;
a file that was never added is not drift, it is absence, and absence was outside every pin's world.
**Found by construction** — by building the thing, not by reading it.

Fixed with a scoped negation (`!/templates/BlazorNative.Templates/content/BlazorNative.App/build/`).
**The second trap was got right too:** `template.json`'s `exclude` narrows to `**/build/**/*.jar`,
**not** `**/build/**` — which would have re-swallowed the same file at *generation* time, one layer
further out, where no pin in the repo could ever have seen it.

## The central proof — the trim line, proven on generated output

**Design decision 3's argument, landing on the phase that wrote it.** A text pin proves *the line is
in the file*. It cannot see ILC silently ignoring an unresolvable root (MSBuild does not validate a
root name — a bogus one passes evaluation with **no error and no warning**), a dropped
`UnmanagedEntryPointsAssembly`, a broken RID condition, a missing `PublishAot`. **And decisively:
8.0's own line was found missing by a PUBLISH, not by a READ.** *"The line is in the file"* and
*"the app survives ILC"* are different claims, and only the second is the one that matters.

**So the lane does not read the template.** `scripts/template-smoke.ps1`: **pack → install from the
nupkg** (the real shipping path, not the checked-in folder) **→ generate → restore** (temp cache,
`--no-cache`, provenance ×3) **→ publish → `assembleDebug`**.

**GREEN, end to end (~1.8 min local):**

```
[arm 1] IL2072 count: 4 ✓
[arm 2] Canonicalized native artifact: TemplateSmokeApp.dll -> publish\BlazorNative.Runtime.dll
[arm 2] Exports found (9): blazornative_dispatch_event, … ✓
[arm 3] Page-presence probe 'BnStarterPage': present ✓ (image 3838 KB)
✓  the generated Android project assembles: TemplateSmokeApp-debug.apk (15590 KB)
✓  template smoke PASS
```

**THE MUTATION — `TrimmerRootAssembly` deleted from the template (run, quoted, reverted):**

```
[arm 1] IL2072 count: 0 ✗ (expected exactly 4)
[arm 2] Exports found (9): … ✓
[arm 3] Page-presence probe 'BnStarterPage': ABSENT ✗ (image 3592 KB)
```

**It reds arm 1 AND arm 3 — and arm 2 STAYS GREEN.** The reason is the insight the phase hands
forward, recorded verbatim:

> **The 9 exports ride out of the *referenced* Runtime assembly and survive a whole-module trim of
> the app. Arms 1–2 are facts about the LIBRARIES; only the probe is a fact about the USER'S CODE.
> That is why arm 3 exists.**

A lane built on arms 1–2 alone would have watched the generated app be deleted by ILC and reported
nine healthy exports. The image also shrank 3838 → 3592 KB — the third arm of 8.0's signature,
visible but unasserted.

**The probe DID red, so the starter page's name is discriminating** — a fact now, not a hope. The
design's fallback (*rename the page until it reds*) was not needed.

**The step was restructured to evaluate all three arms BEFORE any fails.** The trio is a
**signature**, not three checks — 8.0's failure was recognisable because its arms moved *together*.
**A fail-fast at arm 1 could not have quoted arm 3**, which the gate requires. *`0` is not "fewer
warnings" — it is the app being gone.*

## The decisions as proven

- **The pack stays OUT of `src/`.** `src/` membership is load-bearing: `PackagePurityTests` proves
  `src/` holds exactly the six shipped csprojs, and **that proof is what licenses
  `src/Directory.Build.props` to be 8.1's props home at all.** A seventh csproj there **un-licenses
  it**. The six's pins are **assembly-shaped** (type purity off the PE, a sentinel type per package,
  snupkg pairing, `ResolveAssemblyPath`) and a **content-only pack satisfies none of them** —
  joining would mean a special case in **five pinned places**, and **a special case is a hole in a
  pin**. `templates/` is a top-level sibling on its own feed, so the smoke's `6 nupkg + 5 snupkg`
  stays an *exact* count that means something.
- **The version is a pinned MIRROR, not a wildcard.** The disqualifying reason is sharp: **the CI
  lane restores from a local feed holding exactly one version, so `1.0.0-preview.*` always resolves
  and the drift is invisible on CI.** That is a **vacuous pass** — 8.1's headline. **Not twice.**
  The props remains the ONE source; the template's literal is held equal by a pin that reds, the same
  relationship `DEEP_LINK_COMPONENTS` has to the page manifest.
- **`$(AssemblyName)` in the trim root**, not a substituted literal — it removes the substitution
  dependency entirely and is correct for any `-n` the user picks. **Verified on hostile names:**
  `-n Weird.Name.With.Dots -o DifferentFolder` → `"Identity": "Weird.Name.With.Dots"`.
- **The guard order flipped in BOTH copies**, with the behaviour-identical claim **VERIFIED at the
  mechanism** rather than asserted: validation precedes assignment under a lock; there is no callback
  into app code, so reentrancy is unreachable; and both callers are `ModuleInitializer`s, which the
  CLR serializes. One test covers both copies, so **reference and template cannot drift apart on the
  exact line 8.0's review flagged**.
- **The FOURTH Yoga pin** — `android=ios=ci=template=3.2.1`, asserted by a **real read of the
  template's gradle**. Without it a generated app lays out differently from **both** shells,
  silently, on a machine nobody watches. The design called it *"~2 lines and not optional"*; the
  mutation (template's Yoga → 3.3.0 alone) reds both locks.
- **The CI lane lives INSIDE `build-test`** — no fourth required check, no branch-protection change,
  the three gates keep their names and contexts. ~1.8 min local, ~3–5 min CI cold-Gradle, on a job
  with a **45-minute timeout**. **Zero extra publishes:** `build-test` already publishes both bionic
  RIDs, so `-PappPubRoot` points the generated project at the sample's tree and **a real APK
  assembles for free**.
- **`blazornative` is FREE — U1 CLOSED, not carried.** `dotnet new search blazornative` →
  *"No templates found matching: 'blazornative'."* The design recorded this UNPROVEN because the
  sandbox's DNS was down (8.2's offline condition); it was run for real at close.

## The review trail (PASS-blocked → fixed)

**C-1 (Critical)** — the generated app did not compile. The headline above.

**I-1 (Important) — THE SHELL PIN WAS A ROSTER AND DID NOT KNOW HOW MANY FILES IT SHOULD PIN.**
Every pin asserted the **content** of files it was **told the names of**; nothing knew how many files
the template *should* ship. **The reviewer proved it live with three mutations, ALL GREEN:**

1. the byte roster shrunk **15 → 1**;
2. a new shell file in `src/BlazorNative.Jni` the template never gets;
3. **seven unrostered template files DELETED** — `gradlew`, `gradle-wrapper.jar`,
   `settings.gradle.kts`, `AndroidManifest.xml`, `BnStarterPage.razor`, `_Imports.razor`,
   `gradle.properties`.

**A `dotnet new` app missing its wrapper JAR is a tree whose FIRST command fails — and it would have
shipped.** This is the `.gitignore` find restated: that proved a template file can silently never
*exist*; this proved the pin was blind to **7 of the 32 that do**. And it violated **the file's OWN
cited rule** — the header invokes *"the 8.1 I-2 rule: never a roster"* forty lines above a hardcoded
roster.

**The fix pass then found the reviewer's prescribed fix was insufficient for the reviewer's own
class.** A manifest pin closes only the *deletion* mutation — the other two leave the template tree
untouched. **Closed from three sides instead:**

- **The content-manifest pin** — the tree enumerated **from disk**, held equal to a **32-name
  manifest**, red on REMOVE *and* on ADD. The enumeration **matches the pack's own glob**
  (`content/**` minus bin/obj), so **it IS the nupkg's inventory, not a proxy for it**. The manifest
  is an **expected value**, not a roster of subjects — *enumerate what you MEASURE, declare what you
  EXPECT*; the old code inverted it, which is exactly why shrinking the declaration shrank the test.
- **The shell set DERIVED from the template's tree** — *a roster that cannot be shrunk is a roster
  that does not exist.* Deriving **widened it 15 → 19**, revealing that **`gradlew`, `gradlew.bat`,
  the wrapper JAR and `gradle-wrapper.properties` were byte-identical copies NOTHING COMPARED AT
  ALL**.
- **The repo-side pin** — the other direction: both other pins read the *template's* tree, so neither
  sees `src/` growing a file the template never gets. That is the drift the header always claimed to
  catch and did not.

**And the root cause the review missed:** **the design's split table filed the gradle wrapper as
TEMPLATE-OWNED.** It is byte-identical. **That wrong row is why the roster never listed those four
files.** *A wrong row in a table became a hole in a pin — the anti-roster argument, restated.* Table
corrected; the set now derives from the tree, so **the table no longer feeds the pin — it only
describes it.**

**M-1 (Minor) — the guard-order pin was COMMENT-BLIND**, and proven so: the reviewer's vehicle (the
bug present, plus a comment naming `RegisterPages(` above it) → **"Passed! - Failed: 0, Passed: 1"
WITH THE BUG PRESENT**. Comments are stripped first now. ***The pin was finding its own documentation
and calling it code.***

**M-2 (Minor)** — `literals.Count >= 5` while the pin collects **six**. *A floor below the real count
is a floor never reached.* Replaced with a **named set**, not `== 6`: both red on the drop, but **a
count names a number and a set names the FILE**.

**M-3 (Minor)** — the prose staleness is **TWO spots**, and the second is the sharper: `MainActivity.kt:33`
(class KDoc, *"mount **BnDemo**"*) and **`:58`** (companion KDoc, *`Absent → "BnDemo" since Phase 3.4
Gate 4`*) — **which contradicts the code 134 lines below it** (`:192` reads `?: "BnStarterPage"`, the
literal Pin A's excision replaces). *The file tells the reader the app boots into a page it does not
contain, and then boots into a different one.* Ledgered to 8.4 with the principle recorded:
**excisions we already need are free; new excisions are new holes** — leaving both byte-identical is
right **not** because the prose is acceptable, but because un-pinning prose would add a third
uncompared region to a pin the design already calls brittle, to fix a comment.

**Also corrected:** Gate 1 claimed `MainActivity` *"diverges in exactly two places"*; after C-1's fix
it is **three**. Fixed in `ci.yml`'s provenance and the design, both now naming the third and why
only a real compile could find it.

## The iOS doc — its honesty is that it refuses to transcribe

`docs/ios-shell-setup.md`. **The structural fact that decides the whole shape**, verified this phase
rather than recalled: **`project.yml` links against `$(SRCROOT)/vendor/…` and names no publish
directory and no app anywhere.** The link between *"your .NET app"* and *"the iOS shell"* is a
directory of frozen-name files, and **the only thing that populates it today is CI** — ~90 lines of
bash across two workflow steps. No script, no Makefile target, no doc.

**So a transcription would be a FOURTH copy of a recipe that already exists twice in YAML, and it
would rot the day `ci.yml` moved.** The doc points at the live files instead and names what the
reader must change. **`ios-build` is a REQUIRED check, so the referenced material is kept true BY A
GATE rather than by prose discipline** — and the doc says so: *"when this prose and that job
disagree, the job is right."*

**Three claims verified with REAL line numbers:**

1. **`HostViewController.swift:60`** hardcodes `try runtime.start(component: "BnDemo", os: "ios")` —
   a generated app must change it. Its twin is **`BnRuntime.swift:184`**'s default. **Both are SOURCE
   EDITS IN THE SHELL**, and the doc uses that word. A mismatch is not a compile error: the mount
   fails at runtime.
2. **The ATS exemption (`Info.plist:44-48`) exists only for the fixture server** — a new app that
   deletes `BnHostTests` inherits an exemption it never asked for.
3. **A correction the design got wrong:** it cited `AppleShellBridge.swift:102` — **that is the doc
   comment**. The **code** is `fetchBegin` at **`:106`**, `return -1` at **`:108`**. The doc cites the
   function.

**U4 stands and is stated IN THE DOC, not just the ledger:** no lane executes this procedure.
`ios-build` proves the reference shell compiles and links; it does not prove a reader following these
steps arrives at a running app.
