# Phase 4.3 Design — Dev Inner Loop / Fast-Restart

**Date:** 2026-07-11
**Milestone:** M4 — P3: Production-Shippable (closes DoD #5)
**Predecessor:** Phase 4.2 GREEN (PR #39, `c8d69c3`)
**Owns:** the one-command native dev loop the README already promises — file-watcher →
win-x64 publish → JVM preview restart (fast lane), and the scripted ADB lane — with
measured round-trip times. **Fast-restart, not hot-reload**, and the docs say why.

---

## Decisions locked (2026-07-11 brainstorm, survey-grounded)

| Decision | Choice |
|---|---|
| Fast-lane surface | **Console tree dump** — new Kotlin `PreviewHost` (the repo's first JVM `main()`): boots the dll, mounts a component (CLI arg), prints the widget tree as indented text + stage timings, exits. |
| ADB lane | **Scripted + measured** — same script, `-Android` switch: bionic-x64 publish → `installDebug` → `am start` with `EXTRA_COMPONENT` → logcat tail; round-trip measured. |
| Why restart, not reload | Survey-proven: JNA `Native.load` is process-lifetime (no unload API; Windows locks the dll) — a warm JVM cannot pick up a rebuilt dll. Restarting a tiny host process IS the design. |

## 1. PreviewHost (Kotlin, the fast-lane surface)

- **`TreeSnapshot`** (testable core, `main/kotlin/io/blazornative/jni/`): consumes
  `BlazorNativeFrame`s via the existing `NativeFrameAdapter` read path and maintains a
  minimal node model (id, type, text, props, styles, parent, ordered children via
  InsertIndex semantics — the WidgetMapper placement rules, console-shaped). `render()`
  returns the indented text tree. Pure JVM, no Android imports.
- **`PreviewHost.kt`** — thin `main(args)`: parse component name (default `BnDemo`),
  `BlazorNativeRuntime` boot → mount → drain the mount frames into a `TreeSnapshot` →
  print tree + per-stage timings (dll load / init / mount / total) → exit 0 (non-zero on
  mount failure, honest stderr).
- **Gradle**: a `runPreviewHost` `JavaExec` task (JVM classpath + the `jna.library.path`
  the tests already use; `-Pcomponent=` pass-through). Placement note: `main/kotlin` is
  shared with the Android app — PreviewHost there ships ~KBs of dead code in the APK;
  if that offends, the implementer may place it in a JVM-only source set/task instead —
  either is acceptable, document the choice.
- **Tests**: `TreeSnapshotTest.kt` (JVM) — synthetic frames → expected indented text
  (create/text/props/insert-index/remove shapes). PreviewHost's `main` stays thin enough
  to not need its own test; one recorded manual run is the smoke.

## 2. devloop.ps1 (the loop)

`scripts/devloop.ps1`, pwsh 7, house style (`test-android.ps1` conventions: strict mode,
repo-root resolve, JDK/ANDROID_HOME autodetect, colored steps):

- **Default (JVM fast lane):** initial cycle, then `FileSystemWatcher` on
  `src/BlazorNative.{Core,Renderer,Http,Components,Runtime}/**/*.cs` (debounced ~500 ms).
  Cycle = `dotnet publish src/BlazorNative.Runtime -c Release -r win-x64` →
  `gradlew runPreviewHost -Pcomponent=$Component` → tree + **per-stage + total wall
  times printed every cycle**. Ctrl+C exits.
- **`-Android`:** cycle = bionic-x64 publish → `gradlew installDebug` (preBuild already
  re-copies the fresh `.so`) → `adb shell am start -n io.blazornative.shell/.MainActivity
  -e io.blazornative.shell.EXTRA_COMPONENT $Component` → tail `adb logcat -s BlazorNative`
  until the boot marker, stamping the measured round-trip. Requires a running
  device/emulator (fail fast with the `test-android.ps1` pointer if none).
- **`-Once`:** single cycle, no watcher (CI-able smoke / timing capture).
- Parameters: `-Component` (default BnDemo), `-Android`, `-Once`.
- Makefile: `devloop` + `devloop-android` targets.

## Risks

| Risk | Mitigation |
|---|---|
| dll locked by a lingering PreviewHost/gradle daemon breaks the publish step | PreviewHost exits after dump (no daemon holds it); devloop runs gradle `--no-daemon` for the exec task or verifies exit; publish failure message names the lock cause |
| Gradle JVM startup dominates the cycle time | Measure; if gradle overhead swamps, run PreviewHost via direct `java -cp` (classpath emitted once) — decide on evidence |
| TreeSnapshot drifts from WidgetMapper placement semantics | Snapshot tests mirror the WidgetMapper pin shapes (InsertIndex, text-child-on-TextView aliasing documented as N/A on console) |
| Logcat boot marker flaky | Use the established `[BlazorNative]` boot log the shell already emits; timeout with honest failure |
