## THE HEADLINE — the 9.0 ABI bet paid a THIRD time: biometrics + secure storage cost the ABI NOTHING

**Phase 9.0 grew the bridge to 80 bytes / 10 exports ONCE and argued it would be paid once** —
9.1/9.2/9.3 add op constants and touch the ABI at nothing else. Notifications drew first (9.1);
biometrics + secure storage draw second, and **they hold the bet exactly**:

- **Bridge stayed 80 bytes.** No new slot. Both ops ride the existing `HostCallBegin` at offset
  72 (the last slot; nothing appended).
- **Exports stayed 10.** No new symbol. The calls reuse `host_call_complete`; the secure `get`'s
  value rides its **optional payload** (the channel geolocation's fix opened — this is the second
  user, so the channel is proven generic, not one-off). The win-x64/bionic `dumpbin`/`readelf`
  gates and the iOS archive `nm` gate all show the SAME ten `blazornative_*` symbols
  (`all 10 present`).
- **No drift-pin moved, no gate arithmetic.** The three-mirror struct pin (80 on .NET / Kotlin /
  Swift), the export gate (10), and the offset pins are byte-for-byte what 9.0 left. Secure
  storage added only **wire vocabulary**: ops 2+3, the two status enums, the byte cap.

**The reuse proof is made falsifiable, not asserted.** `SecureBiometricsAbiUnchangedTests` pins
`Marshal.SizeOf<BlazorNativeBridgeCallbacks>() == 80`, `OffsetOf(HostCallBegin) == 72`,
`Biometrics == 2`, and `SecureStorage == 3`, *inside the secure suite* so the claim lives next to
the code it guards. The named mutations prove it bites: assert-81-bytes reds the .NET pin, and the
iOS **struct-grow mutant failed to COMPILE** (`missing argument 'mutantEleventhSlot'`, run
**29636616995**) — the Swift type system itself forbids a silent grow. **"Pay once, reuse thrice"
is now demonstrated three times, by construction.**

## The OS-key binding — the security crux, and WHERE it can be proven

This phase's real lesson is not the wire; it is **where a security property can be proven**. The
owner chose **OS-key-level** binding over app-level:

> The OS refuses to hand back plaintext without a fresh authentication — the key material is bound
> to the user's biometric at the OS keystore, so a control-flow bypass in the shell (skip the
> prompt, call decrypt anyway) still fails, because the OS, not the app, holds the gate.

An app-level gate ("check the prompt returned true, then decrypt an unbound key") would be a lie a
single mutated `if` defeats. The OS-key gate is not.

- **Android PROVES it.** The AVD's **software keystore enforces `setUserAuthenticationRequired(true)`**:
  a plain get of an auth-bound secret returns **AuthFailed** — the OS refuses the decrypt with no
  prompt satisfied. This is **mutation-airtight**: drop `setUserAuthenticationRequired` and the
  binding test reds **alone**, naming exactly the removed line.
- **iOS asserts the CONTRACT with OS-enforcement UNPROVEN.** The simulator has **no Secure Enclave**
  and treats `.biometryCurrentSet` as a **no-op**; iOS real-device is deferred (no Apple account).
  So CI asserts the shell **REQUESTS the ACL** (a construction seam — the `SecAccessControl` is
  built and passed) and **enforces the contract** (a plain get → AuthFailed from the shell's own
  auth-bound cache), but the **OS-enforced refusal is named UNPROVEN-until-real-device**.

**This is the milestone's clearest example of a security property provable on one platform and not
the other, recorded honestly rather than smoothed over.**

## Two real platform asymmetries this phase surfaced — both documented, not glossed

1. **An auth-bound `set` PROMPTS on Android but NOT on iOS.** Android's per-use-auth AES key
   requires a fresh authentication to **encrypt**, so writing an auth-bound secret raises the
   prompt; iOS's Keychain ACL gates **retrieval only**, so the write is silent. **Same wire,
   different write-time UX** — a genuine per-platform behavior, documented for anyone building on it.
2. **The iOS simulator enforces no `SecAccessControl`** (no Enclave), where **the Android emulator's
   software keystore DOES enforce** the key-auth requirement. This is why Android can prove the OS
   binding on CI and iOS cannot — the asymmetry above in DoD #4's PROVEN/UNPROVEN split, at the
   emulator level.

## The secret-in-memory security note

The wire is **intra-process and trusted** (the encryption is at-rest, in the keystore/keychain, not
a transport defense). The residual hazard is **non-zeroable plaintext copies**: the .NET `string`,
the marshalled buffer, and the 9.0 TCS window each hold plaintext the GC may relocate and cannot be
wiped. This is **documented for the POC**; the zeroable-buffer pass is **M10** (security hardening).
The **8 KB cap** is enforced at the .NET boundary (`SecretResult.MaxValueBytes`): an oversize value
returns **Error and never crosses** the C boundary.

## Gate 3's 5-run iOS Keychain convergence, honestly

The iOS Keychain maze on a Mac-less setup took five `ios.yml` runs to converge — recorded, not
hidden:

1. **entitlement** — the Keychain needs `keychain-access-groups`; CI got ad-hoc sim signing to
   give the simulator a real Keychain (`036398d`).
2. **sim-no-Enclave** — the OS binding cannot be OS-enforced on the simulator, so it became a
   **construction seam** (`2c347a6`).
3. **a plain-path regression** — the simulator returns a `kSecAttrAccessControl` attribute **even
   for plain items**, so classifying "is this auth-bound?" off the stored attr **misclassified**
   plain secrets; fixed by keying off **the shell's own cache**, not the sim's attribute
   (`edc0e15`).
4. **a real bug — the denial-as-data LAW caught it.** The biometrics deny replied **SYNCHRONOUSLY
   inside `hostCallBegin`**, hanging the bounded await (the completion arrived before the await was
   parked). The milestone's **denial-as-data law** (denial is a status, bounded, never a hang) is
   exactly what surfaced it; fixed via the **notifications deferred-reply pattern** (`8fb455c`).
   **The law found a genuine bug** — the clearest vindication of the standing discipline this phase.

## Verification honesty — the PROVEN / UNPROVEN split

**PROVEN on CI:** both status matrices as data; the Keystore/Keychain round-trip; **Android's
OS-enforced AuthFail** on an auth-bound secret; the gated read via the seam; denial-as-data with no
hang. The 80-byte / 10-export ABI unmoved on all three RIDs.

**UNPROVEN until the owner's physical Android phone** — named, not smuggled:

- **The real fingerprint sensor** — CI drives a fake-biometric AVD path; a genuine sensor is device
  territory.
- **TEE-enforced auth-bound decrypt** — the AVD's *software* keystore proves the binding; the
  hardware TEE enforcing it is the phone's job.
- **Real lockout** — repeated denials escalating to a device lockout no emulator reproduces.

**Biometrics is THE least emulator-like capability of M9** — more of its proof waits on the phone
than any prior phase.

**iOS OS-enforced binding stays DEFERRED — doubly gated:** the simulator has no Enclave to enforce
the ACL, AND iOS real-device needs the Apple Developer account (the M5–M9 trigger). The iOS proofs
here are simulator-scoped, construction-seam + contract, and labeled.
