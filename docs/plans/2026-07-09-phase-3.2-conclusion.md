## The `dispatch_event` contract as-built

From the `DispatchEventCore` header in `Exports.cs` (the contract of record ‚Äî the
Kotlin KDoc on `NativeBindings.blazornative_dispatch_event` carries the same table):

Args are flat JSON via the 3.1 FlatJson pair: `{"name":"click"}` /
`{"name":"change","payload":"‚Ä¶"}`.

| Code | Meaning |
|---|---|
| 0 | dispatched ‚Äî INCLUDING a stale handlerId: delivery is at-most-once, the renderer catches Blazor's `ArgumentException` for a handler that died in a re-render and logs it (a stale tap is not an error) |
| 1 | no session / nothing mounted |
| 2 | dispatch faulted ‚Äî the handler, the resulting re-render, or frame delivery threw (anything routed to `HandleException` inside the dispatch window; detail `ex.ToString()` on stderr ‚Äî Kotlin logs loudly) |
| 3 | malformed or NULL args, including a handlerId outside the int range of the renderer's handler table (guarded BEFORE the `(int)` narrowing ‚Äî silent truncation could alias onto a LIVE handler) |

**SYNCHRONOUS by contract:** the renderer's InlineDispatcher runs the handler, the
re-render, and the FrameSink callback on the calling thread, so everything ‚Äî including
frame delivery to the host ‚Äî has completed when the export returns. Frames therefore
still fire only inside host calls (mount OR dispatch), containing the 3.0d trampoline
hazard. The host-side threading contract (single `BlazorNative-Dispatch` lane, never the
UI thread) lives in `BlazorNativeRuntime.kt`.

## Key discovery 1 ‚Äî Blazor 10 swallows handler throws; rc 2 needed a capture window

The naive rc-2 implementation (catch around the awaited `DispatchUiEventAsync` task)
**cannot work on Blazor 10**: `Renderer.DispatchEventAsync` does NOT propagate handler
exceptions to its caller. They route to `HandleExceptionViaErrorBoundary` ‚Üí (no error
boundary here) ‚Üí `HandleException`, and the returned task completes *successfully*.
Without intervention, a throwing `@onclick` handler would report rc 0 and vanish ‚Äî the
exact silent-swallow failure mode that cost a day each in Phases 2.4 and 2.7 (Bugs A/B)
and motivated DoD #9.

**As-built:** `NativeRenderer` opens a **capture window** around each
`DispatchUiEventAsync` call ‚Äî the first exception Blazor routes to the overridden
`HandleException` while the window is open is remembered and faults the dispatch task,
which the export maps to rc 2 + full `ex.ToString()` on stderr (diagnosable from
logcat). The capture is deliberately a *window*, not a handler hook: the handler itself,
the resulting re-render (`UpdateDisplayAsync` failures), and frame delivery all land in
it. The window is **depth-counted** (Gate 1 polish): nested dispatches (a handler that
dispatches) don't reset it, so an outer handler's throw cannot be discarded by an inner
dispatch. This ships the *partial* DoD #9 ‚Äî handler-window exceptions are now visible;
full renderer strict mode (exceptions outside a dispatch window, e.g. during mount)
remains open. The guarantees assume SYNCHRONOUS handlers; the in-code breadcrumb notes
async handlers (`await` in `@onclick`) move continuations off the dispatch thread and
out of the window ‚Äî 3.3+ revisit.

## Key discovery 2 ‚Äî the re-render addressing bug the device exposed

Gate 3's first live tap incremented the counter in .NET but **the screen never
changed**. Root cause was in the renderer since before this phase, latent because
nothing had ever re-rendered: `ProcessRenderTreeDiff` resolved re-render edits by their
raw `SiblingIndex` against the mount-time sibling map, **ignoring Blazor's
StepIn/StepOut cursor** ‚Äî a diff edit's `SiblingIndex` is RELATIVE to the current
container, and `ReferenceFrameIndex` is batch-relative. Hello's counter `UpdateText`
therefore resolved to the OUTER div (node 1); `ReplaceTextPatch` targeted a container
`LinearLayout` and silently no-opped on Android.

**Why it survived Gates 1 and 2:** every re-render assertion up to that point was
**text-only** ‚Äî the .NET and JVM tests asserted the second frame *contains* a
`ReplaceText` with `taps: 1`, which is true even when its nodeId is wrong. Only a
surface that consumes nodeIds for real (the widget tree on the device) could fail.

**Fix:** a proper diff cursor in `ProcessRenderTreeDiff` ‚Äî `StepIn(siblingIndex)`
descends via ordered-child bookkeeping (`NativeWidgetTree._childOrderMap`, recorded at
node creation where creation order IS render-tree sibling order), `StepOut` pops, and
positional edits (`UpdateText`, `RemoveFrame`, `RemoveAttribute`) resolve through
`GetChildAt(componentId, currentParent, siblingIndex)`. A failed StepIn **poisons the
cursor** (`PoisonedCursor = int.MinValue` ‚Äî never a real node id, never the -1 root
sentinel): edits under an unknown container miss their lookups and `PrependFrame` is
dropped by an explicit guard rather than aliasing the sentinel into a live child-order
bucket (Gate 3 polish).

**Recurrence prevention ‚Äî nodeId is now pinned at three layers** so a wrong-target
patch can never again hide behind matching text: `HelloGoldenTests` (.NET typed golden
asserts the ReplaceText nodeId), `NativeFrameAdapterTest` (Kotlin golden, same), and
`EventRoundTripAndroidTest` (the on-screen text itself ‚Äî the assertion that caught it).
Plus a dedicated `DiffCursorTests` with a `StepIn(1)` case (descend into the *second*
child ‚Äî distinguishes cursor-relative from absolute addressing, which `StepIn(0)`
cannot).

## Review-loop highlights

- **Depth-counted capture window** (Gate 1 polish): the dispatch window is a counter,
  not a bool ‚Äî nested dispatch can't close the outer window or discard its captured
  exception; the first exception wins and is never reset at nested-dispatch start.
- **handlerId overflow guard** (Gate 1 polish): `handlerId > int.MaxValue` ‚Üí rc 3
  BEFORE the `(int)` narrowing ‚Äî silent truncation could alias onto a live handler and
  dispatch the wrong event.
- **`retire()`** (Gate 2 polish): the recreation hazard was re-analyzed as *concurrent
  .NET entry* (old lane's in-flight dispatch vs replacement mount, plus old-lane frames
  delivered to a destroyed Activity's views); `retire()` drains the lane before a
  replacement starts ‚Äî a retired runtime is fully quiescent.
- **`StepIn(1)` cursor test** (Gate 3 polish): review caught that the cursor tests all
  stepped into child 0, where absolute and relative addressing coincide.
- **Truthful-comments sweep** (Gate 3 polish): comments claiming behavior the code
  doesn't have (or no longer has) were corrected across the diff ‚Äî e.g. the recreation
  contract, the mapper's stub-era KDoc, the probe-fate notes.

## Carryovers

**Phase 3.3+ (renderer/diff ‚Äî the five limits documented in-code at
`NativeWidgetTree._childOrderMap`, each safe while component trees are
diff-shape-stable like Hello, each breaks under richer diffs):**
- **(a) Order lists are append-only** ‚Äî `PrependFrame` appends regardless of the edit's
  `SiblingIndex` (mid-list inserts land at the end) and `RemoveFrame` never trims the
  list; after structural churn, cursor indices drift from Blazor's sibling positions.
- **(b) Component frames occupy no order slot** ‚Äî they take a diff sibling slot but the
  renderer skips them without allocating a node, so a child component interleaved
  between elements offsets every cursor index after it (DoD #8 blocker).
- **(c) `SetAttribute` still resolves through the batch-relative
  (componentId, frameIndex) sibling map**, not the cursor ‚Äî only valid for mount-shaped
  batches.
- **(d) `PrependFrame` under a poisoned cursor is dropped** ‚Äî the prepended subtree
  silently never renders (correct vs aliasing, but silent).
- **(e) The renderer never emits `DetachEventPatch`** ‚Äî `on*` RemoveAttribute flows out
  as `UpdatePropPatch(name, null)`, which hosts ignore; until 3.3 routes it, listeners
  are only ever *replaced* by re-attach, never detached. Consequence: a change
  re-attach stacks TextWatchers on the same EditText ‚Äî each fires, the extra dispatches
  carry stale handlerIds, absorbed by the rc-0 stale contract (benign today, waste +
  log noise under churn).

**Phase 3.3+ (other):**
- **Async-handler dispatch window** ‚Äî the rc-2 capture window and the synchronous
  contract assume sync handlers; `await` in a handler moves continuations off the
  dispatch thread and out of the window (in-code breadcrumb on the capture-window doc).
- **Dispatch-lane starvation watch** ‚Äî one lane is deliberate (renderer affinity); a
  slow handler blocks later events. Revisit if M3 components need concurrency.
- **focus/blur unwired** ‚Äî `BuildEventArgs` maps them to `FocusEventArgs` .NET-side;
  no Android listener attaches them.
- **`AppendChild` still TODO in `WidgetMapper`** (the mapper's last stub) ‚Äî pairs with
  DoD #10's emission decision.
