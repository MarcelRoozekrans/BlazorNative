# Phase 3.0 Design — NativeAOT-per-ABI Migration

**Date:** 2026-05-28
**Milestone:** M3 — P2: Real Apps Can Be Built
**Maps to:** [MILESTONE.md](../planning/MILESTONE.md) DoD #1 (runtime architecture decision)
**Predecessor evidence:** [runtime-architecture-eval.md](2026-05-27-phase-2.8-runtime-architecture-eval.md), [phase-2.3-design-revision.md](2026-05-26-phase-2.3-design-revision.md)

---

## Decision

**Runtime architecture for M3+ is NativeAOT-per-ABI**, dropping wasmtime + the .wasm artifact entirely.

This decision commits MILESTONE.md DoD #1 and supersedes Phase 2.8's "evaluate three options" framing.
Two candidates ruled out:

- **wasi-experimental (status quo).** Upstream is no longer actively developed. The three SDK gaps
  identified in Phase 2.3 (no custom `_WasmPInvokeModules`, no `allow-undefined` linker arg, no
  `wit-component` MSBuild bridge) remain unfixed and force the env-var bridge fallback for every new
  host import. M3 DoD #3 needs six more bridge exports; on wasi-experimental, each becomes an
  init-time-only, string-only, one-way env-var. No path to bidirectional events without leaving the
  WASI ABI.
- **componentize-dotnet (NativeAOT-LLVM under wasi-wasm).** A 1-day spike on 2026-05-28 RED'd at
  Gate 2. The renderer's `Bn*` wrapper layer is trim-unsafe: under any NativeAOT trim engine,
  `RenderTreeFrame.ElementName` returns null at runtime because the trimmer cannot statically prove
  the field's reachability through `[UnsafeAccessor]` accessors. M2 carried two trim warnings
  (IL2087 on `NativeRenderer.Mount<TComponent>`, IL2026 on `BlazorInterop.VerifyAccessors`) as
  "harmless" under Mono-AOT; under NativeAOT-LLVM they became silent runtime nulls. Spike concluded
  RED with no shipping path. Spike branch + design + plan + conclusion docs were discarded.

The componentize-dotnet RED surfaced an unavoidable fact: **the renderer trim-fragility wedge applies
to any NativeAOT path**, because NativeAOT-per-ABI uses the same trim engine. That wedge becomes the
prerequisite for this decision: Phase 3.0a closes it before any NativeAOT runtime work starts.

## Why NativeAOT-per-ABI

- **No WASI ABI ceiling.** WASI 0.2 is single-threaded. Mono-WASI has no async `Main`, no
  `Task.Delay`, and a non-inlining default dispatcher. NativeAOT on `linux-bionic-*` has real
  threads, real async, and the standard `System.Threading.Channels` surface.
- **One layer of native interop, not three.** Today: `.wasm` → wasmtime C-API → JNA → Kotlin.
  After Phase 3.0c: `.so` → JNA → Kotlin. Each host import becomes an `[UnmanagedCallersOnly]`
  export + a JNA callback. No wasmtime version compatibility games, no Scudo allocator crashes on
  `wasmtime_component_linker_delete`, no Phase 2.4b 1/N flake.
- **5× cold-start improvement** measured on equivalent codebases (Galaxy Note 10+ NativeAOT 271-331ms
  vs Mono-AOT 1.27-1.41s). M2's ~36s AVD cold-boot was dominated by wasmtime JIT + Mono init; both
  retire.
- **First-class typed C-ABI for the bridge.** M3 DoD #3's six deferred exports become direct
  `[UnmanagedCallersOnly]` functions with typed struct arguments. No JSON-as-env-var shoehorn. No
  WIT-bindgen toolchain hazards.
- **Bidirectional events fall out for free.** M3 DoD #2 wanted a long-running-Main shape so the .wasm
  process could receive event callbacks post-boot. NativeAOT eliminates the runtime-loop concern: the
  library is always loaded; event dispatch is just another C-ABI call.

## What we accept in exchange

- **No "single binary runs anywhere" property.** Three RIDs (win-x64 + linux-bionic-arm64 +
  linux-bionic-x64); a fourth for macOS shell when iOS lands in M4+.
- **Windows dev machine cannot cross-compile to Bionic directly** — WSL Ubuntu required as the build
  host for Android targets. Documented in setup.ps1. Windows-direct path stays available for the
  JVM dev loop (win-x64 `.dll`).
- **Microsoft NativeAOT for `linux-bionic-*` is "limited experience"** per Microsoft docs. Risk
  named in Section 6 below; mitigation is to validate Windows-direct first (3.0b Gate 1-2) before
  sinking time into WSL + cross-compile (Gates 3-5).
- **wasmtime CLI BootSmoke** (the .NET-side subprocess test) retires when the .wasm path leaves
  in 3.0c. JVM + Android instrumented tests provide equivalent coverage on the new runtime.

## Three-phase shape

**Phase 3.0a — Renderer trim safety** (on wasi-experimental, no runtime change)
**Phase 3.0b — NativeAOT runtime works** (JVM-desktop first, then Android, no renderer)
**Phase 3.0c — Native wire protocol + renderer + collapse** (replaces .wasm path)

Each phase is independently revertable. Each phase has gate-by-gate GREEN/RED criteria with a hard
per-gate timebox and the RED-on-ambiguity rule from the componentize spike retrospective. If 3.0a
RED's, 3.0b doesn't start. If 3.0b RED's, 3.0c doesn't start. Calendar ceiling across all three
phases: ~10-14 days.

## Phase 3.0a — Renderer trim safety

**Goal:** zero trim warnings + `HelloComponent` still renders, on the existing Mono-AOT /
wasi-experimental toolchain.

**Why first:** today's componentize-dotnet RED proved that the renderer's `Bn*` wrapper layer is
trim-unsafe. Same NativeAOT trim engine in NativeAOT-per-ABI → same failure. Fixing it on
wasi-experimental first means Phase 3.0b tests a single-variable hypothesis (toolchain swap),
not two compounded ones (toolchain swap + trim fix).

**Scope:**

- Add `<TrimmerSingleWarn>false</TrimmerSingleWarn>` to `BlazorNative.Renderer` +
  `BlazorNative.WasiHost`.
- `dotnet publish src/BlazorNative.WasiHost -c Release -r wasi-wasm` produces full warning list.
- Fix every renderer-touching warning. Patterns:
  - `NativeRenderer.Mount<[DynamicallyAccessedMembers(All)] TComponent>` on the generic parameter.
  - Every `Bn*` ref-struct wrapper that reaches a Blazor public field via `[UnsafeAccessor]` gets
    `[DynamicallyAccessedMembers(All)]` on its enclosing type's `[UnsafeAccessorType]` annotation.
  - `BlazorInterop.VerifyAccessors`: replace `Assembly.GetType(string)` reflection with `typeof()`
    for the finite set of Blazor types we probe, eliminating IL2026.
- Regression test `TrimSafetyTests.Mount_HelloComponent_ElementName_NotNull` locks in the fix at
  the .NET layer.

**Out of scope:** any NativeAOT work, any C-ABI design, anything touching the bridge layer.

**GREEN bar:** `dotnet publish` 0 warnings; `HelloComponent` renders on AVD via `WidgetMapperTest`
unchanged; existing M2 test surface unchanged (.NET 23/2, JVM 11, Android 19) + 3 new
`TrimSafetyTests`.

**Timebox:** 1-2 days, 3 gates.

**Rollback if RED:** revert the annotation commits. If the trim-fragility is deeper than annotations
can solve, **3.0b doesn't start**; the whole NativeAOT direction needs reconsideration.

## Phase 3.0b — NativeAOT runtime works

**Goal:** NativeAOT compiles `BlazorNative.NativeHost` (new csproj) to a native `.dll` (win-x64) and
`.so` (linux-bionic-arm64 + linux-bionic-x64). DI graph + platform-info struct + clean exit.
**No renderer mount, no `HelloComponent`.**

**Why no renderer:** single-variable hypothesis test. If the renderer's trim safety from 3.0a holds,
3.0b proves it on a real NativeAOT trim engine. If 3.0b RED's at the runtime level (compile error,
JNI mismatch, Bionic linker), we don't waste cycles diagnosing whether it's the toolchain or the
protocol redesign.

**Coexistence rule:** the existing `BlazorNative.WasiHost.csproj` + entire wasmtime/.wasm stack stays
intact through all of 3.0b. Everything new is additive. Only on 3.0c GREEN does the final cleanup
commit delete the WASM-era path.

### C-ABI surface (boot-only for 3.0b) — `Exports.cs`

```csharp
[StructLayout(LayoutKind.Sequential)]
public struct BlazorNativeInitOptions
{
    public IntPtr PlatformInfoOs;        // const char* — host-allocated UTF-8
    public int    PlatformInfoApiLevel;
    public IntPtr PlatformInfoNote;      // const char* — optional
}

[StructLayout(LayoutKind.Sequential)]
public struct BlazorNativeInitResult
{
    public int    Status;                // 0 = success
    public IntPtr ErrorMessage;          // const char* — set on Status != 0
    public IntPtr VersionString;         // const char* — static, never freed
}

[UnmanagedCallersOnly(EntryPoint = "blazornative_init")]
public static unsafe BlazorNativeInitResult Init(BlazorNativeInitOptions* opts);

[UnmanagedCallersOnly(EntryPoint = "blazornative_shutdown")]
public static void Shutdown();

[UnmanagedCallersOnly(EntryPoint = "blazornative_version")]
public static IntPtr Version();
```

**String ownership rule:** all input strings are caller-allocated, callee-borrowed (read into managed
`string` during the call, never retained). Result strings are static native memory (compile-time
literals from `Marshal.StringToHGlobalAnsi` at module load); host reads, never frees.

## Phase 3.0c — Native wire protocol + renderer + collapse

**Goal:** `HelloComponent` renders on Android via NativeAOT + native C-ABI struct callback for
frames. wasmtime + .wasm + `WasiHost.csproj` + WASI-shaped wire protocols all delete in the final
commit.

### New types — `PatchProtocolNative.cs`

```csharp
public enum BlazorNativePatchKind : int { CreateNode = 1, ReplaceText, RemoveNode, CommitFrame,
                                          UpdateProp, SetStyle }
public enum BlazorNativeNodeType : int { View = 1, Text, Button, Input, Image, Scroll, Picker }

[StructLayout(LayoutKind.Sequential)]
public struct BlazorNativePatch
{
    public BlazorNativePatchKind Kind;
    public int    NodeId;
    public int    ParentNodeId;          // -1 if root
    public BlazorNativeNodeType NodeType;
    public IntPtr Text;                  // const char* — utf-8, NULL if unused
    public IntPtr PropName;
    public IntPtr PropValue;
}

[StructLayout(LayoutKind.Sequential)]
public struct BlazorNativeFrame
{
    public IntPtr Patches;               // BlazorNativePatch* — native-allocated
    public int    PatchCount;
    public long   FrameSequence;
}

[UnmanagedFunctionPointer(CallingConvention.Cdecl)]
public delegate void FrameCallback(BlazorNativeFrame* frame);
```

**String ownership rule for the frame protocol:**
- All `IntPtr Text` / `PropName` / `PropValue` fields are native-owned, valid only for the duration
  of the callback. Kotlin copies into `String` immediately.
- Patches array is native-owned, valid only for the duration of the callback. Kotlin iterates
  synchronously.
- Per-frame arena allocated + released by the native side. No reference counting. No `Free`
  callback. YAGNI for 3.0c.

### Extended C-ABI exports

```csharp
[UnmanagedCallersOnly(EntryPoint = "blazornative_register_frame_callback")]
public static int RegisterFrameCallback(IntPtr fnPtr);

[UnmanagedCallersOnly(EntryPoint = "blazornative_register_event_callback")]
public static int RegisterEventCallback(IntPtr fnPtr);

[UnmanagedCallersOnly(EntryPoint = "blazornative_mount")]
public static unsafe int Mount(IntPtr componentNameUtf8);

[UnmanagedCallersOnly(EntryPoint = "blazornative_dispatch_event")]
public static unsafe int DispatchEvent(ulong handlerId, IntPtr argsJsonUtf8);
```

`blazornative_mount` does string→Type resolution. 3.0c registers `HelloComponent` via the existing
`[DynamicDependency]` chain. Phase 3.1+ generalizes via an analyzer-emitted mountable-component
registry.

### Renderer rewrite — `NativeRenderer.DispatchFrame`

Today's `Console.WriteLine($"[FRAME] {json}")` becomes:

```csharp
private unsafe void DispatchFrame(RenderFrame frame)
{
    if (_frameCallback == IntPtr.Zero) return;
    using var arena = FrameArena.Rent();
    var patches = arena.AllocPatches(frame.Patches.Count);
    for (int i = 0; i < frame.Patches.Count; i++)
        patches[i] = ToNativePatch(frame.Patches[i], arena);
    var nativeFrame = new BlazorNativeFrame
    {
        Patches = (IntPtr)patches,
        PatchCount = frame.Patches.Count,
        FrameSequence = Interlocked.Increment(ref _frameSeq),
    };
    var cb = (delegate* unmanaged[Cdecl]<BlazorNativeFrame*, void>)_frameCallback;
    cb(&nativeFrame);
    // arena disposes; native buffers reclaimed.
}
```

`FrameArena` is a small `ArrayPool<byte>`-backed util: patch array + short string pool. Avoids
per-frame GC pressure.

### Kotlin shell rewrite

`NativeBindings.kt` (new — replaces `WasmtimeBindings.kt`):

```kotlin
interface NativeBindings : Library {
    fun blazornative_init(opts: BlazorNativeInitOptions.ByReference): BlazorNativeInitResult.ByValue
    fun blazornative_shutdown()
    fun blazornative_version(): Pointer
    fun blazornative_register_frame_callback(callback: FrameCallback): Int
    fun blazornative_register_event_callback(callback: EventCallback): Int

    companion object {
        val INSTANCE: NativeBindings = Native.load("blazornative", NativeBindings::class.java)

interface FrameCallback : Callback {

```

`WidgetMapper.kt` rewires from sealed-class polymorphism to direct struct reads. String fields
copied during callback; patch array iterated synchronously before callback returns.

### Threading model

`Task.Delay` works. `async` methods inside `[UnmanagedCallersOnly]` exports can use `Task.Run`/
`await` freely. The Phase 2.4 `Thread.Sleep` + `#pragma warning disable BN0004` retires.
`InlineDispatcher` for `Mount<T>` (Phase 2.4 finding) becomes optional; don't change in 3.0c
(YAGNI); revisit when M3 DoD #2 ships.

## Risks

| Risk | Where it hits | Mitigation | If RED |
|---|---|---|---|
| **R1: `[UnsafeAccessor]` doesn't survive NativeAOT** | 3.0b Gate 1-2 | 3.0a annotations cover it; `[DynamicallyAccessedMembers(All)]` on enclosing type | RED 3.0b → retry with different accessor pattern (direct field via `[UnscopedRef]`) or stuck on wasi-experimental forever. |
| **R2: NativeAOT for `linux-bionic-*` is "limited experience"** | 3.0b Gate 4-5 | Validate `win-x64` first; sink WSL time only after toolchain proven | RED 3.0b Gate 4 → try `<IlcSystemModule>`/`<IlcCompileBclLink>` tweaks. If still RED, consider BFlat (third-party NativeAOT-like). |
| **R3: WSL friction in dev loop** | Day-to-day post-3.0c | Windows-direct path runs JVM dev loop without WSL | Document as M4 friction; not 3.0 blocker. |
| **R4: Per-frame arena overhead** | 3.0c Gate 2-4 | Arena `ArrayPool<byte>`-backed; alloc O(patches), release on callback return | If frame dispatch >2× M2 JSON path: log as 3.0d work, not RED. |
| **R5: Native string lifetimes wrong → SIGSEGV** | 3.0c Gate 2-3 | Ownership rule explicit; debug-tag field for ABI-drift detection | RED 3.0c Gate 2-3 → struct definition fix, retry. Recurrence = wider redesign. |
| **R6: NDK linker rejects .so** | 3.0b Gate 5 | NDK 26.3.11579264 supports Bionic ELF64; verify with `readelf -d` | RED → NDK upgrade or upstream NativeAOT cross-linker bug. |
| **R7: 3.0a annotations don't fix trim-fragility** | 3.0a Gate 2-3 | Baseline warnings captured Gate 1; each annotation removes a specific warning | RED → consider `TrimmerRootDescriptor` files; if even that fails, RED phase → stuck on wasi-experimental. |

### Cross-cutting

- **Analyzer scoping.** `BlazorNative.Analyzers` reattaches to `BlazorNative.NativeHost.csproj` via
  same `ProjectReference OutputItemType="Analyzer"`. Verified at 3.0b Gate 1.
- **BN0014** (async-handler on `NativeEvents`) — unchanged.
- **BN0004** (`Thread.Sleep` on WASI) — scope rework in 3.0c. Rename to
  `BN0004_BlockingSleepInAsyncPath`; trigger only inside `async` methods. Standalone
  `Thread.Sleep` in `[UnmanagedCallersOnly]` callee paths becomes legal.
- **WSL on this machine.** Recommend `blazornative-ubuntu` as explicitly-named distro to isolate
  from any existing WSL state. setup.ps1 Section 3 handles idempotently.

### Phase 3.0d / future

After 3.0c GREEN, M3 phases 3.1+ inherit a clean foundation:

- **3.1 — Six deferred mobile_bridge exports.** Direct C-ABI exports + Kotlin callbacks. Per-export
  ~50 lines + test.
- **3.2 — Bidirectional event flow (M3 DoD #2).** Event dispatch wired in 3.0c. 3.2 connects
  Android touch events to it. No runtime-loop change needed.
- **3.3 — `Bn*` component library (M3 DoD #4).** Pure .NET work; runtime-neutral.
- **M4 — production-shippable.** CI on Linux GitHub Actions runners builds Bionic targets cleanly
  (Microsoft NativeAOT Linux→Bionic is fully supported).

## Open implementation questions (deferred to Phase 3.0a writing-plans)

1. **`TrimmerRootDescriptor` files for `Microsoft.AspNetCore.Components` types** — needed if
   library-deep trim warnings can't be suppressed at our annotation level. Decide during 3.0a Gate 2
   execution.
2. **`BlazorInterop.VerifyAccessors` refactor strategy** — replace `Assembly.GetType(string)` with
   `typeof()`-keyed lookups (preferred, eliminates IL2026) or `[UnconditionalSuppressMessage]`
   (escape hatch).
3. **`Mount<T>` faulted-task masking carryover** (M3 inherited item from Phase 2.4) — fold into
   3.0a's annotation pass since both touch `Mount<T>`'s generic surface.

These resolve during implementation, not in this design.
